# 算法2.2 调权字段字典与实操流程（2026-03-29）

## 1. 目标

把当前匹配系统从“规则可解释”推进到“规则可校准”：

1. 先用对拍命令确认引擎输出稳定。
2. 再用 outcome 命令确认真实互动漏斗。
3. 最后导出 calibration 数据，做权重更新建议。

---

## 2. 三条核心命令

## 2.1 旧/新引擎对拍

```bash
php artisan app:dev:astro-shadow-compare --limit=100 --only-mismatch
```

用途：

- 看 canonical 改造是否引入异常漂移。
- 识别差异集中维度（`top_diff_dimensions`）。

输出：

- `docs/devlogs/ASTRO_SHADOW_COMPARE.md`
- `docs/devlogs/ASTRO_SHADOW_COMPARE.json`

## 2.2 行为漏斗统计

```bash
php artisan app:dev:pair-outcome-metrics --days=30
```

用途：

- 统计互选、首聊、24h 回复、7天持续互动、解释页查看率。
- 看产品真实互动质量变化。

输出：

- `docs/devlogs/PAIR_OUTCOME_METRICS.md`
- `docs/devlogs/PAIR_OUTCOME_METRICS.json`

## 2.3 调权数据集导出

```bash
php artisan app:dev:export-match-calibration --days=90
```

用途：

- 导出可用于离线调权的数据（CSV + JSON）。
- 支撑“rank_score 与真实互动结果”的相关性分析。

输出：

- `docs/devlogs/MATCH_CALIBRATION_DATASET.csv`
- `docs/devlogs/MATCH_CALIBRATION_DATASET.json`

## 3. 调权字段字典（核心）

## 3.1 基础键

- `match_id`: 匹配记录 ID
- `week_tag`: 周批次
- `created_at`: 匹配时间
- `user_a`, `user_b`: 双方用户 ID（已排序）

## 3.2 总分与聚合特征

- `display_score`: 展示分（用户可读）
- `rank_score`: 排序分（匹配排序主用）
- `overall_confidence`: 整体置信度
- `score_personality_total`
- `score_mbti_total`
- `score_astro_total`
- `score_bazi`
- `score_zodiac`
- `score_constellation`
- `score_natal_chart`

## 3.3 模块特征（按模块展开）

模块集合：

- `personality`
- `mbti`
- `bazi`
- `zodiac`
- `constellation`
- `natal_chart`
- `pair_chart`

每个模块都有：

- `m_<module>_score`
- `m_<module>_confidence`
- `m_<module>_degraded`（0/1）
- `m_<module>_weight`
- `m_<module>_verdict`

示例：

- `m_bazi_score`
- `m_bazi_confidence`
- `m_bazi_degraded`

## 3.4 行为结果标签（观测）

- `mutual_like`: 是否互选
- `first_message`: 是否出现首条私信
- `reply_24h`: 首条私信后 24h 内是否有对方回复
- `sustained_7d`: 7天内是否形成持续互动
- `explanation_view`: 是否查看过匹配解释页

## 3.5 训练标签（建议）

- `label_positive_reply24h`（当前等于 `reply_24h`）
- `label_positive_sustained7d`（当前等于 `sustained_7d`）

建议：

- 轻量版本先优化 `label_positive_reply24h`
- 稳定后再优化 `label_positive_sustained7d`

## 4. 一轮调权实操流程（建议每周一次）

## 步骤 1：对拍检查（防止引擎漂移）

1. 运行 `astro-shadow-compare`
2. 看 `any_diff_rate_pct`
3. 看 `top_diff_dimensions`

停止条件：

- 若差异显著上升，先修引擎/输入问题，不进入调权。

## 步骤 2：漏斗检查（确认当前业务表现）

1. 运行 `pair-outcome-metrics`
2. 重点看：
   - `reply_24h_rate_pct`
   - `sustained_7d_rate_pct`
   - `explanation_view_rate_pct`

## 步骤 3：导出校准数据

1. 运行 `export-match-calibration`
2. 把 CSV 发给顾问做离线分析（相关性/分桶提升）

## 步骤 4：给出“只改一小步”的权重建议

建议约束：

- 每次仅调整 1-2 个权重
- 单次改动幅度不超过 ±10%
- 保留前一版参数可回滚

## 步骤 5：灰度上线

- 先白名单/小流量
- 观察 3-7 天
- 再决定是否全量

---

## 5. 口径与注意事项

- `display_score` 用于解释展示，不建议直接用于排序调参。
- `rank_score` 才是排序主分，调权建议以它为主线。
- 对 `degraded=1` 的样本建议单独分组分析，避免污染主结论。
- MBTI 当前为轻量模型，不建议在核心排序中过度放大。
- 西占高精引擎尚未完全接管前，`natal_chart/pair_chart` 的权重应保持谨慎。

---

## 6. 输出给顾问的最小材料

每次至少提供这三份：

1. `ASTRO_SHADOW_COMPARE.md`
2. `PAIR_OUTCOME_METRICS.md`
3. `MATCH_CALIBRATION_DATASET.csv`

配一句结论：

- “本周目标是提升 reply_24h，不牺牲 sustained_7d；请给出最多 2 个权重调整建议和预计影响方向。”

## 7. 一键执行脚本（已固化）

位置：`scripts/run_astro_calibration_cycle.ps1`

默认执行：

```powershell
./scripts/run_astro_calibration_cycle.ps1
```

常用参数：

```powershell
# 只跑 14 天样本，限制导出 2000 条
./scripts/run_astro_calibration_cycle.ps1 -OutcomeDays 14 -CalibrationDays 14 -CalibrationLimit 2000

# 指定 week_tag
./scripts/run_astro_calibration_cycle.ps1 -WeekTag 2026W13

# 仅预览命令，不实际执行
./scripts/run_astro_calibration_cycle.ps1 -DryRun
```

自动日志：

- `docs/devlogs/CALIBRATION_CYCLE_LOG.md`

说明：

- 脚本顺序固定：对拍 -> 漏斗 -> 校准导出。
- 任一步失败会返回非 0，便于 CI/批处理接入。
