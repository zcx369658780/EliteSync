# 算法 2.3 版本开发规划（可直接下发 Codex）

> 版本目标：在 2.2 已完成“许可证门禁、八字服务端 canonical、MBTI 降权、confidence policy、对拍与调权工具链”的基础上，推进到 **2.3：解释可信、前后端一致、可灰度上线、可回归验收、可持续调权** 的稳定工程版本。
>
> 本文定位：
> 1. 给 Codex 的直接执行规划
> 2. 给项目负责人/顾问的监督清单
> 3. 给后续 2.4 的升级预留技术接口

---

# 0. 2.3 的核心判断

## 0.1 当前 2.2 已达成的基础
- 八字已服务端真源化
- 许可证门禁已建立
- MBTI 已降为轻量因子
- confidence policy 已落地
- 对拍 / 导出 / 调权 / 漏斗脚本已具备
- 西占仍处于受控过渡态，尚未完成“高精 + 合规 + 稳定生产”闭环

## 0.2 2.3 不追求的事
1. **不在本轮强行完成 Swiss 商业授权接入**
2. **不在本轮把所有玄学模块做成极复杂黑箱模型**
3. **不在样本不足时激进重调大权重**
4. **不继续扩写旧近似算法**

## 0.3 2.3 追求的结果
1. **引擎状态 → 置信度 → 前端文案 完全一致**
2. **解释层成为模板化、可回归、可验收的系统**
3. **西占路线做成正式架构决策，不再模糊悬置**
4. **低样本下的调权机制稳定，不因少量样本剧烈震荡**
5. **工程上形成“可灰度、可回滚、可观测”的产品态**

# 1. 2.3 总体目标与验收口径

## 1.1 总体目标
将 2.2 的“工程中间态”推进为“可控产品态”。

## 1.2 总验收标准（必须全部满足）
1. **任何模块只要不是正式 canonical 或输入不完整，就不能乱发高置信 / 证据强**

5. **所有权重调整都能在回归样例与周报里追踪前后变化**
6. **出现异常时可一键降级到旧规则或关闭单模块影响**

# 3. 主线 A：引擎状态 → 置信度 → 前端展示一致性改造

## 3.1 目标
解决“后端其实还是过渡态，前端却像正式高精态一样说话”的问题。

## 3.2 必须新增的标准字段
所有玄学模块统一输出以下元字段：

```json
{
  "engine_source": "lunar_php | legacy_estimate | astronomy_engine | swiss_candidate | manual_fallback",
  "engine_mode": "canonical | legacy | fallback | hybrid",
  "data_quality": "exact_time+location | exact_time_only | date_only | partial_unknown",
  "precision_level": "high | medium | low | estimated",
  "confidence": "high | medium | low",
  "confidence_reason": ["exact_birth_time", "location_missing", "legacy_engine", "partial_input"],
  "display_guard": {
    "allow_high_confidence_badge": true,
    "allow_strong_evidence_badge": false,
    "allow_precise_wording": false
  }
}
```

## 3.3 Badge 与文案规则重写

### 规则 1：高置信 badge 发放门槛
必须同时满足：
- `engine_mode = canonical`
- `precision_level != estimated`
- `data_quality` 至少达到 `exact_time_only`
- 非 fallback / legacy
- 模块自身证据条数达标

### 规则 2：证据强 badge 发放门槛
- 核心证据数量 >= 阈值
- 核心证据不得全部来自低精度估算
- 不允许由单一弱证据“顶上去”

### 规则 3：前端措辞分层
- `high confidence`：可用“结论稳定性较高”
- `medium confidence`：可用“基于较完整信息推断”
- `low / estimated`：必须写“仅供参考 / 基于部分信息估算”

### 规则 4：legacy / fallback 的强约束
以下情况禁止显示“高置信”：
- `legacy_estimate`
- `date_only`
- `partial_unknown`
- `manual_fallback`

## 3.4 Raw tag 映射层
新增：
- `EvidenceTagMapper`
- `EvidenceTagDictionary`

### 目标
将如下内部 tags：
- `moon_sync_high`
- `sun_direction_sync`
- `wu_xing_structure`
- `bazi_branch_same`

统一映射为：
- 情绪节奏较同步
- 相处方向较一致
- 五行结构互补
- 地支节律存在共振

### 要求
- 后端保留 raw tag 供回归与审计
- API 额外输出 `display_tags`
- 前端只消费 `display_tags`

## 3.5 交付物
- `app/Support/ExplanationMetaBuilder.php`
- `app/Support/EvidenceTagMapper.php`
- `config/display_guard.php`
- `docs/confidence_badge_rules_2_3.md`

## 3.6 验收
- legacy 模块在 UI 上不再出现“高置信”误导
- API 可追溯 badge 来源
- raw tag 在用户界面完全消失

# 4. 主线 B：解释层模板化与回归样例体系

## 4.1 目标
让 explanation 不再是“动态拼字串”，而是 **可维护、可比对、可回归** 的模板系统。

## 4.2 统一解释协议
每个模块统一输出 4 层：

```json
{
  "summary": "一句话结论",
  "process": ["形成原因 1", "形成原因 2"],
  "risks": ["风险点 1", "风险点 2"],
  "advice": ["建议 1", "建议 2"],
  "core_evidence": ["..."],
  "supporting_evidence": ["..."],
  "confidence": "medium",
  "priority": "top3"
}
```

## 4.3 模块级模板要求

### 八字匹配
必须说明：
- 偏互补还是偏同频
- 更偏生活节律、情绪支持，还是长期稳定
- 风险点是否来自五行失衡、节律错位、信息不完整
- 给出“适合怎样相处”的建议

### 星盘匹配 / natal_compatibility
必须说明：
- 当前结论更多来自太阳/月亮/上升哪一层
- 哪些因素是核心，哪些只是辅助
- 输入信息不完整时必须降级措辞

### synastry
必须说明：
- 更偏吸引互动、情绪同步、表达协调，还是冲突管理
- 风险是易热易冷、沟通错位、节奏不一致还是长期消耗

### composite_like / 男女合盘
必须说明：
- 这是“关系气质/关系走向”层，不等于单人本命盘
- 当前结论偏初期互动还是长期稳定
- 若输入信息不充分，必须主动提示结论稳定性受限

### MBTI / 人格问卷
必须说明：
- 该结论只是沟通风格参考，不代表决定性结果
- 轻量题结果必须限制措辞强度

## 4.4 模板引擎结构建议
- `ExplanationTemplateRegistry`
- `ExplanationComposer`
- `ExplanationSnapshotExporter`
- `ExplanationRegressionRunner`

## 4.5 回归样例制度
至少建立 **20 组固定样例**，覆盖：
- 完整出生信息 vs 缺失出生时刻
- 高分但高风险
- 中分但长期稳定
- 八字强 / 西占弱
- 西占强 / 八字弱
- MBTI 强 / 玄学弱
- 模块间结论冲突
- fallback / legacy / date_only 场景

### 每组样例必须保存
- 输入快照
- 模块分
- badge
- explanation 全量输出
- 版本号
- 对比结果

## 4.6 交付物
- `docs/explanation_template_spec_2_3.md`
- `tests/Fixtures/explanations/*.json`
- `scripts/run_explanation_regression.ps1`
- `reports/explanation_snapshot_diff/*.md`

## 4.7 验收
- 每次规则调整都能看到 explanation 前后差异
- 不再出现“分高但解释语义空泛”
- 模板可复用，不依赖单点手工写文案

# 5. 主线 C：西占 canonical 正式决策与过渡策略

## 5.1 目标
不再让西占一直处于“未来再说”的模糊状态。

## 5.2 必须产出 ADR 文档
新建：
- `docs/adr_western_canonical_route.md`

## 5.3 ADR 至少比较 3 条路线

### 路线 A：购买 Swiss 商业授权，服务端正式接管
**优点**
- 技术口径与产品表达一致
- 可把星盘/合盘做成长期卖点
- 精度和解释层更稳定

**缺点**
- 成本增加
- 需要法务与采购流程
- 实施周期不确定

### 路线 B：继续闭源安全模式，不启用 Swiss 生产
**做法**
- 保持 legacy / estimate 为过渡
- 用 Astronomy Engine 做数值辅助验证
- 下调西占模块表达强度

**优点**
- 合规风险小
- 工程上最稳

**缺点**
- 西占长期只能做轻量模块
- 产品卖点受限

### 路线 C：临时下调西占权重，等待授权后升级
**做法**
- 在 ranking 和 display 上分层降权
- 保留接口结构不变
- 拿到授权后平滑替换 canonical

**优点**
- 兼顾当前可上线和未来升级

**缺点**
- 需要更严格的文案与 badge 约束

## 5.4 2.3 默认建议
若短期拿不到商业授权：
- **默认执行路线 C**
- 保留 WesternEngine 结构
- 继续禁用 AGPL 生产依赖
- 下调西占权重与前端高置信文案
- 用 Astronomy Engine 做验证而非直接宣称“正式高精星盘”

## 5.5 需要同步的工程动作
- `western_natal.php` 增加 `policy_mode`
- `matching.php` 增加西占权重 profile
- `display_guard.php` 绑定 `policy_mode`

## 5.6 交付物
- `docs/adr_western_canonical_route.md`
- `config/western_policy.php`
- `docs/western_rollout_strategy_2_3.md`

## 5.7 验收
- 文档给出明确推荐路线，不能只列 options 不决策
- 前端措辞、权重、badge 与路线一致
- 未来切 Swiss 时不改 API 契约

# 6. 主线 D：低样本稳健调权与实验约束

## 6.1 目标
避免“用户样本少、权重乱跳、系统今天一个口径明天一个口径”。

## 6.2 调权原则
1. **周维度调整，不做日级大幅波动**
2. **单周权重变动上限 ≤ 10%**
3. **低样本模块使用 shrinkage / 平滑，不直接跟原始转化率走**
4. **display_score 和 rank_score 分离**
5. **保留固定对照 bucket，不跟随所有实验变化**

## 6.3 必须区分的分数

### display_score
- 用于展示
- 更关注用户理解和解释层稳定
- 不应频繁大波动

### rank_score
- 用于排序
- 可受实验与转化反馈影响
- 必须可观测、可回滚

## 6.4 样本不足时的推荐做法
- 模块级最小样本阈值
- 未达阈值时，只允许做提示，不允许自动放大该模块权重
- 使用 Beta/Binomial 或简单经验平滑进行保守更新
- 高噪声模块只允许小步修正

## 6.5 建议新增指标
### 曝光层
- 模块曝光次数
- 高分对数量
- badge 发放比例

### 行为层
- 查看 explanation 率
- 发首条消息率
- 24h 回复率
- 7 天持续互动率
- 互相喜欢率

### 质量层
- 解释被点击但无后续行为的比例
- 高分但快速冷却的比例
- 低置信模块被高频展示的比例

## 6.6 实验约束
- 新 profile 只在白名单 / 小流量启用
- 保持一个 stable bucket 永不跟实验变化
- 周报必须展示：
  - 本周 profile
  - 上周 profile
  - 模块权重变化
  - 行为结果变化
  - 是否超过波动阈值

## 6.7 交付物
- `docs/low_sample_tuning_policy_2_3.md`
- `scripts/check_weight_change_guard.ps1`
- `reports/calibration_guardrail_weekly.md`

## 6.8 验收
- 权重变动有硬限制
- 样本不足时不会出现过度调权
- 展示分和排序分不再混用

# 7. 主线 E：工程质量门禁、灰度、监控、回滚

## 7.1 目标
让 2.3 成为可上线、可维护、可回退的版本，而不是“顾问看起来不错”的版本。

## 7.2 发布前门禁
### 必须通过的检查
- 许可证门禁通过
- explanation 回归样例通过
- badge 规则检查通过
- 关键 API snapshot diff 无异常
- weight guardrail 检查通过
- 前端 display_tags 检查通过

## 7.3 灰度策略
### 灰度维度
- 用户白名单
- 平台白名单
- 城市白名单
- profile version
- 模块级 enable / disable

### 推荐灰度顺序
1. 内部测试账号
2. 顾问验证账号
3. 小流量真实用户
4. 分城市 / 分端放量
5. 全量上线

## 7.4 回滚策略
必须支持：
- 单模块关闭
- 单 badge 规则回退
- 单 profile 权重回退
- explanation 模板回退

## 7.5 监控看板
新增 dashboard 维度：
- 各模块 badge 发放比例
- 高置信比例
- legacy / fallback 占比
- explanation 点击率
- 模块间分数冲突率
- 高分 pair 后续转化率

# 9. 模块级开发要求

## 9.1 八字模块
### 本轮重点
- 不继续扩写历法底层
- 加强解释层稳定性
- 将十神摘要 / 五行互补 / 节律同步固定成模板化证据输出

### 新要求
- 区分“结构互补”和“节律同步”
- 不用神秘化词汇堆砌解释
- 风险层必须指出：过强互补未必代表低摩擦

## 9.2 属相模块
### 本轮重点
- 继续维持低复杂度、低权重、低措辞强度
- 只作为辅助模块存在

### 新要求
- 生肖回退公历年时必须降置信
- 不允许单独把属相结论说得过重

## 9.3 星座模块
### 本轮重点
- 明确其是轻量展示层
- 不与完整西占星盘混用

### 新要求
- 星座只解释基础表达倾向，不承担“高精关系判断”角色

## 9.4 natal_compatibility
### 本轮重点
- 与 synastry / composite_like 明确分层
- 解释时说清楚结论来自太阳/月亮/上升哪一层

## 9.5 synastry
### 本轮重点
- 解释“互动性”和“摩擦点”
- 不直接等同长期关系稳定性

## 9.6 composite_like / 男女合盘
### 本轮重点
- 重命名概念说明：
  - 可以保留前端名称“男女合盘”
  - 但解释层必须写明它衡量的是“关系气质/关系走向”

### 新要求
- 输出必须包含：
  - 该分更偏初期吸引还是长期稳定
  - 哪些项是核心，哪些项是辅助
  - 输入不完整时主动说明结论受限

## 9.7 人格问卷 / MBTI
### 本轮重点
- 保持轻量化
- 不让 MBTI 抢解释主导权

### 新要求
- 结果只解释沟通风格，不给“命定关系”口气

# 11. 你监督 Codex 时，重点看什么

## 第一层：方向是否跑偏
只问 3 个问题：
1. 有没有继续偷偷扩写旧近似算法？
2. 有没有让前端说得比底层实际更重？
3. 有没有把 low sample 的临时结果当长期规律？

## 第二层：工程是否可靠
1. 有无回归样例？
2. 有无灰度方案？
3. 有无回滚方案？
4. 有无周报和 diff？

## 第三层：解释是否可信
1. 解释是否清楚区分结论/过程/风险/建议？
2. 是否仍有内部 tag 泄露到 UI？
3. 是否存在“低置信却高口气”的情况？
