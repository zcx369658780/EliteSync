# 2.4 算法与数据接口总览

日期：2026-03-30  
用途：作为 2.5 版本规划、接口对齐、字段收敛、兼容层清理的统一基线。

## 1. 2.4 当前算法总览

2.4 的核心目标不是继续扩写旧近似算法，而是把现有算法收敛为：

- 服务端 canonical 真源
- 客户端只做输入、展示、缓存、降级提示
- 解释层与置信度门禁统一
- 兼容层可控保留，方便后续删除

### 1.1 匹配总结构

当前匹配由三层组成：

1. 硬性过滤
2. 核心评分
3. 软惩罚与公平调整

#### 硬性过滤

- 仅同城匹配：`city` 必须一致
- 仅异性匹配：`gender` 互补
- 关系目标冲突过滤：`relationship_goal` 不兼容则直接过滤
- 近期重复曝光控制：已配对组合按周/批次过滤

#### 核心评分

当前核心权重：

- 性格匹配：50%
- MBTI/性格问卷：15%
- 玄学总分：35%

公式：

`core_total = personality * 0.50 + mbti * 0.15 + astro_total * 0.35`

#### 软惩罚与调整

- 关系目标部分不一致惩罚
- 年龄差调整
- 同城加成
- MBTI 字母级微调
- 曝光公平性调整

## 2. 2.4 玄学算法结构

2.4 中玄学模块已经统一为可解释、可降级、可门禁的结构。

### 2.1 八字

- 角色：`result` 层，偏长期稳定和结果层解释
- 主来源：`user_astro_profiles.bazi`
- 兜底：`users.private_bazi`
- 置信度会随出生信息完整度下降

解释重点：

- 五行互补
- 长期节律协调
- 结果层稳定性
- 缺少完整资料时给出保守估计

### 2.2 属相

- 角色：`bridge` 层
- 来源：由年柱地支派生，不再按公历年份硬算
- 风险关系：六合、三合、冲、刑、害、同属相、普通关系

解释重点：

- 说明何为六合 / 三合 / 冲刑害
- 输出“为什么匹配 / 为什么不匹配”
- 属相不作为唯一决定因子，只是桥接层

### 2.3 星座

- 角色：`process` 层
- 来源：太阳星座
- 主要表达：互动节奏、表达方式、推进顺滑度

解释重点：

- 同元素 / 互补元素 / 张力元素
- 不做长期结果的单独决定项

### 2.4 星盘

- 角色：`process` 层
- 当前形态：`western_lite`
- 不承诺高精 canonical 生产级结论
- 仅提供轻量、可回退、可解释的辅助能力

解释重点：

- 情绪回应
- 推进节奏
- 上升/月亮/太阳的过程信号
- 数据不完整时明确降级

### 2.5 性格测试 / MBTI

- 前台文案已经统一为“性格测试 / 性格特征”
- 后端兼容键仍保留 `mbti`
- canonical persisted source：`users.public_mbti`
- 历史记录：`mbti_attempts`

解释重点：

- 当前只作为轻量因子，不再作为核心排序模块
- 结果置信度保持中低，不输出过强结论

## 3. 解释层统一契约

匹配解释最终输出采用统一结构：

- `match_reasons`
- `explanation_blocks`
- `compatibility_sections`
- `module_explanations`

常见字段：

- `key`
- `label`
- `layer`
- `score`
- `weight`
- `confidence`
- `confidence_tier`
- `confidence_reason`
- `engine_mode`
- `display_guard`
- `degraded`
- `degrade_reason`
- `highlights[]`
- `risks[]`

### 3.1 layer 含义

- `result`：结果层，偏长期稳定
- `bridge`：桥接层，偏关系类型判断
- `process`：过程层，偏互动推进与情绪同步

### 3.2 display_guard

用于控制展示强度的门禁字段，常见子项：

- `allow_high_confidence_badge`
- `allow_strong_evidence_badge`
- `allow_precise_wording`

### 3.3 confidence 规则

置信度受以下因素影响：

- 数据完整度
- 是否降级估算
- 是否 canonical
- 模块类型
- 引擎模式

## 5. 关键 API 入口

### 5.1 版本检查

- `GET /api/v1/app/version/check`

用途：

- 获取当前平台最新版本
- 返回下载地址、更新日志、强更标记

### 5.2 基础资料

- `GET /api/v1/profile/basic`
- `POST /api/v1/profile/basic`

用途：

- 读写用户基础资料
- 当前承载：昵称、生日、性别、城市、婚恋目标、生肖等基础字段

### 5.3 星盘 / 八字

- `GET /api/v1/profile/astro`
- `POST /api/v1/profile/astro`

用途：

- 读取 / 保存星盘和八字结果
- 当前 canonical 主来源是 `user_astro_profiles`

### 5.4 性格测试

- `GET /api/v1/profile/mbti/quiz`
- `POST /api/v1/profile/mbti/submit`
- `GET /api/v1/profile/mbti/result`

用途：

- 获取性格测试题
- 提交答题结果
- 读取最新性格结果

### 5.5 问卷

- `GET /api/v1/questionnaire/questions`
- `POST /api/v1/questionnaire/answers`
- `POST /api/v1/questionnaire/submit`（兼容）
- `POST /api/v1/questionnaire/draft`（兼容）

用途：

- 获取题库
- 保存作答
- 兼容旧客户端

### 5.6 匹配

- `GET /api/v1/matches/current`
- `POST /api/v1/matches/confirm`
- `GET /api/v1/matches/history`
- `GET /api/v1/matches/{targetUserId}/explanation`

用途：

- 当前匹配结果
- 交互确认
- 历史匹配
- 展开完整解释

### 5.7 消息

- `POST /api/v1/messages`
- `GET /api/v1/messages`
- `POST /api/v1/messages/read/{messageId}`

用途：

- 发送、拉取、已读控制

## 6. 单一真值与兼容策略

### 6.1 当前唯一真值

1. MBTI / 性格测试结果  
   - 主真值：`users.public_mbti`
   - 历史：`mbti_attempts`

2. 八字 / 星盘 / 出生信息  
   - 主真值：`user_astro_profiles`
   - 镜像：`users.private_*`

3. 生肖  
   - 必须由年柱地支派生
   - 不允许仅按公历年份硬算

4. 西占  
   - 当前默认：`western_lite`
   - canonical 高精路线只在后续授权条件满足后再切换

### 6.2 兼容策略

- 旧客户端路径保留到 2.5 前
- 用户可见文案逐步去 MBTI 化，转为“性格测试 / 性格特征”
- 历史报告、调试文档、重复说明逐步进入 `deprecated` / `archive`

---

## 7. 下一版规划建议可直接继承的结论

2.5 之后的规划建议沿用以下边界：

- 玄学模块继续保持服务端真源
- 前端只做展示和缓存，不再承担 canonical 计算
- 所有新算法先过许可证门禁
- 所有新字段先定义唯一真值，再决定是否镜像
- 所有解释模块都必须给出 `confidence`、`engine_mode`、`display_guard`
- 所有灰度 / 回滚都必须有演练证据
