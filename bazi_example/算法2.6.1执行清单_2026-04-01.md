# 算法 2.6.1 执行清单

> 基于《算法 2.6.1 修改方案草案》拆分的可执行清单。
>
> 本轮目标：
> - 把“出生时间”继续固化为个人资料真值。
> - 把“出生地 / 经纬度”真正接入八字、紫微、西占核心计算。
> - 继续保持前端以服务端为真源，减少 session / snapshot 的覆盖风险。

---

## P0：规则定稿与基线冻结

### 目标
- 明确出生地对八字、紫微、西占的具体影响规则，避免“保存了但算法定义不清”。

### 任务
- [ ] 确认出生地对八字的影响方式：
  - [ ] 方案 A：先映射时区，再做真太阳时修正
  - [ ] 方案 B：直接基于经度修正出生时间
  - [ ] 方案 C：仅作为辅助权重，不改主结果
- [ ] 明确出生地对紫微的影响方式：
  - [ ] 是否影响命宫 / 身宫
  - [ ] 是否影响宫位主题
  - [ ] 是否只影响精度 / 置信度
- [ ] 明确出生地对西占的影响方式：
  - [ ] 是否影响 ASC / 宫位 / 真太阳时
  - [ ] 是否影响相位或盘面完整度
- [ ] 冻结当前 2.6 已通过状态，作为 2.6.1 输入基线。

### 验收
- [ ] 顾问确认出生地参与三大算法的规则口径。
- [ ] 当前 2.6 状态可作为 2.6.1 起点。

---

## P1：八字真太阳时链路

### 目标
- 让出生地真正影响八字结果，而不是只影响置信度。

### 任务
- [ ] 新增或收紧“出生地 -> 真太阳时”转换规则。
- [ ] 让 `birth_place / birth_lat / birth_lng` 参与八字核心计算。
- [ ] 调整 `LunarPhpBaziEngine`，让输出结果随出生地变化而变化。
- [ ] 保留当前 `birthday / birth_time / gender` 作为核心基础输入。
- [ ] 设计 fallback 逻辑：
  - [ ] 缺少地点时如何降级
  - [ ] 缺少经纬度时如何处理
  - [ ] 失败时是否沿用旧值或返回低精度

### 关键文件
- [ ] `services/backend-laravel/app/Services/LunarPhpBaziEngine.php`
- [ ] `services/backend-laravel/app/Services/BaziCanonicalService.php`
- [ ] `services/backend-laravel/app/Http/Controllers/Api/V1/ProfileController.php`
- [ ] `services/backend-laravel/app/Http/Controllers/Api/V1/AstroProfileController.php`
- [ ] 如需要，新增地点/真太阳时转换服务

### 验收
- [ ] 同一生日 / 出生时间，不同出生地会导致八字发生可观察变化。
- [ ] `true_solar_time` 随出生地变化而变化，或至少能解释为修正结果。
- [ ] 不允许只改 `confidence`、`notes`，八字本体不变。

## 最终交付标准

只有同时满足以下条件，2.6.1 才算完成：

1. 出生时间在资料页稳定可见、可保存、可回显。
2. 出生地真正影响八字、紫微、西占中的至少核心一项。
3. 编辑页和星盘页都只把服务端作为真源。
4. 百度地点搜索继续可用，异常时明确降级。
5. 测试能证明“改输入 -> 算法结果变化”。
