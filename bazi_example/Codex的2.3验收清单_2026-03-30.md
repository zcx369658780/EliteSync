# Codex 2.3 验收清单

> 用途：用于验收算法 2.3 是否真正完成“主体落地”，而不是只完成了文档整理或局部开发。
>
> 适用对象：Codex、项目负责人、顾问验收。
>
> 使用方式：逐项核对；凡标注为 **必须** 的项，缺任意一项均不建议判定为“2.3 已完成”。

---

# 1. 验收结论分级

## A. 通过（Pass）

满足以下条件：

* 所有“必须项”均已提交
* 所有高优先级红线均已关闭
* 验收样例、灰度、回滚至少各完成一次可复核演示
* 前端展示与后端真实状态一致

## B. 有条件通过（Pass with fixes）

满足以下条件：

* 主体功能已落地
* 仍存在少量收尾项，但不影响真实口径与上线安全
* 有明确修复清单和截止时间

## C. 不通过（Fail）

出现以下任一情况：

* 缺少关键验收材料
* 仍存在“低精度却高口径展示”问题
* 无法证明 explanation 可回归、可比对
* 无灰度或回滚能力
* 西占路线仍停留在“口头讨论”，没有正式 ADR

# 3. 核心验收项

---

## 3.1 显示保护层验收

### 目标

确认“引擎状态 → 置信度 → 前端展示”已经真正打通。

### 必查项

* [ ] 每个玄学模块都输出 `engine_source`
* [ ] 每个玄学模块都输出 `engine_mode`
* [ ] 每个玄学模块都输出 `data_quality`
* [ ] 每个玄学模块都输出 `precision_level`
* [ ] 每个玄学模块都输出 `confidence`
* [ ] 每个玄学模块都输出 `confidence_reason`
* [ ] API 中能追溯 badge 的来源逻辑
* [ ] 前端 badge 与 API 字段一致，不靠前端猜测补逻辑

### 核验问题

1. `legacy_estimate` 是否还会显示“高置信”？
2. `date_only` 是否还会显示“证据强”？
3. `partial_unknown` 是否还会使用强结论措辞？
4. `fallback` 是否被正确降级？

### 通过标准

* 任一低精度/退化路径都不能误发高置信或强结论
* 前端与后端口径一致

---

## 3.2 Raw tag 映射验收

### 目标

确认用户界面不再暴露内部工程 tag。

### 必查项

* [ ] API 同时保留 raw tags 与 display tags
* [ ] 前端只消费 display tags
* [ ] 用户页面不再出现 `moon_sync_high`、`wu_xing_structure`、`bazi_branch_same` 等内部 tag
* [ ] 有统一的 tag 字典或映射层

### 通过标准

* 用户界面完全不出现 raw tag
* display tag 可读、自然、语义稳定

---

## 3.3 Explanation 模板系统验收

### 目标

确认 explanation 已从“拼字串”升级为“模板系统”。

### 必查项

* [ ] 所有模块统一输出四层结构：`结论 / 过程 / 风险 / 建议`
* [ ] explanation 由模板系统生成，而非零散字符串拼接
* [ ] 有模板注册或管理机制
* [ ] explanation 支持 snapshot 导出
* [ ] explanation 支持 diff 对比

### 核验问题

1. 八字模块是否能稳定输出四层解释？
2. 星盘/本命盘模块是否能说明结论来自哪一层？
3. synastry 是否明确是“互动关系”而不是“长期关系总判决”？
4. 男女合盘 / composite_like 是否明确说明它衡量的是“关系气质/关系走向”？
5. MBTI/性格测试是否明确只是辅助判断？

### 通过标准

* explanation 结构统一
* 任意规则调整后都可做前后 diff
* 不再出现“分数有了，但解释空泛或语义失真”的情况

---

## 3.4 回归样例验收

### 目标

确认 explanation 和展示规则可回归，不会改一次坏一次。

### 必查项

* [ ] 至少 20 组固定样例
* [ ] 样例覆盖完整输入与缺失输入场景
* [ ] 样例覆盖高分高风险、中分稳定、模块冲突、fallback 场景
* [ ] 每组样例都保留输入快照、模块分、badge、完整 explanation、版本号
* [ ] 有 snapshot diff 结果

### 通过标准

* 改规则前后，能明确知道哪些 explanation 发生变化
* 变化可解释、可审计

---

## 3.5 西占路线 ADR 验收

### 目标

确认西占 canonical 路线已经正式决策，而不是继续模糊悬置。

### 必查项

* [ ] ADR 文档存在
* [ ] ADR 至少比较 3 条路线
* [ ] ADR 明确给出推荐路线
* [ ] ADR 写清许可证 / 商业授权 / 工程影响
* [ ] 配置与文案已经和 ADR 结论对齐

### 核验问题

1. 现在的默认路线到底是哪条？
2. 若未启用 Swiss 商业授权，前端是否已经同步降调表达？
3. 未来若切换更高精引擎，是否可不改 API 契约？

### 通过标准

* ADR 不是资料汇总，而是正式决策文档
* 产品口径、权重、badge、配置与 ADR 一致

---

## 3.6 低样本调权守门验收

### 目标

确认调权不是“拍脑袋”或“小样本乱跳”。

### 必查项

* [ ] 有明确的权重变动上限
* [ ] display_score 与 rank_score 已区分
* [ ] 有 stable bucket 或对照 bucket
* [ ] 样本不足时有平滑/保守更新机制
* [ ] 有 weekly guardrail 报告

### 核验问题

1. 单周权重变动是否受控？
2. 样本不足时是否仍会激进调权？
3. display_score 是否还和 rank_score 混用？

### 通过标准

* 权重变动可解释、可追踪、可回退
* 没有因为少量样本导致系统大幅波动

---

## 3.7 灰度与回滚验收

### 目标

确认 2.3 不是“只能一次性硬上线”的版本。

### 必查项

* [ ] 有灰度计划文档
* [ ] 有回滚 playbook
* [ ] 已完成至少一次灰度演练
* [ ] 已完成至少一次回滚演练
* [ ] 支持单模块关闭
* [ ] 支持单 profile 回退
* [ ] 支持 explanation 模板回退
* [ ] 支持 western policy 回退

### 通过标准

* 出问题能靠配置先止血
* 不需要立刻改代码才能回退

---

## 3.8 前端一致性验收

### 目标

确认用户看到的内容与后端真实状态一致。

### 必查项

* [ ] 匹配详情页不会再出现纯灰空页
* [ ] 所有模块至少有稳定摘要
* [ ] badge、风险标签、解释块与 API 返回一致
* [ ] 性格测试文案已替换旧 MBTI 强口径
* [ ] 低置信模块已显示“仅供参考 / 基于部分信息估算”等提示

### 通过标准

* 前端不夸大、不漏显、不乱解释

# 5. 建议的验收流程

## 第一步：看文档是否齐

先核对第 2 节“必交材料清单”。

## 第二步：看 API / 配置是否真的接通

抽查 3 个模块：

* 八字匹配
* 星盘匹配
* 男女合盘 / composite_like

重点看：

* `engine_mode`
* `confidence`
* `display_guard`
* `display_tags`
* explanation 四层结构

## 第三步：看前端是否和 API 一致

抽查真实页面或测试环境截图，核对：

* badge
* 摘要
* 风险提示
* tags
* 展开说明

## 第四步：看回归样例和 diff

至少抽查 5 组样例，确认：

* 输入是什么
* 为什么得到当前 explanation
* 与上一个版本差异是什么

## 第五步：看灰度与回滚演练

要求 Codex 现场说明：

* 如何关闭某个模块
* 如何回退某个 profile
* 如何回退 explanation 模板
* 如何切回稳定版本

# 6. 验收时建议直接问 Codex 的问题

1. 现在如果输入只有生日，没有出生时刻，哪些 badge 会被禁止？
2. `legacy_estimate` 路径如何保证不误发高置信？
3. raw tag 在后端如何保留、在前端如何隐藏？
4. 男女合盘现在到底解释的是“关系气质”还是“长期稳定性”？
5. 若下周要下调西占权重，哪些配置改动即可完成？
6. 若 explanation 新模板上线后用户反馈变差，如何在 30 分钟内回退？
7. 20 组固定样例中，最容易跑偏的是哪几组？
8. 当前 stable bucket 和实验 bucket 是如何区分的？
