Update time: 2026-04-25

## Executive Summary

The Android build chain is now healthy:

- JDK 17 is pinned and working
- Flutter stable is installed and aligned
- Project Gradle wrapper works
- `assembleDebug` can produce a fresh APK

So the remaining 4.6P blocker is no longer build infrastructure.
The current blocker is the LiveKit RTC audio chain:

- calls can be created
- heartbeat is working
- join-info is returned correctly
- `Room.connect()` can sometimes succeed far enough to emit track events
- but real audible two-way voice still does not close the loop

Observed symptom split in the latest call (`callId=85`):

- phone can reach `ROOM_CONNECT_OK`, create a local microphone publication, and the local microphone PCM diagnostic is clearly non-zero once the user speaks
- phone still does not stabilize `remoteAudioTrack` reliably and often stays `none`
- emulator can reach `TrackSubscribed` and bind a `RemoteAudioTrack`, but in the failing window the room / peer connection logs still repeatedly cycle through `Connecting / Checking`, and the phone side does not receive a stable remote audio binding
- the emulator-side diagnostic renderer still sees zero-amplitude PCM in the failing window, even though `TrackSubscribed` happened
- both ends continue heartbeating correctly

This makes the current blocker look less like a pure signaling failure and more like a split media-path issue:

1. the phone side mic is alive, so local capture is not the primary blocker anymore
2. one side still does not stabilize remote subscription / binding
3. the other side can bind a remote track but still fails to close the peer/ICE path into a stable audible stream
4. the most likely remaining blocker is now the cloud-side media transport path: UDP media ports, TURN relay, or ICE/DTLS continuity

The latest diagnostic patch now also samples the phone-side local microphone PCM frames directly, so the next live test can distinguish:

- local mic capture is genuinely silent
- local mic capture is alive but remote subscription/rendering is flattening it
- or the phone side is still not stabilizing the remote subscription path

Latest client-side adjustments already tried:

- removed custom Google STUN from `ConnectOptions`
- kept only LiveKit/server-provided routing plus playback retry logic
- added local mic PCM sampling on the phone side
- added peer connection state logs on `TrackSubscribed`

Current strongest hypothesis:

- signaling is good enough to create a room and publish/subscribe events
- the remaining failure is in the media plane, most likely because the server-side UDP/TURN path is still not providing a stable two-way audio route

Official LiveKit deployment docs reinforce this direction:

- ICE/UDP media usually uses the configured UDP range (default `50000-60000/UDP`)
- ICE/TCP uses `7881/TCP`
- LiveKit also supports a single UDP mux port (`7882/UDP`) when configured
- TURN/UDP typically uses `3478/UDP`

Given that the phone-side mic capture is now non-zero, the next cloud-side check should prioritize whether the server's UDP media ports and TURN relay path are actually reachable and enabled end-to-end, not whether the Flutter mic capture or event wiring is functioning.

## RTC audio chain architecture

### 1) Call state machine and heartbeat

The RTC state machine is server-side and not the main blocker now.

Flow:

1. `createCall()` creates a session in `calling` or `busy`
2. `acceptCall()` moves a callee to `connecting`
3. `connectCall()` moves the session to `in_call`
4. `heartbeatCall()` updates `initiator_last_seen_at` and/or `peer_last_seen_at`
5. `refreshExpiredSession()` expires:
   - invite timeout: `calling / ringing` -> `missed`
   - disconnect timeout: `connecting / in_call` -> `ended` with `peer_disconnected`

Current timeout constants:

- `INVITE_TIMEOUT_SECONDS = 10`
- `DISCONNECT_TIMEOUT_SECONDS = 30`

### 2) Join-info payload

`GET /api/v1/rtc/calls/{callId}/livekit` returns the join payload to Flutter.

Client-side entity:

- `RtcLiveKitJoinInfoEntity`
  - `enabled`
  - `url`
  - `token`
  - `roomName`
  - `identity`
  - `participantName`
  - `mode`
  - `expiresAt`

### 3) Flutter connect flow

### 6) Remote audio binding logic

This is likely the most suspicious client-side area now.

Current logic:

- `_remoteAudioTrack` is stored in the service
- `TrackSubscribedEvent` assigns:

  - `_remoteAudioTrackFromEvent = true`

Key intention:

### 4. Track refresh vs event race

We have already reduced the risk of `_refreshRemoteAudioTrack()` clearing a valid event-set track, but this remains a live area to inspect if the next question is ŌĆ£why does one device keep track=null while the other sees framesŌĆØ.
