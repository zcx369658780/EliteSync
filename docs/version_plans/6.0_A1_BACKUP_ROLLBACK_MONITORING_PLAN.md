## 4. DB Backup Gate

本节只设计 gate，不执行任何 backup。

| Gate item | 计划要求 | 证据要求 | 当前状态 |
|---|---|---|---|
| full logical dump plan | 使用 logical dump 模板，目标仅可为 staging DB 或经用户确认的 production DB。 | dump 文件路径、时间、操作者、脱敏 DB 名称、checksum、文件大小。 | planning only |
| schema-only dump plan | 单独导出 schema，包含 routines / triggers。 | schema dump 文件、checksum、表清单。 | planning only |
| pre-migration backup plan | 任何 migration 前必须先有 full + schema-only backup evidence。 | backup evidence index。 | not executed |
| pre-composer-upgrade backup plan | Laravel 12 staging dry-run 若触及 DB 或 config，应先备份 staging DB/config。 | staging snapshot evidence。 | not executed |
| pre-v2-skeleton backup expectation | 即使只读 skeleton，也需明确 DB 不写；任何写接口前必须 backup。 | no-write confirmation 或 backup evidence。 | pending |
| backup storage location policy | backup 不放 repo，不放 public web root，不放 release artifact。 | storage path policy，不含 secret。 | planning only |
| retention policy | 至少保留最近一次进入 migration / runtime 前的可用 backup；生产策略需用户确认。 | retention rule。 | pending |
| checksum / file size verification | backup 完成后必须记录 checksum 和 byte size。 | checksum + size。 | not executed |
| encryption / access control | production backup 必须加密或落在受控目录；不得写入聊天或 git。 | access-control evidence。 | pending |
| owner / operator confirmation | 明确谁执行 backup，执行环境是什么。 | 用户确认记录。 | pending |
| redaction rule | 文档不写 DB host / username / password / token / secret。 | review pass。 | active |

必须遵守：

- production DB backup 只能由用户明确授权后执行。
- staging DB backup / restore 可作为后续演练环境，但也需要用户确认。
- 没有 backup evidence，不允许 migration / DB write / production config change。
- backup command 只能作为 future command template，不得在本文阶段执行。

## 5. Restore Drill Gate

本节只设计 restore drill，不执行 restore。

| Gate item | 要求 |
|---|---|
| restore target | 只能是 staging / isolated DB。 |
| production rule | 不允许 restore 到 production。 |
| target confirmation | restore 前必须确认 target DB 不是 production，并记录脱敏摘要。 |
| schema validation | restore 后核对 schema、表数量、迁移版本、核心表存在性。 |
| row count validation | 只记录脱敏 row count 摘要，不输出真实个人数据。 |
| sample readonly checks | 只允许 staging readonly sample checks，不请求 production API。 |
| sensitive data cleanup | restore 后必须清理 staging sensitive data，或使用脱敏数据集。 |
| pass 标准 | target 隔离、schema pass、row count 合理、readonly sample pass、无 secret leakage。 |
| fail 停止条件 | target 疑似 production、restore error、schema drift、隐私泄露、无法 cleanup。 |

Restore drill 通过前，不得把 backup gate 写成真正可恢复。

| Rollback item | 计划要求 | 当前阶段 |

| DB rollback | migration / DB write 前必须有 backup + restore drill evidence。 | blocked until evidence |
| migration rollback | 每个 migration 必须有 down/restore 策略；当前不执行 migration。 | blocked |
| Redis keyspace cleanup | staging prefix 必须可清理；production keyspace 不碰。 | planning only |
| storage / OSS cleanup | staging storage prefix 可清理；production OSS/CDN 不碰。 | planning only |
| queue worker rollback | worker 部署前必须有 stop/restart/disable 策略。 | carry-over |
| Nginx / PHP-FPM rollback | config 变更前备份原文件和 reload 失败回退步骤。 | planning only |

| release chain rollback | 不改 release chain；如后续改动，需版本链回滚计划。 | protected |
| incident stop condition | 误触 production、secret leakage、schema drift、5xx spike 时立即停止并生成 blocker report。 | active |

必须明确：

- production 不参与当前任务，因此当前不需要执行 production rollback。
- 若后续误触 production，必须立即停止并生成 blocker report。
- 任何无法解释 rollback 的改动不得执行。

## 8. Load-test Gate

本节只设计 load-test gate，不执行 load test。

规则：

- load test 只能在 staging / controlled backend。
- production 真实用户链路不得压测。
- v1 public readonly smoke 与 authenticated readonly smoke 结果不能替代 load test。
- write load test 默认禁止，除非用户另行授权 staging smoke dataset。

未来 load test 前必须确认：

- staging base URL。
- staging DB。
- staging Redis prefix。
- staging storage。
- no production traffic。
- no real user accounts。
- backup / rollback ready。

建议 load test scope：

- health。
- version check。
- authenticated readonly APIs。
- future v2 health / readiness / location contract。

建议指标：

- p50 / p95 / p99 latency。
- error rate。
- throughput。
- DB connection saturation。
- Redis memory / keyspace。
- queue backlog。
- PHP-FPM worker saturation。
- Nginx 5xx / timeout。

Stop conditions：

- 5xx spike。
- DB/Redis saturation。
- response schema drift。
- token leakage。
- privacy leakage。
- production traffic detected。
