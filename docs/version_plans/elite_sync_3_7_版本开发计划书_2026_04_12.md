# EliteSync 3.7 版本开发计划书

更新时间：2026-04-12  
适用对象：项目负责人 / Codex / Claude-mcp / Gemini-MCP / 顾问归档  
版本状态：草案 v1  
当前前置版本：`3.6` 已正式验收通过  
版本定位：`推运与高级解读版`

---

## 0. 本文档定位

本文档用于在 `3.6` 完成归档后，为 EliteSync 制定下一阶段 `3.7` 的正式开发计划。

`3.7` 不是对 `3.6` 的简单扩容，也不是把更多路线按钮继续堆到当前工作台。  
它的定位是：

> **在不破坏 canonical truth 单源、既有路由入口和 `3.6` 双路线基线的前提下，把 Astro Engine 从“古典 / 现代双路线版”推进到“推运与高级解读版”，引入合盘 / 对比盘 / 行运 / 返照 / 限运中的第一批受控能力，并建立高级解读的归档与验收体系。**

本版主要解决：

1. `3.6` 已经解决“路线分流、解释层和样例回归”，但还没有真正进入“时间维度”和“关系维度”。
2. 用户还不能在 App 内看到合盘、对比盘、行运、返照等更接近占星工作台核心价值的能力。
3. 高级能力一旦进入产品，最容易发生的风险就是把派生解释误当成 canonical truth，或把页面做成无法维护的复杂后台。
4. `3.7` 必须成为第一版真正意义上的“高级能力试点版”，但又不能越界成全量专业版。

本版明确不解决：

- 全量专业占星软件级别的参数深度
- 大规模 canonical 引擎替换
- 地理编码 / 登录 / 会话 / 数据库大结构重构
- 与星盘主线无关的产品线扩张

---

## 1. 为什么 3.7 必须单独立项

根据 `3.5` 与 `3.6` 的归档结论，当前项目已经完成：

- Engine Adapter / schema / metadata 基础设施
- 参数工作台 MVP
- 古典 / 现代 / 标准三路线分流
- 轻量解释层
- 多样例样例集、能力矩阵、差异报告与已知偏差归档
- 多 Agent 审查与正式归档链路

但这些能力仍然主要停留在：

1. **单命盘静态视图**
2. **路线差异解释**
3. **显示层与派生层对照**

而 `3.7` 要处理的是另一层复杂度：

1. **关系维度**：合盘 / 对比盘 / 双人视图
2. **时间维度**：行运 / 返照 / 第一批限运口径
3. **高级解读维度**：从路线差异上升到事件型、阶段型、关系型解释

这三者都显著高于 `3.6` 的难度，且更容易越过真值边界。因此：

`3.7` 必须被视为星盘引擎升级第三阶段，而不是 `3.6` 的自然补丁。

## 3. 3.7 总目标

### 3.1 本版要解决的问题

当前 App 在 `3.6` 之后已经能：

- 切换 `standard / classical / modern` 三路线
- 查看路线摘要、路线差异报告与多样例对照
- 保持 `route_mode` 为 derived-only / display-only 上下文

但还不能：

- 对两个人做合盘 / 对比盘展示
- 对同一人做行运 / 返照等时间维度解释
- 输出阶段性、关系型和更高颗粒度的高级解读
- 用多样例、多时间点、多关系样例建立高级能力的回归基线

### 3.2 本版最终应达到的形态

`3.7` 结束后，EliteSync 应达到：

> **在现有稳定产品中形成受控的高级占星能力首版：至少具备一套可用的合盘 / 对比盘入口、一套可用的时间维度入口、一层可归档的高级解读结构，以及可重复执行的样例回归、差异报告和多 Agent 验收链。**

## 4. 3.7 P0（必须完成）

## 4.1 高级能力范围冻结

目标：
- 在开工前冻结 `3.7` 实际纳入的高级能力，防止版本失控

必须完成：
1. 明确 `3.7` 只纳入以下第一批能力：
   - 合盘（Synastry）
   - 对比盘（Bi-wheel / Comparison）
   - 行运（Transit）
   - 返照中的一项最小可用能力（优先月返或日返二选一）
2. 明确以下内容不进入 `3.7 P0`：
   - 法达 / 小限 / 大限 / 流年全量体系
   - 三限 / 次限全量开放
   - 全量事件预测软件式参数工作台
3. 冻结每项能力的最小交付标准：
   - 是否只是 display / derived
   - 是否接入服务端派生接口
   - 是否只做样例 / 内测入口

---

## 4.2 合盘 / 对比盘基础能力

目标：
- 让“关系维度”首次进入产品，但保持受控

必须完成：
1. 建立双人输入 / 双人上下文 schema
2. 支持合盘与对比盘的最小展示：
   - 双人基础信息
   - 双盘点位 / 主要相位 / 关键角点
   - 最小关系摘要
3. 明确字段边界：
   - 对方资料是否为 canonical 输入
   - 合盘结果是否为 derived-only
4. 至少完成一组固定双人样例集
5. 形成 `PAIR_CAPABILITY_MATRIX` 与 `PAIR_PARITY_REPORT`

说明：
- 本版不要求把合盘做成完整专业工作台，但必须具备“可切换、可解释、可回归”的最小闭环。

---

## 4.3 行运 / 时间维度基础能力

目标：
- 让“时间维度”进入产品，但不导致真值层混乱

必须完成：
1. 建立时间上下文 schema：
   - natal context
   - target datetime
   - time mode / transit mode
2. 支持行运最小展示：
   - 当前行运视图
   - 与本命盘的关键相位差异
   - 最小时间摘要卡
3. 选择并落地一项最小返照能力：
   - 月返优先；若实现成本过高，可降级为日返
4. 明确时间维度结果全部属于：
   - derived-only / display-only / advanced context
5. 建立多时间点固定样例集与回归测试

说明：
- 本版做的是“时间维度首版”，不是全量推运平台。

---

## 4.4 高级解读结构首版

目标：
- 让高级能力不是只会“画图”，而是能输出可读的结构化解释

必须完成：
1. 合盘 / 对比盘解释卡：
   - 关系类型摘要
   - 关键相位摘要
   - 可见边界提示
2. 行运 / 返照解释卡：
   - 当前时间窗口
   - 关键主题摘要
   - 与本命盘关系提示
3. 统一高级解释字段来源定义：
   - canonical fields
   - derived analytic fields
   - display-only helper fields
4. 解释卡必须清楚写明：
   - 这是高级派生阅读层
   - 不是 profile truth / canonical truth

---

## 4.5 多样例高级回归体系

目标：
- 不靠“看起来像”，而是建立高级能力的回归底座

必须完成：
1. 建立三类样例：
   - 单人时间维度样例
   - 双人合盘样例
   - 双人对比盘样例
2. 每类样例至少覆盖：
   - 输出计数稳定性
   - 关键摘要稳定性
   - 已知偏差归档
3. 形成归档物：
   - `ADVANCED_CAPABILITY_MATRIX.md`
   - `TRANSIT_PARITY_REPORT.md`
   - `PAIR_PARITY_REPORT.md`
   - `ADVANCED_KNOWN_DEVIATIONS.md`
4. 样例回归必须可重复执行

## 7. 3.7 长期约束（不可违反）

### 7.1 真值约束

1. 服务端 canonical truth 仍是唯一真源。
2. 合盘 / 对比盘 / 行运 / 返照结果不得回写 `profile/basic` 或 canonical natal truth。
3. 双人关系解释与时间维度解释必须明确标记为 derived-only / display-only / advanced context。
4. 任何高级解释字段都不得伪装成“已验证事实”。

### 7.2 架构约束

1. 必须继续坚持 additive change。
2. 不删旧字段、不破坏旧读取路径、不重写 `chart_data` 主结构。
3. 新增高级能力接口必须有兼容 fallback。
4. 不得在没有样例回归矩阵的情况下上线高级解释。

### 7.3 UI / 工作流约束

1. Gemini-MCP 仍为强制 UI / 验收 subagent。
2. Claude-mcp 仍为强制架构边界审查 subagent。
3. Codex 不得跳过 Claude 的 schema 审查就定接口。
4. Codex 不得跳过 Gemini 的阶段验收就定 UI。
5. 高级页面不得演化成“用户无法读懂的半后台”。

### 7.4 发布与回归约束

1. 不得破坏 `3.6` 稳定基线。
2. 旧资料页、旧星盘页、旧设置页、旧版本中心、旧路线入口必须可回归。
3. 高级入口若不稳定，必须支持 feature flag / 版本门禁关闭。
4. 高级能力默认优先内测口径，不强制暴露给全部页面主入口。

```md
4. 阶段 2 和阶段 3 必须让 Gemini-MCP 做 UI / 可读性审查
5. 阶段 4 做高级多样例对照、差异矩阵与已知偏差归档
6. 阶段 5 执行 smoke / regression / rollback / 截图 / handoff / acceptance

你必须始终遵守：
- 服务端 canonical truth 仍是唯一真源
- 高级解释层不得回写真值层
- additive change 优先，不删旧字段
- 不改登录 / 会话 / 地理编码 / 数据库大结构
- 所有外部 subagent 结论必须留痕，能归档
```

1. 建议把高级结果写成新真值层

## 11. 3.7 验收标准

## 11.1 功能验收

必须满足：
1. 合盘 / 对比盘至少有一套可用入口
2. 行运至少有一套可用入口
3. 月返或日返至少有一项最小能力可用
4. 高级解释卡、关系摘要卡、时间摘要卡可见且联动刷新
5. 高级样例集、差异矩阵、已知偏差归档完成

## 11.2 边界验收

必须满足：
1. canonical truth 未被误写
2. 合盘 / 行运 / 返照结果明确属于 advanced derived 层
3. `chart_data`、`engine_info`、`metadata` 旧字段读取逻辑未被破坏
4. 旧缓存、旧客户端、旧入口可兼容
5. `route_mode` display-only 边界不被破坏

## 11.3 流程验收

必须满足：
1. Claude-mcp 阶段 1 / 阶段 3 至少有可归档审查记录
2. Gemini-MCP 阶段 2 / 阶段 3 / 最终截图复核有可归档记录
3. 关键测试、截图、差异报告、回归清单、已知偏差文档齐全
4. handoff / acceptance / regression / review log 一次补齐

## 11.4 不通过条件

出现以下任一情况，`3.7` 不得宣告通过：

1. 高级能力结果被写成 canonical truth
2. 旧页面 / 旧入口 / 旧缓存被破坏
3. Claude 或 Gemini 的强制审查链缺失
4. 没有高级样例矩阵与差异报告
5. 高级页面严重过载、不可读，且未收口
6. 关键测试未通过却仍试图归档

## 13. 一句话结论

`3.7` 的正确方向，不是继续堆更多复杂参数，而是：
