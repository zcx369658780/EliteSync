# EliteSync 3.5 版本开发计划书

更新时间：2026-04-11  
适用对象：项目负责人 / Codex / Gemini-MCP / Claude-mcp / 顾问归档  
版本状态：草案 v1  
当前前置版本：`3.4` 已终验通过  
版本定位：`Astro Engine MVP / 参数化基础版`

---

## 0. 本文档定位

本文档用于在 `3.4` 完成后，为 EliteSync 制定下一阶段 `3.5` 的正式开发计划。

`3.5` 不是普通的 UI 小修版本，也不是试图一次性完成全部专业占星能力的大一统版本。  
它的定位是：

> **在不破坏当前稳定产品基线的前提下，启动星盘引擎升级第一阶段，做出可用的 Astro Engine MVP。**

本版主要解决：

1. 当前 App 星盘页与历史网页原型之间的能力差距
2. 星盘参数化与工作台式交互的最小落地
3. 星盘引擎抽象层、数据结构与绘制层的拆分
4. 基础参数切换与样例对照验证框架

本版明确不解决：

- 古典 / 现代所有能力一次性全做
- 合盘、男女对比盘、行运、三限、次限、月返、日返、法达、小限盘一口气全部落地
- 高风险 canonical 算法全面替换

---

## 1. 为什么 3.5 必须单独立项

根据总计划与总交接稿，当前项目稳定基线已经完成：

- 资料与画像真值链路闭环
- 星盘本地绘制与盘面设置落地
- 设置中心与版本中心已产品化
- health / version check / smoke / regression / rollback 已纳入固定门禁

但顾问已明确判断：

1. 你同事历史网页原型对应的是更高层的“星盘工作台”能力
2. 当前 Kerykeion 路线与该原型不是同一级实现
3. 若要复刻该方向，不能在一个版本里一次性塞满，必须拆成多版本
4. 3.5 更适合做“引擎升级 MVP / 参数化基础版”

因此：

`3.5` 应该被视为一条新主线的第一阶段，而不是继续沿用 `3.4` 的小修思路。

## 3. 3.5 目标

### 3.1 本版要解决的问题

当前 App 已经能：
- 服务端计算真值
- Flutter 本地绘制星盘
- 通过设置页控制部分盘面显示

但还不能：
- 提供接近网页工作台的参数面板
- 提供更完整的星盘参数切换与对照
- 明确区分“引擎层、语义层、绘制层”
- 用系统化样例校验不同参数配置下的输出差异

3.5 的核心任务，就是把这些基础设施搭起来。

### 3.2 本版最终应达到的形态

`3.5` 结束后，EliteSync 应达到：

> **在现有稳定产品中新增一个可控的 Astro Engine MVP：具备基础参数面板、引擎抽象层、盘面精细绘制、参数解读最小版，以及可回归的样例对照框架。**

## 4. 3.5 P0（必须完成）

## 4.1 Astro Engine 抽象层

目标：
- 不再让当前星盘实现直接绑死单一具体库
- 预留未来引擎切换 / 并行计算 / 对照输出能力

必须完成：
1. 建立引擎抽象层（Engine Adapter）
2. 明确输入 schema：
   - 生日
   - 出生时间
   - 出生地点
   - 经度 / 纬度
   - 时区 / 真太阳时相关字段
   - 参数配置对象
3. 明确输出 schema：
   - `chart_data`
   - `planets`
   - `houses`
   - `aspects`
   - `angles`
   - `metadata`
   - `engine_info`
4. 明确哪些字段是 canonical 真值，哪些字段属于派生展示

---

## 4.2 参数化基础面板（MVP）

目标：
- 做出接近历史网页原型的最小参数工作台
- 让用户可选择基础参数并立即看到盘面变化

必须完成：
1. 新增参数面板 MVP
2. 支持以下基础参数切换：
   - 回归黄道 / 恒星黄道（若底层暂不支持全量，必须明确当前支持口径）
   - 核心宫位制（先做最常用的少量）
   - 主要相位开关
   - 基础相位容许度
   - 主要点位 / 主要角点显示开关
3. 参数修改后能够刷新盘面
4. 参数面板要清楚区分：
   - 会影响引擎计算的参数
   - 只影响本地显示的参数

---

## 4.3 盘面精细绘制 MVP

目标：
- 将当前 APP 本地绘制盘面推进到更接近网页原型的工作台风格

必须完成：
1. 标签避让策略升级
2. 相位线过滤与主次层级区分
3. 宫线层次与盘心结构优化
4. 盘面阅读提示与参数状态提示联动
5. 在不污染真值的前提下，让盘面看起来更“工作台化”

---

## 4.4 参数解读最小版

目标：
- 不直接做完整专业解释系统，但先做一层最小可用的参数阅读辅助

必须完成：
1. 参数说明卡 / 参数弹窗最小版
2. 至少覆盖：
   - 黄道制说明
   - 宫位制说明
   - 相位容许度说明
   - 点位显示说明
3. 对用户明确指出：
   - 当前看的是哪组参数
   - 哪些设置影响计算
   - 哪些设置只影响显示

---

## 4.5 样例对照验证框架

目标：
- 不再凭“看起来像不像”判断精度
- 建立固定样例对照体系，为后续 3.6 / 3.7 奠基

必须完成：
1. 建立固定样例集
2. 至少选择一组你同事文档中提到的典型生日对照样例
3. 对比以下输出：
   - 星体落座
   - 宫位落点
   - 主要相位
   - 角点
4. 形成差异报告与已知偏差说明
5. 样例回归必须能重复执行

## 7. 3.5 长期约束（不可违反）

### 7.1 真值约束

1. 服务端 canonical 真值仍是唯一真源
2. Flutter 本地绘制不得回写真值
3. 参数面板必须明确区分“计算参数”和“显示参数”
4. 任何参数切换都必须知道自己是否碰到了真值层

### 7.2 架构约束

1. 不得把新 Astro Engine MVP 直接写死成单一底层库实现
2. 必须先做 schema / adapter，再做更多参数能力
3. 不得在没有回归样例的情况下凭肉眼调算法

### 7.3 UI / 工作流约束

1. 本版涉及 UI 重构，**必须调用 Gemini-MCP 作为 subagent 辅助 Codex**
2. Gemini-MCP 必须参与：
   - 参数面板信息架构审查
   - 盘面可读性审查
   - 阶段验收与最终验收
3. Codex 不得跳过 Gemini-MCP 的阶段验收直接宣告完成

### 7.4 发布与回归约束

1. 所有高风险改动都要配套 smoke / regression / rollback 思维
2. 不得破坏 `3.4` 稳定基线
3. 旧资料页、旧星盘页、旧设置页入口必须可回归

## 10. 多 Agent 分工（本版重点）

### 10.1 Codex

Codex 仍为本版主执行人，负责：
- 计划拆解
- 风险检查
- 编码实现
- 文档归档
- 回归与验收材料整理
- 调度 Gemini-MCP 与 Claude-mcp

Codex 不得：
- 擅自把 MVP 扩成完整版
- 在没有样例回归的情况下硬调算法
- 跳过 Gemini-MCP 的 UI 审查
- 未经论证直接替换 canonical 引擎

### 10.2 Gemini-MCP（本版强制）

**Gemini-MCP 是本版强制 subagent。**

Gemini-MCP 负责：
1. 参数面板信息架构审查
2. 盘面精细绘制与可读性审查
3. 参数说明层、弹窗、提示卡审查
4. 阶段验收与最终验收
5. UI 重构相关截图审核与整改建议

Gemini-MCP 不负责：
- 决定底层算法逻辑
- 主导服务端架构改写
- 取代 Codex 进行主实现

### 10.3 Claude-mcp

Claude-mcp 仅用于关键架构问题咨询：
- Engine Adapter 设计
- chart/options schema 设计
- canonical 真值与 display-only 字段边界
- 向后兼容策略
- 样例回归框架设计

Claude-mcp 不负责：
- 日常 UI 微调
- 普通文案修改
- 替代 Gemini-MCP 做视觉审查
