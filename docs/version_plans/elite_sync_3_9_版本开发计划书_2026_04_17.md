# EliteSync（慢约会）APP 3.9 版本开发计划书

## 0. 文档说明

本计划书用于启动 EliteSync 星盘主线的 `3.9` 版本开发。

本次计划依据以下事实形成：

- `3.8` 已完成第二次验收收口，并可按**正式归档通过**处理。
- 当前星盘主线已完成 `3.5`—`3.8` 的连续推进，已具备：三路线分流、参数工作台、参数/盘面/解读联动、高级能力首版、校准闭环、版本中心口径统一。
- 当前整体判断为：星盘主线约完成 `60%–70%`，剩余 `30%–40%` 仍待推进。
- 下一阶段更适合优先推进：**高级时法 + 更细解释层**，而不是直接进入终局工作台大收口。

因此，`3.9` 的定位不是“终局专业版”，而是：

> **高级时法与细粒度解释层的第一阶段正式版。**

---

## 1. 版本定位

### 1.1 版本代号

`3.9 = Astro Timing Layer & Fine Explanation v1`

### 1.2 版本目标

在不破坏 `3.8` 已归档成果的前提下，完成三件事：

1. **建立可扩展的高级时法框架首版**
   - 不是一次性做完全部时法，而是完成“可扩展骨架 + 第一批正式能力 + 样例闭环”。
2. **把解释层从“块级摘要”推进到“细粒度条目级解释”**
   - 让用户看到的不再只是整页结论，而是更接近专业工作台的逐项说明。
3. **形成新的可验收闭环**
   - 包括：字段来源说明、UI 命名与边界说明、样例矩阵、差异报告、已知偏差、回归验收与正式归档材料。

### 1.3 本版本主线主题

> **时间维度正式进入产品主线，但仍严格停留在显示层 / 解释层 / advanced-context 层，不得伪装为 natal 真值层。**

---

## 2. 3.9 必须完成的核心交付

### 2.1 高级时法框架首版（必须）

本版本必须交付一个**统一的高级时法容器层**，用于承接未来的：

- 法达
- 小限
- 大限
- 流年
- 三限
- 次限
- 其他未来时法

#### 本版本最低要求

至少完成以下内容：

1. **统一的 timing mode 容器设计**

   - 形成前端可识别、后端可扩展、解释层可复用的时法模式结构。
   - 要求模式切换不会污染 natal 主结构。

2. **第一批正式接入的时法能力（建议 1–2 个，不宜贪多）**

   - 推荐优先：
     - `annual/profection-like`（可对应流年/年限类入口）
     - `firdaria-like`（法达）
   - 若实现复杂度或真值来源尚不稳定，可采用：
     - 1 个正式能力 + 1 个占位能力
   - 但必须保证：**至少有 1 个高级时法能力达到“可演示、可审查、可归档”的标准。**

3. **时法结果的高级上下文接入**

   - 时法结果必须以 `advanced-context` 或等价语义挂入解释链。
   - 不允许直接混入 natal truth 字段。

4. **时法样例矩阵**

   - 至少包含：
     - baseline case
     - edge case
     - dense/complex case
   - 用于观察时法结果、解释层输出、UI 命名是否稳定。

### 2.2 细粒度解释层 v1（必须）

本版本必须把解释层从“整段摘要”推进到“条目级解释”。

#### 本版本最低要求

至少完成以下能力：

1. **相位级解释条目化**

   - 相位不只显示存在与否，还要能进入条目级说明。
   - 条目中应包含：对象、类型、强弱/权重表达、解释片段。

2. **点位级解释条目化**

   - 至少支持主要点位/主要对象的独立条目展示。
   - 不要求 3.9 覆盖全部小行星与虚点，但要预留可扩展槽位。

3. **主星/宫主星/宫位关联解释首版**

   - 允许只覆盖“最重要、最稳定的一层”。
   - 重点是建立解释结构，而不是一次性把所有规则写死。

4. **解释层分层展示**

   - 摘要层
   - 条目层
   - 高级时法关联层
   - 三层之间要能说明“谁是摘要、谁是细节、谁是时间因素加成”。

### 2.3 工作台入口收口（必须）

3.9 不做终局大改版，但必须完成一个最小收口：

1. 设置页 / 参数页 / 高级页的命名边界更清楚
2. 时法入口不与 natal 参数入口混淆
3. 高级时法页和普通解释页语义分层更明确
4. 至少修掉一轮 3.8 中被保留为非阻断备注的命名或边界问题

### 2.4 归档材料（必须）

3.9 必须像 3.8 一样具备正式归档所需材料，而不是只做功能。

最低材料清单：

- `3.9_SCOPE.md`
- `3.9_SCHEMA_NOTES.md`
- `3.9_UI_INFORMATION_ARCHITECTURE.md`
- `3.9_SAMPLE_MATRIX.md`
- `3.9_CALIBRATION_REPORT.md`
- `3.9_DIFF_REPORT.md`
- `3.9_KNOWN_DEVIATIONS.md`
- `3.9_REGRESSION_REPORT.md`
- `3.9_ACCEPTANCE_SUMMARY.md`
- `3.9_FINAL_ARCHIVE_NOTE.md`

## 3. 本版本明确不做的事情（任务边界）

为防止范围失控，以下内容**不得被误算为 3.9 必做项**：

### 3.1 不要求一次做完全部高级时法

解释： 3.9 的目标是“建立框架并正式落地第一批能力”，不是在一个版本里把法达、小限、大限、流年、三限、次限全部做完。

验收口径：

- 若团队完成了 1 个正式能力 + 完整框架 + 样例闭环，可视为达到 3.9 主目标。
- 若试图做 5–6 个时法但每个都不稳定，则不视为通过。

### 3.2 不重构 natal 真值主结构

解释： 本次版本聚焦高级时法与解释层，不允许借机重写 `chart_data` 主结构，不允许把高级字段伪装成基础真值字段。

验收口径：

- 任何需要改动 natal canonical truth 才能成立的实现，一律视为越界方案。

### 3.3 不做终局工作台视觉大换代

解释： 3.9 可以做必要 UI 收口，但不能演变成“重画所有页面”“重做整套 IA”“追求终局网页原型 1:1 复刻”。

验收口径：

- 只允许围绕：命名清晰度、层级、入口边界、可读性 做收口。
- 大规模风格重设计留给 `4.x`。

### 3.4 不引入破坏性 schema 迁移

解释： 允许 additive change，不允许破坏旧读取路径、旧缓存、旧入口。

验收口径：

- 老版本样例、老客户端读取、既有演示链路必须继续可用。

## 4. 真值边界与架构边界（硬约束）

以下内容为 3.9 的**红线**，任何 Agent 都不得突破。

### 4.1 真值边界

1. 服务端 canonical truth 仍是唯一真源。
2. `route_mode` 继续保持 `display-only`。
3. `derived-only / display-only / advanced-context` 不得伪装为真值层。
4. 高级时法字段不得回写 `profile/basic` 或 natal 主结构。
5. 解释文本、条目标签、摘要结论均不得反向篡改基础盘真值。

### 4.2 架构边界

1. additive change 优先。
2. 不删旧字段、不破坏旧读取路径。
3. `chart_data` 主结构不动。
4. 旧缓存 / 旧客户端 / 旧入口兼容必须保留。
5. 时法层必须是“挂接式扩展”，不能成为基础盘的侵入式分叉。

### 4.3 工作流边界

1. Codex-CLI 仍是唯一主执行人。
2. Claude-MCP 负责：schema、字段来源、truth boundary、可扩展性风险审查。
3. Gemini-MCP 负责：UI 命名、信息架构、文案可读性、视觉一致性、任务验收。
4. 不得跳过多 Agent 审查链直接宣布通过。

## 5. 版本拆解：建议执行阶段

## Stage 0：立项冻结与范围锁定

### 目标

把 3.9 从“想做很多”收敛成“这次只做该做的”。

### 产出

- `3.9_SCOPE.md`
- 时法优先级列表
- 细粒度解释层字段草案
- 风险清单

### 完成标准

- 明确本次正式时法能力是哪些
- 明确哪些只是占位，不承诺正式交付
- 明确哪些解释条目进入首版，哪些延后

### 未达标表现

- 还在用“都想做”式语言
- timing 能力列表没有 P0/P1/P2 区分
- 解释层粒度没有锁定

---

## Stage 1：时法框架设计与 schema 挂接

### 目标

建立高级时法的统一容器、字段边界和接入方式。

### 必做任务

1. 设计 timing mode 统一入口
2. 设计前端展示所需的最小字段集合
3. 明确字段属于：truth / derived / display / advanced-context 的哪一层
4. 保证兼容旧读取路径
5. 形成 schema notes 与来源说明

### 完成标准

- 至少能稳定承载 1 个正式时法能力
- 不污染 natal 主结构
- 老样例不崩
- 文档能解释每个新字段为什么存在、从哪来、属于哪层

### 未达标表现

- 新字段来源说不清
- 同一字段既像真值又像展示值
- 旧页面/旧缓存被破坏

---

## Stage 2：第一批高级时法能力接入

### 目标

让至少 1 个高级时法真正跑通。

### 建议优先级

#### P0

- 法达 或 年限/流年类一项正式能力

#### P1

- 第二项时法能力（若稳定）

#### P2

- 其他时法仅做占位与文档保留

### 必做任务

1. 完成时法结果生成与前端挂接
2. 完成时法摘要展示
3. 完成时法条目展示
4. 接入高级解释链
5. 形成样例矩阵与边界案例

### 完成标准

- 至少 1 项时法在样例矩阵中稳定可展示
- 用户能看懂当前看到的是哪一种时法视角
- UI 不会误导成 natal 固有属性

### 未达标表现

- 时法页只是一个空标签页
- 结果不可复现
- 输出命名混乱，用户看不出“基础盘”与“时间盘”的区别

## Stage 3：细粒度解释层接入

### 目标

把解释从“摘要页”推进到“条目页”。

### 必做任务

1. 相位解释条目化
2. 点位解释条目化

5. 建立摘要层与条目层映射关系

### 完成标准

- 用户可从摘要进入更细一层的解释
- 至少三类解释条目可以稳定展示
- 文案与标签命名一致，不互相打架

### 未达标表现

- 所谓“细粒度”仍只是更长的大段文本
- 条目结构不稳定，样例切换即崩
- 时法结果和解释条目之间无关联

## Stage 4：UI/IA 最小收口

### 目标

### 必做任务

### 完成标准

- 用户能清楚分辨：基础参数、专业参数、高级时法、解释详情
- 关键页面命名收口一致
- 不引发新的学习成本激增

## 6. 验收标准（必须明确）

以下标准是 3.9 的**必要验收条件**，不是可选建议。

### 6.1 功能验收必要条件

#### 条件 A：至少 1 个高级时法能力正式可用

解释： 3.9 的名称如果包含“高级时法”，就必须有至少 1 个真正可演示、可审查、可归档的时法能力，不接受只做容器不做能力。

#### 条件 B：细粒度解释层必须真实存在

解释： 不能把“更长的摘要文案”冒充为“细粒度解释层”。必须有结构化条目展示。

#### 条件 C：时法与解释层必须连通

解释： 时法能力不能孤立存在，至少要能影响或挂接到高级解释展示。

### 6.2 架构验收必要条件

#### 条件 D：不破坏 3.8 基线

解释： `3.8` 已正式归档，因此 3.9 绝不允许通过“牺牲旧版本稳定性”来换取新能力。

#### 条件 E：字段边界可解释

解释： 每一个新增字段必须说得清：

- 是真值？
- 是导出值？
- 是展示值？
- 还是高级上下文？

说不清来源和归属的字段，不得进入正式归档。

### 6.3 产品验收必要条件

#### 条件 F：命名与入口边界更清晰，而不是更混乱

解释： 3.9 可以不是最漂亮，但必须比 3.8 更清楚，至少不能更难理解。

#### 条件 G：高级页面不呈现“调试工具感”

解释： 即便是专业能力，也要是产品页，不是工程演示面板。

### 6.4 文档与归档验收必要条件

#### 条件 H：必须有样例矩阵 / 差异报告 / 已知偏差

解释： 新能力进入产品，必须留下可复查的证据链。

#### 条件 I：必须通过多 Agent 审查链

解释： Claude 和 Gemini 都必须给出审查结论，允许存在 observation，但不允许跳过。

## 11. 风险预警

### 风险 1：范围膨胀

表现： 想在 3.9 一次做完所有高级时法、所有解释条目、所有视觉重构。

应对： 坚持“框架 + 第一批正式能力 + 可归档闭环”的版本观。

### 风险 2：时法字段越界

表现： 把高级时法结果写进 natal truth，或让 UI 表现误导用户以为那是出生盘固有属性。

应对： Claude 必须专项审查，Codex 必须保留 advanced-context 挂接式实现。

### 风险 3：解释层做成纯文案堆砌

表现： 没有条目结构，只是把摘要写得更长。

应对： 验收时强制要求条目化结构截图与样例矩阵。

### 风险 4：UI 收口变成 UI 大改版

表现： 大量精力消耗在视觉重画，影响真正的时法与解释层交付。

应对： Gemini 仅负责最小收口建议，不推动 4.x 级别重构。
