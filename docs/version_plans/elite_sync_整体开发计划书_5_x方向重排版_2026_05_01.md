# EliteSync 整体开发计划书（5.x 方向重排版）

更新时间：2026-05-01  
适用范围：4.9 稳定化基线之后的 5.x 产品化补强、云端治理增强与测试前路线重排  
当前稳定基线：`0.04.09 / 40900`
当前计划定位：以 4.9 为稳定门禁版本，在不破坏现代 UI、发布门禁、数据库安全与 RTC / 媒体主链的前提下，进入 5.x 产品能力补强阶段。

---

## 0. 本次重排的目的与结论

本计划书用于替代此前以“4.x 基础能力补全”为主的总纲口径，转而建立一个更适合当前现实约束的新总路线。

### 0.1 当前现实约束

1. 公司注册与备案仍未完成，短期内无法进入正式真实运营。  
2. 仍缺少真实外部用户样本，路线判断更多依赖产品推演、竞品拆解与双端联调证据。  
3. 4.9 已经把 UI protected surfaces、rollback / recovery policy、release gate、health、数据库安全、媒体 / 队列 / Worker 可观测性、RTC / LiveKit 可观测性固化为门禁基线。  
4. 现阶段最有价值的，不再是继续堆治理壳层，而是在稳定基线上补齐“未来真实测试一定会高频触达的产品层能力”。  
5. 后端仍应坚持“可扩展单体 + 云端治理增强”的路线，不适合此时做高成本微服务大重构。

### 0.2 重排后的总判断

从本版开始，项目总策略调整为：

- **4.9 视为稳定门禁基线，不再承载新业务功能扩张。**  
- **5.x 重新定义为“产品化补强主线”**，围绕发现、聊天、个人经营页、AI 辅助层、同城与轻关系推进进行增量增强。  
- **竞品拆解只吸收结构能力，不照搬成熟平台的商城、币体系和重商业化玩法。**  
- **后端继续走可扩展单体路线**，重点强化媒体、缓存、异步任务、日志监控、限流、通知、搜索、内容回流与后台治理便利性。  
- **语音通话主链已建立，后续更多承担“关系推进节点”角色**，而不是继续重写 RTC 基础设施。  
- **所有新计划都必须服务于未来测试期可验证、可回归、可扩展，而不是服务于当前并不存在的正式运营场景。**

### 0.3 本计划书的使用方式

本文件作为 5.x 版本计划、范围评审、风险拆解、Codex 执行、顾问复核与后续交接的母计划。后续每一个具体版本计划书，都应以本文件为上位约束，不得脱离本文件另起产品口径。

## 1. 当前稳定基线与必须保留的成果

本节用于固化 4.9 之前已经形成的稳定主干，避免 5.x 产品增强误伤已通过的门禁基线。

### 1.1 已完成且应保留的主线

#### 1.1.1 资料 / 画像 / 星盘真值链

已形成：
- 服务端真值优先的资料保存与重算链；
- 星盘服务端计算 + Flutter 本地绘制路线；
- 前端偏好只影响 derived 展示，不写回 canonical truth；
- 人格、星盘、匹配解释可作为关系引导底座继续复用。

计划口径：
- 5.x 的个人经营页、标签体系、聊天关系摘要、AI 辅助层，都必须建立在这条真值链上；
- 不得绕开既有真值链另起第二真值源。

#### 1.1.2 表达层与互动主链

已形成：
- 动态流基础版；
- 图片消息、视频消息；
- 站内通知中心；
- 1v1 真语音闭环；
- 设置中心、版本中心、健康检查与下载链。

计划口径：
- 5.x 的重点不是重新搭表达与互动主链，而是在其上补产品化结构与转化层；
- 任何增强都必须 additive，不回退已收口的媒体与 RTC 主链。

#### 1.1.3 4.9 稳定化与门禁基线

已形成：
- UI protected surfaces；
- rollback / recovery policy；
- smoke / release gate / health / DB safety；
- 通知中心 slug 降噪；
- 媒体 / 队列 / Worker 可观测性；
- RTC / LiveKit / Heartbeat 可观测性；
- 主页面双端 baseline 与截图证据链。

计划口径：
- 4.9 视为 5.x 的门禁起点；
- 后续版本不得以扩功能为由破坏这些规则；
- 所有版本计划书仍需保留保护面、非目标、验收条件和 blocker 处理规则。

### 1.2 当前保护面

以下内容默认继续列入保护面：

- 资料保存与重算链；
- 星盘 canonical 输出与本地渲染偏好链；
- 动态、图片消息、视频消息、通知中心、1v1 语音通话主链；
- `GET /api/v1/app/health`；
- `GET /api/v1/app/version/check`；
- 云端发布脚本、下载链、SHA256 口径；
- UI protected surfaces；
- 回滚与恢复政策；
- 云端数据库迁移 / 备份 / 恢复 / 写库隔离原则。

### 1.3 5.x 启动前的默认边界

- 本地仍只做前端开发、UI 联调、截图采集、文档整理；后端开发、数据库迁移、备份、恢复和任何会写数据库的操作统一在阿里云端执行。  
- 禁止 repo 级恢复；恢复只能 path-level / file-level / smallest-scope，且恢复前必须 checkpoint。  
- 任何跨层 blocker 先写 blocker report，再做根因缩层，再做最小修复。  
- 任何新功能都必须说明：是否破坏 protected surfaces、是否影响主链、是否会增加测试阶段返工成本。

## 2. 5.x 的核心方向：从“功能骨架已齐”转入“产品化补强”

### 2.1 为什么现在要转到产品化补强

4.3~4.9 已经基本补齐了表达、消息、通知、语音、设置、版本中心、稳定化门禁这些主干能力。继续沿“纯基础设施 / 纯门禁”方向推进，边际收益会下降。5.x 更应该解决的是：

1. 发现页如何更像产品，而不是单一状态流；  
2. 聊天页如何承担首聊、回聊与关系推进；  
3. 个人页如何从资料页升级成经营中枢；  
4. AI 如何真正嵌入慢约会主线，而不是成为孤立页面；  
5. 同城、搜索、轻分享、轻治理动作如何变成正式可用能力。

### 2.2 5.x 的五条产品原则

#### 原则 A：不再盲目扩“更多功能”，而是补“更合理的产品结构”
重点不是再加十个新页面，而是把现有 Discover / Chat / Me / Match 主链做成更完整的产品路径。

#### 原则 B：所有借鉴都必须改写成 EliteSync 语言
Soul 值得吸收的是：
- 发现页分栏与复合入口；
- 聊天页首聊 / 回聊辅助层；
- 聊天中的关系摘要与 AI 续话；
- 个人页经营中枢与 AI 助理；
- 搜索页热词 / 提示卡；
- 轻量分享 / 举报 / 不喜欢 action sheet。

但不应直接照搬：
- Soul 币中心；
- 超级星人；
- 个性商城；
- 重玩法导流；
- 过早商业化装扮。

#### 原则 C：AI 必须嵌入主链
AI 在 5.x 里的职责应是：
- 首聊建议；
- 回聊提醒；
- 续话建议；
- 资料展示优化建议；
- 动态 / 状态文案辅助；
- 主页质量与表达建议。

AI 不应先做成独立重平台。

#### 原则 D：关系推进优先于玩法扩张
5.x 优先解决：
- 如何打招呼；
- 如何继续了解；
- 如何从匹配走向聊天；
- 如何从聊天走向语音；
- 如何让个人资料更值得被继续看。

而不是优先做重娱乐化玩法。

#### 原则 E：云端治理增强必须服务产品化补强
后端增强的重点是：
- 数据 / 搜索 / 内容 / 通知 / 媒体 / 队列治理更方便；
- 管理后台、只读优先 DB 访问、健康检查、备份恢复、日志检索更顺；
- 不是为了好看而做一层层复杂平台。

## 3. 5.x 版本总路线图（重排后）

本版将后续路线拆成四条并行关注点：

1. **发现与状态流补强线**  
2. **聊天与关系推进补强线**  
3. **个人经营页与 AI 助理补强线**  
4. **云端治理与可扩展单体增强线**

### 3.1 推荐推进顺序

推荐顺序：

1. 先做 5.0 Alpha 测试候选版：把最小产品化增强补齐；
2. 再做 5.1 关系转化与内容回流增强版；
3. 再做 5.2 个人经营页与表达层增强版；
4. 再做 5.3 测试运营准备与云端治理增强版；
5. 之后再根据现实反馈进入 5.4 真实小样本吸收版。

### 3.2 暂不进入高优先级的方向

- 重商业化付费体系；
- 多人 RTC；
- 重娱乐互动玩法；
- 大内容社区化；
- 微服务全面拆分；
- 大规模后台运营平台化。

## 4. 5.0：Alpha 测试候选版（产品化补强最小集）

### 4.1 版本目标

5.0 的目标不是再做一轮纯稳定化，而是在继承 4.9 门禁的前提下，补齐“真实小样本测试最容易暴露缺口的产品层”。

### 4.2 5.0 的高优先级范围

#### 4.2.1 Discover / 发现页最小产品化增强

目标：让 Discover 从“有内容流骨架”升级成“真正可用的发现层”。

建议补齐：
- 发现页分栏正式化：推荐 / 同城 / 关注（或推荐 / 同城 / 话题）；
- 搜索页产品化：搜索框 + 热词 + 提示卡；
- 卡片 action sheet：分享 / 不喜欢 / 举报 / 复制链接等轻治理动作；
- 同城内容入口与按需定位权限提示；
- 状态流卡片 CTA 重新设计，避免过强“立即私聊”语义。

#### 4.2.2 Chat / 聊天页关系推进最小集

目标：让聊天页不再只是消息容器，而是承担“首聊、回聊、关系推进”的主路径。

建议补齐：
- 首聊 / 回聊辅助队列；
- 推荐问候模板；
- 输入区 AI 续话 / 破冰入口；
- 聊天页关系摘要条：共同点 / 沟通建议 / 查看主页；
- 聊天页到个人页轻联动；
- 通知、玩法、系统提示的分层口径继续清晰化。

#### 4.2.3 Me / 个人页经营中枢最小集

目标：让个人页从“资料页”升级成“表达 + 内容 + AI 建议 + 关系转化”的中枢。

建议补齐：
- 标签表达体系重排；
- 内容 / 收藏 / 赞过 / 资料建议联动；
- AI 助理轻入口；
- 主页质量分或展示友好度建议；
- 后续声音名片 / 轻语音表达的入口预留。

### 4.3 5.0 非目标

- 不做会员中心与币体系；
- 不做商城与装扮商业化；
- 不做多人互动玩法；
- 不重写 RTC；
- 不重写动态流主链；
- 不把 Discover 做成重社区平台。

## 5. 5.1：关系转化与内容回流增强版

### 5.1 目标

在 5.0 的最小产品化补强基础上，把“关系如何逐步推进”做得更顺。

### 5.2 重点任务

- 首聊 / 回聊队列完善；
- 冷场续话建议；
- 匹配解释到聊天建议联动；
- 动态 / 状态到私聊的低压转化；
- 私聊、关注、查看资料、语音邀请之间的动作节奏设计；
- 通知中心回流文案产品化。

### 5.3 非目标

- 不把首聊系统做成大模型平台；
- 不提前做复杂匹配算法重构；
- 不引入重玩法活动系统。

---

## 6. 5.2：个人经营页与表达层增强版

### 6.1 目标

增强“为什么别人愿意继续看你、回你、和你继续聊”。

### 6.2 重点任务

- 人格、关系风格、生活状态、交友态度标签体系；
- 个人资料聚合区的层级优化；
- 主页质量分 / 展示建议升级；
- AI 帮发 / AI 帮你写一句问候 / AI 整理亮点；
- 声音名片或轻语音自我介绍；
- 个人空间外观层的最小免费版（如背景 / 展示封面 / 风格轻配置）。

### 6.3 非目标

- 不做重商城；
- 不做币体系；
- 不做大型娱乐中心；
- 不做复杂装扮生态。

---

## 7. 5.3：测试运营准备与云端治理增强版

### 7.1 目标

在产品层更像正式产品后，再把管理便利性、后台可观测性与云端治理补顺。

### 7.2 重点任务

- 业务管理后台最小集：用户、资料、动态、会话、通知、举报、测试账号；
- 云端只读优先 DB 访问与审计；
- 搜索、通知、媒体、队列、Worker、日志、健康检查可视化入口；
- 发布链、备份恢复、迁移状态、关键表比对的日常 runbook 固化；
- 继续保持本地前端 / 云端后端严格隔离。

### 7.3 非目标

- 不做重型运营中台；
- 不做全面微服务治理平台；
- 不开放高风险写库后台给本地开发环境。

## 8. 5.4：真实小样本反馈吸收版

### 8.1 目标

在注册 / 备案条件允许、或至少有可控小样本后，对前几版的产品化增强做真实修正。

### 8.2 重点任务

- 吸收首聊 / 回聊 / AI 建议真实反馈；
- 修正 Discover 分栏、同城、搜索热词、提示卡；
- 修正聊天关系摘要与语音邀请路径；
- 修正个人经营页、标签体系、主页质量分的可理解性；
- 继续补充通知、媒体、RTC 失败路径证据。

## 9. 云端后端升级方向（贯穿 5.x）

### 9.1 总原则

后端继续走“可扩展单体”，重点不是微服务数量，而是关键扩展点预埋和治理便利性提升。

### 9.2 重点升级方向

#### 9.2.1 业务域进一步清晰化

建议在单体内继续清晰化这些域：
- auth / account  
- profile / labels / astro  
- match / intro / suggestions  
- conversation / notification  
- media / attachment  
- status_feed / discovery  
- moderation / report  
- rtc_signal / call_state  
- app_release / observability

#### 9.2.2 必补的工程扩展点

- 搜索能力；
- 内容缓存；
- 同城与推荐结果缓存；
- 媒体上传与异步处理状态机；
- 通知与队列；
- 日志检索与告警；
- 限流矩阵长期观察；
- 管理后台与只读优先数据库访问。

#### 9.2.3 数据与安全边界

- 一切写库操作仍在阿里云端执行；
- 本地不直连生产写库；
- 数据迁移、备份、恢复、关键表比对保持制度化；
- 所有高风险修改保持 blocker report + 最小修复流程。

## 10. 版本计划书编写规则（5.x 继续沿用）

后续每一个具体版本计划书都必须包含：

1. 版本定位；
2. 基线与保护面；
3. 范围与非目标；
4. 风险拆解；
5. 任务分层；
6. 验收条件；
7. 多 agent 分工；
8. blocker 处理规则；
9. 单文件 handoff 规则。

同时继续遵守：
- 当前版本 cycle 只保留最小活跃计划文件；
- handoff 内容默认收敛到单一 `HANDOFF_MASTER`；
- 已归档版本不得重新作为当前待开发版本。

## 11. 当前阶段的最终结论

当前项目不再适合继续沿“纯门禁 / 纯稳定化”方向推进，也不适合贸然进入重商业化或重娱乐玩法阶段。

5.x 最合理的路线是：

1. 以 4.9 为稳定门禁基线；
2. 先补 Discover / Chat / Me 的最小产品化增强；
3. 再补关系推进、AI 辅助层与个人经营页；
4. 同时把云端治理、搜索、通知、媒体、队列和后台便利性逐步做顺；
5. 等现实条件允许，再进入小样本真实反馈吸收周期。

一句话概括：

> **5.x 的任务不是“做更多功能”，而是把已经具备的主干能力改造成真正更像产品、也更适合未来测试的慢约会系统。**
