## 4. cache / session 口径统一方案

当前问题：

- Gate 远端观察：`artisan about` 显示 Cache=file、Session=file，但 `.env` 有 `CACHE_STORE=redis`、`SESSION_DRIVER=database`、`QUEUE_CONNECTION=redis`。
- 仓库配置审计显示：
  - `config/cache.php` 读取 `APP_CACHE_STORE`，默认 `file`。
  - `config/session.php` 读取 `APP_SESSION_DRIVER`，默认 `file`。
  - `.env.example` 使用 `CACHE_STORE=redis` 与 `SESSION_DRIVER=database`。
  - `config/queue.php` 读取 `QUEUE_CONNECTION`，再 fallback `APP_QUEUE_CONNECTION`。
- 因此当前存在 env key 命名与 config cache 双重口径不一致风险。

推荐目标：

- Redis 用于 cache / queue / rate limit。
- session 推荐也进入 Redis，但如果 A1 为降低登录态风险，也可先保持 database session，前提是文档明确 reason 与迁移计划。
- 无论 session 选 Redis 还是 database，必须使用一套 app-scoped env key，避免 host-level env 污染。

建议 env 口径：

| 环境 | cache | queue | session | rate limit | 备注 |
| --- | --- | --- | --- | --- | --- |
| local | file 或 redis | sync / redis | file / database | cache default | 开发便利优先 |
| staging | redis | redis | redis 或 database | redis | 与 production 接近 |
| production | redis | redis | redis 或 database | redis | 需容量、持久化和回滚方案 |

进入 skeleton 前必须确认的配置项：

- `APP_ENV`
- `APP_DEBUG`
- `APP_CACHE_STORE` 或最终统一后的 cache key
- `APP_SESSION_DRIVER` 或最终统一后的 session key
- `SESSION_STORE`
- `SESSION_CONNECTION`
- `QUEUE_CONNECTION`
- `REDIS_QUEUE_CONNECTION`
- `REDIS_CACHE_CONNECTION`
- `REDIS_DB`
- `REDIS_CACHE_DB`
- `CACHE_PREFIX`
- `REDIS_PREFIX`

config cache 清理 / 重建流程草案：

1. 备份 `.env` 与当前 cached config 文件状态。
2. staging 设置目标 env key。
3. 执行 `php artisan optimize:clear`。
4. 执行 `php artisan config:cache`。
5. 执行 `php artisan about` 与 `php artisan config:show cache.default`、`session.driver`、`queue.default`。
6. 执行 login/profile/messages smoke。

本阶段不执行上述命令；只记录为后续用户确认后的 staging 步骤。

## 5. queue worker 常驻方案

当前风险：

- Gate 显示 `QUEUE_CONNECTION=redis`，但 Supervisor inactive，未见 `queue:work` / Horizon 常驻进程。
- 媒体 processing、通知、未来 location、match、astro、buddy 异步任务若无 worker，会出现“写入成功但后台处理不运行”的假成功。

Redis queue 目标：

- Redis queue 作为 A1 目标，必须明确 connection、queue names、retry_after、block_for、prefix。
- 队列与 cache/session 不应无边界混用同一逻辑库与淘汰策略。

Supervisor vs systemd：

| 方案 | 优点 | 风险 | 建议 |
| --- | --- | --- | --- |
| systemd | 原生、轻量、与服务器服务管理一致 | 多队列多 worker 伸缩不如 Supervisor 直观 | A1 初期可优先 |
| Supervisor | 多进程组、队列分层、重启策略成熟 | 当前服务 inactive，需要安装/配置/备份 | 后续多队列可采用 |
| Horizon | Laravel queue 可视化强 | 当前未安装，需要 Redis 与 package 评估 | 不作为 A1 前提 |

queue 分层建议：

- `default`：普通业务异步。
- `notifications`：通知创建与推送。
- `media`：媒体处理、缩略图、转码状态。
- `match`：匹配批处理与解释生成。
- `astro`：星盘/八字/紫微重算、Python render fallback。
- `buddy`：搭子请求与匹配后续任务。
- `low`：统计、清理、非关键同步。

failed jobs / retry / backoff / timeout：

- 确认 `failed_jobs` 表存在并可用。
- 每类 job 定义 `tries`、`backoff`、`timeout`。
- Media/astro 类长任务要设置独立 timeout。
- 幂等 job 必须有业务 key，避免重试重复写入。
- queue worker 部署后必须使用 `php artisan queue:restart` 平滑加载新代码。

logs / metrics / alert：

- worker stdout/stderr 独立日志。
- failed job count 告警。
- queue length 与 oldest job latency。
- worker restart count。
- job timeout 和 memory exceeded。

A1 skeleton 前：

- 必须先规划 queue worker 管理方式。
- 若 v2 skeleton 只包含纯读 health/readiness/location schema，可以暂不启用 worker，但 readiness 必须明确 queue check 是 `not_configured`、`degraded` 还是 `skipped`。
- 任何会创建 async job 的 v2 endpoint，不允许在 worker 未常驻前上线。

## 6. Composer 2.7.1 deprecation 处理方案

当前问题：

- Gate 中 `composer check-platform-reqs` 在 PHP 8.4 下成功，但 Composer 2.7.1 输出多条 deprecation notice。
- 该问题不阻断只读文档，但阻断 Laravel 12 staging upgrade 的可信执行环境。

建议处理：

1. staging 只读记录 `composer --version`。
2. staging 只读执行：
   - `composer validate --no-check-publish`
   - `composer check-platform-reqs`
   - `composer outdated --direct`
3. 若 deprecation 仍存在，提交用户确认是否允许 `composer self-update --stable`。
4. self-update 前记录 Composer phar 路径、当前版本、回滚命令。
5. self-update 后只在 staging 验证，不影响 production。

云端执行边界：

- `composer self-update` 是否允许在云端执行：必须先用户确认。
- 本阶段不执行 `composer self-update`。
- 本阶段不执行 `composer update`。
- 本阶段不修改 composer.json / composer.lock。

## 7. Nginx + PHP-FPM 商用压测基线方案

- root / index / try_files。
- PHP fastcgi_pass 与 SCRIPT_FILENAME。
- WebSocket / LiveKit / message ws proxy。

- `client_body_timeout` / `proxy_read_timeout` / `fastcgi_read_timeout`。
- access_log / error_log / log rotation。
- gzip / brotli。
- rate limit / connection limit。
- security headers。

php-fpm pool 审计项：

- listen socket / user / group。
- `pm` 模式。
- `pm.max_children`。
- `pm.start_servers` / min/max spare。
- `pm.max_requests`。
- `request_slowlog_timeout`。
- `slowlog`。

- FPM status path。

opcache 审计项：

- `opcache.enable`
- `opcache.memory_consumption`
- `opcache.max_accelerated_files`
- `opcache.validate_timestamps`
- `opcache.revalidate_freq`

staging 压测前必须记录：

- Nginx effective config 摘要。
- PHP-FPM pool effective config。
- Redis config。
- DB version 与 connection limit。
- 当前 release metadata。

## 9. backup / rollback / monitoring / load test 最小门禁

backup：

- DB 全量备份：`mariadb-dump --single-transaction --quick --routines --triggers --events --all-databases`。
- schema-only 备份：`mariadb-dump --no-data --routines --triggers --events --all-databases`。
- `.env` 备份，脱敏存档。
- Nginx / PHP-FPM / Supervisor 或 systemd 配置备份。
- release 配置备份：`config/app_update.php`、release script、current version metadata。

rollback checklist：

- Git 回滚点。
- DB restore 命令与演练记录。
- `.env` restore。
- Nginx/PHP-FPM config restore。
- queue worker stop/restart rollback。
- v2 feature flag off。
- Flutter base URL 回退。
- v1 smoke 重新通过。

monitoring：

- health/readiness 指标。
- Nginx 4xx/5xx、upstream timeout、request latency。
- Laravel error log、exception rate、deprecation。
- DB connection、slow query、lock wait、disk。
- Redis memory、eviction、blocked clients、keyspace。
- queue length、oldest job age、failed jobs。
- disk usage、storage writeability。

load test：

- 只打 staging，不打 production 真实用户链路。
- 目标先是基线观测，不是追求高并发数字。
- 记录 P50/P95/P99、error rate、PHP-FPM busy workers、DB/Redis 指标、queue latency。
- 工具候选：k6 / wrk；Laravel queue 需要单独 job backlog 测试。

进入 skeleton 前最小门禁：

- 至少完成 backup/rollback checklist 文档。
- 至少完成 staging readiness checklist。
- 至少完成 v1 smoke plan 并执行 P0 smoke。
- 至少记录 Nginx/PHP-FPM/Redis/DB baseline。
