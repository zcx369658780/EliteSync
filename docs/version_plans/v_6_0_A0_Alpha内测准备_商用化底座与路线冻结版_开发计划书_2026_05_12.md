# 0. 一句话目标

冻结 EliteSync 6.0 Alpha 内测准备线的实施路线，把后端 v2、位置链路、Date Drop 式匹配主链、搭子精准陪伴、Claude 横向复评门禁与 A1-A5 版本拆分全部收敛为可执行、可验收、可交接的 planning-only 文档基线。

---

# 1. 当前背景

EliteSync 已从 5.6-5.10 的“玄学能力二次产品化与校准线”切换到 `6.0 Alpha 内测准备线`。5.6-5.10 的结论保留为历史闭环，状态为 `pass with observations`，但当前不继续制定 5.11，也不直接进入 runtime 开发。

6.0 Alpha 的目标不是继续做小功能补丁，而是让 EliteSync 从“功能雏形集合”升级为一个能承接小规模真实内测的慢约会产品。当前需要优先解决的不是单个页面，而是：

1. 后端与第三方链路仍偏开发版，不适合未来真实内测压力；
2. 位置链路语义混杂，出生地、现居地、约会地点、搭子地点需要拆清；
3. Date Drop 式高质量低频匹配主链尚未形成稳定产品闭环；
4. 搭子精准陪伴需要成为独立但服务慢约会的 P1 主线；
5. 基础社交能力、玄学解释产品化和 UI/IA 仍需在后续版本分阶段完成；
6. 每个版本必须建立 Claude 横向复评前置门禁，而不是只依赖 Codex 自报和 GPT 事后验收。

因此，6.0-A0 必须先冻结路线、边界和后续版本拆分，再进入 A1 runtime。A0 不做任何 Flutter / Laravel / DB / API / release chain 运行时代码修改。

# 2. 本版本目标

## 2.1 主目标

6.0-A0 的主目标是完成 6.0 Alpha 的 planning-only 路线冻结。A0 通过后，项目才能进入 6.0-A1“后端 v2 与位置链路最小闭环”。

## 2.2 具体目标

本版本必须冻结以下七类内容：

1. **后端 v2 迁移方案**  
   明确 `contract-first + parallel migration + staged rollout` 路线，定义 v1 contract 审计、v2 contract map、并行期、灰度切换、回滚路径、压测与监控要求。

2. **位置链路重构方案**  
   明确出生地、现居地、约会地点、搭子地点四类位置语义，不再混写；明确去用户侧精确经纬度展示、行政地名优先、外部地图辅助而非强依赖内置地址搜索的方向。

3. **搭子功能产品方案**  
   明确学习搭子、电影搭子、吃饭搭子、健身搭子的最小闭环、数据域、安全边界、反馈闭环，以及与婚恋匹配的边界。

4. **Date Drop 式匹配主链重构方案**  
   明确深度资料、低频高质量推荐、等待期、揭晓、解释、聊天承接、反馈与下一轮优化的主链结构。

5. **Claude 横向复评规则在版本计划中的固定执行方式**  
   将 Claude 使用 Soul + 测测 / CECE 做横向复评的门禁写入 A1-A5 的通用验收流程。A0 本身因 planning-only 采用轻量复评，但不得跳过 Claude 结论。

6. **A1-A5 后续版本拆分**  
   明确：A1 后端 v2 与位置链路最小闭环；A2 Date Drop 式匹配主链；A3 搭子精准陪伴；A4 基础社交补齐；A5 玄学解释与 UI/IA 内测打磨。

7. **项目源索引与 project memory 同步要求**  
   明确 A0 完成后需要同步 `DEVELOPMENT_PLAN_CURRENT.md`、`DOC_INDEX_CURRENT.md`、`project_memory.md`、`version_plans/README.md` 等入口，但不得误写成 runtime 已完成。

# 3. 本版本范围

### 3.1.3 冻结后端 v2 迁移方案

在 A0 计划书中明确：

- v1 API contract 审计范围；
- v2 contract map 初稿；
- v2 服务域拆分；
- 迁移阶段顺序；
- parallel migration 期间兼容策略；
- Flutter base URL / feature flag 切换原则；

- 备份、回滚、压测、监控要求；

### 3.1.4 冻结位置链路重构方案

在 A0 计划书中明确：

| 位置类型 | 主要用途 | A0 冻结口径 |
|---|---|---|
| 出生地 | 星盘 / 八字 / 紫微计算 | 行政地名 + 内部可选坐标；用户侧不展示精确经纬度 |
| 现居地 | 匹配范围 / 生活圈 / 内测分布 | 城市 / 行政区优先，不展示精确位置 |
| 约会地点 | Date Drop 式低频见面建议 | 地点名 / 商圈 / 城市，不作为资料真值 |
| 搭子地点 | 学习 / 电影 / 吃饭 / 健身搭子 | 城市 / 商圈 / 用户确认地点，必须有隐私保护 |

并明确：A0 不接地图 SDK，不改定位逻辑，不改 DB，不改 API。

### 3.1.5 冻结 Date Drop 式匹配主链方案

在 A0 计划书中明确 A2 的未来设计边界：

```text
深度资料 -> 低频高质量推荐 -> 等待期 -> 揭晓 -> 解释 -> 低压聊天 -> 反馈 -> 下一轮优化
```

要求：

- 不做无限刷卡；
- 不做强 Feed；
- 不做泛社交推荐流；
- 不把玄学解释反写为匹配算法真值；
- 不把反馈直接改写 profile 真值；
- 所有算法与 contract 调整必须在 A2 计划中单独审批。

### 3.1.6 冻结搭子精准陪伴方案

在 A0 计划书中明确 A3 的未来设计边界：

```text
搭子类型选择 -> 需求填写 -> 时间/地点/预算/边界 -> 需求卡 -> 1-3 个候选 -> 为什么推荐 -> 低压聊天 -> 活动后反馈
```

首批类型：

- 学习搭子；
- 电影搭子；
- 吃饭搭子；
- 健身搭子。

必须写清：搭子不是泛同城约玩，不是陪玩商业化，不是附近的人无限刷，不默认暴露精确位置。

### 3.1.7 固化 Claude 横向复评规则

A0 必须把以下规则写进后续版本计划模板：

```text
Codex 完成实现与自测 -> Claude 横向复评 -> Codex 处理 Claude 反馈 -> GPT 顾问最终验收 -> 通过后才允许进入下一版本。
```

# 6. 产品与 UX 要求

虽然 A0 不做 UI runtime，但 A0 必须为 A1-A5 锁定产品和 UX 原则。

## 6.1 6.0 Alpha 产品总原则

EliteSync 6.0 Alpha 不是泛社交平台，不是玄学娱乐 App，也不是附近约玩平台。后续所有 runtime 版本必须服务以下主线：

```text
慢约会 / 深资料 / 高质量低频匹配 / 低压聊天 / 共同兴趣陪伴 / 可解释但不命运化的玄学辅助
```

## 6.2 Date Drop 式匹配主链 UX 原则

A2 计划必须承接：

- 用户知道为什么要补资料；
- 用户知道为什么要等待；
- 用户知道本轮匹配如何产生；
- 揭晓页能解释为什么值得聊；
- 用户能低压进入聊天；
- 用户能反馈推荐是否准确；
- 反馈只影响后续推荐优化，不反写资料真值。

禁止把 Match 做成：

- 无限刷卡；
- 强推荐流；
- 强 Feed；
- 看脸滑动；
- 工程状态堆叠；
- 黑箱玄学命运判断。

## 6.3 搭子 UX 原则

A3 计划必须承接：

- 搭子类型清楚；
- 共同目标清楚；
- 时间、地点、预算、边界清楚；
- 初次活动鼓励公共场所；
- 不展示精确实时位置；
- 聊天前有安全提示；
- 活动后可反馈、举报、拉黑；
- 搭子反馈不直接写成婚恋人格判断。

禁止把搭子做成：

- 附近的人无限刷；
- 泛同城约玩；
- 陪玩商业化；
- 强地理曝光；
- 私密地点邀约；
- 性暗示或擦边场景。

## 6.4 位置 UX 原则

A1 计划必须承接：

- 用户侧默认不展示精确经纬度；
- 出生地以行政地名表达；
- 现居地以城市 / 区县表达；

# 8. 多 agent / sub-agent 协作计划

本版本采用 Codex 主执行 + Claude 轻量横向复评 + 必要时 Gemini / 视觉 sub-agent 辅助的多 agent 模式。Codex 不得在跨层 blocker、竞品产品判断、后端 v2 迁移判断、地图 / 定位方案判断、UI/IA 体验判断上长期单独裁决。

## 8.2 Claude 角色

Claude 在 A0 中承担轻量横向复评和架构判断角色，重点检查：

## 8.3 Gemini / 视觉 sub-agent 角色
