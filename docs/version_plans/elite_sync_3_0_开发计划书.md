## 2. 版本背景

## 2.1 2.9 已完成的意义

EliteSync `2.9` 已完成 Beta 上线准备阶段的工程化收口，其核心价值不是扩张前台功能，而是完成：

- 测试与回归基线收口
- 弱网、冷启动、导航稳定性验证
- 健康检查与版本检查对齐
- 灰度演练与回滚演练
- 监控、运维、SOP、事故处理文档补齐

因此，2.9 的完成意味着项目已经从“可演示产品”进入：

> **可进入小规模真实 Beta 的工程准备状态。**

---

## 2.2 3.0 的必要性

2.9 解决的是“能否上线 Beta”的问题，但并没有彻底解决“Beta 用户进来以后，产品能否形成稳定增长与高质量体验”的问题。

因此，3.0 的任务不再是单纯补齐工程门禁，而应转向：

> **在不破坏 2.9 工程底座的前提下，把 EliteSync 从“可运行的 Beta 产品”推进到“可持续运营、可验证留存、可优化匹配质量”的新阶段。**

换句话说：

- `2.9` 的核心是 **Beta Ready**
- `3.0` 的核心是 **Beta Usable + Beta Learnable + Beta Optimizable**

---

## 3. 3.0 版本定位

### 3.1 一句话定位

`3.0` 是 EliteSync 的：

> **Beta 运营与匹配质量提升版**

它的目标不是继续无边界加功能，而是围绕真实 Beta 期间最关键的三件事推进：

1. **让用户更容易完成关键路径**
2. **让匹配结果更可理解、更可行动**
3. **让产品团队能从真实数据中学习并持续优化**

---

### 3.2 版本总目标

3.0 需要同时实现三个层面的推进：

#### A. 用户体验层
- 提升新用户从注册到首次有效互动的完成率
- 降低问卷、匹配、聊天链路中的卡顿感与中断感
- 提升匹配结果的可读性、可信度与下一步行动感

#### B. 产品运营层
- 形成可追踪的 Beta 核心指标体系
- 建立用户反馈到产品改进的闭环
- 提升运营侧对用户状态、问题、内容与节奏的可控性

#### C. 匹配与解释层
- 不大改底层引擎，但要显著提升“解释层”和“行为引导层”的质量
- 让用户不只是“看到一个结果”，而是知道“为什么推荐、接下来怎么做”
- 为后续 3.x 的算法升级打下更可审计、更可迭代的结构基础

---

## 4. 3.0 的核心判断

3.0 不应被理解为“功能大爆发版”，而应被理解为：

> **以真实 Beta 数据为依据，对核心留存链路、匹配解释链路、运营学习链路进行系统升级的版本。**

这意味着 3.0 的成功标准不是“做了多少页面”，而是：

1. 用户关键路径是否更顺
2. 匹配结果是否更能促进继续交流
3. 产品团队是否更能看到真实问题与优化方向
4. 2.9 已建立的工程稳定性是否被守住

---

## 5. 3.0 开发原则

3.0 全程必须遵守以下原则：

1. **服务端 canonical 数据仍然是唯一真源**
2. **前端缓存只做体验优化，不得成为业务真源**
3. **不得为追求增长或演示效果破坏主链路稳定性**
4. **产品改动必须可埋点、可观察、可回滚**
5. **解释层优化优先于底层大规模算法重写**
6. **运营与治理能力必须继续与前台体验解耦**
7. **所有新增能力都要回答“如何验证有效”**

---

## 6. 3.0 版本主题

建议将 3.0 的工作聚焦为四个主题：

### Theme A：新用户转化与首周体验优化
目标：
- 提升新用户完成资料、继续问卷、查看匹配、发起首聊的转化率
- 减少首次使用时的迷失感与冷启动空白感

### Theme B：匹配解释与互动引导升级
目标：
- 让匹配结果更可读、更可信、更有行动建议
- 让用户知道下一步应该做什么，而不只是被动看结果

### Theme C：Beta 数据观测与运营学习闭环
目标：
- 建立面向真实 Beta 的核心指标体系
- 打通用户反馈、异常记录、运营观察与产品迭代之间的闭环

### Theme D：3.0 级产品质量与体验收口
目标：
- 在 2.9 的基础上继续提升稳定性、性能、一致性与主题体验
- 保证 3.0 的改动不会冲击 Beta 主链路

---

## 7. 3.0 范围拆解

## 7.1 P0（必须完成）

### P0-A 新用户关键路径优化
重点目标：
- 优化从注册 / 登录到首页首屏的理解成本
- 优化“继续问卷 / 完善资料 / 查看本周匹配 / 查看星盘画像”等关键 CTA
- 优化首页、匹配页、消息页在“有结果 / 无结果 / 弱网 / 冷启动”下的空态与提示态

建议工作：
1. 首屏关键任务卡统一
2. 首页 CTA 优先级重排
3. 冷启动空态与新手提示优化
4. 问卷未完成态的继续引导优化
5. 无会话、无匹配、无资料完整度时的引导优化

预期结果：
- 用户更清楚下一步做什么
- 首次进入首页的行动成本下降
- 空态不再只是“没有”，而是“下一步怎么做”

---

### P0-B 匹配解释层升级
重点目标：
- 提升“查看本周匹配”后的理解感与信任感
- 让用户清楚知道“为什么推荐”与“怎么开启下一步”

建议工作：
1. 匹配亮点模块结构化重写
2. “为什么适合继续聊”与“交流建议”模块加入标准模板
3. 解释内容分层：结论层 / 证据层 / 行动层
4. 与破冰问题、交流建议、画像亮点联动
5. 对“无充分证据”或“解释置信度有限”的场景做保守展示

预期结果：
- 匹配页不只展示结果，而能促成行为
- 用户对推荐结果的理解与接受度提升
- 为后续解释系统 A/B 优化建立基础

---

### P0-C 首聊与会话启动优化
重点目标：
- 提高匹配后从“看结果”到“开始聊天”的转化率
- 降低首条消息发送的心理门槛

建议工作：
1. 首聊引导卡优化
2. 破冰问题展示节奏优化
3. 匹配页到聊天页的过渡文案优化
4. 会话空态与建议语优化
5. 弱网或会话失败时的恢复引导优化

预期结果：
- 首聊启动率提高
- “有匹配但不聊天”的流失减少

---

### P0-D Beta 指标体系与反馈闭环
重点目标：
- 让 3.0 成为“可学习的 Beta 版本”
- 不再只看页面是否可打开，而要看行为是否有效

必须建立的核心指标：
1. 注册完成率
2. 问卷完成率
3. 资料完善率
4. 匹配查看率
5. 双向确认率
6. 首聊启动率
7. 首聊后 24h 持续互动率
8. 举报率 / 拉黑率
9. 关键页面异常率
10. 弱网恢复成功率

必须补齐的闭环：
1. 用户反馈入口与分类
2. 运营记录模板
3. 问题归因标签体系
4. 每周 Beta 观察总结模板
5. 产品改动前后指标对比方法

---

### P0-E 质量保护与回归升级
重点目标：
- 在推进 3.0 体验优化时，继续守住 2.9 的工程底线

必须做：
1. 3.0 Protected Surfaces 文档
2. 3.0 Beta 回归清单
3. 体验优化项的埋点核对
4. 弱网 / 冷启动 / 空态专项回归
5. 关键文案、状态与主题一致性检查

---

## 7.2 P1（强烈建议完成）

### P1-A 内容与推荐流优化
目标：
- 让首页推荐精选、关系研究、沟通技巧等内容更服务于匹配与互动
- 不再只是资讯展示，而要服务“继续交流”这一主目标

建议工作：
1. 首页内容卡分类收口
2. 内容标签与匹配阶段联动
3. 推荐内容排序策略优化
4. 从内容页返回匹配 / 聊天的转化入口优化

---

### P1-B 画像入口与资料完整度体验优化
目标：
- 提升“完善资料”“星盘画像”“性格结果”这些入口的理解度
- 避免用户觉得这些模块只是装饰，而要让它们成为匹配解释的有机部分

建议工作：
1. 资料完整度提示统一
2. 星盘画像、性格结果与匹配解释串联
3. 资料补充动作与结果价值关联展示
4. 技术性字段隐藏到更合适层级

---

### P1-C 运营侧 Beta 工具增强
目标：
- 让运营不只是“看板 + 审核”，而是能辅助学习和追踪问题

建议工作：
1. Beta 用户分层视图
2. 反馈问题分类与状态流转
3. 重点用户样本观察模板
4. 关键体验问题追踪表

---

## 7.3 P2（可延后）

以下事项不建议进入 3.0 主线，除非 Beta 数据证明其优先级明显上升：

1. 大规模底层匹配算法重写
2. 八字 / MBTI / 星盘规则引擎全面升级
3. 多媒体消息
4. 复杂社交玩法扩张
5. 完整实验平台与复杂推荐架构升级
6. 高风险数据库或服务端结构重构

---

## 8. 3.0 架构与产品边界

## 8.1 3.0 Protected Surfaces

以下链路在 3.0 中属于受保护面：

1. 注册 / 登录主链路
2. 问卷提交流程
3. 资料保存与画像重算链路
4. 每周匹配主流程
5. 双向确认主流程
6. 消息列表与聊天主链路
7. 举报 / 拉黑 / 账号状态反馈链路
8. 后台审核与用户状态治理链路
9. 版本检查 / 健康检查链路
10. 2.9 已建立的 smoke / regression / rollback 体系

---

## 8.2 明确禁止项

3.0 期间禁止以下行为：

1. 为了做增长实验而绕开 canonical 真源
2. 为了快速实现页面效果而写入第二套本地规则
3. 为了做解释层优化而偷偷改变底层匹配主分语义
4. 在无埋点、无回归的前提下上线体验改动
5. 未经过 Claude 边界确认就擅自扩张任务范围
6. 把运营需求直接写死进前台逻辑，导致前后台耦合加深

---

## 9. Claude / Codex / Gemini 分工

## 9.1 Claude：编排与版本控制

Claude 在 3.0 中的职责：

1. 负责版本级拆解与优先级排序
2. 负责定义任务边界与阶段目标
3. 负责协调 Codex 与 Gemini 的调用顺序
4. 负责把多任务结果收敛成统一版本推进节奏
5. 负责决定何时进入下一阶段

Claude 的核心职责不是写最多代码，而是：

> **确保 3.0 的每一步都沿着正确版本目标推进，而不是失控扩张。**

---

## 9.2 Codex：实现与实现侧审查

Codex 在 3.0 中的职责：

1. 根据 Claude 下发的任务执行代码修改
2. 输出影响面、风险点、修改文件清单
3. 执行实现侧自检
4. 补充测试、埋点、脚本与文档
5. 对已完成任务做工程层复查

Codex 的核心要求：
- 优先最小改动
- 优先保护主链路
- 每个任务都要明确未修改范围
- 每个任务都要输出可供 Gemini 验收的交接材料

---

## 9.3 Gemini：规格定义与验收

Gemini 在 3.0 中的职责：

1. 对任务是否符合产品目标进行规格审查
2. 对任务是否越界进行检查
3. 对关键路径、空态、文案、交互一致性进行验收
4. 对回归结果、埋点结果、指标口径进行验收
5. 输出“通过 / 有条件通过 / 不通过”结论

Gemini 的核心要求：
- 以客观验收标准为准
- 明确写出未验证项
- 明确写出剩余风险
- 不因主观审美替代客观验收

## 11. 3.0 验收标准

## 11.1 产品目标验收

必须回答：

1. 首页与关键路径是否更清晰
2. 匹配结果是否更容易理解与行动
3. 首聊启动是否更顺畅
4. 空态与异常态是否更友好
5. 用户是否更清楚下一步做什么

---

## 11.2 指标目标验收

建议作为 3.0 的核心观察指标：

1. 新用户 24h 内问卷继续率
2. 新用户 24h 内资料完善率
3. 匹配结果查看率
4. 匹配后首聊启动率
5. 首聊后 24h 持续互动率
6. 会话加载失败率
7. 首页关键 CTA 点击率
8. 用户反馈提交率与问题分类分布

说明：
- 3.0 不要求所有指标绝对提升后才算通过
- 但必须要求：指标可观测、可对比、可解释

---

## 11.3 工程与稳定性验收

必须保持：

1. 2.9 已验证的健康检查与版本检查链路继续可用
2. 弱网、冷启动、导航切换不可明显退化
3. 关键主链路不可出现 Sev1 / Sev2 新问题
4. 所有 P0 改动都有 smoke / regression 记录
5. 关键体验优化项都有埋点或观测依据

---

## 11.4 验收结论等级

Gemini 对 3.0 子任务与总任务，应统一采用以下结论等级：

- **通过**：目标完成、未越界、受保护面未受损、可进入下一阶段
- **有条件通过**：主体完成，但存在未验证项或剩余风险，需要补件
- **不通过**：目标未完成、越界、误伤受保护面或缺少关键验证
