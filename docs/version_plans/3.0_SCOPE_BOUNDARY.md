# EliteSync 3.0 Scope Boundary

更新时间：2026-04-08

当前基线版本：`0.02.09 / 209`

## 1. 版本边界目标

3.0 的目标是“Beta 运营与匹配质量提升”，不是无边界加功能。

边界控制核心：

1. 守住 2.9 工程稳定底座
2. 优先做 P0 的关键路径、解释层、首聊引导、观测闭环
3. 禁止跨边界扩张与隐性语义改写

---

## 2. In-Scope / Out-of-Scope

### 2.1 P0（必须完成）

#### In-Scope
1. 新用户关键路径优化（首页首屏任务卡、核心 CTA 优先级）
2. 空态/弱网/冷启动提示与恢复引导统一（首页/匹配/消息）
3. 匹配解释层三段式升级（结论/证据/行动）
4. 首聊启动与会话空态引导优化
5. 指标基线、事件字典、反馈分类闭环定义
6. 3.0 保护面与回归清单升级

#### Out-of-Scope
1. 大规模底层匹配算法重写
2. 重做注册/登录架构
3. 重构消息底层协议
4. 大规模数据库 schema 改造

---

### 2.2 P1（强烈建议）

#### In-Scope
1. 首页内容流与匹配阶段联动优化
2. 资料完整度提示与画像入口理解优化
3. 运营侧 Beta 问题追踪与样本观察模板增强

#### Out-of-Scope
1. 完整实验平台建设
2. 多媒体消息体系
3. 大规模推荐架构升级

---

### 2.3 P2（可延后）

#### In-Scope
1. 3.x 后续研究性课题预研
2. 高复杂度玩法评估

#### Out-of-Scope（3.0 主线禁止纳入）
1. 八字/MBTI/星盘规则引擎全面升级
2. 高风险服务端结构重构
3. 与 3.0 主目标弱相关的功能扩张

---

## 3. 明确禁止项

3.0 开发期间，禁止以下行为：

1. 绕开服务端 canonical 真源，新增“第二真源”
2. 解释层优化时偷偷修改底层匹配主分语义
3. 无埋点、无回归、无验收证据就上线体验改动
4. 为了局部体验改动误伤受保护主链路
5. 把运营策略硬编码到前台核心流程导致耦合加深
6. 未经边界确认直接把 P1/P2 内容塞入 P0 批次

---

## 4. 变更控制规则（Change Control）

1. 任何跨 P0/P1/P2 的扩张，必须先由 Claude 明确边界变更说明。
2. 涉及保护面（认证、问卷、资料保存、匹配、消息、治理、版本/健康检查）必须先有风险与回滚点。
3. 任一任务进入实现前，必须明确：
   - 不改什么（Non-goal）
   - 如何验证（最小验证集）
   - 失败如何回退（Rollback）
4. reviewer-codex 发现边界越界时，可直接判定“需返工”。
5. spec-auditor-gemini 对关键路径一致性与验收口径拥有否决权（不通过即不进入下一阶段）。

---

## 5. 边界审计最小清单

每个子任务提交时至少回答：

1. 本任务属于 P0/P1/P2 哪一级？
2. 是否触碰受保护面？触碰了哪些？
3. 是否改变 canonical 数据语义？（默认不允许）
4. 是否新增了可观测事件或验收证据？
5. 是否存在未验证项与剩余风险？

若上述任一问题无法回答，则任务不得进入合并阶段。
