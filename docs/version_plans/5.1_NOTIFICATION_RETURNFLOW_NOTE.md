# EliteSync 5.1E Notification Returnflow Note

更新时间：2026-05-01

## 目标

5.1E 将通知中心从提醒列表增强为关系回流中枢，让消息、动态、匹配、语音通知具备更明确的处理语义。

本轮只做 Flutter 前端 additive 改动，不改后端 notification payload contract，不新增推送平台能力。

## 修改范围

- `apps/flutter_elitesync_module/lib/features/notification/presentation/pages/notification_center_page.dart`
- `apps/flutter_elitesync_module/test/features/notification/presentation/pages/notification_center_page_test.dart`

## 当前实现

### 1. 通知分类

通知卡新增分类 chip：

- `消息回流`
- `动态互动`
- `匹配推进`
- `语音邀请`
- `低噪声提醒`

旧 `kind` 文案仍保留，例如 `新消息`、`动态被喜欢`、`语音来电`。

### 2. 动作语义

通知卡新增显式动作：

- `继续回复`
- `去看看`
- `查看匹配`
- `处理语音`
- `查看结果`
- `打开`
- `稍后处理`
- `标记已读`

这些动作复用现有 `route_name` / `route_args` 分发，不新增 route contract。

### 3. 回流保护

通知点击和有跳转目标的主动作现在先尝试打开目标页面，只有打开成功后才调用 `markRead`。

如果 route target 缺失或 route_name 不支持：

- 不标记已读；
- 显示可恢复提示；
- 保留通知在中心内，避免“已读成功掩盖跳转失败”。

如果通知本身没有 route target，主动作显示为 `仅标记已读`，允许用户明确把这类低噪声通知处理掉；卡片点击仍不会伪装成打开成功。

## 保留主链

- 不改 `NotificationRemoteDataSource`；
- 不改 `NotificationItemEntity`；
- 不改 Laravel `NotificationController` / `NotificationService`；
- 不改 notification payload 语义；
- 不把站内通知写成厂商推送能力；
- 不改消息 / 动态 / 匹配 / RTC 后端主链。

## 风险与观察项

- 仍需后续实测 message 通知中 `conversation_id` 是否为接收方需要的 peer id。
- `稍后处理` 当前只是保留在通知中心并给出本地提示，不写库。
- 未知 `route_name` 仍会降级为不可打开提示。

## 验收口径

- 通知卡能看到分类与产品化动作。
- 未知或缺失 route 不会被错误标记已读。
- 无 route 通知的 `仅标记已读` 主动作可明确标记已读。
- 已知 kind 但无 route 的通知，也必须优先显示 `仅标记已读`，不能显示会误导的回流 CTA。
- `user_id` / `call_id` route 参数兼容数字与数字字符串。
- 消息 / 动态 / 匹配 / 语音回流仍使用既有页面。
- 不出现工程 slug。
