# EliteSync 6.0-A1 IP-based Controlled Staging Plan

## 0. 当前结论

- 由于当前无法完成 ICP 备案，`staging.slowdate.top` 域名 staging 方案短期不可行。
- 可以继续通过 IP 进行受控开发与验证，但不得把 IP 公开服务当作备案替代品。
- 不能把 `https://slowdate.top/` 当 staging。
- 不能把 `http://101.133.161.203/` 根路径当 staging。
- 只有在建立独立端口 / 独立环境 / 独立配置 / 独立测试数据边界后，才能称为 IP-based controlled staging。
- 本计划不代表已经创建 staging。
- 本计划不代表 staging verification passed。
- 本计划不代表 production verification passed。
- 本计划不代表 Candidate C implementation。
- 本计划不代表 A1 final acceptance。

## 2. 为什么仍可用 IP 继续开发

- IP 可用于受控开发、内部测试、只读 metadata verification preparation。
- IP 不应作为公开上线入口。
- IP-based staging 必须是受控测试环境，不得混入 production/default。
- 若涉及公网访问，应尽量通过安全组白名单、VPN、SSH tunnel、临时端口、临时开关降低暴露面。

## 3. IP-based controlled staging 候选方案

### Option A: same server + separate port

示例：

`http://101.133.161.203:<staging_port>/`

要求：

- 独立 Nginx listen port。
- 独立 Laravel `APP_ENV=staging`。
- 独立 DB 或测试库。
- 独立 Redis prefix / DB index。
- 独立 storage path。
- 明确 environment marker。
- security group 只对白名单开放。
- 可随时关闭端口。

风险：

- 与 production 同机，必须严格隔离 Nginx server block、PHP-FPM / Laravel env、DB、Redis、storage。
- 如果安全组或 Nginx 端口暴露控制不严，公网攻击面会增加。
- 操作便利性较高，边界清晰度中等。

### Option B: same server + SSH tunnel only

要求：

- staging 不直接公网开放。
- 通过 SSH tunnel 访问 localhost staging port。
- 最适合只读验证。
- 缺点是 Codex / 外部工具访问流程更复杂。

风险：

- 对只读 metadata verification 最安全，公网暴露面最低。
- 需要稳定 SSH 连接与清晰 tunnel 操作步骤。
- 外部工具、GPT 顾问复核或自动化证据采集流程会更复杂。

### Option C: separate lightweight staging server

要求：

- 独立轻量云服务器。
- 独立 IP。
- 独立 DB / Redis / storage。
- 最清晰，但成本和配置工作量更高。

风险：

- 环境边界最清楚，误触 production 风险最低。
- 需要额外云资源、部署脚本、监控、备份和安全组配置。
- 对当前 A1 只读 metadata verification 来说，成本与准备时间最高。

| Option | Boundary clarity | Exposure | Setup cost | Fit for A1 readonly metadata verification |
|---|---|---|---|---|
| A: same server + separate port | medium | medium if allowlisted | medium | good if security group allowlist is enforced |
| B: same server + SSH tunnel only | medium-high | low | medium | best for narrow readonly verification |
| C: separate lightweight staging server | high | medium if public, low if allowlisted | high | best long-term boundary, heavier for A1 |

## 4. 推荐方案

Codex 推荐：

- 当前阶段优先推荐 `Option B: same server + SSH tunnel only`，因为 A1 目标仍只是 future staging readonly metadata verification，SSH tunnel 能最大限度减少公网暴露面。
- 如果 GPT 顾问或后续执行链需要更稳定、可重复、可被外部工具访问的 staging base URL，则推荐 `Option A: same server + separate port + security group allowlist` 作为操作性更强的替代方案。
- 若后续进入更长期的 staging / QA / release gate，且成本允许，再升级为 `Option C: separate lightweight staging server`。
- 不推荐直接使用 `http://101.133.161.203/` 根路径。
- 不推荐继续尝试 `staging.slowdate.top`，除非未来备案完成。
- 不推荐把 `https://slowdate.top/` 作为 staging。

## 5. 最小环境边界

IP-based controlled staging 至少需要满足：

- `APP_ENV=staging`。
- `APP_DEBUG=false` 或受控。
- 独立 DB / 测试库。
- 独立 Redis prefix / DB index。
- 独立 storage path。
- 不使用 production 用户数据，或必须只读脱敏。
- 不影响 `app_release_versions` production release chain。
- 不影响 Android release/default base URL。
- 不切 Flutter default base URL。
- R1 endpoint allowlist only。
- 明确 environment marker。

4. 是否允许新增 staging env。
5. 是否允许新增 staging DB / 测试库。
6. 是否允许新增 Redis prefix / DB index。

8. staging 端口是否公网开放，还是仅 VPN / SSH tunnel / IP 白名单。
9. 是否允许 Codex 执行服务器命令。
10. 是否需要先备份 production config。
