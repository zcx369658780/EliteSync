# EliteSync 5.3 版本开发计划书

更新时间：2026-05-02  
适用对象：项目负责人 / Codex / Claude / Gemini / 顾问归档  
版本状态：正式版 v1  
当前前置版本：`5.2` 已按 `pass with observations` 收口  
当前稳定门禁基线：`4.9`  
当前对外发布版本：`0.04.09 / 40900`  
版本定位：`功能覆盖收尾版 / Soul 高价值主链第二轮补齐版`

---

## 0. 本文档定位

本文档用于在 `5.2` 完成并正式收口后，为 EliteSync 制定下一阶段 `5.3` 的正式开发计划。

`5.3` 不是旧路线中的“治理优先版”，也不是回头重开 `4.x / 5.0 / 5.1 / 5.2` 主链的补丁容器。  
它的定位已经明确改写为：

> **在不破坏 `4.9` 稳定门禁基线与 `5.0 / 5.1 / 5.2` 已验收成果的前提下，继续补齐 EliteSync 与 Soul 高价值主链对应的剩余关键功能，完成 5.x 第一阶段的功能覆盖收尾。**

本版主要解决：

1. Discover 仍未完全产品化的搜索、热词、同城理由、action sheet 与低压转化问题  
2. Chat 仍停留在候选位或半成品状态的“稍后再聊 / 关系摘要 / AI 续话 / 轻互动”问题  
3. Me / Appearance 已有结构但仍偏浅层的外观、展示标识、经营反馈与内容经营联动问题  
4. Settings 与 Appearance 之间仍需继续收敛的边界、权限前解释与体验开关组织问题  
5. 当前版本验收、截图证据、回归材料与单文件 handoff 收口问题  

本版明确不解决：

- 5.4 才进入的测试运营治理主线  
- 微服务拆分、重型中台、多人 RTC、直播、商城、币体系、礼物、重商业化装扮中心  
- 重新设计 `4.9` 已固定的 release gate / rollback / recovery / database drill / RTC observability 主链  
- 把 5.3 重新写成“继续补后台治理”的版本

## 1. 为什么 5.3 必须单独立项

根据当前主计划与路线图，项目已经完成以下前提：

- `4.9` 已冻结为稳定门禁基线  
- `5.0` 已完成最小产品化覆盖集  
- `5.1` 已完成关系推进与内容转化补齐  
- `5.2` 已完成个人经营中枢与表达层覆盖  
- Soul Stage2 已足够支撑 Discover / Chat / Me / Settings 的第二轮版本规划  

这意味着当前阶段真正的短板，已经不再是：

- 后台治理还不够厚  
- 日志系统还不够全  
- 运维便利性还不够强  

而是：

- 高价值主链虽然已有第一轮覆盖，但仍存在若干“像候选位、不像正式产品能力”的缺口  
- Discover / Chat / Me / Settings 之间的关系推进闭环仍可继续收敛  
- 若此时不完成 5.3，后续直接进入 5.4，会形成“治理很厚但关键产品功能仍有缺口”的倒挂  

因此，`5.3` 必须作为独立版本成立，其核心任务不是重复建设基础设施，而是：

> **在资质获批前，尽量把用户未来高频会用到的高价值主链功能补到更正式、更顺滑、更像产品，而不是像半成品。**

## 2. 承接基线与硬性保护面

## 2.1 承接基线

`5.3` 默认承接以下基线：

- `4.9`：稳定门禁基线  
- `5.0`：最小产品化覆盖集  
- `5.1`：关系推进与内容转化补齐版  
- `5.2`：个人经营中枢与表达层覆盖版  

当前默认验收入口：

- `docs/version_plans/5.2_HANDOFF_MASTER.md`

## 2.2 硬性保护面

`5.3` 期间不得破坏以下保护面：

### 资料 / 画像 / 星盘真值链
- `POST /api/v1/profile/basic`
- `GET /api/v1/profile/basic`
- `GET /api/v1/profile/astro/summary`
- `GET /api/v1/profile/astro/chart`
- `user_astro_profiles`

规则：
- canonical truth 仍只能由服务端写入  
- 前端缓存只能兜底，不能替代真值  
- 标签、展示建议、AI 草稿、经营反馈不得回写真值  

### 消息 / 媒体 / 动态 / 通知主链
- `chat_messages`
- `conversations`
- `conversation_members`
- `media_assets`
- `message_attachments`
- `status_posts`
- `notifications`

规则：
- 5.3 只能做产品层增强与路径收敛  
- 不重写图片 / 视频 / 动态 / 通知已归档主链  
- 不把 Discover / Chat 扩成重社区或重娱乐平台  

### RTC / LiveKit 主链
- `POST /api/v1/rtc/calls`
- `GET /api/v1/rtc/calls/{callId}/livekit`
- `POST /api/v1/rtc/calls/{callId}/heartbeat`
- `rtc_calls`
- `rtc_sessions`

规则：
- 语音继续作为关系推进节点复用  
- 不重写 RTC 状态机  
- 不扩多人、直播、视频房间  
- 不把 5.3 做成媒体基础设施版  

### 版本链与发布门禁
- `GET /api/v1/app/health`
- `GET /api/v1/app/version/check`
- `app_release_versions`
- `apps/android/app/build.gradle.kts`
- `services/backend-laravel/config/app_update.php`
- `scripts/release_android_update_aliyun.ps1`

规则：
- 本版默认不触碰 release chain  
- 如后续确需出包或更新版本中心，必须单独说明，不允许顺手修改  

### UI protected surfaces
- 首页
- Discover
- 匹配页
- 消息页
- 聊天页
- 通知中心
- 个人页
- 设置页
- 版本中心
- 整体现代 UI 结构、卡片、间距、底部导航与入口拓扑

规则：
- 任何功能覆盖不得以 UI 回退为代价  
- 禁止 repo 级恢复  
- 恢复只能 `path-level / file-level / smallest-scope`

## 3. 本版范围与非目标

## 3.1 本版范围

`5.3` 只做下面四大产品域的第二轮补齐与收尾：

### A. Discover
- 搜索页进一步产品化  
- 热词 / 提示卡 / 搜索建议继续补齐  
- 同城理由与推荐理由继续清晰化  
- 内容卡 action sheet 收敛为更正式的轻治理面板  
- 内容 / 状态卡到私聊的低压转化继续收敛  

### B. Chat
- “稍后再聊”从候选位推进为更真实的队列体验  
- 关系摘要继续加厚，但保持轻量  
- AI 续话 / 首聊 / 回聊建议更贴近输入动作  
- 查看主页 / 聊天设置 / 关系节奏化语音入口继续收敛  
- 轻互动候选位继续补齐，但不做重娱乐化  

### C. Me / Appearance
- AI 助理与经营反馈继续加厚  
- 展示标识、内容经营与资料经营联动更顺  
- 外观层细节与文案继续补齐  
- 对 5.2 已有能力做第二轮产品化 polish，但不重开主链  

### D. Settings
- 真设置中心继续收敛  
- Appearance 与 Settings 边界进一步清晰  
- 权限前解释更明确  
- AI 开关 / 辅助开关 / 体验开关组织更清楚  

## 3.2 本版明确非目标

以下内容一律不纳入 `5.3`：

- 业务管理后台、云端只读管理台、审计台  
- 数据库迁移、恢复链重做、只读优先 DB 面板  
- 搜索后端平台化、通知中台化、Worker 统一观测台  
- 商城、币体系、会员体系、礼物、重商业化装扮  
- 多人 RTC、直播、平台化音视频  
- Match 主链重构  
- Message / Media / Notification / RTC contract 重写  
- profile canonical truth 重写  
- 把 5.3 observations 变成 5.2 主链重开

## 4. 版本级执行原则

## 4.1 总原则

`5.3` 必须遵守：

> **先在用户面形成正式入口、正式页面、正式链路，再追求更厚的治理与后台便利性。**

## 4.2 允许的实现方式

当前阶段允许：

- 页面层补齐  
- 前端状态与局部产品结构补齐  
- 轻量本地状态持有  
- 已有接口之上的 additive 承接  
- 不破坏真值链的展示层与交互层增强  
- 对必要小范围 contract 进行边界说明后再实施  

## 4.3 默认禁止的实现方式

默认禁止：

- 一上来重写大页面  
- 为了“更完整”而引入大范围后端改造  
- 为了做 5.3 而顺手动 release chain  
- 让本地开发环境承担高风险写库动作  
- 在无 blocker report 的情况下长时间盲修  
- 以恢复旧 commit 的方式“解决”新问题

## 5. 工作包拆分

# 5.3A Discover 第二轮补齐

## 目标

让 Discover 更明确地体现“复合发现入口”而不是半成品内容页。

## 重点任务

- 搜索页继续产品化：搜索框、热词、提示卡、推荐提示更完整  
- 同城页继续加厚：理由、提示、轻探索路径更自然  
- 推荐理由 / 值得聊理由继续可读化  
- action sheet 收敛为更完整的轻治理与轻转化面板  
- 内容 / 状态卡到私聊的低压转化继续补齐  

## 边界

- 不做重推荐平台  
- 不做重社区 feed 扩写  
- 不强推定位授权  
- 不做强商业化入口  
- 不把分享动作做成主噪音  

## 最小交付判断

满足以下条件可视为 5.3A 成立：

- Discover 搜索入口不再像占位页  
- 同城 / 推荐理由至少有一套正式表达  
- action sheet 能同时承担轻治理与轻转化  
- 至少形成一条“内容 -> 理由 -> 私聊”用户面证据链  

# 5.3B Chat 第二轮补齐

## 目标

让 Chat 更明确地承担关系推进中枢，而不是只在文档里看起来完整。

## 重点任务

- “稍后再聊”继续从候选位走向真实队列体验  
- 关系摘要继续加厚：共同点、节奏说明、查看主页、设置入口更自然  
- AI 首聊 / 回聊 / 续话建议更贴近输入框  
- 轻互动候选位继续补齐，但保持低压  
- 语音入口继续保持“关系推进节点”语义，而不是裸功能按钮  

## 边界

- 不重开 RTC 主链  
- 不扩礼物、商城、直播、多人互动  
- 不把 Chat 做成重运营列表  
- 不引入假成功 AI 自动发送  
- 不做复杂后端回流系统，除非已有接口无法承接且先完成边界说明  

## 最小交付判断

满足以下条件可视为 5.3B 成立：

- Chat 至少有一条更正式的“稍后再聊 / 回聊”队列或回流体验  
- 关系摘要比 5.1 更贴近实际聊天动作  
- AI 建议与输入动作位置关系更自然  
- 查看主页 / 聊天设置 / 语音节奏之间的路径更清楚  

# 5.3C Me / Appearance 第二轮补齐

## 目标

在 `5.2` 已建立个人经营中枢的基础上，把“经营感”与“表达感”做得更像正式能力。

## 重点任务

- AI 助理从建议入口继续扩成更明确的经营反馈结构  
- 内容经营与资料经营联动更清楚  
- 展示标识继续补齐  
- 外观层继续 polish：预览感、说明感、边界感更清楚  
- 对 `5.2` 保留 observation 中适合本轮顺手解决的小 UI 问题做最小收敛  

## 边界

- 不做外观持久化装扮系统  
- 不做付费 dressing center  
- 不做 AI 自动发布  
- 不做声音名片录制闭环  
- 不重写 profile truth  

## 最小交付判断

满足以下条件可视为 5.3C 成立：

- Me 页能更明确表达“被理解 + 被继续了解 + 自我经营”  
- AI 助理 / 展示建议 / 内容经营之间形成更清楚的结构  
- 外观层不与真设置中心混淆  
- 5.2 已有能力在用户面更顺、更像正式产品  

# 5.3D Settings / Permission 第二轮补齐

## 目标

继续拉开 Settings 与 Appearance 的边界，并把体验开关与权限前解释做得更清楚。

## 重点任务

- 真设置中心继续收敛  
- Appearance 不再混淆为设置中心子页  
- 权限请求前增加解释或前置提示  
- AI / 辅助 / 体验类开关的组织更清楚  

## 边界

- 不进入重安全中心重构  
- 不做账号体系重写  
- 不把商业入口和基础设置混排  
- 不在 5.3 展开复杂的系统权限深链  

## 最小交付判断

满足以下条件可视为 5.3D 成立：

- Settings 与 Appearance 的边界可通过截图直接读懂  
- 至少一个权限前解释路径更清楚  
- 设置项组织比当前更像正式产品中心而非杂项聚合页  

# 5.3E 文档、证据与收口

## 目标

确保 `5.3` 不是“功能做了但没法验收”的版本。

## 重点任务

- 完成 5.3 walkthrough 截图证据  
- 完成 5.3 回归清单  
- 完成 5.3 主交接文件  
- 必要时补充 evidence index，但不制造多个平行结论文件  

## 默认文档交付

主文件：

- `docs/version_plans/5.3_HANDOFF_MASTER.md`

- `docs/version_plans/5.3_UI_BASELINE_EVIDENCE_INDEX.md`

如无必要，不再平行生成多个 acceptance / closeout / summary 入口文件。

## 7. 风险拆解

## 7.1 最高风险：版本方向漂移

风险表现：

- 把 5.3 又写回治理优先  
- 借 5.3 顺手重开旧版本主链  
- 因为想做完整功能而把版本范围无限扩大  

应对：

- 所有工作只围绕 Discover / Chat / Me / Settings 四块展开  
- 不做 5.4 才做的治理主线  
- 每轮先声明“本轮只补哪一块，不补什么”  

## 7.2 第二风险：UI 保护面回退

风险表现：

- 为追求功能补齐而破坏现代 UI  
- 借恢复旧文件解决局部问题，导致页面整体回退  

应对：

- 继续把现代 UI 当作 protected surfaces  
- 禁止 repo 级恢复  
- 只能做 path-level / file-level / smallest-scope 恢复  

## 7.3 第三风险：跨层 blocker 被误判为前端问题

风险表现：

- Codex 长时间在 Flutter/UI 层盲修  
- 页面现象与真实根因不一致  
- 实际根因位于后端 / 网络 / 权限 / 协议 / 环境  

应对：

- 一旦出现跨层 blocker，必须先写 blocker report  
- 再由 Claude 做根因缩层  
- 最后由 Codex 做最小修复  
- Gemini 只负责用户面证据与 walkthrough 复核  

## 7.4 第四风险：把候选位误做成重系统

风险表现：

- 为了把候选位做“完整”，引入大后端、大持久化、大运营结构  
- 把 5.3 从收尾版拖成平台化版本  

应对：

- 本版允许“正式入口 + 正式页面 + 最小正式链路”  
- 不要求一次做成完整运营系统  
- 所有实现优先前端产品化承接，谨慎引入后端改造
