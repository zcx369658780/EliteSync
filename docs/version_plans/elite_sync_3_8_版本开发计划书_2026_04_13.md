更新时间：2026-04-13  
适用对象：项目负责人 / Codex-CLI / Claude-MCP / Gemini-MCP / 顾问归档  
版本状态：草案 v1  
当前前置版本：`3.7` 已正式归档通过  
版本定位：`专业工作台深化版 / 参数与解读一体化首版`

---

## 0. 本文档定位

本文档用于在 `3.7` 完成归档后，为 EliteSync 制定下一阶段 `3.8` 的正式开发计划。

`3.8` 不是对 `3.7` 的零散补丁，也不是继续单纯堆叠 advanced preview 页面。  
它的定位是：

> **在不破坏 canonical truth 单源、既有资料链路与 `3.7` 高级能力基线的前提下，把 Astro Engine 从“高级能力演示与归档版”推进到“专业工作台深化版 / 参数与解读一体化首版”，重点补齐：专业参数宽度、解读层深度、精度校准闭环、交互可用性。**

本版主要解决：

1. `3.7` 已完成 pair / transit / return / comparison 的 advanced-context 接入、演示页、样例矩阵和归档，但仍更偏**演示与预览收口**，还不是同事参考图那种“参数 + 排盘 + 解读”一体化专业工作台。
2. 当前参数工作台仍是 MVP，距离同事目标中的大参数面板还有明显差距：宫位制、黄道制、相位容许度、虚点 / 小行星 / 点位集、主星 / 飞宫 / 解读开窗等还不够完整。
3. 当前高级解读层已有骨架，但还不足以支撑“参数、星体、相位、主星、大限 / 流年等逐项点开单独解读”的体验目标。
4. 精度与体验还有未闭环项：参考样例校准、字体重叠、日期输入体验、盘心结构与拥挤控制，需要系统化收口，而不是零散修 bug。

本版明确不解决：

- 全量专业占星桌面软件级别的终局能力
- canonical 引擎全面替换
- 登录 / 会话 / 地理编码 / 数据库大结构重构
- 与星盘主线无关的大规模产品线扩张

## 1. 为什么 3.8 必须单独立项

根据 `3.5`、`3.6`、`3.7` 的归档结果，当前项目已经完成：

- Engine Adapter / schema / metadata 基础设施
- 参数工作台 MVP
- `standard / classical / modern` 三路线分流
- 路线差异报告与样例矩阵
- advanced-context 的 `pair / transit / return / comparison`
- 高级演示页、预览日志、样例矩阵与已知偏差归档

但从同事的参考目标来看，目前还明显欠缺三类东西：

1. **参数宽度不够**：还没有真正接近“专业参数面板”。
2. **解读深度不够**：还没有做到逐类信息点开单独阅读与结构化解释。
3. **校准闭环不够**：还没有把参考样例、宫位偏移、行星标注重叠、日期输入等体验问题系统收口。

因此：

`3.8` 应被视为星盘引擎升级第四阶段——**专业工作台深化与精度闭环版**，而不是 `3.7` 的简单延伸。

## 3. 3.8 总目标

### 3.1 本版要解决的问题

当前 App 在 `3.7` 之后已经能：

- 维持 canonical truth 单源
- 使用三路线工作台与 advanced-context 页面
- 展示合盘 / 对比盘 / 行运 / 返照的演示与归档材料
- 保持截图证据、样例矩阵、回归与 handoff 链路

但还不能：

- 让用户在一个更接近专业网页原型的工作台里完成**参数切换 → 盘面变化 → 解读查看**的闭环
- 对主星、宫主星、飞宫、相位表、点位集给出结构化、可点开的单独解读
- 支持更丰富的宫位制 / 参数集 / 解释视图
- 对同事给出的“校准样例”和“视觉体验问题”形成正式校准报告

### 3.2 本版最终应达到的形态

`3.8` 结束后，EliteSync 应达到：

> **在现有稳定产品中形成首个真正可用的“参数 + 排盘 + 解读”工作台闭环：用户可以在更完整的参数面板中切换专业参数，看到盘面变化，打开解读弹窗 / 单独解读块，并且整个系统已经通过固定校准样例、视觉与交互回归、Claude/Gemini 审查链完成正式归档。**

必须完成：

   - 点位范围 / 点位集
2. 明确参数角色：
   - 哪些是 workbench 参数
   - 哪些是 derived-only 解释参数
   - 哪些尚未进入 canonical truth
3. 提供清晰的当前工作台摘要区
4. 支持参数快速预设 + 细粒度调整并共存
5. 保持旧设置页与旧入口兼容

说明：
- 本版要做的是“专业工作台首版”，不是全量参数海洋。

## 4.2 参数解读与单项弹窗首版

目标：
- 把“参数 / 解读 / 星体 / 相位 / 主星”从整页文字提升到可独立点开的阅读体验

必须完成：
1. 建立一套统一的解读弹窗 / 解读侧卡机制
2. 至少覆盖以下类别的单项解读：
   - 参数解读
   - 星体解读
   - 相位解读

   - 飞宫解读
3. 每类解读必须支持：
   - 当前上下文说明
   - 来源字段说明

4. 允许用户从工作台与本命盘详情页进入这些解读
5. 形成统一的解读块样式，不要每类都各做一套 UI

说明：
- 本版不要求把所有占星语义全写完，但必须先把“可点开、可独立阅读”的机制做出来。

## 7. 3.8 长期约束（不可违反）

### 7.1 真值约束

1. 服务端 canonical truth 仍是唯一真源。
2. 工作台参数、解读弹窗、主星 / 飞宫解释不得回写 `profile/basic` 或 canonical natal truth。
3. 高级能力结果与专业工作台结果都必须明确标记为：
   - canonical truth
   - derived analysis
   - display-only explanation
4. 任何新解释字段都不得伪装成“绝对真值”。

### 7.2 架构约束

1. 必须继续坚持 additive change。
2. 不删旧字段、不破坏旧读取路径、不重写 `chart_data` 主结构。
3. 参数扩展必须有默认 fallback。
4. 不得在没有校准样例和回归矩阵的情况下大改解释逻辑。

### 7.3 UI / 工作流约束

1. Gemini-MCP 仍为强制 UI / 可读性 / 验收 subagent。
2. Claude-MCP 仍为强制架构边界 / 解释字段来源 / 校准策略审查 subagent。
3. Codex-CLI 不得跳过 Claude 的结构审查就定新的解释字段。
4. Codex-CLI 不得跳过 Gemini 的阶段验收就冻结工作台布局。
5. 页面不得再次演化成“只有开发者看得懂的后台面板”。

### 7.4 发布与回归约束

1. 不得破坏 `3.7` 稳定基线。
2. 旧资料页、旧星盘页、旧设置页、旧版本中心、旧高级演示页必须可回归。
3. 新专业工作台能力默认支持 feature flag / 版本门禁。
4. 若高级能力与专业工作台能力冲突，优先保护旧入口与 truth boundary。

## 11. 3.8 验收标准

## 11.1 功能验收

必须满足：
1. 参数工作台已明显深化，不再只是 MVP
2. 至少五类单项解读入口可用
3. 主星 / 宫主星 / 飞宫解释首版可用
4. 日期输入体验已优化
5. 至少一组正式校准样例已纳入并产出报告
6. 参数 → 盘面 → 解读闭环成立

## 11.2 边界验收

必须满足：
1. canonical truth 未被误写
2. 解读层明确属于 derived analysis / display-only explanation
3. `chart_data`、`engine_info`、`metadata` 旧字段读取逻辑未被破坏
4. 旧缓存、旧客户端、旧入口可兼容
5. `route_mode` 与 advanced-context 边界不被破坏

## 11.3 流程验收

必须满足：
1. Claude-MCP 阶段 1 / 阶段 3 至少有可归档审查记录
2. Gemini-MCP 阶段 2 / 阶段 3 / 最终截图复核有可归档记录
3. 关键测试、截图、校准报告、回归清单、已知偏差文档齐全
4. handoff / acceptance / review log / calibration / regression 一次补齐

## 11.4 不通过条件

出现以下任一情况，`3.8` 不得宣告通过：

1. 工作台参数或解释结果被写成 canonical truth
2. 旧页面 / 旧入口 / 旧缓存被破坏
3. Claude 或 Gemini 的强制审查链缺失
4. 没有校准样例与校准报告
5. 工作台严重过载、不可读，且未收口
6. 关键测试未通过却仍试图归档
