## 当前通话算法说明

### 1. 通话状态机是真源

当前通话主链仍然由后端状态机驱动：

- `rtc_sessions`
- `rtc_session_events`
- 呼叫 / 接听 / 拒绝 / 挂断 / 超时 / 忙线

状态机负责的是：

- 谁发起
- 谁接听
- 何时进入 `calling / ringing / connecting / in_call / ended / failed`
- 终态如何回收

它**不负责真正传输语音**，只负责通话生命周期。

### 2. LiveKit 负责语音媒体层

当会话进入 `connecting / in_call` 后，Flutter 端会调用：

- `/api/v1/rtc/calls/{callId}/livekit`

后端返回：

- `url`
- `token`
- `room`
- `identity`
- `participant_name`

Flutter 端用 `RtcLiveKitService` 连接 LiveKit 房间，并执行：

- `Room(roomOptions: adaptiveStream + dynacast)`
- `room.connect(...)`
- `room.startAudio()`
- `room.setSpeakerOn(true)`
- `localParticipant.setMicrophoneEnabled(true)`

### 3. 远端音轨的获取逻辑

当前实现不是只靠一次订阅事件，而是双保险：

- 监听 `TrackSubscribedEvent`
- 监听 `TrackUnsubscribedEvent`
- 监听房间和参与者状态变动
- 连接后从 `room.remoteParticipants` 扫描已订阅的音频轨道

如果发现远端音轨，会把它放入 `remoteAudioTrack`，再驱动通话页上的音频频谱条刷新。

### 4. UI 诊断层

`RtcCallPage` 上新增了 `RtcAudioSpectrumBar`：

- 有远端音轨时，显示实时波形条
- 没有远端音轨时，显示“等待远端音轨…”

这条频谱只用于诊断，不参与业务状态判定。

## 已成立的事实

- 来电能唤起
- 双端能进入通话页
- 双端能自动接通
- 双端能挂断并回到结果页
- 后端 join-info 接口可用
- LiveKit 云端容器可用
- RTC 主链本身没有再崩

## 当前问题

当前仍存在的核心问题是：

- 双端都能连上通话，但**没有实际听到声音**
- 音频频谱条仍然显示“等待远端音轨…”
- 说明前端没有稳定拿到可播放的远端音轨，或者音轨到了但音频路由 / 播放链仍未闭合

当前更像是以下几类问题之一：

- LiveKit 端没有真正把远端音频轨道推到前端
- 音轨订阅成功，但 remote audio track 没有被稳定识别
- 音频播放链路接通了，但设备路由 / 输出未闭合
- 现有诊断 UI 仍不足以直接证明声波已经进入本地播放链

## 证据现状

- 后端 RTC 测试通过
- Flutter RTC 页面和频谱条测试通过
- `RtcLiveKitService` / `RtcCallPage` / `RtcAudioSpectrumBar` 已完成静态检查

  - 呼叫状态已走到 `created -> accepted -> connected -> ended`
  - 但音频频谱仍停在等待态
