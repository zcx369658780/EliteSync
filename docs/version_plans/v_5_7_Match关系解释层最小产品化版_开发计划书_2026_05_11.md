# 2. 版本定位

## 2.1 本版本是什么

5.7 是 Match 关系解释层的最小产品化版本。

它把 5.6 冻结的解释层原则落到一个用户最容易理解的行动点：

```text
用户看到一个匹配对象时，知道为什么值得聊、可以从哪里开场、哪些表达方式更合适、哪些表达方式要避免。
```

5.7 只做最小可验证闭环：

- 在 Match detail / Match result 中新增一个解释区块；
- 解释区块标题建议为 `为什么值得聊`；
- 包含一句关系摘要；
- 包含 2-4 个解释维度；
- 包含建议 / 避免；
- 包含轻追问占位或 disabled / coming soon 入口；
- 包含 display-only 免责声明；
- 保持用户可忽略、可继续使用原 Match 主流程。

## 2.2 本版本不是什么

5.7 不是：

- 玄学合盘完整实现；
- 测测缘分合盘复刻；
- 真实 AI 问答接入；
- 后端解释引擎；
- 新匹配算法；
- 新数据库字段；
- 新 API contract；
- 付费报告；
- 真人咨询；
- 自动代聊；
- 自动发消息；
- release / APK 发布任务。

## 2.3 本版本验收口径

5.7 可以验收为：

```text
pass with observations
```

前提是：

- Match 页面解释层用户可见；
- 解释层保持 display-only；
- 未改 API / DB / release / backend；
- 旧 Match 主流程不被破坏；
- Chat / Me / Discover / Settings protected surfaces 未被误伤；
- 有截图 / XML 或可替代 UI 证据；
- 有回归清单与 5.7 handoff master；
- observations 不阻断进入 5.8 计划阶段。

# 3. 产品目标

## 3.1 用户目标

5.7 要帮助用户解决三个问题：

1. **为什么这个人值得聊？**
   - 给出一句克制的关系摘要。

2. **我可以从哪里开场？**
   - 给出共同点、表达节奏或慢约会适配相关的提示。

3. **我应该注意什么？**
   - 给出建议 / 避免，不做心理诊断，不做确定预测。

## 3.2 产品目标

5.7 要完成从“匹配结果展示”到“匹配理由可理解”的最小升级。

它应增强 Match 的用户行动价值，但不能压过原匹配主流程。

建议产品结构：

```text
Match detail / result
  ├─ 原有匹配对象信息
  ├─ 原有匹配解释 / 转化入口
  ├─ 新增：为什么值得聊
  │   ├─ 一句关系摘要
  │   ├─ 2-4 个解释维度
  │   ├─ 建议 / 避免
  │   ├─ 轻追问占位
  │   └─ display-only 免责声明
  └─ 原有聊天 / 返回 / 继续浏览入口
```

## 3.3 设计语言

解释层文案应使用 EliteSync 自有语言，不复制测测术语。

推荐用词：

- `为什么值得聊`
- `可以从这里开场`
- `你们的共同点`
- `表达节奏`
- `慢约会适配度`
- `建议这样开始`
- `先避免这样表达`
- `仅供关系理解参考，不会写入资料或影响匹配算法`

避免用词：

- `天命`
- `宿命姻缘`
- `必然适合`
- `绝配`
- `灵魂绑定`
- `诊断`
- `预测`
- `付费解锁`
- `大师解读`
- `真人咨询`
- `上传截图`

# 4. 功能范围

## 4.1 必做功能

5.7 必须完成以下最小功能：

### 4.1.1 关系解释卡

新增一张 Match 页面内的关系解释卡。

建议标题：

```text
为什么值得聊
```

建议副标题：

```text
这些提示只帮助你理解开场方向，不会改变匹配结果。
```

### 4.1.2 一句关系摘要

摘要应短、克制、非绝对化。

示例：

```text
你们有相近的表达节奏，适合从轻松的话题慢慢开始。
```

或：

```text
你们的共同点适合做一个自然开场，但不代表关系结果已经确定。
```

### 4.1.3 解释维度

首版建议维度为 3 个：

1. `共同点`
2. `表达节奏`
3. `慢约会适配度`

可选第 4 个：

4. `开场安全感`

每个维度建议包含：

- 维度标题；
- 一句解释；
- 可选轻标签；
- 不使用分数堆叠，不使用黑箱权重。

### 4.1.4 建议 / 避免

解释卡底部增加两个轻量区块：

```text
建议这样开始
先避免这样表达
```

示例建议：

- `从共同兴趣或城市经历问起。`
- `先用一句轻问候，不要急着推进关系。`
- `可以先回应对方资料里的具体细节。`

示例避免：

- `避免一开始就问隐私或强推进见面。`
- `避免把星盘或匹配提示说成确定结论。`
- `避免连续发送太多长消息。`

### 4.1.5 轻追问占位

首版只做 disabled / coming soon 或受控占位，不接真实 AI。

建议文案：

```text
想换个更自然的开场？后续会支持轻追问建议。
```

按钮状态建议：

```text
轻追问建议 · 敬请期待
```

或：

```text
换个开场说法（暂未开启）
```

按钮可以 disabled，或点击后弹出本地说明 dialog，但不得调用网络、不得创建消息、不得进入 Chat 写入。

### 4.1.6 display-only 免责声明

必须清楚可见：

```text
以上内容仅为关系理解与开场参考，不会写入资料，不会改变匹配算法，也不会自动发送消息。
```

# 7. 实现原则

## 7.1 数据来源原则

5.7 不新增 API，只能使用当前 Match 页面已有数据上下文。

解释层数据可以来自：

- 已有匹配对象基本资料；
- 已有匹配解释字段；
- 已有共同点 / 标签 / 城市 / 人格 / 星盘摘要展示信息；
- 本地 deterministic presentation helper；
- 静态 fallback copy。

如果当前 Match 数据不足，宁可展示克制 fallback，不得新增接口。

示例 fallback：

```text
当前资料还不够完整，先从一个轻松问候开始会更稳妥。
```

## 7.2 deterministic helper 原则

首版解释层可使用本地 deterministic helper，根据已有展示字段生成文案。

但 helper 必须：

- 只在 presentation 层；
- 不写入缓存；
- 不写入 server；
- 不改变 repository / API contract；
- 不伪装成 AI；
- 不输出绝对判断。

## 7.3 UI 原则

解释层必须：

- 可读；
- 克制；
- 不像工程面板；
- 不堆技术术语；
- 不干扰原 Match 主行动；
- 不让失败态比成功态更重；
- 不出现强警告压过主流程。

建议视觉结构：

- 卡片式区块；
- 标题 + 简短说明；
- 2-4 个维度 chip / row；
- 建议 / 避免双列或上下分组；
- 轻追问 disabled button；
- 免责声明小字。

## 7.4 可访问性与截图原则

必须确保关键文本能在 Android UI hierarchy 中被读取。

至少应能采集到：

- `为什么值得聊`
- `共同点`
- `表达节奏`
- `慢约会适配度`
- `建议这样开始`
- `先避免这样表达`
- `不会写入资料`
- `不会改变匹配算法`
- `不会自动发送消息`
