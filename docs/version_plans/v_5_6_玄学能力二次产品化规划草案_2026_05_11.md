## 1. 版本定位

5.6 暂定为“玄学能力二次产品化”规划版本。

本版本不是直接实现版本，不进入 Flutter / Laravel / 数据库 / API / 发版链。它的目标是把测测 Stage 1-5 的竞品 UI 观察转化为 EliteSync 自己的产品边界、信息架构、最小切片和验收前置条件。

5.6 要解决的问题不是“把测测功能搬进 EliteSync”，而是回答：

- EliteSync 是否需要一个关系解释 / 个人解释能力层。
- 这些解释能力应该放在 Match、Me、Chat 还是 AI 助手。
- 哪些能力只能作为 derived-only / display-only。
- 哪些能力绝不能反写用户资料真值。
- 第一刀最小切片应该从哪里做。

## 2. 为什么从 5.5 之后进入玄学能力二次产品化

5.5 已经围绕真实小样本反馈做了产品吸收，EliteSync 当前主线从“基础功能覆盖”进入“高价值主链产品化”的阶段。

测测 Stage 1-5 研究说明，玄学能力的产品价值不在于术语本身，而在于它把抽象关系问题包装成可读、可追问、可承接的解释层：

- Stage 1 证明玄学能力可以以入口矩阵方式嵌入主产品，而不是孤立工具。
- Stage 2 证明 AI 解释层和真人服务层必须分离。
- Stage 3 证明测试类能力容易变成互动入口，不能当成真实画像来源。
- Stage 4 证明单人档案上下文可以承接图谱、摘要、解释和轻追问。
- Stage 5 证明双人关系上下文可以承接关系语义、维度评分、建议 / 避免和轻追问。

EliteSync 的后续关键增长点是 Match、Chat、Me 的关系理解和表达效率。玄学能力二次产品化可以作为一种解释层语言，帮助用户更快理解自己、理解匹配对象、理解当前关系节奏。

但该能力必须被严格限制在解释层，不进入资料真值链。

## 3. 测测 Stage 1-5 可吸收结论

### Stage 1：入口矩阵

可吸收：

- 把解释能力拆成多个轻入口，而不是一个重工具页。
- 能力入口可以分布在首页、个人页、关系页和聊天页。
- 入口要服务主链，不应把用户从关系主流程中拉走太远。

不可照搬：

- 不复制高密度工具入口。
- 不把首页改造成玄学工具集合。
- 不引入强商业入口或服务市场。

### Stage 2：AI 问题导航与在线服务分离

可吸收：

- 从具体问题进入解释，比让用户面对空白 AI 输入框更低压。
- AI 追问可以作为解释后的下一步。

不可照搬：

- 不规划真人玄学咨询市场。
- 不规划达人服务大厅。
- 不把 AI 解释入口和真人服务入口混在一起。

### Stage 3：测试频道收窄

可吸收：

- 互动测试适合作为轻量自我理解入口。
- 结果表达可以用于启发用户对关系和自我状态的理解。

不可照搬：

- 不把测试结果写入画像。
- 不把测试结果当成事实判断。
- 不规划大规模题库或创作者测试市场。

### Stage 4：单人档案解释层

可吸收：

- 单人档案上下文可以承接个人解释层。
- 先给结构化对象，再给摘要和解释，有助于降低理解成本。
- AI 追问适合放在解释之后。

不可照搬：

- 不复制测测星盘术语体系。
- 不强收出生资料。
- 不把图谱和解释反写为 profile 真值。

### Stage 5：双人关系解释层

可吸收：

- 双人上下文适合承接 Match 关系解释。
- 关系短句、维度解释、建议 / 避免、轻追问是可产品化结构。
- 示例 / 预览和真实结果必须清晰区分。

不可照搬：

- 不把上传截图作为主路径。
- 不把 `选择完成` 这类状态改变操作变成无提示默认动作。
- 不把入口可见写成完整链路已验证。

## 4. Non-Goals

5.6 规划明确不包含：

- Flutter 代码实现。
- Laravel 代码实现。
- 数据库 schema、迁移、初始化、回填或导入。
- API contract 修改。
- 版本号修改。
- 发布、打包、发版。
- 真人玄学咨询市场。
- 付费报告商店。
- 上传聊天截图作为主路径。
- 强出生资料收集。
- 测试结果反写画像。
- 大规模社区讨论区。
- 复制测测术语体系、页面结构或强玄学诊断口吻。
- 从竞品 UI 推断 EliteSync 后端算法、接口或数据库。

## 5. Protected Surfaces

后续即使进入实现，也必须保护以下面：

- 登录 / session / auth guard。
- 用户资料真值链。
- 出生地、坐标、八字、紫微、星盘等画像字段。
- Match 推荐与匹配解释现有链路。
- Chat 消息发送、草稿、会话状态、RTC / 通话入口。
- Me 个人资料编辑、资料完整度、资料同步提示。
- 通知中心和回流链路。
- 支付、订阅、订单、会员相关入口。
- 数据库迁移、初始化、备份、恢复。
- 版本中心、更新检查、发布链。

本规划阶段不得触碰上述 protected surfaces 的实现。

## 6. Canonical Truth 禁止反写清单

所有玄学 / 关系解释输出不得反写以下真值面：

- `profile/basic`
- `profile/astro/summary`
- `profile/astro/chart`
- `user_astro_profiles`

同时不得反写：

- 用户真实昵称、性别、生日、出生地、坐标。
- 用户身份、关系状态、偏好、标签真值。
- 匹配算法输入真值。
- 聊天消息正文。
- 服务端 canonical profile。

前端缓存只能作为展示兜底，不能抢占服务端真源。

## 7. Derived-Only / Display-Only Contract

5.6 规划中的所有玄学、关系解释、AI 追问和建议文案必须遵守以下 contract：

- `derived-only`：只从已有授权上下文派生，不成为资料真值。
- `display-only`：只用于展示、解释和引导，不进入核心数据写入链。
- `reversible`：用户可以隐藏、关闭或忽略。
- `non-authoritative`：不表达诊断、评判、预测或现实指导承诺。
- `non-persistent by default`：默认不持久化为用户画像，只在用户明确授权后才讨论是否保存偏好。
- `source-visible`：需要让用户理解这是解释层，而不是事实资料。
- `no silent writeback`：任何解释、AI 输出、测试结果都不得静默写回 profile。

推荐免责声明方向：

“以下内容用于帮助你理解互动节奏和表达偏好，不代表事实判断，也不会改写你的资料。”

## 8. Phase A：产品边界与 Contract

目标：先把边界写清楚，再决定是否实现。

交付物：

- 玄学 / 关系解释能力定义。
- Derived-only / display-only contract。
- Non-goals。
- Protected surfaces。
- 用户授权与关闭策略。
- 免责声明文案方向。
- 不反写 canonical truth 的验收清单。

关键问题：

- 哪些上下文允许用于生成解释。
- 是否允许使用聊天摘要。
- 是否允许使用匹配理由。
- 是否允许使用用户主动填写的兴趣 / 标签。
- 解释是否需要短期缓存。

本阶段不产出运行时代码。

## 9. Phase B：Me 个人解释层

目标：在 Me 或个人画像页规划轻量自我理解解释层。

可能形态：

- 个人表达风格卡。
- 自我介绍优化建议。
- 互动偏好解释。
- 资料完善建议。
- AI 展示建议的解释版。

设计原则：

- 只使用已有服务端真源和用户授权上下文。
- 不新增硬玄学输入链。
- 不生成事实画像。
- 不覆盖资料编辑页的真值。
- 允许用户关闭或忽略。

暂不实现内容：

- 星盘图谱。
- 出生资料采集。
- 付费报告。
- 深度人格诊断。

## 10. Phase C：Match 关系解释层

目标：为匹配详情或关系推荐页规划“关系解释层”。

可能结构：

- 一句关系理解摘要。
- 2-4 个关系维度。
- “适合如何开场”建议。
- “避免什么表达方式”提醒。
- 轻追问入口，例如“为什么推荐这个人”“怎么开启对话”“对方可能在意什么”。

设计原则：

- 基于匹配上下文、公开资料、双方已授权展示信息。
- 不使用未授权隐私。
- 不写入匹配算法真值。
- 不把解释当成结论判定。
- 不生成高压诊断、预测或控制性话术。

暂不实现内容：

- 上传截图。
- 真人咨询。
- 付费关系报告。
- 大规模社区评论。
- 测试题结果回写。

## 11. Phase D：Chat 轻追问入口

目标：在聊天页规划低压、可忽略的轻追问或解释建议。

可能形态：

- 冷场时的“换个问法”。
- 首聊时的“从共同点开场”。
- 对关系解释的“继续了解”。
- AI 草稿助手的关系语境提示。

设计原则：

- 不自动发送消息。
- 不修改原聊天内容。
- 不生成强操控话术。
- 不读取未授权私密内容。
- 用户始终可关闭、忽略或改写。

暂不实现内容：

- 自动代聊。
- 自动分析敏感聊天截图。
- 上传截图作为主流程。
- 对对方心理做确定性判断。

## 12. Phase E：AI 助手衔接

目标：把解释层与 AI 助手连接为可选追问，而不是强制转化。

可能形态：

- “为什么会这样推荐？”
- “我可以怎么开场？”
- “这个匹配点是什么意思？”
- “帮我把这句话说得更自然。”

设计原则：

- AI 输出只作为建议。
- 不写回 profile 真值。
- 不进入真人服务市场。
- 不默认持久化。
- 保留用户编辑权。

暂不实现内容：

- 真人服务导流。
- 付费咨询。
- 自动生成长期画像。
- 将 AI 输出作为算法输入。

## 13. 最小切片推荐

推荐最小切片：Match 关系解释层。

建议切片边界：

- 入口：Match detail 或匹配解释区域。
- 内容：一句关系摘要 + 2-4 个解释维度 + 建议 / 避免 + 一个轻追问入口。
- 数据：只用已有匹配上下文和已授权展示资料。
- 写入：无。
- 持久化：无，或仅本地临时展示。
- AI：可先不接真实 AI，只做静态结构或本地文案占位。
- 验收：确认不反写 profile、不影响匹配算法、不改变聊天发送链。

不建议第一刀做 Me：

- Me 更接近资料真值，容易误伤 profile。
- 个人解释层容易被误解为画像判断。
- 需要更严格的免责声明和关闭策略。

不建议第一刀做 Chat：

- Chat 更接近用户真实互动和消息发送链。
- AI 草稿与自动发送边界需要额外保护。

## 14. 为什么优先做 Match 关系解释层

Match 是最适合承接玄学能力二次产品化的第一落点，原因：

- 用户在 Match 场景天然需要理解“为什么推荐这个人”。
- 关系解释可以提升匹配理由的可读性，而不必改动资料真值。
- 建议 / 避免可以服务首聊和关系推进。
- 轻追问可以自然衔接 AI 助手，但不强制进入 AI 对话。
- Match 的解释层可以保持 display-only，不需要写库。
- 相比 Me 和 Chat，Match 更容易控制边界和验收范围。

建议第一版只做“解释结构规划”，不要直接实现 UI。

## 15. 实现前必须确认的问题

进入任何实现前，需要用户明确确认：

1. 5.6 是否正式作为下一阶段规划版本。
2. 第一切片是否选择 Match 关系解释层。
3. 是否允许新增纯文档 spec / PRD。
4. 是否允许后续进入 Flutter UI 方案设计。
5. 是否允许接入 AI，还是第一版只做静态解释结构。
6. 解释层是否允许短期缓存。
7. 是否需要用户设置中的关闭开关。
8. 是否需要单独的免责声明文案。
9. 是否纳入 5.x 主线，还是作为独立实验规划。
10. 验收时是否需要 Claude / Gemini 做 UI 走查或视觉复核。

在这些问题确认前，不进入代码实现。
