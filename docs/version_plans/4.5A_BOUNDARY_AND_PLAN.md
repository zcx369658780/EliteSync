# EliteSync 4.5A 边界与执行计划

更新时间：2026-04-21

## 需求理解

4.5 的目标是建立站内通知与社交转化的最小闭环：通知可产生、可阅读、可回流到正确页面，并能与消息、动态、匹配、首页轻量联动。

本轮只做边界冻结与最小骨架，不做推送平台、RTC、在线状态、营销消息或大规模推荐重构。

## 受影响模块

- UI / 页面：
  - 首页入口与轻量联动
  - 消息页通知入口
  - 通知中心页（新增）
  - 动态作者页 / 详情页回流入口
  - 匹配解释 / 首聊建议回流入口
- 路由：
  - `AppRouteNames.notificationCenter`（待新增）
  - 现有 chat / status / match 路由复用
- 状态管理：
  - 通知列表、未读数、已读状态
  - 首页 / 消息页的轻量提醒状态
- 数据层 / API：
  - `notifications` 站内真值表
  - `NotificationController` 读 / 已读接口
  - 事件来源：messages / status / match / system
- 数据库 / 迁移：
  - 只在现有 `notifications` 骨架上做最小补齐，优先不重做 schema
- 配置 / 环境变量：
  - 推送预留开关、通知偏好预留
- 第三方 SDK：
  - 本轮不接厂商推送 SDK
- 备份 / 恢复 / 发布脚本：
  - 本轮不改发布脚本；云端写库仍作为唯一正式后端路径

## 计划前置评审

### dependency-mapper 结论

- 现有后端已有 `notifications` 表、`NotificationController`、`/api/v1/notifications` 和 `markRead` 路由。
- 前端已有 settings 页的推送开关缓存，但没有独立通知中心页。
- 首页、消息、匹配、动态已存在跳转与轻量联动入口。

### risk-reviewer 结论

- 高风险面：通知写入可能误伤消息 / 动态 / 匹配主链；已读 / 未读可能漂移；本地缓存可能覆盖真实通知状态。
- 回滚点：保持通知功能 additive，随时可关闭写入入口，只保留读取页。

### test-planner 结论

- 最小验证集：通知列表读取、未读数、已读状态、点击回流、空态 / 错误态、旧主链回归。
- 必须覆盖：消息、动态、匹配、版本检查、真值链回归。

### architecture-guardian 结论

- 允许在通知层加新页和新接口。
- 不允许重写消息 / 动态 / 匹配主链。
- 不允许把通知中心做成运营后台。

## 高风险闸门

- 数据库 schema / 初始化 / 回填：只允许最小补齐通知表字段或辅助表，先确认回滚点。
- 路由 / 页面生命周期 / 状态管理：新增通知页时必须保留旧导航行为。
- 配置 / SDK：不引入重型推送 SDK，不修改现有版本检查链。

## 本轮范围

### 做什么

1. 冻结通知边界与事件类型
2. 明确通知读取与回流目标
3. 盘点现有 notification skeleton 是否足够
4. 定义后续实现顺序

### 不做什么

1. 不做厂商推送全适配
2. 不做在线状态 / RTC / 通话
3. 不做营销通知平台
4. 不重构消息 / 动态 / 匹配主链
5. 不改 4.0 / 4.2 / 4.3 / 4.4 / 4.4S 已归档能力

## 回滚与恢复

- 回滚点：若通知写入影响旧主链，可先关闭写入入口，仅保留读取页。
- 恢复方式：保留现有 notifications 读接口，不删除已有主链代码。
- 数据恢复方式：本轮优先不做大规模数据回填，只做最小种子或最小事件写入。
- 版本回退方式：通知功能与主链分离，便于单独回退。

## 验收标准

- 功能验收：能看到通知列表，能点回正确页面。
- 回归验收：消息 / 动态 / 匹配 / 真值链不受影响。
- 旧功能保护验收：版本检查、问卷、图片 / 视频消息、动态流稳定。
- 数据一致性验收：未读数与列表状态一致。
- 日志 / 报错验收：通知读取与点击失败可记录。

## 明确不做

- 推送平台化
- RTC / 通话
- 在线状态
- 推荐算法大改
- 全站 UI 重做
