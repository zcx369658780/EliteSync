## 2. 总体规划原则

### 2.1 真值原则

1. 资料、出生地、坐标、八字、紫微、星盘都以服务端 canonical 真值为主。
2. Flutter 只负责本地展示、阅读性和交互偏好，不抢真值。
3. 本地绘制、盘面设置、展示偏好不得回写真值层。
4. 所有“可观测性”信息必须与“业务真值”明确分离。

### 2.2 风险原则

1. 高风险模块必须先走计划前置、风险拆解、回归门禁。
2. 不允许把低风险 UI 任务膨胀成高风险算法重写。
3. 算法升级必须先有能力清单，再有代码改动。
4. 历史版本只做归档，不回流为当前执行入口。

### 2.3 版本推进原则

1. 稳定基线优先于新功能扩张。
2. 若顾问确认某项能力需要多个版本，必须拆版本，不得一次性塞满。
3. 新版本规划必须写清：目标、边界、保护面、验收、回滚。
4. UI 重构类任务必须引入 `Gemini-MCP` 作为 Codex 的 UI / 验收 subagent。

## 5. 星盘引擎升级方向（新增主线）

## 5.1 顾问结论与总判断

顾问已确认：

1. 当前截图级网页版能力与现有 Kerykeion 路线不完全一致。
2. 星盘引擎升级不能塞进一个普通版本的小修任务里。
3. 这个方向必须按多版本拆解推进，而不是试图在一个版本内一次性做完。
4. 当前项目已有足够好的资料链路、本地绘制和显示层基础，可以承接升级第一阶段。

因此：

> 星盘引擎升级应作为独立的中期产品/技术主线推进，采用多版本路线，而不是混入当前稳定基线的小修任务中。

## 5.2 为什么必须独立立项

原因如下：

1. 目标需求已经不只是“画一个盘”，而是要接近占星工作台。
2. 目标能力包含：古典盘 / 现代盘、合盘、男女对比盘、行运、三限、次限、月返、日返、法达、小限盘。
3. 还要求支持：回归黄道 / 恒星黄道、十几种宫位制、相位容许度、虚点 / 小行星 / 敏感点设置。
4. 后续还需要参数解读、宫主星、主星落宫、宫主飞宫、大限、流年等专业语义层。
5. 这已经不是“继续修星盘页”，而是独立 Astro Engine 项目。

## 5.3 当前技术路线判断

当前项目主线：
- 服务端用现有引擎计算真值
- Flutter 根据 `chart_data` 本地绘制
- 显示层与真值层明确隔离

目标升级方向：
- 需要新的参数化引擎抽象层
- 需要更完整的天文计算 / 占星语义 / 前端绘制拆层
- 需要更强的回归对照与样例验证体系
- 不应直接用当前产品版本节奏把所有高级能力一次性塞进主线

## 5.4 升级总体路线

### 阶段 A：需求与能力边界确认

目标：
- 明确“要复刻到什么程度”
- 明确哪些能力属于 MVP，哪些属于后续专业增强
- 明确同事历史原型与当前项目之间的差异

输出：
- 能力清单
- 样例清单
- 对照校验口径
- UI 工作台草图 / 信息架构

### 阶段 B：引擎与数据结构分层

目标：
- 把底层天文计算、占星语义、前端绘制三层分开
- 定义统一的 chart schema / aspect schema / house schema / options schema
- 让未来能同时承接不同底层库，而不直接绑死当前实现

输出：
- Engine Adapter 抽象层
- 标准化 chart data
- 参数 schema 与选项 schema
- 回归样例接口

### 阶段 C：前端绘制与交互工作台

目标：
- 做出接近历史网页原型的参数面板与精细盘面
- 支持面板切换、预设、显示过滤与参数弹窗
- 完成标签避让、相位过滤、宫线分层、盘心说明等阅读性升级

输出：
- Astro Workbench MVP
- 参数面板
- 盘面精细绘制
- 解释弹窗 / 参数说明

### 阶段 D：回归与兼容性验证

目标：
- 使用固定出生样例对比结果
- 验证不同宫位制 / 黄道制 / 主要参数切换
- 验证旧产品入口不被破坏

输出：
- 回归样例库
- 差异报告
- 兼容性报告
- 风险报告

### 阶段 E：高级能力分期推进

目标：
- 只在前面稳定后，再推进古典 / 现代、推运、合盘等高级能力
- 每一批能力都单独拆版本，不一次性全上

输出：
- 3.6 / 3.7 等后续分期计划

## 5.5 星盘引擎升级的版本拆解建议

### 3.5：Astro Engine MVP / 参数化基础版

定位：
- 星盘引擎升级第一阶段
- 建立引擎抽象层与参数化基础能力
- 做出接近网页原型的参数面板与盘面精细绘制 MVP

适合纳入：
- 引擎抽象层
- chart/options schema
- 基础参数面板
- 回归黄道 / 恒星黄道切换
- 核心宫位制（少量最常用）
- 主要点位、主要相位、基础容许度
- 标签避让、相位线过滤、盘面层次
- 参数解读弹窗最小版
- 样例对照验证框架

### 3.6：古典 / 现代双路线版

定位：
- 建立更完整的古典 / 现代盘口径分流
- 扩充更多宫位制、虚点、小行星、敏感点
- 补更细的解释层

适合纳入：
- 古典 / 现代切换
- 更多宫位制 / 参数开关
- 宫主星、主星落宫、宫主飞宫
- 更细的相位 / 容许度策略
- 更完整参数对照

### 3.7：推运与高级解读版

定位：
- 向专业工作台迈进
- 实现高级推运和解释体系

适合纳入：
- 合盘、男女对比盘
- 行运、三限、次限、月返、日返
- 法达、小限、大限、流年
- 高级解释体系
- 多案例精度回归

## 5.6 当前禁止的做法

以下做法当前禁止：

1. 在没有新计划的情况下直接改星盘 canonical 大逻辑
2. 把 3.5 做成“一次性吃下全部星盘升级”的超大版本
3. 在没有回归样例与能力清单前，凭页面感受硬调算法
4. 让盘面设置页承担算法调参职责
5. 在 UI 重构中跳过 Gemini-MCP 审查

## 8. 当前禁止项（更新版）

以下内容不能在没有新计划的情况下直接扩大：

- 星盘 canonical 大改
- 地点搜索与地理编码的架构替换
- 数据库 schema 大范围重构
- 登录 / 会话 / 路由大改
- 版本中心把可观测性写成业务真值
- 在没有 Gemini-MCP 参与的情况下执行大规模 UI 重构
- 把星盘引擎升级所有能力塞进单一版本

---

## 9. 计划书后续维护规则

- 新版本规划必须写清：目标、边界、保护面、验收、回滚
- 算法升级必须先有能力清单，再有代码改动
- 若顾问确认某项能力需要多个版本，必须拆版本，不得一次性塞满
- UI 重构型版本必须在计划书中显式写明 Gemini-MCP 的职责与调用节点
- 总计划书修订后，必须同步更新版本索引和主交接稿

## 10. 当前一句话结论

EliteSync 当前稳定基线已经形成；后续主线应分为两条：

> **一条继续维护正式产品与 Beta 基线，另一条以多版本方式推进星盘引擎升级。**
