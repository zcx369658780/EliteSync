## 2. 需求理解

- 用户想要什么：在不破坏 3.6 双路线基线、canonical truth 单源和既有入口的前提下，引入关系维度与时间维度的高级占星能力首版。
- 预期结果：形成一套可用、可解释、可回归、可归档的高级能力最小闭环，至少覆盖合盘 / 对比盘、行运和一项最小返照能力。
- 现有问题：3.6 只解决了路线分流、解释层和样例回归，还没有进入关系 / 时间维度；高级能力一旦上线最容易出现 truth boundary 混淆和页面过载。
- 本轮不改什么：不做全量专业占星平台、不做大规模 canonical 引擎替换、不改登录 / 会话 / 地理编码 / 数据库大结构、不破坏 3.6 稳定基线。

## 3. 受影响模块

- UI / 页面：本命盘详情页、合盘 / 对比盘页、行运页、返照页、设置中心、版本中心、截图证据索引。
- 路由：新增高级能力入口路由，保留旧入口可达。
- 状态管理：高级上下文 provider、时间上下文 provider、双人上下文 provider。
- 数据层 / API：`/api/v1/profile/astro`、`/api/v1/profile/astro/render`、`/api/v1/profile/astro/summary`、`/api/v1/profile/astro/chart` 的高级上下文透传；新增派生层接口时保持向后兼容。
- 数据库 / 迁移：本阶段原则上不做 schema 大改，如需新增仅限增量字段且必须有回退策略。
- 地图 / 定位 / 权限：默认不触碰。
- 配置 / 环境变量：仅在高级能力需要 feature flag 或显示开关时评估。
- 第三方 SDK：Claude-mcp、Gemini-MCP 用于审查与验收，不直接作为业务依赖。
- 备份 / 恢复 / 发布脚本：本阶段不主动改动，若后续触碰必须单独补计划。

## 7. 回滚与恢复

- 回滚点：3.6 稳定基线、当前 `chart_data` / `engine_info` / `metadata` 兼容读取路径。
- 恢复方式：保留旧入口、旧读取逻辑与 feature flag，必要时关闭高级入口。
- 数据恢复方式：本阶段默认不改数据库大结构；若后续新增增量字段，须提供回退策略。
- 版本回退方式：保持 3.6 发布物和截图证据可回退。

## 8. 验收标准

- 功能验收：合盘 / 对比盘、行运、最小返照至少有一套可用入口。
- 回归验收：高级样例集、差异矩阵、已知偏差文档形成闭环。
- 旧功能保护验收：3.6 旧入口、旧设置、旧版本中心可达。
- 数据一致性验收：canonical truth 未被误写。
- 日志 / 报错验收：Claude / Gemini 审查留痕可归档。
