# EliteSync 3.7 高级 Schema 与边界说明

更新时间：2026-04-12  
适用阶段：3.7 stage 1（Claude 架构审查落地版）

## 1. 结论

3.7 的高级能力必须以“新增上下文、保持单源真值、保持旧字段兼容”为原则推进。

当前 3.6 已经证明以下模式是安全的：

- `route_mode` 作为 additive display context 可随请求透传
- `engine_info` / `metadata` 可以增量扩展
- 旧客户端可忽略新增字段
- `chart_data` 主结构不能被重写

因此，3.7 的合盘 / 对比盘 / 行运 / 返照不应直接扩写进 `chart_data`，而应通过独立的 advanced context 与派生摘要层表达。

## 2. 当前可复用的现状

### 2.1 Flutter 侧

- 现有 `astroSummaryProvider` / `astroNatalChartProvider` 通过 `route_mode` 查询参数请求 `/api/v1/profile/astro/summary` 与 `/api/v1/profile/astro/chart`。
- 当前 route 相关状态已经独立成 `AstroChartRoutePrefs`、`AstroChartWorkbenchPrefs`、`AstroChartDisplayPrefs`。
- 路线解释卡、路线差异报告卡、参数工作台卡都已经是 display-only / derived-only。

### 2.2 FastAPI 侧

- `/api/v1/profile/astro`
- `/api/v1/profile/astro/summary`
- `/api/v1/profile/astro/chart`
- `/api/v1/profile/astro/render`

均已支持 `route_mode` 透传，并在返回体里携带：

- `chart_data`
- `planets_data`
- `houses_data`
- `aspects_data`
- `route_mode`
- `engine_info`
- `metadata`

### 2.3 Laravel 侧

- `AstroProfileController` 已把 `route_mode` 作为 route context 加回响应。
- 旧缓存读取与 `private_natal_chart` 兜底逻辑仍在。
- 当前 3.6 只是在 display / derived 层做路线扩展，没有破坏 canonical truth。

## 3. 3.7 推荐的上下文拆分

### 3.1 canonical context

仍然只包含本命盘真值来源：

- birthday
- birth_time
- birth_place
- birth_lat
- birth_lng
- tz_str

### 3.2 route context

继续保留 `route_mode`，用于指示当前展示 / 解释路由：

- `standard`
- `classical`
- `modern`

### 3.3 advanced context

3.7 新增的关系 / 时间维度上下文应单独归类为 advanced context：

- `pair_context`
- `time_context`
- `return_context`
- `comparison_context`

这些字段只能用于派生解释和页面展示，不能回写 canonical truth。

## 4. 推荐 schema 结构

### 4.1 metadata 扩展建议

建议在现有 `metadata` 下新增：

```json
{
  "metadata": {
    "schema_version": "astro_engine_mvp_v1",
    "engine_name": "kerykeion",
    "engine_version": "5.x",
    "chart_kind": "natal",
    "route_context": {
      "route_mode": "standard",
      "route_preset": "standard",
      "source": "display_only"
    },
    "advanced_context": {
      "mode": "pair|transit|return",
      "source": "derived_only",
      "scope": "preview|internal|sample"
    },
    "field_roles": {
      "canonical": ["birthday", "birth_time", "birth_place", "birth_lat", "birth_lng", "tz_str"],
      "derived": ["chart_data", "planets_data", "houses_data", "aspects_data", "generated_at"],
      "display_only": ["route_mode"],
      "advanced_context": ["pair_context", "time_context", "return_context", "comparison_context"]
    }
  }
}
```

### 4.2 engine_info 扩展建议

`engine_info` 继续保留“引擎来源 / schema / 图表种类”的角色，不建议塞入过多业务语义。

可增量字段示例：

- `engine_name`
- `engine_version`
- `schema_version`
- `chart_kind`
- `route_mode`
- `advanced_mode`

建议 `advanced_mode` 只描述高级能力类型，不描述业务结论。

## 5. 3.7 API 扩展策略

### 5.1 现有接口继续保留

- `/api/v1/profile/astro`
- `/api/v1/profile/astro/summary`
- `/api/v1/profile/astro/chart`
- `/api/v1/profile/astro/render`

### 5.2 新增接口建议

如果 3.7 需要单独的高级入口，建议采用 additive 命名：

- `/api/v1/profile/astro/pair`
- `/api/v1/profile/astro/transit`
- `/api/v1/profile/astro/return`

这些接口应返回与现有 natal 接口一致的基础结构，并额外带上：

- `advanced_mode`
- `pair_context` / `time_context` / `return_context`
- `advanced_summary`
- `metadata.advanced_context`

### 5.3 向后兼容要求

1. 旧客户端不认识高级字段时必须可正常忽略。
2. 旧接口的 `chart_data` 结构不能改。
3. 旧缓存若只有 natal / route 字段，也必须能继续读取。
4. 高级入口若关闭 feature flag，必须能回退到 3.6 行为。

## 6. 3.7 数据边界

### 6.1 对方资料边界

对方资料在合盘 / 对比盘里应视作输入上下文，不视作 canonical truth。

### 6.2 时间边界

行运 / 返照应把 target datetime 作为 advanced context 输入，而不是修改 natal truth。

### 6.3 结果边界

高级结果只能进入：

- relation summary
- transit summary
- return summary
- display-only helper fields

不得进入：

- `profile/basic`
- natal canonical truth
- 任何会被旧客户端当成真值读取的固定字段

## 7. 风险点

1. `pair_context` 一旦和 `chart_data` 混写，后续很难区分真值与派生层。
2. 时间维度如果复用 natal 保存结构，容易污染现有 `computed_at` / `true_solar_time` 语义。
3. 如果高级摘要直接写进旧 summary 页面，很容易把页面做成第二个后台。
4. 如果新增接口没有 fallback，旧缓存与旧客户端会失去兼容性。

## 8. 结论

3.7 的高级 schema 设计应遵守以下优先级：

1. canonical truth 单源
2. route_mode 继续作为 display-only context
3. pair / time / return 统一归入 advanced context
4. 新增高级接口采用 additive 方式
5. 旧字段、旧缓存、旧入口保持兼容

这份边界说明作为 3.7 stage 1 的基线，后续实现必须围绕它展开。

## 9. stage 2 进展

- `pair` / `transit` / `return` 已从 schema 约束进入后端 scaffold。
- FastAPI 与 Laravel 均已落地对应入口，输出仍保持 derived-only / display-only / advanced-context。
- 后续 UI / 路线工作台 / 高级摘要卡必须继续遵守本文件里的上下文边界。
