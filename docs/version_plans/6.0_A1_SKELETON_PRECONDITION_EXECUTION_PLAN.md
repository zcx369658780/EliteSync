## 2. v1 P0 Smoke Execution Plan

执行环境：

- 只能使用 staging / controlled backend，不打 production 真实用户链路。
- 如必须使用生产等价数据，必须先由用户明确确认账号、范围、可恢复方式和停止条件。
- 执行前必须确认 `git status --short` 干净，避免 smoke 结果与本地未提交 runtime diff 混淆。
- 执行后只输出响应摘要，不输出 token、手机号、精确位置、私密聊天内容、出生完整时间等隐私字段。

测试账号要求：

- 至少准备两个 smoke 账号，用于 messages、RTC、notifications、match 准入等双用户链路。

token 处理要求：

- token 只保存在当前 shell session 或临时安全变量中。
- token 不写入报告、不写入 docs、不写入日志附件、不进入截图。
- 失败日志必须脱敏。

## 4. Env / Cache / Session Config Unification Plan

当前问题：

- Gate 远端观察：`artisan about` 与 `.env` / config 口径不一致。
- `services/backend-laravel/config/cache.php` 使用 `APP_CACHE_STORE`，默认 `file`。
- `services/backend-laravel/config/session.php` 使用 `APP_SESSION_DRIVER`，默认 `file`，并用 `APP_SESSION_CONNECTION` fallback 到 `DB_CONNECTION`。
- `services/backend-laravel/.env.example` 使用 `CACHE_STORE=redis` 与 `SESSION_DRIVER=database`。
- `services/backend-laravel/config/queue.php` 读取 `QUEUE_CONNECTION`，再 fallback `APP_QUEUE_CONNECTION`。
- `config/database.php` 里 DB selector 已偏向 `APP_DB_CONNECTION`，Redis 有 `REDIS_DB` / `REDIS_CACHE_DB`，但 queue/cache/session 的 app-scoped key 仍未统一。

目标口径：

- Redis 用于 cache / queue / rate limit。
- session 可选 Redis 或 database：
  - Redis session：更贴近 cache/session/queue 统一目标，但必须配置独立 DB / prefix / 持久化与淘汰策略，避免 cache eviction 影响登录态。
  - database session：短期降低登录态迁移风险，便于审计与恢复，但会增加 DB 压力；若选择 database，必须写明这是过渡策略。
- local / staging / production 必须分环境说明，不共享同一 Redis prefix、DB、base URL 或 feature flag。
- 避免 host-level env 污染，优先使用 app-scoped key 或统一一套最终 key，不允许 `APP_CACHE_STORE` 与 `CACHE_STORE` 长期并存无解释。

需要统一的 env key 清单：

| Key | 当前问题 / 用途 | skeleton 前要求 |
| --- | --- | --- |
| `APP_ENV` | 区分 local/staging/production | 必须明确，production 不能伪装 staging |
| `APP_DEBUG` | debug 泄露风险 | staging/production 必须 false |
| `APP_CACHE_STORE` 或最终统一 key | config/cache.php 当前读取 | 与 `.env.example` 统一 |
| `APP_SESSION_DRIVER` 或最终统一 key | config/session.php 当前读取 | 与 `.env.example` 统一 |
| `SESSION_STORE` | Redis/session cache store | 若 session=redis 必须确认 |
| `SESSION_CONNECTION` / `APP_SESSION_CONNECTION` | 当前 config 读取 app-scoped connection | 需确定最终 key，避免混淆 |
| `QUEUE_CONNECTION` | config/queue.php 优先读取 | staging/production 目标 redis |
| `REDIS_QUEUE_CONNECTION` | Redis queue connection | 需明确 default/cache 分离 |
| `REDIS_CACHE_CONNECTION` | cache store connection | 需明确 cache connection |
| `REDIS_DB` | default Redis DB | 与 queue/session 使用关系要说明 |
| `REDIS_CACHE_DB` | cache Redis DB | 与 session/queue 不混用或有理由 |
| `CACHE_PREFIX` | cache key prefix | 分环境必须不同 |
| `REDIS_PREFIX` | Redis key prefix | 分环境必须不同 |

后续用户确认后才允许执行的 config cache 流程草案：

1. 备份 `.env`，脱敏记录 key matrix。
2. 备份当前 cached config 状态。
3. 在 staging 修改目标 env key，不改 production。
4. 执行 `php artisan optimize:clear`。
5. 执行 `php artisan config:cache`。
6. 执行 `php artisan about` 与 `php artisan config:show cache.default`、`session.driver`、`queue.default`。
7. 执行 login/profile/messages/notifications smoke。
8. 记录回滚命令与回滚结果。

本阶段禁止执行：

- `php artisan optimize:clear`
- `php artisan config:cache`
- `php artisan cache:clear`
- 修改 `.env`
- 修改 production config

硬结论：

- cache/session 口径不统一前，不得进入 skeleton 商用化。
- 如果用户仅允许完全只读 skeleton，也必须把该问题标为 observation，并由用户明确确认。

## 6. Backup / Rollback / Monitoring / Load-test Gate

backup 最小项：

- DB full dump：全量、含 routines/triggers/events，使用只读或维护窗口策略。
- schema-only dump：用于 migration diff 与 rollback review。
- `.env` backup：必须脱敏归档，不进入 git。
- Nginx config backup：`nginx -T` 摘要与配置文件备份。
- PHP-FPM config backup：pool、php.ini/opcache、slowlog 等摘要。
- Redis config snapshot：bind、protected-mode、requirepass、maxmemory、policy、save/aof、db/prefix。
- Supervisor / systemd config backup：worker/scheduler/service unit 或 supervisor conf。
- release chain readonly snapshot：`build.gradle.kts`、`app_update.php`、release script、当前 version check 响应摘要。

rollback checklist：

1. Git rollback point / branch point。
2. DB restore 命令与 staging restore 演练记录。
3. schema-only 对比方式。
4. `.env` restore。
5. Nginx/PHP-FPM config restore 与 reload/restart 确认点。
6. Redis config restore 与 keyspace 影响说明。
7. queue worker stop/restart rollback。
8. v2 feature flag off。
9. Flutter backend base URL 回退到 v1。
10. v1 P0 smoke 重新通过。

health / readiness 指标：

- health：应用进程存活、版本标识、时间、环境。
- readiness：DB ping、Redis ping、queue connection、storage writable、config loaded、release chain readonly、maintenance mode。
- readiness 不应把 skipped queue worker 写成 pass；必须区分 `ok`、`degraded`、`skipped`、`fail`。

监控范围：

- Nginx：4xx/5xx、upstream timeout、request latency、access/error log。
- Laravel：exception rate、deprecation、job failure、scheduler/worker 状态、storage permission。
- DB：connections、slow query、lock wait、replication/backup status、disk。
- Redis：memory、eviction、blocked clients、keyspace、persistence。
- queue：queue length、oldest job age、failed jobs、retry count。
- disk：storage、logs、backup target space。

staging load test 目标：

- 只打 staging，不打 production 真实用户链路。
- 目标是建立基线，不是追求高并发数字。
- 指标至少记录 P50/P95/P99、error rate、PHP-FPM busy workers、DB/Redis 指标、queue latency、Nginx upstream timeout。
- 工具候选：k6 / wrk；queue load test 需要单独 job backlog 场景。

失败停止条件：

- 任一 backup 文件无法生成或无法校验。
- staging restore 演练失败。
- rollback checklist 缺关键步骤。
- health/readiness 指标无法区分 fail 与 skipped。
- load test 准备误指向 production。
- 任意 DB / migration / production 配置变更前缺少 backup + rollback plan + 用户确认。

## 7. Carry-over Items

### 7.1 queue worker 未常驻

- 纯只读 skeleton 可 carry-over。
- 任何 async / media / notification / location / match / buddy 写链路前必须解决。
- 如果 readiness 包含 queue，应返回 `degraded`、`skipped` 或 `not_configured`，不得伪造 `ok`。
- carry-over 证据：queue worker 管理方案、queue 分层、failed_jobs/retry/backoff/timeout 策略。

### 7.2 Composer 2.7.1 deprecation

- 不阻断只读文档。
- 阻断 Laravel 12 staging upgrade 前置。
- 处理前置：记录 composer version，执行只读 validate/check/outdated，用户确认后才允许 `composer self-update`。
- 未处理时不得执行 dependency resolution，也不得把结果写成 Laravel 12 dry-run 通过。

### 7.3 Nginx / PHP-FPM 商用压测基线不足

- 不阻断只读 contract。
- 阻断 staging 压测和 production v2。
- 压测前必须记录 `nginx -T` 摘要、FPM pool、opcache、client_max_body_size、timeout、logs、slowlog、pm.max_children、pm.max_requests。
- production 参数调整必须用户确认。

## 8. Skeleton 进入判定

是否建议进入 v2 skeleton：`conditional`。

条件说明：

- 当前不建议进入商用化 v2 skeleton。
- 在 SK-G01、SK-G02、SK-G03、SK-G04、SK-G05 完成并经用户确认前，不允许进入会影响 DB、API 写链路、production 流量或 Flutter 默认 base URL 的 skeleton。
- 若用户明确确认，可以只进入极窄的只读 skeleton 类型：
  - 仅允许只读 v2 health/readiness/location contract skeleton。
  - 禁止任何写接口。
  - 禁止 DB/migration。
  - 禁止 production 流量。
  - 禁止 Flutter 默认切 v2。
  - 禁止删除或替换 v1 route。
  - readiness 对 queue/cache/session 等未 resolved 项必须如实标为 `degraded/skipped/not_configured`，不得伪造 pass。

必须先完成的前置项：

1. v1 P0 smoke execution，至少覆盖 health/version/profile/astro/messages/media/questionnaire/RTC/notifications/release readonly。
2. Laravel 12 staging upgrade dry-run plan，并明确 Laravel 11 只作为现状。
3. env / cache / session config unification plan，经用户确认最终 key 与 session 目标。
4. staging / production split checklist，经用户确认 production 默认继续 v1。
5. backup / rollback / monitoring / load-test gate package，至少完成文档化门禁。

可 carry-over 项：

- queue worker 未常驻：仅限完全只读 skeleton carry-over；任何 async/write endpoint 前必须解决。
- Composer 2.7.1 deprecation：只读文档可 carry-over；Laravel 12 dry-run 前必须解决或获确认。
- Nginx/PHP-FPM 商用压测基线不足：只读 contract 可 carry-over；staging 压测和 production v2 前必须解决。

需要用户确认的事项：

- 是否允许执行 v1 P0 smoke，尤其是 write smoke。
- smoke 账号、token 管理、staging / controlled backend 地址。
- Laravel 12 是否固定为 A1 第一升级目标。
- 是否允许 staging composer self-update。
- 是否允许 staging composer update / dependency resolution。
- session 目标选 Redis 还是 database。
- queue worker 管理选 systemd 还是 Supervisor。
- 是否允许下一阶段进入只读 v2 health/readiness/location contract skeleton。

## 9. 下一步建议

按优先级建议：

1. v1 smoke execution package：先生成 smoke 命令包、账号要求、脱敏报告模板和失败停止条件；用户确认后再执行。
2. env / cache / session unification patch plan：先出 patch plan 和 env key matrix；不要直接改 `.env` 或 config cache。
3. Laravel 12 staging dry-run plan：确定 staging branch/clone、Composer 处理策略、测试集和 rollback。
4. backup / rollback / monitoring gate package：补 DB/config/release readonly snapshot、rollback checklist、health/readiness 指标和 load test 目标。
5. 是否允许只读 v2 health/readiness skeleton：仅在用户明确确认后进入，且范围必须限制为只读、无 DB/migration、无 production、无 Flutter 默认切换。
