# EliteSync 4.6A 版本开发计划书

文件名：`elite_sync_4_6A_版本开发计划书_2026_04_23.md`  
更新时间：2026-04-23  
适用范围：`4.6` 当前处于 `conditional pass` 后的最小修正版  
当前承接基线：`4.0` 媒体基础设施、`4.1` 非官方四维人格问卷、`4.2` 图片消息主链、`4.3` 动态流基础版、`4.4` 视频消息版、`4.4S` 媒体链稳定性修正版、`4.5` 通知与社交转化增强版均视为稳定基线；`4.6` 的 RTC 状态机壳层、来电页、权限页与通知回流壳层已基本成立，但真语音闭环仍未完成。  
本版定位：`4.6A LiveKit 连接层修正版 / 真语音复验版`

---

## 0. 本版一句话结论

`4.6A` 不是新的 RTC 功能扩展版，而是一个 **插入式最小修正版**。  
它的目标不是继续扩来电页、通话页或多人能力，而是把 `4.6` 当前暴露出来的 **LiveKit 连接入口错误、旧入口缓存/旧连接态残留、Room.connect 复验缺失、真语音闭环未确认** 这几件事收口到可正式验收的程度。

本版只做：
- 云端 LiveKit 入口层修复后的稳定化确认
- 客户端干净重启后的 join-info / `Room.connect()` 复验
- 真机 + 模拟器双向最小复验
- 只有在 `Room.connect()` 稳定成功后，才继续观察最小 publish / subscribe / playback 证据
- 更新 4.6 的收尾材料，使其具备继续验收的基础

本版明确不做：
- 多人通话
- 视频增强
- 在线状态系统
- 外部推送平台
- 新 RTC 大库替换
- UI 大改版
- 4.5 通知中心重开

换言之，`4.6A` 是一个 **连接层修正版 + 真语音复验版**，目的是先把“入房”和“是否进入真音频阶段”确认清楚，再决定是否需要更下一层的 publish / subscribe 修复。

---

## 1. 为什么需要 4.6A

## 1.1 当前问题已经从“RTC 页面 / 状态机疑似问题”收缩为“LiveKit 入口层问题”

在 `4.6G` 与 `4.6H` 的诊断中，已经能够确认：

- 4.6 的来电页、结果页、权限页、壳层状态机并不是首要阻断点
- 真机和模拟器都曾在 `Room.connect()` 之前的 LiveKit validate / signaling 入口失败
- 旧的 `http://<host>:7880` 公网直连路径并不稳定
- 云端已经完成最小修复：
  - `LIVEKIT_URL` 改为公网根 host
  - nginx 新增 `/rtc/` 反代到 `127.0.0.1:7880`
  - `/rtc/validate` 已可从服务器本机和开发机外部打通

这说明当前版本的首要任务已经不再是继续改 UI 或盲改客户端 RTC 代码，而是：
**确认云端入口修复后，客户端是否能在干净重启后稳定完成 `Room.connect()`。**

## 1.2 为什么不能直接把 4.6 判为通过

因为目前虽然已经确认：

- 旧入口层错误被定位
- 云端代理已补到 `/rtc/validate` 可达

但仍未拿到一轮足够干净的新设备侧证据来证明：

- `Room.connect()` 已稳定成功
- 设备侧不再出现旧的 timeout / reset
- 通话已进入下一阶段的 `TrackPublished / TrackSubscribed / AudioPlayback`

因此当前 4.6 仍然不能直接归档，而更适合插入一个 **4.6A 最小修正版**。

## 1.3 为什么不直接重开 4.6 大版本

因为 `4.6` 的版本边界已经写得很清楚：

- 只做 1v1
- 语音优先
- 不做多人 / 直播 / 在线状态 / 外部推送平台

当前更合理的做法是：
**冻结 4.6 功能边界，在其后插入一个极小的 4.6A 修正版，只修连接层与真语音复验。**

## 2. 版本定位

## 2.1 版本名称

`4.6A LiveKit 连接层修正版 / 真语音复验版`

## 2.2 版本使命

在不破坏 `4.0` 到 `4.5` 已归档主链、且不扩大 `4.6` 边界的前提下，补齐以下能力：

1. 云端 LiveKit 公网入口、反向代理与 join-info 口径统一  
2. 客户端不再继续使用旧的 `:7880` 公网直连入口  
3. 真机与模拟器在干净重启后能重新完成 `Room.connect()` 复验  
4. 若 `Room.connect()` 成功，则继续观察最小 publish / subscribe / playback 证据  
5. 若 `Room.connect()` 仍失败，则把失败点继续收窄到剩余的云端 / 客户端连接层问题，而不是继续泛泛调 UI  
6. 为 `4.6` 是否能升级到 `pass with observations` 提供明确证据基础  

## 2.3 本版不是做什么

本版不是：

- 4.6 第二开发版
- 多人 RTC 版
- 视频通话增强版
- 推送唤醒版
- 在线状态平台版
- UI polish 大版本

## 3. 本版做什么 / 不做什么

## 3.1 本版必须完成的核心目标

### 目标 1：LiveKit 入口与协议口径冻结

至少支持：
- 服务端 `LIVEKIT_URL` 口径统一
- 客户端 join-info 不再下发旧的 `:7880` 直连入口
- nginx `/rtc/` 反代配置固定化
- `/rtc/validate` 从服务器本机和开发机外部都稳定可达

### 目标 2：干净重启后的 `Room.connect()` 复验

至少支持：
- 真机端清理旧进程 / 旧缓存 / 旧连接态
- 模拟器端清理旧进程 / 旧缓存 / 旧连接态
- 双端重新拉取 join-info
- 双端重新发起 1v1 通话
- 明确确认 `Room.connect()` 是否稳定成功

### 目标 3：真机 + 模拟器双向最小通话复验

至少支持以下两组：
- 真机发起 -> 模拟器接听
- 模拟器发起 -> 真机接听

并要求：
- 不只看 UI
- 同时看设备日志与关键事件日志
- 记录是否仍有 timeout / reset

### 目标 4：仅在 `Room.connect()` 成功后，观察最小 publish / subscribe / playback 证据

至少支持：
- `TrackPublished`
- `TrackSubscribed`
- `remoteAudioTrack` 是否非空
- `AudioPlayback` 或等价播放链路证据

说明：
- 本版不承诺一口气把所有深层音频问题都修完
- 但必须先回答：**设备是否已真正进入音频阶段**

### 目标 5：更新 4.6 的收尾材料

至少支持：
- 更新连接层排障说明
- 更新复验结果说明
- 更新 closeout / handoff / acceptance summary
- 明确写出 4.6 当前到底能否升级为 `pass with observations`

## 3.2 本版明确不做的内容

以下内容禁止偷渡进入 `4.6A`：

### RTC 功能面不做
- 多人通话
- 群语音 / 群视频
- 视频通话增强
- 直播 / 房间 / 语聊房
- 通话特效 / 礼物 / AI 摘要

### 通知面不做
- 4.5 通知中心大改版
- 外部推送平台化
- 偏好中心完整产品化
- 营销消息系统

### 架构面不做
- 重写 4.6 RTC 架构
- 替换新的 RTC 大库
- 微服务重构
- 重新设计数据库 schema

### UI 面不做
- 来电页大改版
- 通话页视觉大翻修
- 通知中心全面 polish
- 首页 / 会话页结构重做

## 4. 硬性保护面

## 4.1 真值链保护面

- `POST /api/v1/profile/basic`
- `GET /api/v1/profile/astro/summary`
- `GET /api/v1/profile/astro/chart`
- `user_astro_profiles`
- `questionnaire_*` 真值链

### 解释

本版修的是 LiveKit 连接入口和真语音复验，不是画像、资料、匹配真值链。  
任何通话行为、连接次数、未接、拒绝、失败都不得回写 canonical truth。

## 4.2 消息与媒体保护面

- `GET /api/v1/messages`
- `POST /api/v1/messages`
- `POST /api/v1/media`
- `chat_messages`
- `media_assets`
- `media_processing_jobs`
- `message_attachments`

### 解释

本版允许在通话会话上下文内继续联动消息页，但不允许为了连接层修复去误伤图片 / 视频消息主链。

## 4.3 通知与回流保护面

- `GET /api/v1/notifications`
- `POST /api/v1/notifications/read/{notificationId}`
- `POST /api/v1/notifications/read-all`
- `notifications`
- `notification_center` 前端主链

### 解释

本版只允许保留与通话结果直接相关的最小联动，不得借机重构 4.5 通知中心。

## 4.4 RTC 边界保护面

- `rtc_sessions`
- `rtc_session_events`
- 现有 4.6 状态机页面壳层
- 现有 4.6 权限 / 失败态壳层

### 解释

本版不能借“修连接层”之名，回头重写 4.6 状态机、来电页、结果页或信令模型。

## 4.5 发布与版本保护面

- `GET /api/v1/app/health`
- `GET /api/v1/app/version/check`
- `app_release_versions`
- 宿主 APK 版本口径
- about / version center / release script 绑定链

### 解释

本版虽然偏连接层和云端修复，但仍不得破坏宿主版本链一致性。

## 4.6 环境与数据库边界（长期硬规则）

本版必须严格遵守：

- 本地环境只做前端开发、UI 联调、截图采集、文档整理
- 后端开发、数据库迁移、备份、恢复、写库排障统一在阿里云端执行
- 不允许本地开发库、SQLite、临时数据库污染云端正式后端数据

### 解释

本版涉及 LiveKit 容器、nginx、环境变量、云端入房配置，因此更必须守住“本地前端 / 云端后端”边界。

## 5. 数据与接口边界建议

## 5.1 数据模型建议

优先复用现有核心表与配置，不建议新增 schema：
- `rtc_sessions`
- `rtc_session_events`
- `notifications`
- `app_events`
- `chat_messages`
- `conversations`
- `conversation_members`

本版重点不在新增表，而在：
- `LIVEKIT_URL` 配置
- nginx `/rtc/` 入口
- join-info 下发口径
- 客户端 `Room.connect()` 入参
- 双端复验日志

### 边界解释

本版不允许为了“修连接层”去新增大量 RTC 数据模型。  
若非必须，不新增表、不重做 schema。

## 5.2 接口域建议

### 优先策略

- 继续复用 `GET /api/v1/rtc/calls/{callId}/livekit`
- 继续复用现有 `POST /api/v1/rtc/calls/*` 通话状态接口
- 仅在必须时补充最小调试/诊断日志，不新增大批实验性接口

### 建议最小检查集合

- `GET /api/v1/rtc/calls/{callId}/livekit`：确认 join-info 口径
- 现有 `POST /api/v1/rtc/calls/{callId}/connect`：确认呼叫主链仍正常
- 现有 `POST /api/v1/rtc/calls/{callId}/accept/reject/end`：确认状态机未受伤
- 客户端 `Room.connect()` 入参打印：确认不是旧的 `:7880`

### 边界解释

本版不要求新增完整调试 API。  
核心原则是：
- 入口口径清楚
- 客户端入参可审计
- 设备侧复验可追踪
- 不新造脱离主链的调试体系

## 6. 工作包拆分

## 6.1 4.6A-A：边界冻结与连接层口径说明

### 目标

先把本次修什么、为什么修、修到哪一层写清楚，防止 4.6A 再次扩版本。

### 必做任务

1. 明确本版只修连接层与真语音复验  
2. 明确 4.6 主链不重做  
3. 明确本版不做 publish / subscribe 深层重构，除非连接层复验已通过且仅需最小补丁  
4. 明确本版不改 UI 大结构、不换 RTC 库  
5. 明确本地前端 / 云端后端边界继续生效  

### 必要验收条件

- 产出一份清晰的边界说明
- 明确 4.6A 与 4.6 主版的关系
- 明确哪些内容不在本版范围内

### 解释

没有边界说明就直接继续调 RTC，很容易把 4.6A 做成一个没有上限的“持续排障版”。

---

## 6.2 4.6A-B：云端 LiveKit 入口层固定化

### 目标

把本轮已找到的连接层修复固化下来，避免下一轮设备侧仍吃到旧入口。

### 必做任务

1. 固定云端 backend `.env` 中的 `LIVEKIT_URL` 口径  
2. 固定 nginx `/rtc/` 反向代理配置  
3. 固定 80 端口公网入口与 127.0.0.1:7880 的内层转发关系  
4. 固定服务器本机与开发机外部的 validate 可达验证命令  
5. 记录必要的 LiveKit 容器 / 监听 / 代理检查步骤  

### 必要验收条件

- `/rtc/validate` 从服务器本机可达
- `/rtc/validate` 从开发机外部可达
- join-info 返回的新入口不是旧的 `:7880`
- 不再依赖旧公网直连路径

### 解释

本工作包的目标不是继续调 RTC 页面，而是先确保“入口层修复已经固定，而不是一次性临时操作”。

---

## 6.3 4.6A-C：客户端干净重启与 `Room.connect()` 复验

### 目标

在云端入口修复后，重新做干净客户端验证，确认设备侧是否真的进入下一阶段。

### 必做任务

1. 真机与模拟器都执行干净重启  
   - 退出旧进程
   - 重新拉起 app
   - 重新登录（如有必要）
   - 重新拉取 join-info

2. 做两组复验  
   - 真机发起 -> 模拟器接听
   - 模拟器发起 -> 真机接听

3. 补充新设备日志  
   - `Room.connect()` 成功 / 失败
   - 是否还出现旧的 timeout / reset
   - 当前实际 url / room / identity / callId

### 必要验收条件

- 拿到一轮新的、干净重启后的设备日志
- 能确认客户端是否已吃到新 join-info
- 能确认 `Room.connect()` 是否已进入成功态

### 解释

现在最关键的不是“猜连接成功没有”，而是要拿到干净的设备侧新证据。

---

## 6.4 4.6A-D：最小音频阶段观察（仅在 connect 成功后）

### 目标

只有在 `Room.connect()` 成功之后，才继续观察 publish / subscribe / playback。

### 必做任务

1. 观察 `TrackPublished`  
2. 观察 `TrackSubscribed`  
3. 观察 `remoteAudioTrack` 是否非空  
4. 观察 `AudioPlayback` 或等价播放链证据  
5. 若仍无声音，明确下一阻断点是 publish、subscribe 还是 playback  

### 必要验收条件

- 若 connect 成功，至少能明确音频链进入了哪一层
- 若仍失败，能明确下一步不是继续查入口层，而是转向更下一层诊断

### 解释

本工作包是有条件触发的。  
如果 `Room.connect()` 仍没成功，本版就不应再继续假装已经进入音频阶段。

---

## 6.5 4.6A-E：回归、证据、closeout 收口

### 目标

让这次修复不是“感觉入口好了”，而是能被正式用于 4.6 的后续验收判断。

### 必做任务

1. 更新连接层排障报告  
2. 更新修复说明  
3. 更新修复后验证说明  
4. 补干净重启后的设备侧日志摘录  
5. 更新 4.6 的 acceptance summary / handoff / closeout 口径  
6. 明确本轮修复的是连接层，不是继续扩功能  

### 必要验收条件

- 文档说明与实际验证一致
- 能支撑顾问判断 4.6 是否继续推进、是否仍为 conditional pass、或是否进入下一层小修回合

## 8. 本版验收结论建议口径

## 8.1 何时可以视为 `pass with observations`

仅当以下同时成立：

- 云端入口层修复固定化完成
- 干净重启后的设备侧复验完成
- `Room.connect()` 已稳定成功
- 后续进入 publish / subscribe 观察阶段，且剩余问题只是非阻断 observation

### 建议

`4.6A` 的现实目标不是追求纯 `pass`，而是：  
**让 4.6 从“连接层已定位但设备侧复验未完成”推进到“连接层已确认通过，可继续进入真语音闭环”这一阶段。**

## 8.2 何时仍只能维持 `conditional pass`

若出现以下情况之一：

- 设备侧仍继续吃到旧入口  
- `Room.connect()` 仍不稳定成功  
- 仍出现 timeout / reset 但说不清发生在哪一层  
- 只能证明 validate 可达，不能证明客户端真正完成入房  

则 `4.6` 仍应维持 `conditional pass`，不得强行归档。

## 9. 风险清单

## 9.1 最大风险：把连接层修复做成 RTC 大重构

控制原则：
- 只修入口层、join-info、代理与干净复验
- 不扩成新的 RTC 架构版

## 9.2 第二风险：旧缓存 / 旧进程干扰判断

控制原则：
- 必须干净重启
- 必须记录新 join-info
- 必须确认客户端实际使用的 url

## 9.3 第三风险：入口层已好，但团队继续盲改 UI

控制原则：
- 先证实 `Room.connect()`
- 再进入 publish / subscribe
- 不倒退回页面猜测

## 9.4 第四风险：修复描述超过真实进度

控制原则：
- validate 可达不等于真语音闭合
- 设备侧新证据不到位，不得写成已完成

## 16. 最终建议

本版的最佳策略不是继续盲改 RTC 页面，而是：

- 目标是把 4.6 从“入口层已定位但仍未完成复验”推进到“入口层已复验，可继续进入真语音闭环”这一阶段
