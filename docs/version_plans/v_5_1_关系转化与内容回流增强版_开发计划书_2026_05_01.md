# 0. 一句话结论

5.1 不回头重开 4.x 已归档主链，也不重复再做一版“Discover / Chat / Me 的最小壳层”。

本版本目标是：

> 在 4.9 稳定门禁基线与 5.0 产品化起点之上，把“关系如何继续推进、内容如何自然回流”做成真正可感知的产品链路。

5.1 的核心成果不是再补几个散点功能，而是：

1. 首聊 / 回聊 / 冷场续话形成更完整的安全回流队列；
2. 匹配解释、聊天建议、查看主页、语音邀请之间形成自然节奏；
3. 动态 / 状态 / 通知 / 消息之间出现低压转化链，而不是彼此割裂；
4. 通知中心从“提醒堆叠”升级成“关系回流中枢”；
5. 所有新增能力都在 4.9 release gate / UI baseline / DB safety / media / RTC / notification 主链上 additive 承接；
6. 为后续小样本测试准备真正高频、可复用的关系推进路径，而不是继续堆产品骨架。

# 1. 版本定位

## 1.1 版本名称

**5.1 关系转化与内容回流增强版**

## 1.2 版本类型

> 只从 4.9 + 5.0 additive baseline 承接，不以“关系转化增强”为由重写已归档主链。

# 2. 背景与问题来源

## 2.1 为什么需要 5.1

5.0 已经把 Discover / Chat / Me 从“功能骨架”推进到了“最小产品化起点”，但当前仍有一个非常现实的缺口：

> 用户即使已经能看内容、能进聊天、能看主页，系统仍然缺少一套更顺的“继续聊下去 / 再回来看 / 再推进一步”的产品节奏。

也就是说，5.0 解决的是：

- 页面不像工程页；
- Discover / Chat / Me 具备产品入口；
- AI 开始轻量嵌入主链。

而 5.1 要解决的是：

- 为什么用户愿意继续打开同一个会话；
- 为什么用户会从动态 / 状态回到聊天；
- 为什么用户会在通知里继续处理一个关系；
- 为什么匹配解释不会停留在静态展示，而能变成下一句该怎么聊；
- 为什么语音邀请是“关系节奏中的自然升级”，而不是突兀按钮。

## 2.2 5.1 要解决的核心问题

本版本重点解决以下问题：

1. 首聊、回聊、冷场恢复仍偏散点，尚未形成稳定的队列与推荐节奏；
2. 匹配解释、共同点、主页信息与聊天建议还没有真正串成一条链；
3. 状态流 / 动态流发布后，缺少“回到消息”“继续了解”“低压私聊”的明确回流路径；
4. 通知中心还更像提醒列表，尚未完全承担“关系回流与继续处理”的产品角色；
5. 语音入口虽然已有，但还需要更明确地绑定到关系推进语义，而不是仅作为独立动作；
6. 所有这些增强都必须建立在既有真值链、消息主链、通知主链、RTC 主链之上，而不能另起炉灶。

# 5.2 5.1B：首聊 / 回聊 / 冷场恢复队列增强

## 目标

把聊天页中的首聊、回聊、冷场恢复从单点建议，升级成更清楚的安全回流队列。

## 建议范围

### A. 首聊 / 回聊队列正式化

建议最少形成以下能力：

- `继续聊` 队列；
- `稍后再回` 队列；
- `冷场后重新接起` 队列；
- 当前推荐节奏的轻提示。

### B. 建议来源轻标注

建议轻量说明建议基于：
- 匹配解释；
- 共同点；
- 最近互动；
- 动态 / 状态内容。

### C. 冷场恢复模板

建议至少补齐：
- 追问型；
- 共鸣型；
- 周末安排型；
- 低压回归型。

## 任务

1. 梳理当前首聊 / 恢复建议结构；
2. 设计并接入最小队列层；
3. 增补冷场恢复模板；
4. 产出：
   - `docs/version_plans/5.1_CHAT_QUEUE_NOTE.md`
   - `docs/version_plans/5.1_CHAT_QUEUE_EVIDENCE_INDEX.md`

## 禁止事项

- 不做大模型问答平台；
- 不让建议层盖住真实会话；
- 不主动向真实用户发送任何内容做验证；
- 不把模板做成高压营销式文案。

## 验收标准

- 首聊 / 回聊 / 冷场恢复路径有可见区分；
- 用户能理解“现在为什么推荐这句”；
- 模板轻量、可跳过；
- walkthrough 截图完整。

# 5.3 5.1C：匹配解释到聊天建议联动增强

## 目标

把匹配解释、共同点、主页信息、聊天建议从分散信息点，串成真正可用的关系推进链。

## 建议范围

### A. 关系摘要深化

在 5.0 聊天页关系摘要基础上，继续补：
- 共同点摘要；
- 当前节奏提示；
- 下一步建议；
- 查看主页 / 回到匹配解释入口。

### B. 匹配解释联动

建议至少实现：
- 从匹配解释页或关系摘要中生成追问建议；
- 从共同点快速插入聊天建议；
- 从主页标签生成续话提示。

### C. 关系节奏动作条

建议统一处理：
- 查看资料；
- 继续聊天；
- 语音联动；
- 稍后再聊。

## 任务

1. 梳理当前匹配解释与聊天页联动断点；
2. 增强关系摘要层；
3. 补轻量联动入口；
4. 产出：
   - `docs/version_plans/5.1_RELATIONSHIP_LINKAGE_NOTE.md`

## 禁止事项

- 不重写匹配算法；
- 不把关系摘要变成重诊断页；
- 不生成过度拟人化或过度确定性的 AI 判断。

## 验收标准

- 关系摘要比 5.0 更可执行；
- 查看资料、继续聊、语音联动之间路径自然；

# 5.4 5.1D：动态 / 状态到私聊的低压转化增强

## 目标

把内容表达层和聊天层真正连起来，但坚持慢约会语义，不做高压转化按钮。

## 建议范围

### A. 状态卡片 CTA 重排

建议优先采用：
- `继续了解`；
- `看看资料`；
- `先收藏`；
- `再想想`；
- `低压私聊`。

避免只剩一个“立即私聊”的强动作。

### B. 内容回流

建议补齐：
- 从状态广场回到聊天列表；
- 从通知回到动态原位；
- 从收藏 / 赞过回到资料页或继续了解页；
- 轻量“稍后再聊”存入队列。

### C. 低压转化文案

建议把“打招呼”“继续聊”“看看资料”“先不急”这些动作写得更符合慢约会口径。

## 任务

1. 梳理当前状态流 / 内容卡片动作；
2. 调整 CTA 与动作节奏；
3. 增补内容回流入口；
4. 产出：
   - `docs/version_plans/5.1_CONTENT_TO_CHAT_NOTE.md`
   - `docs/version_plans/5.1_CONTENT_TO_CHAT_EVIDENCE_INDEX.md`

## 禁止事项

- 不把状态流做成强社交 feed；
- 不引入“疯狂私聊”型高压按钮；
- 不让内容流失去慢约会主线。

## 验收标准

- 状态 / 动态卡片 CTA 更像“低压关系转化”；
- 能证明内容到聊天存在自然回流路径；
- 文案不突兀、不强迫。

- 首页；
- Discover / 发现页；
- 消息列表页；
- 聊天页；

- 我的页；

- 星盘页；
- 现代卡片、间距、轻蓝紫配色、底部导航与现代布局骨架。

5.1 可以增强关系转化与回流动作，但不得把现代 UI 重新做回工程页或旧版 UI。

## 6.2 真值链保护面

接口：

- `POST /api/v1/profile/basic`
- `GET /api/v1/profile/basic`
- `GET /api/v1/profile/astro/summary`
- `GET /api/v1/profile/astro/chart`
- `POST /api/v1/profile/astro`

表：

- `user_astro_profiles`
- `questionnaire_questions`
- `questionnaire_answers`
- `questionnaire_attempts`

规则：

- 资料真值仍以服务端重算结果为准；
- 标签表达、AI 建议、关系提示不得回写真值；
- 内容标签区是表达层，不是真值层替代。

## 6.3 消息与媒体主链保护面

接口：

- `GET /api/v1/messages`
- `POST /api/v1/messages`
- `POST /api/v1/messages/read/{messageId}`
- `POST /api/v1/media`
- `GET /api/v1/media/{assetId}`

表：

- `chat_messages`
- `conversations`
- `conversation_members`
- `media_assets`
- `media_processing_jobs`
- `message_attachments`

规则：

- 首聊 / 回聊 / 建议层只做产品增强，不新造平行聊天链；
- 不得破坏图片 / 视频 / 附件发送主链；
- 不得用 UI 建议成功掩盖真实消息失败。

## 6.4 通知与内容主链保护面

表：

- `status_posts`
- `status_post_likes`
- `notifications`
- `moderation_reports`
- `user_blocks`
- `user_relationship_events`

规则：

- 内容回流增强必须建立在现有状态流和通知主链之上；
- 轻治理动作继续保持轻量，不扩成复杂社区系统。

## 6.5 RTC / LiveKit 保护面

# 8. 跨层 Blocker 处理条款

5.1 尽管是产品增强版，但很容易触碰：
- 通知回流；
- 聊天建议来源；
- 动态与消息联动；
- 语音入口与 RTC；
- 发布链。

因此继续强制执行跨层 blocker 规则。

## 8.1 触发条件

以下任一情况都要进入 blocker 流程：

- 页面显示关系建议成功，但真实回流错误；
- 通知按钮看似成立，但跳转页错误；
- 语音邀请按钮存在，但动作链失败；
- 消息、状态、通知之间路径不一致；
- 建议来源、共同点、匹配解释与实际数据不一致；
- 构建、网络、端口、协议、后端环境异常。

## 8.2 标准流程

1. 冻结现象；
2. 先写 Markdown blocker report；
3. 请 Claude 做根因缩层；
4. Codex 只做最小修复；
5. 用用户面截图 + 机器证据双重复验；
6. 写 fix note / verification note / acceptance summary。

## 8.3 禁止事项

- 禁止没有 blocker report 就继续盲改；
- 禁止跨层 blocker 连续多轮只在 Flutter / UI 层死磕；
- 禁止用假成功态掩盖真实回流失败；
- 禁止根因未定前扩大版本范围。

# 9. Rollback / Recovery Policy

## 9.1 禁止 repo 级恢复

默认禁止：

```bash
git reset --hard <old_commit>
git checkout <old_commit> .
git restore --source=<old_commit> .
git clean -fdx
```

也禁止：
- 用旧 zip 覆盖当前仓库；
- 为修 5.1 局部问题恢复整仓；
- 恢复旧 UI 覆盖 protected surfaces。

## 9.2 允许的恢复方式

只允许：
- path-level restore；
- file-level restore；
- smallest-scope restore。

## 9.3 恢复前必须 checkpoint

恢复前必须执行：

```bash
git status --short
git branch --show-current
git rev-parse HEAD
git diff --stat
git diff --name-only
```

并至少执行一种：

```bash
git switch -c safety/pre-recovery-<date>-<topic>
```

或：

```bash
git add -A
git commit -m "safety: checkpoint before <topic> recovery"
```

或：

```bash
git stash push -u -m "safety: checkpoint before <topic> recovery"
```

# 19. 最终执行提醒

5.1 的关键不是“加更多关系功能”，而是：

> **把内容、聊天、通知、语音和资料页之间的低压关系推进链真正串起来。**

4. 这是否只是 additive，而不是 destructive rewrite？
5. 这是否需要先 blocker report，而不是继续盲修？
