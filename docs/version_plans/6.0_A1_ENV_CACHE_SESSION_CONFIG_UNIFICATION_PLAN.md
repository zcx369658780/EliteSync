## 2. 目标配置原则

- Redis 作为 cache / queue / rate limit 的目标承载。
- session 推荐路径：
  - A1 staging / v2 readiness 目标推荐使用 Redis session，但必须独立 connection / prefix / TTL，并确保 maxmemory policy 不会淘汰登录态。
  - production v1 可短期保留 database session 作为过渡口径，理由是降低已通过登录与 Sanctum token 链路的迁移风险；切 Redis 前必须先在 staging 完成 login / authenticated readonly / write smoke。
- local / staging / production 必须分离，不共享 DB、Redis keyspace、cache prefix、session prefix、queue prefix、storage 或 feature flag。
- production 默认继续 v1。
- v2 只能 staging / controlled backend 验证。
- 不允许 host-level env 污染导致 local / production 口径混淆。
- 不允许把 local 环境标识写成 production 可见结论。
- 不允许未确认 config cache 状态就进入 v2 skeleton。
- 所有 production 配置变更必须另开任务、先备份、用户明确确认。

## 4. Cache 统一方案

当前 cache store 来源：

- `config/cache.php` 的 `default` 读取 `APP_CACHE_STORE`，默认 `file`。
- `.env.example` 写 `CACHE_STORE=redis`，当前不会被 `config/cache.php` 直接读取。
- Redis cache store 使用 `REDIS_CACHE_CONNECTION`，默认 `cache`。
- Redis cache DB 在 `config/database.php` 的 `cache` connection 中读取 `REDIS_CACHE_DB`，默认 `1`。

推荐 cache store：

- local：可用 `file`，如需要贴近 staging 可用 `redis`。
- staging：`redis`。
- production：`redis`，但任何变更必须先备份 `.env` / config cache 状态并用户确认。

Redis cache DB / prefix 隔离建议：

- `REDIS_CACHE_CONNECTION=cache`。
- `REDIS_CACHE_DB` 与 queue/session/default 不混用。
- `CACHE_PREFIX=elitesync_<env>_cache_`。
- `REDIS_PREFIX=elitesync_<env>_`。

config cache 生效风险：

- 如果 production 已 cache config，即使 `.env` 里有 `CACHE_STORE=redis`，应用也可能仍读取已缓存的 `APP_CACHE_STORE=file` 或旧值。
- 未执行只读 `php artisan config:show cache` 前，不应把目标 key 写成实际已生效。

进入 skeleton 前必须确认项：

- `APP_CACHE_STORE` 最终值。
- `REDIS_CACHE_CONNECTION` 与 `REDIS_CACHE_DB`。
- `CACHE_PREFIX` 与 `REDIS_PREFIX` 分环境。
- config cache 是否存在以及 loaded value。
- health/readiness 中 cache check 应返回真实状态，不得伪造 pass。

后续允许执行的只读检查命令：

```powershell
php artisan about
php artisan env
php artisan config:show cache
```

后续需要用户确认才允许执行的变更命令：

```powershell
php artisan optimize:clear
php artisan config:clear
php artisan cache:clear
php artisan config:cache
```

本阶段未执行任何 cache 相关命令。

## 5. Session 统一方案

当前 session driver 来源：

- `config/session.php` 的 `driver` 读取 `APP_SESSION_DRIVER`，默认 `file`。
- `.env.example` 写 `SESSION_DRIVER=database`，当前不会被 `config/session.php` 直接读取。
- session connection 读取 `APP_SESSION_CONNECTION`，回退 `DB_CONNECTION`。
- `SESSION_STORE` 仅在 cache-driven session backend 中使用。

推荐 session driver：

- A1 目标推荐：Redis session，用于与 cache / queue / rate limit 的 Redis 目标栈对齐。
- production v1 过渡推荐：如果当前生产已稳定依赖 database session，则短期可保留 database session，直到 staging 验证 Redis session 后再切换。该过渡不得写成 v2 商用目标完成。

如果采用 Redis session：

- `APP_SESSION_DRIVER=redis`。
- `APP_SESSION_CONNECTION=cache` 或后续新增独立 `session` connection。
- `SESSION_STORE=redis` 需与 Laravel 实际 store 定义匹配。
- `SESSION_LIFETIME` 需记录并与登录体验一致。
- session prefix 必须通过 `REDIS_PREFIX` 与 `CACHE_PREFIX` 分环境隔离。
- Redis `maxmemory-policy` 不得淘汰 session；建议不要与 volatile cache 使用同一淘汰策略。
- 必须回归 login、token refresh、auth:sanctum、logout、authenticated readonly smoke。

如果采用 database session：

- 需确认 `sessions` 表存在。
- 若不存在，不得直接 migration；必须另开 migration plan、备份、回滚、staging 验证与用户确认。
- 优点是登录态可审计且不受 Redis eviction 影响。
- 风险是 DB 压力增加，且与 Redis 统一目标不完全一致。

local / staging / production 差异：

- local：`file` 或 `database`，可降低本地依赖。
- staging：建议先试 Redis session，作为 v2 readiness 目标。
- production：保持当前稳定口径，切换前必须完成 staging smoke 与 rollback。

Sanctum / auth:sanctum 风险：

- EliteSync 移动端主要使用 bearer token；但 Sanctum 配置仍包含 stateful domains 与 session middleware。
- session driver 切换可能影响 cookie-based 或 web guard 相关路径。
- 不得在未验证登录、refresh、authenticated GET、logout 的情况下切换 production session driver。

进入 skeleton 前必须确认项：

- `APP_SESSION_DRIVER` 最终 key 与目标值。
- `APP_SESSION_CONNECTION` 或最终 session connection。
- 是否保留 database session 作为 production v1 过渡。
- Redis session 是否有独立 DB/prefix 与 eviction 安全策略。
- 是否已完成 auth smoke。

本阶段未执行 migration，未改变 session driver。

## 6. Queue 统一方案

当前 queue connection 来源：

- `config/queue.php` 的 `default` 读取 `QUEUE_CONNECTION`，再 fallback `APP_QUEUE_CONNECTION`，默认 `database`。
- `.env.example` 写 `QUEUE_CONNECTION=redis`，与 config 优先 key 一致。
- Redis queue connection 读取 `REDIS_QUEUE_CONNECTION`，默认 `default`。
- Redis queue name 读取 `REDIS_QUEUE`，默认 `default`。
- retry 读取 `REDIS_QUEUE_RETRY_AFTER`，默认 `90`。
- failed jobs 使用 `QUEUE_FAILED_DRIVER`，默认 `database-uuids`。

推荐 queue connection：

- local：`sync` 或 `redis`，按开发任务选择。
- staging：`redis`。
- production：`redis`，并且必须有常驻 worker 管理方案。

queue worker 未常驻风险：

- 当前 Gate P1 已记录 queue worker 未常驻。
- 如果有 media、notification、location、match、astro、buddy 异步任务，无 worker 会造成“API 成功但后台处理不运行”的假成功。

queue 分层建议：

- `default`：普通业务异步。
- `notifications`：通知创建、推送和回流。
- `media`：图片 / 视频 / 缩略图 / 转码。
- `match`：Date Drop 与匹配解释相关后续任务。
- `astro`：星盘 / 八字 / 紫微重算与渲染任务。
- `buddy`：搭子请求、匹配、反馈。
- `low`：统计、清理、低优先级任务。

failed_jobs / retry / backoff / timeout 策略：

- `failed_jobs` 必须可用。
- 每类 job 需定义 `tries`、`backoff`、`timeout`。
- media / astro 类长任务必须有独立 timeout。
- 关键写链路必须幂等，避免 retry 重复写入。

systemd / Supervisor：

- systemd：适合 A1 初期少量 worker，部署轻。
- Supervisor：适合多队列多 worker 分组，后续更清晰。
- Horizon：当前未安装，不作为 A1 前置。

queue:restart：

- 只能在后续用户确认后执行。
- production 执行前必须备份 worker 配置并记录回滚方式。

skeleton 前 carry-over：

- 完全只读 v2 health/readiness/location contract skeleton 可 carry-over queue worker 未常驻，但 readiness 必须标为 `degraded`、`skipped` 或 `not_configured`。
- 任何 async / write 链路前，queue worker 常驻与 failed job 策略必须 resolved。

本阶段未启动 worker，未执行 queue 命令。

## 7. Rate Limit 统一方案

当前 throttle / rate limit 状态：

- v1 route 已使用 `throttle:auth`、`throttle:messages`、`throttle:media`、`throttle:notifications`、`throttle:conversations` 等 middleware。
- 本阶段未发现专用 `RATE_LIMIT_STORE` config 读取点。
- Laravel rate limiter 若无专门配置，通常依赖 cache store；因此 cache store 口径不统一会影响 throttle 可信度。

Redis rate limit 目标口径：

- staging / production 的 rate limit 应由 Redis 承载，避免 file cache 或单机 cache 导致限流不一致。
- 建议后续明确 `RATE_LIMIT_STORE=redis` 或在应用层明确 limiter 使用 cache store。
- 限流 key prefix 必须按 environment 分离。

auth / messages / media / notifications / conversations throttle 风险：

- auth：登录 / 注册限流必须稳定，避免暴力尝试。
- messages：发送与列表读取存在聊天保护面，不得因缓存清理而失效。
- media：上传登记需要防止滥用。
- notifications：read/unread 和通知列表不得被错误限流。
- conversations：会话列表与读回执相关，限流需兼顾用户体验。

local / staging / production 分离：

- local 可使用 file / array 进行开发，但测试限流行为时应使用 Redis。
- staging 必须用 Redis 验证 throttle 行为。
- production 必须使用 Redis 或明确等价集中式 store。

进入 skeleton 前判定：

- 对只读 contract skeleton，可作为 observation carry-over。
- 对任何 auth、messages、media、notifications 或 write endpoint skeleton，rate limit store 未明确会成为阻断项。

## 10. Production Safety Gate

- 本阶段不改 production。
- 本阶段不执行 production artisan。
- 本阶段不读写 production `.env` secret。
- 本阶段不改 Nginx / PHP-FPM / Redis / Supervisor。
- 本阶段不启动 queue worker。
- 本阶段不改 release chain，不运行 release script，不生成 APK。
- 本阶段不请求 API，不执行 smoke。
- 任何 production 配置变更必须另开任务、先备份、用户明确确认。
- health 返回 `environment=local` 是 observation，需要后续纳入环境口径修复，但不在本阶段修。
