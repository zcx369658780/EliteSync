# 0. 一句话目标

在不破坏现有 v1 运行链路和 release chain 的前提下，完成 EliteSync 后端 v2 的 contract-first 最小骨架、staging / production 分离基础、位置语义拆分最小闭环，并清理用户侧精确经纬度暴露，为后续 Date Drop 匹配主链与搭子精准陪伴打下商用级底座。

---

# 1. 当前背景

6.0-A0 已完成路线冻结，EliteSync 正式进入 6.0 Alpha 内测准备线。A0 明确了 6.0 Alpha 的 P0 优先级：先处理后端 v2 与位置链路，再进入 Date Drop 式匹配主链、搭子精准陪伴、基础社交补齐、玄学解释与 UI/IA 内测打磨。

A1 是 6.0 Alpha 的第一个 runtime 版本，也是 P0 商用化底座的第一阶段。它不是功能堆叠版本，而是后续 A2-A5 可以安全执行的基础设施版本。A1 的风险高于 A0，因为它可能触碰后端、API、配置、位置展示、环境分离甚至 DB / migration 规划。因此本版本必须采用更严格的执行方式：

```text
只读审计 -> contract map -> migration / rollback plan -> 最小 runtime slice -> staging 验证 -> 保护面回归 -> Claude 横向复评 -> GPT 顾问最终验收
```

A1 不能一次性推倒重写后端，不能直接替换地图 SDK，不能直接改 production DB，不能混入搭子、Date Drop 主链、UI/IA 大改或 release chain。

# 2. 本版本目标

## 2.0 商用级目标栈冻结

A1 必须先把后端 v2 的商用级目标栈写入项目源与 contract 规划。当前目标不是更换到 Go / Java / Node，而是在 Laravel 体系内升级到可长期维护、可分环境部署、可扩容、可监控、可回滚的商用级后端架构。

A1 的商用级目标栈暂定为：

```text
OS：Ubuntu 24.04 LTS
Web：Nginx + PHP-FPM
Backend：Laravel 12 或 Laravel 13，不得以 Laravel 11 作为 v2 商用目标版本
Runtime：PHP 8.4，后续视 Laravel 目标版本评估 PHP 8.5
Database：MariaDB 10.11 LTS 最新补丁版，文档中写作 MariaDB 10.11 / MySQL-compatible，而非笼统 Oracle MySQL
Cache / Queue / Session / Rate Limit：Redis
Async Jobs：Laravel Queue + 独立 worker / supervisor / horizon-like 监控方案
Storage / Media：OSS + CDN，媒体资源不得压在 Laravel 本机磁盘
RTC：独立 LiveKit / RTC 服务，不并入 Laravel 主业务进程
Observability：error tracking + structured logs + metrics + alerting
Backup：DB backup + config backup + release rollback plan
Load Test：staging 压测，不打 production 真实用户链路
Deployment：staging / production 分离，v1 / v2 并行，feature flag / base URL 可回滚
```

A1 执行前必须新增 `Framework / Runtime Support Gate`：

1. 审计当前 Laravel 11.48.0、PHP 8.4.18、MariaDB 10.11.14、Nginx、Redis 的支持状态；
2. 确认 Laravel 11 不作为后端 v2 商用级目标版本；
3. 给出 Laravel 12 与 Laravel 13 的升级选择建议；
4. 检查 composer 依赖、PHP 扩展、Laravel package、queue、cache、session、auth、storage、scheduler、test suite 的兼容风险；
5. 给出 MariaDB 10.11.14 升级到 10.11 LTS 最新补丁版的方案；
6. 明确 MariaDB 与 Oracle MySQL 的文档口径差异；
7. 给出 staging 先行、production 后置、可回滚的升级路径；
8. 在该门禁完成前，不得开始后端 v2 skeleton runtime。

本计划书进入仓库项目源同步阶段时，仍然只代表 A1 planning/source sync，不代表已经允许启动 A1 runtime。只有 Framework / Runtime Support Gate 完成、风险清单与文件清单通过审查，并获得用户明确确认后，才允许进入 contract map / skeleton / 位置链路最小闭环 runtime。

## 2.1 主目标

A1 的主目标是完成“后端 v2 与位置链路最小闭环”，不是完成完整后端重构。

完成后应达到：

1. v1 API contract 被审计并形成保护清单；
2. v2 contract map 初版形成，并与 v1 兼容策略对应；
3. 后端 v2 skeleton 或 v2 namespace / module 具备最小健康检查与 contract 结构；
4. staging / production 分离原则与最小配置链路可验证；
5. 位置链路拆分为出生地、现居地、约会地点、搭子地点四类语义；
6. 用户侧不再暴露精确经纬度、火车站名等不可信位置展示；
7. 外部地图辅助策略形成 POC 或明确后续实现边界；
8. 备份、回滚、压测、监控最小要求被写入并验证；
9. 不破坏现有 v1 登录、资料、星盘、Match、Chat、Media、RTC、版本检查主链；
10. Claude 横向复评与 GPT 顾问最终验收通过。

## 2.2 A1 不是完整后端重构

A1 不要求完成：

- 全量 v2 API；
- 全量数据迁移；
- 完整 staging / production 运维平台；
- 完整地图选点系统；
- 搭子数据表与推荐逻辑；
- Date Drop 匹配算法重构；
- 完整监控告警平台；
- 正式发版。

A1 的成功标准是“底座方向正确、保护面未破坏、最小垂直切片可验证”。

# 3. 本版本范围

## 3.1 做什么

### 3.1.1 v1 API contract 审计

Codex 必须先审计并记录现有 v1 API 的 contract：

```text
Auth / User
Profile basic
Profile astro summary
Profile astro chart
App health
App version check
Messages
Media
Questionnaire
RTC calls / LiveKit / heartbeat
Notifications
```

输出：

```text
docs/version_plans/6.0_A1_V1_CONTRACT_AUDIT.md
```

审计目的不是改代码，而是明确哪些 v1 contract 不得被 A1 破坏。

### 3.1.2 v2 contract map 初版

新增或整理后端 v2 contract map，至少覆盖：

| 域 | A1 要求 |
|---|---|
| app / health | v2 health / readiness / version-compatible contract 初稿 |
| profile | profile truth / display profile / location context 的拆分说明 |
| astro | canonical truth 与 explanation layer 分离说明 |
| match | Date Drop 主链未来 contract 预留，不做 runtime 算法 |
| buddy | 搭子请求 / 匹配 / 反馈 future contract 预留，不做 runtime |
| location | 出生地 / 现居地 / 约会地点 / 搭子地点四类语义 contract |
| feedback | alpha feedback / match feedback future contract 预留 |
| operations | staging / production / logs / monitor / rollback contract 要求 |

输出：

```text
docs/version_plans/6.0_A1_BACKEND_V2_CONTRACT_MAP.md
```

### 3.1.3 后端 v2 skeleton 最小闭环

在不破坏 v1 的前提下，允许建立最小 v2 skeleton。

允许范围：

- 新增 v2 namespace / route group / controller skeleton；
- 新增 v2 health / readiness 只读接口；
- 新增 contract response wrapper 或 DTO 草案；
- 新增 v2 feature flag / config 读取，不影响 v1 默认路径；
- 新增最小测试，证明 v1 health / version check 仍可用，v2 health / readiness 可用。

建议接口：

```text
GET /api/v2/app/health
GET /api/v2/app/readiness
GET /api/v2/contracts/location
```

其中 `GET /api/v2/contracts/location` 可以先返回静态 contract schema 或说明性 JSON，用于验证 v2 contract exposure，不写业务数据。

禁止范围：

- 不迁移 v1 profile 写接口；
- 不迁移 v1 astro 接口；
- 不迁移 v1 messages；
- 不迁移 v1 media；
- 不迁移 v1 RTC；
- 不迁移 v1 version check；
- 不让 Flutter 默认切到 v2；
- 不让 production 用户流量进入 v2。

### 3.1.4 staging / production 分离基础

A1 必须整理环境分离最小方案。

允许完成：

- 文档化当前本地 / staging / production 的配置来源；
- 新增或整理 staging config 模板；
- 明确 Flutter / backend base URL 切换原则；
- 明确 v2 只能先在 staging 或受控环境验证；
- 明确生产环境默认仍走 v1；
- 明确任何生产配置修改必须用户确认。

可以新增：

```text
docs/version_plans/6.0_A1_ENVIRONMENT_SPLIT_PLAN.md
```

除非用户明确授权，本版本不得直接修改 production 环境配置。

### 3.1.5 位置链路重构最小闭环

A1 必须把位置语义拆成四类：

| 类型 | 主要用途 | 用户侧展示 | 是否允许精确坐标 | A1 最小动作 |
|---|---|---|---|---|
| 出生地 | 星盘 / 八字 / 紫微计算 | 行政地名 | 内部可选，用户侧不展示 | 清理用户侧经纬度 / 火车站式展示，保留服务端真值 |
| 现居地 | 匹配范围 / 生活圈 / 内测分布 | 城市 / 行政区 | 默认不展示 | 明确 display 字段与 truth 字段分离 |
| 约会地点 | Date Drop 见面建议 | 地点名 / 商圈 / 城市 | 可选，非资料真值 | 只做 contract 预留，不做推荐 |
| 搭子地点 | 学习 / 电影 / 吃饭 / 健身搭子 | 城市 / 商圈 / 用户确认地点 | 可选，需隐私保护 | 只做 contract 预留，不做搭子 runtime |

A1 最小 runtime 可做：

- 清理 Profile / Astro / location 展示中精确经纬度；
- 将火车站名、坐标串、engine debug 字段从普通用户可见位置移除；
- 增加用户可理解位置展示文案；
- 保留内部计算需要的服务端真值，不反写、不清空；
- 新增位置语义 mapping 文档与测试样例。

输出：

```text
docs/version_plans/6.0_A1_LOCATION_CHAIN_RESTRUCTURE.md
```

### 3.1.6 外部地图辅助 POC

A1 只做策略或轻量 POC，不做正式地图 SDK 替换。

允许：

- 调研当前位置选择链路；
- 明确外部百度地图 / 腾讯地图 / 高德地图 App 辅助定位的可行性；
- 规划“打开外部地图搜索 / 导航 / 手动回填地点名”的低风险方案；
- 记录后续是否需要 Web 选点页或 SDK。

禁止：

- 不直接接地图 SDK；
- 不直接改定位权限；
- 不直接采集实时位置；
- 不强依赖地图 App 返回坐标；
- 不把经纬度展示给普通用户。

### 3.1.7 监控、备份、回滚、压测最小方案

A1 必须为后端 v2 后续迁移建立最小工程约束。

至少输出：

```text
docs/version_plans/6.0_A1_BACKUP_ROLLBACK_MONITORING_PLAN.md
```

内容包括：

- 任何 DB / migration 前必须备份；
- production 写操作必须云端执行并经用户确认；
- staging 验证必须先于 production；
- v2 feature flag 必须可回滚；
- v1 API 必须有回归检查；
- v2 health / readiness 必须可监控；
- 压测不得打 production 真实用户链路；
- A1 不以“压测高并发通过”为必需目标，只要求压测计划与最小 smoke。

### 3.1.8 文档同步与 A1 handoff

A1 完成后必须同步：

```text

docs/DOC_INDEX_CURRENT.md

```

并新增：

```text

```

handoff master 必须是 A1 唯一主交接入口。

## 3.2 不做什么

A1 明确不做以下事项。

### 3.2.1 不做完整后端重写

- 不推倒重写 Laravel；
- 不一次性替换 v1 API；
- 不把 Flutter 默认流量切到 v2；
- 不迁移全部业务域；

- 改 production schema；

- 不改 versionCode；
- 不改 changelog_v0；
- 不改 about_update_0_xx；
- 不改 app_update.php 默认发布配置；

### 3.2.4 不做 A2-A5 功能

- 不做 Date Drop 匹配主链 runtime；
- 不做搭子页面 / DB / API / 推荐；
- 不做基础社交功能补齐；
- 不做玄学入口矩阵；
- 不做大规模 UI/IA 改版；
- 不接真人咨询、付费报告、上传截图主路径。

### 3.2.5 不做高风险地图与位置功能

- 不接地图 SDK；
- 不采集实时位置；
- 不展示精确经纬度；
- 不默认请求定位权限；
- 不把外部地图返回结果写入资料真值；
- 不把出生地、现居地、约会地点、搭子地点混写。

# 5. Protected surfaces

A1 是 runtime 版本，但必须严格保护现有 v1 主链。

## 5.1 绝对保护面

A1 默认不得修改以下 canonical truth 语义：

```text
profile/basic
profile/astro/summary
profile/astro/chart
user_astro_profiles
```

不得：

- 清空出生地；
- 改写出生时间；
- 改写星盘 / 八字 / 紫微真值；
- 把 display location 写回 canonical truth；
- 把位置清理当成资料清理；
