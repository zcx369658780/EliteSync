# EliteSync 4.5 版本开发计划书

文件名：`elite_sync_4_5_版本开发计划书_2026_04_21.md`  
更新时间：2026-04-21  
适用范围：`4.4` 已正式归档、`4.4S` 媒体链稳定性修正版已完成后的下一版本承接  
当前承接基线：`4.0` 媒体基础设施、`4.1` 非官方四维人格问卷、`4.2` 图片消息主链、`4.3` 动态流基础版、`4.4` 视频消息版、`4.4S` 媒体链稳定性修正版均视为稳定基线。  
本版定位：`4.5 通知与社交转化增强版`

---

## 0. 本版一句话结论

`4.5` 不做新的重媒体能力，也不做 RTC / 通话，而是把 EliteSync 从“功能已经具备”推进到“用户能被提醒、能被召回、能被轻量转化”的下一阶段。

本版只做：
- 通知中心最小闭环
- 站内通知事件模型与读取状态闭环
- 首页 / 消息 / 动态 / 匹配之间的轻量转化入口
- 首聊建议、互动提醒、回访入口的产品化收口
- 与现有动态、图片消息、视频消息、匹配解释的 additive 联动

本版明确不做：
- 语音 / 视频通话
- 在线状态系统
- 重型推送平台
- 新推荐算法大改
- 视频动态 / 内容社区扩张
- 微服务重构
- 运营后台平台化扩写

换言之，`4.5` 的目标不是把 EliteSync 做成复杂通知系统，而是建立一个 **可提醒、可回流、可促成互动** 的最小社交转化层。

---

## 1. 版本定位

## 1.1 版本名称

`4.5 通知与社交转化增强版`

## 1.2 版本使命

在不破坏 `3.9 / 4.0 / 4.1 / 4.2 / 4.3 / 4.4 / 4.4S` 已归档主链的前提下，补齐以下能力：

1. 用户能在站内看到与自己相关的关键通知  
2. 通知能覆盖消息、动态互动、喜欢 / 匹配、首聊建议等核心事件  
3. 用户能从通知回流到正确页面，而不是只看到一条孤立提醒  
4. 首页、匹配页、动态页、消息页能形成轻量转化闭环  
5. 通知系统不污染 canonical truth，也不把产品做成高噪声提醒中心  
6. 为未来推送网关、设备 token 与更复杂召回能力预留边界，但本版不做重型扩展  

## 1.3 为什么现在做 4.5

### 原因 A：4.4 与 4.4S 已经把媒体基础链修到可用

现在图片与视频消息、动态表达层都已经具备基础闭环，下一步最自然的问题不是“还能不能发”，而是“发了之后怎样被看见、被回应、被召回”。

### 原因 B：4.5 比 4.6 更适合作为下一承接层

`4.6` 的 RTC / 通话是更重的实时基础设施版本，依赖：
- 更清晰的事件模型
- 更成熟的通知与回流路径
- 更稳定的消息 / 媒体 / 互动链
- 更明确的用户召回与入口设计

先做 `4.5`，可以让“互动提醒与社交转化”先稳定，再承接更重的实时能力。

### 原因 C：当前产品已经有足够多的可触发事件，但缺少统一的回流层

当前已有：
- 匹配解释与首聊建议
- 图片 / 视频消息
- 动态流与治理
- 首页轻量联动

但这些能力还没有被一个清晰的“通知 + 回流 + 转化入口”体系串起来。`4.5` 的职责就是收口这一层。

## 2. 承接基线

本版必须承接并保护以下基线：

### 2.1 已冻结的真值与主链

- `POST /api/v1/profile/basic` 仍是资料真值入口
- `user_astro_profiles` 仍是画像 / 星盘核心真值表
- `chat_messages` 仍是消息主链真值表
- `app_release_versions` 仍是发布真值表
- `questionnaire_attempts / questionnaire_answers / questionnaire_questions` 仍是问卷真值表

### 2.2 已冻结的能力底座

- `4.0` 媒体 / 存储 / 队列 / 缓存 / worker 最小闭环
- `4.1` 非官方四维人格问卷模块
- `4.2` 图片消息主链
- `4.3` 动态流基础版
- `4.4` 视频消息版
- `4.4S` 媒体链稳定性修正版

### 2.3 已存在的相关模块面

- 首页 `Home`：入口聚合、轻量联动、人格摘要与转化入口可用
- 发现 `Discover`：内容分发骨架与动态入口存在
- 匹配 `Match`：解释、首聊建议、转化入口稳定
- 消息 `Chat`：文本 + 图片 + 视频消息主链可用
- 动态 `Status / Feed`：发布、读取、互动、举报 / 拉黑链路已存在
- 我的 `Me / Profile`：资料、设置、版本中心、人格摘要、星盘入口已归位
- 治理 `Moderation`：举报、拉黑、封禁、审核链路已存在

### 2.4 本版最重要的承接口径

`4.5` 是 **通知与社交转化增强版**，不是“推送平台版”，也不是“推荐系统重构版”。  
所有实现都应遵循：

- additive feature，不做 destructive rewrite
- 先建立站内通知闭环，再考虑外部推送扩写
- 先建立事件到页面的回流链，再考虑更复杂召回策略
- 先做低噪声、高相关性的通知，不做铺天盖地事件轰炸
- 先利用现有匹配、动态、消息和媒体主链，不重写它们

## 3. 本版做什么 / 不做什么

## 3.1 本版必须完成的核心目标

### 目标 1：站内通知中心最小闭环

至少支持：
- 通知列表读取
- 通知已读 / 未读状态
- 未读计数最小闭环
- 下拉刷新 / 续读最小闭环
- 点击通知进入对应页面

### 目标 2：核心通知事件模型收口

至少支持以下通知类型中的最小可用集合：
- 新消息通知
- 动态被点赞 / 被互动通知
- 被喜欢 / 匹配成功通知
- 首聊建议 / 回访建议类轻提醒
- 必要的系统通知（如版本提示、服务状态变更）

### 目标 3：社交转化入口闭环

至少支持：
- 从通知回到聊天页
- 从通知回到动态详情或作者页
- 从通知回到匹配解释页或首聊页
- 首页保留轻量“去回应 / 去看看 / 去开启聊天”入口

### 目标 4：通知噪声控制与治理挂点

至少支持：
- 通知文案不工程化
- 同类事件不过度轰炸
- 已失效目标不误导跳转
- 被拉黑 / 被删除 / 被屏蔽内容在通知侧可解释或降噪

### 目标 5：后续推送能力的接口预留

至少支持：
- `notifications` 作为站内真值主表继续稳定
- `device_tokens` / `notification_deliveries` 等结构只做最小预留或最小落表
- 不要求本版把 APNs/FCM/厂商推送全打通

## 3.2 本版明确不做的内容

以下内容禁止偷渡进入 `4.5`：

### 通知能力不做
- 完整推送网关平台
- 厂商推送全适配
- 大规模消息营销体系
- 通知偏好中心全量平台化

### 社交能力不做
- 新的动态大改版
- 新的关系链重构
- 新推荐算法主链
- 内容社区运营化扩张

### 实时能力不做
- 在线状态系统
- 呼叫状态机
- 语音通话
- 视频通话
- RTC 信令

### 架构能力不做
- 微服务重构
- 消息中台化
- 新建复杂事件总线平台
- 对 `4.0` 媒体底座、`4.2` 图片消息、`4.4` 视频消息主链推倒重来

### UI 能力不做
- 全站视觉重做
- 首页 / 消息页 / 匹配页结构重构
- 把通知中心做成大而全运营页面

## 4. 硬性保护面

本版开发前必须先冻结以下保护面，不得误伤：

## 4.1 真值链保护面

- `POST /api/v1/profile/basic`
- `GET /api/v1/profile/astro/summary`
- `GET /api/v1/profile/astro/chart`
- `user_astro_profiles`
- `questionnaire_*` 真值链

### 解释

通知与转化层是“事件与回流层”，不是“资料 / 画像 / 匹配真值层”。  
任何通知点击、未读状态、转化行为，都不得反向写成 canonical truth。

## 4.2 消息与媒体保护面

- `GET /api/v1/messages`
- `POST /api/v1/messages`
- `POST /api/v1/media`
- `chat_messages`
- `media_assets`
- `media_processing_jobs`
- `message_attachments`

### 解释

`4.5` 可以消费消息和媒体事件生成通知，但不能为了通知去改坏消息主链、附件结构或媒体状态机。

## 4.3 动态流与关系链保护面

- `status_posts`
- `status_post_likes`
- `moderation_reports`
- `user_blocks`
- `user_relationship_events`

### 解释

`4.5` 应复用这些事件来源，而不是借机重做动态流、重做关系链或扩大治理平台。

## 4.4 发布与版本保护面

- `GET /api/v1/app/health`
- `GET /api/v1/app/version/check`
- `app_release_versions`
- 宿主 APK 版本口径
- about / version center / release script 绑定链

### 解释

`4.5` 是功能增强版，不是发版链重构版。  
任何通知页面、服务状态提醒、版本提醒都不能破坏版本链的一致性。

## 4.5 环境与数据库边界（长期硬规则）

本版必须严格遵守：

- 本地环境只做前端开发、UI 联调、截图采集、文档整理
- 后端开发、数据库迁移、备份、恢复、写库排障统一在阿里云端执行
- 不允许本地开发库、SQLite、临时数据库污染云端正式后端数据

### 解释

`4.5` 会触发更多事件、更多通知写入，因此更必须守住“本地前端 / 云端后端”这条边界，不得再出现本地覆盖远端数据库的隐患。

## 5. 数据与接口边界建议

## 5.1 数据模型建议

优先复用现有核心表：
- `notifications`
- `app_events`
- `user_relationship_events`
- `chat_messages`
- `status_posts`
- `moderation_reports`
- `user_blocks`
- `conversations`
- `conversation_members`

如仓库中尚未完整落表，再按最小必要原则新增或补齐：
- `notification_deliveries`：仅用于投递记录 / 阅读状态扩展
- `device_tokens`：仅用于后续推送预留，不把本版做成推送平台
- `notification_preferences`：若需要，只做最小偏好骨架，不做全量通知设置中心

### 边界解释

本版不允许为了“以后扩展”一次性建一个大而全通知系统。  
若现有 `notifications` + 事件表已足够承接站内通知，则优先在此基础上补齐。

## 5.2 接口域建议

### 优先策略

- 站内通知读接口：优先收口到 `/api/v1/notifications/*`
- 消息、动态、匹配、治理事件：优先复用现有事件来源，不新造平行事实源
- 首页快捷与轻入口：优先复用 `GET /api/v1/home/*`

### 建议最小接口集合

- `GET /api/v1/notifications`
- `POST /api/v1/notifications/read/{notificationId}`
- `POST /api/v1/notifications/read-all`
- `GET /api/v1/notifications/unread-count`
- `GET /api/v1/notifications/{notificationId}`（若确有必要）

若需要最小偏好或设备预留，可考虑：
- `POST /api/v1/notifications/device-token`
- `DELETE /api/v1/notifications/device-token/{id}`
- `GET /api/v1/notifications/preferences`
- `POST /api/v1/notifications/preferences`

### 边界解释

本版不要求一次性把所有推送、偏好、营销消息、事件订阅都做完。  
核心原则是：
- 通知写路径清晰
- 通知读路径可用
- 通知能回流到正确页面
- 站内先闭环，外推后置

## 6. 工作包拆分

## 6.1 4.5A：边界冻结与通知事件模型收口

### 目标

在动代码前，先把 `4.5` 的版本边界、通知类型、事件来源、回流页面、噪声控制口径写清楚，避免版本越界。

### 必做任务

1. 明确本版通知类型最小集合  
2. 明确每类通知由哪条事实链触发  
3. 明确通知跳转的目标页面和失效回退策略  
4. 明确站内通知与未来推送之间的边界  
5. 明确哪些表复用，哪些表最小新增  
6. 明确本版不做完整推送平台、不做在线状态、不做通话前置  

### 必要验收条件

- 存在一份清晰的边界说明
- 列出新增表 / 新增字段 / 新增接口清单
- 明确哪些旧表和旧接口不可碰
- 明确本版通知是低噪声、回流型，不是运营消息中心

### 解释

没有边界说明直接进入实现，很容易出现：
- 通知类型越做越多
- 事件来源写乱
- 页面跳转混乱
- 消息、动态、匹配各自造通知逻辑而无法收口

因此 `4.5A` 是本版硬前置，不可省略。

---

## 6.2 4.5B：后端通知写入、读取与状态闭环

### 目标

建立通知写入、通知读取、已读状态、未读计数、事件映射的最小后端闭环。

### 必做任务

1. 完成通知写入主链  
   - 新消息类通知
   - 动态互动类通知
   - 匹配 / 喜欢类通知
   - 系统轻提醒类通知

2. 完成通知读取接口  
   - 通知列表
   - 通知详情（如确有必要）
   - 未读数读取

3. 完成已读状态闭环  
   - 单条已读
   - 批量已读 / 全部已读
   - 未读计数与列表状态一致

4. 完成通知失效 / 回退逻辑  
   - 目标已删除时有合理回退
   - 被拉黑 / 被屏蔽 / 被治理内容不误导跳转
   - 失效通知可降噪处理

5. 完成最小 telemetry / 事件记录  
   - 通知创建
   - 通知读取
   - 通知点击
   - 通知回流成功 / 回流失败

### 必要验收条件

- 通知列表能稳定返回
- 已读 / 未读状态切换一致
- 未读计数不漂移
- 通知点击后的回流目标正确
- 失效目标不会把用户带去错误页面

### 解释

本工作包的目标不是做营销触达平台，而是把“事件 -> 通知 -> 用户读取 -> 页面回流”这条链闭合。  
如果后端只有通知表，没有读取与回流状态，实际上不具备真实测试能力。

---

## 6.3 4.5C：Flutter 端通知中心、入口与页面回流

### 目标

让用户能够以产品化方式看到通知、区分状态、进入对应页面，并感知到这是“有用的提醒”，而不是噪声。

### 必做任务

1. 通知中心入口与 IA 收口  
   - 消息页、首页或我的页中明确通知入口位置
   - 不重写底部导航主结构
   - 不把通知页做成运营后台面板

2. 通知列表页  
   - 通知卡片展示
   - 已读 / 未读状态区分
   - 时间、摘要、类型、操作入口
   - 下拉刷新 / 分页最小闭环

3. 通知卡片交互  
   - 点击进入对应页面
   - 未读状态自动或显式更新
   - 失效通知有可理解提示

4. 与关键页面的转化联动  
   - 回到聊天页
   - 回到动态详情 / 作者页
   - 回到匹配解释页 / 首聊建议页
   - 首页轻量承接“去回应 / 去看看 / 去继续聊”入口

5. 必要的轻量空态 / 错误态 / 失败态  
   - 没有通知时不空洞
   - 拉取失败时可重试
   - 不把错误页做得比成功页更重

### 必要验收条件

- 通知入口清晰，不需要工程背景才能理解
- 通知卡片可读，不像调试面板
- 点击通知回流链路可走通
- 未读与已读状态表达清楚
- 文案不工程化、不做运营骚扰感

### 解释

对 `4.5` 而言，前端关键不是页面数量，而是“通知是否真的帮助用户回到正确互动”。  
如果通知页只是事件堆叠，不形成回流，那版本不能算成立。

---

## 6.4 4.5D：转化策略、噪声控制与回归保护

### 目标

在版本收尾前，把通知最容易失控的部分提前收住：噪声、失效回流、状态漂移、误伤旧主链。

### 必做任务

1. 噪声控制最小收口  
   - 同类事件不过度重复
   - 非关键提醒不抢主路径
   - 系统通知不压过真实社交互动

2. 转化入口一致性检查  
   - 首页、消息页、通知页的入口口径一致
   - 回流文案不互相冲突
   - 首聊建议、匹配解释、动态互动不互相抢主线

3. 回归保护  
   - `profile/basic` 真值链回归
   - `astro/summary`、`astro/chart` 回归
   - `messages` 文本 / 图片 / 视频消息回归
   - `status_posts` 动态流回归
   - `questionnaire/history`、`questionnaire/answers` 回归
   - `app/version/check` 回归

4. 日志与错误可见性  
   - 通知创建失败、读取失败、点击回流失败有日志
   - 不允许失败只留在控制台

5. 本地 / 云端边界检查  
   - 本地不做写库通知排障
   - 后端写库、迁移、修复全部在阿里云端执行

### 必要验收条件

- 通知页不出现明显噪声崩坏
- 回流失败可解释
- 回归清单完整，旧主链未被误伤
- 云端数据库边界未被破坏

### 解释

`4.5` 若没有噪声控制与回归保护，就很容易变成“功能都有，但用户体验更乱”。  
因此本工作包不是附录，而是正式验收的一部分。

## 6.5 4.5E：截图证据、验收摘要、handoff、closeout

## 8. 4.5 的验收结论建议口径

## 8.1 何时可以给 `pass`

仅当以下同时成立：
- 通知写入、读取、未读、回流全链路成立
- UI 状态表达清楚
- 回归完整
- 截图证据完整
- handoff / closeout 完整
- 无需再补关键收尾材料

## 8.2 何时给 `pass with observations`

用于：
- 主链已经完整成立
- 可以正式归档
- 仍有少量非阻断观察项，例如：
  - 某些文案仍可继续降噪
  - 某些通知卡片密度还可优化
  - 某些空态还可继续产品化

### 建议

`4.5` 最现实的目标是：**争取做到 `pass with observations` 后冻结主链。**  
不要为了追求更复杂的推送或召回体系而在 `4.5` 上反复开补丁。

## 8.3 何时只能给 `conditional pass`

若出现以下情况之一：
- 通知主链虽跑通，但截图证据包明显不足
- 没有 closeout / handoff
- 回流链路说明不清楚
- 已读 / 未读、未读数或目标跳转存在明显漂移

则应判为 `conditional pass`，并仅允许小回合补收尾，不建议重开大开发。

## 9. 风险清单

## 9.1 最大风险：版本越界成“推送平台版”

表现：
- 站内通知做着做着变成完整运营消息系统
- 厂商推送、营销消息、偏好中心一起做厚
- 把 `4.5` 做成大型通知中台

控制原则：
- 只做站内通知 + 轻量转化闭环
- 不把外推平台和营销系统偷渡进来

## 9.2 第二风险：事件来源混乱

表现：
- 消息、动态、匹配各自写通知，没有统一模型
- 回流页面不一致
- 事件写入漂移，未读数不可信

控制原则：
- 统一通知主表和事件映射口径
- 先收口最小事件模型

## 9.3 第三风险：通知噪声过重

表现：
- 页面像提醒轰炸器
- 用户看不出什么最重要
- 社交提醒与系统提醒混在一起

控制原则：
- 高相关、低噪声优先
- 首聊、消息、匹配、动态互动优先于杂项提醒

## 9.4 第四风险：误伤旧主链

表现：
- 为了通知去改坏消息、动态、匹配、版本链
- 回流跳转引发旧页面路由异常

控制原则：
- 通知层只消费既有事实链，不重写既有主链

## 9.5 第五风险：本地 / 云端写库边界失守

表现：
- 本地后端写库再次污染云端数据
- 临时数据库被误当正式源

控制原则：
- 本地只做前端联调
- 后端写库、迁移、恢复、修复统一云端执行
