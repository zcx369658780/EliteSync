### 2.3 当前阶段判断

当前 EliteSync 已不再处于“从 0 到 1 堆功能”的阶段，而是处于：

> **从可演示产品走向小规模真实 Beta 上线准备的阶段。**

当前的主要问题不再是“缺少前台功能”，而是：

1. 是否能稳定支撑真实 Beta 用户
2. 是否具备完整的测试与验收闭环
3. 是否具备性能、监控、灰度、回滚能力
4. 是否具备最小安全与合规保障
5. 是否具备真实运营接入所需的值班与应急能力

因此，`2.9` 的重点不是新增前台玩法，而是**上线工程化收口**。

## 3. 2.9 版本定位

### 3.1 一句话定位

`2.9` 是 **Beta 上线准备版**，目标不是继续扩张前台功能，而是把当前系统推进到：

> **可以小规模接入真实用户、可监控、可回滚、可灰度、可验收的 Beta 系统。**

---

### 3.2 版本总目标

2.9 版本要回答以下问题：

1. 当前系统能否稳定支撑真实 Beta 用户
2. 关键链路是否可重复验收
3. 出现异常时能否快速发现、定位、回滚
4. 隐私、安全、权限、治理是否足以支撑小规模上线
5. 运营侧是否具备最小审核、值班、应急能力

---

### 3.3 非目标

2.9 明确不做以下事项，避免范围失控：

1. 不回头扩张 `2.7` / `2.8` 的功能范围
2. 不做大规模 UI 重构
3. 不新增多媒体聊天等重功能
4. 不引入前端第二套规则引擎
5. 不在主线上做大规模算法重写
6. 不因为后台或演示需求污染 canonical 真源链路

---

## 4. 当前必须继续遵守的架构原则

后续所有开发必须遵守以下原则：

1. **服务端 canonical 数据是唯一真源**
2. **前端缓存只能兜底，不能成为第二真源**
3. **不要为了 UI / 后台改动误伤匹配真值、资料真源、聊天主链路**
4. **任何高风险改动必须有 smoke、regression、rollback**
5. **后台与治理能力不能反向污染前台体验链**
6. **算法解释层、展示层、排序层继续分离，不混为一体**

## 7. Codex 与 Gemini 的职责分工

## 7.1 Codex 角色定义

Codex 在 2.9 中的角色：

> **执行代理（Implementation Agent）**

Codex 负责：
- 阅读任务说明
- 界定改动边界
- 实施代码修改
- 补充测试与脚本
- 输出变更清单
- 输出风险说明
- 输出待 Gemini 验收的材料

Codex 不负责：
- 自行放宽版本范围
- 自行引入高风险架构重构
- 以“看起来更合理”为由修改 canonical 真源策略
- 用临时缓存替代服务端真值
- 跳过 smoke / regression / rollback 要求

---

## 7.2 Gemini 角色定义

Gemini 在 2.9 中的角色：

> **验收代理（Acceptance Agent）**

Gemini 负责：
- 依据计划书检查任务是否越界
- 依据 Protected Surfaces 检查是否误伤主链路
- 依据验收标准执行验收
- 输出通过 / 不通过结论
- 输出剩余风险与缺口
- 决定是否允许结项或进入下一阶段

Gemini 不负责：
- 直接改动代码作为主执行者
- 擅自扩大需求范围
- 以主观审美替代客观验收标准

## 14. 最终结论

EliteSync 的 `2.9` 不应被理解为普通功能版本，而应被定义为：

> **工程化收口版 / Beta 上线准备版。**

它的核心价值不是再做更多功能，而是让当前已经完成的产品能力真正具备：

- 可重复验收
- 可真实联调
- 可监控
- 可灰度
- 可回滚
- 可小规模接入真实用户

因此，后续所有执行与验收都应围绕这一目标展开，而不是重新回到“继续堆功能”的节奏。
