# 算法 2.5 版本开发规划（可直接下发 Codex）

> 版本定位：在 2.4 已通过并完成尾包归档的基础上，2.5 版本聚焦两件事：
> 1. **关闭性格测试 / MBTI 全链路能力，但保留数据库与兼容数据，不删除历史数据结构**
> 2. **新增“梅花斗数”模块并完成工程落地**：包括计算、测试账号回填、后台匹配算法更新、个人信息展示更新。
>
> 本版本不是继续扩张模块数量，而是完成“旧模块下线 + 新模块落地 + 匹配权重重构 + 解释层对齐”的一次结构性迭代。

# 1. 2.5 的核心判断与工程前提

## 1.1 先把术语问题解决
“梅花斗数”在公开技术生态里的命名并不稳定，2.5 必须先做一个**术语与业务目标确认门**，否则很容易选错技术路线。

### 公开生态里更清晰的两条路线
1. **梅花易数**
   - 常见特征：时间起卦、数字起卦、本卦/互卦/变卦、体用五行分析
   - 更像“事件占断 / 决策辅助”
   - 更接近一次性或场景性推演

2. **紫微斗数**
   - 常见特征：基于出生日期、出生时辰、性别排盘，输出 12 宫、主星、流年等
   - 更适合做“持久化命盘画像”
   - 更自然地进入用户画像、个人信息页、长期匹配算法

## 1.2 工程上的默认结论
**2.5 默认按“梅花易数画像版（Meihua Profile V1）”落地**，但必须把它定义为：
- 服务端可重复计算
- 以用户出生信息为固定种子生成稳定画像变量
- 用于匹配解释与轻量画像展示
- **不宣称它等同于传统“即时占断”意义上的标准梅花易数实战过程**

## 1.3 若产品最终确认其实要的是“紫微斗数”
则 2.5 的梅花分支应立即切换为：
- **紫微斗数路线（Iztro / py-iztro）**
- 不与“梅花易数画像版”混用
- 字段和展示名重新按 Ziwei 体系定义

> 2.5 规划里默认继续按“梅花斗数 = 梅花易数画像版”执行，但必须在 Phase 2.5-0 形成书面确认记录。

# 2. 2.5 总体目标与通过标准

## 2.1 总体目标
完成一次明确的模块替换：
- 让性格测试 / MBTI 从“关闭但残留在接口和算法里”变成“前台关闭、后台禁用、算法清零、数据库保留”
- 让“梅花斗数”从“产品想法”变成“服务端计算 + 数据回填 + 排序接入 + 展示落地”的正式模块

## 2.2 通过标准（必须全部满足）
1. **性格测试 / MBTI 在前台、后台、API、排序、解释层全部关闭**
2. **数据库结构与历史数据保留，不做破坏性删除**
3. **梅花模块有唯一真值、唯一服务端引擎、唯一解释契约**
4. **所有测试账号完成回填并产出批处理报告**
5. **匹配算法权重完成重构，MBTI / 性格权重为 0**
6. **个人信息页、匹配解释页、后台配置页同步支持梅花模块**
7. **新模块通过许可证门禁、解释门禁、灰度与回滚门禁**

# 5. 主线 B：新增梅花模块（Meihua Profile V1）

## 5.1 目标
把“梅花斗数”做成一个可重复、可展示、可进入匹配的服务端模块。

## 5.2 2.5 的工程定义
2.5 默认实现的是：
**Meihua Profile V1（梅花画像版）**

它的特点是：
- 使用用户出生信息作为固定 seed
- 输出稳定、可重复的“梅花画像变量”
- 用于用户画像、匹配解释、模块级评分
- 不把它包装成“即时起卦占断”
- 与八字、属相、星座、western_lite 并列为匹配模块

## 5.3 推荐的输出变量
### 基础结构字段
- `seed_mode`：`birth_datetime_seed | date_only_seed | manual_seed`
- `ben_gua`：本卦
- `hu_gua`：互卦
- `bian_gua`：变卦
- `dong_yao`：动爻
- `ti_gua`：体卦
- `yong_gua`：用卦
- `ti_yong_relation`：体用关系
- `wu_xing_relation`：五行关系
- `gua_element_summary`：卦象五行摘要

### 画像标签字段
- `meihua_style_tags`：性格/相处节律标签
- `meihua_process_tags`：推进方式标签
- `meihua_risk_tags`：摩擦风险标签
- `meihua_advice_tags`：相处建议标签

### 展示与门禁字段
- `confidence`
- `confidence_reason`
- `engine_source`
- `engine_mode`
- `display_guard`
- `degraded`
- `degrade_reason`

## 5.4 推荐的数据表设计
新建：`user_meihua_profiles`

建议字段：
- `id`
- `user_id`（unique）
- `seed_mode`
- `seed_payload`（json）
- `ben_gua`
- `hu_gua`
- `bian_gua`
- `dong_yao`
- `ti_gua`
- `yong_gua`
- `ti_yong_relation`
- `wu_xing_relation`
- `style_tags`（json）
- `process_tags`（json）
- `risk_tags`（json）
- `advice_tags`（json）
- `notes`（json）
- `computed_at`
- `created_at`
- `updated_at`

## 5.5 单一真值原则
- **主真值**：`user_meihua_profiles`
- `users` 表最多只保留轻量镜像摘要字段，不保存完整 canonical

如需镜像，推荐新增：
- `users.private_meihua_summary`（json|null）

## 5.6 服务端实现建议
新增：
- `MeihuaCanonicalService`
- `MeihuaProfileService`
- `MeihuaFeatureExtractor`
- `MeihuaExplanationBuilder`

### 运行模式
- `engine_mode = canonical | fallback`
- 默认禁止客户端本地计算
- 统一由服务端生成

## 5.7 默认输入规则
### 完整输入
- `birthday + birth_time`
- 置信度：中到高

### 缺失出生时刻
- 只用 `birthday`
- 置信度降级
- 前端必须明确显示“基于部分信息估算”

### 无法形成稳定种子
- 禁止输出强结论
- 可输出“资料不足，暂不生成梅花画像”

## 5.8 解释层定位
建议把梅花模块定义为：
- `layer = bridge` 或 `process`
- 偏“关系推进方式、节律互动、体用关系”
- 不抢八字的结果层主导权

## 5.9 交付物
- `docs/meihua_profile_spec_2_5.md`
- `docs/meihua_explanation_contract_2_5.md`
- `services/backend-laravel/app/Services/MeihuaCanonicalService.php`
- `services/backend-laravel/app/Services/MeihuaFeatureExtractor.php`

## 5.10 验收标准
- 新模块可对同一用户稳定重复计算
- 数据落库到唯一真值表
- 解释层与 display_guard 对齐
- APP 与后台都能读取并展示

# 6. 主线 C：重构匹配算法与解释层

## 6.1 目标
在关闭 personality / mbti 之后，重新构造 2.5 的排序结构。

## 6.2 2.5 新的核心评分原则
### 旧结构（2.4）
- personality 50%
- mbti 15%
- astro_total 35%

### 新结构（2.5）
推荐改为：
- `basic_profile_fit`：20%
- `astro_total`：80%

## 6.3 `basic_profile_fit` 的组成
不再来自问卷或 MBTI，而来自现有结构化资料：
- 年龄差适配
- 同城 / 城市层级
- 婚恋目标一致性
- 认证状态
- 活跃度 / 曝光公平性

> 说明：这些因素原本就已经存在于硬过滤与软调节中，2.5 只是把其中一部分显式收敛为非问卷型基础画像分，避免删除 personality 之后只剩玄学单模块主导。

## 6.4 `astro_total` 的 2.5 调整原则
### 原则
- 八字仍保持主模块地位
- 属相、星座仍为辅助
- western_lite 继续低口径、低权重
- 梅花模块以“影子评分 -> 小流量启用 -> 正式接入”的方式逐步加入

### 上线建议
#### Step 1：shadow
- `score_meihua_shadow` 只计算不进最终分

#### Step 2：canary
- `score_meihua` 小权重接入
- 同时输出 diff 报告

#### Step 3：GA
- 正式写入 `score_meihua`
- 并纳入 `score_astro_total`

## 6.5 建议新增字段
### `dating_matches`
新增：
- `score_meihua` `integer|null`
- `score_meihua_shadow` `integer|null`
- `meihua_explanation` `json|null`（如不单列，则继续进入 `match_reasons`）

### 解释输出
新增模块 key：
- `meihua_profile`

### explanation blocks
要求输出：
- 结论
- 过程
- 风险
- 建议
- `confidence`
- `display_guard`
- `layer`

## 6.6 匹配解释层要求
### 梅花模块必须解释
- 本卦/互卦/变卦代表什么样的关系推进特征
- 体用关系对互动意味着什么
- 风险点来自哪里
- 哪些建议是基于中低置信推断

### 禁止
- 不得把梅花模块写成“唯一决定关系成败”的模块
- 不得在低置信时给强口气结论

## 6.7 交付物
- `docs/matching_reweight_2_5.md`
- `docs/meihua_shadow_rollout_2_5.md`
- `reports/meihua_weight_diff_2_5.md`

## 6.8 验收标准
- personality / mbti 权重为 0
- `basic_profile_fit` 替代问卷型基础画像分
- 梅花模块先 shadow，再 canary，再正式接入
- 最终排序与解释都能看到梅花模块

# 7. 主线 D：测试账号回填、灰度、回滚与验收

## 7.1 目标
确保 2.5 不是“只对新用户生效”的版本。

## 7.2 测试账号回填
### 范围
- 所有 `is_synthetic = true` 的测试账号
- 可扩展到顾问白名单账号

### 回填内容
- 生成 `user_meihua_profiles`
- 写入 `users.private_meihua_summary`（如启用镜像）
- 生成 `score_meihua_shadow`
- 生成 explanation snapshot

### 报告要求
输出：
- 总账号数
- 成功数
- 失败数
- 失败原因分布
- 缺失出生时刻占比
- 低置信占比

## 7.3 灰度策略
### 灰度阶段
1. Dev / 本地测试
2. 测试账号批量回填
3. 顾问白名单
4. 小流量真实用户
5. 全量启用

### 需要支持的开关
- `FEATURE_MEIHUA_ENABLED`
- `MATCHING_MEIHUA_SHADOW_ENABLED`
- `MATCHING_MEIHUA_WEIGHT`
- `PROFILE_SHOW_MEIHUA_ENABLED`
- `ALLOW_MBTI_ENDPOINTS`（默认 false）

## 7.4 回滚策略
### 支持回滚
- 关闭梅花展示
- 关闭梅花评分接入
- 只保留 shadow 计算
- 恢复 2.4 评分 profile
- 保留已计算数据但不展示

## 7.5 验收样例
### 最少 30 组回归样例
覆盖：
- 完整出生信息
- 缺失出生时刻
- 高分高风险
- 中分稳定
- 八字高 / 梅花低
- 梅花高 / 八字低
- western_lite 低置信 + 梅花中置信
- 测试账号批量回填失败场景

## 7.6 交付物

## 7.7 验收标准
- 测试账号已批量回填
- 灰度与回滚有真实演练证据
- explanation 样例可复核
