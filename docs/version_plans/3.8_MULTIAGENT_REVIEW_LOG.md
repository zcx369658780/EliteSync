## 1. 审查目标

- 参数工作台深化后的 schema / DTO

## 2. Claude 审查结论

### 2.1 总判断

- 结论：`conditional pass`
- 允许继续推进 3.8，但存在必须修正的边界问题

### 2.2 Claude 识别到的 must-fix

1. **route_mode 与缓存复用的边界需要更明确**
   - 当前 Laravel `appendPythonNatalChart()` 会在存在 `private_natal_chart.chart_data` 时复用缓存，并通过 `withRouteContext()` 仅覆盖路由展示上下文。
   - Claude 认为这会让“同一 chart_data + 不同 route_mode 标签”在校准阶段更容易被误读为引擎或参数差异。
   - 建议：对 route_mode 的 display-only 语义做更明确的文案/测试约束，避免被误当成真值差异。

2. **advanced_context 字段角色描述需要进一步统一**
   - Python 侧 `metadata.field_roles.advanced_context` 与 `metadata.advanced_context` 的呈现方式需要继续保持一致、可追溯。
   - 建议：后续补一层统一的 explanation schema，显式区分 `derived_only`、`display_only`、`advanced_context`。

3. **3.8 的参数工作台 DTO 还缺少版本化的强约束**
   - 当前 `api.py` 仍以 `route_mode` 为主，3.8 的黄道制 / 宫位制 / 相位密度 / 容许度 / 点位集还需要更明确的参数版本化与默认 fallback。
   - 建议：在进入 UI 实现前，先冻结参数命名与默认值策略。

## 3. Claude 给出的边界建议

- 冻结字段：
  - 出生资料 canonical 基座不动
  - `chart_data` 主结构不动
  - `route_mode` 保持 display-only
- 可扩展字段：
  - `metadata.route_context.*`
  - `metadata.advanced_context.*`
  - 新增 explanation blocks，但必须显式标注 `derived_only / display_only / advanced_context`
- 推荐命名：
  - `*_interpretation`
  - `*_explanation`
  - `*_derived_context`
  - `*_display_block`
- 不建议：
  - `*_truth`
  - `*_canonical`
  - 将解释字段并入 natal truth 主结构

## 7. Gemini UI 审查结论

### 7.1 总判断

- 结论：`pass with observations`
- 可以继续推进，但需要优先修正命名与层级边界表达

### 7.2 Gemini 识别到的 must-fix

1. **盘面设置页中的“摘要开关”命名歧义**
   - `行星摘要 / 宫位摘要 / 相位摘要 / 技术参数 / 紧凑密度` 被收在同一层，但名称容易让人误解为“开关摘要”，不是“显示内容开关”。
   - 建议改成更明确的动宾表达，如“显示行星摘要”“显示技术参数”等。

### 7.3 Gemini 识别到的 observations

- `设置口径` 这个概念重要，但需要在页面层更显著地表达“本地偏好 / 服务端真值 / 版本中心”的边界。
- `盘面设置入口 -> 参数工作台页` 的表述与页面内部“工作台参数”区域之间存在轻微歧义，需要明确是独立页面还是页面内分区。
- `只影响本地展示，不影响命盘真值` 的提示应保持高显著性，最好位于首屏可见区域。
- 路线模板 / 快速预设会同步推荐工作台参数，用户需要明确知道这是“推荐值”还是“强制覆盖”。
- `高级演示页`、`视觉门户`、`预览日志` 等命名略偏技术化或泛化，后续可按用户语义再打磨。
- `盘面设置` 与 `参数工作台页` 的 UI 边界需要更明确，避免让用户误以为所有参数都只是展示层偏好。

### 7.4 Gemini 的布局建议

- 首屏优先展示：路线模板、快速预设、核心参数（黄道制 / 宫位制 / 相位密度 / 容许度 / 点位集）
- `盘面元素开关` 建议折叠或分组，避免首屏过载
- `工作台参数`、`摘要开关`、`盘面元素开关` 应分组明确，减少阅读负担
- `玄学总览` 页面建议更明确地区分基础解读入口与高级能力入口
- `高级解读` 页面里的 `预览日志` 更适合改成更用户化的命名

### 7.5 Gemini 的边界结论

- `derived-only / display-only / advanced-context` 提示是做对的，必须继续保持
- `设置中心`、`盘面设置`、`高级解读` 的边界要通过文案、图标和层级继续强化
- Gemini 认为当前 UI 版本可以继续往下做，但必须先修正命名歧义和页面层次
