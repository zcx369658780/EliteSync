## 2. 推荐 implementation 方案

推荐方案：Option A：same server + separate port + security group allowlist。

推荐理由：

- 如果用户希望后续 Codex 可直接做 R1 readonly metadata verification，Option A 的 staging base URL 最直接。
- 既有 server inventory 已确认同机具备 Nginx / PHP-FPM / MariaDB / Redis 基础环境，且已完成 Nginx/config backup，可作为 future rollback reference。
- 独立端口 + 独立 Nginx config + security group allowlist 可以避免把 `101.133.161.203` 根路径误当 staging。
- Option A 的风险主要在同机 staging 误影响 production，需要通过独立 port、独立 staging env、独立 DB / Redis / storage boundary、`nginx -t`、用户授权 reload 和 rollback plan 控制。

Alternative:

- Option B：same server + SSH tunnel only。若用户希望最大限度降低公网暴露，推荐 Option B。风险是 Codex / 外部工具访问流程更复杂，后续 R1 readonly metadata verification 需要 tunnel endpoint 管理。
- Option C：separate lightweight staging server。当前不推荐，除非用户愿意增加服务器成本。优点是隔离最清晰，缺点是成本和配置工作量更高。

明确不推荐：

- 不推荐使用 `101.133.161.203` 根路径。
- 不推荐使用 `slowdate.top`。
- 不推荐使用 `staging.slowdate.top`，除非未来 ICP 备案完成。
