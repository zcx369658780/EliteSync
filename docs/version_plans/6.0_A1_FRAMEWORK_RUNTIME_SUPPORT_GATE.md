## 2. Laravel 11 -> 12/13 升级路线

官方口径：

- Laravel 官方 release policy 显示：Laravel 11 bug fix 到 2025-09-03，security fix 到 2026-03-12；Laravel 12 bug fix 到 2026-08-13，security fix 到 2027-02-24；Laravel 13 已在 2026-01-06 发布，bug fix 到 2027-07-21，security fix 到 2028-01-19。参考：https://laravel.com/docs/releases
- Laravel 13 文档已存在，说明 A1 目标写 Laravel 12/13 是合理的，但直接跳 13 需要包生态和项目代码审计。参考：https://laravel.com/docs/13.x

当前状态：

- Laravel 11.48.0 仍可运行，但不应作为 6.0-A1 后端 v2 商用目标版本。
- 当前项目实际依赖约束仍为 `laravel/framework ^11.31`，因此 v2 目标栈落地前必须先做 composer 依赖路线审计与升级分支验证。

Laravel 12 可行性：

- PHP 8.4 已满足 Laravel 12 的 PHP 版本方向。
- Sanctum `v4.3.1` 要求 `illuminate/* ^11.0|^12.0|^13.0`，从包约束看支持 Laravel 12/13。
- PHPUnit `11.5.55` 可继续作为 12 路线的测试底座候选。
- 需要重点验证：Workerman WebSocket 接入、现有 middleware/auth/session/cache/queue/storage/scheduler 行为、项目自定义服务容器绑定和 route/cache/config cache 行为。

Laravel 13 可行性：

- Laravel 13 已发布，理论上可作为 A1 商用目标候选。
- 但 Laravel 13 会引入更多上游差异，当前项目尚未完成 v1 contract audit / v2 contract map，不建议 A1 第一轮直接跳 13。
- 建议先完成 11 -> 12 staging 验证，再根据 Laravel 13 package 生态、项目自定义代码审计和测试覆盖决定是否 12 -> 13。

## 4. MariaDB 10.11 LTS 最新补丁版升级方案

当前状态：

- 当前 MariaDB：`10.11.14-MariaDB-0ubuntu0.24.04.1`
- `apt-cache policy mariadb-server mariadb-client` 显示 installed/candidate 均为 `1:10.11.14-0ubuntu0.24.04.1`
- 当前 Ubuntu/Aliyun apt 源暂无比 10.11.14 更高的 MariaDB 10.11 包候选
- MariaDB 10.11 为 LTS 分支，官方维护到 2028-02。参考：https://mariadb.org/about/#maintenance-policy

目标口径：

- 文档统一写 `MariaDB 10.11 LTS / MySQL-compatible`
- 不再笼统写 Oracle MySQL
- A1 目标是评估并保持 10.11 LTS 最新补丁版，不跨主版本升级

备份命令草案：

```bash
sudo mkdir -p /opt/elitesync/backups/db
sudo mariadb-dump --single-transaction --quick --routines --triggers --events --all-databases \
  | gzip > /opt/elitesync/backups/db/all-databases-$(date +%Y%m%d-%H%M%S).sql.gz
```

schema-only 导出草案：

```bash
sudo mariadb-dump --no-data --routines --triggers --events --all-databases \
  > /opt/elitesync/backups/db/schema-only-$(date +%Y%m%d-%H%M%S).sql
```

staging 升级步骤：

1. 从 production 只读导出备份与 schema-only。
2. 在 staging 恢复备份，校验表数、关键表行数、字符集、索引、触发器、事件。
3. 确认 staging apt 源可提供更高 10.11 LTS 补丁版；若不可用，不强行替换源。
4. 执行 staging 升级，运行 `mariadb-upgrade`，重启 MariaDB。
5. 运行 Laravel smoke：登录、profile、位置链路、关键 API、queue job、cache/session、慢查询采样。
6. 记录回滚耗时、恢复命令、校验结果。

production 升级前置条件：

- 已有 staging 成功升级证据。
- 已完成恢复演练，不只是备份文件存在。
- 已确认维护窗口、停写策略、回滚点、DB 用户权限、磁盘余量。
- 已确认 v1 继续对外提供服务，v2 不替换 production 默认入口。

回滚策略：

- 优先通过快照/整机备份回滚。
- 次级方案是停服务后用全量 dump 恢复。
- 如只升级补丁版且未做 migration，业务 schema 不应发生变化；禁止与 DB schema 改动合并执行。

## 5. Redis cache / queue / session / rate limit 规划

当前 Redis：

- 版本：Redis 7.0.15
- bind：`127.0.0.1 -::1`
- protected-mode：yes
- requirepass：empty
- maxmemory：0
- maxmemory-policy：noeviction
- appendonly：no
- save：`3600 1 300 100 60 10000`
- databases：16

Laravel 当前配置观察：

- `artisan about`：Cache=file, Queue=redis, Session=file
- `.env`：`CACHE_STORE=redis`, `QUEUE_CONNECTION=redis`, `SESSION_DRIVER=database`
- 由于 Config cached，当前运行配置和 `.env` 存在不一致，A1 进入 skeleton 前必须先明确 local/staging/production 配置来源与缓存刷新流程。

建议配置方向：

- cache：Redis 独立 key prefix，短 TTL 优先，禁止无边界大对象缓存。
- queue：Redis 独立 queue connection，区分 default/notifications/location/media/low_priority。
- session：商用生产建议 Redis 或 database 二选一，A1 需基于登录态一致性、运维可见性和故障恢复选型。
- rate limit：Redis 作为 Laravel rate limiter store，按 API、IP、用户、设备维度设计 key。
- Redis 生产建议设置合理 `maxmemory` 与淘汰策略；cache 可用 `allkeys-lru`，queue/session 不应和易淘汰 cache 混用同一逻辑库或实例。

队列分层建议：

- `critical`：账号、安全、支付或强一致任务。
- `default`：普通业务异步任务。
- `location`：位置链路清理/逆地理/附近推荐等任务。
- `notifications`：通知推送。
- `low_priority`：统计、清理、非关键同步。

failed jobs 与重试策略：

- 启用 failed job 持久化与告警。
- 每类 job 定义 timeout、tries、backoff。
- 对位置链路任务增加幂等键，避免重试造成重复写入。
- 队列 worker 应由 systemd 或 Supervisor 管理；当前 Supervisor inactive，未见 queue worker 常驻进程。

## 7. 队列 / 监控 / 备份 / 压测 / 分环境部署

queue worker 管理：

- 当前 Supervisor inactive。
- 未见 `artisan queue:work` / `horizon` 常驻进程。
- 建议 A1 采用 systemd 或 Supervisor。若继续保持轻量，systemd unit 更直接；若需要多队列、多进程组、动态扩缩，Supervisor 更易管理。
- Horizon 未安装，不能把 Horizon 当作当前前提。

health/readiness 指标：

- HTTP `/health` 或等价 endpoint：应用 boot、DB ping、Redis ping、queue connection、storage writable、版本信息。
- Readiness 需区分 deploy warming、maintenance、DB unavailable、Redis unavailable。

监控范围：

- DB：连接数、慢查询、锁等待、磁盘、备份成功率。
- Redis：内存、keyspace、eviction、blocked clients、persistence 状态。
- queue：队列长度、failed jobs、retry 次数、job latency。
- storage：磁盘余量、上传目录权限、日志增长。
- Nginx：4xx/5xx、upstream timeout、请求耗时。
- Laravel：error log、exception 分类、scheduler/queue worker 状态。

备份与恢复演练要求：

- A1 不只写备份命令，必须在 staging 完成恢复演练。
- 备份应包含 DB dump、schema-only、关键 storage、`.env`/Nginx/systemd 配置的脱敏归档。
- 恢复验收要包含关键 API smoke 与登录态验证。

staging 压测指标与工具建议：

- 工具：k6、wrk、ab 仅作补充；Laravel 队列压测需单独构造 job backlog。
- 指标：P50/P95/P99 latency、error rate、DB QPS/慢查询、Redis ops/memory、PHP-FPM busy workers、Nginx upstream timeout、queue latency。
- 压测前必须冻结 staging 数据集与环境变量，并声明不打 production。

local / staging / production 分离：

- local：开发与单机验证，允许 debug。
- staging：接近 production 的 Nginx/PHP-FPM/Redis/MariaDB/queue/scheduler/backup 配置，禁止真实用户写入。
- production：默认继续 v1；A1 不切换 release chain，不把 v2 skeleton 接入生产入口。
