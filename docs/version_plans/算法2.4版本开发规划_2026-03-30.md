# 算法 2.4 版本开发规划（可直接下发 Codex）

> 版本定位：在 2.3 已达到“有条件通过（Pass with fixes）”的基础上，将系统从“主体落地完成”推进到 **完全通过、准生产闭环、可稳定放量** 的版本。
>
> 2.4 的核心不是继续堆新算法，而是完成 2.3 尚未彻底闭环的部分：
> 1. 西占 canonical 路线从“策略化过渡”推进到“可执行决策”
> 2. 灰度/回滚从“文档+脚本”推进到“可复核演练证据”
> 3. explanation 从“模板可用”推进到“自动化回归门禁”
> 4. 仓库与兼容层从“可运行”推进到“可维护、可审计、可交接”

---

# 0. 2.4 的输入结论（基于 2.3 验收）

## 0.1 2.3 已完成的基础
- 显示保护层已建立：`engine_source` / `engine_mode` / `data_quality` / `precision_level` / `confidence` / `confidence_reason`
- 前后端展示一致性已基本打通
- explanation 模板系统已落地，具备回归样例与 diff 基础
- raw tag 已从用户端收口为 display tags
- 西占路线已形成 ADR 和过渡策略
- 低样本调权守门已建立
- 灰度/回滚文档与脚本已具备

## 0.2 2.3 未完全闭环的部分
1. **西占 canonical 仍处于受控过渡态**，尚未进入“高精+合规+稳定生产”闭环
2. **灰度/回滚证据不够强**，目前偏文档/脚本层，缺少更强的演练留痕
3. **explanation 自动化门禁仍可加强**，需要更严格的发布前阻断机制
4. **历史文档、兼容字段、旧路径残留仍较多**，影响长期维护与交接

## 0.3 2.4 的总目标
把系统推进到：
- **可正式签字通过**
- **可做更大流量灰度**
- **可控风险下逐步进入生产稳定态**

# 1. 2.4 总体目标与通过标准

## 1.1 总体目标
完成 2.3 的收尾与准生产闭环，不再让关键能力停留在“文档上已完成、工程上还在过渡”的状态。

## 1.2 2.4 的通过标准（必须全部满足）
1. **西占路线有明确执行结论**：要么拿到明确授权推进高精 production，要么正式固化为 `western-lite` 长期策略，不再摇摆
2. **灰度/回滚有可复核演练证据**：不仅有文档，还要有日志、截图、结果记录
3. **explanation 成为发布门禁的一部分**：关键回归样例失败时能阻断发布
4. **兼容层与历史残留清理完成一轮**：明确哪些保留、哪些废弃、哪些在 2.5 删除
5. **模块级可观测性提升**：能看到 badge 发放比例、fallback 占比、legacy 占比、低置信展示占比、解释点击率与转化关系

# 3. 主线 A：西占 canonical 执行化与策略收口

## 3.1 目标
结束“ADR 已有，但执行路线仍处于过渡态”的状态，让西占模块进入稳定策略。

## 3.2 2.4 必须做出的执行分叉

### 路线 A：拿到 Swiss 商业授权，推进高精 production
适用条件：
- 商业授权明确可用
- 法务/采购完成
- 服务端接入方案可执行

### 路线 B：继续闭源安全策略，正式固化 `western-lite`
适用条件：
- 2.4 周期内拿不到授权
- 不接受 AGPL 路线
- 优先保证闭源合规与上线稳定

> **2.4 要求：不能继续停留在“口头上两条都可能”**。必须形成当前版本的正式执行结论。

## 3.3 若走路线 A（已拿授权）
### 任务
1. 新建或完善 `WesternCanonicalService`
2. 明确接入高精引擎的运行边界：
   - natal
   - synastry
   - composite_like
3. 将 `western_policy.mode` 切到 `licensed_canonical`
4. 对接现有 `display_guard`
5. 运行 shadow compare：
   - old legacy
   - licensed canonical
6. 输出差异报告与 badge 变化报告

### 验收
- 高精服务端结果可独立运行
- API 契约不变
- 高精仅在合规前提下启用
- 解释层、badge、权重与新模式同步

## 3.4 若走路线 B（未拿授权）
### 任务
1. 正式把当前西占定位固化为：`western-lite`
2. 在 `western_policy.php` 中锁定：
   - 不进入高精 production
   - 不发高强度“高精星盘”类措辞
3. 进一步下调以下内容：
   - badge 强度
   - 模块措辞
   - ranking 权重上限
4. 把 Astronomy Engine 明确为：
   - 数值辅助验证
   - 非占星生产主引擎
5. 输出 `western_lite_contract.md`：定义当前版本承诺与不承诺的能力边界

### 验收
- 产品表达不再超出底层实际能力
- 西占维持轻量可控模块定位
- 为未来高精升级保留接口，不返工 API

## 3.5 需要新增的配置
- `config/western_policy.php`
  - `mode = licensed_canonical | western_lite`
  - `allow_high_precision_wording`
  - `max_rank_weight`
  - `max_display_priority`
- `config/display_guard.php`
  - 增加与 `western_policy.mode` 绑定规则

## 3.6 交付物
- `docs/western_execution_decision_2_4.md`
- `docs/western_lite_contract.md`（若走路线 B）
- `reports/western_shadow_compare_2_4.md`
- `reports/western_badge_weight_diff_2_4.md`

## 3.7 验收标准
- 2.4 必须给出当前版本的单一路线执行结论
- 西占模块的“实际引擎能力、前端措辞、badge 强度、权重上限”四者一致

# 4. 主线 B：灰度 / 回滚 / 发布演练证据化

## 4.1 目标
把“有文档、有脚本”推进到“真的演练过，并留下可复核证据”。

## 4.2 必做演练

### 演练 1：单模块关闭
模拟关闭：
- 八字模块
- 西占模块
- 性格模块

验证：
- 排序是否可继续运行
- explanation 是否优雅降级
- 前端是否出现空白或错位

### 演练 2：单 profile 回退
模拟：
- 新调权 profile 回退到稳定 profile

验证：
- `rank_score` 是否回退成功
- display_score 是否保持稳定
- 周报是否反映变更

### 演练 3：explanation 模板回退
模拟：
- 新模板上线后回退到旧模板集

验证：
- 解释快照是否恢复
- 前端摘要是否恢复

### 演练 4：western policy 回退
模拟：
- 从 `licensed_canonical` 回退到 `western_lite`
- 或从实验模式回退到稳定模式

验证：
- 前后端 badge 和措辞同步变化

## 4.3 证据化要求
每次演练都必须保存：
- 操作时间
- 操作人
- 环境
- 演练目标
- 前置状态
- 操作步骤
- 日志片段
- 截图
- 结果摘要
- 是否通过

- 任一关键问题都能在 30 分钟内切回稳定配置

# 5. 主线 C：Explanation 自动化门禁与样例库升级

## 5.1 目标
让 explanation 不只是“能回归”，而是“失败就阻断发布”。

## 5.2 2.4 必做升级

### 升级 1：样例库从 20 组扩到 40 组
新增覆盖：
- 低置信但高分
- 高置信但高风险
- 多模块冲突
- 西占降级 + 八字高分
- 八字低置信 + 性格高分
- fallback 模块混合出现
- 男女合盘高分但建议保守
- 城市/年龄/沟通风格软调节影响结果

### 升级 2：解释门禁分级
- `P0`：结构错误、字段缺失、空 explanation -> 阻断发布
- `P1`：高置信低依据、低置信高口气 -> 阻断发布
- `P2`：语义不自然、标签不够优雅 -> 可带修复项通过

### 升级 3：发布前自动检查
新增 CI / 脚本：
- `run_explanation_regression`
- `check_explanation_guardrail`
- `check_display_guard_consistency`

### 升级 4：黄金样例机制
指定 10 组黄金样例，要求：
- 版本升级后仅允许可解释的微调
- 不允许无说明大幅漂移

## 5.3 需要统一检查的问题
1. 结论层是否过重？
2. 风险层是否缺失？
3. 建议层是否流于空话？
4. 低置信时是否仍使用“稳定、确定、明显”等词？
5. 男女合盘是否仍把“关系气质”说成“长期稳定判决”？

## 5.4 新增产物
- `tests/Fixtures/explanations_v2/*.json`
- `tests/Fixtures/explanations_gold/*.json`
- `docs/explanation_release_gate_2_4.md`

## 5.5 验收标准
- explanation 成为发布门禁的一部分
- 黄金样例在升级后保持可解释稳定
- 低置信结果不会再出现高口气解释
