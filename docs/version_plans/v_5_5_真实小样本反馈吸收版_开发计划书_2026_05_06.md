## 1. 版本定位

### 1.1 5.5 是什么

`5.5` 是 **真实小样本反馈吸收版**。

它承接 `5.4` 的测试运营准备与只读治理增强结果，不再把重点放在“继续补治理入口”或“继续补 Soul 对应功能覆盖”，而是把当前已经建立的主链、治理入口、runbook、回归清单和证据体系真正用于 **受控小样本测试、问题归因、节奏校正和产品修正**。

一句话定位：

> **5.5 要把 EliteSync 从“已经可被测试、可被观测”推进到“已经开始吸收真实反馈，并用最小改动修正关键体验问题”的阶段。**

### 1.2 5.5 不是什么

`5.5` 不是：

- 不是重开 `5.0–5.3` 的功能覆盖主线；
- 不是继续扩一个更重的运营后台；
- 不是继续堆更多只读治理卡片；
- 不是多人 RTC / 直播 / 房间系统；
- 不是币体系、商城、礼物、会员、重商业化；
- 不是完整推荐算法重写；
- 不是大规模用户测试；
- 不是在本地直接执行生产数据库写入；
- 不是默认发版升级任务，除非用户另行明确要求。

### 1.3 版本验收目标

5.5 完成后，建议验收口径目标为：

> `pass with observations`

原因是 5.5 的本质不是“纯实现版”，而是“吸收真实反馈 + 校正产品结构 + 加厚失败路径”的综合版。它天然会保留少量 observation，但只要：

- 真实小样本反馈有记录；
- 问题归因不是拍脑袋；
- 调整范围克制且对症；
- 保护面未被破坏；
- 失败路径更清楚；
- 交接材料和证据完整；

就可以正式作为 5.x 第一轮功能覆盖后的反馈修正版收口。

## 3. 新版本启动流程要求

本轮是 **新版本开发启动流程**，不是提交 / 推送前流程。

Codex 必须先调用 / 遵守：

- `elitesync-version-start`

并完成只读审计：

```powershell
git branch --show-current
git rev-parse HEAD
git status --short
git diff --stat
```

启动阶段只做：

- 读取计划；
- 确认 5.4 已完成什么；
- 确认真实小样本反馈的来源与可信度；
- 确认本轮拟修正的问题清单；
- 确认保护面；
- 确认非目标；
- 输出 5.5 执行拆解。

启动阶段不要自动进入：

- A/B/C/D 提交分桶；
- staged 清单；
- commit message；
- push 讨论。

只有用户明确要求“准备提交 / push / 整理未提交内容”时，才启用 `elitesync-dirty-worktree` 和固定提交流程。

## 4. 5.5 总目标

5.5 的总目标是建立 **真实小样本反馈吸收最小闭环**：

1. **反馈采集闭环**  
   把真实小样本测试中的体验问题、失败路径、误解点和 friction 记录成可追踪材料，而不是口头印象。

2. **问题归因闭环**  
   区分：是 UI 表达问题、流程节奏问题、本地 explanation surface 问题、回流策略问题，还是跨层 blocker / 真链路问题。

3. **最小修正闭环**  
   只修真实反馈里最影响测试质量和用户理解的高价值问题，不重开大版本主链。

4. **失败路径加厚闭环**  
   对通知、媒体、RTC、回聊、AI 草稿、同城、推荐理由等最容易产生误解的失败态或弱成功态做更清晰表达。

5. **测试记录与版本收口闭环**  
   让 5.5 最终留下：反馈索引、问题分级、改动说明、回归证据、handoff 主文件，便于后续继续测试或准备更正式的 Beta。

## 5. 范围与非目标

### 5.1 本轮允许做

#### A. 小样本反馈采集与问题分级

允许做：

- 小样本 walkthrough 记录；
- 反馈分类：Discover / Chat / Me / Settings / Notification / Media / RTC / Version / Governance；
- 问题分级：阻断 / 高摩擦 / 中摩擦 / observation；
- 反馈与截图、日志、页面、路径绑定；
- 当前版本问题矩阵与优先级表。

#### B. 首聊 / 回聊 / AI 建议节奏修正

允许做：

- 首聊建议文案密度调整；
- 回聊 / 稍后再聊入口表达调整；
- AI 建议位置、数量、默认态修正；
- 低压建议文案与按钮顺序优化；
- 关系摘要内容轻量重排；
- 冷场恢复提示更自然。

#### C. Discover / 同城 / 推荐理由修正

允许做：

- Discover 分栏默认态调整；
- 搜索热词 / 搜索占位文案优化；
- 同城理由、推荐理由文案与信息密度调整；
- 内容卡低压私聊入口可理解性修正；
- 同城 / 推荐 / 活动 / 话题入口的轻量排序修正。

#### D. Me / 标签 / 个人经营区 / 外观层修正

允许做：

- 标签表达可理解性优化；
- 个人经营区说明重写；
- AI 助理 / AI 草稿助手的文案和入口顺序微调；
- 展示标识、外观层、Appearance 边界表达优化；
- 资料完成度、首聊友好度、经营反馈的轻量重排。

#### E. 通知 / 媒体 / RTC 失败路径加厚

允许做：

- 通知回流按钮命名与默认动作优化；
- 通知 payload 对应页面说明加厚；
- 媒体失败重试 / 空态 / content 不可用提示优化；
- RTC 权限页、邀请页、未接通 / 已接听 / 稍后处理路径表达优化；
- chat room 中弱成功态 / 空态更清晰。

#### F. 文档、索引与收口

允许新增：

- `docs/version_plans/5.5_HANDOFF_MASTER.md`
- `docs/version_plans/5.5_REGRESSION_CHECKLIST.md`
- `docs/version_plans/5.5_FEEDBACK_EVIDENCE_INDEX.md`
- `docs/version_plans/5.5_FEEDBACK_MATRIX.md`

如果内容较短，应优先并入 `5.5_HANDOFF_MASTER.md`，不要机械拆文件。

### 5.2 本轮明确不做

- 不重开 `5.0–5.3` 的覆盖主链；
- 不继续扩重型治理后台；
- 不默认新增新的大功能域；
- 不把 `AI 续话` 升级成服务端自动生成 / 自动发送系统；
- 不把 `稍后再聊` 升级成复杂持久化回聊平台，除非用户明确要求且边界清晰；
- 不把 `个人空间外观` 升级成商业化装扮中心；
- 不做完整推荐算法重构；
- 不做多人 RTC / 直播 / 房间；
- 不在本地执行生产数据库写入；
- 不为了吸收反馈而顺手回退现代 UI；
- 不把单个小样本反馈误当成全局产品结论。

## 6.3 5.5C：Chat 节奏与 AI 建议吸收修正

### 目标

根据真实测试反馈，修正首聊 / 回聊 / AI 建议 / 关系摘要的节奏和可理解性。

### 任务

1. 首聊 / 回聊修正：
   - 首聊建议是否太满；
   - 回聊建议是否太早出现；
   - `稍后再聊` / `回聊` / `通知回流` 的顺序是否自然；
   - 会话列表 guidance 是否抢主内容。

2. AI 建议修正：
   - 建议数量；
   - 默认展示密度；
   - AI 按钮位置；
   - 是否过于像自动发送；
   - 是否需要更清晰的“本地草稿帮助”边界。

3. 关系摘要修正：
   - 信息密度；
   - 共性信息的可读性；
   - 通话节奏提示是否自然；
   - 是否与查看主页、聊天设置冲突。

### 验收标准

- 更自然，不更重；
- 更清楚，不更花哨；
- 不把本地建议伪装成 AI 真生成；
- 不改变消息、通知、RTC 的主 contract。

## 6.4 5.5D：Discover / 同城 / 推荐理由吸收修正

### 目标

根据真实小样本反馈，修正 Discover 页最影响理解和转化的表达问题。

### 任务

1. 分栏修正：
   - 推荐 / 同城 / 活动 / 话题默认顺序是否自然；
   - tab 文案是否清楚；
   - 是否存在过度平铺导致的认知噪音。

2. 搜索修正：
   - 搜索占位词；
   - 热词密度；
   - 搜索结果预期；
   - 搜索与 Discover 的边界。

3. 推荐理由 / 同城理由修正：
   - 理由是否过短；
   - 理由是否像空洞营销词；
   - 是否需要把“值得聊理由”说得更慢约会化；
   - 同城内容是否太即时社交化、与产品调性冲突。

4. 内容到私聊的低压转化：
   - CTA 命名是否自然；
   - 是否太推；
   - 是否需要保留更多观察空间。

### 验收标准

- 信息更清楚；
- Discover 不被改成噪音 feed；
- 不变成重社区平台；
- 不强推定位授权；
- 不重写推荐算法。

## 6.5 5.5E：Me / 标签 / 个人经营区吸收修正

### 目标

根据真实测试反馈，修正个人页中最容易引发误解、压迫感或无感的表达层。

### 任务

1. 标签体系修正：
   - 是否太像静态字段；
   - 分组是否自然；
   - 是否能帮助对方理解自己；
   - 是否过于“分类表”。

2. 个人经营区修正：
   - 是否易懂；
   - 是否太工程化；
   - 资料完整度与目标表达是否清楚；
   - AI 助理与经营区是否抢焦点。

3. AI 助理 / AI 草稿助手修正：
   - 是否有帮助；
   - 是否太“像工具”；
   - 是否需要降低存在感；
   - 是否需要更清晰的边界说明。

4. 外观层 / 展示标识修正：
   - 是否容易理解为正式装扮系统；
   - Appearance 与 Settings 边界是否仍清楚；
   - 外观层是否影响资料真值理解。

### 验收标准

- 个人页更像“被理解与被继续了解”的页；
- 不更像营销页；
- 不更像工具箱；
- 不更像商城前台；
- 不回写资料真值。

## 9. 跨层 blocker 条款

5.5 涉及通知、媒体、RTC、日志、反馈归因，因此必须严格遵守跨层 blocker 规则。

一旦出现以下情况：

- 页面反馈和日志不一致；
- 用户说“不能用”，但页面看似成功；
- 通知打开路径与 payload 不一致；
- 媒体页面提示正常，但 content 实际失败；
- RTC 权限、邀请、回流、接通表现不一致；
- Codex 连续两轮无法缩小问题层级；

必须立即停止盲改，进入：

```text
elitesync-cross-layer-blocker
```

并产出 blocker report。

### blocker report 最小格式

```md
# 5.5 <问题名> Blocker Report

## 1. Symptom
- 当前现象：

## 2. Suspected Layer
- 前端 / 后端 / 云端 / 数据库 / 队列 / RTC / 媒体 / 构建环境：

## 3. Affected Surfaces
- 受影响页面 / 接口 / 表 / 日志：

## 4. Established Chain
- 已确认成立的链路：

## 5. Broken Chain
- 尚未成立的链路：

## 6. Tried Changes
- 已尝试：

## 7. Excluded Hypotheses
- 已排除：

## 8. Smallest Safe Next Step
- 下一步最小安全动作：
```

没有 blocker report，不继续跨层盲修。

## 10. Rollback / Recovery Policy 条款

5.5 的恢复规则：

- 禁止 repo 级 restore / reset / clean；

## 17. 最终验收口径

### 17.1 允许 pass with observations 的情况

满足以下条件时，可以建议 `pass with observations`：

- 真实小样本反馈已经形成可追踪材料；
- 高优先级反馈已经吸收进最小修正；
- 问题归因基本成立；
- 通知 / 媒体 / RTC / 回聊 / AI 建议等高摩擦点更清楚；
- 5.0–5.4 产品和治理主链未被破坏；
- evidence index 与截图 / 日志对应；
- handoff master 成为唯一主入口；
- observation 都是非阻断项。

### 17.2 只能 conditional pass 的情况

出现以下情况，只能建议 `conditional pass`：

- 代码基本完成但没有反馈矩阵；
- 只改了页面，但没有说明为什么改；
- 反馈来自口头印象，没有证据绑定；
- 失败路径说变清楚了，但无截图 / 日志证据；
- handoff master 未形成；
- regression checklist 不完整。

### 17.3 必须阻断的情况

出现以下情况必须阻断：

- 为了吸收反馈误改 profile truth chain；
- 误改 messages / media / rtc 主链 contract；
- 破坏 5.4 治理入口的只读边界；
- 回退现代 UI；
- 伪造日志、伪造成功态、伪造真实反馈；
- 跨层 blocker 无 report 继续盲修；
- 把单个反馈过度放大成大重构依据。

## 19. 顾问结论

我建议 5.5 按 **“真实反馈矩阵 + 最小修正 + 失败路径加厚 + 单文件收口”** 来执行。

不要让 Codex 把 5.5 写成大而全的“第二轮产品重构版”，也不要把 5.5 错写成“继续做治理后台版”。

5.5 的成功标准是：后续你已经可以更有把握地回答这些问题：

- 真实测试时，用户到底卡在哪；
- 哪些问题只是文案和节奏问题；
- 哪些问题其实是通知 / 媒体 / RTC 的失败路径不清；
- 哪些问题需要后移，不该现在重开主链；
- 当前版本调整后，首聊 / 回聊 / Discover / 个人经营区是不是更自然；
- 新的修正有没有破坏 5.0–5.4 已建立的基线。

如果这些成立，5.5 就可以按 `pass with observations` 收口。
