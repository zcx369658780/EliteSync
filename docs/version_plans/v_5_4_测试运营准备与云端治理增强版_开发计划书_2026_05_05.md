## 1. 版本定位

### 1.1 5.4 是什么

`5.4` 是 **测试运营准备与云端治理增强版**。

它的核心目标不是继续补 Soul 高价值主链 UI 入口，而是在 `5.0–5.3` 已完成主要产品覆盖之后，把项目推进到更适合后续小样本测试、云端运维、回归复盘、证据追踪和安全治理的状态。

一句话定位：

> **5.4 要把 EliteSync 从“产品主链基本覆盖”推进到“可被稳定测试、可被观测、可被回滚、可被审计、可被小规模运营准备”的阶段。**

### 1.2 5.4 不是什么

`5.4` 不是：

* 不是重开 5.0–5.3 的 Discover / Chat / Me / Settings 产品入口补齐主线；
* 不是继续照搬 Soul 的商城、币体系、会员、礼物、直播、多人房间；
* 不是完整商业化运营后台；
* 不是重型中台；
* 不是微服务拆分；
* 不是多人 RTC / 直播 / 房间系统；
* 不是在本地直接执行生产数据库写入；
* 不是发版版本号升级任务，除非用户另行明确要求。

### 1.3 版本验收目标

5.4 完成后，建议验收口径目标为：

> `pass with observations`

原因是 5.4 涉及云端治理、观测、备份恢复、测试账号、后台最小集等多个跨层点，合理预期会保留若干非阻断 observation；但只要核心治理链路、只读口径、runbook、截图/日志证据和回归保护面成立，就可以正式进入 5.5 或后续真实小样本反馈吸收阶段。

## 3. 新版本启动流程要求

本轮是 **新版本开发启动流程**，不是提交 / 推送前流程。

Codex 必须先调用 / 遵守：

* `elitesync-version-start`

并完成只读审计：

```powershell
git branch --show-current
git rev-parse HEAD
git status --short
git diff --stat
```

启动阶段只做：

* 读取计划；
* 确认基线；
* 确认保护面；
* 确认非目标；
* 确认拟修改范围；
* 输出 5.4 执行拆解。

启动阶段不要自动进入：

* A/B/C/D 提交分桶；
* staged 清单；
* commit message；
* push 讨论。

只有用户明确要求“准备提交 / push / 整理未提交内容”时，才启用 `elitesync-dirty-worktree` 和固定提交流程。

## 4. 5.4 总目标

5.4 的总目标是建立 **测试运营准备最小闭环**：

1. **业务管理后台最小集**
   提供只读优先的运营 / 调试 / 治理入口，能看关键状态，不做重型中台。

2. **云端只读优先 DB 访问与审计**
   明确哪些数据可以只读查看，哪些操作禁止本地执行，哪些远端操作必须留痕。

3. **统一观测入口**
   把版本、健康检查、通知、媒体、队列、RTC、搜索 / Discover、关系推进相关状态整理成最小可读面。

4. **备份恢复与迁移状态 runbook 固化**
   让后续测试前能明确知道：怎么备份、怎么核验、怎么恢复、怎么避免误写生产库。

5. **测试账号与 synthetic 账号治理**
   固化 smoke / regression 使用账号、测试数据状态、账号恢复策略和最小治理说明。

6. **5.4 证据与 handoff 收口**
   用一个主交接文件 + 必要索引 / checklist / runbook 完成收口，避免文件爆炸。

## 6. 工作包拆分

## 6.1 5.4A：启动审计与治理边界冻结

### 目标

确认 5.4 从干净基线启动，冻结治理范围，避免一上来就误改产品主链或生产后端。

### 任务

1. 执行只读 Git 审计：

   * 当前分支；
   * 当前 HEAD；
   * `git status --short`；
   * `git diff --stat`。

2. 读取并摘录当前基线：

   * 5.3 主交接；
   * 5.3 regression checklist；
   * 5.3 evidence index；
   * project memory；
   * development plan current。

3. 输出 5.4 边界确认：

   * 本轮会碰哪些目录；
   * 本轮不会碰哪些目录；
   * 是否涉及后端；
   * 是否涉及数据库；
   * 是否涉及远端云端；
   * 是否涉及发版链；
   * 是否涉及 UI protected surfaces。

4. 确认 5.4 的默认开发方式：

   * 本地只做前端开发、UI 联调、文档整理、只读检查脚本；
   * 后端开发、数据库迁移、备份恢复、写库排障统一在阿里云端执行；
   * 任何生产写库前必须等待用户明确确认。

### 产物

* 5.4 启动审计摘要；
* 5.4 范围与非目标确认；
* 5.4 第一轮风险清单。

## 6.2 5.4B：业务管理后台最小集 / 治理概览入口

### 目标

建立最小治理入口，让后续测试时能快速查看关键状态。

### 建议实现形态

优先选择 **轻量、只读、可回退** 的形态：

* 如果仓库已有管理员页面 / 设置中心入口：在现有入口下新增“测试运营准备 / 治理概览”只读卡片；
* 如果没有成熟后台：先做 Flutter 内部只读页面或 Laravel 只读 endpoint + 文档 runbook；
* 不强行新建完整 Web admin 系统。

### 页面 / 功能建议

治理概览最小应包含：

1. 服务状态：

   * API health；
   * app version check；
   * 当前环境说明；
   * 后端时间；
   * 最近一次检查时间。

2. 发布状态：

   * 当前宿主版本；
   * 当前 version check 结果；
   * 下载包配置状态；
   * 版本中心一致性提示。

3. 测试账号状态：

   * smoke 账号是否存在；
   * 是否可登录；
   * 资料是否完整；
   * 是否有当前匹配 / 消息 / 通知基础数据。

4. 关键链路状态：

   * messages；
   * notifications；
   * media；
   * rtc；
   * status / discover；
   * profile truth chain。

5. 安全边界说明：

   * 当前页面只读；
   * 不提供生产写库按钮；
   * 不提供删除、封禁、批量修改入口；
   * 高风险操作只能走 runbook + 用户确认。

### 验收标准

* 治理入口可访问；
* 页面不会像 debug 面板，文案必须用户 / 运营可读；
* 不出现 `mock success`、`fake ok`、`debug only` 等会误导用户的主文案；
* 所有状态必须来自真实 endpoint、真实配置、真实日志或明确标注为“未接入 / 待接入”；
* 不允许用假数据伪装成功。

## 6.3 5.4C：统一观测入口与状态摘要

### 目标

把测试运营前最容易用到的状态检查统一整理，减少后续排障时在不同文件、接口、命令之间来回跳。

### 重点观测域

#### A. Version / Release

* `apps/android/app/build.gradle.kts`
* `apps/android/app/src/main/assets/changelog_v0.txt`
* `apps/flutter_elitesync_module/assets/config/about_update_0_xx.json`
* `services/backend-laravel/config/app_update.php`
* `GET /api/v1/app/version/check`
* `scripts/release_android_update_aliyun.ps1`

验收目标：

* 证明当前版本链仍未被 5.4 破坏；
* 如果不发版，则明确说明 5.4 不改 `0.04.09 / 40900` 发布基线；
* 版本中心截图必须来自当前安装包。

#### B. Health / Logs

* `GET /api/v1/app/health`
* Laravel 日志位置；
* 关键错误摘要；
* API 可用性；
* 远端服务状态。

验收目标：

* 能从 runbook 或页面知道服务是否健康；
* 异常时能知道下一步查哪份日志；
* 不把网络错误伪装成业务成功。

#### C. Notification / Returnflow

* 通知列表；
* 已读 / 未读；
* 回流目标；
* payload 类型；
* conversation_id / peer_user_id / status_post_id / rtc_call_id 等跳转字段。

验收目标：

* 至少形成真实 payload 跳转复核清单；
* 明确哪些 payload 已验证、哪些仍是 observation；
* 不重开 5.1 通知主链，只做测试准备与证据加厚。

#### D. Media / Queue / Worker

* media_assets；
* media_processing_jobs；
* 图片 / 视频 public_url；
* 队列 worker 状态；
* 失败重试状态。

验收目标：

* 能检查媒体链当前健康状态；
* 能识别 public_url 是否误写 localhost；
* 能知道 worker 是否正常。

#### E. RTC / LiveKit / Heartbeat

* rtc_calls；
* rtc_sessions；
* heartbeat；
* join-info；
* LiveKit 服务状态；
* 断连结束机制。

验收目标：

* 不重写 RTC；
* 只补观测与 runbook；
* 能确认当前 1v1 真语音主链未被破坏；
* 若出现跨层问题，进入 blocker report。

#### F. Discovery / Search / Relationship Progress

* discover feed；
* search terms；
* relationship_progress；
* status_feed；
* content-to-chat 转化状态。

验收目标：

* 不重做推荐算法；
* 只做状态摘要、缓存 / 数据存在性检查、测试可观测性；
* 记录哪些能力仍是本地 explanation surface，哪些已有服务端真值。

### 产物

* 统一观测清单；
* 5.4 observability evidence index；
* 如有页面，则补截图和 XML；
* 如有命令，则补 runbook 输出样例。

## 6.5 5.4E：备份恢复、迁移状态、关键表比对 runbook

### 目标

把后续测试前最关键的恢复能力制度化。

### 任务

1. 固化备份前检查：

   * 当前远端环境；
   * 当前数据库；
   * 当前连接账号；
   * 当前备份目录；
   * 当前时间戳；
   * 当前版本基线。

2. 固化备份命名规范：

```text
backup_elitesync_<env>_<YYYYMMDD_HHMMSS>_<reason>.sql
```

3. 固化恢复前检查：

   * 是否为生产库；
   * 是否已有新备份；
   * 是否已确认目标；
   * 是否有用户明确确认；
   * 是否仅恢复测试账号 / 测试数据；
   * 是否会覆盖真实用户数据。

4. 固化关键表比对清单：

   * users；
   * profile；
   * astro；
   * messages；
   * notifications；
   * media；
   * status；
   * rtc；
   * release versions。

5. 固化迁移状态检查：

   * Laravel migration status；
   * 最近迁移；
   * 是否存在 pending migration；
   * 是否存在本地迁移未上云；
   * 是否存在云端 schema 与代码不一致。

### 产物

建议文件：

`docs/runbooks/BACKUP_RESTORE_AND_MIGRATION_CHECK.md`

## 9. 跨层 blocker 条款

5.4 涉及后端、云端、数据库、队列、日志、RTC、媒体、版本检查，因此必须严格遵守跨层 blocker 规则。

一旦出现以下情况：

* 页面显示健康，但接口 / 日志失败；
* 后端日志成功，但用户面不可用；
* 媒体 / RTC / 通知状态不一致；
* 云端端口、权限、容器、队列、worker 不清楚；
* Gradle / SDK / wrapper / 构建环境失败；
* Codex 连续两轮无法缩小问题层级；

必须立即停止盲改，进入：

```text
elitesync-cross-layer-blocker
```

并产出 blocker report。

### blocker report 最小格式

```md
# 5.4 <问题名> Blocker Report

## 1. Symptom
- 当前现象：

## 2. Suspected Layer
- 前端 / 后端 / 云端 / 数据库 / 队列 / RTC / 媒体 / 构建环境：

## 3. Affected Surfaces
- 受影响页面 / 接口 / 表 / 日志：

## 4. Established Chain
- 已确认成立的链路：

## 5. Broken Chain
- 尚未成立的链路：

## 6. Tried Changes
- 已尝试：

## 7. Excluded Hypotheses
- 已排除：

## 8. Smallest Safe Next Step
- 下一步最小安全动作：
```

没有 blocker report，不继续跨层盲修。

## 10. Rollback / Recovery Policy 条款

5.4 的恢复规则：

* 禁止 repo 级 restore / reset / clean；
* 禁止为了省事恢复旧 commit 覆盖现代 UI；

## 17. 最终验收口径

### 17.1 允许 pass with observations 的情况

满足以下条件时，可以建议 `pass with observations`：

* 治理入口 / runbook / 观测链路成立；
* 只读 DB 与审计规则明确；
* 备份恢复 runbook 成立；
* synthetic 账号治理成立；
* 5.0–5.3 产品主链未被破坏；
* 4.x 保护面未被破坏；
* evidence index 与截图 / 日志对应；
* handoff master 成为唯一主入口；
* observation 都是非阻断项。

### 17.2 只能 conditional pass 的情况

出现以下情况，只能建议 `conditional pass`：

* 代码基本完成但缺截图；
* runbook 只有原则，没有可执行步骤；
* 治理入口显示假状态；
* 只读 / 写库边界不清；
* 版本中心或 health 证据缺失；
* 回归 checklist 不完整；
* handoff master 未形成。

### 17.3 必须阻断的情况

出现以下情况必须阻断：

* 本地误写生产库；
* 未经确认执行生产恢复；
* 破坏 profile truth chain；
* 破坏 message / media / rtc 主链；
* 回退现代 UI；
* 版本链不一致且没有解释；
* 伪造健康 / 伪造成功态；
* 跨层 blocker 无 report 继续盲修。

## 19. 顾问结论

我建议 5.4 按 **“小后台 + 只读治理 + 统一观测 + runbook + synthetic 账号治理 + 单文件收口”** 来执行。

不要让 Codex 把 5.4 扩成完整运营平台，也不要回头重做 5.3 产品层。5.4 的成功标准是：后续你开始真实小样本测试前，能稳定回答这些问题：

* 当前服务健康吗？
* 当前版本链一致吗？
* 测试账号能用吗？
* 通知 / 媒体 / RTC / 队列状态能查吗？
* 数据库能只读核验吗？
* 出问题时知道查哪份日志吗？
* 恢复前知道怎么备份和比对吗？
* 新增治理能力有没有破坏 5.0–5.3 的产品主链？

如果这些成立，5.4 就可以按 `pass with observations` 收口。
