## 0. 3.6 执行摘要

3.6 的目标不是把星盘引擎一口气推到推运版，也不是把 3.5 的工作台 MVP 重新推翻，而是：

> **在不破坏 3.5 稳定基线与 canonical 真值边界的前提下，把 Astro Engine 从“参数化基础版”推进到“古典 / 现代双路线版”，形成可切换、可解释、可比较、可回归的路线能力。**

本轮已完成的前置判断：

- `route_mode` 作为路线标签是可接受的 additive 扩展
- `engine_info` / `metadata` 可以继续做兼容扩展
- 风险最高的点仍然是 canonical 真值边界、旧缓存兼容与路线解释层误写真值
- `Gemini-MCP` 继续承担 UI / 信息架构 / 阶段验收
- `Claude-mcp` 继续承担 schema / 路线边界 / 兼容性审查，并且必须留痕

---

## 1. 3.6 的计划结论

### 1.1 本版必须完成

1. 建立 `route_mode` 及相关路线标识。
2. 让 `standard / classical / modern` 三条路线可追踪、可切换、可回退。
3. 扩展 `engine_info` / `metadata`，但不破坏旧字段读取。
4. 将双路线工作台做成可用的 UI 入口。
5. 增强盘面联动与解释层，让路线差异“看得懂”。
6. 扩展样例对照、差异矩阵、已知偏差归档。
7. 完成 smoke / regression / rollback / 截图 / 归档材料。

### 1.2 本版明确不做

- 合盘、男女对比盘
- 行运、三限、次限、月返、日返
- 法达、小限、大限、流年
- 全量高级解释系统
- canonical 引擎全面替换
- 登录 / 会话 / 地理编码 / 数据库大结构重构

## 2. 3.6 P0 / P1 / 暂缓项

### 2.1 P0

| 项目 | 目标 | 说明 |
|---|---|---|
| `route_mode` | 建立路线骨架 | 至少覆盖 `standard / classical / modern` |
| `engine_info` / `metadata` | 路线扩展 | 保持 additive，保留旧字段可读 |
| 路线默认模板 | 三路线可切换 | 明确黄道制、宫位制、相位、容许度、点位范围 |
| 路线工作台 | 可用入口 | 路线切换不破坏盘面与摘要 |
| 路线摘要 | 可读解释 | 展示路线差异与边界提示 |
| 多样例对照 | 可回归 | 输出能力矩阵 / 差异矩阵 / 已知偏差 |
| 审查留痕 | 可归档 | Claude / Gemini 都要留痕 |

### 2.2 P1

| 项目 | 目标 | 说明 |
|---|---|---|
| 更多宫位制 | 扩充路线模板 | 先做常用集合，不做全量开放 |
| 更丰富相位 / 点位 | 提升解释能力 | 只纳入明确列入路线设计的集合 |
| 路线比较页 | 增强对照 | 将摘要页升级为比较页 |
| 更细解释块 | 提升可读性 | 仍需保持信息层级克制 |
| 大屏工作台预演 | 为后续铺垫 | 不影响当前移动端主线 |

### 2.3 暂缓项

| 项目 | 说明 |
|---|---|
| 推运版 | 留给 3.7 |
| 全量高级解释 | 不在 3.6 主线 |
| canonical 大改 | 绝不进入 3.6 |
