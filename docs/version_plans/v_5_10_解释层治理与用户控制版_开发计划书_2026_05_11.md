# 2. 版本定位

## 2.1 本版本是什么

5.10 是解释层治理与用户控制的最小产品化版本。

它不是继续新增一个解释能力，而是为 5.7-5.9 已落地的三类解释层提供统一用户说明与控制入口：

```text
Match：为什么值得聊
Me / Profile：我的慢约会表达建议
Chat：低压开场建议
Settings：解释层说明、开关、免责声明和用户控制
```

5.10 只做最小可验证闭环：

- 在 Settings 或合适的用户控制入口中新增 `解释与建议设置`；
- 明确三类解释层是什么；
- 明确它们不会写入资料、不会改变匹配算法、不会自动发送消息；
- 给出本地 UI 开关 / 占位开关；
- 开关首版可为本地 presentation state 或说明型入口，不新增 API / DB；
- 增加统一免责声明；
- 保持原 Settings / Profile / Match / Chat 主流程不被替代。

## 2.2 本版本不是什么

5.10 不是：

- 后台治理平台；
- 权限系统重构；
- 真实用户偏好服务端持久化；
- 新数据库设置表；
- 新 API contract；
- AI 安全系统；
- 付费报告开关；
- 真人咨询开关；
- 隐私中心重构；
- Settings 大重构；
- release / APK 发布任务。

## 2.3 本版本验收口径

5.10 可以验收为：

```text
pass with observations
```

前提是：

- Settings 或用户控制入口中新增解释层治理 / 用户控制区；
- 能解释 Match / Me / Chat 三类解释层；
- 有清晰开关或开关占位；
- 有统一 display-only / no-write / no-auto-send 免责声明；
- 未新增 API / DB / release / backend；
- 未改变 Match / Profile / Chat 真值链；
- 原 Settings 主流程不被破坏；
- Match / Me / Chat / Discover / Version Center protected surfaces 未被误伤；
- 有 widget / page test 或等效验证；
- 有截图 / XML 或可替代 UI 证据；
- 有回归清单与 5.10 handoff master；
- observations 不阻断 5.6+ 玄学解释层阶段收口。

# 3. 产品目标

## 3.1 用户目标

5.10 要帮助用户解决四个问题：

1. **这些解释和建议是什么？**
   - 让用户知道 Match、Profile、Chat 中出现的解释层只是辅助理解。

2. **它们会不会改变我的资料或匹配结果？**
   - 明确不会写入资料、不会改变星盘、不会改变匹配算法。

3. **它们会不会自动发消息？**
   - 明确不会自动发送消息，Chat 草稿仍需用户自己确认。

4. **我能不能关闭或减少它们？**
   - 提供本地控制入口或说明型开关，为后续持久化控制预留产品位置。

## 3.2 产品目标

5.10 要完成从“解释层分散落地”到“解释层可治理、可解释、可控制”的最小升级。

它应增强用户信任，避免解释层膨胀成伪真值、伪诊断、伪 AI 决策。

建议产品结构：

```text
Settings / 用户控制入口
  ├─ 原有设置分组
  ├─ 新增：解释与建议设置
  │   ├─ 关系解释提示
  │   ├─ 个人表达建议
  │   ├─ 聊天开场建议
  │   ├─ 统一免责声明
  │   ├─ 本地开关 / 占位开关
  │   └─ 了解这些建议如何工作
  └─ 原有账号 / 隐私 / 版本中心入口
```

## 3.3 设计语言

5.10 应使用 EliteSync 自有语言，强调用户控制、非强制、非真值写入。

推荐用词：

- `解释与建议设置`
- `关系解释提示`
- `个人表达建议`
- `聊天开场建议`
- `辅助理解，不替你决定`
- `不会写入资料`
- `不会改变匹配算法`
- `不会自动发送消息`
- `关闭后仍可使用主流程`
- `仅在本机展示中生效`

避免用词：

- `命运判断`
- `系统判定`
- `AI 决策`
- `算法已确认`
- `关闭会降低匹配率`
- `付费解锁`
- `真人咨询`
- `强制开启`
- `隐私扫描`
- `聊天监控`

# 4. 功能范围

## 4.1 必做功能

5.10 必须完成以下最小功能：

### 4.1.1 解释与建议设置入口

在 Settings 或项目当前最合适的设置 / 个人控制页中新增入口。

建议标题：

```text
解释与建议设置
```

建议副标题：

```text
管理 Match、个人页和聊天中的辅助解释与开场建议。
```

如果当前 Settings 结构不适合新增独立页，可以先在 Settings 页新增一张说明卡，并通过 dialog / bottom sheet 展示详细说明。

### 4.1.2 三类解释层说明

必须说明三类能力：

1. `关系解释提示`
   - 对应 5.7 Match `为什么值得聊`。
   - 说明：帮助理解推荐理由，不改变匹配算法。

2. `个人表达建议`
   - 对应 5.8 Profile `我的慢约会表达建议`。
   - 说明：帮助整理自我表达，不会自动修改资料。

3. `聊天开场建议`
   - 对应 5.9 Chat `低压开场建议`。
   - 说明：帮助整理可编辑草稿，不会自动发送消息。

### 4.1.3 本地开关 / 占位开关

5.10 首版允许以下两种方式之一：

1. **本地 presentation state 开关**
   - 仅在当前页面状态中切换说明；
   - 不持久化到服务端；
   - 不新增 SharedPreferences，除非项目已有明确本地偏好封装且不会影响 API / DB；
   - 不影响实际 Match / Profile / Chat 页面显示逻辑，除非成本极低且安全。

2. **说明型开关占位**
   - 展示开关样式但标注 `本机控制 · 后续完善`；
   - 或提供 `了解与关闭说明`；
   - 不写入任何后端偏好。

推荐首版采用 **说明型开关占位**，降低跨页状态联动风险。

### 4.1.4 统一免责声明

必须清楚可见：

```text
这些解释和建议只用于辅助理解与开场参考，不会写入资料，不会改变星盘或匹配算法，也不会自动发送消息。
```

如果设置页展示开关，还需补充：

```text
关闭后仍可使用匹配、个人资料和聊天主流程。
```

### 4.1.5 了解详情说明

建议增加一个 `了解这些建议如何工作` 入口，点击后展示本地 dialog / bottom sheet。

内容包括：

- 来源：来自已公开展示的资料、匹配说明和本地固定文案；
- 不做：不读取私密聊天、不写真值、不自动发送、不调用真人服务；
- 控制：后续会继续完善可关闭和更细粒度控制。

### 4.1.6 保持原 Settings 主流程

必须保留原 Settings / Version / Account / Appearance 等入口，不得因为 5.10 新增说明卡而替代原设置中心。

# 7. 实现原则

## 7.1 本地说明优先原则

5.10 不新增 API，也不做服务端偏好保存。

首版优先采用：

- Settings 内说明卡；
- 本地 dialog / bottom sheet；
- 说明型开关占位；
- 当前页面临时 toggle state。

不得把这些开关写入后端、profile、match、chat 或 release config。

## 7.2 最小治理原则

5.10 的治理不是后台治理，而是用户理解与用户控制：

- 说明解释层来源；
- 说明不会做什么；
- 提供关闭 / 减少展示的产品位置；
- 为后续 5.x 或 6.x 真正持久化控制预留路径。

## 7.3 UI 原则

解释层控制入口必须：

- 可读；
- 克制；
- 不像工程面板；
- 不制造焦虑；
- 不恐吓用户；
- 不干扰原 Settings 主流程；
- 不把设置页变成大而全控制台；
- 不遮挡账号、隐私、版本中心等原有入口。

建议视觉结构：

- 设置卡片；
- 三项能力说明；
- 三个开关或开关占位；
- 统一免责声明；
- `了解这些建议如何工作` 入口；
- 说明 dialog / bottom sheet。

## 7.4 可访问性与截图原则

必须确保关键文本能在 Android UI hierarchy 中被读取。

至少应能采集到：

- `解释与建议设置`

- `个人表达建议`
- `聊天开场建议`
- `不会写入资料`
