## 5. Feature Acceptance Notes

### 5.2A Tag Expression

Goal:

- Upgrade profile tags from a flat field list into a readable expression system for slow dating.

Implementation:

- `ProfilePage` groups tags into `人格倾向`, `关系风格`, `生活状态`, `资料真值提示`, and `慢约会友好表达`.
- The section is labeled `标签表达体系`.
- Copy states that tags are display and relationship-context hints only.

Boundaries:

- No backend tag model change.
- No questionnaire or matching algorithm change.
- No truth write-back.
- No absolute personality judgement.

Observation:

- Current grouping is presentation-level and keyword-based. A future backend-driven taxonomy could improve ordering and localization.

### 5.2B Profile Operating Hub

Goal:

- Make Me/Profile read like a personal operating hub instead of a long profile field list.

Implementation:

- `个人经营区` summarizes relationship goal, profile completion readiness, and city/birth truth separation.
- `编辑资料` remains reachable from the operating hub.
- `看看状态` links to the status square.

Boundaries:

- No settings center replacement.
- No commercial grid.
- No security center rewrite.
- No backend write.

Observation:

- Current operating hub is static presentation based on `ProfileSummaryEntity`. It should remain lightweight until real product telemetry or backend suggestions are introduced.

### 5.2C AI Display Advice

Goal:

- Upgrade `AI 助理 / 展示建议` from a single entry into actionable display advice.

Implementation:

- Shows `资料完整度建议`, `首聊友好度建议`, `亮点梳理建议`, and a link to existing astrology overview/display advice.
- Copy states that advice does not automatically modify profile data and does not write back to server truth.

Boundaries:

- No LLM panel.
- No fake optimization success.
- No automatic profile edit.
- No backend contract change.

Observation:

- Advice is deterministic presentation text based on profile summary. This is intentional for the 5.2 minimum slice.

### 5.2D AI Draft Helper

Goal:

- Add a minimum AI expression helper for self-introduction, greeting, and highlight organization.

Implementation:

- `AI 草稿助手` contains `帮我写一句自我介绍`, `帮我写一句问候`, and `整理我的亮点`.
- Tapping each action opens a bottom sheet with a local draft suggestion and disclaimer.

Boundaries:

- Drafts are local presentation suggestions only.
- No automatic publishing.
- No automatic message sending.
- No server truth write-back.
- No server-side AI queue.

Observation:

- The helper does not persist drafts. Persistence would require a separate product and contract decision.

### 5.2E Light Voice Expression

Goal:

- Add a light voice expression candidate slot without touching RTC or LiveKit.

Implementation:

- `轻语音表达` contains `声音名片候选位` and `录制前说明`.
- Dialog explains that this is a candidate slot and real recording would request microphone permission only when needed.

Boundaries:

- No RTC state machine change.
- No LiveKit token, join, leave, heartbeat, or permission service change.
- No multiplayer voice, video, live, or audio social platform.
- No recording persistence in this slice.

Observation:

- This slice proves product semantics without introducing an audio contract.

### 5.2F Personal Appearance Layer

Goal:

- Add a lightweight personal appearance layer without turning settings into a commercial dressing center.

Implementation:

- `个人空间外观` contains `展示封面：温柔清爽`.
- Style chips include `清爽`, `安静`, and `真诚`.
- Copy separates appearance from privacy, account, and security settings.

Boundaries:

- No shop.
- No coin system.
- No paid dressing rights.
- No settings route replacement.
- No profile truth write-back.

Observation:

- Current layer is a preview and semantic slot only. Persistent appearance preferences should be designed separately if needed.
