# EliteSync 3.1 版本任务书

更新时间：2026-04-09  
当前产品阶段：`3.0 首轮阶段性交付已通过验收`  
当前客户端版本参考：`0.02.09 / 209`  
下一目标版本：`3.1`  
本轮协作方式：**Codex 主执行 / Gemini 平面设计顾问与 UI 验收支持**

---

## 1. 本文档目的

本文档用于作为 EliteSync `3.1` 的正式任务书，直接服务于新一轮 Codex 开发执行。

3.1 不再以“体验微调”为核心，而是转向：

> **打通恋爱社交产品的最小核心闭环：注册 → 测试账号池 → 匹配 → 首聊 / 互动 → 状态发布 → 广场展示 → 管理员控制。**

本文档重点解决以下问题：

1. 测试账号丢失后如何建立可复用、可重建、可版本化的测试账号池
2. 普通用户、测试账号、管理员三类账号如何在数据层、权限层、前台 / 后台层完成分组
3. 管理员如何在 APP 内拥有最小可用的运营控制能力
4. 用户如何以最小成本发布状态，并让内容进入首页广场流
5. 如何保证每个测试账号有足够完整的画像与公开状态，用于调试匹配和广场推送
6. 如何在不扩张范围的前提下，做出真正可测试、可运营、可复现的最小链路

---

## 2. 3.1 版本定位

### 2.1 一句话定位

`3.1` 是 EliteSync 的：

> **核心社交闭环与测试运营底座打通版**

它的目标不是继续堆更多社交玩法，而是优先把以下能力做实：

- 可批量生成并重建测试账号
- 可区分普通用户 / 测试账号 / 管理员
- 可由管理员发起匹配、查看账号、封禁解封
- 可由普通用户和测试账号发布状态
- 可由首页广场展示服务端状态内容
- 可验证“一个普通账号发状态 → 其他账号可刷到”的最小内容链路

---

### 2.2 版本总目标

3.1 的核心目标分为四条主线：

#### A. 测试底座
建立正式的测试账号生成、标记、重建、画像补齐和状态预置体系。

#### B. 管理底座
让管理员在 APP 内拥有最小可用的业务控制能力，而不只是治理后台能力。

#### C. 社交内容底座
打通用户发布状态、服务端存储、广场展示、用户自删 / 管理员删除的最小闭环。

#### D. 匹配调试底座
让测试账号能参与匹配，并具备可用于调试的基础资料、城市、出生地、八字 / 星盘画像与公开状态。

## 3. 3.1 开发原则

本轮开发必须继续遵守以下原则：

1. **服务端 canonical 数据仍然是唯一真源**
2. **测试账号必须成为正式系统对象，不能继续依赖 SmokeUser / 手机号前缀这类弱约定**
3. **普通用户、测试账号、管理员的区别必须落到数据结构与权限逻辑上，而不是只体现在 UI 上**
4. **状态广场先做最小闭环，不做图片、评论树、复杂推荐、复杂审核流**
5. **管理员触发匹配属于高风险操作，必须具备最小审计与可回查能力**
6. **Gemini 只作为平面设计顾问与 UI 验收支持，不替代 Codex 的实现职责**
7. **本轮优先完成“能打通、能复现、能调试、能验收”，而不是做复杂扩张**

---

## 4. 本轮明确不做的事项

为防止 3.1 失控，本轮明确不做以下内容：

1. 图片状态 / 多媒体状态
2. 帖子编辑功能
3. 评论、回复、点赞、转发等复杂互动链路
4. 广场复杂推荐算法
5. 完整话题系统
6. 管理员复杂报表平台
7. 大规模匹配算法重写
8. 高风险数据库大改或多服务拆分
9. 将测试内容直接默认混入真实用户核心指标

---

## 5. 角色与账号分层设计

## 5.1 账号分层目标

虽然业务上当前主要有三类账号：
- 普通用户
- 测试账号
- 管理员

但实现上建议拆为两层：

### 第一层：角色（Role）
- `user`
- `admin`

### 第二层：账号类型（Account Type）
- `normal`
- `test`
- `synthetic`

这样可以避免把“是不是管理员”和“是不是测试账号”混成单一枚举。

示例：
- 普通用户：`role=user, account_type=normal`
- 测试账号：`role=user, account_type=test`
- 管理员：`role=admin, account_type=normal`
- 后续若需要合成账号：`role=user, account_type=synthetic`

---

## 5.2 建议补充的业务控制字段

建议增加并统一以下字段或等价能力：

- `role`
- `account_type`
- `is_match_eligible`
- `is_square_visible`
- `exclude_from_metrics`
- `is_banned`
- `banned_reason`
- `last_match_batch_id`（可选）

说明：
- `exclude_from_metrics` 用于避免测试账号污染真实运营指标
- `is_square_visible` 用于控制测试账号内容是否进入默认广场
- `is_match_eligible` 用于管理员快速控制账号是否参与匹配

## 6. 3.1 P0 范围（必须完成）

## P0-A 正式测试账号生成与重建脚本

### 目标
建立一套可复用的测试账号脚本，不再依赖临时手工造号。

### Codex 必做内容
1. 设计一套可重复执行的 seed / 命令脚本
2. 可批量生成测试账号
3. 可指定生成数量
4. 可清理并重建测试账号池
5. 可写入明确的 `account_type=test` 标识
6. 可设置 `exclude_from_metrics=true`
7. 可选择是否自动发布状态、是否自动参与匹配

### 测试账号初始资料要求
每个测试账号至少要生成：
- 昵称
- 性别
- 生日
- 所在城市（优先全国省会城市）
- 出生地点（按所在城市搜索候选，前 10 个结果随机）
- 定位信息 / 城市镜像字段

### 画像要求
每个测试账号生成基础资料后，必须触发并确保：
- 八字画像更新
- 星盘画像更新
- 其它当前匹配解释依赖画像更新完成

### 状态要求
每个测试账号应自动生成至少 1 条公开状态，用于首页广场与服务端内容链路调试。

### 验收标准
- 测试账号池可一键生成
- 测试账号池可一键重建
- 生成后能看到明确账号类型标记
- 画像链路实际完成更新
- 状态内容可在广场中被验证

---

## P0-B 管理员账号与 APP 内管理 UI

### 目标
管理员登录 APP 后拥有区别于普通用户的最小管理界面，能完成核心运营控制。

### 管理员最小能力要求
1. 查看用户列表
2. 查看用户基础信息与账号类型
3. 查看用户是否可参与匹配
4. 封禁 / 解封账号
5. 发起每周匹配
6. 查看当前或最近匹配批次信息
7. 查看某个账号的匹配状态
8. 查看某个账号的状态内容
9. 删除用户发布的状态

### 实现建议
- 管理入口在“我的”页或单独管理员入口中暴露
- 仅当 `role=admin` 时可见
- 不要求做复杂后台风格，可做移动端管理面板

### 高风险操作要求
“发起每周匹配”必须最少具备：
1. 二次确认
2. 操作日志或最小审计记录
3. 匹配批次 ID 或等价可回查标识

### 验收标准
- 管理员登录后能看到额外管理入口
- 管理员可以完成封禁 / 解封
- 管理员可以手动触发匹配
- 管理员可以查看至少基础账号信息与状态信息

---

## P0-C 首页状态发布功能 + 底部导航调整

### 目标
将状态发布纳入最小主链路，使内容广场形成真正服务端闭环。

### UI 结构要求
1. 底部导航最中间改为“发布状态”入口
2. 匹配结果入口移动到首页 Hero 区或首页关键卡片中
3. 发布状态操作尽量轻量，不引入复杂编辑器

### 状态结构要求
本轮状态仅支持以下字段：
- 标题
- 信息正文
- 地点（默认取账号所在城市，可编辑或选择是否允许覆盖，由 Codex 在实现期最小化决定）

### 服务端要求
必须新增服务端存储，不允许只保存在本地。
需要有最小数据库结构支持：
- 状态主表
- 作者信息关联
- 可见性 / 删除状态字段
- 时间戳字段

### 权限要求
- 普通用户可发布自己的状态
- 测试账号可发布自己的状态
- 作者本人可删除自己的状态
- 管理员可删除任意状态
- 修改功能以后再做，本轮不要求编辑

### 广场展示要求
- 首页可展示服务端状态流
- 至少能验证一个普通账号发的状态，另一个普通账号能刷到
- 测试账号发布的状态是否默认进入普通流，由 Codex 结合 `is_square_visible` 做最小可控实现

### 验收标准
- 发布状态链路打通
- 服务端确实存储成功
- 首页确实能读到状态流
- 作者本人能删帖
- 管理员能删帖

---

## P0-D 普通用户 / 测试账号参与匹配与互动的最小闭环

### 目标
打通真正可验证的最小社交链路。

### 必须验证的场景
1. 注册一个普通账号
2. 完成基础资料 / 必要问卷
3. 进入匹配池
4. 形成匹配结果
5. 进入首聊 / 会话链路
6. 发布一条状态
7. 另一普通账号能在首页广场刷到这条状态

### 测试账号场景
1. 批量生成的测试账号可进入匹配池
2. 测试账号具备完整画像
3. 测试账号具备至少一条公开状态
4. 管理员能看见测试账号并控制其匹配资格

### 验收标准
- 普通账号链路可跑通
- 测试账号链路可跑通
- 管理员控制链路可跑通
- 至少形成“注册 + 匹配 + 首聊 + 广场状态”的最小闭环

## 9. 明确禁止项

本轮开发明确禁止：

1. 继续依赖 `SmokeUser` 命名约定充当正式测试体系
2. 测试账号无标记混入真实用户数据池
3. 为了方便演示把测试内容写死在前端本地
4. 在无服务端结构支持的情况下伪造状态广场
5. 在无确认与最小审计的情况下直接做危险匹配发布
6. 顺手引入图片、评论、点赞、复杂推荐等范围外功能
7. 因为新增管理员 UI 而破坏普通用户“我的”页主链路

## 12. 3.1 验收标准

## 12.1 功能验收

必须满足：
1. 存在正式测试账号生成脚本
2. 测试账号可重复生成与重建
3. 普通 / 测试 / 管理员三类账号在系统内可区分
4. 管理员在 APP 内有额外管理 UI
5. 用户可发布状态到服务端
6. 首页广场可读到状态流
7. 作者本人可删除自己的状态
8. 管理员可删除任意状态
9. 至少一条普通账号发帖被另一普通账号成功刷到
10. 测试账号生成后能补齐画像并参与匹配

---

## 12.2 稳定性与工程验收

必须满足：
1. 不误伤 3.0 既有首页 / 匹配 / 消息 / 关于页主链路
2. 新增状态功能后，首页加载不可明显退化
3. 新增管理员 UI 后，普通用户视角不出现越权入口
4. 新增脚本与表结构有最小测试或回归说明
5. smoke / regression 至少覆盖：
   - 注册
   - 资料保存
   - 画像更新
   - 匹配结果
   - 消息
   - 状态发布 / 删除
   - 管理员操作入口

---

## 12.3 验收结论等级

Gemini / GPT / 最终外部验收统一使用：
- **通过**：主闭环打通，边界守住，证据齐全
- **有条件通过**：主体完成，但存在补证 / 补件项
- **不通过**：主闭环未打通，或误伤受保护面，或关键能力缺失

## 14. 最终结论

EliteSync 3.1 的核心，不是继续把产品做得更“像社交 APP”，而是先把它做成：

> **一个可测试、可运营、可管理、可复现的最小恋爱社交系统。**

因此，本轮 Codex 的任务重点应牢牢围绕以下四件事：

1. 正式测试账号体系
2. 管理员最小业务控制能力
3. 服务端状态发布与首页广场闭环
4. 普通 / 测试 / 管理三类账号的最小可验证链路

只要这四件事打通，EliteSync 才算真正进入了恋爱社交产品的核心能力建设阶段。
