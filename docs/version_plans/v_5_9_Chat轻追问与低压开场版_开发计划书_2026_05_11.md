# 2. 版本定位

## 2.1 本版本是什么

5.9 是 Chat 场景下的轻追问与低压开场最小产品化版本。

它把 5.7 的 `为什么值得聊` 和 5.8 的 `我的慢约会表达建议` 转化为 Chat 中更靠近行动点的轻提示：

```text
用户准备发第一句话或恢复对话时，系统给出几条低压、可编辑、不自动发送的开场方向。
```

5.9 只做最小可验证闭环：

- 在 Chat room 或首聊 / 回聊入口附近新增一个低压开场建议区；
- 提供从共同点开场、换个更自然说法、低压问候、不要太急提醒；
- 提供可编辑草稿候选，但不自动发送；
- 可选支持“填入输入框但不发送”的本地行为；若实现成本或风险较高，则只展示本地草稿说明，不写入输入框；
- 提供 display-only 免责声明；
- 保持原消息发送、语音、附件、表情、更多工具入口不被替代。

## 2.2 本版本不是什么

5.9 不是：

- 真实 AI 聊天助手；
- 自动代聊；
- 自动发送消息；
- 聊天内容分析系统；
- 私密聊天读取器；
- 新消息 API；
- 新推荐算法；
- 新数据库字段；
- 付费咨询；
- 真人情感顾问；
- 测测 AI 问答复刻；
- Chat 重构；
- release / APK 发布任务。

## 2.3 本版本验收口径

5.9 可以验收为：


前提是：

- Chat 页面新增低压开场 / 轻追问建议，且用户可见；
- 建议保持 display-only / local-only；
- 未改 API / DB / release / backend；
- 未自动发送消息；

# 3. 产品目标

## 3.1 用户目标

5.9 要帮助用户解决四个问题：

1. **我现在该怎么开口？**
   - 给出低压、不冒犯、不强推进的开场方向。

2. **如何把匹配理由转成一句自然的话？**
   - 从共同点、资料细节、关系节奏中生成本地草稿候选。

3. **冷场时能不能换一种说法？**
   - 给出更自然、更短、更轻的替代表达。

4. **我应该避免什么？**
   - 提醒不要一开始问隐私、不要过快推进、不要把解释层写成确定判断。

## 3.2 产品目标

5.9 要完成从“解释层可读”到“解释层可转化为低压开场”的最小升级。

它应增强 Chat 的行动支持价值，但不能代替用户表达，也不能改变 Chat 消息主链。

建议产品结构：

```text
Chat room / 首聊入口 / 回聊入口
  ├─ 原有关系摘要 / 聊天对象信息
  ├─ 原有消息列表
  ├─ 新增：低压开场建议
  │   ├─ 从共同点开场
  │   ├─ 换个更自然的说法
  │   ├─ 低压问候建议
  │   ├─ 不要太急的提醒
  │   ├─ 可编辑草稿候选
  │   └─ display-only / no auto-send 免责声明
  └─ 原有输入框 / 语音 / 表情 / 更多工具 / 发送按钮
```

## 3.3 设计语言

5.9 应使用 EliteSync 自有语言，不复制测测术语，也不把轻建议写成 AI 代聊。

推荐用词：

- `低压开场建议`
- `从共同点开始`
- `换个更自然的说法`
- `轻问候`
- `不要太急`
- `可编辑草稿`
- `只做开场参考，不会自动发送`
- `你可以改完再发`

避免用词：

- `AI 代聊`
- `自动帮你发`
- `最佳回复`
- `必回话术`
- `拿捏对方`
- `心理操控`
- `恋爱军师`
- `大师建议`
- `付费解锁`
- `真人咨询`
- `上传截图分析`

# 4. 功能范围

## 4.1 必做功能

5.9 必须完成以下最小功能：

### 4.1.1 低压开场建议卡

在 Chat room 或首聊 / 回聊行动点新增一张建议卡。

建议标题：

```text
低压开场建议
```

建议副标题：

```text
这些只是本地草稿方向，不会自动发送。
```

### 4.1.2 从共同点开场

新增一个建议项：

```text
从共同点开始
```

示例文案：

```text
可以先从你们都提到的兴趣或城市经历问起。
```

或：

```text
先回应对方资料里一个具体细节，会比泛泛问候更自然。
```

### 4.1.3 换个更自然的说法

新增一个建议项：

```text
换个更自然的说法
```

示例文案：

```text
把长句拆短一点，用一句轻松的问题开始。
```

或：

```text
如果不知道怎么接，可以先表达好奇，不急着推进关系。
```

### 4.1.4 低压问候建议

新增一个建议项：

```text
低压问候建议
```

示例草稿：

```text
你好呀，看到你的资料里提到这个点，感觉挺有意思的。
```

或：

```text
嗨，可以先从一个轻松的话题慢慢聊起吗？
```

### 4.1.5 不要太急的提醒

新增一个建议项：

```text
不要太急
```

示例文案：

```text
刚开始可以先别追问隐私，也别急着推进见面。
```

或：

```text
先用轻问候和具体话题建立安全感。
```

### 4.1.6 可编辑草稿候选

首版可以提供 1-3 条本地草稿候选。

草稿必须是：

- 本地 deterministic 文案；
- 可编辑；
- 不自动发送；
- 不写服务端；
- 不伪装成真实 AI 输出。

可选交互：

1. 只展示草稿，不操作输入框；
2. 点击后复制到输入框，但不发送；
3. 点击后弹出本地 bottom sheet，用户自行复制或确认填入。

推荐首版优先采用 **bottom sheet / 本地说明**，降低误发风险。

### 4.1.7 display-only / no auto-send 免责声明

必须清楚可见：

```text
以上内容仅为开场参考，不会读取私密聊天，不会写入资料，也不会自动发送消息。
```

如果支持填入输入框，还必须显示：

```text
填入后仍需你自己确认发送。
```

# 7. 实现原则

## 7.1 数据来源原则

5.9 不新增 API，只能使用当前 Chat 页面已有数据上下文。

解释层数据可以来自：

- 当前 chat peer 的公开 profile summary；
- 已有 match / relationship summary；
- 当前页面已有 relationship card；
- 已有共同点 / 标签 / 城市 / 人格展示信息；
- 本地 deterministic presentation helper；
- 静态 fallback copy。

如果当前 Chat 数据不足，宁可展示克制 fallback，不得新增接口。

示例 fallback：

```text
当前上下文还不多，可以先用一句轻松问候开始。
```

## 7.2 deterministic helper 原则

首版低压开场建议可使用本地 deterministic helper，根据已有展示字段生成文案。

但 helper 必须：

- 只在 presentation 层；
- 不写入缓存；
- 不写入 server；
- 不改变 repository / API contract；
- 不伪装成 AI；
- 不输出绝对判断；
- 不自动调用发送逻辑；
- 不自动持久化草稿。

## 7.3 输入框填入原则

若实现“填入输入框”功能，必须满足：

- 只改本地 TextEditingController；
- 不触发发送按钮；
- 不调用 message API；
- 不写数据库；
- 填入后必须由用户手动确认发送；
- UI 需清楚提示“仍需你自己确认发送”。

如果现有 Chat 输入架构不适合安全填入，则本版本只做展示 / bottom sheet，不做填入。

## 7.4 UI 原则

解释层必须：

- 可读；
- 克制；
- 不像工程面板；
- 不制造焦虑；
- 不操控用户；
- 不干扰原 Chat 主行动；
- 不遮挡输入框；
- 不遮挡消息列表关键内容；
- 不让失败态比成功态更重。

建议视觉结构：

- 可折叠或轻卡片式区块；
- 标题 + 简短说明；
- 3-4 个建议 chip / row；
- 1-3 条草稿候选；
- disabled / local-only 明确提示；
- 免责声明小字。

## 7.5 可访问性与截图原则

必须确保关键文本能在 Android UI hierarchy 中被读取。

至少应能采集到：

- `低压开场建议`
- `从共同点开始`
- `换个更自然的说法`
- `低压问候建议`
- `不要太急`
- `可编辑草稿`
- `不会读取私密聊天`
- `不会写入资料`
- `不会自动发送消息`
- 如支持填入：`仍需你自己确认发送`
