# 算法 2.6.2 版本修改指令（可直接下发 Codex）

> 目的：基于当前 2.6 初版可视化结果，进行一轮**只聚焦“玄学可视化与信息结构修正”**的收口迭代。
>
> 本轮不再新增术数，不重写算法主链，不改 canonical 计算逻辑；重点解决：
> 1. 八字 / 星盘 / 紫微可视化理解偏差
> 2. 页面信息层级混乱、重复标签过多、主视觉不明确
> 3. 用户看不懂、看不顺、看不到重点的问题
> 4. “我的”页到各详情页的入口与状态表达不够稳定的问题

---

# 0. 先给结论：当前 2.6 可视化为什么不理想

## 0.1 当前问题不是“功能没做”，而是“视觉结构理解错了”
当前版本已经做到：
- 页面能打开
- 数据能读到
- 八字 / 星盘 / 紫微都有详情页
- 服务端真源和前端展示链路基本接上了

但当前问题在于：
- **把“数据展示”误做成了“诊断页堆字段”**
- **把“专业盘面”误做成了“卡片 + 标签拼接页”**
- **把“参考图的主视觉”误理解成“放几个 chip + 放一个示意图”**

所以 2.6.2 的目标不是补数据，而是：
**把已经有的数据，重新组织成正确的信息架构和视觉主结构。**

---

# 1. 2.6.2 的总原则

## 1.1 只做“UI 结构修正”，不扩范围
本轮不做：
- 新算法
- 新模块
- 新接口扩张
- 新商业化功能
- 新专家级参数系统

本轮只做：
- 页面层级重排
- 视觉主结构重做
- 字段展示分层
- 入口与状态表达修正
- 可读性和专业感提升

## 1.2 三条核心设计原则

### 原则 A：一个页面只能有一个主视觉
- 八字页主视觉 = **四柱矩阵 / 四柱主面板**
- 星盘页主视觉 = **本命盘轮盘**
- 紫微页主视觉 = **十二宫盘**

不能再出现：
- 主视觉很弱，标签很强
- 主面板像占位图，标签像正文
- 用户第一眼看不到“这个页面到底在展示什么”

### 原则 B：元信息不得抢主结果的视觉权重
像下面这些信息：
- true solar time
- canonical_server
- confidence
- location shift
- engine
- precision

都属于**参数层 / 元信息层**，不应该在页面最显眼位置大面积占视觉。

### 原则 C：专业图必须“图 + 摘要 + 参数”三段式
每个详情页统一改成：
1. **图**：主视觉
2. **摘要**：用户能读懂的结果说明
3. **参数**：技术细节、引擎、精度、真太阳时、位置修正

不能再把三者揉成一锅。

# 2. 总体信息架构修正

## 2.1 详情页统一结构
所有玄学详情页统一改成以下结构：

```text
详情页
├─ 顶部标题栏
├─ 主视觉区（唯一核心图）
├─ 摘要区（给普通用户看）
├─ 关键结果区（结构化结果）
├─ 参数区（默认折叠/弱化）
└─ 备注区（仅保留必要说明）
```

## 2.2 必须删除的当前错误结构
- 顶部放过多横向 chips 抢主视觉
- 页内出现跨模块导航（如星盘页里又放“八字/紫微/诊断”切换）
- 把 engine / precision / confidence 做成大面积蓝色标签云
- 一个页面同时出现多个“看起来像主视觉”的容器
- 诊断字段直接暴露为正文主要内容

## 2.3 顶部信息规范
详情页顶部只保留：
- 返回
- 标题
- 一个轻量“参数”按钮
- 一个轻量“设置/显示选项”按钮（若必须）

不要在顶部堆：
- 多模块 tab
- 大量状态 chip
- 多层切换

# 3. 八字详情页修改指令

## 3.1 当前问题判断
当前八字页的问题主要有：
- 主视觉还是太弱，四柱矩阵不够像“主面板”
- 元信息（真太阳时、精度、置信）过多用 chip 呈现，显得像调试页
- “生日未填写 / 出生时间 14:25 / 出生地 … / 真太阳时 …”这种字段摆放不统一，阅读顺序混乱
- 五行条形图虽然可用，但页面整体像“管理后台卡片页”，不像成熟排盘页

## 3.2 2.6.2 正确结构

### 第一屏：主视觉必须是四柱主面板
改成：
- 一个完整的四柱矩阵面板
- 年柱 / 月柱 / 日柱 / 时柱为主列
- 至少显示：
  - 天干
  - 地支
  - 十神（轻量）
- 视觉上更接近“表盘 / 排盘面板”，而不是普通 table

### 第二屏：五行图
保留五行条形图，但要：
- 更靠近参考图的密度
- 数值右对齐
- 颜色统一
- 不再用大面积外描边 chip

### 第三屏：辅助信息区
将以下内容统一移入“参数 / 资料来源”区：
- 真太阳时
- 生日
- 出生时间
- 出生地
- 位置修正
- 经纬修正
- 位置来源
- precision / engine / confidence

这些不应该在第一屏占主视觉。

## 3.3 八字页视觉要求
- 减少空白
- 增强矩阵密度
- 不要每个字段都套蓝色描边胶囊
- 用更稳的层次：标题 > 主结果 > 辅助说明 > 参数

## 3.4 八字页必须保留
- 四柱
- 五行
- 一句话摘要
- 参数折叠区

## 3.5 八字页必须下掉/弱化
- 过多 badge
- 过多 outline chip
- 与主面板重复的信息
- 调试口径直接放正文第一屏

# 4. 星盘详情页修改指令

## 4.1 当前问题判断
当前星盘页问题最明显：
- 本命盘轮盘还是“占位示意图感”，不是成熟星盘盘面
- 页面顶部“总览 / 八字 / 紫微 / 诊断”非常破坏星盘页专注度
- `legacy_input` 放在盘中心是严重错误，它是工程参数，不是用户主结果
- 太阳 / 月亮 / 上升的标签漂浮在盘边，但没有形成真正的盘面理解
- 当前页面更像“技术调试版星盘摘要页”，不是“星盘详情页”

## 4.2 2.6.2 正确结构

### 第一屏：只保留本命盘轮盘
轮盘必须升级为：
- 外圈：12 星座名称或符号
- 中圈：盘面分区
- 内圈：中心锚点
- 点位：太阳 / 月亮 / 上升
- 若当前 western_lite 不能稳定提供更多行星或宫位，就不要假装完整

### 中心文案改法
中心只能写：
- `本命盘`
- 可选一行轻量副标题：`基于当前出生资料生成`

绝对禁止再写：
- `legacy_input`
- `canonical_server`
- `engine` 等工程词汇

### 第二屏：星盘摘要
摘要区只讲：
- 太阳星座
- 月亮星座
- 上升星座
- 一句话解释：情绪回应 / 表达方式 / 推进节奏

### 第三屏：参数区
把这些移入参数区：
- 引擎
- 置信
- 真太阳时
- 位置修正
- 经度偏移
- 均时差
- 位置来源

默认折叠，不允许和主盘抢视觉。

## 4.3 必须删除/调整
- 删除详情页内部“跨模块 tab”
- 删除盘面中心工程词汇
- 删除过多漂浮标签
- 若 house/aspect 当前不可靠，不要做“伪专业全盘”

## 4.4 星盘页目标
把当前“轮盘占位图”升级成：
**一个能让用户第一眼就知道自己在看星盘，而不是在看技术摘要卡片的页面。**

# 5. 紫微详情页修改指令

## 5.1 当前问题判断
当前紫微页的核心问题是：
- 12 宫网格已经有方向，但仍然太像“很多小卡片拼出来的列表”
- 宫位 chip（1宫/high/2宫/medium…）过多，视觉噪声大
- 文本密度太高，且压不住
- 还没有形成参考图里的“一个完整命盘”的感觉
- 中心命盘区缺失，宫位像 12 张分散说明卡，而不是同一个盘的一部分

## 5.2 2.6.2 正确结构

### 第一屏：做真正的十二宫主盘
布局改成：
- 外围 12 宫格固定位置
- 中间为中心摘要区
- 命宫 / 身宫高亮
- 关键宫位（夫妻宫、官禄宫、财帛宫等）次高亮

### 中心区必须显示
- 性别 / 局数（若有）
- 出生时间
- 农历时间（若服务端有）
- 命主 / 身主
- 当前层级：本命

### 宫位卡内信息简化
每宫默认只显示：
- 宫名
- 主星
- 1 行摘要

更多内容点宫位后进入底部抽屉。

### 宫位状态标签改法
当前的 `high / medium / normal` 太像开发标注。
统一改成中文轻标签，例如：
- 强
- 中
- 平

而且不要每宫都双 chip。

## 5.3 第二屏：摘要区
命盘下必须有：
- 核心主题
- 关系倾向
- 风险提醒
- 建议

否则普通用户只会觉得“图很复杂但不知道怎么看”。

## 5.4 第三屏：参数区
保留：
- precision
- confidence
- engine
- true solar time
- 修正信息

但全部放到参数抽屉，不许继续占用命盘主视觉空间。

## 5.5 紫微页必须追求的效果
不是“12 张小卡片排成 3x4 网格”，而是：
**一个完整命盘，被拆成 12 宫，但视觉上仍然是一个整体。**

# 6. “我的”页面与入口修改指令

## 6.1 当前问题
目前从验收材料看，入口能跳，但还偏“列表页心态”，不够像统一中枢。fileciteturn24file0

## 6.2 2.6.2 要求
“我的”页只保留三类核心入口：
- 基础资料
- 玄学资料
- 玄学总览

玄学详情页不要直接在“我的”页全部平铺成多个重复入口。

## 6.3 推荐结构
```text
我的
├─ 基础资料卡（编辑）
├─ 玄学资料卡（编辑）
├─ 玄学总览卡（进入总览页）
└─ 设置与系统
```

玄学总览页再统一进入：
- 八字
- 星盘
- 紫微
- 生肖/星座

## 6.4 状态提示要求
“我的”页必须明确显示：
- 资料完整度
- 最近更新时间
- 当前画像状态（最新 / 计算中 / 失败）

# 7. 资料修改与画像实时更新修改指令

## 7.1 当前目标不变
保持“服务端真源”，前端只做输入、触发、刷新。fileciteturn24file0

## 7.2 2.6.2 必须明确的交互
用户修改以下字段：
- 生日
- 出生时间
- 出生地
- 经纬度

保存后必须出现明确状态流：
1. `资料已保存`
2. `正在更新八字 / 星盘 / 紫微画像`
3. `更新完成` 或 `部分失败，可重试`

## 7.3 不允许
- 保存成功但无反馈
- 页面继续显示旧画像而用户不知情
- 某模块更新失败但界面无提示

# 8. 备份与发布流程修改指令

## 8.1 当前方向认可
验收材料里备份脚本与发布脚本接线方向是对的，但 2.6.2 里必须补一个点：
**让“备份已执行”在发布日志中更显式，而不是只存在脚本内部。** fileciteturn24file0

## 8.2 2.6.2 要求
每次版本更新前：
- 自动执行本地备份
- 自动执行云端备份
- 记录备份文件名 / checksum / 时间
- 发布日志里明确写出“备份已完成”
- 如果失败，默认阻断发布

## 8.3 前端不需要展示备份细节
“我的”页只保留轻量“数据安全说明”入口，不要展示技术细节。
