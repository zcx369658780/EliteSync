## 1. 产品目标

5.6A 的目标是在 Match 场景中规划一个轻量的关系解释层，帮助用户理解“为什么推荐这个人”、如何低压开场、哪些表达方式更适合当前关系语境。

本 PRD 只定义产品规格，不进入代码实现。

Match 关系解释层要解决的问题：

- 匹配结果本身只告诉用户“这个人被推荐了”，但没有足够解释“为什么值得聊”。
- 用户看到匹配对象后，常见阻力是开场压力、表达不确定、害怕冒犯或不知道从哪里开始。
- 现有匹配理由如果过于工具化，用户难以转化为实际沟通动作。
- 关系解释层可以把匹配理由翻译成更可读、更低压、更接近聊天启动的表达建议。

为什么第一切片选择 Match，而不是 Me 或 Chat：

- Match 是“关系理解”的第一现场，用户天然需要解释推荐原因。
- Match 可以保持 display-only，不需要写入用户资料真值。
- Match 的解释层可以服务后续 Chat，但不直接碰消息发送链。
- Me 更接近 profile 真值，容易被误解为画像判断。
- Chat 更接近真实消息与自动发送风险，需要更强交互保护。

## 2. 用户场景

### 场景 A：理解推荐原因

用户看到一个匹配结果时，需要快速理解：

- 为什么推荐这个人。
- 双方有哪些可聊的共同点。
- 这个推荐更偏兴趣、表达节奏、互动互补，还是慢约会适配。

### 场景 B：低压开场

用户对推荐对象有兴趣，但不知道如何开始第一句话。关系解释层应提供低压、可改写的开场方向，而不是替用户发送消息。

### 场景 C：表达方式提醒

用户需要知道：

- 哪些表达方式更适合当前匹配。
- 哪些表达方式容易过度、太急或造成误解。
- 如何保持轻松、真诚、不冒犯。

## 3. 页面位置

建议位置：

- Match detail / 匹配详情页。
- 现有匹配解释区域下方或旁侧。
- 作为一个可折叠的轻量解释模块。

页面关系：

- 不替代现有匹配结果页。
- 不改变匹配算法主链。
- 不改变 Match 推荐结果。
- 不阻塞用户原有的查看资料、返回、聊天入口等路径。
- 不强制用户读完解释层才能继续。

## 4. 信息结构

第一版建议包含五个部分。

### 一句关系摘要

目的：用一句克制、非决定性的语言说明当前匹配可以从哪里理解。

示例方向：

- “你们可能更适合从共同兴趣开始轻松聊天。”
- “这组匹配的优势在于表达节奏比较容易接上。”
- “这次推荐更偏慢热型，需要从低压话题开始。”

禁止方向：

- “你们命中注定。”
- “你们一定适合。”
- “这是你的最佳伴侣。”

### 2-4 个解释维度

目的：把推荐理由拆成可扫描的维度。

每个维度建议包含：

- 维度名。
- 一句解释。
- 可选轻量标签或强度提示。

第一版不需要复杂评分、不需要雷达图、不需要生成式长文。

### 建议 / 避免

目的：把解释转化成沟通建议。

建议：

- 给用户一条低压开场方向。
- 鼓励真诚、具体、轻松的表达。

避免：

- 避免过度承诺。
- 避免强判断。
- 避免一上来追问隐私。

### 一个轻追问入口

目的：为后续 AI 助手或解释扩展留入口。

第一版建议只预留入口，不接真实 AI。

入口文案方向：

- “想知道怎么开场？”
- “为什么推荐这个人？”
- “帮我换一种更自然的说法。”

### 免责声明 / display-only 提示

建议固定提示：

“以下内容用于帮助你理解互动节奏和开场方向，不代表事实判断，也不会改写你的资料。”

## 5. 解释维度候选

第一版可从以下维度中选择 2-4 个，不需要全部上屏。

### 共同点

解释双方公开展示资料或匹配上下文中可用于开场的共同元素。

可用方向：

- 共同兴趣。
- 相近生活方式。
- 类似内容偏好。

### 表达节奏

解释双方适合快聊、慢聊、轻松开场还是先观察。

可用方向：

- 慢热。
- 轻松互动。
- 适合从具体话题开始。

### 关系安全感

解释当前匹配是否适合低压、尊重边界、循序渐进的关系推进。

可用方向：

- 更适合温和开场。
- 需要避免压迫式追问。
- 适合先建立轻松感。

### 开场友好度

解释是否容易找到第一句话。

可用方向：

- 有明确共同话题。
- 可以从资料中的具体点切入。
- 需要更轻的破冰方式。

### 互动互补性

解释双方表达方式或兴趣结构是否存在互补启发。

可用方向：

- 一方更主动，一方更慢热。
- 话题类型互补。
- 可以通过提问建立互动。

### 慢约会适配度

解释这次匹配是否适合 EliteSync 的慢约会节奏。

可用方向：

- 不急于推进。
- 先看表达质量。
- 先用小话题建立信任。

## 6. 文案原则

必须遵守：

- 不做强预测。
- 不做心理诊断。
- 不做命运判断。
- 不做现实指导承诺。
- 不使用“命中注定”“必然适合”“一定会喜欢你”等绝对表述。
- 不复制测测术语体系。
- 不使用强玄学诊断口吻。
- 不用操控性、PUA 式或替用户下判断的文案。
- 不暗示系统掌握未授权隐私。

推荐表达：

- “可能更适合……”
- “可以尝试从……开始。”
- “这次推荐更偏向……”
- “如果想低压一点，可以……”

避免表达：

- “对方一定……”
- “你们注定……”
- “系统判断你们最适合……”
- “对方心理是……”

## 7. Derived-Only / Display-Only Contract

Match 关系解释层必须保持：

- derived-only
- display-only
- no silent writeback
- non-authoritative
- reversible
- optional

明确禁止：

- 不反写 `profile/basic`。
- 不反写 `profile/astro/summary`。
- 不反写 `profile/astro/chart`。
- 不反写 `user_astro_profiles`。
- 不反写匹配算法输入。
- 不写聊天消息正文。
- 不自动发送。
- 不把解释结果保存为用户画像。
- 不把 AI 输出作为匹配算法输入。

## 8. 数据边界

第一版只允许使用：

- 已有匹配上下文。
- 用户已授权展示资料。
- 已存在的公开标签或兴趣。
- 当前 Match detail 已经可见的信息。

第一版不允许使用：

- 未授权聊天内容。
- 私密聊天记录。
- 未展示的 profile 字段。
- 本地私有文件。
- 上传截图。
- 新增出生资料强收集。
- 支付、咨询、服务市场数据。
- 竞品 UI 推断出的算法或数据库结构。

第一版不接：

- 真人服务。
- 付费报告。
- 上传分析。
- 强出生资料表单。

## 9. AI 边界

第一版建议先不接真实 AI。

允许规划：

- 预留一个轻追问入口。
- 轻追问未来只生成建议或草稿。
- 用户必须主动点击。
- 输出必须可编辑、可忽略。

禁止规划为第一版能力：

- 自动发送。
- 自动代聊。
- 自动读取未授权聊天内容。
- 自动持久化为用户画像。
- 将 AI 输出写回 profile。
- 将 AI 输出写入匹配算法。
- 接入真人服务或付费咨询。

如果后续允许接 AI，需要另开 AI contract 文档，并单独确认：

- 输入字段。
- 输出字段。
- 是否缓存。
- 用户关闭方式。
- 免责声明。
- 验收和回滚策略。

## 10. 交互边界

用户必须可以：

- 忽略关系解释层。
- 关闭或折叠关系解释层。
- 不读解释也能继续使用 Match detail。
- 不点击轻追问入口。
- 不因为解释层阻塞聊天入口、返回、查看资料等原流程。

交互不得：

- 强制完成。
- 弹窗打断。
- 默认展开过长内容。
- 遮挡核心资料。
- 改变匹配结果。
- 自动进入聊天。
- 自动生成或发送消息。

## 11. 验收标准

### PRD 层验收标准

- 明确产品目标和第一切片范围。
- 明确为什么优先 Match。
- 明确 non-goals。
- 明确 protected surfaces。
- 明确 canonical truth 禁止反写清单。
- 明确 derived-only / display-only contract。
- 明确数据边界和 AI 边界。
- 明确后续实现前需要用户确认的问题。

### 后续 UI 方案验收标准

后续如进入 UI 方案设计，必须确认：

- 模块位置在 Match detail / 匹配解释区域。
- 不替代现有匹配结果页。
- 不阻塞原流程。
- 文案不做强预测、诊断或绝对判断。
- 有 display-only 提示。
- 可折叠或可忽略。
- 不出现真人咨询、付费报告、上传截图主路径。

### 后续实现验收标准

后续如进入实现，必须确认：

- 不修改数据库 schema。
- 不新增迁移。
- 不修改 API contract，除非用户单独批准。
- 不修改版本号。
- 不进入发版链。
- 不反写 profile。
- 不影响匹配算法输入。
- 不自动发送聊天消息。
- 不新增支付 / 咨询 / 上传链。
- 保护面回归通过。

### 保护面不被触碰核对清单

实现前后都要核对：

- `profile/basic` 未被写入。
- `profile/astro/summary` 未被写入。
- `profile/astro/chart` 未被写入。
- `user_astro_profiles` 未被写入。
- Match 推荐结果未被解释层改变。
- Chat 消息正文未被自动生成或发送。
- Me 资料编辑链未被修改。
- Auth / session 未被修改。
- 支付 / 会员 / 订单链未被修改。
- 版本号和发版链未被修改。

## 12. Non-Goals

5.6A 明确不做：

- 真人玄学咨询。
- 付费关系报告。
- 上传截图分析。
- 测试结果回写画像。
- 自动代聊。
- 强诊断或预测。
- 星盘图谱实现。
- 出生资料强收集。
- 大规模社区讨论。
- 服务市场导流。
- 付费转化链。

## 13. 后续实现前仍需用户确认的问题

进入任何实现前，需要用户确认：

1. 是否正式立项 5.6A。
2. 是否允许进入 Flutter UI 方案设计。
3. 是否需要设置页关闭开关。
4. 是否需要免责声明固定文案。
5. 第一版是否完全静态。
6. 是否允许后续接 AI。
7. 是否允许读取哪些匹配上下文字段。
8. 是否允许解释层短期缓存。
9. 是否需要独立验收截图与 XML 证据。
10. 是否需要 Claude / Gemini 做 UI 走查或视觉评审。
