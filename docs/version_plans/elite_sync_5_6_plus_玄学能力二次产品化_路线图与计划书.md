# 第一部分：整体开发计划书

## 1. 本次修订目的与最终结论

本次修订承接两条已完成的研究线：

1. **Soul 拆解线**：继续作为 EliteSync 社交主链参考，服务 Discover、Match、Chat、Me、Settings / Appearance、关系推进、低压开场、个人经营等主链设计。
2. **测测拆解线**：作为 EliteSync 玄学解释层参考，服务星盘、合盘、生辰、紫微、匹配解释、AI 追问、个人解释层和关系解释层的二次产品化。

本次修订后的总判断是：

> **5.0–5.5 作为已完成历史链路，不重写；5.6 起进入“玄学能力二次产品化与校准线”，把已有星盘、画像、匹配解释、关系理解能力转化为 Match / Me / Chat 主链中的 derived-only / display-only / explanation layer。**

这不是把 EliteSync 改造成测测，也不是启动真人玄学咨询、付费报告、上传截图解析、强商业化服务市场。测测只提供结构启发：

- 入口层；
- 上下文层；
- 解释对象层；
- 解释 / 建议层；
- 轻追问层。

其中前四层可以积极吸收，第五层必须克制吸收，不进入强转化和服务市场。

## 2. 5.0–5.5 历史链路处理

5.0–5.5 不在本次修订中重写，只作为已完成或已承接的历史链路保留。

- **5.0**：最小产品化覆盖集，完成 Discover / Chat / Me 的第一轮产品化起点。
- **5.1**：关系推进与内容转化补齐版，完成首聊 / 回聊 / 冷场恢复、关系摘要、状态到聊天低压转化等。
- **5.2**：个人经营中枢与表达层覆盖版，完成个人经营区、标签表达体系、AI 展示建议、AI 草稿助手、轻语音表达候选位、个人空间外观层。
- **5.3**：功能覆盖收尾版，完成 Discover / Chat / Me / Settings 第二轮补齐与护栏回归。
- **5.4**：测试运营准备与云端治理增强版，完成只读运营准备入口、观测入口、runbook、synthetic / smoke 账号治理提示等。
- **5.5**：真实小样本反馈吸收版，完成批准范围内修复与阿里云发布，APK 已到 `0.05.05 / 50500`。

5.0–5.5 的主链不应因为本次测测拆解而重开。后续只能在其稳定基线上做 additive 增量。

## 3. 新阶段定位：玄学能力二次产品化与校准线

5.6+ 的核心不是继续堆新工具，而是把 EliteSync 已经具备的玄学 / 星盘 / 画像 / 匹配解释能力转译成用户可理解、可忽略、可关闭的解释层。

### 3.1 “二次产品化”的含义

二次产品化指：

- 从“能力存在”转向“主链可理解”；
- 从“图谱与资料”转向“关系解释、表达建议、低压开场”；
- 从“结果展示”转向“可被普通用户消费的短句、维度、建议 / 避免”；
- 从“可能的 AI 追问入口”转向“受控、可关闭、不可写回真值的 explanation layer”。

### 3.2 “校准线”的含义

校准线指：

- 校准哪些解释可以进入 Match；
- 校准哪些解释可以进入 Me / Profile；
- 校准哪些解释可以进入 Chat；
- 校准哪些文案不能写成事实、预测、诊断或真值；
- 校准所有解释层不得反写 canonical truth；
- 校准所有 AI 追问、合盘解释、建议文案只能作为 display-only 派生解释。

## 4. 测测 Stage 1–5 可吸收结论

## 4.1 Stage 1：基础 IA 与入口矩阵

### 可吸收

- 玄学能力不应只藏在深层工具页，可以通过首页、Match、Me、Chat、AI 助理等主链入口形成入口矩阵。
- 入口应先解释价值，再允许用户进入深层。
- 首页可以承载能力入口，但不能压过 EliteSync 的慢约会主线与资料真值链。

### 不吸收

- 商业入口、咨询入口、会员入口高密度混排。
- 过早把工具、测试、服务市场和内容流压在同一主屏。
- 用商城、币、卡券、达人等商业元素污染核心主链。

---

## 4.2 Stage 2：AI问 / 在线分层

### 可吸收

- AI 解释层、问题导航层、真人服务层必须分开。
- `AI问` 类结构可以启发 EliteSync 的轻追问入口。
- 关系 / 匹配解释页可以先给轻问题入口，再给可选深问路径。
- 问题卡不应写死，应保留场景化、可轮换、可后续动态配置的余地。

### 不吸收

- 真人 1v1 服务市场。
- `在线` 达人服务大厅。
- 优惠券、连麦、付费咨询、强服务转化。
- 把 `AI问 -> 在线` 写成已验证闭环。

---

## 4.3 Stage 3：测试频道

### 可吸收

- 测试频道 Claude rerun 显示，本轮样本更像互动答题入口层，而不是稳定的结果解释页。
- 可以借鉴“用户先进入轻互动，再获得解释”的节奏，但只能作为可选体验参考。
- `本文含AI生成内容` 这类声明可作为 EliteSync 未来 AI 解释层免责声明参考。

### 不吸收

- 娱乐化 A/B/C/D 测试作为主路径。
- 提交测试答案后反写用户画像。
- 把测试结果当作事实资料或匹配算法输入。
- 把“本轮样本”外推成全量频道事实。
- 把创作者级 `向TA提问` 服务导流迁入 EliteSync 主链。

---

## 4.4 Stage 4：星盘总览 / 图谱 / 档案上下文

### 可吸收

- 单人档案上下文可以承接图谱、摘要、分层解释、轻追问入口。
- 星盘类信息适合“图谱先行 + 结论摘要 + 分层解释”的阅读节奏。
- 档案列表是上下文管理页，不是结果页。
- `基础解读` / AI 解读入口可以启发 EliteSync 的解释层分级，但首版必须控制在 display-only。

### 不吸收

- 把 `DeepSeek解读`、AI 追问、AI问、在线、真人服务混成一条已验证链。
- 未点击的入口不能写成已验证目标页。
- 参数、预览报告、深层解释未验证前不能写成可用链路。
- 高密度星盘术语、复杂宫位图、社区讨论和付费报告不进入近期主线。

---

## 4.5 Stage 5：缘分合盘 / 双人关系解释

### 可吸收

- 双人上下文适合承接关系解释层。
- 可吸收“关系短句 + 维度解释 + 建议 / 避免 + 轻追问入口”的结构。
- Match 可以优先承接关系解释层，因为它天然已有双人上下文。
- 关系解释应先建立“双人关系对象”，再给解释，不应直接把用户扔进空白 AI 对话框。

### 不吸收

- 上传截图作为主路径。
- 真实双人合盘结果页未验证却写成已验证。
- `选择完成` 后续行为未验证却写成结果页入口。
- `合盘 -> AI问 / 在线 / 真人服务` 未验证却写成完整桥接链。
- 真人玄学咨询市场、提问人数、好评率、付费报告等服务市场元素。

---

## 5. 全局产品边界

所有 5.6+ 玄学 / 关系解释输出必须保持：

```text
derived-only / display-only / explanation layer
```

不得反写：

```text
profile/basic
profile/astro/summary
profile/astro/chart
user_astro_profiles
```

不得改变：

- canonical truth；
- profile contract；
- astro contract；
- match algorithm contract；
- release contract；
- API contract；
- 数据库 schema；
- migration；
- 版本号与发版链。

不得提出：

- 真人玄学咨询市场；
- 付费报告商店；
- 上传截图主路径；
- 强出生资料收集；
- 娱乐化测试主路径；
- 测试结果反写画像；
- 大规模社区讨论；
- 测测术语体系复制；
- 从竞品 UI 推断服务端 contract、算法、接口或数据库。

## 6.2 5.7：Match 关系解释层最小产品化版

### 定位

5.7 是 5.6+ 路线批准后的第一个 runtime 候选版本。

### 核心目标

在 Match detail 中增加克制的关系解释层，帮助用户理解：

- 为什么推荐这个人；
- 可以从哪里开场；
- 哪些表达方式适合；
- 哪些表达方式应避免。

### 建议结构

- 标题：`为什么值得聊`；
- 一句关系摘要；
- 2–4 个解释维度；
- 建议 / 避免；
- 轻追问 disabled / coming soon 或受控占位；
- display-only 免责声明。

### 建议首版维度

- 共同点；
- 表达节奏；
- 慢约会适配度；
- 开场友好度。

### 不做

- 不接真实 AI。
- 不新增 API。
- 不新增 DB 字段。
- 不改匹配算法。
- 不写 profile。
- 不写聊天消息。
- 不自动发送。
- 不上传截图。
- 不接真人服务。
- 不做付费报告。

## 6.3 5.8：Me 个人解释层与表达建议版

### 定位

5.8 将解释层从 Match 扩展到 Me / Profile，但因为靠近资料真值，必须更保守。

### 核心目标

帮助用户理解：

- 我如何被看见；
- 我的表达风格是否清晰；
- 资料是否支持慢约会；
- 自我介绍和展示是否需要优化。

### 建议结构

- 我的表达风格；
- 资料展示建议；
- 慢约会友好表达；
- 可以补充什么；
- display-only 免责声明；
- 本地草稿或文案建议入口。

### 不做

- 不重做星盘图谱。
- 不强收出生资料。
- 不深度人格诊断。
- 不自动修改资料。
- 不写 profile。
- 不改 `user_astro_profiles`。
- 不接付费报告。
- 不接真人服务。

## 6.4 5.9：Chat 轻追问与低压开场版

### 定位

5.9 将 Match / Me 的解释层轻量衔接到 Chat，但不做自动代聊。

### 核心目标

帮助用户：

- 在首聊时找到低压开场；
- 冷场时换一种说法；
- 从关系解释转化为可编辑草稿；
- 不让 AI 代替用户表达。

### 建议结构

- 从共同点开场；
- 换个更自然的说法；
- 低压问候建议；
- 不要太急的提醒；
- 可编辑草稿候选；
- 用户确认后才可发送。

### 不做

- 不自动发送。
- 不自动代聊。
- 不读取未授权私密聊天内容。
- 不上传截图。
- 不对对方心理做确定判断。
- 不把 AI 输出写回用户画像。
- 不接真人咨询。

## 6.5 5.10：解释层治理、用户控制与回归校准版

### 定位

### 建议结构

- AI 辅助建议开关；
- 解释层免责声明；
- 关闭后仍可使用主流程；
- 不写入资料说明；
- Match / Me / Chat 回归清单。

### 不做

## 9. Blocker 与停止条件

任一后续版本如出现以下情况，必须停止并回到计划阶段：

- 需要改 API 才能实现；
- 需要新增数据库字段或 migration；
- 需要写 `profile/basic`、`profile/astro/summary`、`profile/astro/chart`、`user_astro_profiles`；
- 需要改变 Match 推荐算法输入；
- 需要自动写聊天消息或自动发送；
- 需要上传截图、图片或文件；
- 需要接真人服务、付费报告或咨询市场；
- 需要把竞品 UI 术语或链路直接复制到 EliteSync；
- 需要使用未验证的竞品路径作为事实前提。

## 2. 测测研究抽象

测测 Stage 1–5 被抽象为五层：

1. **入口层**：首页入口、频道入口、AI问入口、星盘/合盘入口。
2. **上下文层**：单人档案、双人档案、测试上下文、问题上下文。
3. **解释对象层**：星盘、合盘、生辰、紫微、测试卡、关系总览。
4. **解释 / 建议层**：结论句、维度评分、建议 / 避免、免责声明、分层解释。
5. **追问 / 转化层**：AI 追问、上传截图、真人服务、在线服务、更多解读。

EliteSync 重点吸收前四层：

- 入口层：能力入口嵌入 Match、Me、Chat 等主链。
- 上下文层：单人解释依赖个人资料上下文，关系解释依赖双人匹配上下文。
- 解释对象层：把星盘、合盘、生辰、紫微等转译为关系理解、表达偏好、匹配理由。
- 解释 / 建议层：用短句、维度、建议 / 避免组织可读结果。

第五层必须克制：

- 可保留轻追问；
- 不默认接真人服务；
- 不上传截图作为主路径；
- 不做付费报告；
- 不把入口可见写成完整链路已验证。

## 4. 5.7：Match 关系解释层最小产品化版

### 4.1 版本定位

5.7 是 5.6+ 路线批准后的第一个 runtime 候选版本。

### 4.2 核心目标

在 Match detail 中增加克制的关系解释层，帮助用户理解：

- 为什么推荐这个人；
- 可以从哪里开场；
- 哪些表达方式适合；
- 哪些表达方式应避免。

### 4.3 功能边界

建议结构：

- `为什么值得聊`；
- 一句关系摘要；
- 2–4 个解释维度；
- 建议 / 避免；
- 轻追问 disabled / coming soon 或受控占位；
- display-only 免责声明。

建议首版维度：

- 共同点；
- 表达节奏；
- 慢约会适配度；
- 开场友好度。

### 4.4 验收重点

- Match detail 页面可见关系解释层。
- 解释层不影响原有匹配算法。
- 解释层不写 profile / astro / match truth。
- 文案不做确定性诊断，不做预测，不制造伪真值。
- 轻追问只做占位或受控展示。

---

## 5. 5.8：Me 个人解释层与表达建议版

### 5.1 版本定位

5.8 将解释层从 Match 扩展到 Me / Profile，但因靠近资料真值，必须更保守。

### 5.2 核心目标

帮助用户理解：

- 我如何被看见；
- 我的表达风格是否清晰；
- 资料是否支持慢约会；
- 自我介绍和展示是否需要优化。

### 5.3 功能边界

建议结构：

- 我的表达风格；
- 资料展示建议；
- 慢约会友好表达；
- 可以补充什么；
- display-only 免责声明；
- 本地草稿或文案建议入口。

### 5.4 验收重点

- Me 页新增解释层但不替代资料页。
- 资料建议不自动修改资料。
- AI 建议不写回 profile。
- 个人解释不做绝对人格判断。

---

## 6. 5.9：Chat 轻追问与低压开场版

### 6.1 版本定位

5.9 将 Match / Me 的解释层轻量衔接到 Chat，但不做自动代聊。

### 6.2 核心目标

帮助用户：

- 在首聊时找到低压开场；
- 冷场时换一种说法；
- 从关系解释转化为可编辑草稿；
- 不让 AI 代替用户表达。

### 6.3 功能边界

建议结构：

- 从共同点开场；
- 换个更自然的说法；
- 低压问候建议；
- 不要太急的提醒；
- 可编辑草稿候选。

### 6.4 验收重点

- 草稿可编辑。
- 不自动发送。
- 不读取未授权私密聊天内容。
- 不对对方心理做确定判断。
- 不把 AI 输出写回用户画像。

---

## 7. 5.10：解释层治理与用户控制版

### 7.1 版本定位

5.10 是治理与统一控制版本，用于收口 5.7–5.9 的解释层能力。

### 7.2 核心目标

- 给用户统一控制权。
- 固化免责声明。
- 统一解释层开关。
- 统一回归清单。
- 防止解释层膨胀成伪真值、伪诊断、伪 AI 决策。

### 7.3 功能边界

建议结构：

- 关系解释提示开关；
- AI 辅助建议开关；
- 解释层免责声明；
- 关闭后仍可使用主流程；
- 不写入资料说明。

### 7.4 验收重点

- 用户可关闭解释层提示。
- 关闭后主流程不受影响。
- Match / Me / Chat 都有回归证据。
- 不引入付费、真人服务、上传分析、算法训练或社区讨论系统。
