## 1. Smoke 执行环境与前置条件

允许环境：

- local controlled backend：只允许连接本地或隔离测试 DB，不得连接 production DB。
- staging backend：推荐默认 smoke 环境，要求 DB、Redis、storage、release chain 均可独立回滚或隔离。
- 明确授权的 controlled backend：必须由用户确认 base URL、APP_ENV、DB 指向、账号和写入范围。

默认禁止：

- production 真实用户链路。
- 真实用户账号。
- 不可恢复写入。
- 未经确认的远端 DB、真实 media storage、真实 notification、真实 RTC 会话。
- 将任何 token、手机号、私聊正文、精确经纬度、完整出生时间写入报告或 git。

smoke 账号要求：

- 至少两个 smoke 账号：`SMOKE_USER_A` 与 `SMOKE_USER_B`。
- 必须明确 peer user：`SMOKE_USER_B` 是 `SMOKE_USER_A` 的可聊天 / 可 RTC 对象。
- 必须确认是否已有 conversation、media asset、notification、questionnaire baseline、RTC 测试数据。
- 必须确认是否允许创建 RTC call。
- 必须确认是否允许覆盖 profile basic；如允许，需记录执行前 baseline 与恢复方式。

token 安全：

- token 只保存在当前 shell session，例如 `$env:ELITESYNC_TOKEN_A` 或临时变量。
- token 不写入报告。
- token 不进入截图。
- token 不进入 git。
- 任何失败日志都必须脱敏。

执行前必须检查：

- `git status --short` clean。
- 当前 base URL，使用 `<BASE_URL>` 占位记录，不写真实生产 URL。
- 当前 `APP_ENV` 是 local / staging / controlled，而非 production。
- 当前 DB 是 staging / controlled DB，不是 production DB。
- 当前 release chain 只读，不改 versionName / versionCode / app_update / release script。
- smoke 账号、peer user、asset、notification、call 的前置数据均已确认。

执行后必须输出：

- 响应摘要。
- 不输出隐私字段。
- 不输出 token。
- 不输出完整出生时间、手机号、私聊正文、精确经纬度。
- 写 DB smoke 必须记录是否 cleanup 或是否可由 smoke 账号自然覆盖。

## 5. Smoke 数据准备清单

| 准备项 | 必填 | 当前要求 |
| --- | --- | --- |
| smoke user A | 是 | 记录 user id / display name 脱敏标识，不记录手机号或密码 |
| smoke user B | 是 | 记录 user id / display name 脱敏标识，不记录手机号或密码 |
| A/B 是否互为可聊天对象 | 是 | 明确是否有 dating match、conversation 或 allow-list |
| 是否已有 conversation | 是 | 若无，可由 messages write smoke 创建；需说明 cleanup |
| 是否已有 media asset | 是 | 若无，可由 media write smoke 创建；需说明 storage 是否隔离 |
| 是否已有 notification | 是 | 若无，可由 messages/RTC create 触发；不能读真实用户通知 |
| 是否已有 questionnaire baseline | 是 | 记录是否允许新增 attempt，是否允许重置 smoke profile |
| 是否允许创建 RTC call | 是 | 必须确保只通知 smoke peer，不触达真实用户 |
| 是否允许 profile basic 被覆盖 | 是 | 若允许，必须记录执行前 baseline 与恢复方案 |
| 每个写入 smoke 的回收 / 复原方式 | 是 | message/media/questionnaire/rtc/notification/profile 分别记录 |

写入 smoke cleanup 原则：

- profile：恢复执行前 smoke profile baseline。
- messages：保留 smoke message 作为测试数据或在 controlled DB cleanup。

## 8. 失败与停止条件

任一情况出现，停止后续 smoke，不得进入 v2 skeleton：

- app health 失败。
- version check 结构破坏。
- auth token 异常。
- profile truth 写入异常。
- astro summary/chart 失败。
- messages 写入或 read receipt 异常。
- media 写入不可清理。
- RTC LiveKit token 泄露到报告。
- notification read 误读真实用户通知。
- release chain 版本不一致。
- 误打 production 真实用户链路。
- token / 手机号 / 私聊正文 / 精确经纬度进入报告。
- git status 不干净且无法解释。
- 写 DB smoke 无 smoke 账号、无可恢复数据集或无 cleanup 方案。
- 任何 smoke 需要修改 runtime、DB schema、release chain 或 production config。

## 9. Skeleton 关系判定

- v1 P0 smoke package 完成后，是否允许执行 smoke：`conditional`。只允许在用户确认 base URL、APP_ENV、DB、smoke 账号、token 处理、写入范围和 cleanup 后执行。
- smoke 执行通过前，是否允许 v2 skeleton：`no`。
- 如果只做只读 v2 health/readiness/location skeleton，仍建议至少先通过：
  - `V1-P0-01` app health；
  - `V1-P0-02` app version check；
  - `V1-P0-03` profile basic read；
  - `V1-P0-05` astro summary；
  - `V1-P0-06` astro chart；
  - `V1-P0-11` questionnaire history；
  - `V1-P0-16` notification unread-count；
  - `V1-P0-18` release chain readonly check。
- 可后置的 write smoke：profile basic write、messages send、media create、questionnaire answers、RTC create、RTC heartbeat、notification read；但它们后置时必须形成 observation，且不得进入任何写接口 skeleton。
- skeleton 前硬门禁：`V1-P0-01` 到 `V1-P0-18` 的执行结论必须明确为 pass / skipped with approved reason / blocked with stop decision；其中任何 fail 都阻断 skeleton。
