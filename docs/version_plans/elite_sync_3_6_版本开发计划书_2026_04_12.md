# EliteSync 3.6 版本开发计划书

更新时间：2026-04-12  
适用对象：项目负责人 / Codex / Gemini-MCP / Claude-mcp / 顾问归档  
版本状态：草案 v1  
当前前置版本：`3.5` 已正式验收通过  
版本定位：`古典 / 现代双路线版`

---

## 0. 本文档定位

本文档用于在 `3.5` 完成归档后，为 EliteSync 制定下一阶段 `3.6` 的正式开发计划。

`3.6` 不是对 `3.5` 的零散补丁，也不是试图直接跨到 `3.7` 的高级推运版。  
它的定位是：

> **在不破坏当前稳定产品基线与 canonical 真值边界的前提下，把 Astro Engine 从“参数化基础版”推进到“古典 / 现代双路线版”，补齐更完整的参数能力、路线分流和解释层。**

本版主要解决：

1. `3.5` 已建立工作台 MVP，但古典 / 现代只停留在轻量预设层，尚未形成清晰的双路线口径。
2. 用户仍缺少更完整的宫位制、点位、相位、容许度与路线差异解释。
3. 样例对照虽已建立，但还不足以支撑更高颗粒度的路线比较与解释输出。
4. Claude-cli-mcp 链路已恢复，`3.6` 应把 Claude 正式纳入可归档的架构审查链，而不是仅停留在规则层面。

本版明确不解决：

- 合盘、男女对比盘
- 行运、三限、次限、月返、日返
- 法达、小限、大限、流年
- 全量高级解释系统一次性落地
- 高风险 canonical 引擎全面替换

## 1. 为什么 3.6 必须单独立项

根据当前总计划与交接链路，`3.5` 已完成：

- Engine Adapter 与 schema / metadata 基础设施
- 参数工作台 MVP
- 盘面精细绘制 MVP
- 参数解读最小版
- 固定样例对照与差异报告
- smoke / regression / rollback / 归档链路

这意味着 `3.5` 已经把“能不能做参数化星盘工作台”这个问题解决了。  
而 `3.6` 要解决的是另一层问题：

1. 如何把古典与现代做成**清晰、可解释、可比较**的双路线，而不是几个名字不同的预设按钮。
2. 如何在不破坏 canonical 真值主线的前提下，逐步引入更细的宫位、点位、相位与解释层。
3. 如何把路线分流、参数扩展、解释增强与样例回归组织成一套可持续迭代的机制。

因此：

`3.6` 应被视为星盘引擎升级第二阶段，而不是 `3.5` 的补丁集合。

## 3. 3.6 目标

### 3.1 本版要解决的问题

当前 App 在 `3.5` 之后已经能：

- 通过参数工作台控制基础显示与工作台参数
- 展示固定样例对照与差异摘要
- 维持服务端 canonical 真值与本地显示偏好的边界

但还不能：

- 以清楚的产品口径告诉用户“古典路线”和“现代路线”到底差在哪里
- 为古典 / 现代分别提供更完整的默认参数、点位范围、相位集合与摘要解释
- 输出更高颗粒度的路线对照结果，例如宫主星、主星落宫、宫主飞宫、路线摘要
- 用多组固定样例稳定回归古典 / 现代双路线的差异与兼容性

### 3.2 本版最终应达到的形态

`3.6` 结束后，EliteSync 应达到：

> **在现有稳定产品中形成受控的古典 / 现代双路线能力：用户可切换路线、读取更完整参数与解释摘要、查看更细的对照结果，并且整个链路仍保持 canonical 真值边界、兼容旧入口、可回归可归档。**

## 4. 3.6 P0（必须完成）

## 4.1 古典 / 现代双路线骨架

目标：
- 让“古典 / 现代”从预设名称升级为明确的路线口径
- 路线差异要能在 schema、解释和样例回归中被追踪

必须完成：
1. 建立 `route_mode` 或等价路线标识，至少覆盖：
   - `classical`
   - `modern`
   - `standard`（用于兼容当前默认口径）
2. 明确每条路线的默认参数组合：
   - 黄道制口径
   - 宫位制口径
   - 点位范围
   - 相位集合
   - 容许度策略
3. 在 `engine_info` / `metadata` 或等价元信息中补齐路线标记与版本信息
4. 路线切换必须明确说明：
   - 哪些是路线口径
   - 哪些是显示偏好
   - 哪些仍未进入 canonical 主链

说明：
- `3.6` 可以扩展路线输出与工作台结果，但**不得默默把用户 canonical 主画像切换为某条新路线的结果**。

---

## 4.2 参数能力扩展（双路线所需的第一批）

目标：
- 把 `3.5` 中计划后移的参数能力补成双路线的可用基础

必须完成：
1. 在现有核心宫位制基础上，补充更多常用宫位制
2. 在现有点位范围基础上，扩展：
   - 虚点
   - 常用敏感点
   - 第一批常用小行星（只纳入明确列入路线设计的最小集合）
3. 在现有相位密度基础上，补更细的相位分组与筛选
4. 将容许度从“单档位提示”升级为更清楚的路线化策略
5. 对每个新增参数写清：
   - 路线默认值
   - 可选范围
   - 是否影响当前工作台结果
   - 是否只影响本地展示

说明：
- 本版要做的是“受控扩展”，不是把所有专业参数一次性开放。

---

## 4.3 更完整的解释层（MVP+）

目标：
- 让古典 / 现代不仅有参数差异，还能给出结构化阅读结果

必须完成：
1. 增加路线摘要卡：
   - 当前路线
   - 当前路线默认参数摘要
   - 与标准路线相比的关键差异
2. 增加第一批结构化解释：
   - 宫主星
   - 主星落宫
   - 宫主飞宫
3. 增加路线差异摘要：
   - 为什么古典更收敛
   - 为什么现代更开放
   - 当前盘面哪些结果来自路线差异
4. 增加“已知边界提示”：
   - 当前仍是工作台结果还是已接入服务端更深层计算
   - 当前解释层覆盖到什么程度

说明：
- 本版仍然不是全量专业解释系统，但必须让用户感觉“路线不同，阅读结果也真的不同”。

---

## 4.4 双路线工作台与盘面联动

目标：
- 让路线切换、参数变化、盘面刷新和摘要解释形成闭环

必须完成：
1. 在参数工作台中提供清晰的路线入口
2. 路线切换后，盘面摘要、相位数量、点位数量、解释摘要要联动刷新
3. 本命盘详情页需要新增或强化：
   - 当前路线标签
   - 当前路线摘要
   - 路线与显示偏好的边界提示
4. 对路线切换后的视觉复杂度做节制，避免盘面重新变成拥挤噪音

---

## 4.5 扩展样例对照与路线回归

目标：
- 让古典 / 现代双路线的差异不只停留在单一样例

必须完成：
1. 将固定样例集从单一样例扩展到多组样例
2. 每组样例至少对照：
   - 星体落座
   - 宫位落点
   - 主要相位
   - 角点
   - 第一批解释字段
3. 为古典 / 现代 / 标准三条路线生成差异矩阵
4. 形成可归档材料：
   - `ROUTE_CAPABILITY_MATRIX`
   - `ROUTE_PARITY_REPORT`
   - `KNOWN_DEVIATIONS`
5. 回归脚本 / 测试清单必须可重复执行

## 7. 3.6 长期约束（不可违反）

### 7.1 真值约束

1. 服务端 canonical 真值仍是唯一真源。
2. Flutter 本地绘制与本地偏好不得回写真值层。
3. 古典 / 现代路线切换必须明确标记自己属于：
   - canonical 主链
   - 派生结果
   - display-only
4. 任何新解释字段都必须写清来源，不得伪装成已验证的 canonical 事实。

### 7.2 架构约束

1. 不得因为双路线而把现有稳定引擎链路整体推翻。
2. 必须继续坚持 additive change，不删旧字段、不破坏旧读取路径。
3. 任何新路线字段都必须附带向后兼容策略。
4. 不得在没有样例回归矩阵的情况下凭肉眼硬调路线结果。

### 7.3 UI / 工作流约束

1. 本版继续属于 UI 重构型版本，**Gemini-MCP 仍为强制 subagent**。
2. 由于 Claude-cli-mcp 链路已恢复，**Claude-mcp 本版必须留下可归档的架构审查记录**。
3. Codex 不得跳过 Gemini-MCP 的阶段验收与最终验收。
4. Codex 不得跳过 Claude-mcp 的关键架构审查就宣告双路线结构定稿。

### 7.4 发布与回归约束

1. 所有高风险改动都要配套 smoke / regression / rollback 思维。
2. 不得破坏 `3.5` 稳定基线。
3. 旧资料页、旧星盘页、旧设置页、旧版本中心入口必须可回归。
4. 新路线结果不能阻塞旧客户端或旧缓存的读取。

## 10. 多 Agent 分工（本版重点）

### 10.1 Codex

Codex 仍为本版主执行人，负责：
- 计划拆解
- 风险检查
- 编码实现
- 文档归档
- 回归与验收材料整理
- 调度 Gemini-MCP 与 Claude-mcp

Codex 不得：
- 擅自把双路线版扩成推运版
- 在没有样例矩阵的情况下硬调路线差异
- 跳过 Gemini-MCP 的 UI 审查
- 跳过 Claude-mcp 的架构审查
- 未经论证直接替换 canonical 引擎

### 10.2 Gemini-MCP（本版强制）

**Gemini-MCP 是本版强制 subagent。**

Gemini-MCP 负责：
1. 路线入口与参数工作台的信息架构审查
2. 盘面可读性与视觉复杂度审查
3. 路线摘要、解释卡、提示层的可理解性审查
4. 阶段验收与最终验收
5. 正式截图包的视觉复核

Gemini-MCP 不负责：
- 决定底层算法逻辑
- 主导服务端架构改写
- 取代 Codex 完成主实现

### 10.3 Claude-mcp（本版强制留痕）

**Claude-mcp 是本版关键架构 subagent，必须留下可归档审查记录。**

Claude-mcp 负责：
1. 双路线 schema 设计
2. canonical 真值与路线派生字段边界
3. 向后兼容策略
4. 路线能力矩阵设计
5. 多样例回归框架设计

- 日常 UI 微调
