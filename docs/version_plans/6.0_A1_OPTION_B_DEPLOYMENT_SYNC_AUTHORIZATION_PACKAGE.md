## 2. Future sync objective

A future deployment / sync execution should be limited to bringing the already repo-verified R1 readonly v2 route/controller/test surface into the Alibaba Cloud deployed Laravel tree so that a later localhost-only verification rerun can be prepared or executed under a separate gate.

The future objective is narrow:

- synchronize only the R1 readonly v2 route/controller/test surface needed for:
  - `GET api/v2/app/health`
  - `GET api/v2/app/readiness`
  - `GET api/v2/contracts/location`
- preserve existing v1 production behavior;
- avoid DB, migration, composer, cache, queue, Nginx, Flutter, Android, and release-chain mutations unless separately authorized;
- prepare for a later localhost verification rerun, not perform endpoint verification automatically.

The future execution issue must first confirm the exact file list before writing to the server. If the required write set differs from this allowlist, the execution must stop and return to GPT advisor / user decision.

## 4. Future allowed execution shape

After separate user authorization, a future execution issue may allow only:

1. local repo baseline checks;
2. remote backup of the specific target files or directories before overwrite;
3. exact file-list comparison between the repo and deployed tree;
4. controlled sync / copy of the allowlisted files only;
5. readonly syntax checks limited to changed PHP files if available;
6. readonly route inventory check if safe and authorized in that future issue;
7. a redacted deployment / sync execution report.

The future execution should keep each command scoped, redacted, and attributable to the issue that authorizes it.

## 5. Future forbidden scope

Even if deployment / sync is authorized later, the following remain forbidden unless separately authorized:

- full repository deployment;
- broad rsync of the entire repo;
- DB / migration / seed;
- composer install / composer update;
- cache clear / route clear / config clear / optimize clear;
- Nginx reload / restart;
- endpoint verification / curl;
- public staging request;
- production request;
- Flutter / Android / release-chain changes;
- modification of `.env` or credentials;
- reading or outputting credential content;
- runtime features beyond the R1 readonly v2 slice;
- A2 / Date Drop / Candidate C implementation.

## 6. Stop conditions for future execution

Future deployment / sync execution must stop if:

- local repo worktree is dirty;
- target server mismatches the authorized server;
- target path mismatches the authorized deployed Laravel path;
- backup cannot be created;
- intended write set exceeds the allowlist;
- diff indicates unexpected production/default config changes;
- a command would read or output credentials;
- a command would modify DB, cache, composer dependencies, release chain, or Nginx;
- a command would require endpoint verification to continue;
- a command would require broad path discovery or broad file copy.

Stopping on these conditions should produce a blocker or observation report instead of attempting an unreviewed fix.

## 7. Future backup and rollback plan

Future execution must document:

- exact target paths;
- backup paths;
- SHA256 or size/timestamp summary for files where practical;
- rollback command plan for restoring pre-sync files;
- a statement that rollback itself is not executed unless needed and authorized by the future issue.

The backup should cover the pre-sync state of every target file that may be overwritten and any parent directory created by the sync if applicable.

## 8. Future evidence report plan

A future deployment / sync execution report should include:

- issue number;
- branch / HEAD;
- starting remote file presence;
- exact file list synced;
- backup summary;
- syntax / route inventory summary if run;
- what was not done;
- next recommended gate.

The report must remain text-first and redacted. It must not include real `.env` contents, credentials, full Nginx config bodies, tokens, cookies, private user data, or ordinary screenshots.

```text

```

Do not automatically rerun endpoint verification in the deployment / sync task.
