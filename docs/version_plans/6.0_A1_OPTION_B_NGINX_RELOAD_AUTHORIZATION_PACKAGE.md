## 2. Future reload execution goal

If separately authorized by the user, a future issue may:

1. run local repo startup checks;
2. SSH to the authorized EliteSync server;
3. confirm symlink still exists and targets `/etc/nginx/sites-available/elitesync_staging_ip.conf`;
4. run `sudo nginx -t` again immediately before reload;
5. if syntax passes with no warning, run exactly `sudo systemctl reload nginx`;
6. confirm reload command exit status;
7. stop without endpoint verification;
8. write a redacted reload execution report.

## 3. Future allowed scope

Future reload execution may allow only:

```text
SSH to authorized server
redacted symlink existence check
redacted nginx -t
sudo systemctl reload nginx
redacted reload command result
write reload execution report
```

## 4. Future forbidden scope

Even if reload is authorized, still forbid:

- Nginx restart;
- endpoint verification;
- staging request;
- production request;
- SSH tunnel creation;
- `curl http://127.0.0.1:8088`;
- `curl slowdate.top`;
- `curl 101.133.161.203`;
- API / smoke / artisan / PHPUnit / composer / migration;
- `.env` / secret reads;
- modifying Nginx config files;
- modifying production/default server block;
- modifying security group / firewall;
- modifying runtime / tests / DB / Flutter / release chain;
- claiming staging verified or production verified.

## 5. Stop conditions before future reload

Future reload execution must stop if:

- local repo worktree is dirty;
- SSH target mismatch;
- symlink missing;
- symlink target mismatch;
- `nginx -t` warning;
- `nginx -t` failure;
- reload command would require restart;
- any command would read `.env`;
- any command would output secrets;
- endpoint verification is needed to continue;
- staging / production request is needed to continue;
- production/default config appears at risk.

## 6. Future rollback / recovery note

After reload, rollback is not the same as deleting a symlink silently. A future rollback issue would need explicit authorization and should likely:

1. remove the symlink;
2. run `nginx -t`;
3. reload Nginx again only if explicitly authorized;
4. document whether staging was disabled again.

Do not execute rollback in this preparation task.
