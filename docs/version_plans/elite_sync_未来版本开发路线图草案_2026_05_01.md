# EliteSync 未来版本开发路线图草案

更新时间：2026-05-01
定位：5.0 起产品能力补强与测试前演进路线草案
状态：讨论稿

---

## 1. 起点判断

当前项目不应回头重开 4.3~4.9 已收口主链，而应在 4.9 的稳定化、治理、可观测性与发布链基线上继续向前推进。当前可承接的新路线，必须默认继承：现代 UI protected surfaces、rollback / recovery policy、smoke / release gate / health / DB safety、媒体链与 RTC 语音主链、通知中心主链，不得为了补新功能破坏这些基线。

同时，整体后端路线继续坚持“可扩展单体”而非提前微服务化：保留单体主干，继续加厚媒体、缓存、异步任务、日志监控、限流、实时信令预留等关键扩展点，不因为“未来百万级”预期而过早大重构。

---

## 2. 新路线的总体目标

未来版本不再按“哪里缺一个功能就补一个功能”的方式推进，而改为三条并行但有主次的路线：

1. **产品能力补强线**：把首页 / 发现 / 聊天 / 个人页从“已可用骨架”升级到“更像产品”的状态。
2. **关系推进与 AI 辅助线**：把匹配解释、首聊草稿、聊天页、动态页、个人页中的 AI 能力真正串起来。
3. **测试前工程化线**：继续补 smoke、baseline、runbook、失败态证据与云端运维治理，但不再把它当成唯一主题。

这意味着 5.x 的版本规划要在“保持 4.9 基线不破”的前提下，重新回到产品面，但只做对未来真实测试一定会保留的能力。

---

## 3. 来自 Soul 拆解后的产品结论（映射到 EliteSync）

### 3.1 必须吸收的结构结论

- **发现页不是纯 Feed，而是复合入口层**：推荐 / 同城 / 搜索 / 分享 / moderation 应在一个统一 Discover 内协同组织。
- **聊天不是纯 inbox，而是关系推进层**：首聊 / 回聊、关系摘要、查看主页、输入区 AI、通话升级入口应自然联动。
- **个人页不是静态资料页，而是用户经营中枢**：资料展示、标签表达、内容经营、AI 建议、功能中心要在同一个“自己”页内形成闭环。
- **设置与个人空间外观应分层**：真正设置中心继续保留，但个人空间外观、背景、展示风格可作为轻量二级层，不应污染主设置路径。

### 3.2 明确不作为当前高优先级的方向

- 币体系、商城、复杂装扮、成熟平台玩法中心
- 互动小游戏/强娱乐化分支
- 大规模推荐系统重构
- 多人 RTC、直播、复杂视频通话增强
- 为未来超大规模并发提前做微服务重构

---

## 4. 新的版本路线建议

## 4.1 5.0：Alpha 测试候选版（产品化补强最小集）

### 版本定位

5.0 不再定义为“继续稳定化”，而定义为：

> **Alpha 测试候选版 / 产品化补强最小集**

### 核心目标

在不破坏 4.9 治理与稳定化基线的前提下，完成最小但高价值的产品增强，让 EliteSync 从“功能骨架齐”进一步进入“可以带小规模真实用户试用”的状态。

### 本版应做

#### A. Discover 发现页产品化
- 顶部分栏明确化：建议至少形成 `推荐 / 同城 / 关注或话题` 三分结构
- 搜索页产品化：搜索框 + 热词 + 提示卡，不再只是空入口
- 卡片 action sheet 统一化：分享 / 举报 / 不喜欢 / 复制链接 / 轻互动统一收口
- 同城内容分支成型，但权限按需申请，不先弹定位权限

#### B. Chat 聊天页关系推进增强
- 首聊 / 回聊辅助层：形成“稍后继续了解 / 再打个招呼”的安全回流队列
- 聊天页关系摘要条：共同点、关系提示、沟通建议、查看主页入口
- 输入区 AI 轻集成：一键插入建议问候、续话建议、基于匹配解释的追问建议
- 聊天页与语音入口轻联动：保持 1v1 语音已通闭环，但只做更自然入口，不重写 RTC 主链

#### C. Me / Profile 个人经营页增强
- 标签表达体系升级：人格、关系风格、生活状态、交友态度共同表达
- 个人页从资料归档页升级成“内容经营页”
- AI 助理最小版：资料完整度 / 展示质量 / 首聊友好度 / 状态建议
- 声音名片或轻语音自我介绍作为候选最小项（二选一，不铺太大）

### 本版不做
- 复杂会员体系
- 装扮商城 / 币中心
- 多人音视频 / 视频通话增强
- 大推荐算法改造
- 运营后台大扩张

### 验收口径
- 新增能力必须与 4.9 UI baseline 共存，不退回旧 UI
- Discover / Chat / Me 三个主模块新增 UI 必须都有 screenshot baseline
- AI 建议类能力必须可关闭、可跳过、不可伪成功
- 所有新增入口都要有空态 / 失败态 / 回读路径

---

## 4.2 5.1：关系转化与内容回流增强版

### 版本定位

> **关系转化增强版 / 动态与聊天联动版**

### 核心目标

把 5.0 里的产品化增强继续推进到“能提高开口率、回聊率、资料可读性”的层面，但仍然不进入重运营和大社区路线。

### 本版应做
- 状态流与聊天联动：从状态卡可进入“问一句 / 继续了解 / 收藏稍后聊”
- 个人页的内容经营反馈：谁看过我、哪些资料更吸引人、哪些状态更易获得回应（轻量，不做强排行榜）
- 匹配解释与聊天建议联动深化：从匹配结果直接产出不同风格的首聊模板
- 通知中心文案产品化：从工程状态型提醒进一步升级成产品语言
- 轻量收藏 / 稍后看 / 稍后聊体系补齐

### 本版不做
- 独立内容社区大改版
- 强公共广场运营体系
- 复杂实验平台
- 复杂付费漏斗

### 验收口径
- 以开口率 / 回聊链完整度 / 页面回读清晰度为主要产品验收维度
- 通知、聊天、动态、个人页之间的跳转必须可截图复验
- 不允许把产品提示写成工程日志式文案

---

## 4.3 5.2：个人空间与表达层增强版

### 版本定位

> **轻量个人空间版 / 表达层增强版**

### 核心目标

在核心关系链已经更顺的前提下，再补用户的个性化表达层，但控制范围，不把项目带向重装扮平台。

### 本版应做
- 个人空间外观层最小版：背景、展示样式、声音名片封面等轻量配置
- 资料展示模版化：不同人格/关系风格可映射不同展示模版
- AI 帮发 / AI 帮写状态最小版：服务“低压表达”，不是做内容工厂
- 搜索 / 热词 / 提示卡可加入与慢约会更相关的主题词

### 本版不做
- 商城
- 虚拟货币
- 超级会员装扮权益
- 重娱乐玩法

### 验收口径
- 所有外观能力都必须是可选增强，不影响主功能使用
- 相册 / 录音 / 媒体权限全部按需申请
- 视觉增强不能破坏 4.7 之后建立的 protected UI surfaces 基线

---

## 4.4 5.3：测试运营准备与云端治理增强版

### 版本定位

> **测试运营准备版 / 云端治理增强版**

### 核心目标

在产品面已经达到 Alpha 可试状态后，补齐真正面对小规模真实测试时最容易出问题的运维与治理能力。

### 本版应做
- 业务管理后台最小版：用户、资料、匹配、消息、状态、通知、举报的只读/轻治理视图
- 云端只读优先数据库访问与 runbook 固化
- 版本、健康、队列、Worker、媒体、RTC、通知链的统一观测入口
- 失败态证据补齐：媒体 / 队列 / RTC / Heartbeat / 通知的失败路径
- 测试账号与 synthetic 账号治理体验补强

### 本版不做
- 大而全运营中台
- 任意写库型桌面工具
- 生产库直连乱改
- 提前微服务化

### 验收口径
- 所有后端管理动作默认云端执行，本地仍只做前端/UI/截图/文档
- 只读优先、审计优先、恢复优先
- 至少形成一版可给团队复用的运维/发布/恢复统一手册

---

## 4.5 5.4：Alpha 后反馈吸收版（预留）

### 版本定位

> **真实小样本反馈吸收版**

这是预留版，不预先写死大量功能。只有在 5.0~5.3 之后确实进入小规模真实试用，才根据用户反馈决定是否：

- 调整首聊建议策略
- 调整标签体系
- 调整同城与推荐分栏
- 调整 AI 助理提示密度
- 调整通知、回流和关系推进节奏

原则是：**先收用户反馈，再动结构，不预先脑补一个 5.4 巨型功能包。**

---

## 5. 三条并行工作包（贯穿 5.x）

### 5.1 产品工作包
- Discover 分栏、搜索、同城、action sheet
- Chat 首聊/回聊、关系摘要、AI 输入建议
- Me 经营中枢、标签体系、AI 助理、轻语音表达

### 5.2 工程工作包
- 持续保护 4.9 release gate 基线
- 补齐失败路径证据
- 继续加厚缓存、队列、媒体与 RTC 可观测性
- 云端 runbook 与数据库恢复演练制度化

### 5.3 设计与验收工作包
- 所有新页面都必须提供现代 UI 截图基线
- Gemini 继续负责视觉与验收观察
- Claude 仅在跨层 blocker、结构判断、根因缩层时介入
- Codex 保持主执行位，但严禁 repo 级回滚与跨层盲修

---

## 6. 新路线的优先级排序

如果资源有限，建议严格按下面顺序推进：

1. **5.0：Discover / Chat / Me 的最小产品化补强**
2. **5.1：关系转化与内容回流增强**
3. **5.3：测试运营准备与云端治理增强**
4. **5.2：个人空间与表达层增强**
5. **5.4：真实反馈吸收版**

这里刻意把 5.3 提前于 5.2 的原因是：如果真的要进入小规模测试，治理、runbook、观测、恢复往往比“更漂亮的空间装扮”更关键。

---

## 7. 对团队的执行提醒

### 7.1 后续计划书必须继续继承的硬规则
- 不重开已归档版本主链
- 继续从 4.9 additive baseline 承接
- 本地只做前端开发、UI 联调、截图采集、文档整理
- 后端开发、数据库迁移、备份、恢复、写库排障统一在阿里云端执行
- 现代 UI 仍是 protected surfaces
- 恢复只能 path-level / file-level / smallest-scope，恢复前必须 checkpoint
- 跨层 blocker 必须先写 blocker report，再做最小修复

### 7.2 后续版本计划书的固定结构
后续正式版本开发计划书仍应默认包含：
- 版本定位
- 承接基线
- 范围与非目标
- 工作包拆分
- 硬性保护面
- 数据 / 接口 / 表边界
- blocker 条款
- UI protected surfaces 条款
- rollback / recovery policy 条款
- Codex / Claude / Gemini 分工
- 回归清单
- 截图 / 日志证据要求
- 最终验收口径

---

## 8. 最终结论

新的未来路线不应再写成“继续完善已有功能”这种宽泛表述，而应明确为：

> **以 4.9 稳定化基线为承接点，优先补 Discover、Chat、Me 三大模块的产品化与关系推进能力；以 AI 助理、首聊/回聊、标签表达、同城探索、搜索提示卡为主增量；同时保持可扩展单体路线与云端治理纪律，不提前走向重商业化、重娱乐化或重微服务化。**

这条路线既承接现有项目基线，也吸收了 Soul 拆解后真正高价值的结构启发。
