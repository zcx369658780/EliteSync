## 2. R2 候选目标

### Candidate A：GET /api/v2/profile/display

| 检查项 | 判断 |
| --- | --- |
| 目标 | 提供只读 profile display wrapper，面向前端展示安全资料摘要。 |
| 是否 readonly | 是；未来只允许 GET，不允许写 profile/basic。 |
| 是否 authenticated | 是；profile display 涉及用户资料，应要求 auth 边界。 |
| 是否涉及 DB | 可能涉及只读读取既有 profile / user / astro display 来源。 |
| 是否涉及 migration | 不应涉及；一旦需要 migration，应停止。 |
| 是否涉及 production | 不应涉及；production 默认继续 v1。 |
| 是否涉及 Flutter | 不应涉及；Flutter 默认不切 v2。 |
| 是否涉及 v1 contract risk | 中等；必须保护 `GET/POST /api/v1/profile/basic` 和 v1 auth payload。 |
| 是否涉及 canonical truth risk | 中高；不得写 profile truth，不得反写 astro canonical truth。 |
| 是否涉及 privacy / location risk | 中高；不得暴露 `birth_lat` / `birth_lng` / `engine` / `debug` / `hash` / `signature` / precise coordinate fields。 |
| 是否可作为 R2 narrow slice | 可作为未来候选，但需要 GPT 顾问先确认 auth、字段 allowlist、v1 保护和 no-write 证据边界。 |
| 阻断项 | auth smoke 边界、profile/basic 保护说明、字段脱敏清单、no canonical truth write 证明、targeted feature test 范围。 |
| 推荐 / 不推荐理由 | 不作为当前首选。它能直接服务 display cleanup，但 profile truth 与 auth 风险高于 R1，需要更强保护和独立审查。 |

### Candidate B：GET /api/v2/location/context

| 检查项 | 判断 |
| --- | --- |
| 目标 | 提供只读 location context 摘要，明确 `birth_location`、`current_location`、`date_place`、`buddy_place` 四类语义边界。 |

| 是否 authenticated | 建议是；若返回用户上下文摘要，应走 auth 边界。若只返回 schema/static rules，则可保持 public readonly。 |
| 是否涉及 DB | 取决于范围。schema/static rules 可不读用户 DB；用户上下文摘要可能需要只读现有 user / astro profile 字段。 |
| 是否涉及 migration | 不应涉及；当前 contract map 标记 location context blocked until DB plan and privacy review。 |
| 是否涉及 production | 不应涉及；不得请求 production，不得读取真实用户链路。 |
| 是否涉及 Flutter | 不应涉及；Flutter 默认不切 v2。 |
| 是否涉及 v1 contract risk | 中等；不得改变 v1 geo/profile/astro location surfaces。 |
| 是否涉及 canonical truth risk | 中等；不得把 date_place / buddy_place / display location 写回 profile truth 或 astro truth。 |
| 是否涉及 privacy / location risk | 高；不得展示精确坐标、实时定位、住址、`birth_lat` / `birth_lng`、`position_signature` 或 `location_source`。 |

## 3. 推荐 R2 方向

Codex 建议将 Candidate C 提交 GPT 顾问审查：先选择 `staging / production split readonly verification package`，而不是立即进入新的 endpoint runtime。

理由：

- Candidate A 直接触碰 authenticated profile display、profile/basic 保护和 canonical truth display wrapper，风险高于 R1。
- Candidate B 与 A1 location chain 更贴近，但 `current_location`、`date_place`、`buddy_place` 仍有 truth 来源、DB plan 和 privacy review 风险。
- Candidate C 不新增 endpoint，不触碰 DB/migration/Flutter/production，更符合 R1 stage accepted 后的保守推进。
- 若 GPT 顾问要求 R2 必须是 endpoint runtime，建议优先让 GPT 顾问在 A/B 之间选择，并要求 Claude horizontal review 先复核 auth、v1 protection、privacy/location 边界。

本建议只表示“建议提交 GPT 顾问审查”，不表示已获实施授权。

## 6. R2 若获授权时的测试与证据计划

本节仅作为未来建议，不执行。

- route-list 计划：未来只在用户授权后执行 `php artisan route:list --path=api/v2`，确认新增 route 仅为获授权 candidate，且 R1 3 个 endpoint 仍可见。
- php -l 计划：未来只检查新增 / 触碰的 R2 PHP 文件和 R1 相关 PHP 文件。
- targeted feature tests 计划：未来只运行获授权 candidate 的 Feature Test 与 R1 3 个 Feature Test，必要时加入 v1 protected-surface focused checks。
- forbidden grep 计划：检查 R2 controllers/tests 中不得出现 DB write、migration、dispatch、notification、storage write、middleware bypass、seeding、production URL、secret output。
- no production request 证明：记录未请求 production URL，未使用 production token，未执行 production ssh/API smoke。
- no DB write / no migration 证明：记录 `database/` diff 为空，migration 文件未新增，controller 仅 GET readonly 行为。
- v1 protected surface regression 计划：确认 v1 health/version/profile/basic/profile/astro/messages/media/questionnaire/rtc/notifications contracts 没有被改写；如需执行 smoke，必须另行授权。
- privacy / location display 证明：确认 response 不包含 precise coordinates、`birth_lat`、`birth_lng`、`engine`、`debug`、`hash`、`signature`、`position_signature`、`location_source` 或住址级字段。
- auth boundary 证明：若 candidate 需要 authenticated route，必须证明 unauthenticated request 被拒绝且 authenticated readonly request 不写数据；该证明需要另行授权执行。
