# 0. 一句话结论

EliteSync 从 6.0 起正式切换为：

```text
Alpha 内测准备线：商用级底座重构 + Date Drop 式高质量低频匹配 + 搭子精准陪伴 + 基础社交功能补齐 + 玄学解释产品化 + UI/IA 内测打磨
```

5.6–5.10 已完成第一轮“玄学能力二次产品化与校准线”闭环，但当前项目发起人判断：EliteSync 的大部分功能仍是雏形，尚不具备高压力内测能力。

因此，后续不应继续简单下达 5.11 小版本功能补丁，而应进入面向一个月后内测的 6.0 Alpha 规划。

# 2. 当前项目判断

## 2.1 项目初衷

EliteSync 的设计初衷是：

```text
学习 Date Drop 的高质量、低频、结果导向匹配方式，结合星座、八字、紫微等中国用户更熟悉的解释资源，构建高效婚恋匹配产品。
```

EliteSync 不是 Soul 的复制品，也不是测测的复制品。

三类参考的角色必须重新校准：

| 参考对象 | 在 EliteSync 中的角色 | 可吸收重点 | 不吸收内容 |
|---|---|---|---|
| Date Drop | 匹配机制母版 | 低频高质量匹配、深度资料、开放题、反馈驱动、长期关系导向 | 不做浅层刷卡、不做无限推荐流 |
| Soul | 社交表达参考 | 交友意图、自我介绍、关系节奏、低压破冰、个人经营 | 不吸收强 Feed、关注制、匿名快聊、礼物商城、强运营活动 |
| 测测 / CECE | 玄学解释层参考 | 入口矩阵、档案上下文、字段引导、维度解释、摘要/详情分层 | 不吸收真人咨询、付费报告、上传截图主路径、娱乐化测试、消息模拟 |

## 2.2 当前状态

截至 5.10，EliteSync 已具备较多功能雏形：

- 登录与资料；
- 星盘 / 八字 / 紫微等画像能力；
- 匹配结果与关系解释；
- Chat 低压开场建议；
- Profile 个人表达建议；
- Settings 解释层治理；
- 图片 / 视频 / 语音 / RTC 等基础能力；
- 动态 / 通知 / 版本中心 / 运维入口等基础工程能力。

但从内测视角看，仍存在关键问题：

1. 后端与第三方服务链路仍偏开发版，不适合未来十万级用户压力预期；
2. 地图 / 定位 / 位置选择链路需要重新设计，不能继续依赖不够稳健的内置地址搜索；
3. 基础社交功能与成熟 App 差距明显；
4. 玄学能力仍偏“原材料展示”，缺少婚恋场景解释；
5. 部分页面信息密度低、页面过长、文字气泡滥用；
6. 部分工程术语、内部状态、测试语义暴露给用户；
7. 当前还没有建立“每个版本必须经过竞品横向复评”的固定验收制度。

# 3. 6.0 Alpha 总目标

6.0 Alpha 的目标不是继续堆功能，而是让 EliteSync 在一个月后具备小规模内测前的产品与工程底座。

总目标：

```text
让 EliteSync 从“功能雏形集合”升级为“可承接真实内测用户的高质量婚恋匹配产品 Alpha 版”。
```

具体目标：

1. 完成商用级后端 v2 与位置链路重构方案；
2. 建立 Date Drop 式匹配主链：深资料、低频高质量匹配、揭晓、反馈；
3. 新增“搭子”精准陪伴产品线；
4. 补齐 Soul / 测测对照中暴露的基础功能差距；
5. 把星座 / 八字 / 紫微从原材料改造成关系解释能力；
6. 完成主要页面 UI / IA 重构，降低页面长度与信息噪声；
7. 建立每版本 Claude 横向复评门禁；
8. 为一个月后内测准备数据、日志、反馈、监控、回滚与运营入口。

# 5. P0：商用化底座与后端 v2

## 5.1 问题判断

当前后端与第三方 SDK 链路更适合开发验证，不应直接承接未来较高用户压力。

需要重点处理：

- 用户注册与资料真值链；
- 匹配计算与结果发放；
- Chat / message / media；
- 通知与回流；
- 星盘 / 八字 / 紫微计算链；
- 位置、城市、约会地点；
- 反馈与内测数据回收；
- 日志、监控、告警、备份、恢复；
- 测试环境、staging 环境、production 环境隔离。

## 5.2 后端 v2 原则

后端更换不应采用“推倒重写 + 一次性迁移”的高风险方式。

建议采用：

```text
contract-first + parallel v2 + staged migration
```

即：

1. 先冻结 v1 API contract；
2. 设计 v2 API contract；
3. 新老后端并行；
4. Flutter 通过配置切换 base URL；
5. 先迁移读接口；
6. 再迁移低风险写接口；
7. 最后迁移资料、匹配、消息、媒体等核心写链；
8. 每一步都保留回滚路径。

## 5.3 后端 v2 需要覆盖的核心域

| 领域 | v2 要求 |
|---|---|
| Auth / User | 支持真实内测账号体系、风控、登录态、注销保护 |
| Profile | 资料真值、展示资料、关系目标、兴趣标签分层 |
| Astro | 星盘 / 八字 / 紫微真值与解释层分离 |
| Match | 低频高质量匹配、等待期、揭晓、反馈、下一轮优化 |
| Buddy / 搭子 | 兴趣陪伴需求、共同兴趣匹配、时段与地点偏好、搭子反馈 |
| Chat | 消息、首聊、冷场恢复、关系上下文、建议草稿 |
| Media | 图片 / 视频 / 语音资源与审核策略 |
| Notification | 匹配揭晓、搭子推荐、聊天回流、反馈提醒 |
| Feedback | 匹配反馈、搭子反馈、内测问题反馈 |
| Operations | 内测用户管理、灰度、日志、告警、导出 |
| Release | 版本检查、更新、强制升级、兼容策略 |

## 5.4 位置链路重构

当前不建议继续强化内置地址搜索。

建议改为：

```text
去内置地图搜索化 + 位置语义分层 + 外部地图辅助
```

位置数据至少拆成四类：

| 类型 | 用途 | 展示精度 | 是否需要经纬度 |
|---|---|---|---|
| 出生地 | 星盘 / 八字计算 | 城市 / 区县 / 手工校正 | 内部可用，用户侧不展示精确经纬度 |
| 现居地 | 匹配地域范围 | 城市 / 行政区 | 可选，默认不展示精确经纬度 |
| 约会地点 | 线下见面建议 | 地点名称 / 商圈 / 城市 | 可选，不作为资料真值 |
| 搭子地点 | 学习 / 吃饭 / 健身等陪伴 | 城市 / 商圈 / 用户确认地点 | 可选，需隐私保护 |

第一阶段建议：

- 去掉用户侧精确经纬度展示；
- 去掉火车站名等误导性地名展示；
- 出生地和现居地改为行政地名；
- 约会地点和搭子地点可先保存文本 / 城市 / 商圈；
- 外部百度地图 / 腾讯地图 App 仅作为导航或位置参考，不强依赖返回坐标；
- 后续如确需选点回传，再单独评估地图选点组件或 Web 选点页。

# 6. P1：Date Drop 式匹配主链

## 6.1 产品定位

EliteSync 的婚恋主链应以 Date Drop 式机制为母版：

```text
深度理解用户 -> 低频高质量推荐 -> 揭晓 -> 解释 -> 聊天 -> 反馈 -> 下一轮优化
```

不要变成无限刷卡、强 Feed、强私聊或泛社交平台。

## 6.2 核心模块

### 6.2.1 深度资料输入

需要从“基础资料表单”升级为“关系理解资料”。

应包含：

- 基础资料；
- 关系目标；
- 城市 / 距离 / 见面意愿；
- 兴趣偏好；
- 价值观；
- 生活节奏；
- 恋爱观；
- 不能接受的点；
- 开放题；
- 可选语音表达；
- 星座 / 八字 / 紫微等解释资料。

### 6.2.2 低频匹配机制

应明确：

- 不是无限刷人；
- 不是强推荐流；
- 每轮推荐数量有限；
- 匹配结果有等待期和揭晓期；
- 揭晓后给出明确解释；
- 用户反馈会影响后续推荐。

### 6.2.3 等待期体验

当前 Match 等待期缺少“为什么等待”的产品解释。

应增加：

- 本轮匹配正在准备；
- 系统会综合哪些信息；
- 用户可以补充哪些资料提高质量；
- 可查看自己的表达建议；
- 可先进入搭子 / 自我介绍 / 话题准备。

### 6.2.4 揭晓页

揭晓页应包括：

- 推荐对象摘要；
- 为什么值得了解；
- 共同点；
- 差异点；
- 适合怎样开始；
- 先避免怎样表达；
- 是否愿意继续了解；
- 低压开场建议。

### 6.2.5 反馈闭环

用户需要能够反馈：

- 推荐是否准确；
- 哪些点吸引 / 不吸引；
- 是否愿意继续；
- 聊天体验如何；
- 是否见面；
- 是否希望类似对象更多 / 更少。

反馈只用于后续匹配优化，不反写星盘 / 资料真值。

# 7. P1：搭子精准陪伴功能

## 7.1 功能定位

搭子功能是 6.0 Alpha 新增主线之一。

产品定义：

```text
官方基于共同兴趣、共同目标、时段、地点和关系边界，为用户提供低压力、精准陪伴型匹配。
```

搭子不是婚恋匹配的替代品，而是：

- 降低社交启动压力；
- 增加真实互动频率；
- 给慢约会提供自然接触场景；
- 为婚恋匹配提供补充行为反馈；
- 在内测阶段提高用户留存与体验密度。

## 7.2 首批搭子类型

首批建议做四类：

| 类型 | 场景 | 核心匹配因子 | 是否适合内测首批 |
|---|---|---|---|
| 学习搭子 | 图书馆、自习、备考、线上打卡 | 目标、时间、地点、学习方向、监督强度 | 是 |
| 电影搭子 | 看电影、观影偏好、片单讨论 | 片类偏好、时间、距离、交流意愿 | 是 |
| 吃饭搭子 | 探店、工作餐、周末饭局 | 口味、预算、位置、时间、AA 偏好 | 是 |
| 健身搭子 | 健身房、跑步、运动打卡 | 运动类型、强度、时段、地点 | 是 |

后续可扩展：

- 旅行搭子；
- 读书搭子；
- 展览搭子；
- 游戏搭子；
- 备考搭子；
- 工作互助搭子；
- 情绪陪伴搭子。

## 7.3 搭子与婚恋匹配的关系

必须明确边界：

| 维度 | 婚恋匹配 | 搭子匹配 |
|---|---|---|
| 目标 | 长期关系 / 婚恋可能性 | 共同兴趣 / 陪伴活动 |
| 节奏 | 低频高质量 | 可相对高频但仍克制 |
| 解释 | 关系适配 / 价值观 / 星盘合盘 | 兴趣、目标、时间、地点、活动偏好 |
| 风险 | 情感预期过高 | 线下安全与边界 |
| 反馈 | 是否合适、是否继续、关系进展 | 活动是否成功、是否守时、是否舒服 |

搭子功能不能把 EliteSync 变成泛同城约玩平台。它必须服务于慢约会主线：低压力认识人、通过共同活动建立信任。

## 7.4 搭子功能最小闭环

6.0 Alpha 最小闭环建议：

1. 用户选择搭子类型；
2. 填写搭子需求；
3. 选择时间 / 地点 / 线上或线下；
4. 系统生成搭子需求卡；
5. 系统推荐 1–3 个候选；
6. 展示为什么推荐；
7. 用户选择感兴趣；
8. 进入低压聊天；
9. 活动后反馈；
10. 反馈进入后续搭子推荐。

首版可以不做复杂发布广场，优先做官方匹配式搭子推荐。

## 7.5 搭子数据结构建议

首版至少需要：

- 搭子类型；
- 搭子目标；
- 期望时间；
- 期望地点 / 城市 / 商圈；
- 线上 / 线下；
- 活动预算；
- 交流强度；
- 性别偏好是否开放；
- 安全边界；
- 是否接受多人；
- 反馈状态。

## 7.6 搭子安全边界

搭子涉及线下活动，必须在计划书中前置安全边界：

- 不鼓励初次私密地点见面；
- 推荐公共场所；
- 明确 AA / 预算 / 时间；
- 聊天前展示安全提示；
- 活动后可反馈不适；
- 支持举报 / 拉黑；
- 不展示精确实时位置；
- 不将搭子反馈直接写成婚恋人格判断。

# 8. P2：基础社交能力补齐

## 8.1 目标

补齐与成熟社交 App 的基础差距，但不破坏 EliteSync 的低频高质量匹配定位。

## 8.2 应优先补齐

### 8.2.1 用户交友意图内容化

当前资料字段不应只是“信息表”。应让用户表达：

- 我想认识什么样的人；
- 我最近想做什么；
- 我适合怎样被打招呼；
- 我愿意从什么话题开始；
- 我目前更偏婚恋 / 搭子 / 朋友 / 先了解。

### 8.2.2 自我介绍发布 CTA

学习 Soul 的“自我表达入口”，但不做重 Feed。

可以做：

- 发布一段自我介绍；
- 发布一个最近想找的搭子；
- 发布一个约会 / 观影 / 学习意图；
- 发布一个可被匹配系统读取的轻状态。

### 8.2.3 个人标签可视化

Profile 不应只是长表单。

应形成：

- 关系目标标签；
- 兴趣标签；
- 生活节奏标签；
- 价值观标签；
- 搭子偏好标签；
- 星座 / 八字 / 紫微摘要标签；
- 慢约会友好表达标签。

### 8.2.4 Chat 关系节奏提示

Chat 不应只是输入框。

应提供：

- 为什么可以聊；
- 从哪里开始；
- 先不要太急；
- 是否适合语音；
- 是否适合约搭子活动；
- 冷场后如何恢复。

### 8.2.5 通知与回流

通知应围绕主链：

- 匹配即将揭晓；
- 匹配已揭晓；
- 对方回应；
- 搭子候选已生成；
- 搭子活动待反馈；
- 资料可补充；
- 内测反馈邀请。

不做强运营骚扰。

## 8.3 暂不做

- 关注制社交；
- 重 Feed；
- 礼物商城；
- 表情 / GIF / 贴纸系统；
- 付费会员；
- 真人咨询；
- 自动代聊；
- 强 AI 问答；
- 娱乐化测试广场。

# 9. P3：玄学解释层产品化重构

## 9.1 问题

当前星盘 / 八字 / 紫微信息偏原材料展示，用户难以理解其婚恋价值。

需要从：

```text
专业数据展示
```

转为：

```text
对关系、表达、节奏、搭子、约会有帮助的解释
```

## 9.2 入口矩阵

建议在 6.0 中形成玄学入口矩阵：

- 我的星盘；
- 我的八字；
- 我的紫微；
- 我的表达建议；
- 我和 TA 为什么适合聊；
- 我和 TA 的关系节奏；
- 搭子活动适配提示；
- 今天适合怎样表达。

## 9.3 分层解释结构

每个解释页都应采用：

```text
摘要句 -> 维度卡 -> 详情页 -> 建议 / 避免 -> 可选追问
```

不得再把全部信息堆在一个超长页面。

## 9.4 与婚恋主链结合

玄学解释必须服务于：

- 我如何被看见；
- 我适合怎样表达；
- 我们为什么值得聊；
- 我们可能在哪些节奏上不同；
- 适合从什么话题开始；
- 搭子活动中如何相处更舒服。

## 9.5 边界

所有玄学解释保持：

```text
derived-only / display-only / explanation layer
```

不得：

- 反写 profile/basic；
- 反写 profile/astro/summary；
- 反写 profile/astro/chart；
- 反写 user_astro_profiles；
- 改匹配算法 contract；
- 改 API / DB / release chain；
- 接真人咨询；
- 接付费报告；
- 上传截图作为主路径；
- 复制测测术语体系；
- 把玄学解释写成确定命运判断。

# 10. P4：UI / IA 内测前打磨

## 10.1 目标

让 App 从“功能堆叠雏形”变成“用户可理解、可探索、可反馈”的内测产品。

## 10.2 重点问题

需要处理：

- 部分页面文字气泡滥用；
- 页面信息密度过低；
- 页面过长；
- 玄学解释页过度堆叠；
- 工程术语暴露；
- 测试语义暴露；
- 入口之间层级不清；
- 用户不知道下一步该做什么。

## 10.3 页面级方向

| 页面 | 方向 |
|---|---|
| Home | 从功能入口堆叠改为今日主任务：补资料、看匹配、找搭子、继续聊天 |
| Discover | 不做重 Feed，聚焦搭子 / 自我介绍 / 轻状态发现 |
| Match | 等待期、揭晓期、详情期分层 |
| Chat | 增加关系上下文、低压建议、搭子活动建议 |
| Profile | 从资料表单改成个人经营与被理解页面 |
| Astro | 长页拆分，摘要优先，详情后置 |
| Buddy | 搭子类型入口、需求卡、候选推荐、反馈闭环 |
| Settings | 账号、安全、通知、解释与建议、隐私、外观分层 |

## 10.4 内测反馈入口

每个主模块都应有低成本反馈：

- 看不懂；
- 不准确；
- 不想看；
- 推荐不合适；
- 页面太长；
- 想要某类搭子；
- 遇到问题。

反馈进入内测看板，不直接影响用户真值。

# 11. Claude 横向复评门禁

## 11.1 新增强制规则

从 6.0 Alpha 起，每个版本完成后，必须先通过 Claude 横向复评，再提交 GPT 顾问最终验收。

标准流程：

```text
Codex 完成实现与自测
-> Codex 生成证据包与版本报告
-> Claude 调用 Soul + 测测作为横向参考进行复评
-> Claude 输出竞品对照复评报告
-> Codex 根据 Claude 复评修正或解释不采纳原因
-> GPT 顾问最终验收
-> 通过后才允许 commit / push / 进入下一版本
```

## 11.2 Claude 复评角色

Claude 的角色不是替代 GPT 顾问，而是：

- 独立产品体验复查员；
- Soul / 测测横向对照评估员；
- UI / IA 可理解性审查员；
- 功能完整性与用户路径审查员；
- 竞品吸收边界审查员。

## 11.3 每版 Claude 复评必须检查

1. 本版本目标是否实际可见；
2. 用户是否能自然找到入口；
3. 页面是否能解释“为什么这个功能有用”；
4. 与 Soul 同类功能相比，是否缺少基础社交能力；
5. 与测测同类功能相比，是否缺少解释层、字段引导、摘要/详情分层；
6. 是否出现竞品不应吸收的商业化、娱乐化或服务化路径；
7. 是否暴露工程术语；
8. 是否破坏 Date Drop 式低频高质量匹配定位；
9. 是否误把搭子做成泛同城约玩；
10. 是否有用户安全、隐私、位置展示风险。

## 11.4 Claude 复评输出文件

每个版本必须输出：

```text
CLAUDE_<VERSION>_HORIZONTAL_REVIEW.md
CLAUDE_<VERSION>_SOUL_COMPARISON.md
CLAUDE_<VERSION>_CECE_COMPARISON.md
CLAUDE_<VERSION>_ACTION_MATRIX.md
```

如果版本与 Soul / 测测无强关系，也必须说明：

```text
本版本不强依赖 Soul / 测测对照，原因是……
```

但不能省略 Claude 复评。

## 11.5 复评通过标准

Claude 复评结论允许：

- `pass`：可提交 GPT 顾问验收；
- `pass with observations`：可提交 GPT 顾问验收，但必须带 observation；
- `conditional pass`：需 Codex 补证据或小修后再提交；
- `fail`：不得提交 GPT 顾问最终验收。

只有 `pass` 或 `pass with observations` 可以进入 GPT 顾问最终验收。

`conditional pass` 必须先补材料。

`fail` 必须返工。

# 13. 版本开发通用边界

所有 6.0 Alpha 版本必须遵守：

## 13.1 不破坏保护面

不得无计划改动：

- profile/basic；
- profile/astro/summary；
- profile/astro/chart；
- user_astro_profiles；
- dating_matches；
- chat_messages；
- media_assets；
- rtc_calls / rtc_sessions；
- app_release_versions；
- release chain。

## 13.2 不复制竞品

允许：

- 观察公开可见 UI；
- 抽象产品结构；
- 学习功能分层；
- 结合 EliteSync 自身定位重新实现。

禁止：

- 解包；
- 逆向；
- 抓包；
- 接口分析；
- 复制 UI 资产；
- 复制文案体系；
- 复制商业链路；
- 绕过权限或付费。

## 13.3 不把搭子做成泛同城约玩

搭子必须服务慢约会和共同兴趣陪伴。

禁止：

- 附近的人无限刷；
- 强地理曝光；
- 私密地点邀约；
- 无边界线下约玩；
- 陪玩商业化；
- 付费陪伴；
- 性暗示或擦边场景。

## 13.4 不把玄学解释写成命运判断

玄学解释只能是：

- 理解参考；
- 表达建议；
- 关系节奏提示；
- 可忽略、可关闭、可反馈。

不得写成：

- 绝对性格判断；
- 命运断言；
- 婚姻保证；
- 医疗 / 财务 / 法律建议；
- 强迫式决策依据。
