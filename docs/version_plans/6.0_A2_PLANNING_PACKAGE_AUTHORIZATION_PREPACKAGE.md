## 7. Evidence requirements for future A2 runtime gates

Before any future A2 runtime implementation gate can execute, the runtime authorization package should prove or define:

1. Exact runtime slice:
   - route / page / module scope;
   - files expected to change;
   - files explicitly out of scope;
   - whether backend, Flutter, DB, or release-chain are involved.
2. Product acceptance target:
   - what Date Drop user journey segment becomes visible;
   - what remains placeholder / future-only;
   - how users understand waiting, reveal, explanation, feedback, and next round.
3. Protected-surface review:
   - v1 contract protection;
   - profile truth protection;
   - astro explanation derived-only / display-only boundary;
   - Match algorithm and DB write boundaries;
   - Chat/message side-effect boundaries;
   - release-chain diff must remain empty unless separately authorized.
4. Data and privacy plan:
   - no production DB write without separate authorization;
   - no migration without backup / rollback plan;
   - no hidden feedback write side effects;
   - no private user data in evidence.
5. Environment plan:
   - local / staging / production boundary;
   - whether endpoint requests are needed;
   - whether staging or production requests are forbidden or separately authorized;
   - no use of `slowdate.top` or `101.133.161.203` as staging without explicit confirmation.
6. Test plan:
   - targeted unit / feature / widget tests as relevant;
   - protected-surface diffs;
   - no broad smoke unless separately authorized;
   - no endpoint smoke unless separately authorized.
7. Text-first evidence plan:
   - Codex self-review;
   - evidence index;
   - action matrix if observations exist;
   - screenshot inventory only when user-visible UI is touched.
8. Claude / Gemini review plan:
   - Claude horizontal review is required before GPT final acceptance for user-visible 6.0 Alpha runtime;
   - Gemini may be used for visual / long-context review if needed;
   - neither may bypass authorization gates.
9. Stop conditions:
   - runtime needs DB / migration / production / endpoint / SSH / cache / composer / service operation not pre-authorized;
   - branch or worktree mismatch;
   - unplanned current docs / release-chain / Flutter base URL changes;
   - any command would output secrets.
