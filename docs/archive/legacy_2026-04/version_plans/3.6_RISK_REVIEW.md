# EliteSync 3.6 风险评审

更新时间：2026-04-12  
适用对象：Codex / 项目负责人 / Claude-mcp / Gemini-MCP  
版本状态：阶段 0 风险评审

---

## 1. 结论

3.6 的总体方向是可控的 additive 扩展，不是 canonical 重写。  
但这版的风险明显高于 3.5，主要原因是它开始引入：

- `route_mode`
- 古典 / 现代 / 标准三路线
- 更细的解释层
- 更复杂的样例对照与差异矩阵

其中最需要警惕的不是“加字段”，而是：

1. 路线派生结果被误沉淀为真值
2. 旧缓存 / 旧客户端在 schema 扩展后失配
3. UI 工作台因为解释层扩展而再次变成“半个后台”

---

## 2. 高风险清单

| 风险 | 等级 | 说明 |
|---|---|---|
| `route_mode` 误写真值 | 高 | 路线标签只能是派生上下文，不能替代 canonical profile |
| `engine_info` / `metadata` 扩展失配 | 高 | 旧客户端必须继续可读，新增字段只能附加 |
| `astro_chart_preferences_v1` 膨胀 | 高 | 本地偏好若混入路线真值，会破坏显示 / 真值分层 |
| `chart_data` 旧结构被改坏 | 高 | Flutter 渲染器、样例对照、旧缓存会一起受影响 |
| 路线入口把 UI 重新做成“后台” | 中高 | 3.6 允许更强工作台，但不能让页面失去阅读性 |
| 多样例对照矩阵不稳定 | 中高 | 若样例矩阵不稳定，后续差异解释会失去可信度 |
| 旧路由入口失效 | 中 | 资料页、星盘页、设置中心、版本中心都必须保留 |

---

## 3. 潜在回归点

### 3.1 API / schema

- `/api/v1/profile/basic`
- `/api/v1/profile/astro/summary`
- `/api/v1/profile/astro/chart`
- `/api/v1/profile/astro`
- `/api/v1/profile/astro/render`

这些接口不能因为 3.6 的双路线而改坏旧字段。

### 3.2 Flutter 读取链

- `astro_profile_provider.dart`
- `natal_chart_svg_builder.dart`
- `astro_chart_settings_provider.dart`
- `astro_chart_sample_comparison.dart`
- `astro_chart_sample_fixture.dart`

这些模块对旧字段、默认值和样例结构都有隐式耦合。

### 3.3 路由与入口

- `AstroOverviewPage`
- `AstroProfilePage`
- `AstroNatalChartPage`
- `AstroChartSettingsPage`
- `AstroBaziPage`

任何路线切换都不能破坏旧入口返回栈和加载行为。

---

## 4. 回滚建议

1. 任何新增字段只做 additive，不改名、不删字段、不改类型。
2. 保留 `standard` 作为 3.5 兼容 fallback。
3. 所有未知 `route_mode`、未知枚举、坏缓存都要自动退回 3.5 行为。
4. 新旧缓存要支持缺省补齐和忽略未知 key。
5. 服务端和 Flutter 侧都要有一键关闭双路线的 feature flag 或版本门禁。
6. 若出现 schema / 显示分歧，先停在只读验证，不继续向下扩。
7. 回滚优先“禁用新路线”，不优先删除数据。

---

## 5. 兼容性提示

- `chart_data` 必须继续兼容 3.5 旧结构。
- `engine_info` / `metadata` 只能加新子对象，不能重排旧字段。
- 路线能力矩阵要明确“计算能力”与“展示能力”不是一回事。
- `astro_chart_preferences_v1` 继续只做 display-only。
- 旧入口必须保留：
  - `/api/v1/profile/basic`
  - `/api/v1/profile/astro/summary`
  - `/api/v1/profile/astro/chart`
  - `AstroOverviewPage`
  - `AstroNatalChartPage`
  - `AstroChartSettingsPage`
- 任何 route-specific explanation 都只能定位为“附加解释层”，不能混入 canonical 真值。
