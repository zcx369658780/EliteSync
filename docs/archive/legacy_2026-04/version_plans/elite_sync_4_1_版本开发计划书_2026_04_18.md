# EliteSync 4.1 版本开发计划书

日期：2026-04-18  
版本定位：`4.1 非官方四维人格问卷与人格能力补全版`  
当前承接基线：`4.0 正式通过 / pass with observations`  
计划适用对象：Codex-CLI、Claude-MCP、Gemini-MCP、项目顾问验收

---

## 0. 本版定位与背景

### 0.1 为什么 4.1 现在启动

4.0 已完成基础能力基建版的主要目标：
- 会话、媒体、关系事件、通知的领域骨架已建立；
- 对象存储主路径最小闭环已接通；
- 队列 / 缓存 / worker 最小闭环已落地；
- Flutter 附件上传状态骨架已成型；
- 3.x 真值链与既有保护面未受破坏。

这意味着 4.x 已经具备承接“正式产品能力接入”的基础条件。根据整体开发总纲，**最适合在 4.0 之后优先进入的版本，不是图片消息正式版，也不是动态流，而是人格能力补全版**。原因有三：

1. 四维人格问卷不依赖 4.0 之外的新基础设施，可以在现有基建上稳定推进；  
2. 人格倾向与 EliteSync 的匹配解释、人格表达、首聊引导天然相关，属于差异化核心能力；  
3. 若不先把人格测试从“已有表结构”推进成“正式产品能力”，后续主页、匹配解释、状态表达会缺少一条重要的人格主线。

### 0.2 当前现实约束

本版规划以以下现实约束为前提：

1. 公司尚未注册，短期内仍无法通过真实运营获取大量产品反馈；
2. 因此 4.1 必须优先做“未来无论如何都会保留”的人格能力，而不是为了运营包装而扩面；
3. 4.1 应在 4.0 已建立的工程底座之上推进，不允许重复搭基础设施；
4. 人格问卷必须以“版本化、可回溯、可复测、可解释”为原则，不得做成一次性写死的小测试页；
5. 人格结果可以轻量接入个人主页、匹配解释和首聊建议，但不得在本版中绑死整个匹配系统；
6. **本版不走“官方 MBTI 测试接入”路线，不依赖所谓公开官方题库，不复制网上流传题目原文，而是实现一个“基于四维偏好的非官方人格问卷系统”。**

### 0.3 本版正式定位

4.1 的正式定位不是“做一个官方 MBTI 页面”，而是：

> **4.1 非官方四维人格问卷与人格能力补全版：把四维人格测试从已有数据表与结果字段，升级为正式产品能力、结构化算法输出与可持续迭代的人格模块。**

换句话说，4.1 要解决的是：
- 用户如何进入人格测试；
- 如何完成问卷；
- 结果如何生成与解释；
- 结果如何存储与版本化；
- 结果如何进入个人主页、人格表达与匹配系统；
- 后续如何扩展、重测、比较与迭代。

### 0.4 产品与合规口径

本版必须统一以下口径：

1. **产品展示不得宣称“官方 MBTI 测试”。**  
2. **产品文案应使用“人格四维测试”“16 型人格倾向测试”“基于四维偏好的非官方人格问卷”等表达。**  
3. **题库必须为自研题库，不得直接照搬公开流传的题目原文。**  
4. **结果页必须明确：该结果反映当前问卷下的人格倾向，仅供自我理解与社交参考，不构成专业心理诊断。**  
5. **人格结果不是平台唯一真相，只是人格模块之一。**

## 1. 4.1 的核心目标

本版只围绕以下 **6 个核心目标** 展开。

### 1.1 目标一：把人格测试从“数据库表存在”升级为“正式产品流程”

必须形成：
- 测试入口；
- 题目流程；
- 提交流程；
- 结果生成；
- 结果页；
- 结果回看入口。

不允许继续停留在只有 `mbti_attempts` 表结构和分数字段、但前台没有完整流程的状态。

### 1.2 目标二：建立题库版本化与答题结果版本化机制

必须形成：
- 题库版本号；
- 答题记录与结果版本号；
- 分数结构、置信度结构、tie-break 结构可回看；
- 后续题库升级不破坏旧结果可解释性。

### 1.3 目标三：建立正式的人格结果解释系统

必须形成：
- 4 个维度的结构化解释；
- 16 型结果页；
- 结果摘要；
- 信度/不确定性提示；
- 用户可理解、非工程化的结果说明。

### 1.4 目标四：建立人格能力与个人主页/匹配解释的轻量联动

必须形成：
- 人格结果可进入个人主页；
- 匹配解释可读取人格结果作为一个模块；
- 首聊建议 / 风格提示可轻量使用人格结果；
- 但不允许在本版中把人格结果绑成唯一人格真相或唯一匹配真相。

### 1.5 目标五：建立重测、历史记录与版本对比能力

必须形成：
- 用户可再次测试；
- 旧记录保留；
- 不同版本题库结果可区分；
- 至少能看出“当前结果”和“历史尝试”的关系。

### 1.6 目标六：明确 4.1 不做什么，避免范围失控

必须明确：
- 4.1 不做动态流；
- 4.1 不做图片消息正式版；
- 4.1 不做音视频通话；
- 4.1 不做新算法主线升级；
- 4.1 不做复杂 AI 陪聊人格引擎；
- 4.1 不做复杂 A/B 平台；
- 4.1 不做“官方 MBTI 接入工程”。

## 2. 本版明确不做的内容

为了避免 4.1 演变成“人格能力大杂烩”，以下内容明确列为 **本版不做**。

### 2.1 不做官方 MBTI 题库 / 官方 MBTI 产品接入

不做：
- 不采购或接入官方 MBTI 电子测试产品；
- 不直接复刻所谓“官方 MBTI 题库”；
- 不从 GitHub 或公开网页直接复制题目原文上线；
- 不在产品中声称“官方 MBTI 测试”。

原因：本版走的是**非官方四维人格问卷系统**路线，重点是自研题库、版本化结构和产品能力，不走版权和商标风险较高的复刻路径。

### 2.2 不做完整动态流与状态流产品

不做：
- 发布状态页；
- 点赞评论全链路；
- 动态推荐排序；
- 人格内容 feed 化。

原因：这些属于 4.3 主线，不应混入 4.1。

### 2.3 不做完整图片 / 视频消息正式产品

不做：
- 图片消息正式交互闭环；
- 视频消息正式闭环；
- 缩略图、预览、播放等完整产品行为。

原因：这属于 4.2 / 4.4 的主任务，4.1 只消费 4.0 已有基础设施，不反向扩面。

### 2.4 不做音视频通话与实时状态

不做：
- 实时呼叫；
- 在线状态；
- RTC 信令；
- 通话页面。

原因：这是 4.6 的单独高风险版本。

### 2.5 不做四维人格之外的人格体系大扩容

不做：
- Big Five 正式接入；
- 九型人格正式接入；
- AI 人格测评引擎；
- 多种人格体系同时上线。

原因：4.1 的重点是先把四维人格问卷做完整、做扎实，而不是同时引入多个新体系。

### 2.6 不做匹配算法主线大重构

不做：
- 用人格结果全面替换现有匹配解释结构；
- 用人格结果大改总分算法；
- 调整星盘 / pair_chart / timing 主线权重。

原因：4.1 只做“人格模块正式接入”，不是“匹配算法重算版”。

## 3. 4.1 的工作包拆分

本版建议拆为 **5 个工作包**，统一作为 4.1 版本主线实施。

- **4.1A：题库、答题结构与版本化骨架**
- **4.1B：测试流程与结果生成闭环**
- **4.1C：结果解释页、历史记录与复测能力**
- **4.1D：个人主页 / 匹配解释 / 首聊建议的轻量联动**
- **4.1E：回归、可观测性、文档与交付收口**

这些工作包不是独立产品版本，而是 4.1 的分阶段执行结构。

## 4. 4.1A：题库、答题结构与版本化骨架

### 4.1A 目标

先把人格问卷的“内容层”和“结构层”做扎实，避免后续整个测试系统建在不稳定题库上。

### 4.1A 必做任务

#### 4.1A.1 设计自研题库数据结构

必须明确：
- 题目主表或等价结构；
- 题目版本号；
- 维度映射（E/I, S/N, T/F, J/P）；
- 反向题/正向题标记；
- 显示顺序与分组；
- 题目启用状态；
- 题目来源标记为自研。

建议至少具备：
- `question_code`
- `question_version`
- `dimension`
- `polarity`
- `prompt`
- `option_a`
- `option_b`
- `is_reverse`
- `is_active`
- `sort_order`
- `source_type`（固定为自研或等价口径）

#### 4.1A.2 明确正式版 / 快速版口径

建议本版至少支持两类问卷配置：
- **正式版**：48~64 题，作为主结果依据；
- **快速版**：20~28 题，作为轻量入口或后续 onboarding 预留。

本版可以先完成正式版，快速版至少预留结构与版本口径。

#### 4.1A.3 明确答题记录结构

必须明确：
- 每次测试 attempt 的唯一标识；
- 用户选择项；
- 题目版本；
- 问卷类型（正式版 / 快速版）；
- 答题时间；
- 提交时间；
- 分数快照；
- 置信度快照；
- tie-break 日志快照；
- 最终结果字母。

可以继续沿用并补强 `mbti_attempts`，但不得让旧字段失去解释能力。

#### 4.1A.4 形成版本化规则

必须明确：
- 题库升级如何处理；
- 不同版本结果如何区分；
- 旧 attempt 是否保留；
- 旧题库是否仍可回看；
- 结果页如何展示题库版本与问卷类型。

### 4.1A 任务边界解释

允许做：
- 新表或新字段；
- 自研题库 seed 数据；
- 版本字段；
- 后端 scoring 结构补强。

禁止做：
- 一边做题库一边引入多个全新人格体系；
- 为了快速上线写死题目在前端代码里；
- 使用来源不明的公开题目原文直接上线；
- 让题库版本和结果版本脱节。

### 4.1A 验收标准

必须满足：
- 题库可配置；
- 题目来源为自研口径；
- attempt 可版本化；
- 分数结构可回溯；
- 未来改题不会破坏历史结果的解释口径。

## 5. 4.1B：测试流程与结果生成闭环

### 4.1B 目标

把四维人格问卷真正做成用户可以完成的一条正式流程，而不是停留在数据结构层。

### 4.1B 必做任务

#### 4.1B.1 建立测试入口

必须完成：
- 在合适入口提供人格测试进入点；
- 入口文案清晰，避免工程感；
- 避免出现“官方 MBTI”字样；
- 若测试耗时较长，应有预期提示。

建议入口文案：
- 人格四维测试
- 16 型人格倾向测试
- 基于四维偏好的非官方人格问卷

#### 4.1B.2 建立答题流程页

必须完成：
- 单题或分组答题页面；
- 当前进度展示；
- 下一题 / 上一题（或等价导航）；
- 未答完时的提示；
- 提交前确认逻辑。

#### 4.1B.3 建立提交与结果生成逻辑

必须完成：
- 提交答卷；
- 计算四维得分；
- 计算结果字母；
- 写入 `score_json / confidence_json / tie_break_log_json` 或等价结构；
- 返回结果页所需数据。

#### 4.1B.4 建立最小不确定性提示

必须完成：
- 边缘结果或 tie-break 场景有基本说明；
- 不把结果表达成绝对不可改变的“命定标签”；
- 保留“当前测试结果”“当前问卷结果”的口径。

#### 4.1B.5 建立正式免责声明口径

结果页或测试页必须有清晰说明：
- 该结果为非官方四维人格问卷结果；
- 仅供自我理解与社交参考；
- 不构成专业心理诊断。

### 4.1B 任务边界解释

允许做：
- 答题 UI；
- 后端 scoring；
- 提交流程；
- 基础置信度提示；
- 正式免责声明文案。

禁止做：
- 为了炫技引入复杂 AI 实时分析；
- 把测试流程做成过度游戏化产品；
- 把答题结果直接绑定成账号不可修改真值；
- 使用“官方 MBTI 测试结果”等高风险措辞。

### 4.1B 验收标准

必须满足：
- 用户可完整完成一次测试；
- 结果可生成、可存储、可回看；
- 异常情况与边缘情况有最小解释；
- 流程能跑通而不是停在 demo 态；
- 页面文案不冒充官方 MBTI 产品。

## 6. 4.1C：结果解释页、历史记录与复测能力

### 4.1C 目标

让人格测试结果不只是四个字母，而是可理解、可复查、可继续演进的人格能力模块。

### 4.1C 必做任务

#### 4.1C.1 建立结果主页

必须完成：
- 16 型结果展示；
- 结果摘要；
- 四维分布或等价表达；
- 用户语言解释；
- 结果版本 / 题库版本 / 问卷类型提示；
- “当前结果更偏 XX 型”的表述，而非绝对化定性。

#### 4.1C.2 建立维度解释页或维度展开结构

必须完成：
- E/I
- S/N
- T/F
- J/P
每一维至少有：
- 当前倾向；
- 简明解释；
- 不确定性提示（如适用）。

#### 4.1C.3 建立历史记录与复测能力

必须完成：
- 用户能看到自己的历史 attempt；
- 用户可重新测试；
- 当前结果与历史记录有区分；
- 复测不会覆盖掉旧记录。

#### 4.1C.4 建立结果分享/展示的最小产品化形式

至少考虑：
- 个人主页可见摘要；
- 可供后续状态流或聊天页轻量展示的标签结构；
- 但本版不要求完成社交分享产品链。

### 4.1C 任务边界解释

允许做：
- 结果页；
- 历史页；
- 复测入口；
- 维度解释模块。

禁止做：
- 做成大而全的内容社区；
- 为每种类型写超长内容站式文章产品；
- 把历史结果复杂可视化到过重程度；
- 把结果页面写成“官方认证人格报告”。

### 4.1C 验收标准

必须满足：
- 结果页清楚；
- 维度解释可读；
- 历史记录可区分；
- 复测不破坏旧数据；
- 结果展示具备最小产品感；
- 结果口径保持非官方、倾向性表达。

---

## 7. 4.1D：个人主页 / 匹配解释 / 首聊建议的轻量联动

### 4.1D 目标

让人格结果不只停留在“做过一次测试”，而是正式进入 EliteSync 的人格表达与轻量匹配链路中。

### 4.1D 必做任务

#### 4.1D.1 接入个人主页

必须完成：
- 个人主页展示人格结果摘要；
- 展示方式简洁，不挤压主页面；
- 没有结果时的空态明确。

#### 4.1D.2 接入匹配解释模块

必须完成：
- 人格结果作为一个模块进入匹配解释读取层；
- 至少能输出“人格倾向匹配提示”或等价标签；
- 不允许直接大改总匹配分逻辑。

#### 4.1D.3 接入首聊建议 / 互动风格提示

必须完成：
- 基于人格结果生成轻量风格建议；
- 例如：更偏理性、偏感受、偏外向、偏内向的首聊提示；
- 但不得做成复杂 AI 对话系统。

#### 4.1D.4 保持人格模块只是“模块之一”

必须明确：
- 人格结果是人格模块之一，不是唯一人格真相；
- 不覆盖星盘、画像、pair_chart 等其他模块；
- 不在本版中重新定义整个匹配体系。

### 4.1D 任务边界解释

允许做：
- 主页轻量展示；
- 匹配解释读取；
- 首聊建议轻量提示；
- 结果摘要卡。

禁止做：
- 大改匹配总分；
- 强行让没有测试结果的用户无法使用系统；
- 把人格结果做成平台唯一标签系统；
- 把人格结果与其他人格/玄学模块写成互斥关系。

### 4.1D 验收标准

必须满足：
- 人格结果能被主页与解释系统读取；
- 不影响原有主链稳定；
- 首聊提示有增益但不过度扩张；
- 人格模块间边界清晰。

---

## 8. 4.1E：回归、可观测性、文档与交付收口

### 8.1E 目标

让 4.1 不是“能测一次就结束”的版本，而是后续还能维护、调试、升级的正式能力版。

### 8.1E 必做任务

#### 8.1E.1 建立日志与错误记录

至少记录：
- 进入测试；
- 提交测试；
- 结果生成失败；
- 题库版本不匹配；
- attempt 写入失败；
- 历史记录读取异常。

#### 8.1E.2 建立回归清单

至少覆盖：
- 人格测试入口；
- 答题流程；
- 结果页；

## 9. 本版必须保护的边界（硬性保护面）

- `GET /api/v1/profile/astro/summary`

- `GET /api/v1/app/version/check`

- `app_release_versions`
- `moderation_reports`
- synthetic 批次审计字段
- 4.0 已落地的对象存储主路径与媒体状态机边界
- 4.0 已落地的会话、媒体、关系事件骨架
- 3.x 已归档的高级时法边界说明口径

### 9.1 为什么这些边界必须保护

#### 人格模块不应反向破坏 3.x 真值链
人格测试是新增模块，不是资料/命盘/画像主链重构，不得因为加测试系统而误伤既有真值链。

#### 人格模块不应干扰 4.0 的媒体基础设施
4.1 的重点是人格能力，不是媒体系统二次改造。现有对象存储、媒体状态机、附件骨架只允许被复用，不允许被重写。

#### 人格结果不是 canonical 画像替代物
人格结果可以作为人格模块参与表达和匹配解释，但不能替代 `profile/basic`、星盘画像或其他既有模块的真值语义。

## 10. 本版不允许出现的错误倾向

为避免 4.1 失控，以下倾向明确禁止：

1. 借人格测试之名扩大成完整人格系统平台；
2. 借结果展示之名引入动态流或内容社区；
3. 借人格结果接入之名大改匹配总分主线；
4. 借“更智能”之名引入复杂 AI 人格引擎；
5. 把人格结果写成永久不可变真值；
6. 把题库写死在前端，导致无法版本化；
7. 忽略历史记录与结果版本差异；
8. 为了展示炫酷而牺牲结果解释清晰度；
9. 使用来源不明题目原文，导致后续版权/商标风险；
10. 用“官方 MBTI”包装非官方问卷产品。
