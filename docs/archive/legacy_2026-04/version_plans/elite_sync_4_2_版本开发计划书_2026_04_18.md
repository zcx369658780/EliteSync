# EliteSync 4.2 版本开发计划书

日期：2026-04-18  
版本定位：`4.2 图片消息正式接入版`  
当前承接基线：`4.1 正式通过 / pass with observations`  
计划适用对象：Codex-CLI、Claude-MCP、Gemini-MCP、项目顾问验收

---

## 0. 本版定位与背景

### 0.1 为什么 4.2 现在启动

截至当前，EliteSync 已具备以下连续基线：

- **3.x** 已完成收口，真值链、星盘边界、发布链、版本中心、保护面已基本稳定；
- **4.0** 已完成基础能力基建版：对象存储主路径、媒体状态机、队列/缓存最小闭环、附件状态骨架与最小可观测性已经落地；
- **4.1** 已完成非官方四维人格问卷模块：版本化问卷、历史记录、复测、轻量主页/匹配联动与 walkthrough 证据链已建立。

这意味着：

> **现在最自然、最工程上顺畅的下一步，不是继续打磨 4.1，也不是跳去动态流或音视频通话，而是把 4.0 已经铺好的媒体基础设施，正式接到聊天主链上，完成图片消息能力。**

### 0.2 当前现实约束

本版规划以以下现实约束为前提：

1. 公司尚未注册，短期内仍无法通过真实运营去验证复杂社交流程；
2. 因此 4.2 必须优先完成“未来测试阶段一定会高频使用”的核心能力：**聊天发图、图片展示、图片失败重试、图片预览与最小治理**；
3. 4.2 必须建立在 4.0 已落地的对象存储、状态机、缓存与附件骨架上，不得绕开现有底座另起炉灶；
4. 4.2 是“图片消息正式接入版”，不是“媒体宇宙大版本”，不能把视频消息、动态流、音视频通话一起塞进来；
5. 图片消息一旦进入主聊天链路，后续所有版本都会受它影响，因此本版必须把错误态、失败态、预览、缓存与回归做扎实。

### 0.3 本版正式定位

4.2 的正式定位不是“给聊天页加个发图按钮”，而是：

> **4.2 图片消息正式接入版：把 4.0 的媒体基础设施与附件状态骨架，正式接到会话主链中，形成可发送、可接收、可预览、可失败重试、可缓存、可回归的图片消息能力。**

换句话说，4.2 要解决的是：
- 用户如何选择图片；
- 图片如何上传到对象存储主路径；
- 图片如何绑定到消息；
- 消息列表如何展示图片；
- 缩略图、原图、失败态、处理中态如何表现；
- 发送失败如何重试；
- 会话主链如何不被拖垮；
- 后续视频消息、状态流媒体与更完整媒体治理如何在此基础上继续扩展。

## 1. 4.2 的核心目标

本版只围绕以下 **7 个核心目标** 展开。

### 1.1 目标一：建立图片消息的正式发送链路

必须形成：
- 用户选图；
- 选图后进入上传态；
- 上传到 4.0 已落地的对象存储主路径；
- 生成/绑定 `media_assets`；
- 建立消息与附件关系；
- 会话列表与消息列表中能看到图片消息。

### 1.2 目标二：把附件状态机真正接入聊天主链

必须形成：
- 待发送；
- 上传中；
- 处理中；
- 发送成功；
- 失败；
- 重试；
- （必要时）取消/移除草稿态。

不允许附件状态仍停留在 4.0 的“只展示状态骨架”层面。

### 1.3 目标三：建立缩略图 / 预览 / 全屏查看的最小闭环

必须形成：
- 会话列表中优先展示缩略图或轻量图片卡；
- 点击可进入更清晰预览；
- 不要求做完整相册产品，但必须具备最小可用的图片查看能力；
- 不能因加载原图导致聊天页面明显卡顿。

### 1.4 目标四：建立失败重试与弱网可理解性

必须形成：
- 用户能知道上传失败了；
- 用户知道是“发送失败”还是“处理中”；
- 用户可以点击重试；
- 至少有基础的失败文案与状态提示。

### 1.5 目标五：建立图片缓存与清理策略的最小闭环

必须形成：
- 缩略图与原图分层；
- 不把大图长期塞满内存；
- 聊天列表滚动时优先轻量展示；
- 为未来更大规模媒体消息留出缓存边界。

### 1.6 目标六：建立最小治理与最小安全边界

必须形成：
- 类型限制；
- 大小限制；
- 图片 mime 校验；
- 基础错误码；
- 基础举报/阻断挂点预留；
- 不允许把图片消息写成“无约束上传”。

### 1.7 目标七：明确 4.2 不做什么，避免范围失控

必须明确：
- 4.2 不做视频消息正式版；
- 4.2 不做动态流图片媒体产品；
- 4.2 不做音视频通话；
- 4.2 不做完整聊天媒体相册；
- 4.2 不做图片 AI 审核平台；
- 4.2 不做复杂 CDN / 签名 URL 工业化大升级；
- 4.2 不做重型聊天系统改造。

## 2. 本版明确不做的内容

为了避免 4.2 演变为“全媒体大版本”，以下内容明确列为 **本版不做**。

### 2.1 不做视频消息正式产品

不做：
- 视频选取；
- 视频上传；
- 视频封面生成；
- 视频卡片；
- 视频播放链路。

原因：这些属于 **4.4** 的主任务，不能和图片消息一起硬绑。

### 2.2 不做动态流 / 状态流媒体产品

不做：
- 状态发布配图；
- 动态图文 feed；
- 图片评论链；
- 图片点赞与推荐。

原因：这些属于 **4.3** 主线，不应在 4.2 里偷渡。

### 2.3 不做音视频通话与实时在线状态

不做：
- 呼叫；
- RTC；
- 在线状态；
- 实时推流。

原因：这是 **4.6** 的高风险版本。

### 2.4 不做完整媒体治理平台

不做：
- 完整图片审核平台；
- 高级内容风控后台；
- AI 内容识别重系统；
- 多级人工审核工作台。

原因：4.2 只需要保留治理挂点与最小安全策略。

### 2.5 不做复杂上传工业化方案

不做：
- 分片上传；
- 断点续传；
- 客户端直传完整 SDK；
- 大规模签名 URL 平台；
- 复杂图片编辑器。

原因：4.0 已建立对象存储最小闭环，4.2 的重点是接入聊天主链，而不是把上传协议做成平台级产品。

### 2.6 不做匹配、人格、星盘主线改造

不做：
- 修改 4.1 人格问卷主线；
- 修改 3.x 星盘 / pair_chart / timing 主线；
- 修改匹配总分主结构。

原因：4.2 的重点是图片消息，不是解释系统或人格系统扩容。

## 3. 4.2 的工作包拆分

本版建议拆为 **5 个工作包**，统一作为 4.2 版本主线实施。

- **4.2A：后端图片消息契约与附件绑定闭环**
- **4.2B：Flutter 选图、上传、失败重试与消息发送闭环**
- **4.2C：图片渲染、缩略图预览与查看器最小闭环**
- **4.2D：缓存、错误处理、弱网体验与最小安全治理**
- **4.2E：回归、可观测性、截图证据与交付收口**

这些工作包不是独立产品版本，而是 4.2 的分阶段执行结构。

---

## 4. 4.2A：后端图片消息契约与附件绑定闭环

### 4.2A 目标

把 4.0 的媒体基础设施真正接入消息主链，形成“图片上传成功后能成为一条正式消息”的结构闭环。

### 4.2A 必做任务

#### 4.2A.1 明确图片消息的消息模型

必须明确：
- 文本消息与图片消息的共存方式；
- `chat_messages` 既有主链如何保持；
- `message_attachments` 如何绑定图片资产；
- 一条消息是否允许带单图 / 多图；
- 本版建议优先只做 **单图消息**，多图仅预留结构。

#### 4.2A.2 建立图片发送接口闭环

必须形成：
- 上传图片；
- 创建 / 获取 `media_assets`；
- 校验图片状态达到可发送条件；
- 创建消息；
- 绑定附件；
- 返回可被聊天页消费的消息结构。

#### 4.2A.3 明确消息状态与附件状态的关系

必须明确：
- 图片附件处于 `uploading` 时，消息如何表现；
- 图片附件处于 `processing` 时，消息如何表现；
- 图片附件 `ready` 后消息如何表现；
- 图片附件失败时，消息与附件关系是否保留；
- 失败重试是否复用原消息草稿或创建新消息草稿。

#### 4.2A.4 建立发送幂等与错误码基础口径

必须明确：
- 重复点击发送如何处理；
- 上传成功但绑定失败如何处理；
- 绑定成功但列表未刷新如何处理；
- 基础错误码如何表达给前端。

### 4.2A 任务边界解释

允许做：
- 新接口；
- `message_attachments` 与 `media_assets` 绑定；
- 后端状态流补强；
- 单图消息发送闭环。

禁止做：
- 大改 `chat_messages` 文本主链；
- 一次性上多图、长图、相册消息；
- 引入视频或音频附件；
- 因图片消息重写整个会话协议。

### 4.2A 验收标准

必须满足：
- 图片可经由媒体主路径进入消息主链；
- 消息与附件结构稳定；
- 单图消息能够被查询、展示与重试；
- 文本消息主链未受破坏。

---

## 5. 4.2B：Flutter 选图、上传、失败重试与消息发送闭环

### 4.2B 目标

让用户真正能在聊天页完成“选择图片 → 上传 → 发送 → 失败重试”的产品动作。

### 4.2B 必做任务

#### 4.2B.1 接入选图入口

必须完成：
- `MessageInputBar` 附件按钮真正接入图片选择；
- 选择后进入待发送预备态；
- 失败或取消时能回退，不污染输入区。

#### 4.2B.2 建立上传与发送状态骨架

必须形成用户可见状态：
- 已选择待发送；
- uploading；
- processing；
- failed；
- retry；
- sent / ready。

#### 4.2B.3 建立失败重试逻辑

必须完成：
- 失败态有明确可点击动作；
- 用户知道重试的是“重新上传”还是“重新绑定消息”；
- 重试成功后 UI 状态能正确更新；
- 不允许失败后只剩一条无法处理的废消息。

#### 4.2B.4 发送体验不破坏既有文本输入主链

必须明确：
- 有图片时是否允许带文案；
- 没图片时文本发送逻辑不变；
- 图片消息不会导致输入框或会话列表出现明显错乱。

### 4.2B 任务边界解释

允许做：
- 选图接入；
- 待发送状态；
- 失败重试；
- 上传与消息发送联动。

禁止做：
- 一次性做多图编辑器；
- 一次性做图片裁剪美化工具；
- 因图片发送重写整套聊天页结构；
- 为演示效果绕开真实状态机。

### 4.2B 验收标准

必须满足：
- 用户能实际选图并发送；
- 失败时能看到状态并重试；
- 文本发送主链仍正常；
- 发送体验不是“看似可用、实则不可恢复”的假闭环。

---

## 6. 4.2C：图片渲染、缩略图预览与查看器最小闭环

### 4.2C 目标

让图片消息不仅能“发出去”，还要在聊天中“看得见、点得开、不会拖垮性能”。

### 4.2C 必做任务

#### 4.2C.1 会话中展示缩略图

必须完成：
- 聊天列表优先展示缩略图或轻量图片卡；
- 图片消息与文本消息区分清楚；
- 上传中、处理中、失败态的视觉与成功态不同。

#### 4.2C.2 建立点击预览 / 查看器

必须完成：
- 点击图片消息进入更大视图；
- 最小可用的图片查看器；
- 支持返回聊天页；
- 不要求做完整相册或复杂手势系统，但至少要可查看大图。

#### 4.2C.3 建立缩略图与原图分层策略

必须明确：
- 聊天列表加载什么；
- 查看器加载什么；
- 原图是否按需拉取；
- 失败加载时的占位策略。

### 4.2C 任务边界解释

允许做：
- 缩略图展示；
- 全屏或半屏查看器；
- 基础加载占位；
- 轻量图片卡样式。

禁止做：
- 完整媒体相册；
- 图片编辑器；
- 长图识别、OCR、智能标签等扩展功能；
- 图片社交化二次玩法。

### 4.2C 验收标准

必须满足：
- 图片消息在聊天里可见；
- 点击后可查看更清晰版本；
- 会话滚动不被原图加载明显拖垮；
- 成功态与失败态可区分。

---

## 7. 4.2D：缓存、错误处理、弱网体验与最小安全治理

### 4.2D 目标

让图片消息不是“实验性跑通”，而是具备最小产品可信度。

### 4.2D 必做任务

#### 4.2D.1 建立图片缓存边界

必须明确：
- 缩略图缓存；
- 原图按需加载；
- 内存与磁盘缓存分层；
- 清理策略或最小回收策略。

#### 4.2D.2 建立错误提示与弱网体验

必须完成：
- 上传失败提示；
- 下载失败提示；
- 处理中提示；
- 网络较差时的基本用户反馈；
- 不允许用户误以为“已经发出去”但其实失败了。

#### 4.2D.3 建立图片安全与限制策略

必须明确：
- mime 白名单；
- 大小限制；
- 非图片文件误上传处理；
- 图片异常损坏处理；
- 基础举报/阻断挂点（至少保留结构）。

#### 4.2D.4 建立最小消息治理边界

必须考虑：
- 被 blocked 的图片怎么展示；
- 被删除的图片怎么展示；
- 错误码如何映射到用户文案；
- 不能让危险状态直接穿透到 UI 裸露工程异常。

### 4.2D 任务边界解释

允许做：
- 缓存策略；
- 弱网提示；
- 大小 / 类型校验；
- 错误文案映射。

禁止做：
- 重型风控系统；
- 复杂内容审核平台；
- 为 4.2 引入极重的 CDN/签名系统改造；
- 因治理挂点去改变 3.x/4.1 业务主链。

### 4.2D 验收标准

必须满足：
- 图片消息在弱网下不会假成功；
- 错误态对用户可理解；
- 缓存行为不至于拖垮聊天体验；
- 基础类型与大小限制有效。

---

## 8. 4.2E：回归、可观测性、截图证据与交付收口

### 8.2E 目标

让 4.2 成为一个可归档、可交接、可继续扩展的正式图片消息版本，而不是只在聊天里勉强跑通。

### 8.2E 必做任务

#### 8.2E.1 建立最小 telemetry / observability

至少覆盖：
- 图片选择打开；
- 图片上传开始；
- 图片上传成功；
- 图片上传失败；
- 图片重试；
- 图片发送成功；
- 图片查看器打开；
- 图片查看失败（如有）。

要求：
- 仅做最小可观测性；
- 不泄露敏感图片内容；
- 不把 telemetry 当真值。

#### 8.2E.2 建立回归清单

至少覆盖：
- 文本消息主链；
- 图片消息发送；
- 图片失败重试；
- 图片预览；
- 缓存 / 弱网基础行为；
- 3.x 真值链保护面；
- 4.1 人格模块不受误伤。

#### 8.2E.3 建立截图证据链

至少覆盖：
- 选图入口；
- 上传中态；
- 处理中态；
- 失败态；
- 重试成功态；
- 聊天中图片消息；
- 图片预览页；
- 版本中心（如仍需对齐当前版本）。

#### 8.2E.4 建立文档与 handoff

必须产出：
- 4.2 执行记录；
- 4.2 验收摘要；
- 4.2 多 Agent 审查日志；
- 4.2 截图证据索引；
- 4.2 截图验收说明；
- 4.2 handoff 文档；
- 已知限制与后续承接说明。

### 8.2E 任务边界解释

允许做：
- 日志；
- 测试；
- 截图证据；
- handoff；
- 文档补齐。

禁止做：
- 只做代码不补材料；
- 用埋点取代业务验证；
- 为了截图而临时造不真实的产品路径；
- 在文档里掩盖当前未做内容。

### 8.2E 验收标准

必须满足：
- 错误可定位；
- 回归可执行；
