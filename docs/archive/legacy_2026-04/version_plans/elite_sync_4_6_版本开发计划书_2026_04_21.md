# EliteSync 4.6 版本开发计划书

文件名：`elite_sync_4_6_版本开发计划书_2026_04_21.md`  
更新时间：2026-04-21  
适用范围：`4.5` 已正式归档通过后的下一版本承接  
当前承接基线：`4.0` 媒体基础设施、`4.1` 非官方四维人格问卷、`4.2` 图片消息主链、`4.3` 动态流基础版、`4.4` 视频消息版、`4.4S` 媒体链稳定性修正版、`4.5` 通知与社交转化增强版均视为稳定基线。  
本版定位：`4.6 RTC / 通话基础设施版（语音优先）`

---

## 0. 本版一句话结论

`4.6` 不做“大而全实时社交平台”，也不做多人房间、直播或复杂在线体系，而是把 EliteSync 从“异步消息 + 通知回流”推进到“可发起、可响铃、可接通、可挂断”的 **1v1 实时通话最小闭环**。

本版只做：
- 1v1 会话内实时通话的最小基础设施
- **语音通话优先**，视频通话只做骨架预留或受控最小闭环
- 通话发起、响铃、接受、拒绝、挂断、超时、占线等基础状态机
- 权限检查、设备准备、网络状态与失败态最小收口
- 与现有消息页、通知中心、媒体链、匹配链的 additive 联动
- 顺手修掉与通话页直接相关的 4.5 轻量观察项（如通知卡片工程化 type slug 暴露）

本版明确不做：
- 多人通话
- 直播 / 房间 / 语聊房
- 在线状态全平台体系
- 视频美颜 / 滤镜 / 贴纸
- 复杂通话计费 / 礼物 / 运营玩法
- 完整厂商推送平台化
- 微服务重构

换言之，`4.6` 的目标不是把 EliteSync 做成 RTC 平台，而是建立一个 **收敛、可信、可被验收的 1v1 实时沟通基础层**。

## 1. 版本定位

## 1.1 版本名称

`4.6 RTC / 通话基础设施版（语音优先）`

## 1.2 版本使命

在不破坏 `3.9 / 4.0 / 4.1 / 4.2 / 4.3 / 4.4 / 4.4S / 4.5` 已归档主链的前提下，补齐以下能力：

1. 用户可从现有会话或明确入口中发起 1v1 通话  
2. 对方可收到站内实时来电提示，并进行接受 / 拒绝 / 忽略  
3. 通话状态机最小闭环成立：发起、响铃、接通、挂断、拒绝、超时、占线、失败  
4. 权限、设备、网络失败态可解释，不把底层异常直接暴露给用户  
5. 通话与既有消息、通知、媒体、匹配链形成 additive 联动，而不重写它们  
6. 为后续视频通话增强、外部推送唤醒、在线状态精细化预留边界，但本版不做大扩展  

## 1.3 为什么现在做 4.6

### 原因 A：4.5 已经建立了通知与回流层

现在用户不仅能收到消息和互动提醒，也已经有站内通知和页面回流能力。实时通话再往前走，才有明确的提醒入口和召回路径。

### 原因 B：4.4 / 4.4S 已把媒体读取与播放链修到可用

图片和视频消息、媒体 URL、播放链已经稳定，说明音视频资源与设备侧媒体基础能力已有承接条件，适合进入 RTC 的下一层。

### 原因 C：继续做“异步增强”边际收益开始下降

当前已有：
- 图片 / 视频消息
- 动态流与互动
- 通知中心与社交转化
- 匹配解释与首聊建议

下一步最自然的升级不再是继续堆异步入口，而是让“从匹配 -> 消息 -> 通知 -> 实时通话”的关系链成立。

### 原因 D：必须继续坚持小步版本策略

如果 4.6 一次性做成“音频 + 视频 + 在线状态 + 外推唤醒 + 多端同步 + 厂商推送”，几乎必然越界。  
因此本版必须坚持：
- **语音优先**
- 视频仅做预留或受控最小闭环
- 先建立信令 / 状态机 / 权限 / 页面基本可信度

## 2. 承接基线

本版必须承接并保护以下基线：

### 2.1 已冻结的真值与主链

- `POST /api/v1/profile/basic` 仍是资料真值入口
- `user_astro_profiles` 仍是画像 / 星盘核心真值表
- `chat_messages` 仍是消息主链真值表
- `app_release_versions` 仍是发布真值表
- `questionnaire_attempts / questionnaire_answers / questionnaire_questions` 仍是问卷真值表

### 2.2 已冻结的能力底座

- `4.0` 媒体 / 存储 / 队列 / 缓存 / worker 最小闭环
- `4.1` 非官方四维人格问卷模块
- `4.2` 图片消息主链
- `4.3` 动态流基础版
- `4.4` 视频消息版
- `4.4S` 媒体链稳定性修正版
- `4.5` 通知与社交转化增强版

### 2.3 已存在的相关模块面

- 首页 `Home`：轻量联动与社交转化入口可用
- 匹配 `Match`：解释、首聊建议、转化入口稳定
- 消息 `Chat`：文本 + 图片 + 视频消息主链可用
- 通知 `Notification Center`：消息 / 动态 / 匹配类通知与回流已成立
- 动态 `Status / Feed`：发布、读取、互动与治理挂点稳定
- 我的 `Me / Settings`：权限说明、通知设置、版本中心已具备承载能力

### 2.4 本版最重要的承接口径

`4.6` 是 **RTC / 通话基础设施版**，不是“通话完整版”，更不是“直播 / 房间 / 在线社交大版本”。  
所有实现都应遵循：

- additive feature，不做 destructive rewrite
- 先做 1v1、先做语音优先，再考虑视频增强
- 先做站内实时通话闭环，再考虑外部推送唤醒
- 先做状态机可信，再做视觉 polish
- 先保护现有消息 / 通知 / 媒体主链，不重写它们

## 3. 本版做什么 / 不做什么

## 3.1 本版必须完成的核心目标

### 目标 1：1v1 通话发起与响铃闭环

至少支持：
- 从会话页或明确入口发起语音通话
- 对方在应用内收到来电界面或来电提示
- 对方可接受 / 拒绝 / 忽略
- 发起方能看到正在呼叫、对方拒绝、超时未接等状态

### 目标 2：通话状态机最小闭环

至少支持：
- `idle`
- `calling`
- `ringing`
- `connecting`
- `in_call`
- `rejected`
- `missed`
- `busy`
- `ended`
- `failed`

并要求：
- 前后端状态口径一致
- UI 不得伪装成功
- 超时 / 占线 / 网络失败要有合理降级

### 目标 3：权限、设备与失败态闭环

至少支持：
- 麦克风权限检查与请求
- 视频预留时的摄像头权限骨架（若本版不做视频可只做预留）
- 设备不可用、权限被拒、网络中断、信令失败等错误态
- 失败态用户可理解，而不是只暴露底层异常

### 目标 4：通知与回流闭环

至少支持：
- 站内来电提醒与回流
- 未接来电 / 已拒绝 / 已结束在通知层的最小可见记录
- 从通知回到对应会话或通话结果页
- 不要求本版打通外部推送，但要为后续预留 token / delivery 边界

### 目标 5：与现有页面的轻量联动

至少支持：
- 会话页可见明确通话入口
- 匹配页或首聊建议页若已有高相关入口，可轻量联动到“先聊再考虑通话”或“发起通话”
- 不重写首页、动态页、消息主导航结构

### 目标 6：附带清理项（仅限直接相关 observation）

仅允许顺手修掉下列与通话页 / 通知页直接相关、已知且轻量的观察项：
- `4.5` 通知卡片上暴露的 `message / status_like / match_success` 这类工程化 type slug，改成更自然的用户面文案或弱化展示
- 通话相关通知、未接来电卡片和会话摘要的产品化口径统一

> 说明：这些是 **附带清理项**，不是单独开新版本，也不是 4.5 重开。只能在你本轮明确触达通知与会话页时顺手修，不得扩成通知大改版。

## 3.2 本版明确不做的内容

以下内容禁止偷渡进入 `4.6`：

### 通话能力不做
- 多人通话
- 群语音 / 群视频
- 直播 / 语聊房 / 房间制 RTC
- 通话录音 / 转写 / AI 实时摘要
- 通话内礼物、贴纸、互动特效

### 视频能力不做或仅做预留
- 视频美颜
- 滤镜 / 背景虚化
- 屏幕共享
- 通话内图片 / 文件协作
- 复杂的视频升级 / 降级策略

### 通知能力不做
- 厂商推送全量接入
- APNs / FCM 全量平台化
- 通知偏好中心完整产品化
- 营销或运营消息系统

### 在线状态不做
- 全局在线状态体系
- 最后在线时间产品化
- 跨端 presence 完整同步
- “正在输入 / 正在通话 / 正在浏览” 全量状态层

### 架构能力不做
- 微服务重构
- 即时通讯 / RTC 中台化
- 新建复杂事件总线平台
- 对 `4.0 / 4.2 / 4.4 / 4.5` 已归档主链推倒重来

### UI 能力不做
- 全站视觉重做
- 消息页大改版
- 首页 / 匹配页 / 通知中心全面翻修
- 把通话页做成大而全音视频控制台

## 4. 硬性保护面

本版开发前必须先冻结以下保护面，不得误伤：

## 4.1 真值链保护面

- `POST /api/v1/profile/basic`
- `GET /api/v1/profile/astro/summary`
- `GET /api/v1/profile/astro/chart`
- `user_astro_profiles`
- `questionnaire_*` 真值链

### 解释

通话层是“实时互动层”，不是“资料 / 画像 / 匹配真值层”。  
任何通话行为、接听率、时长、拒绝、未接，都不得反向写成 canonical truth。

## 4.2 消息与媒体保护面

- `GET /api/v1/messages`
- `POST /api/v1/messages`
- `POST /api/v1/media`
- `chat_messages`
- `media_assets`
- `media_processing_jobs`
- `message_attachments`

### 解释

`4.6` 可在消息页上增加通话入口与通话记录语义，但不能为了通话去改坏文本 / 图片 / 视频消息主链和媒体状态机。

## 4.3 通知与回流保护面

- `GET /api/v1/notifications`
- `POST /api/v1/notifications/read/{notificationId}`
- `POST /api/v1/notifications/read-all`
- `notifications`
- `notification_center` 前端主链

### 解释

`4.6` 会消费通知能力，但不能借机重构 4.5 的通知中心主链；本版只允许做与来电 / 未接 / 通话结果直接相关的最小联动与轻量文案降噪。

## 4.4 发布与版本保护面

- `GET /api/v1/app/health`
- `GET /api/v1/app/version/check`
- `app_release_versions`
- 宿主 APK 版本口径
- about / version center / release script 绑定链

### 解释

`4.6` 是实时基础设施功能版，不是发布链重构版。  
任何新页面、新服务状态提醒都不能破坏版本链一致性。

## 4.5 环境与数据库边界（长期硬规则）

本版必须严格遵守：

- 本地环境只做前端开发、UI 联调、截图采集、文档整理
- 后端开发、数据库迁移、备份、恢复、写库排障统一在阿里云端执行
- 不允许本地开发库、SQLite、临时数据库污染云端正式后端数据

### 解释

`4.6` 会引入更多实时状态与写库事件，更必须守住“本地前端 / 云端后端”边界，不得再出现本地覆盖远端数据库的隐患。

## 5. 数据与接口边界建议

## 5.1 数据模型建议

优先复用现有核心表：
- `notifications`
- `app_events`
- `chat_messages`
- `conversations`
- `conversation_members`
- `user_relationship_events`
- `device_tokens`（若 4.5 已预留）

在此基础上，按最小必要原则新增或补齐：
- `rtc_sessions`：1v1 通话会话主表，仅记录通话状态、双方、开始/结束、失败原因等
- `rtc_session_events`：通话事件流水表，仅用于状态迁移与排障
- `call_notifications`（如无必要不单独建表，优先复用 `notifications` 类型）
- `rtc_device_capabilities`（若确有必要，仅做最小设备/权限缓存，不做复杂 profile）

### 边界解释

本版不允许为了“以后扩展多人通话”一次性建一个大而全 RTC 域模型。  
如果 `rtc_sessions + rtc_session_events + notifications` 足够承接 1v1 最小闭环，就优先保持收敛。

## 5.2 接口域建议

### 优先策略

- 实时信令：优先收口到 `/api/v1/rtc/*` 或等价最小命名域
- 通知回流：继续复用 `/api/v1/notifications/*`
- 消息页通话入口：复用既有会话与用户关系事实源，不新造用户真假关系表

### 建议最小接口集合

- `POST /api/v1/rtc/calls`：发起通话
- `POST /api/v1/rtc/calls/{callId}/accept`：接受通话
- `POST /api/v1/rtc/calls/{callId}/reject`：拒绝通话
- `POST /api/v1/rtc/calls/{callId}/end`：挂断通话
- `GET /api/v1/rtc/calls/{callId}`：读取当前通话状态
- `POST /api/v1/rtc/calls/{callId}/heartbeat` 或等价机制：仅在确有必要时用于状态保活
- `GET /api/v1/rtc/config`：返回 RTC 配置、token、服务参数（如确有必要）

若需要最小结果页或历史预留，可考虑：
- `GET /api/v1/rtc/history`

### 边界解释

本版不要求一次性做完完整 RTC 平台 API。  
核心原则是：
- 呼叫路径清晰
- 状态路径可信
- 失败路径可解释
- 页面能真实响应信令状态
- 不新造一堆脱离主链的试验性接口

## 6. 工作包拆分

## 6.1 4.6A：边界冻结与 RTC 状态机收口

### 目标

在动代码前，先把 `4.6` 的版本边界、1v1 通话范围、状态机、权限与环境边界写清楚，避免版本越界。

### 必做任务

1. 明确本版是 **语音优先的 1v1 通话基础设施版**  
2. 明确视频是预留还是仅做最小受控闭环  
3. 明确通话状态机与状态迁移图  
4. 明确响铃超时、占线、拒绝、挂断、失败的口径  
5. 明确本版是否需要设备 token / 前台保活 / 外推预留  
6. 明确本版不做多人、直播、运营推送、在线状态全量体系  
7. 明确仅允许顺手收 4.5 的轻量通知文案 observation，不扩大为通知大改版  

### 必要验收条件

- 存在一份清晰的边界说明
- 列出新增表 / 新增字段 / 新增接口清单
- 明确哪些旧表和旧接口不可碰
- 明确语音优先与视频预留的边界

### 解释

没有边界说明直接进入 RTC，最容易出现：
- 一版做成语音+视频+在线状态+推送一锅炖
- 信令与通知、会话、消息、状态写乱
- UI 先做很炫，但状态机不可信

因此 `4.6A` 是本版硬前置，不可省略。

---

## 6.2 4.6B：后端 RTC 会话、信令与状态闭环

### 目标

建立 1v1 通话会话、基础信令、状态更新、超时与失败处理的最小后端闭环。

### 必做任务

1. 完成通话发起与会话创建  
   - 发起呼叫
   - 判断目标可否进入响铃状态
   - 写入 `rtc_sessions`

2. 完成基础状态迁移  
   - 发起 -> 响铃
   - 接受 -> 连接中 -> 通话中
   - 拒绝 / 超时 / 占线 / 失败 -> 结束
   - 挂断 -> 结束

3. 完成最小信令传递  
   - 站内可用信令或最小 WebSocket / SSE / polling 方案
   - 只要能可靠触达双方页面即可
   - 不追求一版做复杂中台

4. 完成通话结果回流  
   - 未接、拒绝、挂断、通话结束可形成通知或会话结果卡片
   - 与消息会话保持一致语义

5. 完成最小 telemetry / 事件记录  
   - `rtc_call_started`
   - `rtc_call_ringing`
   - `rtc_call_accepted`
   - `rtc_call_rejected`
   - `rtc_call_missed`
   - `rtc_call_ended`
   - `rtc_call_failed`

### 必要验收条件

- 发起、接受、拒绝、挂断的后端状态一致
- 超时与占线可被识别
- 状态不漂移、不假成功
- 失败原因可记录

### 解释

本工作包的目标不是做成熟 RTC 平台，而是把“可发起、可响铃、可结束”的后端主链闭合。  
如果后端只有一个“call 按钮”，没有状态机和信令，那版本不能算成立。

---

## 6.3 4.6C：Flutter 通话页、来电页与会话入口

### 目标

让用户能够在产品化界面中发起通话、接收来电、看到状态、结束通话，并理解当前发生了什么。

### 必做任务

1. 会话页通话入口  
   - 在聊天页增加清晰但不过度突兀的通话入口
   - 不把消息页重做成通话控制台

2. 来电界面 / 响铃界面  
   - 来电人信息
   - 接受 / 拒绝
   - 超时或对方取消时的状态变化

3. 通话进行页  
   - 语音优先的基础通话 UI
   - 麦克风静音、挂断、扬声器（如需要）等最小控制
   - 若视频仅预留，则不要做假视频页

4. 通话结果页或回退态  
   - 拒绝、未接、失败、已结束可解释
   - 回到会话页或通知页路径清楚

5. 顺手收口 4.5 轻量观察项  
   - 去掉用户面通知卡片中明显工程化的 type slug 直出
   - 统一来电 / 未接 / 已结束相关通知与摘要的产品化文案

### 必要验收条件

- 入口清晰，不需要工程背景才能理解
- 来电与通话中页面可读，不像调试页
- 接受 / 拒绝 / 挂断状态可见
- 失败态可理解，不把底层异常大段裸露给用户

### 解释

对 `4.6` 而言，前端的关键不是动画多炫，而是“状态变化是否可信、是否有足够产品感”。

## 6.4 4.6D：权限、网络、失败态与回归保护

### 目标

在版本收尾前，把 RTC 最容易翻车的部分提前收住：权限、网络、失败态、旧主链回归。

### 必做任务

1. 权限收口  
   - 麦克风权限
   - 视频预留时的摄像头权限骨架
   - 被拒绝后的引导

2. 网络与信令失败兜底  
   - 网络波动
   - 对方离线 / 不可达
   - 信令丢失 / 超时
   - 页面切后台或前后台切换最小处理

3. 回归保护  
   - `profile/basic` 真值链回归
   - `astro/summary`、`astro/chart` 回归
   - `messages` 文本 / 图片 / 视频消息回归
   - `notifications` 列表 / 已读 / 回流回归
   - `status_posts` 动态流回归
   - `questionnaire/history`、`questionnaire/answers` 回归
   - `app/version/check` 回归

4. 本地 / 云端边界检查  
   - 本地不做后端写库 RTC 排障
   - 云端执行信令、写库、迁移、修复
   - 明确数据库隔离仍有效

### 必要验收条件

- 权限失败与网络失败可解释
- RTC 不会明显打坏消息、通知、动态主链
- 回归清单完整
- 云端数据库边界未被破坏

### 解释

因此本工作包不是附录，而是正式验收的一部分。

## 8. 4.6 的验收结论建议口径

## 8.1 何时可以给 `pass`

仅当以下同时成立：
- 发起、响铃、接受、通话中、挂断、失败态全链路成立
- UI 状态表达清楚
- 回归完整
- 截图证据完整
- handoff / closeout 完整
- 无需再补关键收尾材料

## 8.2 何时给 `pass with observations`

用于：
- 主链已经完整成立
- 可以正式归档
- 仍有少量非阻断观察项，例如：
  - 某些通话页视觉细节仍偏系统化
  - 某些失败态文案还可继续降噪
  - 视频增强能力仍停留在预留层

### 建议

`4.6` 最现实的目标是：**争取做到 `pass with observations` 后冻结主链。**  
不要为了追求“更完整实时社交平台”而在 `4.6` 上反复开补丁。

## 8.3 何时只能给 `conditional pass`

若出现以下情况之一：
- 通话主链虽跑通，但截图证据包明显不足
- 没有 closeout / handoff
- 信令状态、失败态、页面回流说明不清楚
- 只能演示单侧页面，无法证明双侧状态链

则应判为 `conditional pass`，并仅允许小回合补收尾，不建议重开大开发。

## 9. 风险清单

## 9.1 最大风险：版本越界成“完整实时平台版”

表现：
- 音频、视频、在线状态、外推唤醒一起做厚
- 继续堆多人、直播、房间、运营玩法
- 把 `4.6` 做成半年大版本

控制原则：
- 只做 1v1、语音优先、站内最小闭环
- 不把后续版本内容偷渡进来

## 9.2 第二风险：信令与状态机不可信

表现：
- 页面显示响铃，但后端已结束
- 双方状态不同步
- 已挂断还显示通话中

控制原则：
- 先收口状态机
- 先保证状态可信，再做 polish

## 9.3 第三风险：误伤消息 / 通知主链

表现：
- 为了通话去改坏消息会话页
- 通话通知把 4.5 通知中心写乱
- 通话结果污染消息摘要语义

控制原则：
- 通话层只消费并联动既有主链，不重写主链
- 只允许顺手修与通话直接相关的 4.5 轻量 observation

## 9.4 第四风险：权限与失败态处理粗糙

表现：
- 麦克风权限被拒后用户无从理解
- 网络失败直接抛底层报错
- 对方离线 / 超时无解释

控制原则：
- 失败态必须产品化降噪
- 不能把底层异常直接当主文案

## 9.5 第五风险：本地 / 云端写库边界失守

表现：
- 本地后端写库再次污染云端数据
- 临时数据库被误当正式源

控制原则：
- 本地只做前端联调
- 后端写库、迁移、恢复、修复统一云端执行

- 弱网下重试或失败解释

## 14.4 数据一致性回归

- 双方状态一致
- 通话结果与通知结果一致
- 通话行为不会污染画像 / 匹配真值层
- 本地环境不会触发云端数据库污染

## 15. 版本结束时的文档要求

本版结束时必须至少形成：

1. `4.6` 边界说明  
2. 实现摘要  
3. 回归清单与执行结果  
4. walkthrough 截图证据索引  
5. closeout note  
6. advisor handoff  
7. 若有 observation，必须明确写清为何不阻断归档  

---

## 16. 最终建议

本版的最佳策略不是贪大求全，而是：

**把实时通话层做成一个收敛、真实、可被验收、可被承接的基础版本。**

也就是说：
- 不回头重开 `4.5`
- 不把 `4.6` 做成多人通话 / 在线状态 / 外推平台大版本
- 不破坏真值链、消息主链、媒体主链与通知主链
- 严格执行本地前端 / 云端后端边界
- 仅顺手收口与通话直接相关的 4.5 轻量文案 observation
- 优先争取 `pass with observations` 后冻结主链，进入下一版本

如果本计划被严格执行，那么 `4.6` 完成后，EliteSync 将从“能异步沟通、能被通知回流”进一步推进到“能进行可信的 1v1 实时沟通”的可测试产品状态，为后续更完整的视频通话增强、外部推送唤醒和 presence 能力建立健康承接基线。
