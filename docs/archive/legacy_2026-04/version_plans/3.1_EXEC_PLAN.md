# EliteSync 3.1 执行计划

更新时间：2026-04-09  
目标版本：`3.1`  
当前阶段：**账号分层底座 + 状态发布底座实施中**

## 1. 版本目标

3.1 的目标是打通最小社交闭环与测试运营底座：

- 注册
- 测试账号池
- 匹配
- 首聊 / 互动
- 状态发布
- 广场展示
- 管理员控制

## 2. 需求理解

本轮核心不是 UI 细调，而是把“可生成、可复用、可管理、可调试”的最小社交链路做实。

### 3.1 需要解决的主问题
1. 测试账号丢失后如何重建可复用账号池
2. 普通用户、测试账号、管理员如何分层
3. 管理员如何在 APP 内获得最小控制面
4. 用户如何发布状态并进入广场展示
5. 如何保证测试账号具备完整可调试画像

## 3. 明确不做

- 图片状态 / 多媒体状态
- 帖子编辑
- 评论 / 回复 / 点赞 / 转发
- 复杂推荐算法
- 完整话题系统
- 管理员大报表平台
- 大规模匹配算法重写
- 高风险数据库大改或多服务拆分

## 4. 受影响模块

### 4.1 后端
- 测试账号生成 / 重建脚本
- 账号类型与权限字段
- 管理员 API
- 状态发布 / 广场流
- 匹配调试开关

### 4.2 前端
- “我的”页管理员入口
- 管理员用户列表 / 审核 / 匹配控制
- 发布状态入口
- 首页广场展示

### 4.3 数据层
- `users`
- 测试账号标识 / 账号类型字段
- 状态发布表 / 广场数据源
- 运营指标隔离字段

## 5. 实施顺序

1. 补齐账号分层与测试账号池定义
2. 落地测试账号生成 / 重建脚本
3. 落地管理员 APP 内控制面
4. 落地状态发布与广场最小闭环
5. 接入最小回归与验收材料

## 6. 验证原则

- 所有新字段默认值必须安全
- 生成脚本必须可重建、可重复执行
- 测试账号不得污染真实指标
- 管理员高风险动作必须有最小审计与可回查能力
- 新链路不得破坏注册 / 登录 / 匹配 / 首聊主链路
