# 算法 2.6 版本开发规划（可直接下发 Codex）

> 版本定位：在 2.5 已通过并完成“MBTI/性格测试关闭 + 紫微斗数服务端 canonical + 排序接入 + 回填验证 + 回滚与影子对拍闭环”的基础上，2.6 版本进入 **稳定运营基础设施 + 输入链路修复 + 多模块可靠运行 + 后续多场景扩展准备** 阶段。
>
> 2.6 不追求堆更多新术数，也不直接跳到复杂关系智能，而是优先完成：
>
> 1. **服务器用户数据库自动备份（云端 + 本地）**
> 2. **修复用户输入信息、计算玄学指标的入口**
> 3. **把 2.5 的新旧模块切换后的数据链路真正跑稳**
> 4. **为 2.7 的多场景匹配做 schema 与接口准备**

# 3. 主线 A：数据库自动备份（云端 + 本地）

## 3.1 目标

让当前用户数据库从“只有线上主库”提升到“可定时备份、可跨介质保存、可恢复演练、可追踪失败告警”的安全级别。

## 3.2 备份范围

### 必须包含

* `users`
* `user_astro_profiles`
* `dating_matches`
* `messages`
* `app_release_versions`
* 与画像、匹配、消息、版本相关的所有核心业务表

### 建议包含

* 结构（schema）
* 数据（data）
* 迁移版本信息
* 备份 manifest

## 3.3 备份策略

### 本地备份

* 备份位置：服务器本地独立目录，如 `/opt/backups/elitesync/mysql/`
* 频率：

  * 每日增量 / 每日全量（二选一，结合现有规模）
  * 至少每日一份完整可恢复备份
* 保留策略：

  * 最近 7 天全量保留
  * 最近 30 天按日/周保留

### 云端备份

* 推荐：阿里云 OSS / 对象存储
* 频率：

  * 每日自动上传最新完整备份
* 保留策略：

  * 最近 30 天完整保留
  * 月度归档保留 3-6 个月（按成本评估）

## 3.4 备份方式建议

### 优先方案

* `mysqldump` + 压缩 + 校验和
* 生成：

  * SQL dump
  * 压缩包
  * checksum 文件
  * manifest 元信息

### manifest 应至少包含

* 备份时间
* 数据库名
* 文件大小
* 表数量
* 行数摘要（可选）
* SHA256
* 备份来源环境
* 备份脚本版本

## 3.5 自动化执行

### 建议新增脚本

* `scripts/db_backup_mysql_full.sh`
* `scripts/db_backup_mysql_upload_oss.sh`
* `scripts/db_backup_verify.sh`
* `scripts/db_backup_prune.sh`

### 调度方式

* Linux `cron`
* 统一写入日志文件
* 失败触发告警

## 3.6 恢复能力要求

必须能完成：

1. 恢复到单独测试库
2. 验证关键表存在
3. 验证关键字段存在
4. 验证至少一条用户画像数据能正常读取
5. 验证匹配结果接口可基于恢复库跑通基本查询

## 3.7 告警要求

备份失败时至少发送到：

* 管理员邮件 / 企业微信 / 钉钉 / Telegram 中任一渠道

### 触发告警条件

* 备份脚本非 0 退出
* OSS 上传失败
* checksum 校验失败
* 备份文件大小异常（过小）
* 恢复演练失败

## 3.8 交付物

* `docs/db_backup_strategy_2_6.md`
* `docs/db_restore_playbook_2_6.md`
* `docs/archive/legacy_2026-04/reports/db_backup_run_log_2_6.md`
* `docs/archive/legacy_2026-04/reports/db_restore_drill_2_6.md`
* 备份脚本与 cron 配置样例

## 3.9 验收标准

* 云端 + 本地双备份可自动跑
* 至少完成 1 次恢复演练
* 失败会告警
* 备份文件可校验

# 4. 主线 B：修复用户输入信息与玄学计算入口

## 4.1 目标

解决“用户资料改了，但玄学画像没及时更新 / 多入口口径不一致 / 入口触发链不稳定”的问题。

## 4.2 2.6 要修的核心问题

1. 用户资料入口分散
2. 出生信息修改后未统一触发 canonical 重算
3. 基础资料与画像镜像字段存在不同步风险
4. 前端可能存在旧入口、重复入口或弱提示入口
5. 某些字段填写后未触发紫微/八字/生肖/星座/western_lite 全量联动

## 4.3 必须统一的输入入口

### 基础资料类

* 昵称 / 性别 / 城市 / 婚恋目标

### 出生资料类（玄学关键入口）

* 生日
* 出生时间
* 出生地
* 出生经纬度（若有）

### 统一原则

* **所有玄学相关资料统一经一个 canonical input pipeline 进入后端**
* 禁止在前端直接各模块分别提交、分别计算

## 4.4 建议的数据流改造

### 旧模式（问题）

* 基础资料单独保存
* 玄学资料可能在不同页面、不同接口提交
* 保存后是否重算依赖零散逻辑

### 新模式（2.6 目标）

1. 前端统一提交 `profile/basic` 与 `profile/astro` 的最小规范数据
2. 服务端进入 `ProfileInputOrchestrator`
3. 识别变更字段
4. 写入 canonical 主表
5. 触发相关模块重算：

   * 八字
   * 生肖
   * 星座
   * western_lite
   * 紫微
6. 更新镜像字段
7. 生成或刷新 explanation 所需摘要
8. 写入事件日志

## 4.5 建议新增服务

* `ProfileInputOrchestrator`
* `AstroRecomputeDispatcher`
* `AstroFieldChangeDetector`
* `UserProfileMirrorSyncService`
* `AstroProfileFreshnessService`

## 4.6 触发规则

### 必须触发重算的字段

* `birthday`
* `birth_time`
* `birth_place`
* `birth_lat`
* `birth_lng`
* `gender`（若紫微/术数模块依赖）

### 仅触发展示更新、不触发玄学重算的字段

* `name`
* `city`
* `relationship_goal`

## 4.7 幂等要求

* 同样的输入重复提交，不应重复写入多份 canonical 结果
* 连续保存多次，只保留最新有效快照
* 异步队列 / 重试不能导致画像状态错乱

## 4.8 失败处理

若某个模块重算失败：

* 不应阻断基础资料保存
* 应记录失败原因
* 应允许该模块单独重跑
* 前端应提示“部分画像暂未更新完成”而不是静默失败

## 4.9 前端入口修复要求

### 必须检查

* 是否仍有旧的资料入口 / 旧弹窗 / 旧路由
* 是否有“改了出生资料但页面仍显示旧画像”的缓存问题
* 是否有保存后未提示重新计算状态的问题
* 是否有必填项不明确、导致 low confidence 偏高的问题

### 前端建议新增

* 资料完整度提示
* “保存后将更新八字/紫微/星座等画像”的提示
* 计算中状态
* 失败重试入口
* 最近更新时间显示

## 4.10 交付物

* `docs/profile_input_pipeline_2_6.md`
* `docs/astro_recompute_rules_2_6.md`
* `docs/archive/legacy_2026-04/reports/profile_input_fix_report_2_6.md`
* `docs/archive/legacy_2026-04/reports/astro_recompute_e2e_2_6.md`

## 4.11 验收标准

* 修改生日/出生时间/出生地后，相关玄学模块都能正确重算
* 镜像字段同步更新
* 前端页面不再出现“用户输入已改，画像未变”的问题
* 失败可见、可重试、可追踪

* `precision`
* `confidence`
* `confidence_reason`
* `computed_at`
* `degraded`

## 5.4 新鲜度控制

新增概念：`profile_freshness`

* `fresh`
* `stale`
* `pending_recompute`
* `failed_recompute`

### 用途

* 前端展示“画像是否为最新”
* 后台观测哪些模块经常算失败

## 5.5 事件日志

建议新增：`profile_recompute_logs`

建议字段：

* `id`
* `user_id`
* `trigger_source`（profile_save / manual_retry / backfill / migration）
* `changed_fields`
* `modules_targeted`
* `modules_succeeded`
* `modules_failed`
* `error_summary`
* `started_at`
* `finished_at`

## 5.6 幂等与锁

* 同一用户同一时刻只允许一条画像重算任务
* 同一资料快照不重复计算
* 避免并发保存时覆盖新结果

# 10. 你监督 Codex 时重点看什么

## 第一类：别让备份只停留在“脚本存在”

重点看：

* 是否真的跑了
* 是否真的上传到云端
* 是否有 checksum
* 是否有恢复演练
* 是否有失败告警

## 第二类：别让“修复入口”变成只改前端按钮

重点看：

* 后端有没有统一 orchestrator
* 修改出生资料后是不是所有相关模块都重算
* 镜像字段有没有同步更新
* failure 是否可见、可重试

## 第三类：别让重算链路越来越乱

重点看：

* 有没有 freshness
* 有没有 recompute logs
* 有没有并发锁
* 有没有旧结果覆盖新结果的风险

# 1. 设计原则

## 1.1 页面定位
“我的”页面不再只是资料页，而是升级为：
1. **个人基础资料中心**
2. **玄学资料录入中心**
3. **玄学画像查看与可视化入口中心**
4. **资料更新状态中心**

## 1.2 基本原则
- 服务端是唯一真源
- 前端只负责输入、展示、缓存、状态提示
- 可视化优先强调“易读、分层、可解释”，不要一开始追求复杂动效
- 低置信度时必须降级展示，不允许强口气结论
- 图形展示与文字说明必须并存，不能只给盘面不给摘要

## 1.3 本轮不做的事
- 不在客户端本地做 canonical 八字 / 星盘 / 紫微计算
- 不做复杂关系图谱或多对象对比图
- 不做商业化皮肤系统
- 不做专家级排盘参数自定义大而全页面

- `birthday`
- `birth_time`
- `birth_place`
- `birth_lat`
- `birth_lng`

- 八字重算
- 生肖重算
- 星座重算

- 紫微重算

- `正在更新画像…`

- `紫微：更新中`

### Step 5：完成后自动刷新
- 卡片摘要刷新
- 详情页刷新
- 若某模块失败，明确提示

## 4.3 状态枚举建议
- `fresh`
- `pending_recompute`
- `failed_recompute`
- `stale`

## 4.4 前端文案建议
- `保存后将自动更新八字 / 星盘 / 紫微画像`
