# EliteSync 3.4 版本开发计划书

更新时间：2026-04-11  
适用对象：项目负责人 / Codex / Gemini-MCP / Claude-mcp / 后续顾问归档  
版本状态：草案 v1  
当前前置版本：`3.1` 已通过、`3.2` 已终验通过、`3.2a` 已完成补丁收口、`3.3` 已终验通过

---

## 0. 本文档定位

本文档用于在 `3.3` 完成后，为 EliteSync 制定下一阶段 `3.4` 的正式开发计划。

`3.4` 的任务不是回头推翻 `3.2 / 3.2a / 3.3` 已通过能力，也不是直接推进高风险星盘 canonical 算法升级，而是：

1. 继续把产品做得更像正式 App
2. 把资料编辑、星盘显示、设置中心、版本中心这一批核心页面再收一轮体验与口径
3. 为真实 Beta 运营与回归做准备
4. 继续严格隔离“显示层优化”和“算法真值升级”两条路线

---

## 1. 整体计划回顾

### 1.1 当前项目已完成的阶段性成果

#### 3.1 已完成

`3.1` 已把项目推进到“最小社交闭环 + 测试运营底座”阶段，核心包括：

- 账号分层底座
- 状态发布 / 广场最小闭环
- 管理员移动端控制面
- 匹配调试底座
- 普通用户可见性证据
- 临时证据与临时账号清理

#### 3.2 已完成

`3.2` 已把项目推进到“转化链路 + 治理视图 + 测试规模化”阶段，核心包括：

- `normal / test / synthetic` 测试账号体系正式落地
- 1000 synthetic 默认池与 10000 扩展能力补证
- Admin 客观治理视图增强
- 匹配结果 -> 首聊 -> 恢复建议转化链路补强
- 广场轻分层
- 终验证据包与口径澄清

#### 3.2a 已完成

`3.2a` 是 `3.2` 的显示层补丁版，核心包括：

- 星盘主视觉从服务端 SVG 依赖切到 APP 端本地绘制
- 盘面设置页上线：恢复默认、预设档位、元素开关
- 关于页 / 更新历史 / 版本检查与发布信息同步
- Android 宿主 API / WS 基线注入补齐
- 高风险星盘算法升级继续冻结

#### 3.3 已完成

`3.3` 已把项目推进到“资料真值链路 + 设置中心 + 版本中心正式化”阶段，核心包括：

- 资料保存 -> 服务端重算 -> 页面重绘链路收口
- `profile/basic` 保存响应返回 `astro_profile` 快照与 `recomputed_at`
- 设置中心结构化分组
- 盘面设置与星盘页边界说明收口
- 版本中心与服务可观测性口径收口
- 文档、测试、截图证据补齐并完成终验归档

### 1.2 当前项目最重要的判断

到 `3.3` 为止，项目已不再缺“有没有功能”。

当前更重要的是：

1. 继续做正式产品体验打磨，而不是盲目堆新功能
2. 把设置、资料、星盘、版本页的口径和体验继续做顺
3. 让 Beta 运营、版本检查、弱网回归、回滚检查更稳
4. 坚持将星盘显示优化与 canonical 算法升级隔离

## 3. 为什么 3.4 应这样定义

### 3.1 当前不适合直接做什么

当前不适合把主线直接改成：

- 星盘 canonical 算法大改
- 宫位制 / 黄道制 / 相位容许度 / 虚点的大规模专业化开放
- 高风险引擎替换
- 大规模推荐算法重构
- 重内容平台化扩张
- 重社群 / 重直播 / 重打赏逻辑

原因：

1. 这些内容会直接碰真值层，风险高
2. 相关算法资料目前仍应沿用单独算法交接路径处理
3. 当前最值钱的不是“再加功能”，而是把已有产品做得更顺、更稳、更正式

### 3.2 当前最值得做什么

当前最值得做的是：

1. 页面口径统一收尾
2. 资料编辑链路体验再收一轮
3. 星盘阅读性精修
4. 设置中心继续产品化
5. Beta 运营准备与最小回归材料固化

---

## 4. 3.4 核心目标

### 4.1 P0（必须完成）

1. **页面口径统一收尾**
   - “关于与更新”与“版本中心”命名统一
   - 当前版本、环境行版本、更新历史标题、版本检查口径统一
   - 清理旧文案、中英文混杂与测试期遗留口径

2. **资料编辑链路体验收口**
   - 保存资料后的反馈更明确
   - 出生地搜索、已选出生地、城市字段关系更清楚
   - 保存失败 / 重算失败 / 慢网反馈更可见
   - 让用户更容易理解“改资料为什么会影响画像 / 星盘”

3. **星盘阅读性精修**
   - 优化标签拥挤、相位线可读性、盘心留白与层次
   - 优化预设档位文案
   - 补轻量阅读帮助，不碰 canonical 真值

4. **设置中心正式产品化再收一轮**
   - 优化信息层级、留白、分组关系
   - 统一开关、按钮、说明卡视觉风格
   - 减少“正式功能 / 调试项”歧义

5. **Beta 运营准备收口**
   - 重新跑 health / version / check 核对
   - 重新跑弱网首进验证
   - 重新跑最小回滚检查
   - 固化 3.4 的最小 Beta 回归清单

6. **归档材料标准化**
   - handoff / acceptance / risk review / regression checklist 一次补齐
   - 截图证据一次归档到位

### 4.2 P1（尽量完成，做不完可后移）

1. 隐私与安全细节完善
   - 隐私设置“可见性边界”说明再细一点
   - 修改密码 / 退出登录 / 资料可见范围的反馈更清楚
   - 为未来用户协议 / 隐私政策正式上线预留接口

2. 内容缓存与性能偏好优化
   - 性能模式文案更具体
   - 内容缓存清理、重置偏好的风险提示更清晰
   - 清理后的反馈更可见

3. 匹配后转化链路轻优化
   - 首聊建议与恢复建议继续微调
   - 解释 -> 行动 -> 会话路径减少跳跃感

4. Admin 小修
   - 统计口径继续保持自解释
   - 卡片数字、筛选标签、用户明细之间关系更直观

### 4.3 明确暂不纳入 3.4

以下内容不进入 `3.4` 主线：

1. 星盘 canonical 算法升级
2. 宫位制 / 黄道制 / 相位容许度的专业级开放
3. 更复杂的虚点体系
4. 大规模推荐算法重构
5. 内容平台化扩张
6. 重社群 / 重直播 / 重打赏逻辑

## 6. 长期约束（3.4 不可违反）

### 6.1 真值约束

1. 服务端 canonical 真值仍是唯一真源
2. 前端本地绘制结果不得回写真值层
3. 本地盘面设置只影响显示，不改后端画像结果
4. 页面文案必须明确区分“真值”和“显示偏好”

### 6.2 玄学约束

1. 八字 / 紫微 / 星盘 / 星座的解释层仍与计算层分离
2. 盘面设置页不得扩展成算法调参面板
3. 任何高风险算法入口必须先进入单独算法交接文档，不得直接混入产品页

### 6.3 发布与运维约束

1. 版本号、更新历史、版本检查、下载信息必须同步
2. 所有高风险修改都要带 smoke / regression / rollback 思维
3. 不得破坏 3.2 / 3.2a / 3.3 已通过能力

### 6.4 数据与治理约束

1. synthetic / test / normal 分层保持不变
2. synthetic 默认 `exclude_from_metrics = true`
3. Admin 统计口径必须继续自解释
