# EliteSync 4.4 版本开发计划书

文件名：`elite_sync_4_4_版本开发计划书_2026_04_19.md`  
更新时间：2026-04-19  
适用范围：`4.3` 已正式归档通过后的下一版本承接  
当前承接基线：`4.0` 媒体基础设施、`4.1` 非官方四维人格问卷、`4.2` 图片消息主链、`4.3` 动态流基础版均视为稳定基线。  
本版定位：`4.4 视频消息版`

---

## 0. 本版一句话结论

`4.4` 不做视频社区，也不做 RTC 通话，而是把 EliteSync 的聊天主链从“文本 + 图片”推进到“文本 + 图片 + 单视频消息”的可用状态。

本版只做：
- 聊天中的单视频消息最小闭环
- 视频选择、上传、处理中、失败重试、发送确认
- 视频封面 / 预览图、气泡展示、点击播放
- 基于 `4.0` 媒体底座的异步处理与状态机收口
- 基于 `4.2` 消息附件链的安全扩展
- 与现有聊天页保持 additive 承接，不重写会话主结构

本版明确不做：
- 视频动态
- 多视频消息
- 视频剪辑 / 滤镜 / 美颜 / 配乐
- 工业级分片上传
- 自动播放瀑布流
- 语音 / 视频通话
- 大型媒体平台化改造

换言之，`4.4` 的目标不是把 EliteSync 做成短视频产品，而是把聊天主链补到“可以发送并消费短视频消息”的基础媒体层。

## 1. 版本定位

## 1.1 版本名称

`4.4 视频消息版`

## 1.2 版本使命

在不破坏 `3.9 / 4.0 / 4.1 / 4.2 / 4.3` 已归档主链的前提下，补齐以下能力：

1. 用户能在聊天中选择并发送单个视频附件  
2. 视频附件能进入既有媒体状态机，并异步处理  
3. 会话中能清楚看到“上传中 / 处理中 / 可播放 / 失败”状态  
4. 用户能点击视频气泡进入播放页或预览页  
5. 失败视频能重试、替换或清除，不误导用户为已发送成功  
6. 视频消息可被会话列表、摘要与回读链正确承接  

## 1.3 为什么现在做 4.4，而不是继续回头补 4.3 或直接进入通话能力

### 原因 A：4.3 已完成表达层，应冻结主链

`4.3` 的职责是补齐动态表达、轻互动与治理，不是继续扩大为更重的视频平台。现在若把视频能力直接叠在动态流，会让 `4.3` 与 `4.4` 失去边界。

### 原因 B：视频消息是聊天主链更自然的下一步

相比视频动态或 RTC，视频消息更直接承接：
- `4.0` 的媒体底座
- `4.2` 的消息附件主链
- `4.3` 之后更完整的用户表达场景

先把聊天视频消息做稳，再考虑更高复杂度能力，版本节奏更健康。

### 原因 C：通话能力仍属于单列高风险版本

RTC / 通话会涉及：
- 信令
- 在线状态
- 权限与设备状态
- 弱网重连
- 更复杂的会话状态机

这些都不应混进 `4.4`。`4.4` 应专注于“异步视频消息”，而非“实时音视频通话”。

## 2. 承接基线

本版必须承接并保护以下基线：

### 2.1 已冻结的真值与主链

- `POST /api/v1/profile/basic` 仍是资料真值入口
- `user_astro_profiles` 仍是画像 / 星盘核心真值表
- `chat_messages` 仍是消息主链真值表
- `app_release_versions` 仍是发布真值表
- `questionnaire_attempts / questionnaire_answers / questionnaire_questions` 仍是问卷真值表

### 2.2 已冻结的能力底座

- `4.0` 媒体 / 存储 / 队列 / 缓存 / worker 最小闭环
- `4.1` 非官方四维人格问卷模块
- `4.2` 图片消息上传、附件绑定、预览、失败重试、摘要回读
- `4.3` 动态流基础版与相关治理挂点

### 2.3 本版最关键的承接口径

`4.4` 是 **聊天视频消息版**，不是“视频动态版”、不是“短视频广场版”、也不是“通话版”。  
所有实现都应遵循：

- additive feature，不做 destructive rewrite
- 先扩展聊天媒体，不重写动态流
- 先做短视频消息最小闭环，不做大型视频平台
- 先复用 `4.0 / 4.2` 媒体能力，不重造第二套媒体基础设施

## 3. 本版做什么 / 不做什么

## 3.1 本版必须完成的核心目标

### 目标 1：聊天视频选择与发送闭环

至少支持：
- 从聊天页选择单个视频
- 发送前展示待发送状态
- 可取消 / 清除当前待发送视频
- 发送后在会话中可见状态变化

### 目标 2：视频上传与异步处理闭环

至少支持：
- 视频上传走既有 `/api/v1/media`
- 视频进入 `uploaded / processing / ready / failed` 状态链
- 不允许把转码或预览图生成阻塞在同步请求里
- 失败有明确错误态，不假成功

### 目标 3：视频消息展示与播放闭环

至少支持：
- 聊天气泡展示视频封面或预览图
- 展示播放入口、时长、基础状态
- 点击进入预览/播放页
- 处理未完成时展示“处理中”而非假播放态

### 目标 4：失败重试与治理挂点闭环

至少支持：
- 上传失败可重试
- 处理失败可见
- 用户可重新选择视频或清除
- 保留最小治理 / 日志 / telemetry 挂点

### 目标 5：会话链与摘要链正确承接

至少支持：
- `POST /api/v1/messages` 继续使用 `attachment_ids`
- 会话摘要正确显示为视频消息语义
- 会话列表、消息气泡、详情页对状态口径一致

## 3.2 本版明确不做的内容

以下内容禁止偷渡进入 `4.4`：

### 内容能力不做
- 视频动态
- 视频广场
- 自动播放内容流
- 多视频轮播
- 视频专题页 / 推荐页

### 编辑能力不做
- 视频裁剪
- 滤镜 / 美颜
- 贴纸 / 字幕
- 音乐叠加
- 复杂封面编辑器

### 实时能力不做
- 语音通话
- 视频通话
- 呼叫状态机
- 在线状态系统
- RTC 信令

### 媒体工程能力不做
- 工业级分片上传
- 断点续传完整版
- 大视频创作平台
- 大规模离线媒体管线

### 架构能力不做
- 微服务重构
- 对 `4.0` 媒体底座推倒重来
- 对 `4.2` 图片消息主链推倒重来
- 对 `4.3` 动态流进行视频化扩写

## 4. 硬性保护面

本版开发前必须先冻结以下保护面，不得误伤：

## 4.1 真值链保护面

- `POST /api/v1/profile/basic`
- `GET /api/v1/profile/astro/summary`
- `GET /api/v1/profile/astro/chart`
- `user_astro_profiles`
- `questionnaire_*` 真值链

### 解释

视频消息是聊天表达层的一部分，不是资料、画像、匹配的真值层。  
任何视频消息、观看状态、失败状态，都不得反向写成匹配或画像的 canonical truth。

## 4.2 消息与媒体保护面

- `GET /api/v1/messages`
- `POST /api/v1/messages`
- `POST /api/v1/media`
- `chat_messages`
- `media_assets`
- `media_processing_jobs`
- `message_attachments`

### 解释

`4.4` 必须在既有消息附件主链上扩展视频支持，而不是另外发明一套“视频消息平行链”。  
尤其不能因为视频而把图片消息链打坏，也不能让 `message_attachments` 失去清晰边界。

## 4.3 发布与版本保护面

- `GET /api/v1/app/health`
- `GET /api/v1/app/version/check`
- `app_release_versions`
- 宿主 APK 版本口径
- about / version center / release script 绑定链

### 解释

`4.4` 是媒体功能版本，不是发布链重构版本。  
新增视频能力不能破坏版本检查、about、changelog 与下载包的一致性。

## 4.4 动态流保护面

- `status_posts`
- `status_post_likes`
- `HomePage` 轻量动态预览
- `StatusAuthorPage`
- `4.3` 治理挂点

### 解释

`4.4` 默认不扩写动态视频。  
本版所有视频能力都应收口在聊天主链中，避免误把动态广场改造成视频平台。

## 4.5 UI / IA 保护面

必须继续遵守：
- 页面要像产品页，不要像内部调试面板
- 聊天页不能因为视频变成媒体管理器
- 失败态可见，但不能比成功态更重
- 预览/播放入口清楚，但不堆砌过多工程状态文案

## 5. 数据与接口边界建议

## 5.1 数据模型建议

优先复用现有核心表：
- `media_assets`
- `media_processing_jobs`
- `message_attachments`
- `chat_messages`
- `conversations`
- `conversation_members`
- `notifications`
- `app_events`

如仓库中需要最小补充，优先采用“扩字段 / 扩 metadata / 扩处理任务”而非新建平行主表。建议最小扩展点：

- `media_assets.media_type`：支持 `video`
- `media_assets.mime_type`、`size_bytes`：继续作为基础字段
- `media_assets.status`：继续走统一状态机
- `media_assets` 或其 metadata 中增加最小视频元信息：
  - `duration_ms`
  - `width`
  - `height`
  - `thumbnail_key` 或等价预览图引用
  - `playback_variant` / `transcode_profile`（若已有 metadata 容器则优先放 metadata）

### 边界解释

本版不建议为了视频单独新建“大型视频资产系统”。  
应优先在既有 `media_assets` 的统一媒体抽象上扩展，使图片与视频共用状态机、对象存储与处理任务，但在 UI 与播放逻辑层保持清晰区分。

## 5.2 接口域建议

### 优先策略

- 媒体上传：继续使用 `POST /api/v1/media`
- 消息发送：继续使用 `POST /api/v1/messages` + `attachment_ids`
- 消息读取：继续使用 `GET /api/v1/messages`
- 视频详情 / 播放若需额外接口，优先最小补充，不重建消息域

### 建议最小接口集合

- `POST /api/v1/media`：支持视频上传与视频 metadata 返回
- `GET /api/v1/media/{assetId}`：返回状态、预览图、播放地址或等价播放信息
- `POST /api/v1/messages`：支持视频 `attachment_ids`
- `GET /api/v1/messages`：返回视频消息状态、封面 / 预览 / 播放字段

若需要最小辅助接口，可考虑：
- `POST /api/v1/media/{assetId}/retry` 或在既有重试语义内完成
- `GET /api/v1/media/{assetId}/playback` 仅在签名 URL / 多清晰度确实需要时新增

### 边界解释

`4.4` 不要求一次性做完视频平台的所有 API。  
核心原则是：
- 上传路径清晰
- 发送路径沿用既有消息主链
- 播放信息可获取
- 状态链清楚
- 不新增一堆脱离主链的试验性接口

## 6. 工作包拆分

## 6.1 4.4A：边界冻结与视频媒体规格收口

### 目标

在动代码前，先把 `4.4` 的版本边界、视频规格、状态机口径、预览与播放策略写清楚，避免版本越界。

### 必做任务

1. 明确本版仅支持“聊天中的单视频消息”  
2. 明确视频时长、体积、分辨率上限  
3. 明确视频是否强制异步处理、在哪些状态下可发送/可播放  
4. 明确封面图 / 预览图生成策略  
5. 明确失败类型：上传失败、处理中失败、播放不可用  
6. 明确本版不做视频动态、不做通话、不做编辑器  

### 必要验收条件

- 存在一份清晰的边界说明
- 列出新增字段 / metadata / 处理任务 / 接口清单
- 明确哪些旧表和旧接口不可碰
- 明确视频仍复用统一媒体管线，而非平行重造

### 解释

没有边界说明就直接实现视频，最容易出现的结果是：
- 把消息版做成创作平台
- 同步请求里塞转码逻辑
- 聊天页被做成媒体管理页
- 版本越界到 RTC 或动态视频

因此 `4.4A` 是本版硬前置，不可省略。

---

## 6.2 4.4B：后端视频上传、处理与消息绑定闭环

### 目标

建立视频上传、异步处理、状态更新、消息绑定、读取回读的最小后端闭环。

### 必做任务

1. 完成视频上传进入媒体主链  
   - `POST /api/v1/media` 支持 `video/*`
   - 记录最小视频元信息
   - 进入统一状态机

2. 完成视频处理任务  
   - 预览图 / 封面生成
   - 必要的转码或播放变体生成
   - 处理失败可回写状态与错误码
   - 不阻塞同步上传请求

3. 完成消息绑定  
   - `POST /api/v1/messages` 继续使用 `attachment_ids`
   - 视频附件绑定到 `chat_messages`
   - 会话摘要与消息列表能正确识别视频消息

4. 完成读取回读  
   - `GET /api/v1/messages` 返回视频所需展示字段
   - `GET /api/v1/media/{assetId}` 可查询处理结果
   - ready 前后状态返回一致

5. 完成最小 telemetry / 事件记录  
   - 选择视频
   - 开始上传
   - 上传成功
   - 上传失败
   - 开始处理
   - 处理成功
   - 处理失败
   - 视频消息发送
   - 视频播放打开

### 必要验收条件

- 上传请求不会因视频处理被长时间阻塞
- 视频能在刷新后正确回读状态
- 处理中与 ready 状态区分明确
- 失败状态可记录，不假成功
- 视频消息不会破坏已有文本 / 图片消息读取

### 解释

本工作包的目标不是做完整媒体平台，而是把“可上传、可处理、可绑定、可回读”的后端主链闭合。  
如果后端只有视频文件上传，没有状态机、处理任务和回读字段，就不能算视频消息成立。

---

## 6.3 4.4C：Flutter 聊天页视频发送器、气泡与播放页

### 目标

让用户能够在聊天中以产品化方式选择视频、发送视频、看到视频状态、打开视频播放。

### 必做任务

1. 聊天输入区视频入口  
   - 保持与文本 / 图片入口风格协调
   - 不把输入区做成复杂媒体工具栏

2. 待发送视频卡片  
   - 展示时长、缩略封面、待发送状态
   - 支持清除 / 重新选择
   - 发送按钮状态清楚

3. 视频消息气泡  
   - ready 时显示封面 + 播放入口 + 时长
   - processing 时显示处理中状态
   - failed 时显示失败与重试入口
   - 不误导用户点击不存在的播放能力

4. 视频播放页 / 预览页  
   - 点击视频气泡进入播放页
   - 不自动全局自动播放
   - 支持基础播放 / 暂停 / 返回
   - 弱网或不可播放时有明确提示

5. 会话摘要与列表语义  
   - 会话列表摘要正确显示视频消息语义
   - 图片、视频、文本三类消息不混淆

### 必要验收条件

- 视频入口清楚，不需要工程背景才能理解
- 发送中 / 处理中 / ready / failed 状态可辨识
- 视频气泡可读，不像调试组件
- 播放页足以完成基本消费，不需要外跳到系统杂乱页
- 从待发送到会话内气泡的状态过渡连贯

### 解释

对 `4.4` 而言，前端关键不是“播放器有多炫”，而是状态表达是否清楚、聊天主链是否顺畅。  
如果播放器存在但用户分不清何时可发、何时可播、何时失败，则版本不能算成立。

---

## 6.4 4.4D：治理、缓存、性能与回归保护

### 目标

在版本收尾前，把视频最容易带来的性能、治理与回归风险提前收住。

### 必做任务

1. 视频治理挂点检查  
   - 沿用既有 moderation / report 体系的最小挂点
   - blocked / deleted / failed 状态在 UI 中可解释

2. 缓存与性能最小收口  
   - 聊天列表优先展示封面或预览图
   - 不在消息列表中自动拉取大视频
   - 原视频播放按需加载
   - 避免因视频导致聊天列表明显卡顿

3. 回归保护  
   - `profile/basic` 真值链回归
   - `astro/summary`、`astro/chart` 回归
   - `messages` 文本 / 图片 / 视频消息回归
   - `media` 图片 / 视频上传回归
   - `status_posts` 与 `4.3` 动态流回归
   - `questionnaire/history`、`questionnaire/answers` 回归
   - `app/version/check` 回归

4. 日志与错误可见性  
   - 上传失败、处理失败、播放失败均有日志
   - 不允许失败只在控制台里出现

### 必要验收条件

- 弱网 / 失败场景可复现且可解释
- 视频不会明显拖垮聊天滚动性能
- 回归清单完整，旧主链未被误伤
- 视频能力不会反向伤及图片消息、动态流与版本链

### 解释

视频是 4.x 当前最重的媒体能力之一。

## 8. 4.4 的验收结论建议口径

## 8.1 何时可以给 `pass`

仅当以下同时成立：
- 视频上传、处理、绑定、展示、播放、失败重试全链路成立
- UI 状态表达清楚
- 回归完整
- 截图证据完整
- handoff / closeout 完整
- 无需再补关键收尾材料

## 8.2 何时给 `pass with observations`

用于：
- 主链已经完整成立
- 可以正式归档
- 仍有少量非阻断观察项，例如：
  - 某些视觉细节仍偏系统播放器风格
  - 某些 processing 文案仍可继续降噪
  - 某些封面展示密度仍可优化

### 建议

`4.4` 最现实的目标是：**争取做到 `pass with observations` 后冻结主链。**  
不要为了追求“更炫的播放器”而在 `4.4` 上反复开补丁。

## 8.3 何时只能给 `conditional pass`

若出现以下情况之一：
- 视频主链虽跑通，但截图证据包明显不足
- 没有 closeout / handoff
- 治理与回归材料薄弱
- 能上传或能播，但状态机、失败态、回读说明不清楚

则应判为 `conditional pass`，并仅允许小回合补收尾，不建议重开大开发。

## 9. 风险清单

## 9.1 最大风险：版本越界到视频平台或通话版

表现：
- 聊天视频做着做着变成视频动态
- 把通话入口、在线状态、RTC 一起做进来
- 把 `4.4` 做成媒体平台化大版本

控制原则：
- 只做 single video message / lightweight playback
- 不把后续版本内容偷渡进来

## 9.2 第二风险：同步链路塞入重处理逻辑

表现：
- 上传接口长时间阻塞
- 同步请求里做封面生成或转码
- 用户误以为发送已完成

控制原则：
- 上传与处理分离
- 统一状态机驱动 UI

## 9.3 第三风险：误伤图片消息与聊天主链

表现：
- 为视频改坏 `attachment_ids` 语义
- 图片消息预览、失败重试或会话摘要被连带打坏

控制原则：
- 视频能力扩展自 `4.2`，不是替代 `4.2`
- 文本 / 图片 / 视频三类消息并存验证

## 9.4 第四风险：UI 工程感过重

表现：
- 聊天页像媒体调试台
- 视频气泡堆满工程状态文案
- 处理状态影响阅读节奏

控制原则：
- 由 Gemini 重点审查页面可读性与用户感知
- 所有主文案避免过强工程语义

## 9.5 第五风险：性能与弱网问题被低估

表现：
- 聊天页滚动卡顿
- 大视频误自动加载
- ready 前后状态错乱

控制原则：
- 预览图优先，原视频按需播放
- 弱网与失败态必须纳入正式回归
