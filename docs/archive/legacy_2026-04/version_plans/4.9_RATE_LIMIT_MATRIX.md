# 4.9 关键接口限流矩阵

更新时间：2026-04-28

## 目的

在 4.8 已完成 Alpha smoke 的基础上，为 5.0 Alpha 前的真实路径复验建立最小限流矩阵，降低重复点击、脚本误触发、弱网重试、异常客户端和后台任务抖动对核心链路的影响。

## 现有实现概览

后端已存在的 limiter：

- `auth`
- `messages`
- `conversations`
- `media`
- `relationships`
- `notifications`

当前路由中已挂载的 throttle 位置主要包括：

- 登录 / 注册：`throttle:auth`
- 会话列表 / 创建 / 详情：`throttle:conversations`
- 媒体列表 / 上传 / 展示 / 处理：`throttle:media`
- 关系链：`throttle:relationships`
- 通知列表 / 未读 / 全部已读 / 单条已读：`throttle:notifications`
- 消息发送 / 列表：`throttle:messages`

## 4.9 当前优先建议

1. 先补最容易引发滥用或真值污染的写接口：
   - `profile/basic`
   - `profile/astro`
   - `questionnaire/answers`
   - `match/{targetUserId}/explanation`
   - `status/posts`
   - `rtc/calls`
   - `rtc/calls/{callId}/heartbeat`
   - `notifications/read-all`
2. 读接口可先以已有 limiter 或轻限流复核为主；
3. heartbeat 不得因限流误杀正常通话；
4. version check / health 保持较高频，但要避免无限刷屏；
5. 若后续发现接口已被 Nginx / API Gateway / client 端节流覆盖，可在本矩阵里降级为 observation。

## 5. 现有可复用点

- `AppServiceProvider.php` 已有基础 limiter 定义；
- `routes/api.php` 已在部分路由挂载 throttle；
- 4.9 可优先补最小缺口，不要求大重构。

## 6. 结论

当前 4.9 的限流工作应以“先矩阵、后最小补强”为原则，避免因过度限流影响 smoke 和 Alpha 复验。
