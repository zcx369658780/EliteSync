# 4.6P 音频播放链定向修正版

更新时间：2026-04-24

## 定位

4.6P 不是新的 RTC 功能版，也不是继续重查连接层的大排障版，而是一个只盯“听到声音”这一 blocker 的定向修正版。

本版只修：
- 音频播放链
- 音频路由
- `TrackPublished / TrackSubscribed / remoteAudioTrack / AudioPlayback` 的真实证据补齐
- 真机与模拟器双向最小复验
- 仅限与音频诊断直接相关的少量调试暴露项

本版不做：
- 多人通话
- 视频增强
- 在线状态平台
- 外部推送平台
- 新 RTC 库替换
- 通话页大改版
- 连接层重做
- 通知中心重做

## 当前已知问题

- 状态机主链已跑通
- LiveKit 连接层已修正到可连接 / 可 heartbeat 阶段
- 真机侧没有稳定音频帧，模拟器侧虽有帧但无可听声音或无波形
- 当前 blocker 更像 publish / subscribe / remoteAudioTrack / playback / routing 中的一层或多层

## 本版工作包

### 4.6P-A 边界冻结与音频问题重写
- 明确本版只修音频播放链与音频路由
- 明确 4.6 / 4.6A 主链不重做
- 明确本版不换 RTC 大库、不改 UI 大结构
- 明确当前问题优先级顺序：publish / subscribe / remoteAudioTrack / playback / routing

### 4.6P-B publish / subscribe 证据补齐
- 在真机与模拟器两端补齐 TrackPublished / TrackSubscribed / participant / publication / sid / source / track runtime type 日志
- 明确两端 local / remote publication 与 publication.track 是否非空
- 明确到底是哪一端先掉链

### 4.6P-C `remoteAudioTrack` 绑定与播放链修复
- 明确 `remoteAudioTrack` 的赋值来源与页面 rebuild 链
- 明确频谱组件是否拿到真实非空 track
- 明确当前播放链是否需要显式 attach / play / sink

### 4.6P-D Android 音频路由与设备差异收口
- 检查 speakerphone / earpiece / audio focus / audio mode / mute / gain / sink
- 明确真机与模拟器的差异
- 若有必要，给出最小设备侧修复

### 4.6P-E 回归、证据、收尾材料更新
- 更新音频排障报告 / 修复说明 / 复验记录
- 更新证据索引与验收口径
- 更新 handoff / closeout

## 验收建议

- 若能证明真机 / 模拟器至少一条真实可听声音链路成立，则可考虑将 4.6 结论推进到 `pass with observations`
- 若仍不能听到声音，但能把问题明确收窄到 publish / subscribe / playback / routing 的单一子层，则可进入下一轮最小修复
