# EliteSync 4.4S 版本开发计划书

文件名：`elite_sync_4_4S_版本开发计划书_2026_04_19.md`  
更新时间：2026-04-19  
适用范围：`4.4` 当前停留在 `conditional pass` 后的收尾修正版  
当前承接基线：`4.0` 媒体基础设施、`4.2` 图片消息主链、`4.3` 动态流基础版已归档；`4.4` 视频消息版代码主链已基本成立，但媒体读取 / 播放链仍存在阻断问题。  
本版定位：`4.4S 媒体链稳定性修正版`

---

## 0. 本版一句话结论

`4.4S` 不是新功能版，而是一个 **后端媒体链收口版 / 稳定性修正版**。  
它的目标不是继续扩展视频能力，而是把当前已经暴露出来的 **图片加载失败、视频播放 Source error、媒体 URL / 读取链 / 播放链不稳定** 的问题修到可正式归档的程度。

本版只做：
- 图片 / 视频媒体读取链排障与修复
- 媒体对象存储访问链、预览链、播放链、响应头与权限链收口
- 图片消息与视频消息的成功读取、成功播放、失败态解释与日志补齐
- 让 `4.4` 从 `conditional pass` 升到 `pass with observations` 的最小阻断修复

本版明确不做：
- 新的社交功能
- 视频动态
- RTC / 通话
- 视频编辑器
- 大规模后端重构
- 推送 / 通知增强
- 新推荐逻辑

换言之，`4.4S` 是一个 **插入式修正版**，目的是在不打乱版本节奏的前提下，把媒体后端链路修实。

---

## 1. 为什么需要 4.4S

## 1.1 当前问题已经从“取证不足”转为“媒体链阻断”

在 `4.4` 的补证据回合中，已经能够证明：
- 视频可以被选中并返回聊天页
- 会话摘要已经能显示“视频消息”语义
- 视频消息气泡与播放页已可到达

但新的截图同时暴露了真正的阻断问题：
- 图片仍出现“图片加载失败”
- 视频播放页出现 `ExoPlaybackException: Source error`

这说明当前阻断点不再只是“成功态截图不够”，而是：
**媒体读取链、对象存储访问链、播放器可读链本身仍不稳定。**

## 1.2 为什么不能把这个问题推迟到 4.5 或 4.6

因为：
- `4.5` 原本更偏通知 / 转化增强，不适合背负媒体底层修复主任务
- `4.6` 是更高复杂度的 RTC / 通话基础设施版，如果当前连图片 / 视频消息读取都不稳，往后只会放大风险
- 图片与视频读取链属于当前 `4.0 / 4.2 / 4.4` 的基础质量问题，不应被拖成未来版本的技术债

## 1.3 为什么不直接重开 4.4 大版本

因为 `4.4` 的版本边界已经定义得很清楚：
- 单视频消息最小闭环
- 不做平台化
- 不做通话
- 不做大媒体体系

当前更合理的做法是：
**冻结 4.4 功能边界，在其后插入一个稳定性修正版 4.4S。**

## 2. 版本定位

## 2.1 版本名称

`4.4S 媒体链稳定性修正版`

## 2.2 版本使命

在不破坏 `3.9 / 4.0 / 4.1 / 4.2 / 4.3 / 4.4` 已有边界的前提下，补齐以下能力：

1. 图片消息能稳定加载  
2. 视频消息能稳定读取播放源或最小可播放版本  
3. 对象存储地址、预览图地址、播放地址、签名/权限/响应头链路正确  
4. 失败态能真实反映“上传失败 / 处理失败 / 读取失败 / 播放失败”  
5. 日志与排障证据足以支撑后续版本继续承接  

## 2.3 本版不是做什么

本版不是：
- 4.5 通知版的替代
- 4.6 通话版的前置开发
- 微服务重构版
- 新功能扩展版
- UI 大改版

## 3. 本版做什么 / 不做什么

## 3.1 本版必须完成的核心目标

### 目标 1：图片读取链收口

至少支持：
- 已发送图片消息能正常显示缩略 / 预览
- 点击图片能正常查看或进入预览页
- 不再出现“图片加载失败”作为常态成功面
- 图片失败态能区分是上传问题、资源缺失问题还是读取问题

### 目标 2：视频播放源收口

至少支持：
- 视频消息点击后，播放器拿到有效可读源
- 视频封面 / 预览图 / 原视频地址的关系明确
- `Source error` 不再作为当前主路径常态
- 若处理未完成，则明确展示“处理中 / 暂不可播放”，而不是直接报播放器底层错误

### 目标 3：媒体地址与访问权限链收口

至少支持：
- `media_assets.public_url` 或等价访问字段返回真实可访问地址
- 图片 / 视频 / 缩略图的访问路径口径统一
- 本地、测试环境、模拟器环境的媒体访问路径一致可解释
- 不再出现 localhost、不可达内网地址、错误签名或错误 path 拼接

### 目标 4：媒体处理结果与状态机收口

至少支持：
- `pending / uploading / uploaded / processing / ready / failed / blocked / deleted` 状态真实对应
- UI 不在错误状态下伪装成功
- ready 前后读取字段一致
- 处理失败时能返回清楚错误码或最小可解释原因

### 目标 5：日志、回归与证据收口

至少支持：
- 后端日志能看出媒体资源落盘、路径生成、处理结果、读取结果
- Flutter 侧能区分上传失败、读取失败、播放失败
- 截图证据能证明“图片成功显示 + 视频成功播放”这两个关键事实

## 3.2 本版明确不做的内容

以下内容禁止偷渡进入 `4.4S`：

### 功能面不做
- 新的动态能力
- 新的关系链能力
- 新的推荐流
- 视频动态 / 视频广场
- 通知中心 / 推送扩写

### 媒体面不做
- 多视频消息
- 视频编辑器
- 美颜 / 滤镜 / 裁剪
- 工业级断点续传完整版
- 大型转码平台升级

### 架构面不做
- 微服务拆分
- 全量 OSS / CDN 架构改造
- 大规模权限系统重构
- 新建媒体后台平台

### UI 面不做
- 聊天页整体翻修
- 消息页大改版
- 首页 / 动态流视觉体系重构

## 4. 硬性保护面

## 4.1 真值链保护面

- `POST /api/v1/profile/basic`
- `GET /api/v1/profile/astro/summary`
- `GET /api/v1/profile/astro/chart`
- `user_astro_profiles`
- `questionnaire_*` 真值链

### 解释

本版修的是媒体读取链，不是画像、资料、匹配真值链。  
任何媒体修复都不得绕道修改 canonical truth 逻辑。

## 4.2 消息与媒体保护面

- `GET /api/v1/messages`
- `POST /api/v1/messages`
- `POST /api/v1/media`
- `chat_messages`
- `media_assets`
- `media_processing_jobs`
- `message_attachments`

### 解释

本版必须在既有主链上修，不允许另起“临时兼容链”。  
也就是说：
- 不允许为了修视频播放而新造平行消息表
- 不允许为了修图片加载而绕开 `media_assets`
- 不允许靠前端硬编码资源路径掩盖后端问题

## 4.3 已归档版本保护面

- `4.0` media / storage / attachment pipeline
- `4.1` 非官方四维人格问卷模块
- `4.2` 图片消息主链
- `4.3` 动态流基础版
- `4.4` 视频消息功能边界

### 解释

`4.4S` 只能修媒体稳定性，不能借机扩大为：
- 4.4 第二开发版
- 4.5 前置开发版
- 动态视频版
- 通话前置版

## 4.4 发布与版本保护面

- `GET /api/v1/app/health`
- `GET /api/v1/app/version/check`
- `app_release_versions`
- 宿主 APK 版本口径
- about / version center / release script 绑定链

### 解释

本版虽然偏后端收口，但仍不得破坏宿主版本链一致性。

## 5. 本版优先排查的问题清单

## 5.1 地址问题

必须优先确认：
- `public_url` 是否真实可访问
- 是否错误写入 localhost / 内网 host / 不可达容器地址
- 图片与视频是否共用了错误 host / path
- 缩略图路径与原始资源路径是否分离正确

## 5.2 对象存储问题

必须优先确认：
- 文件是否真实上传成功
- 存储 key 是否正确
- 上传后的对象是否真的存在
- 读取时是否命中了错误 bucket / prefix / endpoint

## 5.3 响应头与播放器兼容问题

必须优先确认：
- `Content-Type` 是否正确
- 视频返回是否支持播放器读取
- 是否支持必要的 `Range` 请求
- 是否存在 302/403/404/无效签名导致 ExoPlayer Source error

## 5.4 处理链问题

必须优先确认：
- 视频是否真的完成转码/预处理
- ready 状态是否过早写入
- 缩略图生成是否成功
- 播放地址是否指向尚未完成或不可读的产物

## 5.5 UI 状态口径问题

必须优先确认：
- “已完成”是否真的意味着可发送 / 可播放
- 如果只是上传完成、处理未完成，UI 是否误导用户
- 错误状态是否被压成泛化的“加载失败”而丢失可解释性

## 6. 工作包拆分

## 6.1 4.4S-A：媒体链排障与边界冻结

### 目标

先把问题拆明白，确认本次到底修哪条链，而不是盲目改代码。

### 必做任务

1. 画清媒体读取链：上传 -> 存储 -> 处理 -> URL 生成 -> 消息回读 -> 前端显示 -> 播放  
2. 对图片与视频分别列出当前实际失败点  
3. 列出当前环境中的 host / path / storage provider / access mode  
4. 明确是地址、权限、响应头、对象缺失、还是转码产物问题  
5. 冻结本版边界：只修稳定性，不扩功能  

### 必要验收条件

- 产出一份“媒体链排障说明”
- 能明确说出图片失败与视频失败是否同源
- 能明确说出修复优先级顺序

### 解释

如果不先定位，Codex 很容易继续在 UI 层绕圈，而真正的后端问题会被掩盖。

---

## 6.2 4.4S-B：后端媒体地址与读取链修复

### 目标

把图片 / 视频的资源地址、读取链、存储访问链修正到真实可读。

### 必做任务

1. 修复 `public_url` / 播放地址 / 缩略图地址生成逻辑  
2. 校正 host、prefix、endpoint、storage key 拼接  
3. 确保图片与视频资源在测试环境中真实可访问  
4. 若使用签名 URL，校正签名过期、生成口径与播放器可用性  
5. 校正后端返回字段，避免前端拿到无效地址  

### 必要验收条件

- 图片资源 URL 在当前测试环境可打开
- 视频资源 URL 在当前测试环境可被播放器读取或最小下载验证读取
- 后端返回字段与真实资源一致

### 解释

这是本版最关键工作包。  
如果地址链没修好，前端再怎么补播放器和 UI 都没有意义。

---

## 6.3 4.4S-C：媒体处理与状态机收口

### 目标

确保“ready / processing / failed”真正代表资源状态，而不是假状态。

### 必做任务

1. 检查视频处理任务何时写 `ready`  
2. 确保缩略图 / 播放源没生成成功时不提前给 ready  
3. 补充或修正错误码 / error_code / failure reason  
4. 确保图片加载失败不是被错误状态吞掉  
5. 若必要，收紧 UI 对“已完成”的表述，使其与真实媒体状态一致  

### 必要验收条件

- ready 前后行为一致
- processing 态不会误导用户点开无效视频
- failed 态能解释是读取失败、处理失败还是资源不存在

### 解释

现在截图里“已完成”与“视频无法播放”并存，这说明状态机和前端文案至少有一侧没对齐。

---

## 6.4 4.4S-D：Flutter 端媒体读取与播放兜底修正

### 目标

在后端链修正后，让前端真正把正确状态表达出来，并避免把底层播放器异常直接裸露给用户。

### 必做任务

1. 图片气泡 / 预览页在资源存在时正常显示  
2. 视频气泡点击进入播放页时，若源有效则正常播放  
3. 若播放失败，给出用户可理解的失败说明，不直接把大段底层异常暴露成主文案  
4. 若资源还在 processing，展示“处理中 / 暂不可播放”而不是 Source error  
5. 校正“选择图片 / 视频”“已完成”等文案口径，避免误导  

### 必要验收条件

- 图片成功态可见
- 视频成功播放态可见
- 失败态可解释但不工程化
- 文案与真实状态一致

### 解释

本版不是只修后端。  
如果后端已好但前端仍把状态表达错，验收仍然过不了。

---

## 6.5 4.4S-E：回归、截图证据、closeout 收口

### 目标

让这次修复不是“感觉好了”，而是能被正式验收。

### 必做任务

1. 补图片成功加载截图  
2. 补视频成功播放截图  
3. 补会话列表视频消息摘要截图  
4. 补聊天页待发送 / ready / 播放链路截图  
5. 更新执行记录、验收摘要、handoff、closeout  
6. 明确本轮修复的是后端媒体链，不是继续扩功能  

### 必要验收条件

- 截图证据形成真正成功链路
- 文档说明与截图一致
- 能支撑顾问把 `4.4` 从 `conditional pass` 提升到 `pass with observations`

## 8. 本版验收结论建议口径

## 8.1 何时可以给 `pass with observations`

仅当以下同时成立：
- 图片成功加载链已打通
- 视频成功播放链已打通
- 会话摘要语义正确
- 回归完整
- 截图证据完整
- 剩余问题只是不影响主链的轻微 observation

### 建议

`4.4S` 的现实目标不是追求纯 `pass`，而是：  
**把当前 4.4 从 `conditional pass` 推进到 `pass with observations`。**

## 8.2 何时仍只能是 `conditional pass`

若出现以下情况之一：
- 图片仍频繁加载失败
- 视频播放仍报 source error
- 只能证明路径可达，不能证明成功播放
- 文案与状态仍对不上
- 后端日志与实际资源链仍说不清

则仍应维持 `conditional pass`，不得强行归档。

## 9. 风险清单

## 9.1 最大风险：把稳定性修复做成后端大重构

控制原则：
- 只修媒体主链阻断
- 不扩展成 OSS / CDN 全面改造项目

## 9.2 第二风险：继续用前端假状态掩盖后端问题

控制原则：
- 不允许靠 UI 掩盖不可读资源
- 必须让资源真正可读

## 9.3 第三风险：修视频时再次打坏图片主链

控制原则：
- 图片与视频一起回归
- 不允许“修好视频、打坏图片”

## 9.4 第四风险：底层错误直接暴露给用户

控制原则：
- 开发阶段可以保留详细日志
- 用户面要做可理解降噪
