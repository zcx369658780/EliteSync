# 4.9 媒体 / 队列 / Worker 可观测性说明

更新时间：2026-04-28

## 目标

让图片、视频、媒体读取、媒体处理任务在失败时可定位，不再只能依赖用户截图判断。

## 当前链路

### API / 控制层

- `POST /api/v1/media`
- `GET /api/v1/media/{assetId}`
- `GET /api/v1/media/{assetId}/content`
- `POST /api/v1/media/{assetId}/process-demo`

### 服务层

- `MediaAssetService`
- `MediaStorageService`
- `MediaCacheService`
- `MediaStateMachine`
- `MediaUploadPolicy`

### 队列 / Worker

- `ProcessMediaAssetJob`
- `app:media:process-queue`
- `media_processing_jobs`

### 真值表

- `media_assets`
- `media_processing_jobs`
- `message_attachments`

## 现有日志点

### 控制层日志

- `media_upload_failed`
  - 上传入口异常；
  - 记录 `owner_user_id`、`media_type`、`message`。
- `media_asset_saved`
  - 媒体资产创建或更新成功；
  - 记录 `asset_id`、`owner_user_id`、`media_type`、`storage_disk`。
- `media_content_stream_failed`
  - 媒体内容读取异常；
  - 记录 `asset_id`、异常信息。
- `media_process_demo_triggered`
  - 手动触发 demo 处理；
  - 记录 `asset_id`、`owner_user_id`。

### 服务层失败码

- `mime_not_allowed`
- `upload_too_large`
- `storage_write_failed`
- `processing_failed`
- `asset_missing`
- `awaiting_upload`

### 队列 / 处理结果

- `queued`
- `running`
- `succeeded`
- `failed`

## 建议日志字段

为了 5.0 前更好排障，建议媒体事件日志至少统一以下字段：

- `request_id`
- `user_id` / 脱敏 id
- `asset_id`
- `media_type`
- `storage_provider`
- `storage_disk`
- `storage_key`
- `public_url`
- `status_before`
- `status_after`
- `error_code`
- `error_message`
- `job_id`
- `job_type`
- `attempt_count`
- `max_attempts`
- `retry_count`
- `duration_ms`

## 失败码建议含义

| 失败码 | 含义 | 常见触发位置 |
|---|---|---|
| `mime_not_allowed` | 文件类型不允许 | 上传校验 / worker |
| `upload_too_large` | 文件超出限制 | 上传校验 |
| `storage_write_failed` | 存储写入失败 | 对象存储 / 文件系统 |
| `processing_failed` | 处理任务失败 | 同步处理 / job 执行 |
| `asset_missing` | 处理任务找不到资产 | 队列延迟 / 脏任务 |
| `awaiting_upload` | 等待上传完成 | 处理队列 / demo 命令 |

## 建议的最小可观测性动作

1. 保留 `media_upload_failed` 和 `media_asset_saved`；
2. 在 worker / queue 路径补齐 job_id / attempt_count / duration_ms；
3. 失败时必须携带 `asset_id` 和 `error_code`；
4. `public_url` 只做显示，不允许写回 localhost；
5. `content` 端点失败时必须能看到是磁盘、storage_key 还是 URL 解析问题。

## 不做什么

- 不重写媒体主链；
- 不新造媒体平台；
- 不把 `public_url` 硬编码为前端路径；
- 不把 localhost 写回正式数据；
- 不用 UI 假成功掩盖媒体读取失败。

## 结论

4.9D 的目标不是增加媒体功能，而是把图片 / 视频 / 媒体读取 / 队列处理失败变成可查证、可复验、可收敛的工程化链路。
