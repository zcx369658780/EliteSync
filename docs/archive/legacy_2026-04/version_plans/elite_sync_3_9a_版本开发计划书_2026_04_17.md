# EliteSync 3.9 版本开发计划书

日期：2026-04-17  
版本定位：`3.9 收尾计划 / pass with observations 后续收口版`  
当前基线：`0.03.09 / 30900`  
计划适用对象：Codex-CLI、Claude-MCP、Gemini-MCP、项目顾问验收

---

## 0. 计划背景与本版定位

### 0.1 当前背景

3.9 版本已通过顾问验收，当前归档口径为：

> **pass with observations / 正式归档通过（保留非阻断观察项）**

当前顾问结论已经明确：
- 3.9 **不需要打回重做**；
- 已完成高级时法框架首版、细粒度解释层接入、截图证据索引、样例矩阵与日志摘要链路；
- `canonical truth` 边界已守住；
- 允许正式归档，但保留若干非阻断观察项，建议继续收尾。

### 0.2 本版计划的真正目标

本计划书**不是新一轮功能扩展计划**，也不是 4.x 基础能力补全计划的替代品，而是：

> **针对 3.9 已通过但仍保留 observation 的部分，进行一次严谨、有限、可控的收尾开发。**

换言之，本版是一个**“收尾计划书”**，不是“重功能版本计划书”。

### 0.3 本版的总体原则

必须遵守以下原则：

1. **不推翻 3.9 已归档结论。**  
2. **不把 3.9 收尾做成 4.x 的功能前置。**  
3. **不新增大功能面。**  
4. **只处理 observation、算法版本化收口、证据链加厚、工程语义降噪、回归材料补厚。**  
5. **所有工作都必须以“守住 canonical truth 边界”为前提。**

## 1. 3.9 当前归档结论与收尾必要性

### 1.1 当前顾问认可的已完成内容

以下内容视为 3.9 已完成，不得再作为“待实现目标”重新发散：

- 高级时法框架首版已落地；
- 细粒度解释层已接入预览页 / 演示页；
- 页面命名边界与显示边界已基本建立；
- 样例矩阵与日志摘要链路已形成；
- `display-only / local preference / canonical truth` 边界已建立；
- 至少 1 个正式时法能力与 1 个占位能力并列展示方案已被接受；
- 多 Agent 审查链已闭环；
- 版本已归档通过。

### 1.2 当前仍保留的 observation

根据顾问验收结论与交接口径，当前 observation 主要有 3 类：

#### observation A：细粒度解释层的可视证据仍然不够强
问题本质：
- 文档链和验收链都说“有”，但聊天窗口里直接能看见的条目级卡片证据还不够强；
- 证据存在，但呈现厚度不足；
- 下一次归档时，最好让外部审阅者不用“相信文档”，而是能直接“从截图和页面中看见”。

#### observation B：高级时法页仍有少量工程口径外露
问题本质：
- `display-only`、`advanced-context`、`annual/profection-like`、`firdaria-like` 等词对顾问和工程验收友好；
- 但对终端用户来说仍然带有明显工具感和工程语义；
- 当前不阻断归档，但应在收尾版中进一步降噪。

#### observation C：回归材料偏薄
问题本质：
- 当前回归材料更像保护面说明，而不是足够厚的执行记录；
- 保护面列得清楚，这是优点；
- 但如果未来还要继续扩展高级能力，则需要更厚的逐项回归结果作为基线。

### 1.3 本版收尾的必要性

虽然 3.9 已通过，但收尾依然必要，原因有三：

1. 这是 **3.x 线正式收口前的最后机会**，适合把“能低成本补强”的内容补齐；  
2. 当前 observation 都属于“现在不修，以后更难统一口径”的问题；  
3. 4.x 方向已经重整为基础能力补全，若现在不把 3.9 收尾，后续容易形成两套混杂基线。

## 2. 本版版本定位与目标边界

### 2.1 本版正式定位

本版定位为：

> **EliteSync 3.9 收尾版（3.9A / 3.9B / 3.9C 合并规划口径）**

它不是 3.9 的返工，而是：
- 对 observation 的定向补强；
- 对 3.x 高级能力链路的最后一次结构化收口；
- 为 4.x 提供更干净的历史基线。

### 2.2 本版必须完成的目标

本版只围绕以下 5 个目标展开：

#### 目标 1：补强细粒度解释层的可视证据
- 单独形成“条目级解释卡片”的清晰可验收页面证据；
- 把“有此能力”提升为“看得见、截得到、能归档”。

#### 目标 2：继续降低高级时法页的工程语义暴露
- 在不破坏边界表达的前提下，减少页面工程术语；
- 把页面从“工具感”进一步向“产品感”推进。

#### 目标 3：把回归材料从“保护面说明”加厚为“执行结果记录”
- 将当前回归口径升级为更明确的测试执行清单、结果记录、截图映射、风险结果。

#### 目标 4：把星盘 / timing / pair_chart / advanced-context 的版本边界再明确一层
- 不做大算法扩张；
- 只做版本化和边界清晰化收尾；
- 为未来 4.x 及算法线单独评审提供更稳基线。

#### 目标 5：不破坏当前通过结论与保护面
- 所有改动都必须可回归；
- 不得为了收尾而伤害已归档能力。

### 2.3 本版明确不做的内容

本版严禁做以下事项：

- 不新增媒体消息能力；
- 不新增动态流、状态流；
- 不新增 MBTI 正式产品链路；
- 不做音视频通话；
- 不做 4.x 功能前置；
- 不改版本主线定位；
- 不重构现有匹配主链路；
- 不触碰用户资料真值链路；
- 不发散为新的玄学功能扩展版。

## 3. 本版任务拆分总览

本版建议拆分为 3 个收尾包，按顺序实施：

- **3.9A：细粒度解释层证据补强与展示降噪**
- **3.9B：回归材料加厚与截图证据索引完善**
- **3.9C：星盘 / 高级时法 / pair_chart 的版本边界收口**

这 3 个包均属于 3.9 收尾范围，不视为 4.x。

## 4. 3.9A：细粒度解释层证据补强与展示降噪

### 4.1 目标

解决当前 observation A 和 B 的大部分问题：
- 让细粒度解释层“看得更见”；
- 让高级时法页“更像产品页而不是工具页”。

### 4.2 必做任务

#### 4.2.1 条目级解释卡片证据页补齐
必须完成：
- 单独梳理“条目级解释卡片”页面或清晰展示区域；
- 能展示至少 3 类不同层级的解释结构；
- 每个解释卡片都能看出“证据 → 解释 → 提示”的关系；
- 页面足够清楚，可单独截图归档；
- 至少形成 1 张用于顾问验收的标准截图。

建议结构：
- 标题层
- 证据标签层
- 中间判断层
- 用户语言解释层
- 补充提示层

#### 4.2.2 高级时法页命名降噪
必须完成：
- 对当前页面中工程感较强的文字进行梳理；
- 保留内部结构边界，但对用户可见文本做更产品化命名；
- 避免继续增加新工程术语；
- 保留必要的专业能力区分，但不写成工程对象名。

示例方向：
- 页面展示文案可以更偏“年度重点 / 节律关联 / 解读入口”等产品化表达；
- 工程字段名、内部 feature 名、日志标签不直接裸露给用户。

#### 4.2.3 页面层级与视觉密度优化
必须完成：
- 梳理高级时法页的信息层级；
- 减少页面一屏内的术语密度；
- 优化卡片间距、标题层级、说明文案位置；
- 降低“开发工具页”观感。

### 4.3 可选优化

可做但非必做：
- 增加一页仅供验收使用的演示态页面；
- 为不同解释层级增加更明显的视觉分层；
- 增加“查看依据”一类轻量入口，但不得引发大结构改动。

### 4.4 任务边界解释

本包允许改：
- 页面命名
- 用户可见说明文案
- 卡片层级
- 展示排序
- 截图证据页

本包禁止改：
- canonical truth
- 星盘计算真值
- 数据库 schema
- 匹配总分逻辑
- 高级能力的核心算法输出

### 4.5 验收标准

必须同时满足：
- 能清晰展示细粒度解释层卡片；
- 顾问无需依赖文字说明即可从截图中看懂条目级解释存在；
- 高级时法页工程术语明显减少；
- 页面层次比 3.9 原版更清楚；
- 不破坏原有高级时法框架首版结构。

## 5. 3.9B：回归材料加厚与截图证据索引完善

### 5.1 目标

解决 observation C：让回归材料从“有保护面”提升到“有执行记录、可复查、可归档”。

### 5.2 必做任务

#### 5.2.1 形成正式的回归执行清单
必须完成：
- 把当前保护面转换成明确的测试执行项；
- 每项要有：
  - 测试对象
  - 测试动作
  - 预期结果
  - 实际结果
  - 结论
  - 风险说明（如有）

至少覆盖：
- `profile/basic` 保存与重算链路
- `profile/astro/summary`
- `profile/astro/chart`
- 盘面设置页本地偏好边界
- 高级时法预览页 / 演示页
- 样例矩阵
- 日志摘要输出
- 旧入口路由
- 版本中心 / about 页
- 版本检查接口

#### 5.2.2 截图证据索引加厚
必须完成：
- 为关键功能建立截图编号；
- 每张图映射到对应测试项与文档说明；
- 至少覆盖：
  - 细粒度解释层卡片
  - 高级时法框架页
  - 样例矩阵页
  - 日志摘要页
  - 盘面设置页边界说明
  - 版本中心页

#### 5.2.3 把“通过但保留观察项”的口径固化进文档
必须完成：
- 在收尾材料中明确写出本版归档结论；
- 明确 observation 已被处理到什么程度；
- 明确哪些问题已关闭、哪些问题仍保留到 4.x 或算法线。

### 5.3 可选优化

可做但非必做：
- 增加回归结果表格模板；
- 增加截图与 commit / PR / 页面路径的三向映射；
- 增加风险项分级。

### 5.4 任务边界解释

本包允许改：
- 回归文档
- 截图索引
- 页面证据编排
- 测试记录形式

本包禁止改：
- 为了“让回归更好看”而修改业务逻辑；
- 为了“增加截图”而新增无必要页面；
- 把回归文档写成空泛口号而非执行记录。

### 5.5 验收标准

必须同时满足：
- 回归材料不再只是保护面说明；
- 有逐项执行记录；
- 截图证据编号清晰；
- 顾问可通过文档与截图快速复查；
- 回归材料与页面证据之间可相互指认。

## 6. 3.9C：星盘 / 高级时法 / pair_chart 的版本边界收口

### 6.1 目标

本包不是大改算法，而是把 3.x 已引入的高级能力边界再收紧一层，为后续算法演进线提供更清楚的历史基线。

### 6.2 必做任务

#### 6.2.1 星盘输出结构边界再次确认
必须完成：
- 确认当前星盘输出的 schema 边界；
- 确认哪些字段属于 canonical truth；
- 确认哪些字段属于 derived-only / display-only；
- 确认前端本地渲染偏好不写回真值链。

#### 6.2.2 timing / advanced-context 的 derived-only 边界强化
必须完成：
- 在代码与文档中再次明确：
  - `summary`
  - `entry`
  - `timing-association`
  属于展示/解释层；
- 它们不承担写回，不替代星盘真值；
- 如存在实验能力，应明确 feature flag 或实验口径。

#### 6.2.3 pair_chart 的版本口径收尾
必须完成：
- 明确 pair_chart 当前是否已进入稳定匹配链；
- 如果未进入，则明确它当前只是“版本化预留能力”；
- 如果已局部进入，则必须明确其读取边界和权重边界；
- 不允许 pair_chart 在本收尾版中变成新的大开发面。

#### 6.2.4 证据标签与解释模板边界说明
必须完成：
- 解释标签的来源要明确；
- 模板与证据不要混写；
- 文案模板可以优化，但不得伪装成算法输出本身；
- 日后算法升级应另起版本规划，不在本收尾版中偷偷推进。

### 6.3 可选优化

可做但非必做：
- 为算法边界补一份简明的版本说明表；
- 对 pair_chart / timing / advanced-context 增加一份“当前阶段口径说明”。

### 6.4 任务边界解释

本包允许改：
- 版本边界说明
- 文档口径
- feature flag 标注
- derived / canonical 的注释与结构说明

本包禁止改：
- 大规模重写算法引擎；
- 升级星盘引擎版本并引发大量结果变化；
- 在 3.9 收尾版中引入新的高风险算法能力；
- 把 derived 层误写进 canonical truth。

### 6.5 验收标准

必须同时满足：
- canonical / derived / display-only 边界更清楚；
- pair_chart / timing / advanced-context 的当前阶段口径更明确；
- 不新增新的算法混乱点；
- 文档与代码表达一致。

## 7. 本版必须保护的边界（硬性保护面）

以下保护面在本版本中默认**禁止破坏**：

- `POST /api/v1/profile/basic`
- `GET /api/v1/profile/astro/summary`
- `GET /api/v1/profile/astro/chart`
- `GET /api/v1/app/health`
- `GET /api/v1/app/version/check`
- `astro_chart_preferences_v1`
- `user_astro_profiles`
- `app_release_versions`
- `moderation_reports`
- synthetic 批次审计字段
- 地点搜索与地理编码链路
- 权限 / 路由 / 生命周期 / 状态缓存
- 发布脚本版本检查 / 下载地址 / SHA256 口径

### 7.1 为什么这些边界必须保护

#### `profile/basic` / `astro/summary` / `astro/chart`
因为它们是资料真值链与星盘展示链的底座，一旦破坏，将导致整个画像和解释系统失稳。

#### `astro_chart_preferences_v1`
因为 3.9 的重要成果之一就是建立“本地展示偏好不影响命盘真值”的边界，这条边界不能回退。

#### `app_release_versions` 与版本检查链路
因为 3.x 已完成发布、归档与版本中心收口，本次收尾不允许破坏版本检查的产品可信度。

#### `user_astro_profiles`
因为它是当前 canonical 快照的重要承载点，3.9 收尾不应触碰其核心语义。

## 8. 本版不允许出现的错误倾向

为避免 Codex 在执行过程中发散，本版明确禁止以下倾向：

1. 借“收尾”名义扩展 4.x 功能；
2. 借“解释层优化”名义改动真值层；
3. 借“术语降噪”名义删除必要边界说明；
4. 借“回归加厚”名义制造大量无意义截图；
5. 借“版本边界明确”名义启动新算法重构；
6. 为追求漂亮页面而伤害原有稳定路径；
7. 在没有顾问评审前同时推进多条高风险链路。

## 14. 本计划的最终定位总结

这份计划书的意义，不是要求团队再“重做一次 3.9”，而是：

- 用最小的成本补齐 3.9 的 observation；
- 用最小的风险加厚证据与回归链；
- 用最明确的边界收紧 3.x 的算法与展示口径；
- 为 4.x 的基础能力补全主线腾出一个更干净、更稳定的历史起点。

因此，本版的最佳执行策略应当是：

> **小步、有限、保守、可回归地完成 3.9 收尾，而不是把它做成新的大开发版本。**
