# EliteSync 2.7 版本 PRD + 执行清单 + 多 Agent Prompt

更新时间：2026-04-06
适用版本：`2.7`
当前基线版本：`0.02.06`（2.6.4 已结项）
当前阶段定位：**慢约会核心体验补完版**

---

## 0. 版本摘要

### 0.1 一句话目标

**2.7 的目标不是再做更多页面，而是把 EliteSync 真正做成“有仪式感的慢约会产品”。**

### 0.2 这一版要补的灵魂能力

1. `Drop` 倒计时系统
2. 匹配揭晓页与揭晓动画
3. 盲盒式资料解锁机制
4. 破冰问题系统
5. 匹配后反馈机制

### 0.3 2.7 与 2.6.4 的关系

2.6.4 已经把：

- smoke / regression / release gate 固化
- 发布门禁收口
- 文档索引归档
- 回滚与验收流程固化

因此 2.7 可以放心进入“产品体验补完”，但必须继续继承 2.6.4 的工程纪律：

- 不误伤数据库与认证主链路
- 不破坏资料真源与画像链路
- 每一块新体验都必须有最小可验收闭环

## 1. 版本定位

## 1.1 当前产品所处位置

截至 2.6.4 结束，EliteSync 已具备：

- 注册 / 登录
- 问卷作答
- 每周匹配
- 匹配解释
- 双向确认
- 聊天
- 首页内容流
- 资料编辑
- 玄学总览 / 八字 / 星盘 / 紫微 / 诊断
- 回归、发布、回滚门禁

这意味着产品已经完成“功能闭环”，但还没有真正完成“情绪闭环”。

## 1.2 2.7 的核心问题

目前的痛点不是“没有匹配”，而是：

- 用户不知道什么时候最值得回来打开 App
- 匹配结果出现时不够有仪式感
- 确认前后信息边界不够明确
- 聊天开始太冷，缺少开场引导
- 用户体验反馈没有被系统化记录

因此 2.7 的目标是补足 **等待 -> 揭晓 -> 选择 -> 解锁 -> 开聊 -> 反馈** 这条情绪链。

## 1.3 2.7 不做的事情

本版本默认不做：

- 不重构认证体系
- 不做学信网 / 高校邮箱正式接入
- 不做运营后台大版本
- 不大改聊天底层架构
- 不新增大体量玄学模块
- 不修改资料保存真源逻辑
- 不改数据库核心 schema，除非极小增量且可回滚
- 不在本轮推进付费 / 会员系统

## 2.1 用户体验目标

让用户在 2.7 中感受到：

1. **我知道什么时候会有结果**
2. **结果揭晓是值得期待的事件**
3. **我不是一上来就看到一切，而是逐步解锁**
4. **我不是冷启动聊天，而是有系统帮我破冰**
5. **我用完之后，系统会收集并学习我的感受**

## 2.2 业务目标

- Drop 页面打开率

- 首条消息发送率
- 破冰问题点击率
- 破冰问题发送率

## 3. 2.7 的产品结构

## 3.1 总体链路

```text
首页 / MatchTab
  -> Drop 倒计时
  -> 9 点后进入揭晓页
  -> 显示悬念版匹配结果
  -> 用户选择 喜欢 / 略过
  -> 若双方都喜欢 -> 进入解锁态
  -> 显示更完整资料 + 破冰问题
  -> 进入聊天
  -> 匹配后反馈
```

## 3.2 状态机（核心）

建议统一定义为：

### A. Match Cycle State

- `pre_drop`：未到揭晓时间
- `countdown_live`：倒计时展示中
- `drop_open`：可揭晓
- `revealed_teaser`：已看到悬念版
- `liked_pending`：我已喜欢，等待对方
- `skipped`：我已略过
- `double_like_unlocked`：双方已喜欢，已解锁
- `chat_started`：已进入聊天
- `feedback_pending`：待填写反馈
- `feedback_done`：已提交反馈

### B. Profile Reveal State

- `teaser_only`：只展示悬念信息
- `partial_reveal`：展示更多标签与亮点
- `full_reveal`：展示完整资料与聊天入口

### C. Icebreaker State

- `not_ready`：未解锁
- `ready`：可显示推荐问题
- `sent`：已发送至少一条
- `refresh_available`：可换一组问题

## 4. 2.7 五大模块

---

## 4.1 模块一：Drop 倒计时系统

### 目标

让用户知道“结果什么时候来”，并形成每周固定的期待感。

### 设计原则

- 首页和 MatchTab 都能看到倒计时入口
- 倒计时信息必须明确而不打扰
- 倒计时要和每周匹配任务真实状态一致
- 若后端结果尚未 ready，要有降级提示

### 前端表现

#### pre_drop / countdown_live
- 顶部大倒计时卡片
- 文案示例：
  - 距离本周 Drop 揭晓还有 `02天 03小时 18分`
  - 今晚 21:00，新的匹配将准时揭晓
- 可搭配轻动画，但先不要做过重粒子特效

#### drop_open
- 按钮变成：`立即揭晓本周匹配`
- 主 CTA 必须明显

### 后端 / 数据建议

新增或兼容提供：

- 当前匹配周期 ID
- 本周 drop 时间
- 当前用户在该周期的可见状态
- 是否已有匹配结果
- 是否允许进入揭晓页

### 完成定义

- 用户在未到时能看到准确倒计时
- 到点后 CTA 自动切到可揭晓状态
- 后端未 ready 时前端不误导用户

---

## 4.2 模块二：揭晓页与悬念版结果

### 目标

把匹配结果从“普通列表展示”升级为“事件化揭晓”。

### 设计原则

- 先给惊喜，再给信息
- 初次揭晓只给悬念版，不直接给完整资料
- 首屏信息要少而准

### 首屏建议信息结构

揭晓后先展示：

- 模糊头像 / 半遮挡头像
- 昵称缩写
- 学校 / 城市（必要时做脱敏）
- 3 个匹配亮点
- 一句简短解释文案

### 匹配亮点来源建议

优先顺序：

1. 问卷共鸣标签
2. 生活方式相近或互补标签
3. 玄学画像中的温和型亮点
4. 已经存在的匹配解释摘要

### 交互按钮

- `喜欢` / `继续了解`
- `略过`

### 完成定义

- 用户打开匹配结果时有明显“揭晓”感
- 首屏不是信息轰炸，而是悬念信息 + 亮点
- 用户能清楚做出选择

---

## 4.3 模块三：盲盒式资料解锁

### 目标

让“双方确认”真正有价值，而不是形式化点击。

### 设计原则

- 信息必须分层，不要一开始全部给完
- 解锁逻辑要直观，不要让用户困惑
- 保持神秘感，但不能影响基本判断

### 建议三层结构

#### 第一层：悬念版（揭晓后）
展示：
- 模糊头像
- 昵称缩写
- 学校 / 城市脱敏信息
- 3 个匹配亮点
- 1 段简短匹配说明

不展示：
- 全头像
- 完整详细资料
- 完整玄学画像
- 联系方式

#### 第二层：单边喜欢后
展示：
- 更多兴趣标签
- 更多生活方式标签
- 部分个性 / 画像卡片
- “等待对方回应”的状态提示

#### 第三层：双边喜欢后
展示：
- 完整头像与昵称
- 更完整资料卡
- 破冰问题
- 聊天入口

### 文案建议

- 单边喜欢后：`你已表达好感，若对方也愿意了解，你们将解锁完整资料。`
- 双边喜欢后：`恭喜，双方都愿意继续了解，资料已解锁。`

### 完成定义

- 用户能理解为什么当前只能看到部分信息
- 用户能理解什么时候会解锁完整资料
- 双边确认后明显感受到“解锁成功”

---

## 4.4 模块四：破冰问题系统

### 目标

把聊天从“自己硬想第一句话”改为“系统辅助开场”。

### 设计原则

- 问题数量少而精
- 可直接发送
- 不要太尴尬、太模板化、太玄学重口
- 优先从双方真实共性出发

### 推荐结构

每组 3-5 个问题，来源分层：

#### A. 共鸣型
- 你们都喜欢安静型周末，如果周末只能选一个活动，你会选什么？
- 你们都偏爱稳定关系，你觉得一段关系里最重要的安全感来自哪里？

#### B. 差异型
- 你更偏向计划型，他/她更偏向随性型，你会怎么安排一次第一次见面？

#### C. 轻松型
- 如果把这次匹配当作一部电影开场，你觉得第一幕会发生什么？

### 功能按钮

- `发送这句话`
- `换一组`
- `我自己写`

### 显示位置

建议放在：

- 双边喜欢后的解锁页
- 聊天页顶部 card

### 完成定义

- 至少能稳定展示 3-5 条可读问题
- 一键发送可用
- 换一组逻辑可用
- 破冰问题不会影响正常输入消息

---

## 4.5 模块五：匹配后反馈

### 目标

让系统知道这次匹配体验是否有效，为后续排序和体验优化留数据。

### 触发时机建议

- 聊天开始 24 小时后
- 或匹配后 3-7 天
- 或对话结束后

### 最小反馈结构

1. 你觉得这次匹配是否值得继续了解？
2. 这次匹配解释是否准确？
3. 破冰问题是否有帮助？
4. 你愿意继续参加下周匹配吗？
5. 可选文本意见

### 数据用途

- 先作为产品体验数据
- 暂不直接重写主排序逻辑
- 后续可进入 2.8 / 2.9 的优化数据池

### 完成定义

- 反馈入口能触发
- 反馈数据能提交并落库
- 不阻塞用户正常使用

## 8. 2.7 用户故事

### 故事 1：等待
作为用户，我希望知道本周匹配什么时候揭晓，这样我会在正确时间回来打开 App。

### 故事 2：揭晓
作为用户，我希望匹配结果揭晓时有明显仪式感，而不是像普通列表刷新一样平淡。

### 故事 3：试探
作为用户，我希望在决定前先看到少量高价值信息，而不是被全部资料直接淹没。

### 故事 4：双向确认
作为用户，我希望双方都表达愿意继续了解后，再解锁更完整资料，这样会更有郑重感。

### 故事 5：开场
作为用户，我希望系统给我几句自然的破冰问题，降低聊天开场压力。

### 故事 6：反馈
作为用户，我希望使用后可以快速反馈这次匹配是否合适，而不必写复杂长评。

## 9. 2.7 验收标准

## 9.1 功能验收

- 用户能看到倒计时
- 到点后能进入揭晓页
- 揭晓页能展示悬念版匹配结果
- 喜欢 / 略过能正常提交
- 双边喜欢后能进入解锁态
- 破冰问题能展示、换一组、发送
- 反馈入口能触发并提交

## 9.2 状态验收

- 未到时间不会误显示结果
- 已到时间未 ready 时有降级提示
- 单边喜欢后能显示等待态
- 双边喜欢后状态能正确切换
- 破冰问题加载失败不影响聊天主入口
- 反馈加载失败不影响主功能

## 9.3 工程验收

- 2.6.4 smoke / regression 不倒退
- 资料保存与画像链路不受影响
- 聊天主链路不受影响
- 新页面失败不拖垮首页或 MatchTab
