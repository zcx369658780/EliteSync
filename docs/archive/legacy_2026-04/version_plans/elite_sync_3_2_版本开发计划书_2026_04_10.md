## 1. 3.2 版本定位

### 1.1 一句话定义

`3.2 = 匹配与互动转化优化 + 玄学可信度修复 + 测试运营增强 + 广场轻分层版`

### 1.2 为什么这样定义

3.1 已经把账号分层、状态发布/广场最小闭环、管理员移动端控制面、匹配调试底座、可见性证据与清理流程打通。3.2 不应再回到“有没有系统”的阶段，而应围绕以下四个问题推进：

1. 匹配之后能不能更顺畅地聊起来、持续聊下去
2. 玄学结果是否可信、易读、可稳定回归
3. 测试账号体系是否规模化、可重建、可清理、可隔离
4. 广场是否能更好服务真实互动而不是只做状态展示

### 1.3 本版核心目标

#### P0（必须完成）

1. 建立正式的测试账号生成与管理体系
2. 首阶段默认支持 `1000` 个 `synthetic` 测试账号池
3. 支持按批次扩展到 `10000` 个 `synthetic` 测试账号用于压力验证
4. 保留 `20-50` 个高质量人工 `test` 账号用于人工回归与验收
5. 测试账号资料完整，便于匹配、广场、聊天与玄学算法联调
6. 修复玄学排盘 correctness 与 readability 的 P0 问题
7. 强化匹配后的首聊、恢复、持续互动转化
8. 广场做轻量分层与审核/可见性增强
9. 形成完整的阶段验收与最终归档包

#### P1（本版可选，做不完可后移）

1. 关系进度条 / 熟悉度 / 轻量互动任务
2. 更细的广场频道标签与排序策略
3. 更细的管理查询与批次统计面板
4. 更丰富的 synthetic 行为脚本模板

#### 暂不纳入 3.2 主线

1. 全量专业占星模式扩张（如大量进阶盘型的一次性完整落地）
2. 校园垂直化战略大转向
3. 重直播、重打赏、重语音房模式
4. AI 虚拟恋人主产品化

## 3. 长期约束与保护面

以下约束在 3.2 全程不可违反：

### 3.1 真源约束

1. 服务端 canonical 数据是唯一真源
2. 前端缓存不能成为第二真源
3. 玄学计算结果以服务端 canonical 为准，前端只负责输入、展示、交互与容错

### 3.2 玄学约束

1. 八字、节气、星盘、宫位、时区、经纬度、真太阳时链路不得重新回到手写近似公式
2. 计算层与解释层分离
3. 展示层与排序层分离
4. 先修 correctness，再做 depth expansion

### 3.3 测试数据约束

1. `normal / test / synthetic` 必须继续分层
2. `synthetic` 默认 `exclude_from_metrics = true`
3. `synthetic` 是否进入普通流必须受控，不能默认无限混入
4. 必须支持 batch create / rebuild / cleanup / disable
5. 环境不得长期被临时验收账号与证据污染

### 3.4 变更安全约束

1. 所有高风险改动都要配套 smoke / regression / rollback 思维
2. 每个阶段都要有可截图、可复核、可追溯的证据
3. 修改不得破坏 3.1 已通过能力

### 3.5 保护面

重点保护以下 surfaces：

- 注册 / 登录
- 基础资料保存
- 出生地搜索
- 画像重算
- 匹配解释
- 每周匹配
- 首聊 / 会话链路
- 状态发布 / 广场显示
- 管理员移动端控制面
- synthetic batch 创建与清理

## 4. 多 Agent 协作架构

### 4.1 总体原则

- `Codex`：默认启动多 Agent 模式，担任总调度者、主架构师、主程序员、主 Reviewer、主执行人
- `Gemini-MCP`：负责平面 UI 设计审阅、交互视觉把关、阶段验收、最终验收
- `Claude-mcp`：只处理重要架构咨询，不承担日常实现与频繁 review

### 4.2 Agent 任务边界

#### Codex 边界

Codex 负责：

1. 阅读并消化仓库主交接与本计划书
2. 拆解版本任务与依赖顺序
3. 编码实现、迁移、脚本、接口联调、页面修改、日志补强
4. 主动调用 Gemini-MCP 做 UI / 阶段验收
5. 在遇到关键架构问题时调用 Claude-mcp 咨询
6. 自身完成代码 review、风险检查、回归执行与文档归档

Codex 不得：

1. 未经论证擅自扩大版本范围
2. 为了省事跳过阶段验收
3. 把前端临时计算伪装成 canonical 真源
4. 把 synthetic 数据默认写入真实运营指标
5. 在没有批次管理与清理能力前直接长期常驻 10000 账号

#### Gemini-MCP 边界

Gemini-MCP 负责：

1. UI 布局、视觉层次、可读性、拥挤度、弹窗/表单体验审查
2. 对星盘页、状态广场页、首聊引导页、管理页进行视觉审阅
3. 对每阶段交付做验收判定：通过 / 有条件通过 / 打回
4. 给出明确整改意见与截图要求

Gemini-MCP 不负责：

1. 直接主导数据库结构与后端架构设计
2. 直接决定 canonical 计算逻辑
3. 取代 Codex 进行主编码实现

#### Claude-mcp 边界

Claude-mcp 只负责以下重要架构问题：

1. synthetic 批量账号与真实环境隔离架构
2. 玄学 canonical 计算链路的 source-of-truth 设计
3. 批量生成、重建、幂等、清理与日志模型设计
4. 匹配/广场/管理查询在 1000 -> 10000 规模扩张时的结构风险
5. 复杂数据流与模块边界问题

Claude-mcp 不负责：

1. 日常 UI 微调
2. 一般代码格式 review
3. 简单 bug 修补
4. 普通页面文案润色

## 9. 阶段 1：测试账号体系与数据生成

### 9.1 目标

把 3.1 的测试运营底座推进为正式可规模化使用的测试账号体系。

### 9.2 账号体系设计要求

#### 人工 test 账号

- 数量：`20-50`
- 用途：人工回归、可见性验证、聊天验证、管理后台验证、运营演示
- 特点：资料真实感强、可手工登录、可长期保留、标识清晰

#### synthetic 账号

- 默认池：`1000`
- 扩展池：支持按批次扩展到 `10000`
- 用途：匹配批处理、广场可见性、排序、状态流、压力验证、数据分布验证
- 特点：可批量创建、可重建、可清理、可禁用、可审计

### 9.3 synthetic 账号资料完整性要求

每个 synthetic 账号至少需要具备以下信息：

#### 基础身份字段

- `nickname`
- `gender`
- `birthday`
- `age`
- `province`
- `city`
- `location_label`
- `birth_place_query_city`
- `birth_place_name`
- `birth_place_address`
- `birth_place_lat`
- `birth_place_lng`
- `profile_avatar_seed` 或同等可复现字段

#### 资料与匹配字段

- 自我介绍/简介
- 兴趣标签
- 恋爱/交友目标标签
- 基础偏好字段
- 至少一组用于匹配解释的结构化特征
- 是否可参与匹配
- 是否可进入广场
- 是否排除统计

#### 玄学相关字段

- 出生年月日时（若系统需要）
- 出生地点真实候选信息
- 经度纬度
- 时区/真太阳时链路可用
- 八字可重算
- 星盘可重算
- 玄学摘要页可展示
- 图表/summary 获取正常

#### 管理字段

- `account_type = synthetic`
- `batch_id`
- `seed`
- `created_at`
- `generation_version`
- `status`
- `visibility_scope`
- `exclude_from_metrics = true`
- `is_match_eligible`
- `is_square_visible`
- `cleanup_token` 或等价清理标识

### 9.4 位置与出生地点生成规则

1. 位置按中国省会城市随机分布，不要只集中在少数城市
2. 要有基础分布控制，避免极端不均衡
3. 出生地点不能只写“某市”作为占位
4. 出生地点必须先用地址搜索定位到用户所在城市，再在候选结果中随机选一个真实地点
5. 出生地点应保留搜索关键词、候选来源与最终选中结果，便于回归与排错

### 9.5 数据生成原则

1. 可复现：同 seed / 同 batch 配置可重建同批数据
2. 可分布：年龄、地区、兴趣、活跃度要有基础分布设计
3. 可清理：一个 batch 能整体禁用或整体清除
4. 可混流：允许受控地让部分 synthetic 进入普通流验证可见性
5. 可隔离：默认不污染真实运营指标

### 9.6 功能要求

必须提供：

- batch create
- batch rebuild
- batch cleanup
- batch disable
- batch summary
- batch audit log
- batch visibility control

### 9.7 阶段验收要求（Gemini + Codex 自检）

1. 至少完成一批 `1000` synthetic 账号生成
2. 至少保留 `20-50` 个人工 test 账号
3. 随机抽检若干账号，其昵称、生日、位置、出生地点完整
4. 随机抽检若干账号，其八字、星盘 summary/chart 均可用
5. 管理后台可查到批次
6. 账号可清理、可禁用、可重建
7. synthetic 默认不污染真实指标
8. 有截图和日志证据

### 9.8 建议提交给 Claude-mcp 的问题

- synthetic 批次模型与真实环境隔离如何设计最稳
- 1000 -> 10000 扩展时哪些字段和索引要预留
- batch rebuild 与 cleanup 如何保持幂等与可审计

## 10. 阶段 2：玄学可信度修复

### 10.1 目标

解决当前最影响用户信任与测试价值的玄学 correctness 与 readability 问题。

### 10.2 P0 任务

1. 校验星盘结果正确性
2. 校验宫位与度数偏移问题
3. 校验 Whole / 回归黄道等关键配置是否与目标口径一致
4. 检查时区、经纬度、真太阳时、house system 参数传递链路
5. 修复星盘页字体过大、重叠、拥挤问题
6. 增加更易用的日期选择器
7. 简化相位线，仅保留主要必要相位
8. 为参数、解读、星体、相位、主星、大限、流年等提供弹窗式阅读能力（做 P0 范围内最小可用版）

### 10.3 不在本阶段强行全做

以下内容可先做结构预留，不强求本版全量落地：

- 大量专业盘型的一次性完整支持
- 全量高级参数体系
- 非 P0 的深度占星扩展

### 10.4 阶段验收要求（Gemini）

1. 至少一个基准生日案例能与目标参考盘对齐到可接受范围
2. 页面不存在明显文字重叠、字号失衡、信息拥挤
3. 日期输入比当前更易用
4. 相位视觉噪音下降
5. 核心参数与解读入口清晰
6. 相关页面截图完整

### 10.5 建议提交给 Claude-mcp 的问题

- correctness 问题究竟是 canonical 源、配置参数、house system、时区、经纬度还是真太阳时传递导致
- 前端展示与后端返回如何解耦，避免前端再次解释计算

---

## 11. 阶段 3：匹配与互动转化优化

### 11.1 目标

把“匹配成功”推进为“真正开聊并持续互动”。

### 11.2 P0 任务

1. 优化首聊引导
2. 优化沉默后的恢复建议
3. 根据双方画像生成更可执行的聊天方向提示
4. 形成最小的互动进度感知（如是否已破冰/是否进入持续聊阶段）
5. 补充埋点与摘要，便于比较优化前后效果
6. 让匹配解释里的“行动建议”真正可被点击和使用

### 11.3 设计原则

1. 不做 AI 虚拟陪伴主产品
2. 不把聊天辅助做成喧宾夺主
3. 以“帮助真人互动”为目标，而不是取代真人聊天
4. 所有建议都要服务转化，不只是装饰文案

### 11.4 阶段验收要求（Gemini）

1. 首聊建议入口明显但不打扰
2. 恢复建议有场景感，不生硬
3. 页面路径清晰，普通用户能理解下一步该怎么做
4. 至少通过 synthetic/test 账号完成若干正向会话样例
5. 有前后对比截图与说明

### 11.5 建议提交给 Claude-mcp 的问题

- 聊天建议是实时生成、缓存生成还是事件触发生成更稳
- 埋点与转化摘要如何设计，既可分析又不污染主链路

## 12. 阶段 4：广场轻分层与运营增强

### 12.1 目标

让广场更好服务真实互动转化，而不是停留在最小状态流。

### 12.2 P0 任务

1. 为状态广场增加轻量标签/频道结构
2. 优化首页广场预览与广场页的入口逻辑
3. 增强可见性与审核控制
4. 支持更清晰的 synthetic / test / normal 可见性验证
5. 对管理后台补最小批次查询与内容治理视角

### 12.3 设计原则

1. 轻分层，不做大而全内容平台
2. 优先服务匹配与互动，不偏离慢约会主线
3. 管理与运营能力先于花哨玩法

### 12.4 阶段验收要求（Gemini）

1. 广场层次更清楚
2. 用户能理解不同标签/频道的意义
3. 管理端对内容可见性与删除能力可演示
4. synthetic/test/normal 的展示边界明确
5. 有完整截图证据
