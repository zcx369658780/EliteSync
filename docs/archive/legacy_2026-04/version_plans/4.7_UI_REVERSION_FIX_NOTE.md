# 4.7 UI Reversion Fix Note

## Cause

The UI regression happened when a repo-level recovery restored the tree to a prior functional baseline while fixing cross-layer blockers. That recovery recovered buildability and call / message paths, but it also overwrote later UI evolution because the modern surfaces were not protected.

## Triggering actions

- restoring the repo to an older known-good baseline
- recovering build artifacts and runtime chain state together
- treating build recovery as higher priority than UI preservation

## Impacted surfaces

- main navigation / bottom tabs
- home page
- messages / chat entry placement
- notifications center
- match / profile / questionnaire / version center entry discovery
- modern visual spacing and card treatments

## Recovery approach

- restore the modern UI baseline from the protected surface definition
- only use path-level restores for cross-layer fixes
- keep build / backend / RTC / media fixes isolated from UI files

## Forbidden git operations

- repo-wide hard reset
- repo-wide restore from old commit
- repo-wide clean of tracked/untracked state
- old zip or old worktree overwrite

## Path-level restore rule

Only restore the smallest file set needed for the blocker. If a path is in the protected UI surface list, stop and ask for approval before restoring it.

## Recovery checkpoint rule

Before any future recovery, capture:

1. `git status --short`
2. `git branch --show-current`
3. `git rev-parse HEAD`
4. `git diff --stat`
5. `git diff --name-only`

Then create a safety checkpoint by branch / commit / stash before restoring anything outside the smallest target path.

## Re-verification

After any future recovery, re-check the UI baseline evidence index and the UI baseline regression section of the checklist before release.
