# EliteSync 4.0 版本开发计划书

日期：2026-04-18  
版本定位：`4.0 基础能力基建版`  
当前承接基线：`3.9 收尾整体通过 / pass with observations`  
计划适用对象：Codex-CLI、Claude-MCP、Gemini-MCP、项目顾问验收

---

## 0. 本版定位与背景

### 0.1 为什么进入 4.0

3.x 版本线已经完成了从原型到可归档稳定基线的收口：
- 资料与画像真值链已形成；
- 星盘服务端计算 + Flutter 本地绘制路线已固定；
- 匹配解释与首聊转化链路已形成；
- 设置中心、版本中心、发布链、治理视图与文档索引已初步成型；
- 3.9 及其收尾任务（3.9A / 3.9B / 3.9C）已经把 observation 基本压平。

因此，项目不应继续停留在“收口、归档、降噪”的阶段，而应进入一个新的主线：

> **把 EliteSync 从“匹配解释型原型”推进为“具备基础社交能力承载底座的可测试产品”。**

### 0.2 当前现实约束

本版规划以以下现实约束为前提：

1. 公司尚未注册，短期内不能进入正式运营阶段；
2. 暂无大规模真实用户反馈，不能以运营指标驱动迭代；
3. 未来测试阶段一定会高频使用的功能尚未补齐；
4. 后端目前更适合走“可扩展单体”路线，而不适合直接大规模微服务重构；
5. 如果现在不把媒体、会话、缓存、队列、状态机等基础设施打好，后续图片/视频/状态流/音视频功能会面临高返工风险。

### 0.3 本版正式定位

4.0 的正式定位不是“功能大而全版”，也不是“运营准备版”，而是：

> **4.0 基础能力基建版：只做未来必需基础能力的承载底座，不追求一次性补齐所有社交功能。**

更具体地说，4.0 要完成的不是“把所有用户想要的功能都做完”，而是：

- 为图片消息、视频消息、状态流、关系链、通知、未来实时通话预埋稳定工程底座；
- 为 4.1、4.2、4.3 之后的功能版本提供清晰边界；
- 在不破坏 3.x 稳定基线的前提下，建立一个可持续迭代的 4.x 主干。

## 1. 4.0 的核心目标

本版只围绕以下 **6 个核心目标** 展开。

### 1.1 目标一：建立会话域、媒体域与关系域的清晰模块边界

必须明确：
- 会话（conversation）负责什么；
- 消息（message）负责什么；
- 附件（attachment）负责什么；
- 媒体（media）负责什么；
- 关系（relationship）负责什么；
- 通知（notification）未来应如何接入。

本版不是要把所有业务都做完，而是先把未来的模块边界划定清楚。

### 1.2 目标二：建立对象存储 + 媒体元数据 + 上传状态机基础设施

必须形成：
- 媒体不直存应用服务器本地；
- 数据库只持久化媒体元数据与状态；
- 上传、处理中、成功、失败、阻断等状态有统一状态机；
- 前后端对媒体状态的理解一致。

### 1.3 目标三：引入异步任务与缓存基础设施的最小闭环

必须形成：
- 队列 / worker 的最小可用链路；
- Redis 或等价缓存方案的最小可用链路；
- 能支撑后续图片压缩、视频转码、审核、通知等后台任务；
- 能为未来高频访问接口提供缓存扩展点。

### 1.4 目标四：建立前端附件组件、上传组件与错误态/重试态组件

必须形成：
- 会话里附件消息的通用组件；
- 上传中的占位态；
- 失败重试态；
- 处理中的过渡态；
- 空态、错误态、禁用态的一致表达。

### 1.5 目标五：建立基础日志、监控、限流与调试口径

必须形成：
- 上传链路日志；
- 关键 API 错误日志；
- 队列任务失败记录；
- 最小限流与防刷口径；
- 后续定位问题所需的基础可观测性。

### 1.6 目标六：明确 4.0 不做什么，避免功能发散

本版最重要的目标之一，就是**不乱做**。必须明确：
- 音视频通话不在 4.0 正式实现；
- MBTI 完整产品化不在 4.0 正式实现；
- 动态流完整交互不在 4.0 正式实现；
- 算法主线升级不在 4.0 正式实现；
- 不允许一边做基础设施，一边悄悄塞入大量前台产品功能。

## 2. 本版明确不做的内容

为了避免 4.0 演变为“什么都想做”的失控版本，以下内容明确列为 **本版不做**：

### 2.1 不做完整 MBTI 产品链路

不做：
- 正式题库体验页；
- 完整测试流程；
- 历史重测与结果对比；
- MBTI 正式接入匹配系统。

原因：这些属于 4.1 的主目标，不应与 4.0 基建混版。

### 2.2 不做完整状态流 / 动态流产品

不做：
- 完整发布页；
- 评论、点赞、删除、举报全链路；
- 推荐排序。

原因：动态流属于 4.3 的产品功能，不应在 4.0 先把基础设施版做成半成品主功能。

### 2.3 不做音视频通话正式功能

不做：
- 实时呼叫；
- 信令；
- RTC 集成；
- 在线状态系统。

原因：音视频通话是单独高风险版本，必须在 4.6 再规划。

### 2.4 不做算法主线升级

不做：
- 星盘引擎升级；
- pair_chart 运行时扩张；
- timing 进入主匹配链；
- 新玄学模块扩容。

原因：4.0 重点是产品基础设施和后端承载底座，而不是算法重构。

### 2.5 不做微服务大重构

不做：
- 全面拆服务；
- 分库分表；
- 跨区域多活；
- 超大规模推荐架构。

原因：现阶段还没有真实流量去支撑这种高成本重构，且会显著拖慢基础能力补全进度。

---

## 3. 4.0 的工作包拆分

本版建议拆为 **5 个工作包**，统一作为 4.0 版本主线实施。

- **4.0A：领域边界与数据骨架建立**
- **4.0B：对象存储与媒体状态机接入**
- **4.0C：缓存 / 队列 / 任务处理最小闭环**
- **4.0D：Flutter 附件组件与上传体验骨架**
- **4.0E：日志、监控、限流与交付文档补齐**

这些工作包不是独立产品版本，而是 4.0 的分阶段执行结构。

## 4. 4.0A：领域边界与数据骨架建立

### 4.1 目标

为后续功能版本先把“骨头”搭好，而不是先堆“肉”。

### 4.2 必做任务

#### 4.2.1 明确后端领域边界

必须至少明确以下领域：
- `conversation`
- `message`
- `media`
- `relationship`
- `notification`
- `rtc_signal`（预留）

要求：
- 每个域要有职责说明；
- 每个域与现有 `profile / astro / match / release / moderation` 的关系要写清楚；
- 不能出现“所有功能都直接挂在 messages”这种临时拼接结构。

#### 4.2.2 设计基础数据模型骨架

建议新增或明确以下表结构方向：
- `conversations`
- `conversation_members`
- `messages`
- `message_attachments`
- `media_assets`
- `media_processing_jobs`
- `user_relationship_events` 或等价关系表
- `notifications`（可先预留）

注意：
- 本版的目标是设计与建立骨架，不要求一次用满；
- 字段必须兼顾后续扩展；
- 要有创建时间、更新时间、状态字段、幂等/追踪字段；
- 软删除与审计字段要提前考虑。

#### 4.2.3 设计基础接口域

建议预留或建立：
- `/api/v1/conversations/*`
- `/api/v1/messages/*`
- `/api/v1/media/*`
- `/api/v1/relationships/*`
- `/api/v1/notifications/*`（预留）
- `/api/v1/rtc/*`（仅预留）

要求：
- 接口命名要统一；
- 不允许将媒体上传逻辑继续塞在现有杂糅接口里；
- 要有最小错误码约定。

### 4.3 任务边界解释

允许做：
- 建表 / 调整新表；
- 新增接口骨架；
- 新域模块拆分；
- 追踪字段、状态字段设计。

禁止做：
- 改动 3.x 保护面真值链；
- 把新域逻辑直接混入旧 `profile/basic` 或 `astro` 接口；
- 以“边界设计”为名进行大范围无计划重构。

### 4.4 验收标准

必须满足：
- 域边界文档清楚；
- 基础表结构与接口结构能支持后续图片消息扩展；
- 新骨架未破坏 3.x 已归档主路径；
- 至少形成一份“数据骨架与接口边界说明”。

## 5. 4.0B：对象存储与媒体状态机接入

### 5.1 目标

让未来的图片/视频功能不再依赖本地文件或临时方案，而是站在正确的存储模型上。

### 5.2 必做任务

#### 5.2.1 接入对象存储

必须完成：
- 选定对象存储（优先阿里云 OSS）；
- 建立上传 key / bucket / 路径策略；
- 数据库只保存媒体元数据、状态、存储 key、URL 或签名访问相关信息；
- 不允许继续把媒体主体依赖在应用服务器本地磁盘上。

#### 5.2.2 设计媒体资源模型

`media_assets` 或等价表中建议至少包含：
- `id`
- `owner_user_id`
- `media_type`（image/video/other）
- `storage_provider`
- `storage_key`
- `mime_type`
- `size_bytes`
- `width`
- `height`
- `duration_ms`（视频预留）
- `sha256` 或等价校验字段
- `status`
- `error_code`
- `created_at`
- `updated_at`

#### 5.2.3 建立统一媒体状态机

至少统一：
- `pending`
- `uploading`
- `uploaded`
- `processing`
- `ready`
- `failed`
- `blocked`
- `deleted`

要求：
- 前端、后端、数据库、日志对状态的理解一致；
- 不允许前端自行臆造状态；
- 不允许后端返回模糊无意义状态。

#### 5.2.4 建立上传协议与错误处理口径

必须明确：
- 上传失败如何返回；
- 用户取消如何返回；
- 对象存储失败如何记录；
- 同一文件重复上传如何处理；
- 安全限制（体积、格式）如何表达。

### 5.3 任务边界解释

允许做：
- 对象存储接入；
- 上传 API 建设；
- 新媒体表；
- 状态机实现；
- 文件校验与限制。

禁止做：
- 视频转码重逻辑全量上线；
- 用户前台视频消息正式功能；
- 将媒体存储硬编码为不可替换方案；
- 将对象存储逻辑泄露到 UI 层。

### 5.4 验收标准

必须满足：
- 媒体上传能落到对象存储；
- 媒体元数据与状态能持久化；
- 状态机可以驱动前端后续体验；
- 不需要通过本地磁盘才能走完整链路。

## 6. 4.0C：缓存 / 队列 / 任务处理最小闭环

### 6.1 目标

让后续图片压缩、视频转码、审核、通知、推荐预处理等能力有一个可持续扩展的后台处理底座。

### 6.2 必做任务

#### 6.2.1 引入 Redis 或等价缓存 / 队列基础设施

必须完成：
- 本地开发与目标环境都具备最小可运行链路；
- 支持队列任务派发；
- 支持最小缓存存取；
- 明确 key 命名规则与 TTL 规则。

#### 6.2.2 建立最小 worker 处理链

至少支持：
- 图片压缩 / 处理预留任务；
- 视频任务预留壳子；
- 审核任务预留壳子；
- 通知任务预留壳子。

要求：
- 即使只有 mock 任务，也必须形成真实的 task lifecycle；
- 失败记录、重试口径、状态回写要明确。

#### 6.2.3 缓存对象与边界设计

建议优先缓存：
- 用户基础资料（只读快照层）
- 会话列表元信息
- 未读计数（预留）
- 地理编码结果
- 版本检查结果
- 媒体签名 / 访问控制短期结果

要求：
- 不允许缓存覆盖 canonical truth；
- 缓存只做提速与兜底；
- 关键真值仍以数据库或正式接口为准。

### 6.3 任务边界解释

允许做：
- 引入 Redis；
- 任务队列与 worker；
- 基础缓存封装；
- 简单 mock 任务链路。

禁止做：
- 为了“先有队列”写一套未来注定推翻的杂乱实现；
- 用缓存去覆盖资料与画像真值；
- 把 4.0 做成复杂分布式系统实验场。

### 6.4 验收标准

必须满足：
- 队列与 worker 能真实运行；
- 至少一个媒体相关任务能走完整状态链；
- 缓存层可工作且不破坏真值链；
- 后续 4.2 / 4.4 可以直接在此基础上扩展。

## 7. 4.0D：Flutter 附件组件与上传体验骨架

### 7.1 目标

不是现在就完成图片消息产品，而是先让 Flutter 端具备接入图片/视频消息的 UI 骨架与交互状态机。

### 7.2 必做任务

#### 7.2.1 建立附件消息组件骨架

至少预留：
- 图片附件卡片样式；
- 视频附件占位样式；
- 上传中样式；
- 处理中样式；
- 失败重试样式；
- 禁用 / blocked 样式。

#### 7.2.2 建立上传状态展示组件

必须完成：
- 进度展示（可简化但不可缺失）；
- 上传失败说明；
- 可点击重试；
- 成功后状态切换；
- 若服务端处理中，前端展示清楚等待态。

#### 7.2.3 建立通用错误态与空态组件

必须完成：
- 上传失败统一文案；
- 网络问题统一文案；
- 权限问题统一文案；
- 空白占位与加载占位统一风格。

#### 7.2.4 保持与 3.x UI 风格兼容

要求：
- 不允许突然做成全新风格；
- 仍应保留 EliteSync 当前的产品感；
- 高级时法、画像、设置等现有页面不应被附件组件骨架改乱。

### 7.3 任务边界解释

允许做：
- 组件骨架；
- Mock 数据驱动的附件演示；
- 上传态与失败态组件；
- 会话页中预留插槽。

禁止做：
- 强行在 4.0 做成可商用图片聊天成品；
- 做大量真实用户不可用但工程很重的 UI 复杂动效；
- 为了 demo 好看而绕开状态机。

### 7.4 验收标准

必须满足：
- Flutter 端能体现媒体状态机；
- 后续图片消息接入时不需要重写整个 UI 结构；
- 错误态、失败态、处理态都有统一表现；
- 与当前产品风格不冲突。

## 8. 4.0E：日志、监控、限流与交付文档补齐

### 8.1 目标

让 4.0 不是“代码做完但无法定位问题”的版本，而是具备最小工程可观测性的版本。

### 8.2 必做任务

#### 8.2.1 上传链路与媒体链路日志

必须记录：
- 上传开始；
- 上传完成；
- 上传失败；
- 后台任务开始；
- 后台任务失败；
- 状态切换异常。

#### 8.2.2 API 错误与异常记录

至少覆盖：
- 媒体上传接口；
- 会话接口骨架；
- 媒体读取接口；
- 签名 / 访问控制错误；
- Redis / 队列连接错误。

#### 8.2.3 基础限流与防刷口径

建议至少建立：
- 上传次数限制；
- 单文件体积限制；
- 媒体格式限制；
- 接口基础速率限制。

#### 8.2.4 文档与 handoff 补齐

必须产出：
- 4.0 执行记录；
- 数据骨架说明；
- 媒体状态机说明；
- 队列与缓存接入说明；
- 关键保护面回归记录；
- 发布 / 构建影响说明（如有）。

### 8.3 任务边界解释

允许做：
- 日志；
- 错误码整理；
- 基础限流；
- 文档补齐。

禁止做：
- 只写代码不补文档；
- 把限流写死到以后无法调；
- 为了日志完整而泄露敏感数据。

### 8.4 验收标准

必须满足：
- 出现上传或任务错误时可定位；
- 基本限流可工作；
- 文档可支撑顾问与后续版本理解 4.0 成果；
- 4.0 可以顺利交接给 4.1 / 4.2。

## 9. 本版必须保护的边界（硬性保护面）

以下保护面在 4.0 中默认 **禁止破坏**：

- `POST /api/v1/profile/basic`
- `GET /api/v1/profile/astro/summary`
- `GET /api/v1/profile/astro/chart`
- `GET /api/v1/app/health`
- `GET /api/v1/app/version/check`
- `astro_chart_preferences_v1`
- `user_astro_profiles`
- `app_release_versions`
- `moderation_reports`
- synthetic 批次审计字段
- 地点搜索与地理编码链路
- 权限 / 生命周期 / 路由基础结构
- 3.9 已归档的高级时法边界说明口径

### 9.1 为什么这些边界必须保护

#### 资料与画像真值链不能被 4.0 冲击
4.0 是基础能力基建版，不是资料系统重构版。如果为了媒体、会话、缓存引入新基础设施而误伤资料真值链，将导致整个项目基线被破坏。

#### 本地展示偏好不能污染 canonical truth
3.x 的核心成果之一，就是把本地展示偏好与命盘真值彻底分开。4.0 不得回退这条边界。

#### 版本检查与发布链路不能被非必要改动打乱
4.0 虽不是运营版，但仍应保持现有发布链的基本可用性和一致性。

## 10. 本版不允许出现的错误倾向

为避免 4.0 失控，以下倾向明确禁止：

1. 借“基础设施”名义扩大成大规模重构；
2. 借“媒体接入”名义强行上线完整图片/视频社交功能；
3. 借“缓存提速”名义覆盖真值链；
4. 借“队列预埋”名义引入未来肯定要推翻的临时结构；
5. 借“UI 骨架”名义做脱离状态机的假页面；
6. 借“方便开发”名义把对象存储逻辑泄露到前端；
7. 一边做 4.0 一边偷塞 MBTI、动态流、音视频通话大功能；
8. 为了演示效果破坏 3.x 既有页面与产品调性。

## 16. 本计划的最终定位总结

这份计划书的意义，不是催促团队“快点把所有功能都做出来”，而是：

- 在功能补全之前先把工程底座搭好；
- 把未来一定会遇到的媒体、状态、任务、缓存、队列问题提前处理到合适层面；
- 用最小的结构性改动，为后续产品能力版本提供稳定基线；
- 防止 4.x 一开始就因为范围失控而重走原型式开发的老路。

因此，4.0 的最佳执行策略应当是：

> **克制、清晰、以基础设施为中心地完成最小闭环，而不是一口气做成完整社交产品。**
