## 1. 结论先行

### 1.1 当前进度判断

综合当前进度书与最初 SDD，可以把 EliteSync 的状态判断为：

- **第一阶段（基础架构与核心模块）基本完成**
  - 注册 / 登录 / 问卷 / 资料录入主链路已可用。
- **第二阶段（匹配引擎与即时通讯）基本完成**
  - 每周匹配、匹配解释、双向确认、聊天、首页内容流已跑通。
- **第三阶段（仪式感体验与运营后台）部分完成**
  - 已有“匹配解释 + 玄学画像 + 可视化”能力，但“Drop 仪式感”“盲盒机制”“破冰问题”“运营后台”“内容审核”仍未形成完整闭环。
- **第四阶段（测试优化与上线准备）只完成底座**
  - 已具备备份 / 恢复 / hotfix / 回归意识，但还没有达到可稳定灰度和正式上线的工程化标准。

### 1.2 当前项目的真实阶段

项目已经不再是“从 0 到 1 搭功能”，而是进入了：

> **从“功能可演示”走向“核心体验完整 + 风险可控 + 可灰度发布”的阶段。**

换句话说，接下来最重要的不是继续堆新页面，而是完成三件事：

1. **把 2.6.3a 已有成果稳定下来**，避免 UI 重构再次伤及数据库、定位、接口链路。
2. **补完最初慢约会产品真正缺失的体验层**，尤其是倒计时、揭晓、盲盒、破冰、反馈。
3. **补齐上线前必须具备的运营与安全能力**，包括审核、后台、监控、告警、回滚、压测、灰度。

## 2.2 部分完成但尚未闭环的部分

这些模块不是完全没有，而是**已有局部实现，但尚未形成产品级闭环**。

### A. 慢约会的仪式感体验

最初 SDD 里的以下内容，当前仍属于“待补完”：

- `Drop` 倒计时
- 匹配揭晓动画 / 仪式化出场
- 盲盒式资料解锁机制
- 双方确认前后的信息层级控制
- 破冰问题推荐

### B. 信任与审核体系

目前虽然已有基础资料链路，但严格意义上的“可信准入”还没有完全落地：

- 学信网 / 高校邮箱 / 人工审核的完整认证链路仍需设计成可扩展架构
- 举报、拉黑、申诉、封禁、风控留痕还未形成一体化流程
- 聊天内容审核、头像审核、资料审核尚未形成后台闭环

### C. 运营后台与数据回路

最初方案里强调的运营管理能力仍明显不足：

- 用户列表 / 认证状态 / 人工审核后台
- 内容审核台
- 核心指标看板
- 匹配成功率 / 双向确认率 / 聊天活跃度 / 留存指标
- 问卷完成率与匹配解释点击率

### D. 测试与发布门禁

当前已有 smoke / hotfix / 回滚意识，但仍缺：

- 自动化回归矩阵
- 高风险页面专项验收清单
- 性能基线
- 崩溃监控
- 灰度发布标准
- 发布后观测与回滚阈值

## 2.3 当前最危险的风险点

### 风险一：继续做 UI 优化时误伤关键链路

这是项目已经发生过的真实风险，后续仍是头号风险：

- UI 改动误伤数据库初始化
- UI 改动误伤地图 / 定位 / 权限
- 页面跳转收口时丢失旧行为
- 本地缓存反客为主，覆盖服务端真源

### 风险二：玄学模块越来越完整，但慢约会主体验反而不完整

当前玄学画像部分进展很快，但如果再继续把开发重点压在图表和盘面上，容易出现：

- 展示很强，但匹配揭晓体验很弱
- 页面很多，但转化链路不强
- 可视化很专业，但用户第一次匹配的惊喜感不足

### 风险三：没有后台和指标，后续会失去判断力

没有运营后台和核心指标时，团队会无法回答以下关键问题：

- 新用户卡在哪一步流失？
- 问卷是否过长？
- 匹配解释是否真的提高确认率？
- 玄学页是增强留存，还是只是看起来很酷？
- 哪类 bug 最影响转化？

## 3. 接下来的总体开发原则

## 3.1 总原则

后续开发必须遵循以下六条原则：

1. **先稳定，后扩展。**
2. **先补慢约会核心体验，再继续美化周边页面。**
3. **前端不产生第二真源。**
4. **高风险改动必须先备份、再改动、后验收。**
5. **每个版本必须有明确“可验收的完成定义”。**
6. **多 Agent 协作必须服务于降风险，而不是放大修改面。**

## 3.2 当前阶段不建议优先做的事情

以下事项建议暂缓，不作为最近两到三个版本的重点：

- 继续扩张新的玄学门类
- 大规模数据库 schema 改造
- 支付、会员体系深度商业化
- 视频 / 语音重型互动
- 复杂社交广场 / 内容社区
- 过早追求“功能大全”

原因很简单：你们现在更需要的是**把最小可发布闭环做扎实**。

## 4.1 版本 2.6.4：稳定性与发布门禁收口版

### 目标

把 2.6.3 / 2.6.3a 的视觉与画像成果真正固化下来，建立“以后能安全迭代”的门禁。

### 重点任务

#### A. 关键链路稳定化

- 资料编辑保存链路专项回归
- 出生地搜索 -> 保存 -> 重算 -> summary / chart 回读一致性校验
- 无网 / 超时 / 空数据 / 脏数据场景兜底
- 本地缓存与服务端真源的一致性检查

#### B. 高风险表面保护

把以下模块列入 protected surfaces：

- 数据库初始化 / migration
- 登录态恢复
- 定位权限与地点搜索
- `POST /api/v1/profile/basic`
- `GET /api/v1/profile/astro/summary`
- `GET /api/v1/profile/astro/chart`
- 匹配解释页进入链路
- 聊天入口与房间路由

#### C. 发布门禁建立

- 发布前 smoke checklist 固化
- 回归 checklist 与页面截图基线
- 数据备份 / 恢复 / 回滚演练模板固定化
- 崩溃日志与关键接口日志统一采集
- 发布后 24 小时重点观察指标建立

#### D. 视觉收尾只做“小而确定”的问题

- 页面间间距、字号、边界状态统一
- 玄学页 skeleton / loading / empty / error 态统一
- 设置页的本地偏好和实际展示行为一致

### 交付物

- 2.6.4 回归验收单
- 关键链路 smoke 清单
- protected surfaces 清单
- 发布与回滚 SOP
- 稳定版安装包 / 测试包

### 完成定义

满足以下条件才算 2.6.4 完成：

- 资料保存链路无阻塞级 bug
- 四个主要玄学页无致命渲染问题
- summary / chart 分流无互相拖慢问题
- 回滚演练能在规定步骤内完成
- 至少有一套固定可重复的回归清单

## 4.2 版本 2.7：慢约会核心体验补完版

### 目标

把最初 SDD 里“真正像 Date Drop/慢约会”的部分做出来，让产品从“能匹配”升级为“有仪式感地匹配”。

### 重点任务

#### A. Drop 仪式感系统

- 每周匹配倒计时组件
- 匹配开放时刻状态机
- 揭晓动画 / 结果出场页
- 匹配已解锁 / 未解锁 / 已错过状态定义

#### B. 盲盒式资料展示

建议拆成三层信息权限：

1. **未确认前**：只看匹配亮点、模糊标签、有限画像摘要
2. **单边确认后**：展示更多信息，但仍保留部分神秘感
3. **双边确认后**：开放完整资料与聊天入口

#### C. 破冰问题系统

- 根据问卷答案与匹配解释生成 3-5 个开场问题
- 匹配确认后直接插入聊天页上方
- 支持“换一组问题”与“复制发送”

#### D. 匹配解释页升级

- 把“算法解释”“玄学解释”“生活方式标签”分层展示
- 控制每层的篇幅与置信标签
- 避免解释过长导致首屏负担过重

#### E. 匹配后反馈机制

- 是否愿意继续了解
- 是否觉得解释准确
- 是否愿意再次参与下周匹配
- 负向反馈进入后续匹配过滤因子

### 交付物

- Drop 倒计时与揭晓页
- 盲盒资料解锁链路
- 破冰问题组件
- 匹配解释 2.0 页面
- 匹配后反馈结构

### 完成定义

- 新用户能清楚知道“什么时候出结果”
- 匹配结果出现时有明显仪式感
- 确认前后信息边界清晰
- 聊天不是冷启动，而是自带破冰入口
- 匹配后的正负反馈能被记录

## 4.3 版本 2.8：信任安全与运营后台版

### 目标

建立一套最小可用的“准入、审核、风控、运营”体系，为后续真实测试用户做准备。

### 重点任务

#### A. 认证架构先行

考虑到外部接口资质可能尚未完全齐备，建议采取“两层设计”：

- **接口抽象层先完成**：认证 provider 统一接口
- **真实能力分阶段接入**：先 mock / 手工审核，再逐步切真实学信 / 高校邮箱能力

建议支持：

- 手机号验证码
- 高校邮箱验证占位 / 真接入
- 学籍认证占位 / 真接入
- 人工审核入口

#### B. 举报与风控闭环

- 举报用户
- 拉黑用户
- 资料违规标记
- 聊天违规标记
- 封禁 / 解封 / 申诉状态
- 审核备注与操作日志

#### C. 运营后台 MVP

最低限度先做四块：

1. 用户管理
2. 认证审核
3. 内容审核
4. 数据看板

#### D. 指标体系建立

建议从以下指标开始：

- 注册完成率
- 问卷完成率
- 匹配打开率
- 单边确认率
- 双边确认率
- 首次发言率
- 破冰问题点击率
- 7日留存
- 举报率 / 封禁率

### 交付物

- 后台 MVP
- 举报 / 拉黑 / 封禁 / 申诉流程
- 认证 provider 抽象层
- 审核日志与风控留痕
- 核心运营指标面板

### 完成定义

- 测试用户出现违规行为时可以被处理
- 后台能查看用户、认证、举报、审核状态
- 团队能用数据判断产品卡点，而不是凭感觉讨论

## 6. 建议的团队协作方式（适配 Codex + Gemini + GPT）

## 6.1 角色分工建议

### GPT

负责：

- 版本拆解
- 风险评审
- PRD / 验收标准 / 文档
- 复杂改动前的结构审查

### Codex

负责：

- 具体代码落地
- 重构实施
- 自动化脚本
- 回归清单执行辅助

### Gemini

负责：

- UI / 视觉 / 页面连贯性检查
- 多模态对照验收
- 页面结构遗漏发现

## 6.2 多 subagent 工作流建议

每个版本都走以下固定流程：

1. **需求拆解 agent**：拆成模块、页面、接口、风险点
2. **架构保护 agent**：标记 protected surfaces，不允许误伤
3. **实现 agent**：分块落地代码
4. **回归 agent**：逐项走 checklist
5. **验收 agent**：输出“保住了什么 / 丢了什么 / 未验证什么”

这样做的目的不是增加仪式感，而是**降低 UI 重构误伤关键链路的概率**。

## 7. 每个版本都必须输出的文档

后续建议固定输出以下文档，不再随意发挥：

- `EXEC_PLAN.md`：本次改动计划
- `RISK_REVIEW.md`：风险拆解
- `CHANGE_SCOPE.md`：本次允许改动的表面
- `REGRESSION_CHECKLIST.md`：回归清单
- `ACCEPTANCE_REPORT.md`：验收报告
- `ROLLBACK_PLAN.md`：回滚方案

## 8. 我对项目当前状态的最终判断

EliteSync 目前最可贵的地方，不是已经做了很多页面，而是已经形成了以下三个正确趋势：

1. **慢约会闭环已经存在，不再只是 PPT 产品。**
2. **玄学画像链路已经建立单向真源，后续可以持续升级。**
3. **团队已经意识到回归、备份、恢复、门禁的重要性。**

但它距离“可以放心给真实用户使用”的差距，也同样很明确：

- 仪式感体验还没有完全长出来；
- 运营后台和风控体系还太弱；
- 工程门禁和灰度能力还不够；
- 目前仍容易在 UI 改动中误伤核心链路。

所以，接下来的正确路线不是“再做更多功能”，而是：

> **先把 2.6.4 做稳，再用 2.7 把慢约会灵魂补齐，再用 2.8/2.9 建立真实测试与上线能力。**
