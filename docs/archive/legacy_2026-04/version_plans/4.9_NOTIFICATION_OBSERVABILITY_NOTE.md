# 4.9 通知 / 事件 / 回流可观测性说明

更新时间：2026-04-28

## 目标

确保通知中心不仅展示自然文案，还能在回流失败、已读失败、事件生成失败时被追踪。

## 当前链路

### 通知生成

通知生成由 `NotificationService::createForUser()` 统一入口处理。

当前使用场景包括：

- `MessageController`：新消息通知；
- `StatusPostController`：动态点赞等通知；
- `MatchController`：匹配结果通知；
- `RtcSessionService`：RTC 来电 / 接听 / 拒绝 / 结束通知。

### 通知读取 / 已读

- `GET /api/v1/notifications`
- `GET /api/v1/notifications/unread-count`
- `POST /api/v1/notifications/{notificationId}/read`
- `POST /api/v1/notifications/read-all`

### 通知回流

前端通知中心会根据 `route_name` 与 `route_args` 回流到：

- 聊天页；
- 匹配解释 / 匹配结果页；
- 动态作者页；
- RTC 来电页 / 通话页 / 通话结果页；
- 设置页。

## 现有真值字段

### 后端通知表

`notifications` 表当前已包含：

- `user_id`
- `kind`
- `title`
- `body`
- `payload`
- `read_at`
- `expires_at`

### payload 常见内容

- `route_name`
- `route_args`
- `call_id`
- `call_key`
- `peer_user_id`
- `peer_name`
- `mode`
- `actor_user_id`

## 现有日志 / 事件点

### 通知控制层

- `notification_marked_read`
- `notification_marked_all_read`

### 生成侧上游事件

- 消息发送成功后会写通知；
- 动态点赞 / 匹配成功 / RTC 事件会写通知；
- `NotificationService` 在 5 分钟窗口内做去重。

## 建议的可观测字段

建议通知事件日志至少包含：

- `notification_id`
- `type` / `kind`
- `display_title`
- `display_body`
- `target_type`
- `target_id`
- `user_id` / 脱敏 id
- `read_state_before`
- `read_state_after`
- `route_target`
- `route_result`
- `error_code`
- `source_domain`（message / status / match / rtc）

## 失败模式

### 1. 生成失败

可能原因：

- 上游事件没触发；
- 通知 payload 不合法；
- 服务端异常；
- 去重窗口命中了重复数据。

建议查看：

- 上游 controller / service 日志；
- `notifications` 表是否有新记录；
- `payload` 是否完整。

### 2. 已读失败

可能原因：

- 通知不存在；
- 当前用户不是目标用户；
- 通知已过期或状态异常。

建议查看：

- `notification_marked_read`；
- `notification_marked_all_read`；
- `read_at` 是否变化。

### 3. 回流失败

可能原因：

- `route_name` 为空；
- `route_args` 缺少必要字段；
- 前端 route 映射缺失；
- 通知类型存在但目标页面不可达。

建议查看：

- 前端 `NotificationCenterPage` 路由分发；
- `NotificationItemEntity.routeName` 和 `routeArgs`；
- 通知中心是否把工程 slug 错误暴露给用户。

## 4.9 的最小要求

1. 通知中心展示自然文案，不直接暴露工程 slug；
2. read / read-all 能追踪；
3. 通知回流失败时可以定位 route_name / route_args；
4. RTC / message / status / match 的通知都能被自然打开；
5. 去重逻辑保留，但不要把真实新通知吞掉。

## 不做什么

- 不扩成厂商推送平台；
- 不改通知主链 schema；
- 不改 read / read-all 的行为语义；
- 不让工程 slug 回到普通用户页面。

## 结论

4.9E 的任务是把通知的“生成 - 展示 - 已读 - 回流”做成可追踪链路，而不是重写通知系统。
