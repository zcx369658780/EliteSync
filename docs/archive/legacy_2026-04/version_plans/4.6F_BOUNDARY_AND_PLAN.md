# 4.6F LiveKit 语音接入边界与执行计划

更新时间：2026-04-23

## 定位

4.6F 是 4.6 RTC / 通话基础设施版的语音媒体接入子任务，不是新功能版，不是多人通话，不是直播，不是在线状态平台。

本轮只做一件事：

- 把现有 4.6 的通话状态机接到真实语音媒体层，优先采用 LiveKit 作为现成开源方案

## 为什么要有 4.6F

4.6 之前已经把以下部分跑通：

- 呼叫创建、接听、拒绝、挂断、超时、忙线
- 来电页 / 通话页 / 结果页
- 来电通知与轮询兜底
- 返回消息页的路由收口

但 4.6 仍缺真实语音传输：

- 目前仓库里没有可工作的音频媒体引擎
- 现有 RTC 只是在做状态机与路由，不是真语音

因此 4.6F 的目标是：

- 在不破坏现有 RTC 状态机的前提下，新增真实语音媒体层
- 让双方在同一通话房间里建立音频连接

## 选型

当前建议路线：

- WebRTC 作为底层协议
- LiveKit 作为现成的开源 RTC 媒体栈
- Laravel 后端负责生成 token 与校验参与者
- Flutter 负责连接房间、开启麦克风、接收远端音频

不选的方向：

- 自己从零实现完整 P2P 信令
- 直接扩成多人 / 直播 / 在线状态平台
- 把 4.6F 变成通话产品大版本

## 本轮任务清单

### 后端

- 新增 LiveKit 配置项
- 新增通话房间 token 生成接口
- 复用现有 `rtc_sessions` 作为房间状态真源
- 保证只有会话参与者才能拿到 join token
- 保留现有状态机与通知链

### Flutter

- 新增 LiveKit 客户端服务
- 在通话页 / 来电页接入语音房间连接
- 接听后自动进入房间并开启麦克风
- 发起方在通话建立后自动加入房间
- 通话终态安全离开房间

### 验证

- 验证 token 生成与参与者校验
- 验证 Flutter 可在没有 LiveKit 配置时安全降级
- 验证接入后不误伤消息 / 通知 / 图片 / 视频主链

## 明确不做

- 不做多人通话
- 不做直播
- 不做完整 RTC 平台化
- 不做视频优先版
- 不做在线状态平台
- 不重写 4.0 / 4.2 / 4.4 / 4.5 主链

## 验收口径

4.6F 的最低验收标准：

- 语音通话能真正建立音频连接
- 双方都能加入同一 LiveKit 房间
- 麦克风开启后能传输语音
- 通话终态能正常退出房间
- 现有 RTC 状态机仍然正常
