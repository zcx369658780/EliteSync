## 0. 计划结论

`3.5` 不是普通的 UI 小修版本，也不是一次性完成全部专业占星能力的大一统版本。  
它的定位是：

> **在不破坏当前稳定产品基线的前提下，启动星盘引擎升级第一阶段，先做出可控、可回归、可扩展的 Astro Engine MVP。**

本轮计划前置已经完成以下只读评审：

- `dependency-mapper`
- `risk-reviewer`
- `test-planner`
- `architecture-guardian`

结论一致：

1. 当前稳定基线必须保持不变。
2. `3.5` 的第一刀应该落在引擎抽象、schema 版本化、元信息补充和回归框架上。
3. 不能把 `3.5` 一次性做成完整专业工作台。
4. UI 工作在需要可视化判断时，必须继续让 Gemini-MCP 参与阶段验收。
5. 关键架构边界问题只在必要时再咨询 Claude-mcp。

## 1. 当前基线

当前项目的稳定前提是：

- 对外发布版本：`0.03.02a / 30201`
- 当前已归档阶段：`3.4`
- 当前总交接稿：`docs/HANDOFF_MASTER_CURRENT.md`
- 当前文档主入口：`docs/DOC_INDEX_CURRENT.md`
- 当前版本规划入口：`docs/version_plans/README.md`
- 当前运行手册入口：`docs/runbooks/README.md`
- 历史材料统一归档到：`docs/archive/legacy_2026-04/`

星盘当前稳定链路是：

1. `POST /api/v1/profile/basic`
2. 服务端重算 canonical 画像
3. 保存响应返回更新后的 `astro_profile`
4. Flutter 优先消费重算快照并刷新页面
5. `GET /api/v1/profile/astro/summary`
6. `GET /api/v1/profile/astro/chart`
7. Flutter 使用 `chart_data` 在本地绘制星盘

当前还必须保留的约束：

- 服务端 canonical 真值是唯一真源
- Flutter 本地绘制只负责显示，不改真值
- `astro_chart_preferences_v1` 仅影响展示
- 盘面设置页不能升级成算法真值调参面板

## 2. 3.5 目标

### 2.1 本版一句话定义

`3.5 = Astro Engine MVP / 参数化基础版`

### 2.2 本版要解决的问题

当前 App 已经能：

- 服务端计算真值
- Flutter 本地绘制星盘
- 通过设置页控制部分盘面显示

但还不能：

- 提供接近历史网页原型的参数工作台
- 提供更系统的引擎抽象层与 schema 版本化
- 用结构化元信息明确区分 canonical / derived / display-only
- 用固定样例体系做可重复验证

### 2.3 本版最终应达到的形态

`3.5` 结束后，EliteSync 应达到：

> **在现有稳定产品中新增一个可控的 Astro Engine MVP：具备基础参数面板、引擎抽象层、盘面精细绘制、参数解读最小版，以及可回归的样例对照框架。**

## 3. 3.5 范围

### 3.1 P0

必须完成：

1. Astro Engine 抽象层与 schema 版本化
2. 标准化输入 / 输出元信息
3. 参数面板 MVP
4. 盘面精细绘制 MVP
5. 参数解读最小版
6. 样例对照验证框架
7. 回归、验收、归档

### 3.2 P1

尽量完成，做不完可后移到 3.6：

1. 更多宫位制支持
2. 更多点位 / 敏感点 / 小行星支持
3. 参数面板信息层级优化
4. 参数预设模板
5. 更细的相位显示分组
6. 参数结果摘要页

### 3.3 明确不做

以下内容不进入 `3.5` 主线：

- 古典 / 现代全部能力一次性全做
- 合盘、男女对比盘
- 行运、三限、次限、月返、日返
- 法达、小限、大限、流年
- 大规模解释系统
- 全量宫位制 / 黄道制 / 虚点 / 相位体系一口气开放
- 高风险 canonical 引擎替换

## 4. 执行阶段

## 阶段 1：Engine Adapter 与 Schema 定义

### 目标

先把结构立住，不碰现有稳定页面的核心读取逻辑。

### 任务

1. 为星盘响应补充稳定的 `engine_info` 与 `metadata`
2. 标记 canonical / derived / display-only 字段角色
3. 给 Python astro 服务建立 schema 版本号
4. 保持 `chart_data` 结构兼容
5. 保持 Laravel 保存 / 读取 / 缓存链路不变

### 验收标准

1. 新 schema 可读且可扩展
2. 现有 `chart_data` 读取逻辑不被破坏
3. 文档清楚说明哪些字段属于真值、派生、显示层
4. 后端 feature tests 通过

## 阶段 2：参数面板 MVP

### 目标

做出最小可用的参数工作台 UI。

### 任务

1. 实现参数面板 MVP
2. 支持最小参数切换
3. 参数状态可见
4. 参数修改与盘面刷新联动
5. 参数说明与帮助入口可达

### Gemini-MCP 必须参与

Gemini-MCP 必须至少审查：

1. 参数面板布局
2. 参数分组与信息层级
3. 参数开关、预设、提示的可理解性
4. 移动端与桌面 / 大屏适配思路（若有网页工作台入口）

### 验收标准

1. 参数面板可用
2. 参数切换反馈清楚
3. UI 没有明显拥挤、错位、不可读问题
4. Gemini-MCP 阶段验收通过

## 阶段 3：盘面精细绘制 MVP

### 目标

把当前本地绘制盘面推进到更接近工作台原型的形态。

### 任务

1. 标签避让升级
2. 相位线过滤与层次控制
3. 宫线 / 盘心 / 环带层次优化
4. 参数状态与盘面提示联动
5. 保持本地绘制与服务端真值边界清楚

### Gemini-MCP 必须参与

Gemini-MCP 必须至少审查：

1. 盘面可读性
2. 标签拥挤度
3. 相位线视觉噪音
4. 盘心留白与层次

### 验收标准

1. 新盘面比 `3.4` 更接近目标原型
2. 页面说明清楚写明真值与显示层边界
3. 旧入口未被破坏
4. Gemini-MCP 阶段验收通过

## 阶段 4：参数解读最小版 + 样例对照回归

### 目标

补一层参数理解辅助，并建立基础校验框架。

### 任务

1. 参数解读最小版
2. 固定样例对照
3. 差异报告生成
4. 已知偏差说明
5. 可重复执行的回归脚本 / 检查清单

### 验收标准

1. 参数说明能帮助用户理解当前配置
2. 至少一组典型生日样例可重复验证
3. 差异有记录，不再只凭截图判断
4. 回归材料可归档

## 阶段 5：最终回归、验收与归档

5. 写最终交接稿
