# 4.9 RTC / LiveKit / Heartbeat 可观测性说明

更新时间：2026-04-28

## 目标

把 RTC 从“看起来能连”变成“每一层都能查到证据”：

- 呼叫创建与释放；
- 心跳与超时收口；
- LiveKit join-info 与 room.connect；
- track subscribe / un-subscribe；
- 音频播放与路由；
- 真机 / 模拟器双端差异；
- 失败时能直接定位到 backend / RTC / 媒体 / 播放哪一层。

## 失败模式

### 1. Join-info 拿不到 / 401 / 404

可能原因：

- call 不属于当前用户；
- session 已过期；
- 鉴权失败；
- backend 真值链未建立。

看：

- `rtc_livekit_join_info_requested`
- `GET /api/v1/rtc/calls/{callId}/livekit` 返回值
- `rtc_sessions` 当前状态

### 2. Room.connect 超时或失败

可能原因：

- LiveKit URL / 域名错误；
- 安全组 / UDP / TURN 未通；
- 证书 / 代理 / 协议错配；
- 嵌套 WebRTC 连接未收敛。

看：

- `RTC_LIVEKIT_ROOM_CONNECT_START`
- `RTC_LIVEKIT_ROOM_CONNECT_OK`
- `RTC_LIVEKIT_ROOM_CONNECT_FAIL`
- `RTC_LIVEKIT_PEER_CONNECTION`

### 3. 有 TrackSubscribed 但没有声音

可能原因：

- 本地采集正常，但播放路由失败；
- `startAudio()` / `setSpeakerOn()` 顺序不对；
- 远端帧仍然是静音；
- 设备音频焦点未到位。

看：

- `RTC_LIVEKIT_TRACK_SUBSCRIBED`
- `RTC_LIVEKIT_AUDIO_PLAYBACK_*`
- `RTC_LIVEKIT_REMOTE_AUDIO_STATS`
- `RTC_LIVEKIT_AUDIO_FRAME`
- `RTC_LIVEKIT_AUDIO_ROUTE`

### 4. 有连接但没有帧

可能原因：

- publish 未真正成功；
- 订阅未完成；
- 远端 participant 没有 audio publication；
- 连接停在 checking / connecting。

看：

- `RTC_LIVEKIT_LOCAL_AUDIO_BIND`
- `RTC_LIVEKIT_REMOTE_PARTICIPANT`
- `RTC_LIVEKIT_ROOM_SNAPSHOT`
- `RTC_LIVEKIT_PEER_CONNECTION`

### 5. 心跳超时导致会话被收口

可能原因：

- 前端 heartbeat timer 未启动；
- 页面已离开但会话仍然 active；
- 网络断开；
- 设备后台被杀。

看：

- `rtc_call_heartbeat`
- `heartbeat_timeout`
- `initiator_last_seen_at`
- `peer_last_seen_at`

## 最小可观测性动作

1. 后端要能按 `call_id` 串起 session / events / heartbeats / livekit join-info；
2. Flutter 要能在 `room.connect` 成功后记录连接状态快照；
3. `TrackSubscribed` 出现时必须记录 track id / kind / subscribed；
4. 远端音频帧统计必须区分“没帧”和“静音帧”；
5. 心跳只负责收口，不要被错误限流杀掉正常通话；
6. 通话失败时必须能判断是 backend / signaling / media / playback 哪层。

## 不做什么

- 不扩成多人通话；
- 不重写 RTC 底层；
- 不把 UI 诊断位误当业务流程；
- 不拿截图代替日志；
- 不用 repo 级回滚解决 RTC 问题。

## 结论

4.9F 的目标是把 RTC / LiveKit / Heartbeat 从“能用”变成“可证据化排障”，为 5.0 的真实路径复验和发布门禁打底。
