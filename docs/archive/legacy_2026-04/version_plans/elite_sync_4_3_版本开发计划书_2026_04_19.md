# EliteSync 4.3 版本开发计划书

文件名：`elite_sync_4_3_版本开发计划书_2026_04_19.md` 更新时间：2026-04-19\
适用范围：`4.2` 已冻结后的下一版本承接\
当前承接基线：`4.2 图片消息正式接入版` 已通过并冻结主链；`4.0` 媒体基础设施、`4.1` 非官方四维人格问卷、`4.2` 图片消息主链均视为稳定基线。\
本版定位：`4.3 动态流基础版`

---

## 0. 本版一句话结论

`4.3` 不做“大而全内容平台”，而是把 EliteSync 补到一个真正具备 **表达、被看见、轻互动、可治理** 的基础社交层。

本版只做：

- 最小可用的动态发布链路
- 最小可用的动态流读取链路
- 最小可用的轻互动与治理挂点
- 与现有首页 / 发现 / 个人主页的轻量联动
- 与 `4.0` 媒体底座、`4.2` 图片链路的安全复用

本版明确不做：

- 视频动态
- 重型评论系统
- 复杂推荐算法
- 关注/粉丝体系
- 大而全关系链重构
- 动态平台化、社区化、运营化扩张

换言之，`4.3` 的目标不是把 EliteSync 做成内容社区，而是为后续 `4.4` 视频消息与更完整互动层建立一个收敛、可回归、可验收的动态表达基线。

---

## 1. 版本定位

## 1.1 版本名称

`4.3 动态流基础版`

## 1.2 版本使命

在不破坏 `3.9 / 4.0 / 4.1 / 4.2` 已归档主链的前提下，补齐以下能力：

1. 用户能发动态
2. 用户能看动态
3. 用户能进行轻互动
4. 用户能举报 / 拉黑 / 删除
5. 动态可以复用既有媒体基础设施，但不把媒体平台化
6. 动态入口和资料/匹配/发现形成轻量联动，而不是重写产品结构

## 1.3 为什么现在做 4.3，而不是继续回头补 4.2 或直接跳 4.4

### 原因 A：4.2 已经完成主目标，应冻结主链

`4.2` 的职责是把图片消息正式接入聊天主链，而不是继续扩写成更重的社交产品面。本版若继续在 `4.2` 上补大量表达能力，会把“聊天媒体扩展”误做成“动态流版本”，导致版本边界失焦。

### 原因 B：动态流比视频消息更适合作为下一步承接

`4.4` 的视频消息会更依赖：

- 更清晰的分发入口
- 更稳定的媒体读取与预览习惯
- 更完整的治理边界
- 更明确的用户表达场景

先做 `4.3`，可以先把“表达与被看见”这一层建立起来，再承接更重的视频能力。

### 原因 C：Discover / Feed 骨架已有承接条件

当前项目中 `Discover` 已有 feed / tab / search / cache 骨架，因此 `4.3` 不是从零开始，而是在既有表面上做真正可走通的产品闭环。

## 2. 承接基线

本版必须承接并保护以下基线：

### 2.1 已冻结的真值与主链

- `POST /api/v1/profile/basic` 仍是资料真值入口
- `user_astro_profiles` 仍是画像 / 星盘核心真值表
- `chat_messages` 仍是消息主链真值表
- `app_release_versions` 仍是发布真值表
- `questionnaire_attempts / questionnaire_answers / questionnaire_questions` 仍是问卷真值表

### 2.2 已冻结的能力底座

- `4.0` 媒体 / 存储 / 队列 / 缓存 / worker 最小闭环
- `4.1` 非官方四维人格问卷模块
- `4.2` 图片消息上传、附件绑定、预览、失败重试、摘要回读

### 2.3 已存在的相关模块面

- 首页 `Home`：入口聚合与轻量联动可用
- 发现 `Discover`：feed / tab / search / cache 具备骨架
- 消息 `Chat`：文本 + 图片主链可用
- 我的 `Me / Profile`：资料、设置、人格摘要、星盘入口已归位
- 治理 `Moderation`：举报、拉黑、封禁、审核链路已存在

### 2.4 本版最重要的承接口径

`4.3` 是 **动态流基础版**，不是“新社交主干重构版”。\
所有实现都应遵循：

- additive feature，不做 destructive rewrite
- 先补表达链路，不重写主算法
- 先做轻互动，不做复杂平台化
- 先复用 `4.0 / 4.2` 的媒体能力，不重造上传体系

## 3. 本版做什么 / 不做什么

## 3.1 本版必须完成的核心目标

### 目标 1：动态发布链路闭环

至少支持：

- 纯文本动态
- 文本 + 单图动态
- 发布前预览 / 清除
- 发布成功后的回读确认
- 发布失败后的可见失败态

### 目标 2：动态流读取链路闭环

至少支持：

- 发现页或独立动态流页拉取动态列表
- 作者、时间、正文、图片、基础状态展示
- 刷新与分页 / 续读的最小闭环
- 被拉黑用户 / 被屏蔽内容的过滤

### 目标 3：轻互动链路闭环

至少支持：

- 点赞 / 取消点赞
- 删除自己的动态
- 举报他人动态
- 从动态卡片进入作者资料页

### 目标 4：治理挂点齐全

至少支持：

- 举报入口可见
- 拉黑后列表过滤正确
- 删除只作用于本人动态
- 基础埋点与事件记录可追踪

### 目标 5：与现有页面轻量联动

至少支持：

- 首页或发现页存在明确动态入口
- 作者主页中存在动态入口或最近动态区
- 不改写匹配总分主线
- 不改写人格/星盘主线

## 3.2 本版明确不做的内容

以下内容禁止偷渡进入 `4.3`：

### 内容能力不做

- 视频动态
- 多图相册流
- 长视频内容页
- 话题广场
- 搜索运营平台化
- 热榜、趋势榜、专题页

### 互动能力不做

- 重型评论楼中楼系统
- 私信外的复杂互动体系
- 转发 / 引用 / 收藏全套体系
- 关注 / 粉丝 / 订阅完整图谱
- 社群 / 圈子 / 群组

### 算法能力不做

- 复杂推荐排序
- 个性化内容平台算法
- 将动态行为直接写入匹配 canonical truth
- 借本版重写匹配解释主逻辑

### 媒体能力不做

- 视频上传
- 视频封面生成
- 视频转码
- 工业级分片上传
- 大型相册 / 图库产品

### 架构能力不做

- 微服务重构
- 大规模事件总线改造
- 新建复杂社区后台
- 对 `4.0` 媒体底座做推倒重来

## 4. 硬性保护面

本版开发前必须先冻结以下保护面，不得误伤：

## 4.1 真值链保护面

- `POST /api/v1/profile/basic`
- `GET /api/v1/profile/astro/summary`
- `GET /api/v1/profile/astro/chart`
- `user_astro_profiles`
- `questionnaire_*` 真值链

### 解释

动态流是“表达层”，不是“真值层”。\
任何动态内容、互动行为、点赞记录，都不得反向污染用户画像 canonical truth，也不得把动态行为直接写成匹配算法的真值输入。

## 4.2 消息与媒体保护面

- `GET /api/v1/messages`
- `POST /api/v1/messages`
- `POST /api/v1/media`
- `chat_messages`
- `media_assets`
- `media_processing_jobs`
- `message_attachments`

### 解释

动态图片必须复用既有媒体上传与对象存储链路，但不能为了动态流去改坏聊天图片主链，更不能把 `message_attachments` 误抽象成一套混乱的“万能附件”结构而伤及消息域。

## 4.3 发布与版本保护面

- `GET /api/v1/app/health`
- `GET /api/v1/app/version/check`
- `app_release_versions`
- 宿主 APK 版本口径
- about / version center / release script 绑定链

### 解释

`4.3` 是功能版本，不是发布链重构版本。\
任何新页面、新接口都不能破坏版本检查、版本展示和下载链的一致性。

## 4.4 UI / IA 保护面

必须继续遵守：

- 页面要像产品页，不要像内部调试面板
- 动态页不能做成工程仪表盘
- 失败态可见，但不能比成功态更重
- 图片展示清楚，但不能把动态页做成相册页

## 5. 数据与接口边界建议

## 5.1 数据模型建议

优先复用现有核心表：

- `status_posts`
- `moderation_reports`
- `user_blocks`
- `notifications`
- `app_events`
- `user_relationship_events`
- `media_assets`

如仓库中尚未完整落表，再按最小必要原则新增：

- `status_post_media`：仅用于动态与媒体资源的关联
- `status_post_likes`：仅用于点赞关系
- `status_post_comments`：本版默认不作为必须项；若仓库已有基础实现，可保留接口预留但不做重型展开

### 边界解释

本版不允许为了“未来扩展”一次性新建一整套大型社区数据结构。\
若现有 `status_posts` 已足够承接文本 + 单图动态，则优先在此基础上补齐。只有在缺失必要关联能力时，才允许最小新增关联表。

## 5.2 接口域建议

### 优先策略

- feed 读接口：优先收口到现有 `GET /api/v1/home/feed`、`GET /api/v1/discover/feed`
- post 写接口：建议收口到 `/api/v1/status/*`
- 媒体上传：继续使用 `POST /api/v1/media`
- 举报 / 拉黑：继续挂接现有 moderation 体系

### 建议最小接口集合

- `POST /api/v1/status/posts`
- `GET /api/v1/status/posts/{postId}`
- `DELETE /api/v1/status/posts/{postId}`
- `POST /api/v1/status/posts/{postId}/likes`
- `DELETE /api/v1/status/posts/{postId}/likes`
- `POST /api/v1/status/posts/{postId}/report`
- `GET /api/v1/profile/{userId}/statuses` 或等价接口

### 边界解释

本版不要求一次性把所有 feed、主页、搜索、话题、评论接口都做完。\
核心原则是：

- 写路径清晰
- 读路径可用
- 互动路径最小闭环
- 现有聚合页尽量复用

## 6. 工作包拆分

## 6.1 4.3A：边界冻结与最小数据收口

### 目标

在动代码前，先把 `4.3` 的版本边界、数据边界、读写口径、媒体复用口径写清楚，避免版本越界。

### 必做任务

1. 明确 `status_posts` 在本版中的字段语义与状态口径
2. 明确动态是否允许单图，以及如何绑定 `media_assets`
3. 明确是否新增 `status_post_media`、`status_post_likes`
4. 明确 Discover / Home / Profile 三处入口的职责
5. 明确删除、举报、拉黑的作用域
6. 明确本版不做评论楼中楼、不做视频动态

### 必要验收条件

- 存在一份清晰的边界说明
- 列出新增表 / 新增字段 / 新增接口清单
- 明确哪些旧表和旧接口不可碰
- 明确动态图片复用媒体链路，而非新造上传逻辑

### 解释

没有边界说明就直接进入实现，最容易出现的结果是：

- feed 越做越大
- 聊天媒体被误改
- 首页 / 发现 / 个人主页被同时重构
- 后面验收时无法界定什么算本版完成

因此 `4.3A` 是本版硬前置，不可省略。

---

## 6.2 4.3B：后端最小动态读写闭环

### 目标

建立动态发布、拉取、删除、点赞、举报、过滤的最小后端闭环。

### 必做任务

1. 完成动态发布接口

   - 支持文本动态
   - 支持文本 + 单图动态
   - 媒体引用走既有 `media_assets` 链路

2. 完成动态读取接口

   - feed 列表读取
   - 个人主页动态读取
   - 作者、时间、正文、媒体、互动计数最小字段

3. 完成动态轻互动接口

   - 点赞 / 取消点赞
   - 删除自己的动态
   - 举报他人动态

4. 完成过滤与治理最小逻辑

   - 屏蔽 / 拉黑用户后动态不再出现在正常流中
   - 已删除内容不会继续正常展示
   - 举报事件有记录

5. 完成埋点 / 事件记录

   - 发布动态
   - 点赞动态
   - 删除动态
   - 举报动态
   - 打开作者资料页

### 必要验收条件

- 接口可在本地 / 测试环境稳定返回
- 发布后的动态能在刷新后回读到
- 删除后不再继续以正常可见态出现
- 点赞状态切换一致，不出现前后端状态漂移
- 举报动作可记录，且不误删内容
- 拉黑过滤有效

### 解释

本工作包的目标不是把后端做成内容平台，而是把“可表达、可被看见、可轻互动、可治理”的链路闭合。\
若后端只做发布，不做删除 / 举报 / 过滤，则实际上不具备真实测试能力。

---

## 6.3 4.3C：Flutter 端发布器、动态流与个人主页入口

### 目标

让用户能够以产品化方式进入动态流、发布动态、查看动态、进入作者资料页，并理解当前状态。

### 必做任务

1. 动态流入口与 IA 收口

   - 明确是放 Discover 主入口、二级 tab，还是首页轻入口
   - 不重写首页主结构
   - 不把发现页做成内容社区首页

2. 动态发布器

   - 文本输入
   - 选图 / 清除
   - 发布按钮状态
   - 上传 / 发布中状态
   - 失败提示与重试

3. 动态流列表

   - 动态卡片展示
   - 作者信息、时间、正文、单图、点赞数、操作入口
   - 下拉刷新 / 分页续读最小闭环

4. 动态卡片操作

   - 点赞 / 取消点赞
   - 举报入口
   - 自己的动态可删除
   - 点击作者进入资料页

5. 个人主页动态入口

   - 用户主页可看到动态入口或最近动态
   - 不重写现有资料真值页结构

### 必要验收条件

- 动态入口清晰，不需要工程背景才能理解
- 发布链路中上传中 / 发布中 / 失败 / 成功状态可辨识
- 动态卡片可读，不像调试面板
- 图片显示清楚，但不喧宾夺主
- 删除和举报操作不会误导用户
- 从动态进入作者资料页链路可走通

### 解释

对 `4.3` 而言，前端的关键不是页面数量，而是状态表达是否清楚。\
如果发布器和动态流让人看不懂，哪怕接口已通，也不能算版本成立。

---

## 6.4 4.3D：治理、缓存、性能与回归保护

### 目标

在版本收尾前，把最容易被忽略但实际决定可测试性的部分补齐。

### 必做任务

1. 治理挂点检查

   - 举报路径是否可见
   - 拉黑后过滤是否生效
   - 删除权限是否严格

2. 缓存与性能最小收口

   - feed 列表避免无节制重复拉取
   - 单图动态优先缩略图 / 轻量展示
   - 不让原图直接拖垮列表滚动

3. 回归保护

   - `profile/basic` 真值链回归
   - `astro/summary`、`astro/chart` 回归
   - `messages` 文本 / 图片消息回归
   - `media` 上传回归
   - `questionnaire/history`、`questionnaire/answers` 回归
   - `app/version/check` 回归

4. 日志与错误可见性

   - 发布失败、点赞失败、拉取失败有日志
   - 不允许失败只在控制台里出现

### 必要验收条件

- 弱网 / 失败场景可复现且可解释
- feed 列表不出现明显性能崩坏
- 回归清单完整，旧主链未被误伤
- 动态图片不会反向伤及聊天图片主链

### 解释

`4.3` 若没有缓存、失败态、回归保护，就很容易在看似“功能跑通”的情况下把老能力打坏。\
因此本工作包不是附录，而是正式验收的一部分。

## 6.5 4.3E：截图证据、验收摘要、handoff、closeout

### 目标

把 `4.3` 做成一个可以正式归档、后续可承接 `4.4` 的版本，而不是只留一堆零散截图和口头说明。

### 必做任务

1. walkthough 级截图证据
2. 回归执行记录
3. 验收摘要
4. closeout note
5. advisor handoff / 下一版本承接说明

### 建议截图顺序

1. 发现页 / 动态入口
2. 发布文本动态
3. 发布单图动态
4. 动态流正常展示
5. 点赞 / 取消点赞
6. 举报动态
7. 删除自己的动态
8. 从动态进入作者资料页
9. 拉黑过滤或治理证据
10. 关键旧功能未受伤的回归证据

### 必要验收条件

- 截图不是切片堆砌，而是完整链路
- 文档说明不是空话，能明确本版做了什么 / 没做什么
- 可直接支撑进入 `4.4`，不需要顾问回头猜上下文

### 解释

没有完整证据链，`4.3` 即使代码主链成立，也只能算 `conditional pass`。\
若要正式归档，必须把截图、回归、closeout、handoff 做齐。

## 7. 必要验收条件总表

## 7.1 代码实现层必须满足

1. 用户可发布文本动态
2. 用户可发布文本 + 单图动态
3. 动态流可正常读取和刷新
4. 用户可点赞 / 取消点赞
5. 用户可删除自己的动态
6. 用户可举报他人动态
7. 拉黑 / 过滤有效
8. 个人主页存在动态入口或最近动态区

## 7.2 产品体验层必须满足

1. 入口清晰
2. 发布状态可理解
3. 成功态与失败态都可见
4. 动态卡片不是工程面板风格
5. 图片展示清楚但不把页面做成图库
6. 资料页与发现页没有被重写得面目全非

## 7.3 工程保护层必须满足

1. `4.0` 媒体底座未受伤
2. `4.2` 图片消息未受伤
3. 真值链未受伤
4. 版本检查与发布链未受伤
5. 日志、失败态、回归说明齐全

## 8. 4.3 的验收结论建议口径

## 8.1 何时可以给 `pass`

仅当以下同时成立：

- 动态发布、读取、点赞、删除、举报、过滤全链路成立
- UI 状态表达清楚
- 回归完整
- 截图证据完整
- handoff / closeout 完整
- 无需再补关键收尾材料

## 8.2 何时给 `pass with observations`

用于：

- 主链已经完整成立
- 可以正式归档
- 仍有少量非阻断观察项，例如：
  - 某些视觉细节仍偏系统页
  - 某些文案还可继续降噪
  - 某些卡片密度仍可优化

### 建议

`4.3` 最现实的目标是：**争取做到 **``** 后冻结主链。**\
不要为了追求“绝对完美视觉”而在 `4.3` 上反复开补丁。

## 8.3 何时只能给 `conditional pass`

若出现以下情况之一：

- 动态主链虽跑通，但截图证据包明显不足
- 没有 closeout / handoff
- 治理与回归材料薄弱
- 动态读写能跑，但过滤/删除/失败态说明不清楚

则应判为 `conditional pass`，并仅允许小回合补收尾，不建议重开大开发。

## 9. 风险清单

## 9.1 最大风险：版本越界

表现：

- 动态流做着做着变成社区平台
- 把评论、关系链、推荐、搜索一起做厚
- 把 `4.3` 做成一个半年大版本

控制原则：

- 只做 feed / post / lightweight interaction
- 不把后续版本内容偷渡进来

## 9.2 第二风险：误伤聊天媒体链

表现：

- 为动态图片去改 `media_assets` / `message_attachments` 语义
- 聊天图片上传、预览或失败重试被连带打坏

控制原则：

- 动态图片复用 `4.0 / 4.2` 管线，但不重写消息域

## 9.3 第三风险：UI 工程感过重

表现：

- 动态页像调试台
- 发布器像内部工具页
- 状态提示语义太技术化

控制原则：

- 由 Gemini 重点审查页面可读性与用户感知
- 所有主文案避免 scaffold / debug / internal / demo 等工程语义

## 9.4 第四风险：治理挂点不完整

表现：

- 能发布但不能举报
- 拉黑后仍能看到对方动态
- 删除权限控制不严格

控制原则：

- 把治理挂点列为本版正式验收项，不作为“以后再说”的附录

## 9.5 第五风险：用 demo 内容充当产品完成证据

表现：

- 成功态全靠过于明显的 demo 图片
- 真实状态不够自然

控制原则：

- 成功态样例要尽量接近真实产品感
- 失败态可以保留为排障证据，但不能掩盖成功链路

## 16. 最终建议

本版的最佳策略不是贪大求全，而是：

**把动态表达层做成一个收敛、真实、可被验收、可被承接的基础版本。**

- 不回头重开 `4.2`

- 不破坏真值链与媒体主链

如果本计划被严格执行，那么 `4.3` 完成后，EliteSync 将从“有匹配、有消息”进一步推进到“有表达层、有轻互动层”的可测试产品状态，为后续 `4.4` 视频消息与更完整的关系/通知能力建立健康承接基线。
