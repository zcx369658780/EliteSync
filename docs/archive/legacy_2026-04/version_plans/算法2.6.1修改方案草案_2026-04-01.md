# 算法 2.6.1 修改方案草案

> 目的：在 2.6 的资料输入与备份底盘已基本成型的基础上，把“出生时间 / 出生地”从单纯的资料字段，升级为真正影响八字、紫微、西占的核心输入。
>
> 当前说明：
> - 出生时间在当前仓库里已经接入个人资料页、保存链路、星盘页回显。
> - 出生地也已经可保存、可回显、可搜索。
> - 但现状是：**出生地并未真正进入八字核心计算**，紫微和西占也仍有一部分只是影响精度/置信度，而不是核心排盘结果。
>
> 因此 2.6.1 的目标不是“继续补字段”，而是把字段真正变成算法输入。

---

## 一、2.6.1 的目标

1. 让个人资料页中的出生时间、出生地成为稳定输入。
2. 让保存出生时间 / 出生地后，八字、紫微、西占都能按产品规则重算。
3. 让星盘页和个人资料页都只依赖服务端真值，不依赖旧缓存。
4. 明确“出生地到底影响哪些算法、影响到什么程度”。
5. 补齐回归测试，避免以后又出现“能保存但算法不变”的情况。

---

## 二、当前状态基线

### 2.1 已完成的部分

- 个人资料页已可输入并回显：
  - 生日
  - 出生时间
  - 出生地
  - 经度 / 纬度
- 保存后后端会重算画像。
- 星盘页已改成服务端优先读取。
- 出生地搜索已接入百度 Web 服务 API。

### 2.2 仍需补强的部分

- 八字引擎尚未把出生地真正作为结果因子。
- 紫微引擎里出生地主要还是影响：
  - 置信度
  - 精度说明
  - notes
- 西占引擎仍偏“简化画像”或“退化输入”，还没有完全把出生地 / 经纬度用于核心天文盘计算。
- 前端保存后回显链路仍应继续收紧，避免旧 session / snapshot 覆盖新真值。

---

## 三、修改原则

1. **服务端为唯一真源**
   - 资料保存、画像计算、结果展示都以服务端为准。
   - 前端 session / cache 只能作为 fallback。

2. **出生地要真正进入算法**
   - 不能只影响置信度。
   - 不能只写 notes。
   - 不能只作为展示信息。

3. **先定义规则，再改算法**
   - 不允许直接把出生地“随便加权”。
   - 必须明确：
     - 出生地是否映射时区
     - 是否用于真太阳时修正
     - 是否参与 seed
     - 是否只影响部分派生字段

4. **改动范围最小化**
   - 尽量集中在后端 canonical 层。
   - 前端只做输入、保存、展示、回显。
   - 不新增第二套本地算命逻辑。

---

## 四、算法修改方案

### 4.1 八字：把出生地接入真太阳时 / 地理修正链路

#### 目标
- 让出生地改变时，八字结果有明确的、可解释的变化。

#### 推荐规则
- `birthday + birth_time` 仍是八字基础输入。
- `birth_place / birth_lat / birth_lng` 通过以下任一路径进入八字：
  1. 先换算为时区与经度修正，再算真太阳时
  2. 由真太阳时驱动八字分支

#### 涉及文件
- `services/backend-laravel/app/Services/LunarPhpBaziEngine.php`
- `services/backend-laravel/app/Services/BaziCanonicalService.php`
- `services/backend-laravel/app/Http/Controllers/Api/V1/ProfileController.php`
- `services/backend-laravel/app/Http/Controllers/Api/V1/AstroProfileController.php`

#### 必做项
- 设计一个可复用的“地点 -> 真太阳时”转换服务。
- 让 `birth_lat / birth_lng` 参与八字计算，而不是只影响 confidence。
- 明确当出生地变化但时间不变时，八字哪些字段会变化。

#### 验收标准
- 同一生日 / 出生时间，不同出生地应产生可观察的八字差异。
- `true_solar_time` 需随出生地变化而变化或至少能被解释为修正结果。
- 不能只改 `notes` 或 `confidence`。

---

### 4.2 紫微：让出生时间 + 出生地共同参与排盘规则

#### 目标
- 紫微要从“只看时间的简化画像”升级为“时间 + 地点”的持久命盘画像。

#### 当前问题
- 当前紫微里出生地主要是：
  - `birth_place_present`
  - `confidence`
  - `precision`
  - `notes`
- 还没有真正影响：
  - 命宫
  - 身宫
  - 12 宫位
  - 宫位主题
  - 星曜分布

#### 推荐规则
- `birth_time` 继续是核心。
- `birth_place / birth_lat / birth_lng` 至少要进入以下一项：
  1. 真太阳时修正
  2. 命身宫 seed 修正
  3. 宫位主题偏移

#### 涉及文件
- `services/backend-laravel/app/Services/ZiweiCanonicalService.php`
- `services/backend-laravel/app/Http/Controllers/Api/V1/ProfileController.php`
- `services/backend-laravel/app/Http/Controllers/Api/V1/AstroProfileController.php`
- `services/backend-laravel/app/Services/UserAstroMirrorService.php`

#### 必做项
- 定义“出生地如何影响紫微”的业务规则。
- 如果暂时不能直接影响所有宫位，至少要让：
  - 命宫 / 身宫
  - `major_themes`
  - `summary`
  其中一项随着出生地可观察变化。

#### 验收标准
- 保存出生地后，紫微结果必须有实质变化。
- 不能只有 `confidence` 变，结果完全不变。
- `profile/astro` 里可看到新值。

---

### 4.3 西占：让出生地真正参与上升、宫位、星盘计算

#### 目标
- 出生地要进入西占核心盘面，而不是只作为资料展示。

#### 当前问题
- 目前西占结果仍偏简化或退化输入。
- 出生地主要影响完整度或降级，而不是完整天文盘。

#### 推荐规则
- 使用出生地经纬度影响：
  - 上升点
  - 宫位
  - 必要的天文修正
- 若当前引擎还不能完整实现，则先把“地点对盘面影响”限定到可解释的最小子集。

#### 涉及文件
- `services/backend-laravel/app/Services/WesternNatalEngine.php`
- `services/backend-laravel/app/Services/WesternNatalCanonicalService.php`
- `services/backend-laravel/app/Services/LegacyInputWesternNatalEngine.php`
- `services/backend-laravel/app/Http/Controllers/Api/V1/ProfileController.php`
- `services/backend-laravel/app/Http/Controllers/Api/V1/AstroProfileController.php`

#### 必做项
- 明确当前是“过渡引擎”还是“正式天文引擎”。
- 如果短期不能上完整实现，也要至少让出生地影响：
  - `asc_sign`
  - `true_solar_time`
  - `precision`

#### 验收标准
- 改出生地后，星盘页能看到合理变化。
- 不能只改精度标签，不改盘面结果。

## 五、前端修改方案

### 5.1 个人资料页

#### 目标
- 出生时间、出生地成为完整资料输入项。

#### 涉及文件
- `apps/flutter_elitesync_module/lib/features/profile/presentation/pages/edit_profile_page.dart`
- `apps/flutter_elitesync_module/lib/features/profile/presentation/pages/profile_page.dart`
- `apps/flutter_elitesync_module/lib/features/profile/presentation/pages/astro_profile_page.dart`
- `apps/flutter_elitesync_module/lib/features/profile/presentation/providers/profile_providers.dart`
- `apps/flutter_elitesync_module/lib/shared/providers/session_provider.dart`

#### 必做项
- 保存后优先回显服务端数据。
- 避免旧 snapshot 覆盖新值。
- 出生时间、出生地都要能看到已保存状态。

### 5.2 星盘页

#### 目标
- 星盘页只读服务端画像。

#### 必做项
- 进入页面即拉取服务端数据。
- 不依赖手动刷新按钮。
- 显示出生时间 / 出生地 / 真太阳时 / 置信度 / 精度。

### 5.3 出生地搜索

#### 目标
- 继续只走百度 Web API。
- 搜索结果必须带地址和经纬度。

#### 必做项
- 搜索成功：
  - 显示名称 / 地址 / 经纬度
- 搜索失败：
  - 明确提示服务不可用
- 不允许回退到本地城市列表作为假结果。

### Phase 2.6.1-1
把出生时间在个人资料页彻底固化：
- 保存
- 回显
- session 同步
- profile 读取

### Phase 2.6.1-2
定义“出生地如何影响八字”：
- 真太阳时修正
- 或地理经度修正
- 或其他明确规则

改八字引擎，让出生地真正影响结果。

### Phase 2.6.1-4

### Phase 2.6.1-5
改西占引擎，让出生地影响上升 / 宫位 / 真太阳时。

### Phase 2.6.1-6

## 八、明确不做的事

- 不新增第二套本地排盘逻辑。
- 不把出生地仅当作标签展示。
- 不靠 session / snapshot 伪装最新真值。
- 不把“保存成功”当成“算法已变”的替代。

---

## 九、最终判断

2.6.1 的核心不是再加字段，而是：

- **把出生时间真正固化为个人资料字段**
- **把出生地真正纳入八字 / 紫微 / 西占的计算规则**
- **把前端回显收敛到服务端真源**

如果后续要落地，建议先从八字的真太阳时修正开始，因为这条最容易验证，也最能体现“出生地真的影响结果”。
