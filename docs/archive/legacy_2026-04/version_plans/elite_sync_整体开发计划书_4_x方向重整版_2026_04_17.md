# EliteSync 整体开发计划书（4.x 方向重整版）

更新时间：2026-04-17  
适用范围：3.x 收尾、星盘算法完善、4.x 基础能力补全、后端扩展式改造、后续测试前基线划定  
当前基线：`0.03.09 / 30900`

---

## 0. 本次重整的目的与结论

本计划书用于替代此前以“3.x 收口、治理与发布准备”为主的阶段性思路，转而建立一个更符合当前现实约束的新总纲。

### 0.1 当前现实约束

1. 公司尚未注册，短期内无法进入正式运营阶段。  
2. 暂无真实外部用户，缺少真实反馈样本。  
3. 现阶段最紧缺的不是运营治理能力，而是未来无论如何都会保留的核心产品能力。  
4. 如果在基础社交能力尚未补齐的情况下提前进入测试，后续改动会更大，返工成本更高。  
5. 当前后端仍适合按“可扩展单体”方向演进，不适合现在就做高成本的微服务大重构。

### 0.2 本次计划重整后的总判断

从本版开始，项目总策略调整为：

- **3.x 不再继续扩展大功能面**，以收尾、补齐算法短板、稳定基线、完善证据链与文档为主。  
- **4.x 重新定义为“基础能力补全主线”**，优先补齐未来测试期一定会使用的核心产品功能。  
- **算法升级优先级上调**，但必须与展示层优化、运营治理、发布包装拆开管理。  
- **后端架构不做全面推倒重来**，采用“扩展式改造”路线：保留当前单体主干，同时预埋媒体、缓存、异步任务、实时信令、监控等关键扩展点。  
- **音视频通话属于基础设施级能力**，必须单列版本规划，不与普通页面功能混做。  
- **所有新计划都必须服务于未来测试期可验证、可回归、可扩展，而不是服务于当前并不存在的运营场景。**

### 0.3 本计划书的使用方式

本文件作为后续版本规划、风险评审、任务拆分、Codex 执行与顾问验收的总入口文件。后续每一个具体版本计划书，都应以本文件为母计划进行收敛，不得脱离本文件另起口径。

## 1. 当前产品基线与既有成果（必须保留）

本节用于固化 3.x 已形成的基线，避免后续基础能力补全时误伤已通过的结构。

### 1.1 已完成且应保留的产品主线

#### 1.1.1 账号与治理

已形成：
- `normal / test / synthetic` 三层账号口径；
- synthetic 账号生成、重建、清理、停用链路；
- Admin 治理视图；
- `exclude_from_metrics = true` 的 synthetic 默认排除策略。

计划口径：
- 这部分视为“测试与治理基础设施”，不再作为 4.x 的开发重心，但必须保留；
- 后续仅允许为测试便利性做小修补，不允许反复推翻概念口径。

#### 1.1.2 资料与画像真值链

已形成：
- `POST /api/v1/profile/basic` 保存后触发服务端重算；
- 资料页、画像页、星盘页已统一“服务端真值优先”；
- 前端缓存只能兜底，不能覆盖真值。

计划口径：
- 这是后续所有算法升级与资料功能扩展的底座；
- 4.x 新功能只能在该真值链上扩展，不得绕开该链路私自建立第二真值源。

#### 1.1.3 星盘服务端计算 + Flutter 本地绘制路线

已形成：
- 服务端负责计算与保存；
- Flutter 根据 `chart_data` 本地绘制；
- 星盘设置页只控制本地渲染偏好；
- `astro_chart_preferences_v1` 不写回 canonical truth。

计划口径：
- 当前路线继续保留；
- 星盘算法的完善只能通过“服务端 canonical 输出升级 + 前端 derived 展示升级”完成；
- 不回退到服务端生成最终 SVG 作为主路径。

#### 1.1.4 匹配解释与首聊转化链路

已形成：
- 匹配解释增强；
- 匹配结果到首聊草稿导流；
- 解释页与首聊入口联动；
- 保留旧路径回归门禁。

计划口径：
- 后续即使大规模补功能，这条链路仍应继续保留；
- 解释性与转化性继续是 EliteSync 的差异化核心之一。

#### 1.1.5 设置中心、版本中心、文档与发布脚本

已形成：
- 设置中心产品化；
- 版本中心/关于页可展示版本、服务状态、更新历史；
- 发布脚本与阿里云发布链路已可用；
- 文档索引体系已形成。

计划口径：
- 这些能力从重心上后移，不再作为主版本目标；
- 但必须作为工程底座继续维护，不能失效。

### 1.2 当前应视为保护面的稳定对象

以下内容默认列入保护面，除非专项评审，否则不得随意修改：

- `POST /api/v1/profile/basic`
- `GET /api/v1/profile/astro/summary`
- `GET /api/v1/profile/astro/chart`
- `GET /api/v1/app/health`
- `GET /api/v1/app/version/check`
- `astro_chart_preferences_v1`
- `user_astro_profiles`
- `app_release_versions`
- `moderation_reports`
- synthetic 批次审计字段
- 发布脚本中的版本检查、下载地址、SHA256 口径
- 地点搜索与地理编码链路
- 权限 / 生命周期 / 路由基础结构

### 1.3 当前已知但未彻底关闭的观察项

这些不构成返工原因，但应进入后续版本的修补计划：

- 细粒度解释层的截图证据仍可增强；
- 高级时法页仍有少量工程语义外露；
- 回归材料厚度仍可提升；
- UI / IA 虽已清楚，但仍需继续降噪；
- 版本同步流程仍需保持严谨，不得因功能扩张而破坏发布一致性。

## 2. 总体战略调整：从“运营准备优先”切换到“能力补全优先”

### 2.1 旧方向的问题

此前 3.x 后半段的计划重心更多放在：
- 稳定性；
- 发布链路；
- 治理视图；
- 版本中心；
- 文档与归档；
- 解释层可读性与展示收口。

这套方向在“即将进入运营或 Beta”时是合理的，但在当前阶段存在三类问题：

1. **没有真实用户反馈，运营治理的边际收益有限。**  
2. **基础社交能力未补齐，测试阶段会面临大面积补课。**  
3. **若先围绕治理与包装做得很厚，后续再补媒体、状态、实时能力时反而会对当前结构造成更大扰动。**

### 2.2 新方向的核心原则

从本计划开始，EliteSync 的整体方向调整为：

#### 原则 A：先补齐未来一定会保留的核心能力
优先开发：
- MBTI 人格测试全链路；
- 用户个人状态 / 动态；
- 聊天图片消息；
- 聊天视频消息；
- 媒体上传与缓存；
- 会话体验补全；
- 社交关系链基础能力；
- 后续语音 / 视频通话的基础设施前置。

#### 原则 B：算法升级上调，但必须体系化版本化
- 星盘 / 八字 / 紫微 / pair chart / timing 不再零散修修补补；
- 每一类升级都必须带版本号、证据标签、快照回归、解释模板；
- 不能再把“展示层扩展”与“canonical 算法升级”混成一件事。

#### 原则 C：后端走“可扩展单体”路线
- 当前不做微服务大重构；
- 先把未来一定会卡脖子的环节抽象出来：媒体、缓存、异步任务、实时信令、日志监控；
- 普通业务继续留在单体内快速迭代。

#### 原则 D：把未来测试阶段的修改成本压低
所有计划都要回答一个问题：

> 这个功能如果以后进入测试期，被真实用户频繁使用或提出修改意见，当前实现是否便于修改？是否已经避免了明显的结构性返工？

若答案是否定的，该项设计不得进入稳定主线。

## 3. 版本总路线图（重整后）

本版计划将项目拆为三条长期并行主线：

1. **3.x 收尾与算法补完线**  
2. **4.x 基础能力补全线**  
3. **架构扩展与测试前工程化线**

### 3.1 三条主线的定位

#### 3.1.1 3.x 收尾与算法补完线
目标：
- 不再扩展大功能面；
- 完成星盘算法与高级时法体系中尚未收口的部分；
- 完成解释层证据增强、工程语义降噪、算法版本化基线；
- 为 4.x 提供稳定算法底座。

#### 3.1.2 4.x 基础能力补全线
目标：
- 补齐未来测试阶段必需的核心社交产品功能；
- 建立媒体消息、状态流、关系链、推送、会话体验；
- 将 EliteSync 从“匹配解释型原型”推进为“具备基础社交闭环的可测试产品”。

#### 3.1.3 架构扩展与测试前工程化线
目标：
- 不改变单体主干；
- 逐步补充对象存储、CDN、队列、缓存、日志、监控、限流、信令接口；
- 为未来中高并发和音视频能力预埋工程底座。

### 3.2 阶段顺序原则

推荐顺序：

1. 先完成 3.x 剩余算法与解释层基线；
2. 再做 4.0~4.3 的基础功能补齐；
3. 再进入 4.4 之后的实时与复杂交互能力；
4. 音视频通话不得在媒体消息尚未稳定前提前上线。

## 4. 3.x 收尾线：尚未完成的星盘算法完善与解释体系补完

本节是本次重整新增的关键内容。目标不是继续泛化“玄学功能”，而是把已经纳入系统主线的星盘 / 八字 / 紫微 / timing / pair_chart 能力做成真正可长期维护的算法底座。

### 4.1 当前状态回顾

目前已知现状：
- 服务端负责星盘计算与保存；
- Flutter 本地根据 `chart_data` 绘制；
- 当前依赖 `kerykeion==5.12.7`；
- 数据结构已包含 `chart_data / planets / houses / aspects`；
- 高级时法框架首版、细粒度解释层已完成首轮归档；
- pair_chart / timing / advanced-context 被明确列为后续算法演进线内容。

### 4.2 3.x 尚未完成的星盘算法完善总目标

3.x 剩余算法工作的目标不是继续加页面，而是完成下列五个收口：

1. **输入口径收口**：出生时间、地点、时区、经纬度、缺失值、精度问题统一。  
2. **canonical 输出收口**：星盘输出结构标准化、版本化、可回归。  
3. **解释层证据收口**：所有解释文案均可回溯到结构化证据。  
4. **高级时法边界收口**：timing / advanced-context 从“演示卡片”逐步进入“版本化 derived 层”。  
5. **对拍与快照收口**：算法升级具备 golden cases、快照回归与版本差异说明。

### 4.3 3.x 星盘算法完善的具体任务清单

#### 4.3.1 输入标准化与数据校验

目标：解决“同一用户因输入差异导致盘面不稳定”的问题。

必须完成：
- 出生日期与时间的标准化解析；
- 时区统一存储与换算逻辑；
- 出生地名称到经纬度的解析与缓存；
- 地理编码失败时的 fallback 逻辑；
- 缺失精确出生时间时的处理策略；
- 前后端对出生数据合法性的统一校验；
- 服务端写入 `input_meta` 或等价元数据，用于解释与回归。

建议补充：
- 明确“未知出生时分”时的保守策略；
- 明确未来是否支持夏令时识别与校正；
- 明确地点别名与标准地点名映射表。

#### 4.3.2 canonical 星盘输出 schema 版本化

目标：从“当前能跑通”提升为“以后能稳定演进”。

必须完成：
- 为星盘输出结构建立明确 schema；
- 至少包含：
  - `metadata`
  - `chart_data`
  - `planets`
  - `houses`
  - `aspects`
  - `calculation_version`
  - `engine_version`
  - `source_input_hash`
- 为每个字段定义可空规则、单位、枚举、兼容策略；
- 记录算法版本号，支持历史快照对拍；
- 对旧记录建立兼容读取层，禁止粗暴破坏旧数据。

建议补充：
- 为未来 `points / angles / nodes / optional_objects` 预留兼容字段；
- 为前端绘图提供稳定的 derived adapter，而不是让前端依赖底层 engine 原始字段。

#### 4.3.3 相位、宫位与解释规则收口

目标：让“盘面结果”与“解释文本”之间有明确桥梁。

必须完成：
- 相位规则参数化（例如 orb、主相位/次相位开关等）；
- 宫位与行星位置的结构化输出；
- 关键解释标签的结构化定义；
- 前端显示的解释项必须来自结构化 evidence，而不是散乱字符串。

建议补充：
- 建立 `evidence_tags` 体系；
- 建立解释文案模板层；
- 将“证据 → 中间判断 → 最终文案”的链路拆开。

#### 4.3.4 高级时法 / timing / advanced-context 的版本化过渡

目标：把 3.9 的首版演示框架从“可展示”推进到“可维护”。

必须完成：
- 明确 `summary / entry / timing-association` 的结构定义；
- 明确它们属于 derived-only 层，不直接写回 canonical truth；
- 为 timing 相关输出增加算法版本号与证据来源；
- 为样例矩阵、日志摘要、截图证据建立统一索引格式；
- 明确“预览版”“实验版”“稳定版”的口径边界。

建议补充：
- 为 advanced-context 建立 feature flag；
- 对实验算法结果增加 `confidence` 或 `stability_hint`；
- 明确哪些解释只允许展示，不允许进入匹配总分。

#### 4.3.5 pair_chart / synastry / 关系盘能力的边界重构

目标：把未来恋爱社交核心差异化能力做成可演进模块。

必须完成：
- 明确 `pair_chart` 的输入输出格式；
- 区分个人 natal chart 与双人关系盘的真值边界；
- 为 pair_chart 增加结构化分项与解释标签；
- 明确关系盘结果是否进入总匹配分、以何种权重进入；
- 保留版本对拍与旧结果兼容策略。

建议补充：
- 区分 `compatibility_score`、`relationship_style`、`risk_points`、`communication_points` 等不同维度；
- 避免将关系盘直接写成“最终结论”，而应保留多维标签解释。

#### 4.3.6 八字 / 紫微 与西式星盘的跨模块一致性

目标：避免不同模块各说各话，影响产品可信度。

必须完成：
- 明确各模块的数据来源、计算周期、版本号；
- 统一模块级 evidence 输出格式；
- 统一模块解释模板接口；
- 统一匹配系统读取各模块结果的方式。

建议补充：
- 对不同模块输出的“重叠语义”进行规范化；
- 避免前端为每个算法写完全独立的解释与展示逻辑。

#### 4.3.7 算法回归与快照体系

目标：以后每次升级算法，都能知道自己改坏了什么。

必须完成：
- 建立 golden cases（至少覆盖典型出生时间、地点、模糊输入、跨时区、关系盘样例）；
- 保留算法快照结果；
- 对算法升级前后差异生成 diff 报告；
- 为解释文本建立截图证据与结构证据双重回归；
- 把算法回归纳入版本门禁。

### 4.4 3.x 收尾线建议拆分的小版本

建议将剩余 3.x 工作拆为下列收尾包：

#### 3.9A：证据层与截图回归增强
目标：
- 强化细粒度解释层证据；
- 完成截图证据索引加厚；
- 清理高级时法页工程语义。

#### 3.9B：星盘 canonical schema 与输入口径标准化
目标：
- 完成输入标准化；
- 完成星盘输出 schema 版本化；
- 建立基础 golden cases。

#### 3.9C：pair_chart / timing / advanced-context 版本化基线
目标：
- 完成 derived 层结构化定义；
- 为高级时法与关系盘建立版本边界；
- 不追求一次做到最终效果，但必须做到以后可继续演进。

### 4.5 3.x 收尾线的验收标准

必须全部满足：
- 不新增大功能面；
- canonical truth 未被破坏；
- 星盘输出结构具备版本号；
- 解释层可回溯到结构证据；
- 高级时法页工程术语减少；
- 算法快照与 golden cases 可复现；
- 回归材料厚度明显提升；
- 不影响 `0.03.09` 级别发布链的稳定口径。

## 5. 4.x 主线：基础能力补全与算法升级并重

4.x 不再定义为“治理 / 运营准备版”，而定义为：

> **把 EliteSync 从匹配解释型原型推进为具备基础社交闭环、未来可测试、未来可迭代的核心产品版本线。**

### 5.1 4.x 的总体目标

必须实现以下四个目标：

1. **补齐基础社交能力**：聊天媒体、个人状态、动态流、关系链、通知等。  
2. **完成人格测试全链路**：MBTI 从数据结构升级到完整产品能力。  
3. **搭建媒体与实时能力底座**：为图片、视频、未来音视频通话预埋基础设施。  
4. **让算法升级真正服务于社交转化**：算法不再只是展示，而要更深地作用于匹配、首聊、状态推荐与关系引导。

### 5.2 4.x 的优先级矩阵

## 7. 算法升级在 4.x 中的定位

### 7.1 算法升级不是后置项，而是并行主线

尽管 4.x 以基础能力补全为主，但算法升级不能完全后置。原因是 EliteSync 的差异化并不只是“像普通聊天 App 一样发消息”，而是要把人格、命盘、关系解释、破冰文案、状态引导做出自己的独特价值。

因此 4.x 中的算法升级应遵循：

- 与基础功能并行推进；
- 不破坏 stable baseline；
- 每个升级都必须有版本号和回归；
- 优先服务于匹配、首聊、关系引导和人格表达。

### 7.2 4.x 算法升级的重点方向

#### 方向 A：MBTI 正式进入匹配与表达系统
- 进入个人主页、状态卡片、匹配解释；
- 不作为唯一真相，只作为人格模块之一；
- 与首聊风格建议、互动建议联动。

#### 方向 B：关系盘 / pair_chart 更深地服务社交转化
- 为匹配解释提供结构化 evidence；
- 为聊天破冰生成更具体的互动建议；
- 为关系发展阶段提供不同表达标签。

#### 方向 C：timing / advanced-context 从展示演示进入实验化产品能力
- 以 feature flag 控制；
- 先做实验，不直接写入主匹配分；
- 观察其在首聊与状态建议中的价值。

#### 方向 D：解释模板系统化
- 把散乱文案沉淀为模板体系；
- 支持版本化、A/B 实验、回归对拍；
- 支持不同页面复用统一解释源。

## 8. 后端架构路线：不提前做大厂级重构，但必须做扩展式改造

### 8.1 总体判断

当前后端不应立即进行大规模微服务化重构，因为：

1. 当前尚无真实运营流量；
2. 需求仍在快速演进；
3. 团队当前更需要的是快速补齐产品基础能力；
4. 过早拆分会显著拉高复杂度与维护成本。

但与此同时，当前后端也不能继续按“轻量原型接口集合”方式前进，因为图片、视频、状态流、通知、实时通话都会对后端提出新要求。

因此推荐路线为：

> **保留单体主干，进行领域化拆层与关键扩展点预埋。**

### 8.2 推荐的后端演进模式：可扩展单体

建议将后端逐步整理为以下域：

- `auth`：认证与账号
- `profile`：资料、画像、命盘输入
- `astro`：星盘/八字/紫微/pair_chart/timing 算法域
- `match`：匹配与解释
- `conversation`：会话、消息、附件
- `media`：媒体上传、转码、存储、访问控制
- `status_feed`：状态流 / 动态流
- `relationship`：喜欢、匹配、屏蔽、拉黑
- `notification`：通知与推送
- `moderation`：举报、审核、治理
- `telemetry`：行为埋点与实验
- `release`：版本检查与升级
- `rtc_signal`：未来音视频信令预留

### 8.3 当前阶段必须预埋的六大扩展点

#### 8.3.1 对象存储与 CDN

必须引入：
- 阿里云 OSS 或等价对象存储；
- CDN 分发静态媒体；
- 数据库只存 key、元数据、状态，不直接承担大文件本体。

适用对象：
- 头像；
- 聊天图片；
- 聊天视频；
- 状态图片 / 视频；
- 未来通话截图、封面等。

#### 8.3.2 异步任务队列

必须引入：
- Redis 队列或等价任务系统；
- Worker 处理后台任务。

适用任务：
- 图片压缩；
- 视频转码；
- 封面生成；
- 内容审核；
- 通知分发；
- 推荐预计算；
- 算法快照生成。

#### 8.3.3 缓存层

必须引入：
- Redis 或等价缓存。

优先缓存对象：
- 用户基础资料；
- 匹配候选；
- 解释结果快照；
- 会话列表；
- 未读计数；
- 热门状态流；
- 地理编码结果；
- 版本检查结果。

#### 8.3.4 媒体处理状态机

必须建立统一状态：
- `pending`
- `uploading`
- `uploaded`
- `processing`
- `ready`
- `failed`
- `blocked`
- `deleted`

应用于：
- 图片附件；
- 视频附件；
- 动态媒体；
- 未来通话录制等扩展对象。

#### 8.3.5 日志、监控与限流

必须引入：
- 请求日志；
- 错误日志；
- 慢查询日志；
- 核心接口耗时监控；
- 队列任务失败告警；
- 基础限流与防刷。

#### 8.3.6 实时信令预留

即使 4.0 不做通话，也应提前预留：
- 呼叫会话对象模型；
- 在线状态接口位置；
- 未来 WebSocket / 推送 / RTC 信令抽象层。

### 8.4 当前不建议立刻做的架构动作

暂不建议：
- 全面微服务拆分；
- 提前做复杂分库分表；
- 提前做跨区域多活；
- 提前做超大规模推荐平台；
- 提前做复杂数据湖或离线数仓。

这些动作会在未来需要时再进入专项规划，目前不应干扰产品基本盘搭建。

## 9. “百万到千万用户规模”问题的规划口径

### 9.1 必须先区分三种规模

#### 场景 A：累计注册用户达到百万
这种情况下，只要：
- 数据库设计合理；
- 索引齐全；
- 图片视频走对象存储；
- 热点接口有缓存；
- 异步任务体系可用；

那么一个可扩展单体仍有机会承载。

#### 场景 B：十万级到百万级日活
这种情况下，必须更早补齐：
- 队列；
- 缓存；
- 媒体存储；
- 读写热点拆分；
- 更明确的监控与限流。

#### 场景 C：高并发社交 + 大量实时音视频
这已经不只是“普通后端接口”问题，而是：
- 实时状态系统；
- WebSocket/信令；
- RTC 服务；
- 推送系统；
- 媒体处理管线；
- 强监控与风控；
- 更成熟的扩容策略。

### 9.2 现在的正确做法

现在不应为了“将来可能的千万级规模”而过早推翻当前结构，但必须在 4.x 期间完成以下前置准备：

1. 领域边界清楚；
2. 媒体与业务 API 解耦；
3. 引入缓存与任务队列；
4. 为实时能力预留接口；
5. 为日志、监控、限流建立最小闭环。

### 9.3 真正需要考虑更大架构升级的触发条件

当出现以下任意两项以上时，再启动更高阶架构专项评审：

- 外部测试用户显著增长；
- 图片 / 视频消息已高频使用；
- 推送与状态流进入高频互动；
- 语音 / 视频通话成为日常主路径；
- 数据库热点已明显出现瓶颈；
- 缓存与队列仍无法兜住核心压力；
- 真实并发与存储成本开始显著上升。

- 在未建立回归样本的前提下升级算法；
- 把运营治理页面扩展混成基础能力版本的主目标；
- 一次版本同时做多条高风险链路。

## 14. 统一验收原则

### 14.1 产品验收
- 能形成真实的用户闭环；
- 页面语言能被普通用户理解；
- 复杂能力不会因工程术语而削弱产品感。

### 14.2 工程验收
- 保护面不破坏；
- 异常状态可解释；
- 日志与监控能看到关键问题；
- 关键功能有状态机或显式流程。

### 14.3 算法验收
- 结果有版本号；
- 结果可回溯；
- 升级前后能对拍；
- 解释文案不脱离结构证据。

### 14.4 发布验收
- 版本号、构建号、changelog、about、下载包、`app_release_versions` 一致；
- 发布脚本可复现；
- PostCheck 通过；
- handoff 与文档索引更新完成。

## 16. 最终结论与本计划的长期基线意义

从本次计划重整开始，EliteSync 不再以“当前缺少运营能力”为主要问题，而以“未来一定需要的核心能力尚未补齐”为主要问题。

因此，项目的整体策略正式调整为：

> **3.x 负责把算法与解释体系收口，4.x 负责把基础社交能力补齐，后端以可扩展单体路线持续演进，为未来测试与真实增长预埋工程底座。**

这份计划书的意义不是给出一份短期 TODO，而是明确未来几个阶段都必须遵守的底层口径：

- 什么东西已经形成基线；
- 什么东西必须继续完善；
- 什么能力必须优先补齐；
- 什么架构现在不该做；
- 什么扩展点现在必须提前埋好；
- 什么验收与边界必须反复遵守。

后续所有版本计划书，都应从本文件派生，并与本文件保持一致，不得再重新回到“运营包装优先、基础能力后补”的旧路线。
