# EliteSync 3.3 版本开发计划书

更新时间：2026-04-10  
适用对象：项目负责人 / Codex / Gemini-MCP / Claude-mcp / 后续顾问归档  
版本状态：草案 v1  
当前前置版本：`3.1` 已通过、`3.2` 已终验通过、`3.2a` 已完成补丁收口

---

## 0. 本文档定位

本文档用于在 `3.2` 与 `3.2a` 完成后，给 EliteSync 制定下一阶段的正式开发计划。

`3.3` 的任务不是回头重做已经通过的内容，也不是立刻推进高风险的星盘算法大改，而是：

1. 把 `3.2 / 3.2a` 留下的展示、设置、版本、资料真值链路进一步收口
2. 把“星盘精细化显示”中已经可做的 **显示层** 工作推进到稳定版本
3. 把会影响 canonical 真值的算法升级继续冻结，等待需求材料成熟后单独立项
4. 把产品从“可演示、可验收”继续推进到“更像正式可用 App”

---

## 1. 回顾整体计划谱系

### 1.1 总方向没有变

EliteSync 的总方向仍保持不变：

- 产品主线仍是 **慢约会**
- 核心链路仍是：注册 / 资料 / 问卷 / 匹配 / 首聊 / 持续互动 / 状态广场
- 玄学画像仍是匹配解释与可视化的重要补充，而不是独立替代主产品
- 服务端 canonical 真值仍是唯一真源
- 前端只负责展示、输入、本地偏好和容错，不得抢真值
- 每个版本都必须带 smoke / regression / rollback / 验收材料

### 1.2 3.1 已完成什么

`3.1` 已把项目推进到：

- 最小社交闭环打通
- 账号分层底座打通
- 状态发布 / 广场最小闭环打通
- Admin 控制面落地
- 匹配调试底座落地
- 普通用户可见性与临时数据清理证据补齐

### 1.3 3.2 已完成什么

`3.2` 已把项目推进到：

- 测试账号体系规模化（默认 1000 synthetic，验证过 10000 扩展路径）
- Admin 客观治理视图增强
- 匹配结果 -> 首聊 -> 恢复建议 这一段转化链路补强
- 广场轻分层收口
- 玄学样本证据、synthetic 样本证据、Admin 统计口径证据补齐

### 1.4 3.2a 已完成什么

`3.2a` 是 `3.2` 的显示层补丁版，已完成：

- 星盘主视觉由服务端 SVG 依赖切到 APP 端本地绘制
- 星盘设置页上线（恢复默认、预设档位、显示元素开关）
- 关于页 / 更新历史 / 版本检查同步到 `0.03.02a`
- Android 宿主 API / WS 基线注入补齐
- 星盘精细化显示需求中“先做展示层”的部分已经落地

### 1.5 当前最重要的结论

到 `3.2a` 为止，项目已经不缺“有没有功能”，当前更需要解决的是：

1. 资料 -> canonical -> 本地绘制 / 本地设置 的链路是否完全自洽
2. 星盘显示控制是否真正稳定、易理解、可追溯
3. 设置中心、关于页、隐私与账号安全是否更像正式产品而不是验收壳
4. 版本口径、页面口径、统计口径、配置口径是否彻底统一
5. 高风险算法升级如何继续冻结，避免显示需求反向污染真值层

## 2. 3.3 一句话定义

`3.3 = 资料与画像真值收口 + 星盘显示深化 + 设置与版本中心完善版`

---

## 3. 为什么 3.3 应该这样定义

### 3.1 现在不适合直接做什么

当前不适合直接把版本主线改成：

- 星盘算法重写
- 宫位制 / 黄道制 / 相位容许度 / 虚点的大规模专业化升级
- 高风险 canonical 引擎替换
- 大规模推荐算法重写
- 重直播 / 重内容平台 / 重话题系统

原因很明确：

1. 相关算法资料目前不完整，已明确冻结
2. 3.2a 刚把本地绘制与设置页做起来，应该先稳定显示层与资料链路
3. 继续直接改算法，会把“显示问题”和“真值问题”混在一起，风险过高

### 3.2 现在最值得做什么

当前更值得做的是：

1. 把星盘本地绘制、盘面设置、本地偏好、出生资料编辑之间的关系彻底讲清并做顺
2. 把设置中心、隐私、安全、关于与更新做成正式产品级入口
3. 把版本口径、统计口径、页面说明统一，减少顾问、同事、用户三方的理解歧义
4. 为未来的算法升级预留干净接口，而不是在本版直接硬做

## 4. 3.3 核心目标

### 4.1 P0（必须完成）

1. **资料与画像真值链路收口**
   - 编辑出生资料 -> 服务端重算 -> 本地盘面刷新 形成清晰闭环
   - 明确“哪些是 canonical 真值，哪些是本地显示偏好”

2. **星盘显示深化**
   - 本地绘制盘面继续稳定
   - 盘面设置对实际显示的影响可见、可恢复、可验证
   - 盘面阅读性进一步优化（拥挤、层次、说明、切换体验）

3. **设置中心正式化**
   - 设置首页结构收口
   - 隐私设置、账号安全、通知偏好、性能偏好、盘面设置入口统一整理
   - “关于与更新”从验收页升级为正式产品入口

4. **版本中心与口径统一**
   - 当前版本、更新历史、版本检查、下载信息、release log 对齐
   - 清理页面里残余的不一致口径

5. **3.2 / 3.2a 遗留小问题收尾**
   - Admin 统计显示一致性收尾
   - 关于页 / 更新历史版本口径一致性收尾
   - 星盘主页面与设置页的说明文案收口

### 4.2 P1（本版尽量做，做不完可后移）

1. 星盘阅读辅助层
   - 更多说明提示
   - 元素说明浮层
   - 预设档位文案更易懂

2. 设置与性能偏好
   - 夜间模式
   - 简化模式 / 性能轻量模式
   - 通知与提醒偏好

3. 画像诊断与用户自助理解
   - 让用户更清楚看到：出生资料、真太阳时、经纬度、位置修正、当前图像结果之间的关系

### 4.3 明确暂不纳入 3.3

以下内容明确冻结，不进入 3.3 主线：

1. 星盘 canonical 算法大改
2. 虚点 / 黄道制 / 宫位制 / 相位容许度的专业级参数全面开放
3. 任何会影响服务端真值的高风险引擎替换
4. 大规模推荐算法重写
5. 内容平台化扩张
6. 重直播 / 重社群 / 重打赏逻辑

---

## 5. 3.3 目标版本形态

`3.3` 结束后，EliteSync 应该达到的状态是：

> **用户能更清楚地理解自己的出生资料、画像真值与星盘显示之间的关系；设置中心、关于与更新、隐私与安全入口更像正式 App；星盘显示控制稳定、可恢复、可说明；而高风险算法升级仍被严格隔离。**

---

## 6. 长期约束（3.3 不可违反）

### 6.1 真值约束

1. 服务端 canonical 真值仍是唯一真源
2. 前端不得把本地绘制结果写回真值层
3. 本地盘面设置只影响显示，不改后端画像结果
4. 任何页面说明必须显式区分“真值”和“显示偏好”

### 6.2 玄学约束

1. 八字 / 紫微 / 星盘 / 星座的解释层仍与计算层分离
2. 盘面设置页不得扩展成算法调参面板
3. 任何高风险算法入口必须先进入单独交接文档，不得直接混入产品页

### 6.3 发布与安全约束

1. 版本号、更新历史、版本检查、下载地址必须同步
2. 所有高风险修改要有 smoke / regression / rollback 材料
3. 不得破坏 3.2 已通过能力
4. 不得破坏 3.2a 的本地绘制与设置页基础能力

### 6.4 数据与治理约束

1. synthetic / test / normal 分层保持不变
2. synthetic 默认 `exclude_from_metrics = true`
3. Admin 统计口径必须自解释，不再制造歧义

## 阶段 1：资料与画像真值链路收口

### 目标

让“用户修改出生资料 -> 服务端重算 -> 页面刷新 -> 本地盘面重绘”这条链路更可见、更稳定、更少歧义。

### P0 任务

1. 资料编辑页补清晰说明
2. 修改出生资料后，盘面刷新路径可观察
3. 资料更新失败 / 重算失败时，给出明确反馈
4. 资料页、画像页、盘面页对出生时间 / 地点 / 经纬度的展示口径统一
5. 让用户能理解“为什么改资料会影响盘面”

### 验收标准

1. 修改出生资料后，星盘页能可靠刷新
2. 相关字段展示一致
3. 不出现“显示变了但用户不知道原因”的情况
4. 有前后对比截图与回归记录

---

## 阶段 2：星盘显示深化

### 目标

在不动 canonical 算法的前提下，把星盘显示层做得更稳定、更可读、更易调。

### P0 任务

1. 继续优化本地星盘绘制稳定性
2. 盘面设置页与主盘页联动验证
3. 恢复默认 / 三档预设 / 元素开关做实
4. 优化盘面层次、文字拥挤、信息说明
5. 收掉“SVG 缺失时代”的遗留文案与假设

### P1 任务

1. 更清晰的预设解释
2. 显示项分组更合理
3. 局部说明浮层

### 验收标准

1. 设置变更能真实影响盘面显示
2. 恢复默认可靠
3. 本地绘制不依赖旧 SVG 才能出壳
4. 页面说明能明确区分显示层与真值层
5. 有主盘页 / 设置页 / 前后对比图

---

## 阶段 3：设置中心与关于页正式化

2. 次级页面空态与异常态优化

1. 设置首页像正式 App，而不是调试页
2. 入口层级清楚
3. 当前版本、服务状态、更新历史一致
4. 至少完成设置页、关于页、盘面设置页三页联动截图证据

- 阶段 5：1 - 2 天

### 总体建议

`3.3` 作为一个显示与设置中心收口版，建议以 **1.5 - 2.5 周** 为主要交付周期。  
如果同时加入大量额外显示细化或更完整的隐私/账号安全页面，则可能拉长到 **3 周左右**。

## 14. 3.3 最终通过标准

`3.3` 只有在以下条件同时满足时，才可判定为通过：

1. 资料编辑 -> 服务端重算 -> 页面刷新链路清楚可用

4. 设置中心、关于页、隐私与安全入口结构清楚
5. 当前版本、更新历史、版本检查口径一致
6. Admin 统计显示口径不再误导

8. smoke / regression / rollback 完成
9. 最终交接稿与验收报告齐备
