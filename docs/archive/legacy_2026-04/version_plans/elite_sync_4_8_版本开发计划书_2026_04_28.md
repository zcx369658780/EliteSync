## 0. 一句话结论

4.8 不再扩展大功能，不重开 4.7，也不回滚旧版本。  

本版本目标是：

> 把 4.0–4.7 已完成的媒体、人格、图片消息、动态、视频消息、通知、RTC 语音通话、现代 UI 保护与发布链路，整合成一条可供 5.0 Alpha 测试前复验的真实 smoke 基线。

本版的核心成果不是“新增功能数量”，而是：

1. 双账号真实路径能跑通；
2. Alpha smoke matrix 可重复执行；
3. UI baseline 不再被旧 UI 覆盖；
4. 通知中心工程 slug 降噪；
5. live-call runtime state 补证据；
6. release gate 与版本链一致；
7. 跨层 blocker 处理流程被实际演练并固化；
8. 形成 5.0 前可继承的测试、截图、日志、runbook 与 handoff 材料。

## 1. 版本定位

### 1.1 版本名称

**4.8 Alpha Smoke 与测试前真实路径复验版**

### 1.2 版本类型

本版本属于：

- 测试前稳定化版本；
- Alpha Smoke 版本；
- release gate 复验版本；
- UI baseline regression 版本；
- 4.x 主链整合验收版本。

它不是：

- 新功能扩展版；
- 大 UI 重构版；
- 新推荐算法版；
- 视频通话增强版；
- 运营后台版；
- 付费 / 会员 / 商业化版本；
- 数据库重建或后端大改版。

### 1.3 当前承接基线

默认承接以下已完成或已冻结基线：

- `3.9`：高级时法框架、细粒度解释层、真值链保护已收口；
- `4.0`：媒体 / 存储 / 队列 / 缓存 / worker 基础设施已收口；
- `4.1`：非官方四维人格问卷已收口；
- `4.2`：图片消息主链已收口；
- `4.3`：动态流基础版已收口；
- `4.4`：视频消息版已收口；
- `4.4S`：媒体链稳定性修正版已收口；
- `4.5 / 4.5E`：通知与社交转化增强版已收口；
- `4.6P`：1v1 真语音可听闭环已收口；
- `4.7`：UI protected surfaces、rollback / recovery policy 与测试前质量门禁已收口。

4.8 的第一原则是：

> 不把任何已归档版本重新当成待开发版本，不把 observation 当成重开旧主链的理由。

## 2. 背景与问题来源

### 2.1 为什么需要 4.8

4.0–4.7 已经补齐了主要产品能力：

- 用户资料与画像真值链；
- 星盘 / 人格 / 匹配解释；
- 图片消息；
- 视频消息；
- 动态流；
- 通知中心；
- 1v1 语音通话；
- 现代 UI protected surfaces；
- rollback / recovery policy；
- 跨层 blocker 处理规则。

但这些能力目前仍主要以“版本内验收”形式存在。要进入 5.0 Alpha 测试准备，必须把它们串成一条真实可执行路径：

> 登录 → 资料 → 问卷 → 匹配 → 动态 → 消息 → 图片 → 视频 → 通知 → 语音通话 → 设置 → 版本中心 → 发布链检查。

4.8 就是为了完成这条路径的总复验。

### 2.2 4.8 要解决的核心问题

本版本重点解决以下问题：

1. 4.0–4.7 主链是否能作为一个整体被真实跑通；
2. 通知中心是否仍暴露工程 slug；
3. 语音通话是否具备补充 runtime state 证据；
4. UI 是否在跨层修复后仍保持现代 UI baseline；
5. 版本号、下载包、版本中心、后端版本检查是否一致；
6. 后端 / 数据库 / 云端操作是否严格遵守“本地前端-only、云端后端-only”；
7. 若出现跨层 blocker，是否能按 blocker report → Claude 定点分析 → Codex 最小修复 → 双证据复验执行。

10. 形成截图证据索引、日志证据索引、验收摘要、handoff 与 closeout 文档。

## 5.4 4.8D：通知中心工程 slug 降噪

### 目标

解决 4.7 后保留 observation：通知中心仍可能显示 `rtc_call_invite`、`rtc_call_ended`、`rtc_call_missed` 等工程 slug。

### 范围

仅做用户面文案映射，不重写通知主链。

### 任务

1. 梳理通知类型：
   - `rtc_call_invite`
   - `rtc_call_ended`
   - `rtc_call_missed`
   - `message_new`
   - `status_like`
   - `status_comment`
   - `match_success`
   - 其他现有通知类型
2. 增加用户面展示映射：
   - `rtc_call_invite` → “你收到一通语音来电”
   - `rtc_call_ended` → “语音通话已结束”
   - `rtc_call_missed` → “你有一通未接来电”
   - `message_new` → “你收到一条新消息”
   - `status_like` → “有人赞了你的动态”
   - `match_success` → “你有新的匹配”
3. 工程 type 仅保留在 debug / log / telemetry 中；
4. 对未知类型提供兜底文案：
   - 不显示原始 slug；
   - 展示“你有一条新通知”或同等自然文案。

### 禁止事项

- 不新建平行通知系统；
- 不改通知表结构，除非已有字段完全无法表达；
- 不重写通知 read / read-all 主链；
- 不把通知中心改成营销推送平台。

### 验收标准

- 通知中心主界面不再直接显示工程 slug；
- 通知详情或 debug 环境可保留 type；
- 通知回流路径不受影响；
- 旧通知数据也能通过映射显示自然文案。

## 5.5 4.8E：live-call runtime state 补证据

### 目标

补齐 4.6P / 4.7 保留 observation：独立 live-call runtime state 截图和日志证据。

### 范围

只补证据与最小可读状态，不扩展 RTC 能力。

### 任务

1. 复验 1v1 语音通话链路：
   - 发起；
   - 接听；
   - in_call；
   - heartbeat；
   - hangup；
   - ended；
   - missed / rejected / timeout 兜底态。
2. 截取 live-call runtime state 页面或可视状态：
   - call id 可打码；
   - user id 可打码；
   - 保留状态、时长、连接状态、音频状态等必要证据。
3. 保存关键日志：
   - join-info 返回成功；
   - Room.connect 成功；
   - remote audio track 或音频链路状态；
   - heartbeat 正常；
   - end call 状态正常。
4. 可选：补测模拟器反向发言链。

### 禁止事项

- 不做视频通话；
- 不做多人通话；
- 不做直播；
- 不做在线状态平台；
- 不重写 RTC 状态机；
- 不盲改 Flutter 页面掩盖云端 / LiveKit / 端口问题。

### 验收标准

- 至少有一组 live-call runtime state 截图；
- 至少有一组机器日志证据；
- 语音可听闭环不倒退；
- 如果模拟器反向发言链未复测，必须作为 observation 写入 closeout，而不是阻断本版。

## 8. 跨层 Blocker 处理条款

本版本必然涉及媒体、RTC、通知、版本链和云端配置，因此必须显式执行跨层 blocker 规则。

### 8.1 触发条件

出现以下任一情况，立即进入 blocker 流程：

- 页面显示成功，但真实效果失败；
- 媒体上传成功但播放失败；
- 通话状态机通但没声音；
- join-info 可用但 Room.connect / 音频不通；
- 通知可见但回流目标错误；
- 版本中心显示与后端 version check 不一致；
- Gradle / JDK / wrapper / Flutter AAR 构建异常；
- OSS / public_url / Nginx / LiveKit / 端口 / UDP / TCP / 代理异常；
- 后端日志与 UI 现象不一致。

### 8.2 标准流程

1. 停止盲改；
2. 冻结当前现象；
3. 写 Markdown blocker report；
4. 请 Claude 做根因缩层；
5. Codex 只做最小修复；
6. 用机器证据 + 用户面证据双重复验；
7. 写 fix note / verification note / acceptance summary。

### 8.3 禁止事项

- 禁止 Codex 连续多轮只在 Flutter / UI 层盲修；
- 禁止没有 blocker report 就继续试；
- 禁止用 UI 假成功掩盖后端 / 端口 / 网络问题；
- 禁止根因未定前大范围重构或换库；
- 禁止把跨层问题当成纯前端问题处理。

## 9. Rollback / Recovery Policy

### 9.1 禁止 repo 级回滚

默认禁止：

```bash
git reset --hard <old_commit>
git checkout <old_commit> .
git restore --source=<old_commit> .
git clean -fdx
```

也禁止：

- 用旧 zip 覆盖当前项目；
- 用旧工作树覆盖当前项目；
- 为了修构建或后端 blocker 恢复整仓代码。

### 9.2 允许的恢复方式

只允许：

- path-level restore；
- file-level restore；
- smallest-scope restore。

### 9.3 恢复前必须 checkpoint

恢复前必须执行：

```bash
git status --short
git branch --show-current
git rev-parse HEAD
git diff --stat
git diff --name-only
```

并至少执行一种：

```bash
git switch -c safety/pre-recovery-<date>-<topic>
```

或：

```bash
git add -A
git commit -m "safety: checkpoint before <topic> recovery"
```

或：

```bash
git stash push -u -m "safety: checkpoint before <topic> recovery"
```

### 9.4 涉及 UI protected surfaces 的恢复

如果恢复涉及 UI protected surfaces，必须停止并请用户确认。Codex 不得自行恢复相关文件。

## 16. 最终验收口径

满足以下条件可建议 `pass`：

- release gate 通过；
- 截图 / 日志证据充分；
- handoff / closeout 完整；
- 保护面未受影响；
- 无阻断 observation。

### 16.2 pass with observations

满足以下条件可建议 `pass with observations`：

- 主链可正式归档；
- Alpha smoke 能支撑 5.0 前继续推进；
- 仍有少量非阻断项，例如：
  - 模拟器反向发言链未补测；
  - 个别截图可更完整；
  - 个别通知类型仍需后续文案润色；
  - 某些 runbook 可在 4.9 继续增强；
- observation 已写入 closeout，且不需要重开 4.8 主链。

### 16.3 conditional pass

出现以下情况，只能建议 `conditional pass`：

- 代码基本完成但缺少截图证据；
- smoke matrix 未执行完整；
- release gate 未完成；
- UI baseline regression 未完成；
- live-call runtime state 无证据；
- closeout / handoff 不完整；
- 仍有 blocker 未缩层；
- 无法确认是否使用旧 APK 截图。

### 16.4 fail / 不通过

出现以下情况，不得归档：

- 现代 UI 回退旧版；
- 执行 repo 级回滚导致保护面受损；
- 本地数据库覆盖远端；
- 消息 / 媒体 / 通知 / RTC 主链被重写且未验证；
- 语音通话可听闭环倒退；
- release gate 严重不一致；
- 伪成功 / mock 成功掩盖真实失败。

## 18. 最终执行提醒

4.8 的关键不是“多做一点”，而是“确认已有东西真的能一起工作”。

Codex 执行本版本时，每一次改动都要回答：

1. 这是否服务 Alpha Smoke？
2. 这是否会误伤 protected UI？
3. 这是否会误伤 canonical truth？
4. 这是否会误伤媒体 / 消息 / 通知 / RTC 主链？
5. 这是否需要云端执行而不是本地执行？
6. 如果失败，是否应先 blocker report，而不是继续盲修？

只要答案不清楚，就先停下来写边界说明。4.8 的成功标准，是让 EliteSync 从“版本功能完成”进入“Alpha 前可复验基线”。
