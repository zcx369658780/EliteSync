# EliteSync 4.6P 版本开发计划书

文件名：`elite_sync_4_6P_版本开发计划书_2026_04_24.md`  
更新时间：2026-04-24  
适用范围：`4.6 / 4.6A` 当前均停留在 `conditional pass` 后的下一插入式修正版  
当前承接基线：`4.0` 媒体基础设施、`4.1` 非官方四维人格问卷、`4.2` 图片消息主链、`4.3` 动态流基础版、`4.4` 视频消息版、`4.4S` 媒体链稳定性修正版、`4.5` 通知与社交转化增强版均视为稳定基线；`4.6` 的 1v1 RTC 状态机、来电页、结果页、通知回流壳层已基本成立；`4.6A` 已将 LiveKit 入口层从旧 `:7880` 直连问题推进到可连接 / 可 heartbeat 阶段，但真实语音播放链仍未闭合。  
本版定位：`4.6P 音频播放链定向修正版`

---

## 0. 本版一句话结论

`4.6P` 不是新的 RTC 功能版，也不是继续重查连接层的大排障版，而是一个 **只盯“听到声音”这一 blocker 的定向修正版**。  
它的目标不是继续扩来电页、通话页、多人能力或视频能力，而是把当前已经被收敛出来的阻断点——**真机无音频帧、模拟器有音频帧但无可听声音/无波形、音频播放链与音频路由未闭合**——修到可正式判断的程度。

本版只做：

- 音频播放链与音频路由的定向排障与定向修复
- `TrackPublished / TrackSubscribed / remoteAudioTrack / AudioPlayback` 的真实证据补齐
- 真机与模拟器双向最小复验，明确问题是 publish、subscribe、playback 还是 routing
- 清理与音频页面直接相关的极少量调试暴露项
- 更新 4.6 / 4.6A 的收尾材料，使其具备继续验收或继续下一小修回合的基础

本版明确不做：

- 多人通话
- 视频通话增强
- 在线状态系统
- 外部推送平台
- 新 RTC 库替换
- 通话页大改版
- 连接层重做
- 通知中心重做

换言之，`4.6P` 是一个 **音频播放链定向修正版**，目的是把“能进通话页但没声音”的问题继续缩到单一可执行故障层，而不是再把 RTC 做成一个没有边界的大版本。

## 1. 为什么需要 4.6P

## 1.1 当前首要 blocker 已经从“连接层”下沉到“播放链 / 路由链”

在 `4.6G`、`4.6H`、`4.6A` 连续排障后，已经能够确认：

- 4.6 的来电页、结果页、权限页、状态机壳层不是首要阻断点
- LiveKit 旧的公网 `:7880` 直连入口问题已基本被定位并修过
- 当前 heartbeat 与通话结果页已能成立
- 当前新的主阻断点是：
  - 真机侧没有音频帧或没有进入有效播放态
  - 模拟器侧即使有音频帧，也没有可听声音或没有真实播放证据
  - 历史音频诊断仍停留在“等待远端音轨”或等价未闭合状态

这说明当前版本的首要任务已经不再是继续改连接入口，也不再是继续改壳层 UI，而是：
**把播放链 / 路由链 / 远端音轨绑定链继续收口。**

## 1.2 为什么不能继续把问题笼统叫“RTC 还没通”

因为目前已经有足够证据说明：

- 状态机主链已可运行
- 连接层已不是最早的首要阻断点
- heartbeat 可见
- 通话页面与结果页可达

如果继续把问题笼统归为“RTC 还没通”，会导致 Codex 继续：

- 重复排查连接层
- 盲改 UI
- 反复怀疑信令 / 结果页 / 入口口径

而真正该做的是：
**把问题正式改写成“音频播放链 / 音频路由 / 设备差异定向修复”。**

## 1.3 为什么不直接重开 4.6 大版本

因为 `4.6` 与 `4.6A` 的边界已经很清楚：

- 只做 1v1
- 语音优先
- 不做多人 / 直播 / 在线状态 / 外部推送平台
- 4.6A 只修连接层与复验

当前更合理的做法是：
**冻结 4.6 功能边界与 4.6A 连接层修正版边界，在其后插入一个更小的 4.6P，只修播放链与路由链。**

## 2. 版本定位

## 2.1 版本名称

`4.6P 音频播放链定向修正版`

## 2.2 版本使命

在不破坏 `4.0` 到 `4.5` 已归档主链、且不扩大 `4.6 / 4.6A` 边界的前提下，补齐以下能力：

1. 设备侧能够确认是否真的进入 `TrackPublished / TrackSubscribed` 阶段  
2. 设备侧能够确认 `remoteAudioTrack` 是否真实非空并被正确绑定  
3. 真机与模拟器各自的音频播放链、扬声器路由、播放 attach / play 状态可被明确观测  
4. 若能修通，则至少形成一条“真机 + 模拟器”之间可听声音的最小成功链  
5. 若仍不能修通，则把问题继续明确收窄到 publish、subscribe、playback、routing、真机采样或模拟器限制中的单一子层  
6. 为 4.6 是否继续推进到 `pass with observations` 提供新的决定性依据  

## 2.3 本版不是做什么

本版不是：

- 4.6 第二开发版
- 4.6A 的连接层重修版
- 多人 RTC 版
- 视频通话增强版
- 推送唤醒版
- UI polish 大版本

## 3. 本版做什么 / 不做什么

## 3.1 本版必须完成的核心目标

### 目标 1：真实音频阶段证据补齐

至少支持：

- 明确观察 `TrackPublished`
- 明确观察 `TrackSubscribed`
- 明确观察 `remoteAudioTrack` 是否非空
- 明确观察 `AudioPlayback` 或等价播放链是否进入成功态
- 不再只停留在 heartbeat 或结果页层面

### 目标 2：播放链路闭环

至少支持：

- 远端音轨拿到后，设备侧存在真实播放路径
- 播放路径的 attach / play / sink / route 状态可审计
- 若播放失败，能明确失败点在 attach、playback、route 或 native audio path
- 不把“有音频帧”误写成“用户能听到声音”

### 目标 3：真机 / 模拟器差异收口

至少支持：

- 真机发起 -> 模拟器接听
- 模拟器发起 -> 真机接听
- 两组方向都明确记录：
  - 是否 publish 正常
  - 是否 subscribe 正常
  - 是否播放正常
  - 是否可听声音

### 目标 4：音频路由与设备侧最小兜底

至少支持：

- 扬声器 / 听筒 / 默认输出路由的最小明确口径
- Android 音频焦点、音频模式、音频会话设置的最小排查
- 如果真机和模拟器表现不同，能明确是模拟器限制还是通用问题
- 弱网 / 前后台切换不作为本版主任务，但不能明显干扰音频判断

### 目标 5：附带清理项（仅限直接相关 observation）

仅允许顺手修掉下列与本版直接相关、已知且轻量的观察项：

- 通话相关页面中持续暴露的工程化调试标签，如 `ended` 直出、过多 heartbeat 列表等，可在保留诊断能力前提下适度降噪
- 会话列表中不应出现 token / 密文 / 调试串样式的摘要；若当前与通话联调链直接相关，可顺手改回正常摘要语义

> 说明：这些是 **附带清理项**，不是单独开新版本，也不是重开 4.5 / 4.6 UI 改版。只能在你本轮明确触达相关页面时顺手修，不得扩成视觉大翻修。

## 3.2 本版明确不做的内容

以下内容禁止偷渡进入 `4.6P`：

### RTC 功能面不做

- 多人通话
- 群语音 / 群视频
- 视频通话增强
- 直播 / 房间 / 语聊房
- 通话录音 / 转写 / AI 摘要

### 通知与转化面不做

- 4.5 通知中心大改版
- 外部推送平台化
- 匹配 / 动态 / 首页的新转化链设计

### 架构面不做

- 新 RTC 大库替换
- 微服务重构
- 重做 LiveKit 连接层大逻辑
- 重做数据库 schema

### UI 面不做

- 来电页大改版
- 通话页大翻修
- 首页 / 消息页整体 IA 重构
- 结果页视觉重构

## 4. 硬性保护面

## 4.1 真值链保护面

- `POST /api/v1/profile/basic`
- `GET /api/v1/profile/astro/summary`
- `GET /api/v1/profile/astro/chart`
- `user_astro_profiles`
- `questionnaire_*` 真值链

### 解释

本版修的是音频播放链，不是画像、资料、匹配真值链。  
任何通话行为、播放状态、失败原因都不得回写 canonical truth。

## 4.2 消息与媒体保护面

- `GET /api/v1/messages`
- `POST /api/v1/messages`
- `POST /api/v1/media`
- `chat_messages`
- `media_assets`
- `media_processing_jobs`
- `message_attachments`

### 解释

本版允许继续在通话会话上下文里验证消息摘要与回流，但不允许为了修音频播放链去误伤文本 / 图片 / 视频消息主链。

## 4.3 通知与回流保护面

- `GET /api/v1/notifications`
- `POST /api/v1/notifications/read/{notificationId}`
- `POST /api/v1/notifications/read-all`
- `notifications`
- `notification_center` 前端主链

### 解释

本版只允许保留与通话结果直接相关的最小联动，不得借机重构 4.5 通知中心。

## 4.4 RTC 边界保护面

- `rtc_sessions`
- `rtc_session_events`
- 现有 4.6 状态机页面壳层
- 现有 4.6 权限 / 失败态壳层
- 现有 4.6A 连接层修复口径

### 解释

本版不能借“修播放链”之名，回头重写 4.6 状态机、来电页、结果页、连接层或信令模型。

## 4.5 发布与版本保护面

- `GET /api/v1/app/health`
- `GET /api/v1/app/version/check`
- `app_release_versions`
- 宿主 APK 版本口径
- about / version center / release script 绑定链

### 解释

本版虽然偏音频与设备侧修复，但仍不得破坏宿主版本链一致性。

## 4.6 环境与数据库边界（长期硬规则）

本版必须严格遵守：

- 本地环境只做前端开发、UI 联调、截图采集、文档整理
- 后端开发、数据库迁移、备份、恢复、写库排障统一在阿里云端执行
- 不允许本地开发库、SQLite、临时数据库污染云端正式后端数据

### 解释

本版主要是设备侧与播放链排障，但仍可能触达云端诊断接口与日志，因此更必须守住“本地前端 / 云端后端”边界。

## 5. 数据与接口边界建议

## 5.1 数据模型建议

优先复用现有核心表与配置，不建议新增 schema：

- `rtc_sessions`
- `rtc_session_events`
- `notifications`
- `app_events`
- `chat_messages`
- `conversations`
- `conversation_members`

本版重点不在新增表，而在：

- 设备侧播放链日志
- LiveKit 房间事件日志
- `remoteAudioTrack` 绑定日志
- Android 音频路由 / 播放状态日志
- 双端复验记录

### 边界解释

本版不允许为了“修音频”去新增一堆 RTC 新表。  
若日志和既有事件模型足够承接，就优先保持收敛。

## 5.2 接口域建议

### 优先策略

- 继续复用 `GET /api/v1/rtc/calls/{callId}/livekit`
- 继续复用现有 `POST /api/v1/rtc/calls/*` 通话状态接口
- 仅在必要时补充最小调试日志与播放链状态输出，不新增大批实验性接口

### 建议最小检查集合

- `GET /api/v1/rtc/calls/{callId}/livekit`：确认 join-info 口径未回退
- 现有 `POST /api/v1/rtc/calls/{callId}/connect`：确认呼叫主链仍正常
- 现有 `POST /api/v1/rtc/calls/{callId}/accept/reject/end`：确认状态机未受伤
- 客户端播放链日志：确认不是只有 connect/heartbeat，而是进入真实音频阶段

### 边界解释

本版不要求新增完整调试 API。  
核心原则是：

- 播放链路清楚
- 设备侧状态可审计
- 失败点可下钻
- 不新造脱离主链的调试体系

## 6. 工作包拆分

## 6.1 4.6P-A：边界冻结与音频问题重写

### 目标

先把当前问题从“RTC 还没通”正式改写成“音频播放链 / 路由链 blocker”，防止团队再次回到 UI 或连接层乱试错。

### 必做任务

1. 明确本版只修音频播放链与音频路由  
2. 明确 4.6 / 4.6A 主链不重做  
3. 明确本版不换 RTC 大库、不改 UI 大结构  
4. 明确当前问题优先级顺序：publish / subscribe / remoteAudioTrack / playback / routing  
5. 明确本地前端 / 云端后端边界继续生效  

### 必要验收条件

- 产出一份清晰的边界说明
- 明确 4.6P 与 4.6 / 4.6A 的关系
- 明确本版不是继续重查连接层
- 明确哪些内容不在本版范围内

### 解释

没有边界说明，Codex 很容易继续在 “连通了没”“UI 看起来像不像” 这些旧方向上绕圈。

---

## 6.2 4.6P-B：publish / subscribe 证据补齐

### 目标

把“有没有真正进入音频阶段”这件事做成可证明的事实，而不是靠截图猜测。

### 必做任务

1. 在真机与模拟器两端分别补齐：
   - `TrackPublished`
   - `TrackSubscribed`
   - participant / publication / sid / source / track runtime type 日志  
2. 明确两端各自：
   - local audio publication 是否存在  
   - remote audio publication 是否存在  
   - publication.track 是否非空  
3. 明确到底是哪一端先掉链：
   - publish 端  
   - subscribe 端  
   - 两端都正常，但 playback 不成立  

### 必要验收条件

- 拿到一轮可对齐的双端 publication / subscription 日志
- 不再只凭 heartbeat 判断“好像已进入音频阶段”
- 能明确 publish 与 subscribe 是否都成立

### 解释

如果这一层仍说不清，就不应继续假装已经进入播放层。

---

## 6.3 4.6P-C：`remoteAudioTrack` 绑定与播放链修复

### 目标

在 publish / subscribe 成立的前提下，把远端音轨真正送入设备播放链。

### 必做任务

1. 明确 `remoteAudioTrack` 的赋值来源：
   - `TrackSubscribedEvent`
   - remote participants 扫描
   - 其他 fallback 路径  
2. 明确 `remoteAudioTrack` 赋值后页面是否 rebuild  
3. 明确频谱组件是否拿到真实非空 track  
4. 明确当前播放链是否依赖 SDK 默认行为，还是需要显式 attach / play / sink  
5. 若需要，修复：
   - attach 逻辑  
   - playback 启动逻辑  
   - 播放状态同步逻辑  

### 必要验收条件

- 能证明 `remoteAudioTrack` 是否真实非空
- 能证明页面拿到的 track 与 service 内 track 一致
- 若 playback 成功，至少有一条真机或模拟器可听成功证据
- 若 playback 仍失败，明确失败点不再是 track 绑定层

### 解释

本工作包的目标不是做“更漂亮的频谱”，而是做“真正的声音输出”。

---

## 6.4 4.6P-D：Android 音频路由与设备差异收口

### 目标

把“为什么没有可听声音”继续往 native audio path 与设备路由层收口。

### 必做任务

1. 明确真机与模拟器的音频输出路由差异  
2. 检查并收口：
   - speakerphone route  
   - earpiece / media route  
   - audio focus  
   - audio mode  
   - mute / gain / sink 相关设置  
3. 明确模拟器是否天然不适合作为可靠播放端或可靠发音端  
4. 若真机侧无音频帧、模拟器有帧但无声，分别给出最小解释与最小修复  

### 必要验收条件

- 至少能明确真机与模拟器哪一端问题更大
- 至少能明确问题更偏 publish、playback 还是 route
- 不再笼统地把所有问题叫“没声音”

### 解释

本工作包是 4.6P 的关键，因为你们现在的材料已经说明：连接层和状态机都不是最大 blocker 了。

---

## 6.5 4.6P-E：回归、证据、收尾材料更新

### 目标

让这轮修复不只是“又试了两天”，而是能被正式用于 4.6 系列的下一次验收判断。

### 必做任务

1. 更新音频排障报告  
2. 更新修复说明  
3. 更新双端复验记录  
4. 补新的成功态 / 失败态证据索引  
5. 更新 4.6 / 4.6A 的 acceptance summary / handoff / closeout 口径  
6. 明确本轮修复的是播放链 / 路由链，不是继续扩功能  

### 必要验收条件

- 文档说明与实际复验一致
- 能支撑顾问判断 4.6 是否仍为 conditional pass、是否可升级、或是否还需要更小一轮 native audio 修复

---

## 7. 必要验收条件总表

## 7.1 实现层必须满足

1. 状态机主链未受伤  
2. join-info 与连接层未回退到旧错误  
3. 双端 publication / subscription 证据齐全  
4. `remoteAudioTrack` 绑定链可审计  
5. 至少一条设备侧播放链成功证据，或至少把失败点继续压缩到单一子层  
6. 会话页与结果页不再继续暴露明显调试串  

## 7.2 产品体验层必须满足

1. 本版不继续扩 UI  
2. 通话相关失败态与成功态可被解释  
3. 不再把 heartbeat / ended / 调试状态直接当用户主文案  
4. 会话列表不应暴露 token / 密文 / 调试摘要  

## 7.3 工程保护层必须满足

## 8. 本版验收结论建议口径

## 8.1 何时可以视为 `pass with observations`

仅当以下同时成立：

- 连接层未回退  
- publish / subscribe / playback 链已至少形成一条真实成功链  
- 双端复验完成  
- 剩余问题只是轻微 observation，例如某些页面文案仍偏系统态  

### 建议

`4.6P` 的现实目标不是追求纯 `pass`，而是：  
**把 4.6 从“连接层已修但真实音频未闭合”推进到“音频播放链已至少形成最小成功证据”这一阶段。**

## 8.2 何时仍只能维持 `conditional pass`

若出现以下情况之一：

- 仍无法明确 publish / subscribe / playback 谁先掉链  
- 双端都拿不到决定性日志  
- 仍没有任何一条设备侧可听成功证据  
- 会话页 / 通话页仍大量暴露调试态，且团队还说不清功能真实完成度  

则 `4.6` 仍应维持 `conditional pass`，不得强行归档。

## 9. 风险清单

## 9.1 最大风险：把播放链修复做成“继续乱试 RTC”

控制原则：
- 只修播放链与路由链  
- 不再回到连接层大排障  
- 不换库、不扩 UI  

## 9.2 第二风险：只有 heartbeat，没有真实音频证据

控制原则：
- heartbeat 不得作为真语音已闭合的代替证据  
- 必须补 publication / playback 级证据  

## 9.3 第三风险：模拟器表现误导真机判断

控制原则：
- 必须分开记录真机与模拟器  
- 明确哪一端是 publish 问题、哪一端是播放问题  
- 不把模拟器偶发行为写成通用结论  

## 9.4 第四风险：调试串继续污染用户面

控制原则：
- 只保留必要诊断能力  
- 用户面摘要与主文案必须降噪

## 16. 最终建议

本版的最佳策略不是继续盲改 RTC 页面，而是：

**先用一个插入式修正版把音频播放链与音频路由链修实，再决定 4.6 是否真正进入可归档阶段。**

- 不回到连接层大排障  
- 不跳到新的 RTC 大架构  
- 不继续猜 UI / 波形 / 摘要层  
- 优先确认 publish / subscribe / playback 谁先掉链  
- 目标是把 4.6 从“状态机已成立但仍听不到声音”推进到“至少有一条真实音频成功链，或至少把剩余问题收窄到单一 native audio blocker”这一阶段  

如果本计划被严格执行，那么 `4.6P` 完成后，EliteSync 的 RTC 路线会比现在清晰得多：要么确认真实音频已经闭合并可以继续收口验收；要么明确只剩最后一个更小的 native 播放 / 路由问题，而不是继续在 UI、连接层和状态机之间来回摇摆。
