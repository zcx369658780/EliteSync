# 1. 先给 Codex 的硬性原则

## 1.1 禁止继续做的事

1. **禁止继续手写生产级天文/历法底层**
   - 月亮黄经
   - ASC / MC
   - 宫位
   - 主要相位识别
   - 节气边界
   - 干支换算
2. **禁止继续把 Android 客户端近似星盘当 canonical 数据源**
3. **禁止继续把生肖按公历年份 **``** 直接推断作为主算法**
4. **禁止继续把 3 题轻量 MBTI 当核心排序模块**
5. **禁止在未核验许可证前，把 Swiss 相关栈直接用于闭源商用客户端**

## 1.2 必须改成的事

1. **八字 / 节气 / 干支 / 生肖：优先走成熟中历库**
2. **西占天文计算：优先走 Swiss Ephemeris 体系，或以其为对拍基准**
3. **客户端：只负责采集出生信息、展示结果、展示置信度，不再负责 canonical 盘面计算**
4. **人格模块：MBTI 退到展示层，后台逐步转向 Big Five / IPIP / 沟通风格 / 依恋风格**
5. **所有第三方依赖都必须补“许可证与商用闭源风险说明”**

# 2. 先统一概念，避免 Codex 混淆

## 2.1 “生肖 / 星座 / 星盘”不是一个复杂度层级

### 生肖（属相）

- 来源：**中历年柱地支**或约定明确的历法边界
- 不能简单用公历年份硬算
- 关键难点：立春、春节、年柱切换边界

### 太阳星座

- 来源：公历日期所在黄道十二宫区间
- 适合做轻量展示字段
- 不等于完整星盘

### 完整星盘（Natal Chart）

- 包含：
  - 行星位置
  - ASC / MC
  - 宫位
  - 相位
  - house system
  - 时间与地理位置修正
- 这是高精度天文计算问题，**必须交给成熟引擎**

### 合盘 / 关系盘

- 不应再混成一个总名词
- 至少拆为：
  - `natal_compatibility`：两人基础特征比较
  - `synastry`：两人盘之间的互动关系
  - `composite_like` 或 `composite`：关系气质、关系主题

# 3.1 八字 / 农历 / 节气 / 干支 / 生肖

## A. 生产优先资源

### 1) lunar-java

- 仓库：`https://github.com/6tail/lunar-java`
- 许可证：**MIT**
- 适用语言：Java / Kotlin / JVM
- 能力：
  - 公历 / 农历转换
  - 节气
  - 干支
  - 八字

  - 对干支纪年支持多种边界口径
  - Android / Kotlin / JVM 接入成本低
- 推荐角色：
  - **JVM 服务端 canonical 排盘主实现**
  - 或 **Android/JVM 校验工具**
- 结论：
  - **可直接进入生产候选**

### 2) lunar-python

- 仓库：`https://github.com/6tail/lunar-python`
- 许可证：通常与同系列保持宽松许可，生产前仍需本地再核
- 适用语言：Python
- 能力：
  - 批量排盘
  - 离线对拍
  - 批处理回填
- 推荐角色：
  - **Python BaziEngine 第一版主实现**
  - **离线回填脚本**
  - **JVM 版结果对拍**
- 结论：
  - **可直接进入生产候选**

### 3) lunar-php

- 仓库：`https://github.com/6tail/lunar-php`
- 许可证：**MIT**
- 适用语言：PHP / Laravel

  - **迁移桥接方案**

### 4) lunar-javascript

- 仓库：`https://github.com/6tail/lunar-javascript`
- 许可证：通常与同系列保持宽松许可，生产前仍需本地再核

  - 管理后台校验工具

## C. 高精对拍与跨语言封装候选

### 6) sxtwl / sxtwl\_cpp

- 常见入口：
  - PyPI：`sxtwl`

# 3.2 西方占星 / 天文计算 / 本命盘 / 合盘

## A. 生产核心资源

### 1) Swiss Ephemeris（官方主线）

- 官网：`https://www.astro.com/swisseph/`
- 官方定位：高精度星历底层
- 能力：
  - 行星位置
  - 时间转换
  - ASC / MC
  - 宫位
  - 恒星时

  - **双路线：AGPL 或 Professional License**
  - 闭源商用要非常谨慎
- 重要提醒：
  - 不能因为仓库公开就默认“可闭源商用”

### 2) Swiss Ephemeris 源码镜像 / 适配层仓库

- 常见镜像：`https://github.com/aloistr/swisseph`
- 价值：
  - 看源码
  - 编译
  - 写适配层
- 结论：
  - **源码可得 != 商用闭源可直接上**
  - 仍受 Swiss 官方授权模式约束

### 3) pyswisseph

- 仓库：`https://github.com/astrorigin/pyswisseph`
- 许可证：**AGPL-3.0**
- 定位：Python 对 Swiss Ephemeris 的常用封装
- 推荐角色：
  - Python `WesternEngine` 原型
  - 高精服务端实验
  - 对拍基线
- 结论：
  - **技术上很好用，许可证上必须强提醒**
  - 闭源商用后端若不接受 AGPL 义务，就必须另行解决授权问题

## B. 原型 / 验证 / 解释层实验资源

- 能力：
  - chart 对象组织
  - 相位
  - 宫位
  - 传统占星建模
- 推荐角色：
  - **原型验证**
  - **规则实验**
  - **测试夹具生成**
- 结论：
  - **可以研究，但不应在忽略 Swiss 授权的前提下直接上生产**

### 5) Kerykeion

- 官网：`https://kerykeion.net/`
- 仓库：`https://github.com/g-battaglia/kerykeion`
- 许可证：**AGPL-3.0**
- 能力：
  - Natal
  - Synastry
  - Composite
  - 结构化 chart 数据

  - **快速研究**

  - 不应再说“完全不存在开源关系评分能力”
  - 更准确的说法是：
    - **存在可研究的开源 synastry / relationship scoring 实现**
    - **但没有一个可以直接当你们商业排序主模型的通用权威总评分库**

### 6) Astronomy Engine（可选补充）

- 仓库：`https://github.com/cosinekitty/astronomy`
- 许可证：**MIT**
- 定位：通用天文计算库
- 能力：
  - 太阳 / 月亮 / 行星位置
  - 天象事件预测
  - 多语言支持
- 价值：
  - 许可证宽松

- 局限：
  - 不是一站式占星业务框架
  - 宫位 / 占星解释层仍需自己补
- 结论：
  - **可作为轻量验证与备选底层，不替代 Swiss 在占星业务里的主线地位**

## C. Android 客户端特别说明

### Android 客户端可以做什么

- 采集出生信息
- 做输入校验
- 展示盘面结果
- 展示精度标签 / 置信度
- 做缓存与降级展示
- 在极少数情况下做轻量演示级估算

### Android 客户端不要做什么

- 不要做生产级 canonical 星盘计算
- 不要在客户端直接扩展 Swiss 全链路逻辑作为长期方案
- 不要把旧近似月亮 / ASC / 宫位算法继续扩写成主路线
- 不要把 AGPL / Swiss 许可证不清晰的栈直接塞进闭源商用客户端

### 推荐策略

- **客户端只负责输入与展示**
- **所有高精盘面统一服务端生成**
- **旧客户端近似算法降级为 legacy\_estimate 或异常兜底**
