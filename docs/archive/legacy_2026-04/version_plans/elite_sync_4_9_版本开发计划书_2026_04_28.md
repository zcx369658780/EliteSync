## 0. 一句话结论

4.9 不做新业务功能，不重开 4.8，不扩展 RTC，不重构 UI。  

本版本目标是：

> 在 4.8 Alpha Smoke 已经通过的基础上，把测试前最容易出事故的治理、限流、监控、告警、发布链、云端 runbook、备份恢复演练与通知文案降噪补齐，为 5.0 Alpha 测试候选版建立工程化安全基线。

4.9 的核心成果不是“增加页面”，而是：

1. 通知中心工程 slug 降噪完成；
2. 关键接口限流策略明确；
3. 媒体 / 通知 / RTC / 版本链具备最小可观测性；
4. 队列、媒体处理、RTC、通知失败具备可查证日志；
5. release gate 可复验；
6. 云端后端操作 runbook 明确；
7. 数据库备份与恢复演练有最小流程；
8. 跨层 blocker 处理流程继续固化；
9. 现代 UI protected surfaces 继续不回退；
10. 形成 5.0 Alpha 前的工程化交接材料。

## 1. 版本定位

### 1.1 版本名称

**4.9 测试前治理、限流、监控、发布链强化版**

### 1.2 版本类型

本版本属于：

- 测试前工程化版本；
- Alpha 前风险收敛版本；
- 限流与监控补强版本；
- release gate 强化版本；
- 云端 runbook 与备份恢复演练版本；

## 2. 背景与问题来源

### 2.1 为什么需要 4.9

4.8 已经证明：

- 现代 UI baseline 仍在；
- 4.0–4.7 主链可以进行 Alpha smoke；
- 双端 RTC 真实通话可以跑通；
- 音频帧已经在现场联测中出现；
- 主要页面截图证据足以支持进入下一阶段。

但 4.8 仍保留几个关键 observation：

1. 通知中心仍可见少量工程 slug，例如 `rtc_call_invite`、`rtc_call_accepted`、`rtc_call_ended`；
2. release gate 仍需要在 5.0 前做一次最终复核；
3. 媒体 / 通知 / RTC / 版本链的失败可观测性仍可加强；
4. 云端后端操作、数据库备份恢复、跨层 blocker runbook 需要更适合真实测试阶段；
5. 进入 5.0 Alpha 前，应先完成限流、监控、告警、健康检查和恢复演练的最小闭环。

因此，4.9 的定位不是“再加功能”，而是：

> 把 4.8 证明能跑通的主链，变成测试阶段可监控、可限流、可排障、可恢复、可发布、可交接的工程化基线。

### 2.2 4.9 要解决的核心问题

本版本重点解决以下问题：

1. 用户面通知是否仍暴露工程 slug；
2. 关键接口是否具备基础限流策略；
3. 媒体上传 / 读取失败是否可查；
4. 视频处理 / 图片处理 / 队列失败是否可查；
5. 通知读取 / 已读 / 回流失败是否可查；
6. RTC call create / accept / heartbeat / end / livekit join-info 失败是否可查；
7. release gate 是否能在发布前一键或半自动复核；
8. 数据库备份与恢复是否有测试前最小演练流程；
9. 云端后端操作是否有明确 runbook，避免本地误写库；
10. 若出现跨层 blocker，是否能继续执行 blocker report → Claude 定点分析 → Codex 最小修复 → 双证据复验。

## 5.2 4.9B：通知中心工程 slug 降噪收口

### 目标

解决 4.8 主要 observation：通知中心仍可见 `rtc_call_invite`、`rtc_call_accepted`、`rtc_call_ended` 等工程 slug。

本工作包只做用户面文案映射，不重写通知主链。

### 范围

涉及页面：

- 通知中心列表；
- 通知卡片；
- 可能的通知详情；
- 消息页顶部通知入口若有相关展示，也只做文案映射。

涉及通知类型至少包括：

- `rtc_call_invite`
- `rtc_call_accepted`
- `rtc_call_ended`
- `rtc_call_missed`
- `rtc_call_rejected`
- `message_new`
- `status_like`
- `status_comment`
- `match_success`
- 其他当前已有通知类型。

### 建议用户面文案

| 工程 type | 用户面标题 | 用户面副文案 |
|---|---|---|
| `rtc_call_invite` | 语音来电 | 对方邀请你语音通话 |
| `rtc_call_accepted` | 通话已接听 | 对方已接听通话 |
| `rtc_call_ended` | 通话已结束 | 对方结束了通话 |
| `rtc_call_missed` | 未接来电 | 你有一通未接语音通话 |

| `message_new` | 新消息 | 你收到一条新消息 |
| `status_like` | 动态被喜欢 | 有人赞了你的动态 |
| `status_comment` | 动态有新回复 | 有人回复了你的动态 |
| `match_success` | 新匹配 | 你有新的匹配结果 |
| unknown | 新通知 | 你有一条新的站内提醒 |

### 任务

1. 梳理通知中心当前 type 展示逻辑；
2. 增加 notification type → 用户面标题 / 副文案 / icon / action label 的映射层；
3. 工程 type 只保留在 debug / log / telemetry；
4. 对未知 type 做自然文案兜底；
5. 补截图：通知中心降噪前后对比；
6. 输出：
   - `docs/version_plans/4.9_NOTIFICATION_DENOISE_NOTE.md`
   - `docs/version_plans/4.9_NOTIFICATION_DENOISE_EVIDENCE.md`

- 不改通知表结构，除非已有字段完全无法展示自然文案；
- 不重写 `GET /api/v1/notifications`；
- 不重写 read / read-all；
- 不改变通知回流路由；
- 不把通知中心扩展成营销推送系统；
- 不把工程 type 直接显示给普通用户。

### 验收标准

- 通知中心主界面不再直接显示工程 slug；
- 旧通知数据也能通过映射显示自然文案；
- 通知回流路径不受影响；

## 5.3 4.9C：关键接口限流矩阵

### 目标

建立测试前最小限流策略，防止 Alpha 期间因为重复点击、脚本误触发、弱网重试或异常客户端造成核心接口压力和状态污染。

### 任务

创建：

`docs/version_plans/4.9_RATE_LIMIT_MATRIX.md`

矩阵建议包含以下列：

| 模块 | 接口 | 风险 | 限流对象 | 建议阈值 | 超限响应 | 是否已有实现 | 本轮动作 | 证据 |
|---|---|---|---|---|---|---|---|---|

### 必须覆盖接口

#### 账号 / 登录

- 登录接口；
- 注册或验证码相关接口，如已有；
- 修改密码接口。

#### 资料 / 画像 / 星盘

- `POST /api/v1/profile/basic`
- `POST /api/v1/profile/astro`
- `GET /api/v1/profile/astro/summary`
- `GET /api/v1/profile/astro/chart`

#### 问卷

- `GET /api/v1/questionnaire/questions`
- `POST /api/v1/questionnaire/answers`
- `GET /api/v1/questionnaire/history`

#### 匹配

- `GET /api/v1/match/current`
- `GET /api/v1/match/history`
- `GET /api/v1/match/{targetUserId}/explanation`

#### 消息 / 媒体

- `GET /api/v1/messages`
- `POST /api/v1/messages`
- `POST /api/v1/messages/read/{messageId}`
- `POST /api/v1/media`
- `GET /api/v1/media/{assetId}`

#### 动态 / 治理

- `GET /api/v1/discover/feed`
- `POST /api/v1/status/posts` 或当前等价接口；
- 点赞接口；
- 删除接口；
- 举报接口；
- 拉黑接口。

#### 通知

- `GET /api/v1/notifications`
- `POST /api/v1/notifications/read/{notificationId}`
- `POST /api/v1/notifications/read-all`

#### RTC

- `POST /api/v1/rtc/calls`
- `POST /api/v1/rtc/calls/{callId}/accept`
- `POST /api/v1/rtc/calls/{callId}/reject`
- `POST /api/v1/rtc/calls/{callId}/end`
- `POST /api/v1/rtc/calls/{callId}/heartbeat`
- `GET /api/v1/rtc/calls/{callId}`
- `GET /api/v1/rtc/calls/{callId}/livekit`

#### 发布 / 健康

- `GET /api/v1/app/health`
- `GET /api/v1/app/version/check`

### 建议策略

1. 登录 / 验证码：按账号、IP、设备维度限流；
2. 媒体上传：按用户、文件大小、频率限流；
3. 消息发送：按用户、会话、短时间频率限流；
4. 动态发布：按用户、短时间频率限流；
5. 举报 / 拉黑：按用户行为频率限流，防误伤；
6. 通知 read-all：按用户短时间频率限流；
7. RTC call create：按用户和目标用户维度限流，防止频繁呼叫；
8. heartbeat：只做合理频率校验，不得误杀正常通话；
9. version check / health：允许较高频，但要避免无限刷新打爆。

### 本轮边界

4.9 优先建立矩阵与最小实现验证；已有 limiter 复用，不要求大规模重构。

### 验收标准

- 限流矩阵完整；
- 高风险接口有建议阈值；
- 至少对登录、媒体上传、消息发送、RTC call create、通知 read-all 建立最小验证；
- 超限响应不泄露内部错误；
- 限流不会破坏正常 smoke。

## 5.4 4.9D：媒体 / 队列 / Worker 可观测性

### 目标

让图片、视频、媒体读取、媒体处理任务在失败时可定位，不再只能依赖用户截图判断。

### 覆盖对象

- `media_assets`
- `media_processing_jobs`
- `message_attachments`
- 对象存储 adapter；
- 图片上传；
- 视频上传；
- 视频封面 / 处理任务；
- public_url / playback_url / thumbnail_url；
- worker / queue。

### 任务

1. 梳理媒体链现有日志点；
2. 建立媒体事件日志字段建议：
   - `request_id`
   - `user_id` 或脱敏 id
   - `asset_id`
   - `media_type`
   - `storage_provider`
   - `storage_key`
   - `status_before`
   - `status_after`
   - `error_code`
   - `duration_ms`
   - `job_id`
   - `retry_count`
3. 梳理失败码：
   - 上传失败；
   - 存储失败；
   - public_url 无效；
   - 处理超时；
   - 文件类型不支持；
   - 文件过大；
   - 读取权限失败；
   - 对象不存在；
   - worker 未处理；
   - job failed。
4. 补充或确认最小日志输出；
5. 输出媒体排障 runbook：
   - `docs/runbooks/MEDIA_PIPELINE_TROUBLESHOOTING.md`

### 验收标准

- 发生媒体失败时能定位到 asset / job / error_code；
- 图片 / 视频 smoke 不倒退；
- worker 状态可查；

## 8. 跨层 Blocker 处理条款

本版本涉及通知、媒体、RTC、release gate、云端配置、数据库安全，因此必须继续执行跨层 blocker 规则。

### 8.1 触发条件

出现以下任一情况，立即进入 blocker 流程：

- 页面显示成功，但真实效果失败；
- 媒体上传成功但播放失败；
- worker 显示正常但媒体状态不变；
- 通知显示正常但回流目标错误；
- 通知已读显示成功但后端未更新；
- 通话状态机通但没声音；
- join-info 可用但 Room.connect / 音频不通；
- heartbeat 正常但 call busy 残留；
- version check 与版本中心不一致；
- Gradle / JDK / wrapper / Flutter AAR 构建异常；
- OSS / public_url / Nginx / LiveKit / 端口 / UDP / TCP / 代理异常；
- 后端日志与 UI 现象不一致。

### 8.2 标准流程

1. 停止盲改；
2. 冻结当前现象；
3. 写 Markdown blocker report；

### 8.3 禁止事项

## 9. Rollback / Recovery Policy

### 9.1 禁止 repo 级回滚

默认禁止：

```bash
git reset --hard <old_commit>
git checkout <old_commit> .
git restore --source=<old_commit> .
git clean -fdx
```

也禁止：

- 用旧 zip 覆盖当前项目；
- 用旧工作树覆盖当前项目；
- 为了修构建或后端 blocker 恢复整仓代码；
- 为了恢复文档而覆盖当前 UI。

### 9.2 允许的恢复方式

只允许：

- path-level restore；
- file-level restore；
- smallest-scope restore。

### 9.3 恢复前必须 checkpoint

恢复前必须执行：

```bash
git status --short
git branch --show-current
git rev-parse HEAD
git diff --stat
git diff --name-only
```

并至少执行一种：

```bash
git switch -c safety/pre-recovery-<date>-<topic>
```

或：

```bash
git add -A
git commit -m "safety: checkpoint before <topic> recovery"
```

或：

```bash
git stash push -u -m "safety: checkpoint before <topic> recovery"
```

### 9.4 涉及 UI protected surfaces 的恢复

如果恢复涉及 UI protected surfaces，必须停止并请用户确认。Codex 不得自行恢复相关文件。

- 执行 repo 级回滚导致保护面受损；
- 本地数据库覆盖远端；
- 消息 / 媒体 / 通知 / RTC 主链被重写且未验证；
- 语音通话可听闭环倒退；
- release gate 严重不一致；
- 伪成功 / mock 成功掩盖真实失败；
- 日志泄露敏感密钥或用户密码。

---

## 17. 4.9 完成后对 5.0 的承接建议

如果 4.9 达到 `pass` 或 `pass with observations`，后续建议进入：

> **5.0 Alpha 测试准备版 / 小规模真实用户测试候选版**

5.0 不应再继续堆大功能，而应做：

1. Alpha 测试账号与测试名单；
2. 正式 Alpha 包构建与下载链；
3. 测试说明与反馈收集表；
4. 真实弱网 / 真机 / 双端路径复测；
5. 隐私、协议、退出登录、账号安全最终检查；
6. 版本中心、健康检查、通知、媒体、RTC 的最终 walkthrough；
7. 5.0 Alpha Handoff。

进入 5.0 前必须满足：

- 主链 smoke 完整；
- UI baseline 稳定；
- 远端数据库安全；
- 版本链一致；
- 媒体、通知、RTC 可复验；
- 失败态不伪成功；
- 跨层 blocker 有 runbook；
- 限流、日志、release gate、备份恢复有最小闭环。

## 18. 最终执行提醒

4.9 的关键不是“做更多功能”，而是“让已有功能在测试阶段不容易失控”。

Codex 执行本版本时，每一次改动都要回答：

1. 这是否服务 Alpha 前工程化稳定？
2. 这是否会误伤 protected UI？
3. 这是否会误伤 canonical truth？
4. 这是否会误伤媒体 / 消息 / 通知 / RTC 主链？
5. 这是否需要云端执行而不是本地执行？
6. 这是否会把工程日志或 slug 暴露给普通用户？
7. 如果失败，是否应先 blocker report，而不是继续盲修？

只要答案不清楚，就先停下来写边界说明。4.9 的成功标准，是让 EliteSync 从“Alpha Smoke 可跑通”进入“Alpha 测试前可监控、可限流、可恢复、可发布”的工程化基线。
