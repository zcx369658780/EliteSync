# 0. 一句话结论

5.2 不回头重开 4.x 已归档主链，不重复再做 5.0 的最小产品化起点，也不把 5.1 的关系转化 observation 扩写成新主线。

本版本目标是：

> 在 4.9 稳定门禁基线、5.0 产品化起点、5.1 关系转化增强之上，把 **Me / Profile** 真正推进为“个人经营页 + 表达层 + AI 建议 + 轻语音表达候选位”的产品中枢。

5.2 的核心成果不是再补一页“更长的资料页”，而是：

1. 标签表达体系从零散字段升级成更清晰的自我介绍层；
2. 主页质量分 / 展示建议从点状提示升级成更可执行的个人经营建议；
3. AI 从“查看建议”升级为服务资料展示、状态表达、问候组织的轻量助手；
4. 轻语音表达（声音名片或轻语音自我介绍）形成最小候选位；
5. 个人空间外观层保持轻量、免费、可选，不做商城化与重装扮生态；
6. 所有新增能力继续建立在 profile canonical truth、Match / Chat / Notification / RTC 主链之上，不引入第二真值源或商业中心式页面。

# 1. 版本定位

## 1.1 版本名称

**5.2 个人经营页与表达层增强版**

## 1.2 版本类型

本版本属于：

- 5.x 第三阶段正式版本；
- 个人经营中枢增强版；
- 表达层补强版；
- AI 展示建议增强版；
- 轻语音表达候选版；
- 轻量个人空间外观层版。

它不是：

- 继续做 5.1 的关系回流证据补采集版；
- 商城 / 币体系 / 超级会员装扮版；
- 多人 RTC / 视频通话 / 直播版；
- 重娱乐化玩法版；
- 大推荐算法改造版；
- 微服务重构版；
- 安全深区 / 账号中心重写版；
- 后台运营平台版。

## 1.3 当前承接基线

默认承接以下基线：

- `4.9`：测试前治理、限流、监控、发布链强化版，已冻结为 5.x 稳定门禁基线；
- `5.0`：Alpha 测试候选版 / 产品化补强最小集，已完成 Discover / Chat / Me 的最小产品化起点，并按 `pass with observations` 收口；
- `5.1`：关系转化与内容回流增强版，已按 `pass with observations` 收口；
- 当前对外发布版本基线仍按 `0.04.09 / 40900` 理解，除非 5.2 明确触发新的发包动作；
- 当前 5.x 总路线已明确：先建立产品化起点，再增强关系推进，随后补个人经营页与表达层。

5.2 的第一原则是：

> 只从 4.9 + 5.0 + 5.1 additive baseline 承接，不以“个人经营页增强”为由重写已通过的 Match / Chat / Notification / RTC 主链。

# 2. 背景与问题来源

## 2.1 为什么需要 5.2

5.0 已经把 Discover / Chat / Me 推进到“可见的产品起点”，5.1 又把关系推进、内容回流、通知回流、语音节奏做得更自然。当前剩下的明显缺口是：

> 用户已经能看内容、能聊天、能从通知回流，但“为什么别人愿意继续看我、回我、继续和我聊”这一层仍然不够强。

目前 Me 页虽然已具备：
- 基础资料区；
- 资料完整度；
- 同步提示；
- AI 展示建议入口；
- 内容标签区；

但它仍更像“个人经营中枢的起点壳层”，还没有完成真正的表达层组织与经营逻辑。

## 2.2 5.2 要解决的核心问题

本版本重点解决以下问题：

1. 标签表达还不够体系化，仍偏字段堆叠；
2. AI 展示建议还偏入口级，不够可执行；
3. 个人资料页与状态表达、首聊建议、关系推进之间联动仍偏弱；
4. 轻语音表达尚未形成正式候选位；
5. 轻量个人空间外观层还没有真正落成产品语义；
6. 所有这些增强都必须保持免费基础能力优先，不提前走重商业化装扮路线。

# 3. 本轮做什么

5.2 必做内容如下：

1. 标签表达体系重排；
2. 个人资料聚合区层级优化；

6. 轻量个人空间外观层最小版；
7. 5.2 截图基线、回归清单、验收证据与单文件主交接。

# 4. 本轮不做什么

5.2 明确不做：

1. 不重开 4.0~5.1 已归档主链；
2. 不重做 5.0 的 Discover / Chat / Me 基础骨架；
3. 不把 5.1 的截图 observation 继续扩成主开发任务；
4. 不做商城、币体系、超级会员装扮权益；
5. 不做大型娱乐中心；
6. 不做多人 RTC、直播、视频通话增强；
7. 不做重社区内容平台；

9. 不做本地写库型后端改造；
10. 不把 Me 页改造成安全深区或商业中心替代页；
11. 不为了表达层增强破坏 profile canonical truth、Match / Chat / Notification / RTC / version gate 主链。

# 5.2B：标签表达体系重排

## 目标

把 Me 页中的人格、关系风格、生活状态、交友态度等表达信息，从零散字段升级成更清晰、更适合慢约会阅读的标签体系。

## 建议范围

### A. 标签分层

建议至少形成以下层次：

- 人格倾向；
- 关系风格；
- 生活状态；
- 交友态度；
- 慢约会友好表达。

### B. 标签语义优化

要求：
- 更像“自我介绍语言”；
- 不像后台字段；
- 不把人格结果写成唯一真相；
- 不输出过度绝对化判断。

### C. 标签与关系推进联动

建议让标签能轻量服务：
- 首聊建议；
- 继续了解；
- 查看资料；
- 共同点提示。

## 任务

1. 梳理当前标签与资料字段；
2. 设计最小分层方案；
3. 调整 Me 页标签展示结构；
4. 产出：
   - `docs/version_plans/5.2_TAG_SYSTEM_NOTE.md`
   - `docs/version_plans/5.2_TAG_UI_EVIDENCE_INDEX.md`

## 禁止事项

- 不把标签做成大而全测评系统；
- 不让标签覆盖 canonical truth；
- 不让标签区变成密集字段表。

## 验收标准

- 标签层次清晰；
- 文案更像产品表达；
- 标签能为后续关系推进提供轻量输入；
- walkthrough 截图完整。

# 5.2C：个人资料聚合区层级优化

## 目标

让个人页更像“可经营的主页”，而不是“冗长字段清单”。

## 建议范围

### A. 资料区分层

建议至少区分：
- 核心身份信息；
- 关系相关信息；
- 展示亮点；
- 资料完整度 / 可信度提示。

### B. 轻量经营区

建议补齐：
- 我的亮点；
- 最近状态或表达入口；
- 收藏 / 赞过 / 资料建议的轻联动。

### C. 可读性优化

要求：
- 优先用户阅读；
- 不做后台字段堆叠；
- 不做商业化宫格中心。

## 任务

1. 梳理当前 Me 页资料区层级；
2. 调整信息分组和呈现顺序；
3. 增补最小经营区；
4. 产出：
   - `docs/version_plans/5.2_PROFILE_LAYOUT_NOTE.md`
   - `docs/version_plans/5.2_PROFILE_LAYOUT_EVIDENCE_INDEX.md`

## 禁止事项

- 不把 Me 页做成杂货铺；
- 不引入安全深区替代页；
- 不把资料区做成工程表单镜像。

## 验收标准

- 资料层级明显比 5.0 / 5.1 更清楚；
- 经营区存在且轻量；
- 页面仍保持现代 UI 基线。

# 5.2D：AI 展示建议升级

## 目标

把 5.0 的“查看 AI 展示建议”从入口级能力，升级成更可执行的个人经营助手。

## 建议范围

### A. 展示建议增强

建议最小覆盖：
- 资料完整度建议；
- 展示友好度建议；
- 首聊友好度建议；
- 资料亮点梳理建议。

### B. 建议落地方式

要求：
- 轻量可读；
- 可关闭、可跳过；
- 不伪成功；
- 不写成独立 AI 平台。

### C. 与主链联动

建议建议层可轻量跳往：
- 修改资料；
- 写一句问候；
- 看看状态；
- 回到聊天。

## 任务

1. 梳理当前 AI 展示建议入口；
2. 补最小可执行建议列表；
3. 补轻量联动动作；
4. 产出：
   - `docs/version_plans/5.2_AI_DISPLAY_NOTE.md`
   - `docs/version_plans/5.2_AI_DISPLAY_EVIDENCE_INDEX.md`

## 禁止事项

- 不做大模型面板；
- 不伪造 AI 自动优化成功；
- 不把建议写成唯一正确答案。

## 验收标准

- AI 建议比 5.0 更可执行；
- 建议与资料、状态、聊天存在自然联动；
- 证据链完整。

# 5.2E：AI 帮发 / AI 帮你写一句问候 / AI 整理亮点最小增强

## 目标

让 AI 真正服务表达层，但仍保持“低压、轻量、可跳过”。

## 建议范围

### A. AI 帮发

建议最小能力：
- 帮你写一句状态；
- 帮你写一句自我介绍；
- 帮你写一句问候。

### B. AI 整理亮点

建议最小能力：
- 根据资料真值和标签，整理“值得先被看见的点”；
- 生成轻量展示建议草稿；
- 不自动发布、不自动发送。

### C. 草稿逻辑

要求：
- 只写入草稿；
- 由用户确认；
- 不直接发送消息；
- 不回写服务端真值。

## 任务

1. 设计最小草稿流；
2. 打通资料页 / 状态表达 / 问候组织的轻入口；
3. 产出：
   - `docs/version_plans/5.2_AI_DRAFT_HELPER_NOTE.md`
   - `docs/version_plans/5.2_AI_DRAFT_EVIDENCE_INDEX.md`

## 禁止事项

- 不做自动代聊；
- 不做服务端队列化 AI 助手；
- 不把 AI 草稿当真实发送。

## 验收标准

- AI 帮发 / 问候 / 整理亮点最小流成立；
- 用户能看懂“这只是草稿建议”；
- 不误伤消息主链。

# 6. 硬性保护面

以下页面与视觉结构继续保护：

- 首页；
- Discover / 发现页；
- 消息列表页；
- 聊天页；
- 匹配页；
- 我的页；
- 设置页；
- 版本中心；
- 通知中心；
- 通话页；
- 星盘页；
- 现代卡片、间距、轻蓝紫配色、底部导航与现代布局骨架。

5.2 可以增强 Me / Profile 与表达层，但不得把现代 UI 重新做回工程页或旧版 UI。

## 6.2 真值链保护面

接口：

- `POST /api/v1/profile/basic`
- `GET /api/v1/profile/basic`
- `GET /api/v1/profile/astro/summary`
- `GET /api/v1/profile/astro/chart`
- `POST /api/v1/profile/astro`

表：

- `user_astro_profiles`
- `questionnaire_questions`
- `questionnaire_answers`
- `questionnaire_attempts`

规则：

- 资料真值仍以服务端重算结果为准；
- 标签表达、AI 建议、展示建议不得回写真值；
- 个人经营页增强只能建立在 canonical truth 之上，而不是替代 canonical truth。

## 6.3 消息、通知与媒体主链保护面

接口：

- `GET /api/v1/messages`
- `POST /api/v1/messages`
- `POST /api/v1/messages/read/{messageId}`
- `POST /api/v1/media`
- `GET /api/v1/media/{assetId}`

表：

- `chat_messages`
- `conversations`
- `conversation_members`
- `notifications`
- `media_assets`

# 8. 跨层 Blocker 处理条款

5.2 尽管是个人经营页增强版，但仍可能触碰：
- profile canonical truth；
- AI 草稿层；
- 标签表达与聊天联动；
- 轻语音表达与权限 / RTC；
- 版本与发布链。

因此继续强制执行跨层 blocker 规则。

## 8.1 触发条件

以下任一情况都要进入 blocker 流程：

- 标签或展示建议与真实资料不一致；
- AI 草稿看似写入成功，但实际链路错误；
- 轻语音表达入口存在，但权限 / 回流 / 候选位语义异常；
- 个人空间外观层干扰真正设置中心；
- 构建、网络、端口、协议、后端环境异常。

## 8.2 标准流程

1. 冻结现象；
2. 先写 Markdown blocker report；
3. 请 Claude 做根因缩层；
4. Codex 只做最小修复；
5. 用用户面截图 + 机器证据双重复验；
6. 写 fix note / verification note / acceptance summary。

## 8.3 禁止事项

- 禁止没有 blocker report 就继续盲改；
- 禁止跨层 blocker 多轮只在 Flutter / UI 层死磕；
- 禁止用假成功态掩盖真实失败；
- 禁止根因未定前扩大版本范围。

# 9. Rollback / Recovery Policy

## 9.1 禁止 repo 级恢复

默认禁止：

```bash
git reset --hard <old_commit>
git checkout <old_commit> .
git restore --source=<old_commit> .
git clean -fdx
```

也禁止：
- 用旧 zip 覆盖当前仓库；
- 为修 5.2 局部问题恢复整仓；
- 恢复旧 UI 覆盖 protected surfaces。

## 9.2 允许的恢复方式

只允许：
- path-level restore；
- file-level restore；
- smallest-scope restore。

## 9.3 恢复前必须 checkpoint

恢复前必须执行：

```bash
git status --short
git branch --show-current
git rev-parse HEAD
git diff --stat
git diff --name-only
```

并至少执行一种：

```bash
git switch -c safety/pre-recovery-<date>-<topic>
```

或：

```bash
git add -A
git commit -m "safety: checkpoint before <topic> recovery"
```

或：

```bash
git stash push -u -m "safety: checkpoint before <topic> recovery"
```

# 19. 最终执行提醒

> **让用户更容易表达自己、被理解、被继续了解，同时保持慢约会语义和轻量产品路径。**

1. 这是否真的增强了“为什么别人愿意继续看我、回我、和我继续聊”？

4. 这是否会误把个人页做成商业中心或安全中心？
5. 这是否需要先 blocker report，而不是继续盲修？
