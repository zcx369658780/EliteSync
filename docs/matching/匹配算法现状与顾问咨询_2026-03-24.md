# 匹配算法现状与顾问咨询（2026-03-24）

## 1. 当前系统已实现（后端）

### 1.1 硬过滤（必须满足）
- 同城匹配：`matching.hard_filters.same_city_only=true`
- 异性匹配：`matching.hard_filters.opposite_gender_only=true`
- 关系目标极端冲突过滤：`casual` vs `marriage` 直接过滤
- 近14天重复配对过滤：`exclude_recent_pair_days=14`

### 1.2 当前配对打分主链路（MatchingEngineService）
- 已计算以下中间量：
  - `profileSimilarity`（问卷画像向量余弦）
  - `questionCompatibilityDirectional`（双向可接受答案匹配）
  - `interestSimilarity`
  - `activityScore`
  - `freshnessScore`
  - `fairnessMultiplier`
- 但当前 `base` 分数实际直接采用 `AstroCompatibilityService.total/100`
  - 即：当前最终核心分几乎由玄学模块主导
- 最终：
  - `base = astro_total`
  - `final = base * penalty_product`
  - `fair_adjusted = final * fairness_multiplier`

### 1.3 玄学模块（AstroCompatibilityService）
- 权重：
  - 八字 50%
  - 属相 30%
  - 星座 10%
  - 星盘 10%
- 输出：
  - 子分：`score_bazi/score_zodiac/score_constellation/score_natal_chart`
  - `match_verdict`（high/medium/low）
  - `match_reasons`（summary/match/mismatch/confidence）
- 已落库到 `dating_matches`，并通过 `match/current` 返回

### 1.4 合成账号数据
- 已批量补全合成账号扩展画像（出生地、时间、八字、星盘字段）
- 命令：`app:dev:fill-synthetic-astro`

## 2. 当前问题（产品反馈）

### 问题A：性格测试与MBTI未有效进入匹配主分
- 现状：
  - MBTI仅作为 `penaltyFactors` 中轻微乘子（字母位匹配微调）
  - 进入APP时的人格测试（大五/画像向量）虽然有计算，但在当前主分里被弱化，且未形成显式“人格匹配分”展示
- 影响：用户体感“做了题，但分数没体现”

### 问题B：匹配理由理论解释不够细
- 现状：文案仍偏模板化，例如“八字五行互补度较好”
- 需求：给出更细节且可解释的规则依据（例如五行分布差异、生肖关系类型、星座元素关系、星盘关键项）

### 问题C：理由排版与可读性不佳
- 现状：理由可读性不足，结构密集
- 需求：
  - 每种理由单独一行
  - 按分数高低使用颜色层级（高分偏亮/低分警示）
