## 2. 总体产品判断

测测不是单一玄学工具页集合，而是一套“入口矩阵 + 档案上下文 + 解释层 + AI/问题追问 + 社区/服务转化”的复合产品。

Stage 1-5 当前能稳定支持的总体判断：

- 首页承担高密度入口分发：星盘、星座、测试、AI问、在线、缘分合盘等能力以快捷入口和频道形态并列。
- `AI问` 是问题导航 + AI解释入口 + 部分真人导流混合层。
- `在线` 是真人服务市场 / 达人目录 / 强转化服务层。
- `测试` 在 Claude 抽样中更像互动答题入口层，而不是全量内容化结果页。
- `星盘` 是单人档案上下文下的图谱先行解释模块。
- `缘分合盘` 是双人档案上下文下的关系解释入口。
- `档案列表` 是贯穿星盘、生辰、紫微、合盘的上下文管理层，不是结果页。

必须保留的总边界：

- 多个入口可见，不等于完整链路已验证。
- AI 追问入口可见，不等于已验证 `AI问` 目标页。
- 上传入口可见，不等于上传流程可用或安全。
- 真人 / 在线服务在某些频道可见，不等于所有解释页都导向服务成交。
- 示例 / 预览结果可见，不等于真实结果页已验证。

## 3. Stage 1：基础 IA 与首页骨架

Stage 1 的主要价值是建立测测的全局产品地图。

当前结论：

- 底部主导航围绕首页、消息、AI问、在线、我的等核心入口组织。
- 首页不是单一 feed，而是功能入口、内容推荐、频道导航和商业入口混合页。
- 测测把“玄学能力”包装成多个可点入口，而不是单一测试工具：
  - 星盘
  - 星座
  - 测试
  - AI问
  - 在线
  - 缘分合盘
  - 生辰 / 紫微 / 合盘等细分模块
- 个人页和设置页包含账号、外观、权限、商业和安全相关入口。

对 EliteSync 的启发：

- 不应把玄学能力做成孤立页面，应作为多个轻入口嵌入 Match、Me、Chat、Discover 等主链。
- 首页入口密度可以高，但 EliteSync 需要更克制，避免把慢约会产品做成工具集合。
- 能力入口应先做“解释层 / 建议层 / 轻追问”，不要提前做强交易或强服务市场。

## 4. Stage 2：`AI问` 与 `在线`

Stage 2 Claude rerun 将 `AI问` 与 `在线` 的职责拆清楚。

### `AI问`

当前定位：

- 问题导航页。
- AI 解释入口。
- 部分达人 / 真人推荐混合层。
- 用户从具体问题进入解释，而不是从空白对话框开始。

典型结构：

- 顶部 AI / 问题语义。
- 问题卡。
- 推荐卡 / 达人卡。
- 提问、追问、咨询等 CTA。

### `在线`

当前定位：

- 真人服务大厅。
- 达人目录。
- 强转化服务页。

典型结构：

- 达人列表。
- 状态、价格、评分、服务标签。
- 筛选 / 排序。
- 咨询、连麦、真人服务相关 CTA。

Stage 2 对 EliteSync 的启发：

- AI 解释层和真人服务层必须分开设计。
- EliteSync 可以吸收“从问题卡进入 AI 解释”的低压方式。
- 不建议把真人服务市场提前压到主链上；在 EliteSync 当前阶段，真人服务应远后置或不做。

## 5. Stage 3：`测试` 频道

Stage 3 Claude rerun 推翻并收窄了旧版判断。

旧版倾向把 `测试` 概括为内容化解释流 / 结果式文章流 / 作者互动流。Claude 真机重跑后，当前更稳的判断是：

- `测试` 频道在本轮抽样中更像互动答题入口层。
- 多个详情页先展示 A/B/C/D 答题前入口，不是提交后的真实结果页。
- 部分页面存在作者 / 社区互动和创作者服务导流迹象。
- 真实提交后的测试结果页仍未验证。

必须保留的边界：

- 不能写“测试频道 100% 都是互动答题型”。
- 不能写“测试结果页导向 AI问 / 在线 / 真人服务”已验证。
- 当前更安全表述是：本轮抽样显示多个测试详情页强引导互动答题入口，但尚不能证明所有详情页均如此。

对 EliteSync 的启发：

- 互动测试适合做轻入口，不适合成为事实画像来源。
- 测试结果应是 derived-only / explanation-only，不能回写用户真值。
- 可借鉴答题前详情页的引导节奏，但不要复制高压导流和创作者服务包装。

## 6. Stage 4：`星盘`、`生辰`、`紫微` 与档案上下文

Stage 4 Claude rerun 确认 `星盘` 是单人档案上下文下的图谱先行解释模块。

当前结论：

- `星盘` 绑定 `示例档案` 等档案上下文，不是匿名工具页。
- 第一层是图谱 / 盘面 / 盘型切换。
- 下方是基础解读、DeepSeek 解读入口、结论句、报告卡、社区内容。
- `DeepSeek解读` 和 AI 追问卡可见，但未点击验证目标页。
- 保存证据中未发现页面内容级 `AI问`、`在线`、真人 / 达人 / 咨询服务入口。

`生辰` / `紫微` 对照：

- `生辰` 更偏参数表、时间与命理数据说明。
- `紫微` 更偏宫位图谱、大限 / 流年与工具型结构。
- 两者只做浅层对照，不能证明深层没有解释或追问。

档案上下文：

- `档案列表` 是上下文管理 / 选择页，不是结果页。
- `添加档案`、编辑、删除、`选择完成` 都属于写入或确认边界。
- `合盘` 可进入档案选择状态，但 Stage 4 未验证合盘结果页。

对 EliteSync 的启发：

- 画像 / 星盘能力应围绕已有 profile 上下文组织，而不是要求用户重复输入。
- 图谱和解释可以分层，但解释层必须是 display-only。
- AI 追问适合放在解释之后，不适合作为首屏主路径。

## 7. Stage 5：`缘分合盘` / 关系合盘

Stage 5 Claude rerun 验收通过，口径为 pass with observations。

当前结论：

- `缘分合盘` 是双人档案上下文下的关系解释入口。
- 页面包含两个 `请选择档案` 槽位、`契合度` 和关系语义句。
- 档案列表层可见搜索、排序、筛选、`选择了1人：示例档案`、`添加档案`、`选择完成`。
- 关系总览可见六维评分：
  - `羁绊`
  - `激情`
  - `玩乐`
  - `亲密`
  - `默契`
  - `承诺`
- 示例 / 预览解释层可见：
  - `建议 / 避免`
  - `上传截图秒懂ta`
  - `ta说这些话是什么意思`
  - `ta的心理是什么`
  - `更多解读`
  - `时间 / 星象`

必须保留的边界：

- 真实双人合盘结果页未验证。
- `选择完成` 未点击，后续是结果页、校验、登录、保存还是生成链未知。
- `上传截图秒懂ta` 未点击，上传链未知。
- 追问入口未点击，不能写成导向 `AI问`、DeepSeek 或聊天。
- 页面内容级 `DeepSeek解读`、`AI问`、`在线`、真人 / 达人 / 咨询 / 连麦入口未验证。

对 EliteSync 的启发：

- 关系解释页最值得借鉴的是“双人上下文 -> 关系语义 -> 维度评分 -> 建议 / 避免 -> 轻追问”的节奏。
- 上传截图不适合作为 EliteSync 主路径；应优先使用已有聊天和匹配上下文。
- 关系解释必须是 derived-only / display-only，不得反写资料真值。

## 8. 横向结构模型

Stage 1-5 合并后，测测的玄学能力可以抽象成五层：

1. 入口层
   - 首页快捷入口、频道 tabs、模块 tabs。
2. 上下文层
   - 单人档案、双人档案、示例档案、档案列表。
3. 解释对象层
   - 测试题、星盘图谱、生辰表、紫微宫位、关系合盘。
4. 解释 / 建议层
   - 基础解读、结论句、维度评分、建议 / 避免、更多解读。
5. 追问 / 转化层
   - AI 追问、上传、社区讨论、报告预览、真人 / 在线服务。

对 EliteSync 来说，最有价值的是前四层，最需要克制的是第五层。

## 9. EliteSync 路线图判断

结论：建议进入“玄学能力二次产品化”版本规划，但不建议立刻进入大规模实现。

更准确的路线图状态应是：

`进入版本规划 / 产品化设计阶段`，不是 `立即进入 runtime implementation`。

建议原因：

- Stage 1-5 已经证明玄学能力可以作为关系理解、个人画像解释和轻追问的产品化素材。
- Stage 4 / Stage 5 对 EliteSync 的 Match、Me、Chat 主链有直接启发：个人解释、关系解释、建议 / 避免、轻追问。
- Stage 2 证明 AI 解释层与真人服务层必须分离，适合 EliteSync 避免高压转化。
- Stage 3 证明测试类能力容易误伤事实真值，必须限制在娱乐 / 引导 / 解释层。

不建议立刻实现的原因：

- EliteSync 的 canonical truth 边界必须先固化。
- 玄学能力需要明确 derived-only / display-only contract。
- 需要先设计用户授权、免责声明、可撤回、可隐藏、不可反写资料真值等保护。
- 当前竞品证据仍有多个未验证链路，不能直接复刻。

## 10. 建议版本规划草案

建议把“玄学能力二次产品化”拆成一个后续规划包，而不是塞进当前 5.x 主链临时实现。

### Phase A：产品边界与 contract

目标：

- 定义所有玄学 / 关系解释输出为 derived-only。
- 明确禁止反写 `profile/basic`、`profile/astro/summary`、`profile/astro/chart`、`user_astro_profiles`。
- 定义解释层来源、展示范围、免责声明和用户控制。

### Phase B：Me 个人解释层

目标：

- 在 Me 或个人画像页提供轻量自我理解卡。
- 只使用已有服务端真源和用户授权上下文。
- 不新增硬玄学输入链，不生成事实画像。

### Phase C：Match 关系解释层

目标：

- 为匹配详情增加关系解释模块。
- 结构参考 Stage 5：关系短句、维度解释、建议 / 避免、轻追问。
- 不做上传，不做真人服务，不做高压转化。

### Phase D：Chat 轻追问入口

目标：

- 在 Chat 中提供基于当前关系上下文的轻追问建议。
- 追问生成草稿或解释建议，不自动发送。
- 用户可关闭或忽略。

### Phase E：AI 助手衔接

目标：

- 将解释层与 AI 助手连接，但只作为可选追问。
- 不把 AI 输出写回资料真值。
- 不引入真人服务市场。

## 11. 不应进入路线图的内容

当前不建议规划：

- 真人玄学咨询市场。
- 付费报告商店。
- 上传聊天截图作为主路径。
- 真实出生资料强收集。
- 用测试结果改写用户画像。
- 大规模社区讨论压入关系结果页。
- 复制测测术语体系或强玄学诊断口吻。

## 12. 后续最小行动

下一步建议不是继续拆测测 Stage 6，而是先做 EliteSync 内部规划文档：

1. 新建“玄学能力二次产品化”版本规划草案。
2. 先写 non-goals 和 protected surfaces。
3. 定义 derived-only / display-only 数据边界。
4. 选择一个最小产品切片：优先 `Match 关系解释层`，其次 `Me 个人解释层`。
5. 等用户确认后，再进入实现计划。

当前建议评级：

`go to planning, do not implement yet`
