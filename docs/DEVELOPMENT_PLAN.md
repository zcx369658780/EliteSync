# EliteSync 开发拆解与详细计划（V1）

> 依据：`README.md` 中的系统目标、架构、模块划分、接口设计、数据模型与实施计划章节。

## 1. 项目目标与范围拆解

### 1.1 前端 Android App 目标
- 支持「注册登录 → 学生认证 → 66题问卷 → 每周 Drop 匹配 → 双边确认 → 即时聊天 → 个人中心」完整闭环。
- 围绕「慢约会」建立仪式感体验：每周二晚 9 点匹配揭晓、匹配亮点展示、破冰问题引导。
- 满足基础可用性指标：首屏、关键页面加载与消息实时性体验可接受。

### 1.2 后端服务目标
- 提供用户、认证、问卷、匹配、消息五大核心服务。
- 实现定时匹配任务（预计算 + Drop 时刻发布）与实时消息服务（HTTP + WebSocket）。
- 满足合规与安全要求：鉴权、数据加密、风控审计、隐私保护。

### 1.3 首版（MVP）范围建议
- **必须实现（P0）**：账号体系、认证流程（先支持邮箱/人工审核占位）、问卷系统、每周匹配、双向确认、文本聊天、基础运营后台。
- **次要实现（P1）**：破冰问题推荐、通知系统完善、举报拉黑、埋点分析看板。
- **后续迭代（P2）**：多媒体消息、复杂推荐策略、灰度实验平台、智能风控升级。

## 2. 总体技术与仓库结构建议

## 2.1 技术栈建议
- **Android 客户端**：Kotlin + Jetpack Compose + MVVM + Retrofit/OkHttp + Room + Hilt + WebSocket(STOMP)。
- **后端服务**：Java/Kotlin + Spring Boot（或 Go + Gin，二选一建议先统一 Java 技术栈）+ PostgreSQL + Redis + MQ + MinIO/OSS。
- **算法与定时任务**：Python（离线计算）+ Celery/Airflow（或 Spring Scheduler）+ 向量检索引擎（后续接入 Milvus）。
- **运维与交付**：Docker + Docker Compose（开发）/K8s（生产）+ CI/CD（GitHub Actions）。

### 2.2 仓库目录建议（单仓）
- `apps/android/`：Android App。
- `services/api/`：后端 API 与 WebSocket。
- `services/match-engine/`：匹配计算任务。
- `services/admin/`：运营后台（可后续再拆）。
- `infra/`：容器编排、数据库脚本、监控配置。
- `docs/`：设计文档、接口文档、测试计划。

## 3. 工作分解结构（WBS）

## 3.1 产品与设计
1. PRD 细化：将 README 的宏观设计转为可验收的用户故事。
2. 交互原型：登录、认证、问卷、匹配、聊天、个人页全流程原型。
3. UI 规范：颜色、字体、组件、动效、空态与错误态。

### 3.2 Android 客户端
1. 工程初始化：架构分层、路由、网络层、日志与崩溃收集。
2. 账号与认证模块：注册登录、Token 刷新、实名认证/学生认证状态机。
3. 问卷模块：题目拉取、分步作答、进度保存、断点续答、提交。
4. Drop 模块：倒计时、结果揭晓、匹配亮点、意向选择（愿意/不愿意）。
5. 聊天模块：会话列表、消息列表、发送接收、已读状态、断线重连。
6. 个人中心：资料编辑、认证状态、隐私设置、帮助反馈。
7. 质量保障：单元测试、UI 自动化、性能监控、埋点。

### 3.3 后端服务
1. 基础能力：统一鉴权（JWT）、用户中心、配置中心、日志追踪。
2. 认证服务：学校/邮箱认证流程、审核队列、状态回传。
3. 问卷服务：题库管理、答卷提交、答案加密存储、版本管理。
4. 匹配服务：用户向量生成、匹配打分、稳定匹配、结果入库。
5. Drop 发布服务：定时任务、结果缓存、消息推送。
6. 聊天服务：HTTP 消息接口 + WebSocket 推送 + 已读回执。
7. 风控与合规：举报、拉黑、审计日志、敏感词/内容安全预留。
8. 运维支持：健康检查、指标监控、告警、灰度发布。

### 3.4 数据与算法
1. 数据库建模：users/questions/user_answers/matches/messages 主表 + 索引。
2. 数据安全：字段加密策略、脱敏策略、密钥轮换策略。
3. 算法迭代：规则版（MVP）→ 向量版（V1）→ 多目标优化（V2）。
4. 算法评估：匹配满意度、双向确认率、7日留存、聊天启动率。

### 3.5 测试与发布
1. 测试体系：单元、集成、接口契约、端到端、性能、安全测试。
2. 发布策略：开发/预发/生产三环境，灰度+回滚预案。
3. 运营准备：审核 SOP、客服应急手册、值班机制。

## 4. 分阶段开发计划（16 周）

## 4.1 阶段 0：项目启动（第 1 周）
- 目标：需求冻结、技术选型冻结、基础工程搭建。
- 交付：
  - 产品需求清单（含 P0/P1/P2）。
  - 前后端脚手架初始化。
  - 数据库初版 ER 图。
- 验收标准：可启动 Demo（空页面 + 健康检查接口）。

### 4.2 阶段 1：核心账户与问卷（第 2-5 周）
- Android：登录注册、认证状态页、问卷作答页（含保存草稿）。
- 后端：用户服务、认证服务、问卷服务、题库管理接口。
- 数据：users/questions/user_answers 表落库与迁移脚本。
- 验收标准：新用户可完成注册、认证提交、问卷提交全流程。

### 4.3 阶段 2：匹配与 Drop（第 6-9 周）
- Android：Drop 倒计时、匹配结果卡片、意向确认页。
- 后端：匹配任务、匹配结果查询、双向确认逻辑、推送通知。
- 算法：先上线规则版匹配（问卷权重 + 基础约束）。
- 验收标准：可模拟每周匹配并完成双向确认。

### 4.4 阶段 3：即时通讯与安全（第 10-12 周）
- Android：会话列表、聊天页、WebSocket 连接管理、离线重连。
- 后端：消息接口、WebSocket 网关、已读状态同步。
- 安全：限流、审计日志、敏感词预留接口、关键数据加密。
- 验收标准：匹配成功双方可实时聊天，消息可追溯。

### 4.5 阶段 4：稳定性与上线准备（第 13-16 周）
- 性能：压测 Drop 高峰、数据库与缓存调优。
- 质量：回归测试、崩溃率与ANR指标治理。
- 发布：预发灰度、生产上线、监控告警全开。
- 验收标准：达成上线门槛（见第 6 节）。

## 6. 关键里程碑与验收门槛

| 里程碑 | 时间 | 验收结果 |
|---|---:|---|
| M1 需求/架构冻结 | W1 末 | PRD、技术方案、ER 图评审通过 |
| M2 问卷闭环完成 | W5 末 | 注册-认证-问卷链路端到端可用 |
| M3 匹配闭环完成 | W9 末 | Drop 发布与双向确认可用 |
| M4 聊天闭环完成 | W12 末 | 匹配后实时聊天稳定可用 |
| M5 上线就绪 | W16 末 | 性能/安全/回归通过，具备灰度上线条件 |

上线门槛（建议）：
- 核心接口 P95 < 500ms（Drop 高峰除外设单独阈值）。
- 崩溃率 < 0.5%，ANR < 0.3%。
- Sev1/Sev2 级缺陷清零。
- 数据加密、审计、权限控制通过安全检查。

## 7. 风险清单与应对策略

1. **认证链路复杂、外部接口不稳定**  
   - 对策：先做邮箱认证 + 人工审核兜底，学信网逐步接入。
2. **匹配算法效果不达预期**  
   - 对策：先定义业务 KPI（确认率、聊天启动率），采用 A/B 策略渐进优化。
3. **Drop 时刻并发冲击**  
   - 对策：T-1 预计算 + Redis 缓存 + 限流降级。
4. **聊天服务稳定性风险**  
   - 对策：WebSocket 网关集群 + ACK/重试 + 离线补偿。
5. **合规与隐私风险**  
   - 对策：最小化采集、分级权限、全链路审计、敏感数据加密。
