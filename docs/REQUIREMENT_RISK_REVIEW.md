# 需求评审与风险拆解规范

> 这是 `dependency-mapper + risk-reviewer + test-planner + architecture-guardian` 的可复用执行规范。每次复杂任务必须先产出这份评审结论，再允许进入实现。

## 1. 评审目标

- 识别需求会影响哪些模块
- 找出高风险改动面
- 明确不可破坏的旧功能
- 给出回滚点和验证策略

## 2. 必须并行完成的只读评审

1. `dependency-mapper`
2. `risk-reviewer`
3. `test-planner`
4. `architecture-guardian`

## 3. 统一输出格式

### 3.1 需求理解

- 需求是什么
- 为什么现在要做
- 本轮不做什么

### 3.2 影响面

- 页面 / 入口
- 状态层
- 数据层
- 数据库
- 配置 / SDK
- 地图 / 定位
- 路由 / 生命周期

### 3.3 高风险点

- 数据破坏风险
- 路由丢失风险
- 定位 / 权限风险
- 缓存污染风险
- 兼容层断裂风险

### 3.4 旧功能保护

- 原按钮行为
- 原入口路径
- 原数据读写
- 原异常处理

### 3.5 回滚与恢复

- 回滚点
- 恢复方式
- 数据恢复方式

### 3.6 验证计划

- smoke
- 回归
- 手工验收
- 边界条件

## 4. 决策门槛

以下任一项不清楚时，不得写代码：

- 旧功能映射不明确
- 备份/回滚点缺失
- 数据库或定位链路未评审
- 验证命令未定义

## 5. 输出模板

```
需求理解：

受影响模块：

高风险点：

不可破坏的旧功能：

回滚/恢复点：

验收标准：

本轮明确不改什么：
```
