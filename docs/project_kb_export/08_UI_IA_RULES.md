# UI / IA 规则

更新时间：2026-04-19

## 总体原则

- 页面要表达产品能力，不要像内部调试面板。
- 轻量联动优先，避免把单模块做成“工程仪表盘”。
- 版本中心、问卷、图片 / 视频消息、星盘页都要保持“用户可读”优先。

## 不要太工程化的页面

- 版本中心
- 问卷结果页
- 高级时法 / 玄学展示页
- 图片消息上传 / 预览页
- 视频消息预览页

这些页面应避免：
- `scaffold`
- `debug`
- `demo`
- `internal`
- `mock`
等过强工程语义出现在主文案里。

## 强调轻量联动的模块

- 首页人格摘要卡
- 匹配解释页的人格提示
- 问卷历史入口
- 版本中心历史与服务状态

这类联动应当是 additive，不要改变主流程结构。

## 截图验收中的经验总结

- 最好的证据链是：入口 -> 操作 -> 结果 -> 历史 / 回读。
- 版本中心截图必须和宿主 APK 版本一致。
- 失败态要可见，但不要把失败态做得比成功态更重。
- 图片预览要清晰，但不要把聊天页改成相册页。
- 视频播放页要清晰，但不要把聊天页改成播放器列表页。

## 当前 UI 口径

- 星盘：真值优先，展示偏专业但要保持可读
- 问卷：非官方、版本化、结果只讲当前倾向
- 图片消息：清楚展示上传状态、失败重试和预览
- 视频消息：清楚展示上传状态、播放预览和失败恢复
- 首页 / 匹配：轻量联动，不重写主算法

## 设计禁区

- 不要把人格结果写成唯一真相
- 不要把版本中心做成纯工程页面
- 不要把消息页做成多页面媒体平台
- 不要把星盘页回退成纯技术面板
