# UI / IA 规则

更新时间：2026-05-12

## 6.0 Alpha UI/IA 原则

- 页面要表达真实用户任务，不要像内部调试面板。
- 首页从功能堆叠改为今日主任务。
- 星盘长页必须拆分，优先摘要 / 维度 / 详情分层。
- 清除工程术语、内部状态、测试语义和 mock / debug 文案。
- 入口要低压、清晰、安全，尤其是 Match、Buddy / 搭子、Chat 与位置相关页面。

## Buddy / 搭子入口

- Buddy / 搭子入口要低压、清晰、安全。
- 不把搭子做成泛同城约玩。
- 学习搭子、电影搭子、吃饭搭子、健身搭子等场景必须体现共同兴趣和边界。
- 不默认暴露精确位置、私密资料或高风险见面暗示。
- 必须给用户明确的退出、举报、拉黑和反馈路径规划。

## Match / Date Drop 式匹配

- Match 主链要服务低频高质量匹配，不做无限推荐流。
- 页面应解释“为什么值得聊 / 为什么适合等待 / 如何低压开启”，而不是只给结论。
- 等待期、揭晓、反馈闭环要清楚，不使用工程状态词。

## Astro / 玄学解释

- 玄学解释必须用户可读，避免像工程面板。
- 星盘、八字、紫微等长内容要拆成摘要、维度、详情。
- display-only / derived-only 免责声明必须清楚可见。
- 不出现真人咨询、付费报告、上传截图主路径或娱乐化测试主路径。

## Chat / Profile / Settings

- Chat 建议只能辅助开场和回聊，不自动发送，不替换输入栏。
- Profile 建议只能辅助表达，不自动修改资料。
- Settings 负责解释边界、用户控制和隐私说明，不做后台治理平台。

## 截图验收经验

- 最好的证据链是：入口 -> 操作 -> 结果 -> 历史 / 回读。
- 版本中心截图必须和宿主 APK 版本一致。
- 失败态要可见，但不要比成功态更重。
- UI 文案必须面向普通用户，不面向工程人员。
