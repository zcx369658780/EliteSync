# Codex 协作规则

更新时间：2026-04-19

## 默认角色

- Codex：默认主编排器，负责计划、实现、修复、文档收口和最终整合。
- Claude：只做高价值架构顾问和边界风险审查。
- Gemini：只做验收、视觉 / UI / walkthrough、长上下文总结。

## 默认执行顺序

1. 先做需求拆解和边界确认
2. 再做风险清单
3. 再做最小实现
4. 再做测试与回归
5. 再做验收摘要、handoff、closeout
6. 最后才考虑 PR / 发布

## 新版本开发流程

- 先看当前计划书
- 先冻结不改什么
- 先列保护面
- 先确认版本号与发版脚本
- 再进入实现
- 实现后必须有测试和验收材料
- 发版前先同步 `apps/android/app/build.gradle.kts`、`docs/CHANGELOG.md`、`services/backend-laravel/config/app_update.php` 与更新历史

## 什么时候先做边界说明

- 涉及数据库 schema
- 涉及真值链
- 涉及权限 / 路由 / 状态缓存
- 涉及媒体 / 存储 / 上传
- 涉及版本号与发版脚本

这类任务必须先写清边界，再动代码。

## 什么时候可以先做实现

- 仅是局部 UI 文案
- 仅是无状态展示修正
- 仅是不会影响保护面的轻量改动

即便如此，也要先确认不会误伤版本中心、缓存、真值链。

## Claude / Gemini 失效时的处理

- Claude 因预算、配额、错误不可用时，先通知并停止重试。
- Gemini 因额度、配额、错误不可用时，改用已授权 subagent 或本地测试自证，不要让任务停摆。
- 不可用时，Codex 继续承担主工作，不允许整个版本开发链断掉。

## 本仓库固定规则

- 版本号、版本检查、下载包必须绑定同一条链。
- 发版优先使用 `scripts/release_android_update_aliyun.ps1`。
- 截图证据、回归材料、closeout 文档是验收的一部分，不是附录。
- 任何新版本若触碰高风险面，必须先做计划、风险评审、测试计划，再实施。
- 本地只做前端开发与联调；后端、迁移、备份、恢复和写库操作默认在阿里云端执行。
