# 回滚计划

> 适用于所有高风险改动、发布门禁、线上热修与回归失败。回滚不是事后补救，而是每次变更前必须确认的执行路径。

## 1. 回滚原则

- 优先回到最近一个已验证可用版本
- 优先回滚配置和路由，再回滚业务逻辑
- 不在未确认备份可用的情况下做生产回滚
- 回滚后必须补做最小 smoke 与关键页面验收

## 2. 回滚触发条件

- 发布门禁失败
- 关键页面 loading / 空白 / 崩溃
- 登录态异常
- 资料保存失败或画像回读异常
- 地点搜索、summary、chart 任一关键链路不可用
- 线上数据与页面展示明显不一致

## 3. 回滚层级

### 3.1 配置回滚

- 恢复最近一次稳定的环境变量 / 路由配置 / SDK 配置
- 先回退最小配置，不直接删除业务代码

### 3.2 代码回滚

- 回退最近一次提交或 release 分支 hotfix
- 优先回退与高风险 surface 相关的文件
- 不在回滚时顺手扩大重构范围

### 3.3 数据回滚

- 仅在数据变更不可逆时启用
- 先恢复备份，再验证关键记录
- 保持数据恢复与代码版本一致

## 5. 标准回滚步骤

1. 确认故障类型：配置 / 代码 / 数据 / 部署
2. 冻结新变更
3. 恢复最近一次稳定版本
4. 校验关键接口
5. 校验关键页面
6. 记录回滚原因与影响面

## 6. 回滚后最小验收

- 登录成功
- 资料页可打开
- 保存资料不报错
- summary 可读
- chart 可读或可降级
- 玄学四页不白屏
- 地点搜索可用
