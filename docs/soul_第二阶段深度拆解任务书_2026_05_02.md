# 0. 任务一句话结论

本轮不再停留于 Soul 第一阶段的“模块级浅层观察”，而进入 **第二阶段深度拆解**：

> 以 **Discover / Chat / Me / Settings 分层** 为主轴，继续向下拆 Soul 的功能闭环、页面联系、实现逻辑与设计意图，形成一套能直接服务 EliteSync 后续产品开发、交互设计、边界判断与验收复核的结构化参考资料。

本轮目标不是“抄 Soul”，也不是“把 Soul 所有入口都点一遍”，而是：

1. 深入分析 Soul 各功能模块的功能和联系；
2. 梳理 Soul 功能可能的实现方式与设计逻辑；
3. 为完善 EliteSync 的各项功能积累足够的参考资料和证据；
4. 尽量减少 EliteSync 后续因产品结构不清、交互路径不顺、功能边界模糊而反复返工。

# 1. 背景与本轮启动原因

第一阶段拆解已经证明：Soul 最值得参考的不是某个孤立小功能，而是其 **产品结构组织方式**。

已初步确认的高价值结论包括：

- Discover 不是纯 Feed，而是复合入口层；
- Chat 不是纯消息列表，而是关系推进层；
- Me 不是静态资料页，而是个人经营中枢；
- Settings 与个人空间外观应分层；
- AI、首聊建议、回聊、资料卡、语音邀请等能力是串联关系推进的，而不是各自孤立存在；
- 商业化、权限申请、互动玩法与主社交流程是明确分层的。

但当前已完成的仍然主要是 **浅层结构识别**，还不足以支持 EliteSync 后续“尽可能少走弯路地补全大部分高频功能”。

因此需要进入第二阶段：

- 从“看到了哪些页面”升级为“这些页面如何构成闭环”；
- 从“识别有哪些按钮”升级为“这些按钮为什么放在这里、节奏如何安排”；
- 从“描述功能表象”升级为“推断可能的实现方式、状态流与设计逻辑”。

# 2. 本轮总目标

## 2.1 主目标

### A. 功能与联系分析

继续拆清 Soul 各主模块之间的关系，尤其是：

- 内容流如何导向聊天；
- 匹配 / 推荐如何导向首聊；
- 首聊如何进入回聊与关系推进；
- 个人资料如何服务于聊天与继续了解；
- AI 如何嵌入资料、聊天、状态而不喧宾夺主；
- 语音邀请如何作为关系推进节点出现；
- 设置、外观、权限、商业化如何与主社交流程分层。

### B. 实现方式与设计逻辑分析

在不做逆向工程和高风险操作的前提下，基于页面结构、文案、入口关系、状态表现、返回路径、权限提示、空态与失败态，推断 Soul 各能力的可能实现方式与设计意图。

### C. EliteSync 参考资料积累

形成足够厚的文字结论、截图证据、层级梳理与功能映射，供 EliteSync 后续：

- 制定版本计划；
- 调整页面 IA；
- 设计低压转化路径；
- 补齐 AI 草稿与关系摘要；
- 校验 Discover / Chat / Me / Settings 的产品合理性；
- 降低“做了功能但不好用”或“修一周才发现方向错”的风险。

## 2.2 次目标

- 为未来 GPT 项目源保留一份可长期复用的 Soul 深拆资料；
- 为 Codex 之后的实现回合提供更清晰的“应该像什么、为什么这样做”；
- 为顾问后续验收提供竞品参照框架，而不是只看页面是否能点开。

# 3. 本轮不做什么

以下内容默认不纳入本轮第二阶段深拆范围，除非后续单独批准：

## 3.1 不进入高风险或不必要区域

- 支付确认页
- 会员购买链路
- 币体系充值
- 删除账号
- 账号安全深层页
- 实名、申诉等敏感治理流程
- 实际向陌生人发送真实内容
- 需要授权相机、麦克风、相册后才可深入的高风险链路

## 3.2 不以“全量覆盖”为目标

本轮不要求把 Soul 每一个入口都拆完。

目标是：

- 优先拆高价值闭环；
- 优先拆 EliteSync 后续高概率会做的能力；
- 优先拆能减少返工的结构逻辑；
- 不浪费时间在重商业化、币体系、装扮商城、娱乐玩法中心。

## 3.3 不做盲目模仿

本轮不是为了复制 Soul 页面，而是为了提炼：

- 结构启发；
- 节奏启发；
- 分层启发；
- 状态表达启发；
- 低压社交转化启发。

所有结论都必须改写成 EliteSync 可用语言，而不是原样照搬 Soul。

# 4. 第二阶段拆解的总方法

## 4.1 从“模块扫描”升级为“闭环拆解”

每个子任务都不能只回答：

- 页面上有什么；
- 按钮叫什么。

而必须尽量回答：

1. 这个页面在整体产品中承担什么角色；
2. 它和上游入口、下游动作如何连接；
3. 它服务的是曝光、筛选、首聊、回聊、关系推进还是经营表达；
4. 这些组件为什么这样排布；
5. 对 EliteSync 有哪些可吸收结论；
6. 哪些不能学。

## 4.2 从“截图堆积”升级为“结构化记录”

每轮拆解必须同时产出：

- 文字结构结论；
- 页面截图；
- UI hierarchy / XML；
- 页面关系表；
- 关键路径摘要；
- EliteSync 对应参考点。

不能只留截图，不写结论。

## 4.3 从“功能名词”升级为“设计推断”

在不越界的前提下，允许做谨慎推断，但必须显式标注为：

- 观察事实；
- 合理推断；
- 不确定点；
- 后续待验证点。

例如：

- 某个入口之所以放在聊天页而不是资料页，可能是为了降低切页成本；
- 某个语音邀请前先出现节奏提示，可能是为了降低首聊阶段的社交压力；
- 某个 AI 草稿不自动发送，可能是为了避免越权表达。

这类推断可以写，但必须和“页面实际看到了什么”分开。

# 5. 本轮优先级顺序

## 5.1 第一优先级：Chat 深拆

本轮最优先继续深拆 Chat，因为 Soul 报告已经证明：聊天页才是其关系推进的关键层。

### 必拆重点

#### A. 首聊 / 回聊 / 稍后再聊闭环

继续围绕 `奇遇铃-稍后再聊` 深入拆：

- 这个队列保留了哪些人；
- 哪些标签是系统生成，哪些可能是关系推断；
- 首聊建议与回聊建议如何区分；
- 它和聊天详情页、资料页、关注、语音邀请的关系是什么；
- 它在 Soul 中承担的是“补救冷启动”还是“关系再激活”。

#### B. 聊天页关系摘要层

继续拆清：

- 匹配度、共同点、认证、星球、引力签等内容如何组合；
- 哪些信息是为了降低尴尬、降低社交压力；
- 哪些信息是为了引导继续了解；
- 哪些信息更像“聊天前摘要”而不是“完整资料”。

#### C. 输入区 AI / 语音 / 文本节奏

重点拆：

- AI 入口在输入区中的位置；
- 语音邀请前后是否存在节奏提示、关系前提或 follow-up；
- 首聊阶段为何更适合文字、何时才切换语音；
- 表情、更多、语音、AI 的层次关系。

#### D. 聊天设置与查看主页浅层联动

在不进入高风险区的前提下，尽量拆：

- `聊天设置` 的浅层信息结构；
- `查看主页` 与聊天页之间的跳转关系；
- `关注后可邀请通话` 的节奏设计；
- 这些入口是否在服务“关系推进”，而不是纯配置。

### 预期输出

- `SOUL_CHAT_DEEPDIVE_STAGE2_REPORT.md`
- `SOUL_CHAT_RELATIONSHIP_PROGRESS_MAP.md`
- 对应截图与 XML 证据目录

## 5.2 第二优先级：Discover 深拆

第一阶段已经确认 Discover 是复合入口层，本轮要继续拆清它的“导流与治理逻辑”。

### 必拆重点

#### A. 推荐 / 同城 / 关注（或话题）分栏逻辑

拆清：

- 三个分栏分别承担什么；
- 用户从这里最容易走向的动作是什么；
- 哪些是曝光型内容，哪些是关系型内容；
- 同城与推荐的权重差异在 UI 上如何表达。

#### B. 内容卡到聊天的转化路径

拆清：

- `私聊` 按钮在卡片中承担什么角色；
- 点赞、评论、关注、私聊、分享、举报这些动作为什么这样排列；
- 什么动作是正向社交，什么动作是治理，什么动作是低压探索；
- 是否存在“直接进资料”与“直接进私聊”的不同节奏。

#### C. Search / 热词 / 提示卡逻辑

拆清：

- 搜索页为什么不只是空搜索框；
- 热词、恋爱提示卡、推荐提问承担什么作用；
- 这是“内容发现”还是“社交破冰辅助”；
- EliteSync 后续该如何借鉴而不变成娱乐热榜。

#### D. Share / Report / Dislike action sheet

拆清：

- 为什么分享与举报在同一个 sheet 中；
- 为什么内外部动作并列；
- “不喜欢 / 举报 / 复制链接 / 关注 / 私聊” 的排列说明了什么；
- 对 EliteSync 的轻治理动作应该如何设计。

### 预期输出

- `SOUL_DISCOVER_DEEPDIVE_STAGE2_REPORT.md`
- `SOUL_DISCOVER_CONVERSION_AND_GOVERNANCE_MAP.md`
- 对应截图与 XML 证据目录

## 5.3 第三优先级：Me / 个人经营中枢深拆

本轮需要把 Soul 的 Me 从“资料页”进一步拆成“个人经营系统”。

### 必拆重点

#### A. 个人经营区与 AI 助理关系

拆清：

- AI 助理是在做资料体检、内容经营、魅力反馈，还是社交建议；
- `主页吸引力`、任务、帮发、展示建议之间的关系；
- 这是运营化设计，还是用户表达辅助。

#### B. 标签体系与资料层级

拆清：

- 人格、职业状态、交友态度、感情状态这类标签如何组合；
- 哪些标签服务于“快速理解我”；
- 哪些标签服务于“关系推进”；
- 哪些标签服务于展示而不是判断。

#### C. 内容、收藏、赞过、AI 助理、功能中心的整体 IA

拆清：

- 为什么内容经营能力和资料展示放在同一页；
- 为什么函数入口会进入功能中心；
- Soul 是如何把“自我表达”与“功能运营”叠加的；
- EliteSync 该学哪些，不该学哪些。

#### D. 个人空间外观层

继续分析：

- 外观层与真实设置如何分层；
- 图片背景、互动背景、装扮等能力是如何与安全设置隔离的；
- 这类能力是否应该在 EliteSync 中保持免费轻量预览层。

### 预期输出

- `SOUL_ME_DEEPDIVE_STAGE2_REPORT.md`
- `SOUL_PERSONAL_OPERATING_HUB_MAP.md`
- 对应截图与 XML 证据目录

# 6. 每个子模块都必须回答的问题

本轮不论拆哪个模块，报告里都必须尽量回答以下统一问题：

## 6.1 页面角色

- 这是曝光页、筛选页、转化页、回流页、资料页、经营页还是治理页？

## 6.2 上下游关系

- 它从哪里来；
- 最自然导向哪里；
- 它和哪些页面形成闭环；
- 关闭 / 返回之后回到哪里。

## 6.3 组件层级

- 顶部入口、中部内容、底部动作分别服务什么；
- 哪些组件是主动作，哪些是辅助动作，哪些是治理动作。

## 6.4 社交节奏

- 这是在鼓励曝光、打招呼、继续了解、回聊、关注、邀请语音还是表达自我？

## 6.5 设计逻辑推断

- 为什么这个动作放在这里；
- 为什么不是放在别的页面；
- 为什么是轻量提示而不是强制流程；
- 为什么先文字后语音；
- 为什么先资料再通话或先关注再邀请。

## 6.6 EliteSync 参考价值

- 哪些结论适合直接吸收；
- 哪些只适合作为结构启发；
- 哪些阶段性不适合做；
- 哪些明确不能学。

# 7. 对“可能实现方式”的分析要求

本轮除了页面拆解，还必须补一层“可能实现方式与设计逻辑分析”。

## 7.1 可推断但不能过度想象

允许分析的内容：

- 可能的数据分层；
- 可能的状态流；
- 可能的前端 IA 与后端读写关系；
- 可能的推荐/排序/回流设计；
- 可能的权限申请策略；
- 可能的通知与跳转模型；
- 可能的 AI 草稿工作方式；
- 可能的 voice rhythm gating。

禁止：

- 伪装成已知源码事实；
- 编造后端表结构细节；
- 声称已知 Soul 内部实现；
- 把产品推断写成确定结论。

## 7.2 推荐的实现分析维度

每个高价值功能尽量补以下推断：

### A. 可能的数据对象

例如：

- 稍后再聊队列可能需要什么数据；
- 聊天关系摘要可能需要什么聚合信息；
- AI 草稿建议可能基于什么输入；
- 个人经营页的展示建议可能来自哪些用户资料字段。

### B. 可能的状态流

例如：

- 内容卡 -> 私聊 -> 回聊建议 -> 语音邀请；
- 个人资料修改 -> AI 展示建议更新；
- 搜索热词 -> 内容浏览 -> 关注 / 私聊；
- 设置外观 -> 权限申请 -> 媒体选择 -> 背景更新。

### C. 可能的前后端边界

例如：

- 哪些应是服务端真值；
- 哪些只应是本地展示态；
- 哪些可能是可缓存内容；
- 哪些可能必须从服务端重新拉取。

### D. 可能的产品逻辑

例如：

- 为什么 AI 草稿只生成、不自动发送；
- 为什么语音需要节奏提示；
- 为什么 Discover 把举报、不喜欢、分享、私聊收口到一个动作层；
- 为什么 Me 把资料、内容、AI 助理、标签放在一个经营中枢里。

# 10. 风险与纪律

## 10.1 低风险优先

本轮继续保持低风险原则：

- 不发送真实消息；
- 不支付；
- 不删除；
- 不进行高风险账号操作；
- 不授权不必要权限；
- 不进入明显超出产品结构分析价值的路径。

## 10.2 遇到 blocker 的处理方式

如遇：

- 权限拦截；
- 会员付费拦截；
- 账号安全深层页；
- 删除账号；
- 商业化深链；
- 需要真实互动才可继续的路径；

则：

1. 停在当前安全边界；
2. 记录 blocker 页面；
3. 输出 blocker report；
4. 不为追求“继续拆”而越界。

## 10.3 不允许只凭猜测写结论

任何“可能实现方式”分析，都必须与页面实际观察分开写。

报告中建议固定分栏：

- `观察事实`
- `推断`
- `不确定点`
- `对 EliteSync 的启发`
