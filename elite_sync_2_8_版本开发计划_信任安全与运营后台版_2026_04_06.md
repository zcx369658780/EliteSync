# EliteSync 2.8 版本开发计划（信任安全与运营后台版）

更新时间：2026-04-06  
适用版本：`2.8`  
当前基线版本：`0.02.06`（2.7 已结项）  
当前阶段定位：**信任安全与运营后台补完版**

---

## 0. 版本摘要

### 0.1 一句话目标

**2.8 的目标不是继续美化慢约会体验，而是为真实用户接入建立“可信准入、可审核、可管理、可观测”的最小后台与安全闭环。**

### 0.2 这一版要补的核心能力

1. 用户举报 / 拉黑 / 封禁 / 申诉闭环
2. 内容审核与资料审核最小后台
3. 认证架构占位与人工审核链路
4. 运营后台 MVP
5. 核心指标看板与事件埋点收口

### 0.3 2.8 与 2.7 的关系

2.7 已经把：

- Drop 倒计时
- 揭晓与盲盒分层解锁
- 破冰问题
- 匹配后反馈

做成了完整体验链。

因此 2.8 不再以“产品仪式感”作为第一优先级，而是进入：

- 用户接入可信度
- 风险用户处理能力
- 后台运营效率
- 数据可见性
- 小规模真实测试支撑能力

## 1. 版本定位

## 1.1 当前产品所处位置

截至 2.7 结束，EliteSync 已经具备：

- 注册 / 登录
- 问卷作答
- 每周匹配
- 匹配解释
- 双向确认
- 聊天
- 首页内容流
- 资料编辑
- 玄学总览 / 八字 / 星盘 / 紫微 / 诊断
- Drop 倒计时 / 揭晓 / 盲盒解锁 / 破冰问题 / 匹配后反馈
- smoke / release / rollback / 回归门禁

这意味着产品已经具备了“可用 + 有体验”的前台闭环，但还不具备“可放心接入真实用户”的后台治理能力。

## 1.2 2.8 的核心问题

目前的缺口主要不是用户看不到什么，而是团队管不住什么：

- 没有完整的举报与封禁闭环
- 没有人工审核台来处理风险资料
- 没有运营后台快速查看用户与匹配状态
- 没有清晰的核心指标来判断产品卡点
- 没有一套可扩展的认证 provider 架构

因此 2.8 的目标是补足：

> **准入 -> 审核 -> 风控 -> 运营 -> 指标** 这条治理链。

## 1.3 2.8 不做的事情

本版本默认不做：

- 不大改前台匹配体验主流程
- 不重构聊天底层
- 不重新设计问卷体系
- 不推进大规模商业化付费
- 不全面上线学信网 / 高校邮箱真实外部接口（可先做 provider 抽象与人工审核占位）
- 不大改数据库核心 schema，除非极小、明确、可回滚
- 不把 2.8 做成“大而全的管理系统”

## 2. 版本目标

## 2.1 产品目标

让团队在 2.8 后具备以下能力：

1. 能处理违规用户与投诉
2. 能审核资料与认证状态
3. 能快速查看用户、匹配、反馈的基本情况
4. 能通过数据看出产品漏斗问题
5. 能为后续真实用户灰度测试建立最小治理能力

## 2.2 业务目标

2.8 重点建立并观察以下指标：

- 注册完成率
- 问卷完成率
- 本周 Drop 打开率
- 单边确认率
- 双边确认率
- 首条消息发送率
- 举报率
- 拉黑率
- 封禁率
- 反馈提交率
- 7 日留存

## 2.3 工程目标

- 不破坏 2.6.4 的 smoke / release / rollback 门禁
- 不破坏 2.7 的体验主链
- 审核、举报、反馈、用户管理能力尽量模块化，后续可替换或扩展
- 后台能力先 MVP，不追求一次做满

## 3. 2.8 总体结构

## 3.1 总体链路

```text
前台用户
  -> 注册 / 登录 / 填资料 / 匹配 / 聊天
  -> 举报 / 拉黑 / 反馈
  -> 审核相关状态展示

后台运营
  -> 用户列表
  -> 认证/资料审核
  -> 举报工单处理
  -> 风险动作（封禁/解封/备注）
  -> 数据看板
```

## 3.2 治理状态机（建议）

### A. 用户状态 User Moderation State

- `normal`：正常
- `reported`：被举报待处理
- `under_review`：审核中
- `restricted`：受限
- `banned`：封禁
- `appealing`：申诉中
- `restored`：恢复

### B. 认证状态 Verification State

- `not_submitted`：未提交
- `pending_manual_review`：待人工审核
- `approved`：通过
- `rejected`：驳回
- `resubmission_required`：需补充材料

### C. 举报工单状态 Report Ticket State

- `new`：新建
- `triaged`：已分级
- `investigating`：处理中
- `action_taken`：已处理
- `dismissed`：驳回
- `closed`：关闭

## 4. 2.8 五大模块

---

## 4.1 模块一：举报 / 拉黑 / 封禁 / 申诉闭环

### 目标

让前台用户有最小可用的安全出口，让后台运营有最小可执行的风险处理能力。

### 前台需要具备的能力

- 举报用户
- 举报聊天内容
- 拉黑用户
- 查看“已拉黑”状态（最低限度）
- 在被限制或封禁时收到明确状态说明

### 后台需要具备的能力

- 查看举报列表
- 查看举报详情
- 标记优先级
- 记录处理结论
- 执行封禁 / 限制 / 忽略
- 添加人工备注

### 申诉建议

先做最小版：

- 封禁用户可提交简短申诉说明
- 后台可查看并恢复 / 驳回

### 完成定义

- 用户可以举报与拉黑
- 后台能查看并处理举报
- 封禁动作可执行
- 申诉最小入口可用

---

## 4.2 模块二：认证架构占位与人工审核链路

### 目标

先建立“可扩展的认证架构”，不急着在 2.8 全量接通真实外部平台。

### 设计原则

- 先做 provider abstraction
- 先支持人工审核流程
- 再逐步接通真实外部认证能力

### 建议支持的 provider 抽象

- `phone_basic`
- `school_email_placeholder`
- `student_status_manual_review`

### 前台能力

- 提交认证资料入口
- 显示认证状态
- 驳回后提示补充或重提

### 后台能力

- 审核列表
- 审核详情
- 通过 / 驳回 / 补充材料
- 备注原因

### 完成定义

- 认证状态不再只是静态展示
- 团队能通过后台完成最小人工审核
- 架构上为未来真实接入预留清晰入口

---

## 4.3 模块三：运营后台 MVP

### 目标

为运营、测试与顾问协作提供一个最小可用的管理入口。

### MVP 建议四块能力

#### A. 用户管理
- 用户列表
- 搜索用户
- 查看基础状态
- 查看认证状态
- 查看封禁状态

#### B. 举报管理
- 举报列表
- 举报详情
- 处理动作

#### C. 审核管理
- 资料审核
- 认证审核
- 审核备注

#### D. 数据看板
- 注册
- 问卷完成
- 匹配打开
- 双边确认
- 首条消息
- 举报
- 反馈

### 技术建议

- 后台先做内部可用，不追求复杂权限系统
- 页面先以列表 + 详情抽屉 / 详情页为主
- 优先保证查询效率与处理效率

### 完成定义

- 团队可以通过后台处理用户与举报，而不必直接查库
- 团队可以通过后台看见核心指标，而不必靠口头汇报

---

## 4.4 模块四：内容审核与资料审核最小闭环

### 目标

避免头像、昵称、资料文本、聊天内容完全裸奔。

### 范围建议

先做最小闭环：

- 头像审核状态
- 昵称 / 简介文本审核状态
- 聊天内容举报后的人工复核入口

### 不建议 2.8 做得过重的内容

- 不要一开始做全量复杂 AI 审核平台
- 不要一开始做实时重风控阻断全链路
- 不要让审核策略拖垮聊天主体验

### 完成定义

- 用户资料有最小审核入口
- 举报后的聊天内容可以被人工查看并记录处理结论

---

## 4.5 模块五：埋点与核心指标看板

### 目标

让团队第一次能比较客观地看到“产品到底卡在哪”。

### 看板第一版建议

## 8. 2.8 用户故事

### 故事 1：用户安全
作为用户，我希望遇到骚扰或异常对象时，能立刻举报或拉黑，而不是被迫继续接触。

### 故事 2：被处理说明
作为被限制的用户，我希望知道自己当前状态与申诉入口，而不是完全没有反馈。

### 故事 3：运营处理
作为运营人员，我希望不用查数据库就能处理举报、查看认证和标记用户状态。

### 故事 4：认证占位
作为团队，我希望现在先用人工审核跑通流程，为未来学信网 / 高校邮箱接入留接口。

### 故事 5：数据判断
作为产品负责人，我希望看到注册、问卷、Drop、双向确认、首条消息、举报、反馈这些核心指标，帮助判断产品下一步重点。

## 9. 2.8 验收标准

## 9.1 功能验收

- 用户能举报和拉黑

- 举报后能产生工单
- 工单状态可流转
- 封禁 / 限制状态能生效
- 审核通过 / 驳回状态能正确展示
- 后台查询失败不拖垮前台体验

## 9.3 工程验收

- 2.6.4 smoke / regression 不倒退
- 2.7 核心体验不倒退
- 资料保存与画像链路不受影响
- 聊天主链路不受影响
- 新后台模块失败不影响前台主功能
